# 1. U3 신규 패턴

## PAT-U3-01 외부 어댑터 서킷브레이커·폴백 체인 — 벤더 폴백 D08 `[U3]`

**문제**: U3의 외부 의존(카카오·TourAPI·네이버)은 우리가 제어할 수 없는 가용성을 가진다. 무한 대기는 탐색 스레드 고갈로 증폭되고, 벤더 1종 장애가 타 벤더 호출로 전파되면 격리가 무너지며, 침묵 실패는 사용자를 빈 화면에 방치한다(ADR-0011). 외부 소스는 전부 Medium(§6.3)이므로 **폴백이 필수**다.

**적용(설계)**: 전 외부 호출은 포트 뒤에서 U1 **PAT-RES-01 4요소 계약**을 의무화하고(계약 누락은 설계 리뷰 차단), Resilience4j(U3-S1)로 구현한다. 벤더별 **독립 서킷**(벌크헤드) + **폴백 체인**(D08 순서):

| 의존 (Port) | 타임아웃 | 재시도 | 서킷 | 폴백 체인 |
|---|---|---|---|---|
| 카카오 장소 검색(`PlaceSearchPort`) | 3초 | 없음(쿼터 보호 — 사용자 재시도) | 벤더 독립 | ① 네이버 2차(D08) → ② TTL 캐시 잔존 → ③ '확인 불가'(G8) |
| 카카오 지오코딩(`GeocodingPort`) | 3초 | 없음 | 벤더 독립 | ① 네이버 2차 → ② 수동 핀 지정(US-E3-08) |
| TourAPI(`TourContentPort`) | 3초 | 없음 | 벤더 독립 | ① 카카오 검색 대체 → ② 정본 테이블 잔존 → ③ '미확인' 배지 |
| 네이버(2차 폴백) | 3초 | 없음 | 벤더 독립 | TTL 캐시 잔존 → '확인 불가' |
| OTA(`DeeplinkBuilder` — 외부 호출 없음) | N/A | N/A | N/A | 링크 불가 시 타 OTA(D17)/숙소명 복사 |

```text
[탐색 요청]
  → 캐시 조회(정본 + TTL) ──히트──> 응답(외부 미호출)
  → 미스: 카카오·TourAPI 병렬 팬아웃 (어댑터별 3초·독립 서킷)
       ├ 카카오 실패/open ──> 네이버 2차(D08) ──> TTL 캐시 잔존 ──> '확인 불가'(G8)
       └ TourAPI 실패/open ──> 정본 테이블 잔존 ──> '미확인' 배지
  → 일부 성공: 부분 응답 + "일부 정보가 빠졌을 수 있어요"(US-E3-11, 침묵 실패 금지)
  → 전체 실패: "지금 불러올 수 없어요" + 재시도 + [직접 등록] 우회(US-E3-08·10·11)
```

**설계 원칙**: (a) 재시도는 멱등 GET이라도 **쿼터 보호 우선으로 미적용**(사용자 버튼이 재시도), (b) 서킷 open/half-open은 계측(A12), (c) 폴백은 반드시 사용자 고지·계측과 짝(침묵 실패 금지), (d) 폴백 순서·타임아웃은 remote config(OPS-01 변경 관리), (e) **최종 폴백은 항상 수동 등록 우회**(NFR-U3-RES-03 — 외부 장애가 여정을 완전 차단 불가). **TMap 경계**: U3 거리는 직선거리 로컬 계산(G34)이라 TMap 미호출 — 라우팅 폴백(G106)은 C2/U5(NFR-U3-RES-04).

**U3 적용 지점**: M7 어댑터(`PlaceSearchPort`·`GeocodingPort`·`TourContentPort`)·네이버 폴백 어댑터, M3 탐색 오케스트레이션(병렬 팬아웃·부분 응답), M5 `DeeplinkBuilder`. 구현은 Resilience4j(U3-S1), fake 실패 주입 테스트(D37).

**검증 기준**: RESILIENCY-10 — 전 어댑터 명시 타임아웃·독립 서킷·폴백 체인(코드 리뷰+계약 테스트), fake로 타임아웃·5xx·서킷 open 주입 시 폴백 실동작(카카오 다운→네이버, 전체 다운→수동 등록 우회 노출) PR CI 통과. RESILIENCY-09 — 서킷/쿼터 계측이 A11·A12에 존재.

## PAT-U3-02 POI 하이브리드 캐싱 — 정본 영구 vs 실시간 TTL (소스별 D13) `[U3]`

**문제**: 지도 API 약관은 **영구 캐싱을 금지**(D13)하고, 그 적법성(스냅샷 영구 저장 포함)은 **P2 법무 확인 전까지 미확정**이다. 동시에 매 탐색마다 외부를 재호출하면 지연·쿼터(G196)를 소모한다. 소스별로 저장 정책이 갈리는데 이를 코드가 혼동하면 약관 위반(법적 리스크)이 된다.

**적용(설계)**: 저장 정책을 **소스 태그로 코드·스키마 레벨에서 분기**(혼동 원천 차단):

| 소스 | 저장 | TTL·정리 | 근거 |
|---|---|---|---|
| TourAPI 등 **허용 공공 소스** | **정본 테이블 영구**(RDS) | 일 1회 배치 갱신(G196) | D13 (a) · **P4 캐싱 조건 확인** |
| 지도 API(카카오·네이버) | **인메모리 TTL만**(Caffeine, 영구 저장 표면 0) | 약관 허용 TTL·만료 시 재조회 강제 | D13 (b) · **P2 지도 약관** |
| 사용자 확정 스냅샷 | 최소 필드 영구 + **보존 정책 컬럼**(source·retention_policy·cached_at) | 정책 변경 시 소급 축소 | D13 · **P2 적법성** |
| 라이브 정확 가격 | **저장 안 함** | 조회 후 폐기 | D09 |

**설계 원칙**: (a) **영구 캐싱 금지 강제** — 지도 소스는 인메모리 TTL 외 영구 경로가 코드에 존재하지 않게(아키텍처 테스트로 "지도 소스 → 정본 테이블 쓰기" 검출), (b) canonical POI ID 무결성(좌표 50m + 유사도, G133/G148)은 **하드 제약**(NFR-U3-DATA-02) — 매칭 멱등·대칭 PBT, (c) 캐시는 정본이 아님(미스=재호출, NFR-U1-AV-03 상속) — 단 TourAPI 정본 테이블은 외부 순단 시 폴백 소스(PAT-U3-01), (d) **출처 표기**(D13). **P2·P4 대응**: 정책 컬럼으로 약관 확정 후 보존 범위를 소급 조정, 지도 소스 스냅샷은 P2 확정 전 보류(NFR-U3-DATA-03).

**U3 적용 지점**: M7 정본 저장소(정본 테이블·소스 매핑 D17)·인메모리 TTL 캐시·스냅샷 발급(값 객체, component-dependency §3.1 소유권 요약 — 약관 리스크를 M7 어댑터 한 곳에 격리), canonical ID 발급기(유사도 매칭 U3-S5).

**검증 기준**: D13 — 소스별 저장 경로 분기 테스트(지도 소스가 정본 테이블에 영구 저장되지 않음 아키텍처 테스트), TTL 만료 후 재조회 강제 PBT(unit-of-work.md U3 DoD). NFR-U3-DATA-02 — canonical 매칭 멱등·대칭 PBT + 골든 데이터셋 회귀(오매칭=그라운딩 오염 하드 제약). §6.5 지도 약관 — 출처 표기·영구 캐싱 금지 확인.

## PAT-U3-03 라이브 가격 온디맨드 호출 — 목록 일괄 금지 `[U3]`

**문제**: 정확 가격·재고는 외부 OTA가 확정하며 변동한다(US-E3-01). 목록 전체를 일괄 조회·캐싱하면 (a) 파트너 약관·쿼터(G196)를 위반하고, (b) 낡은 가격을 저장해 사용자를 오도한다(D09 — 정확 가격 캐싱 금지).

**적용(설계)**:

- **온디맨드 단건**: '가격 보기' 탭 **시점에만** 날짜·인원 단건 조회(G33) — 목록 렌더 시 일괄 조회 금지. 입력값은 **세션 내 재사용**(재입력 마찰 감소)하되 **결과는 저장하지 않는다**(D09). 타임아웃 5초(단건, NFR-U3-PERF-04).
- **호출 억제**: 사용자·세션 단위 rate-limit(U1 PAT-SEC-03 레이트리미터 재사용 — LC-3 전역 컴포넌트, "후속 유닛의 고비용 엔드포인트가 동일 메커니즘 재사용"에 해당) → 파트너 약관 최소치 준수(G196).
- **MVP 실현 경계**: 1차는 OTA API 계약(P5) 전이므로 '가격 보기'가 **날짜·인원 파라미터를 담은 OTA 검색 딥링크 핸드오프**로 실현될 수 있다(정확 가격은 OTA 측 확정, D09). 파트너 가격 API 확보 시 동일 온디맨드 계약으로 교체(어댑터 뒤 격리).

**U3 적용 지점**: M3 '가격 보기' 유스케이스(바텀시트·세션 입력), M5 딥링크(날짜·인원 파라미터), 사용자별 rate-limit 필터.

**검증 기준**: D09 — 목록 렌더 시 일괄 가격 조회 미발생·정확 가격 미저장 테스트. G196 — rate-limit 발동 테스트. G33 — 세션 내 입력 재사용·결과 비저장 확인.

## PAT-U3-04 URL 파싱 SSRF 가드 — 화이트리스트·fetch 없음 `[U3]`

**문제**: 사용자가 붙여넣는 OTA 링크(US-E3-08)는 **신뢰할 수 없는 입력**이다. 서버가 이 URL을 fetch하면 내부망·메타데이터 엔드포인트 접근(SSRF)의 문이 열리고, 파싱 결과를 무검증 신뢰하면 오염된 좌표·명칭이 등록된다.

**적용(설계)**: **fetch 없는 패턴 파싱 + 다층 가드**(G31):

- **SSRF 원천 차단**: OTA 링크는 **URL 패턴 파싱만 · 페이지 fetch 없음**(G31) — 사용자 제공 URL로의 **아웃바운드 요청이 코드 경로에 존재하지 않는다**(SSRF 구조적 불가). 아키텍처 테스트로 "붙여넣은 URL을 인자로 하는 HTTP 호출" 부재 검출.
- **입력 검증(SECURITY-05)**: (a) 스킴 http/https만, (b) **지원 도메인 화이트리스트**(remote config) 내 URL만 파싱, (c) 길이 상한·형식 검증, (d) 화이트리스트 밖·파싱 실패는 즉시 **수동 핀 지정 흐름 폴백**(US-E3-08 — 침묵 실패 금지).
- **파싱 결과 불신**: 추출한 숙소명·좌표도 신뢰하지 않고 **지오코딩 재검증**(GeocodingPort) + 국내 좌표 범위 검증(G120) 후 확정.

**U3 적용 지점**: M4 링크 파싱 유스케이스(US-E3-08 경로 2), 화이트리스트 remote config, 폴백 라우팅(수동 핀).

**검증 기준**: G31/SECURITY-05 — 화이트리스트 밖 URL 전부 거부·파싱 실패 시 폴백 경로 PBT(unit-of-work.md U3 DoD), 붙여넣은 URL 대상 아웃바운드 호출 부재 아키텍처 테스트(SSRF 구조적 차단), 파싱 좌표의 지오코딩 재검증·좌표 범위 차단 테스트.

## PAT-U3-05 복귀 핸드오프 상태 머신 `[U3]`

**문제**: OTA 딥링크 이탈 후 예약했는지 앱은 알 수 없다(포스트백 자동 등록은 1차 제외, D09/G29). 그렇다고 방치하면 등록 온램프가 끊기고, 반복 노출하면 사용자를 괴롭힌다(G32 — 1회만).

**적용(설계)**: **최근 이탈 숙소 1건 기준 상태 머신**(G32):

```text
[딥링크 이탈] → 클릭 로그 기록(숙소 ID·이탈 시각) → 상태: PENDING_RETURN(최근 1건만)
   │  (24시간 창)
[24h 내 첫 복귀] → 핸드오프 카드 1회 노출("방금 본 [숙소명], 예약하셨나요? → 거점 등록")
   ├ 사용자 [등록] → 등록 폼 자동 채움(US-E3-06) → 상태: CONSUMED
   └ 사용자 무시/닫기 → 상태: DISMISSED (재노출 없음)
[24h 경과] → 상태: EXPIRED (노출 없음)
```

- **설계 원칙**: (a) **최근 이탈 숙소 1건만** 추적(다중 누적 금지, G32), (b) 24h 창·첫 복귀·1회 노출을 상태 전이로 명문화(무시 후 재노출 없음), (c) 클릭 로그(아웃바운드 클릭)는 소유권 스코핑(accountId) + 제휴 고지 계측, (d) 결정성은 clock 주입 순수 함수로 테스트(창 경계·복귀 판정).

**U3 적용 지점**: M5 클릭 로그·복귀 판정, 핸드오프 카드(클라이언트), 등록 폼 자동 채움 연결(M4).

**검증 기준**: G32 — 24h 내 첫 복귀 1회 노출·무시 후 재노출 없음·경계(24h 직전/직후) 상태 전이 테스트(clock 주입 결정성), 최근 1건만 추적 확인, 클릭 로그 소유권 스코핑 테스트(SECURITY-08 상속).

## PAT-U3-06 커버리지 게이트 데이터 품질 검증 `[U3]`

**문제**: TourAPI 데이터 품질(좌표 확보율·영업시간 채움률)이 낮으면 탐색·일정 생성 그라운딩이 오염되고(unit-of-work.md U3 리스크), 이를 측정하지 않으면 "출시해도 되는가"를 답할 수 없다. 빈 필드를 방치하면 사용자에게 고장 인상을 준다.

**적용(설계)**:

- **정량 출시 게이트(G192)**: 지역별 **좌표 확보율 ≥ 95% · 영업시간 채움률 ≥ 70%**를 측정하는 **배치 잡**(기존 ShedLock 스케줄러 재사용, infrastructure-design.md §5)이 커스텀 메트릭 발행 → 대시보드·게이트 판정. 미달 지역은 출시 제한(운영 판단).
- **'미확인' 표기(ADR-0011)**: 누락·미달 필드는 빈칸이 아니라 **'미확인'**(US-E3-03·G199 통일). 좌표 미확정 등록은 "지도에서 위치 확인" 강제(US-E3-06·08).
- **영업시간 갱신 경계**: 정기 변경만 자동 감지(TourAPI·지도 재조회, 일 1회) — U3는 정본 영업시간 채움·측정까지, Plan-B 트리거 승격은 M9(U6, G192).

**U3 적용 지점**: M7 커버리지 측정 잡, 정본 저장의 채움률 계측, '미확인' 표기 규약(클라이언트·M7 DTO).

**검증 기준**: G192 — 지역별 좌표·영업시간 채움률 산출 정확성 테스트, 게이트 임계(95%/70%) 판정 로직 테스트, 미달·누락 필드의 '미확인' 표기 테스트(빈칸 아님). RESILIENCY-05 — 커버리지 메트릭이 대시보드에 존재(SI §8 편입).
