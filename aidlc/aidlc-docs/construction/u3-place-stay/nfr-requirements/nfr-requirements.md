## 0. U3 NFR 프로필 — "데이터 척추 · 외부 의존 최다"의 NFR 본질

U3의 NFR 본질은 **우리가 제어할 수 없는 외부 데이터 소스 위에서 사용자 경험 품질과 적법성을 지키는 것**이다:

1. **복원력(핵심)** — 외부 API 4종(카카오·네이버·TourAPI·OTA 아웃바운드)의 가변 가용성이 탐색·등록을 마비시키지 않게 한다(RESILIENCY-10, M3·M5 = Medium §6.3 — 폴백 필수).
2. **캐싱·적법성** — D13 하이브리드 저장(TourAPI 정본 영구 + 지도 API TTL·영구 캐싱 금지)을 NFR·인프라로 강제하되, **P2 지도 약관·P4 TourAPI 캐싱 조건**이 확정될 때까지 영구 저장 표면을 최소화한다.
3. **데이터 품질 게이트** — TourAPI 좌표 확보율·영업시간 채움률이 탐색·일정 품질을 좌우하므로 **출시 게이트(G192)** 를 측정 파이프라인으로 구현한다.
4. **입력 방어** — 사용자 붙여넣기 URL(G31)·외부 응답 스키마는 신뢰할 수 없는 입력이다(SECURITY-05, SSRF·오매칭 방어).

보안·무상태·관측성·배포의 서버 규약은 **U1 전역 정본을 그대로 상속**하고, U3는 외부 어댑터 계층의 증분만 소유한다.

### 0.1 워크로드 중요도 (requirements.md §6.3)

| 컴포넌트 | 중요도 | 불가용 시 영향·U3 대응 |
|---|---|---|
| 숙소 탐색(M3·M5) | **Medium** | 계획 단계 기능 — **수동 등록 우회 존재**(US-E3-08). 외부 소스 전체 실패 시에도 직접 등록으로 여정 지속 |
| POI 정본(M7) 공급 | 기반(Medium) | U4~U8이 이 데이터 위에서 동작 — 캐시 정본/스냅샷으로 외부 순단 흡수(component-dependency §2.1) |
| 등록 숙소(M4)·위시리스트 | 조회/저장은 Critical 경로(DB·Auth) 상속 | 자체 데이터 — 외부 의존 없음. 소유권 검증(SECURITY-08)만 증분 |

**핵심 원칙**: 외부 의존은 전부 **Medium 이하 경로**이며(§6.3 "외부 의존성은 전부 High 이하, 폴백 필수"), 어떤 외부 소스도 U3의 가용성 바닥이 되어서는 안 된다 — 마지막 폴백은 항상 **수동 등록·직접 입력**(사용자 자체 데이터).

## 1. 성능 (D38, §6.1)

§6.1이 U3에 직접 부과하는 목표는 **일반 화면 전환 300ms**뿐이며(AI 생성은 U5·U6), 이는 클라이언트 내비게이션 예산이다. U3 성능 요구의 본질은 **외부 API 지연이 탐색 체감을 지배하지 않게 타임아웃·부분 응답·캐시로 분리하는 것**이다.

### 1.1 U3 API 응답 목표 (제안 — 서버 처리 시간 기준, 네트워크 RTT 별도)

| API (M3/M7 인터페이스 그룹) | p50 | p95 | 상한(타임아웃) | 비고 · 추적 |
|---|---|---|---|---|
| 숙소·POI 탐색(캐시 히트 — 정본/TTL 캐시 서빙) | 100ms | 300ms | 3초 | 인메모리/정본 조회 — 외부 미호출(§4.2 캐싱) |
| 숙소·POI 탐색(캐시 미스 — 외부 병렬 호출) | 800ms | 3,000ms | **어댑터별 3초 + 부분 응답** | 카카오·TourAPI **병렬** 호출(직렬 금지) — 어느 하나 지연·실패해도 가용 소스로 부분 응답(NFR-U3-PERF-03·US-E3-11) |
| POI 후보 풀 조회(closed-set — M8 그라운딩 공급) | 100ms | 300ms | 3초 | 정본/캐시 조회 — 외부 미호출(component-dependency §2.1, G115) |
| 라이브 가격 조회('가격 보기' — 온디맨드 단건) | — | — | **5초(단건)** | **목록 일괄 조회 금지·캐싱 금지**(D09·G33) — 탭 시 날짜·인원 단건만, 세션 내 입력 재사용(§1.3) |
| 지오코딩·역지오코딩(등록 좌표 확정) | 200ms | 800ms | 3초 | 카카오 지오코딩 → 네이버 폴백(D08). 등록 크리티컬 패스 아님(비동기 확정 가능) |
| 위시리스트·등록 숙소 CRUD(자체 데이터) | 50ms | 200ms | 3초 | 외부 의존 없음 — U1 CRUD 성능 규약 상속 |

- **NFR-U3-PERF-01 (탐색 캐시 우선)**: 탐색은 **캐시(정본 영구 + 지도 TTL) 우선 서빙**하고, 미스일 때만 외부 병렬 호출한다. 캐시 히트 경로가 지배적이도록 설계해(§4) 외부 지연이 상시 노출되지 않게 한다.
- **NFR-U3-PERF-02 (외부 호출 병렬·타임박스)**: 다중 소스(카카오·TourAPI) 탐색은 **병렬 팬아웃 + 어댑터별 3초 타임아웃**. 총 대기는 최장 어댑터 타임아웃(3초)을 넘지 않는다 — 순차 합산 금지.
- **NFR-U3-PERF-03 (부분 응답 허용 — 침묵 실패 금지)**: 일부 소스 미응답 시 나머지로 목록을 구성하고 "일부 정보가 빠졌을 수 있어요"를 표기한다(US-E3-11). 빈칸이 아니라 **부분 응답 또는 오류·재시도 안내**(ADR-0011).
- **NFR-U3-PERF-04 (라이브 가격 온디맨드)**: '가격 보기'는 **탭 시점 단건 조회만** — 목록 전체 일괄 조회·정확 가격 캐싱을 하지 않는다(D09). 입력(날짜·인원)은 세션 내 재사용(G33)하되 결과는 저장하지 않는다.
- **NFR-U3-PERF-05 (지도 렌더 성능·클라이언트)**: 카카오 지도 SDK(config plugin) 로딩·핀 렌더가 상세·핀 지정 화면의 300ms 전환을 차단하지 않게 — 지도는 지연 로드, 로드 실패 시 목록/좌표 텍스트 뷰 폴백(§2.3, tech-stack §2).

### 1.2 클라이언트 성능

- 탐색 결과는 페이지네이션/무한스크롤(US-E3-01)로 초기 페이로드를 제한 — 지역당 POI 5천(G142)을 한 번에 로드하지 않는다.
- 필터·정렬(직선거리 G34)은 **클라이언트 로컬 계산 우선**(Haversine — 외부 미호출)으로 전환 지연을 네트워크와 분리한다(§1.4).

## 2. 복원력 (핵심 · §6.2·§6.3, RESILIENCY-10, ADR-0011)

U3 복원력의 정본은 **component-dependency.md §6 외부 어댑터 격리 표**와 **U1 PAT-RES-01 4요소 계약(타임아웃+재시도+서킷+폴백)**이다. 본 절은 U3 어댑터별 계약을 설계 정본으로 승격한다.

### 2.1 외부 어댑터 4요소 계약 (RESILIENCY-10 — 계약 누락은 설계 리뷰 차단)

| 의존 (Port · 소유 M7/M5) | 타임아웃 | 재시도 | 서킷 브레이커 | 폴백 체인 (RESILIENCY-10) |
|---|---|---|---|---|
| **카카오 로컬 장소 검색** (`PlaceSearchPort`) | 3초 | 없음(멱등 GET이나 쿼터 보호 우선 — 사용자 재시도 버튼) | 벤더별 독립 서킷(실패율 임계 초과 시 open) | **① 네이버 2차 폴백(D08) → ② TTL 캐시 잔존분 → ③ '확인 불가' 안내(G8)** |
| **카카오 지오코딩** (`GeocodingPort`) | 3초 | 없음 | 독립 서킷 | ① 네이버 2차(D08) → ② **수동 핀 지정 흐름**(US-E3-08 — 좌표 미확정 등록은 "지도에서 위치 확인" 강제) |
| **TourAPI 콘텐츠** (`TourContentPort`, M3 계약 공유) | 3초 | 없음 | 독립 서킷 | ① 카카오 검색 대체 → ② **정본 테이블 잔존분**(허용 소스 영구) → ③ '미확인' 배지(US-E3-03) |
| **네이버 장소**(2차 폴백 전용) | 3초 | 없음 | 독립 서킷 | 폴백의 폴백 — 실패 시 TTL 캐시 잔존분 → '확인 불가' |
| **OTA 딥링크** (`DeeplinkBuilder`, M5 — **외부 호출 없음**) | N/A(URL 조립만) | N/A | N/A | 링크 생성 불가 시 **타 OTA 링크 또는 숙소명 복사 안내**(US-E3-05·D17 다중 OTA) |
| **클라이언트 카카오 지도 SDK**(`shared/map` 브리지) | 지도 로드 타임아웃(클라) | — | — | 지도 로드 실패 시 **목록 뷰/좌표 텍스트 폴백**(unit-of-work.md U3 리스크 완화) |

- **NFR-U3-RES-01 (전 어댑터 타임아웃·서킷 의무)**: 카카오·네이버·TourAPI 각 어댑터는 명시적 타임아웃 + 독립 서킷 브레이커를 가진다(벤더 1종 장애가 타 벤더로 전파 금지 — 벌크헤드). 서킷 open/half-open 전이는 계측 이벤트(SI §8.2 A12).
- **NFR-U3-RES-02 (벤더 폴백 체인 — D08)**: 지도 계열은 **카카오 → 네이버 2차 폴백**(D08 정본)을 어댑터 내부에서 처리 — M3/M7 도메인 로직은 벤더 무지(component-dependency §6 격리 원칙). 폴백 순서는 remote config로 조정 가능(NFR-U1-MT-04 상속).
- **NFR-U3-RES-03 (최종 폴백 = 수동 등록 우회)**: **외부 소스 전체 실패 시** 탐색은 "지금 숙소 정보를 불러올 수 없어요" + 재시도, 그리고 **직접 등록(US-E3-08) 우회 경로**를 제공한다(US-E3-11). 데이터 부족 지역도 동일 — "숙소를 직접 등록하면 일정을 만들어 드려요"(US-E3-10·D09). **어떤 외부 장애도 사용자 여정을 완전 차단하지 않는다.**
- **NFR-U3-RES-04 (거리 = 직선거리 로컬 계산 · TMap 경계)**: U3 거리 필터·정렬은 **검색 여행지 중심 좌표 기준 직선거리(Haversine — 로컬 계산)** 만 사용하며 소요시간을 표시하지 않는다(G34·D25). **따라서 U3는 TMap(도로거리)을 호출하지 않는다.** TMap `RoadDistancePort`·라우팅 실패 시 직선거리×우회계수(G106) 폴백은 **C2 소유로 U5 범위**다(component-dependency §6·§2.3). U3의 직선거리는 폴백이 아니라 **1차 정본**이므로 외부 라우팅 의존 자체가 없다(가용성 우위).
- **NFR-U3-RES-05 (부분 실패·오매칭 격리)**: 소스 부분 실패는 부분 응답(NFR-U3-PERF-03), POI 오매칭(동명·근접 병합 오류)은 **매칭 임계 보수 설정 + 운영자 보정 여지**(D17)로 격리 — 오매칭이 일정 생성 그라운딩을 오염시키지 않게(§5 매칭 검증).

### 2.2 캐시가 가용성 요소가 아님 (NFR-U1-AV-03 상속)

- **NFR-U3-RES-06**: 탐색 TTL 캐시·인메모리 캐시는 **지연 최적화일 뿐 가용성 요소가 아니다** — 캐시 소실은 외부 재호출로 충족한다. 단, TourAPI **정본 테이블**은 캐시가 아닌 정본이며(허용 소스 영구), 외부 순단 시 폴백 데이터 소스로 기능한다(component-dependency §2.1 "캐시 정본 잔존분으로 응답").

## 3. 규모 (§6.8 G142 — DAU 1천 · 지역당 POI 5천)

### 3.1 데이터 볼륨

- **NFR-U3-SC-01 (POI 정본 볼륨)**: 지역당 POI 후보 **5천**(G142). 국내 주요 지역 수십 개 기준 정본 테이블 **수십만 행** 수준 — 단일 PostgreSQL로 충분(SI §3, NFR-U1-SC-03 파티셔닝 여지 상속, 샤딩 불요). 인기 장소 집계·소스 매핑·스냅샷 포함 U3 신규 테이블 ~8~9개(infrastructure-design.md §2).
- **NFR-U3-SC-02 (탐색 트래픽)**: 탐색은 계획 단계 저빈도(Medium §6.3) — DAU 1천 기준 정상 피크 수 RPS 미만. **캐시 히트가 지배적**이면 외부 호출은 캐시 미스·라이브 가격에 한정. SI §1.2 태스크 2개로 흡수(추가 컴퓨트·오토스케일 산정 불요, G142).

### 3.2 외부 호출 rate-limit·쿼터 (G196, RESILIENCY-09)

- **NFR-U3-SC-03 (라이브 가격 호출 상한)**: 라이브 가격 조회는 **파트너 약관 최소치 준수**(G196) — 온디맨드 단건·목록 일괄 금지(D09)로 호출량을 구조적으로 억제. 사용자·세션 단위 rate-limit(U1 PAT-SEC-03 레이트리미터 재사용, LC-3 전역 컴포넌트).
- **NFR-U3-SC-04 (정적 콘텐츠 갱신 주기)**: OTA·TourAPI 정적 콘텐츠는 **일 1회 배치 갱신**(G196) — 실시간 반복 호출 금지. 배치는 기존 ShedLock 스케줄러 프레임(SI §1.2) 재사용.
- **NFR-U3-SC-05 (외부 쿼터 80% 알람)**: 카카오·TourAPI·네이버 각 어댑터 호출량 커스텀 메트릭 대비 문서화된 쿼터의 **80% 알람**(SI §8.2 A11, NFR-U1-SC-04 상속) — U3가 A11·A12의 첫 소비자다. 쿼터 한도는 P4(TourAPI)·벤더 콘솔 확인 후 문서화.

## 4. 캐싱·데이터 (D13 소스별 정책의 NFR화)

D13 하이브리드 저장을 NFR·인프라 강제 규칙으로 승격한다. **적법성(P2·P4)이 이 절을 지배**한다.

### 4.1 소스별 캐싱 정책 (D13 · component-dependency §3.1)

| 소스 | 저장 정책 | NFR 강제 규칙 | 근거·선결 |
|---|---|---|---|
| **TourAPI 등 캐싱 허용 공공 소스** | **정본 테이블 영구 저장**(D13 (a)) | 캐싱 **허용분만** 정본 영구 — 허용 범위는 보존 정책 컬럼으로 소급 축소 가능 | **P4 캐싱 조건 확인 선결** |
| **지도 API(카카오·네이버)** | **실시간 호출 + 약관 허용 TTL 캐시**(D13 (b)) | **영구 캐싱 금지** — TTL 상한 준수, 만료 후 재조회 강제. 영구 저장 표면 최소화(§4.2 인메모리 권고) | **P2 지도 약관 선결** |
| **사용자 확정 스냅샷**(등록 숙소·필수 방문지·방문 체크) | 확정 시점 '사용자 입력 데이터'로 영구 스냅샷(D13) | **스냅샷 필드 최소화** + 소스별 보존 정책 컬럼 | **P2 스냅샷 적법성 법무 확인 선결** |
| **정확 가격(라이브)** | **저장 안 함** | 캐싱 금지(D09) — 조회 시점 소비 후 폐기 | US-E3-01 |

- **NFR-U3-DATA-01 (영구 캐싱 금지 강제)**: 지도 API 소스 데이터는 **TTL 없는 영구 저장을 금지**한다 — TTL·만료 정리·출처 표기를 데이터 모델·인프라 차원에서 강제(§6.5 지도 약관). TTL 만료 속성은 PBT 대상(만료 후 재조회 강제 — unit-of-work.md U3 DoD).
- **NFR-U3-DATA-02 (canonical POI ID 무결성 — 하드 제약 D37)**: 동일 실세계 장소는 **정확히 1개 canonical POI ID**(좌표 근접 50m + 명칭 유사도, G133/G148). 소스별 place_id N:1 매핑(D17 패턴). 매칭 멱등성·대칭성은 PBT·골든 데이터셋 회귀로 상시 검증(unit-of-work.md U3 DoD·리스크 완화). 오매칭은 그라운딩 오염(§2.1)이므로 하드 제약.
- **NFR-U3-DATA-03 (스냅샷 필드 최소화 — P2 대응)**: 사용자 확정 스냅샷은 일정·거점 기능에 필요한 **최소 필드만**(명칭·좌표·카테고리·체류 기본값 등) 저장하고, 소스·보존 정책 컬럼(source·retention_policy·cached_at)을 둔다 — 약관 변경 시 정책 컬럼 기준 소급 대응(unit-of-work.md U3 리스크 완화). **P2 확정 전 지도 API 소스 스냅샷은 보류**(TourAPI 허용분만 영구).
- **NFR-U3-DATA-04 (출처 표기)**: 지도 API·TourAPI 데이터는 화면·응답에 **출처 표기**(D13·§6.5 지도 약관 요건).

### 4.2 캐시 계층 (규모 대비 — 상세 인프라는 infrastructure-design.md §3)

- **NFR-U3-DATA-05**: 캐시는 **2계층** — (1) TourAPI **정본 테이블**(RDS 영구, 허용 소스), (2) 지도 API 소스·핫 탐색 결과의 **단기 TTL 캐시**. 단기 캐시는 **인메모리(Caffeine) 권고**(정본 아님·미스=재호출·영구 저장 표면 0 → P2 리스크 격리·신규 인프라 0). **ElastiCache는 미도입** — 규모(수 RPS·POI 5천/지역)·정본 아님 대비 이득 0(SI §4 트리거와 통합: 실측 50 RPS 초과 또는 U11 팬아웃 시 재평가).

## 5. 보안 (Security Baseline 상속 — §6.4)

U3는 SECURITY 규칙의 신규 1차 구현 지점이 아니다 — 전역 보안 설정(deny-by-default·헤더·에러 핸들러·구조화 로깅·시크릿 관리)은 U1 스캐폴드가 소유하고 U3는 상속한다. U3 증분은 **외부 입력·외부 키·소유권** 세 지점이다.

- **NFR-U3-SEC-01 (외부 API 키 = 시크릿)**: 카카오 REST 키·TourAPI 서비스 키·네이버 클라이언트 시크릿·(U5용 TMap 키는 U5)은 **소스·이미지·설정 파일에 미포함** — Secrets Manager 주입(NFR-U1-SEC-11 상속, PAT-SEC-05). 어댑터별 키 분리, 시크릿 스캔 CI 게이트(SECURITY-10).
- **NFR-U3-SEC-02 (URL 파싱 SSRF 방어 — G31)**: OTA 링크 붙여넣기는 **URL 패턴 파싱만 · 페이지 fetch 없음**(G31) — 사용자 제공 URL로의 아웃바운드 요청이 **구조적으로 발생하지 않으므로 SSRF가 원천 차단**된다. 추가 가드: (a) **지원 도메인 화이트리스트**만 파싱, (b) URL 형식·길이·스킴(http/https) 검증(SECURITY-05), (c) 파싱 추출값(숙소명·좌표)도 신뢰하지 않고 지오코딩 재검증, (d) 화이트리스트 밖·파싱 실패는 **수동 핀 지정 흐름 폴백**(US-E3-08). 화이트리스트 밖 URL 전부 거부·폴백 경로는 PBT 대상(unit-of-work.md U3 DoD).
- **NFR-U3-SEC-03 (외부 응답 스키마 검증)**: 카카오·TourAPI·네이버 응답은 **외부 입력**이다 — 응답 스키마 검증(SECURITY-05), 비정상·부분 응답은 구조화 처리(누락 필드는 '미확인'으로 저장, 오류 fail-safe). 외부 문자열(숙소명)은 저장 전 새니타이즈·저장 후 출력 이스케이프(XSS 방어, NFR-U1-SEC-14 상속).
- **NFR-U3-SEC-04 (크롤링 금지 준수)**: OTA는 **데이터 크롤링 없이 숙소명 검색 딥링크만**(D09·§6.5) — `DeeplinkBuilder`는 외부 호출 없는 URL 조립만(component-dependency §6). 리뷰·평점 미표시(OTA 위임), 제휴 수수료 고지(US-E3-05).
- **NFR-U3-SEC-05 (소유권 — SECURITY-08)**: 위시리스트·등록 숙소·저장 POI 접근은 **토큰 주체(accountId) 기준으로만 조회**(IDOR 원천 차단, NFR-U1-SEC-18 상속). 저장은 로그인 계정 귀속(D22 — 기기 변경 유지, US-E3-04). 등록 숙소는 계정 레벨 풀(D15).
- **NFR-U3-SEC-06 (국내 좌표 범위 검증)**: 등록·핀 지정 좌표는 국내 좌표 범위(bounding box/역지오코딩) 검증(G120 — 관련 강제는 U4 완성이나 좌표 확정은 U3) — 범위 밖 차단·사유 안내.

## 6. 커버리지 게이트 — 데이터 품질 출시 판정 (G192)

TourAPI 데이터 품질(좌표 확보율·영업시간 채움률)이 탐색·일정 품질을 좌우하므로(unit-of-work.md U3 리스크), **출시 게이트를 측정 파이프라인으로 구현**한다.

- **NFR-U3-GATE-01 (출시 게이트 지표)**: **좌표 확보율 ≥ 95% · 영업시간 채움률 ≥ 70%**(G192)를 지역별로 측정한다. 게이트 미달 지역은 출시 판정에서 **차단 또는 제한 노출**(운영 판단) — 게이트는 데이터 품질의 정량 출시 기준이다.
- **NFR-U3-GATE-02 (측정 잡)**: 지역별 POI 정본의 좌표·영업시간 채움률을 **주기 측정하는 배치 잡**(기존 ShedLock 스케줄러 재사용, infrastructure-design.md §5)이 커스텀 메트릭으로 발행 — 대시보드·게이트 판정 입력.
- **NFR-U3-GATE-03 ('미확인' 표기)**: 미달·누락 필드는 **빈칸이 아니라 '미확인'으로 표기**(US-E3-03·G199 통일) — 침묵 실패 금지(ADR-0011). 좌표 미확정 등록은 "지도에서 위치 확인" 단계 강제(US-E3-06·08).
- **NFR-U3-GATE-04 (영업시간 변경 감지 경계)**: 정기 영업시간 변경만 자동 감지(TourAPI·지도 재조회, 일 1회), 당일 임시휴무는 수동 트리거(G192) — 자동 감지는 M9 휴무 폴링(U6)으로 승격되나, U3는 **정본의 영업시간 필드 갱신·채움률 측정**까지 소유.

## 7. 미결·선결 항목

### 7.1 선결 과제 — 설계 차단 요인 (P2·P4)

U3는 **비개발 선결 2건이 Functional Design(스냅샷·정본 테이블 확정 스키마) 착수를 차단**한다. 본 NFR 산출물은 벤더·정책 중립 요건까지 확정하며 차단되지 않는다.

| 과제 | 차단 대상 | 본 문서의 중립 대응 |
|---|---|---|
| **P2 지도 API 약관 검토·계약**(D13 스냅샷 적법성 법무 확인) | 지도 API 소스 캐싱 TTL 상한·사용자 확정 스냅샷 영구 저장 가부 | 지도 소스 영구 저장 표면 0(인메모리 TTL — §4.2), 스냅샷 필드 최소화·정책 컬럼(NFR-U3-DATA-03), P2 확정 전 지도 소스 스냅샷 보류 |
| **P4 TourAPI 활용 신청·캐싱 조건 확인** | 정본 영구 저장 필드 범위(D13 (a)) | `TourContentPort` 격리, 허용분만 정본 영구·보존 정책 컬럼으로 소급 축소(NFR-U3-DATA-01) |
| P5 OTA 제휴 계약(개발 무차단) | 파트너 딥링크 정책·수수료 문안 | 딥링크 템플릿·화이트리스트 remote config화(NFR-U3-SEC-02·04) |

### 7.2 벤더 종속 미결 — 없음 (전역 결정 상속)

**U3는 신규 인프라·벤더 종속 미결 항목이 없다.** U1이 PD-1~9를 전부 종결했고(클라우드·시크릿·로그·알림·캐시 정책), U3는 그 종결 결과와 GD-1~7·SI를 상속한다. U3의 유닛 로컬 결정(지도 SDK·서킷브레이커 라이브러리·POI 캐시 계층·유사도 알고리즘)은 **기존 스택 내 선택**이며 신규 벤더가 아니다(tech-stack-decisions.md).

| 잠재 결정 항목 | U3 처리 | 근거 |
|---|---|---|
| 클라우드·CI/CD·배포·롤백 | GD-1~7 상속(재질문 없음) | u1-foundation-nfr-design-plan.md §2 |
| 컴퓨트·DB·시크릿·로그·알림 스택 | SI 정본 상속(재정의 금지) | SI §1·3·6·8, U1 PD-1~9 종결 |
| 외부 API 아웃바운드 정책 | NAT 경유·SG 아웃바운드 443 상속(목적지 IP 가변 SaaS 예외) | SI §2.2 |
| 단기 캐시 계층 | 인메모리(Caffeine) 권고 — ElastiCache 미도입(§4.2) | SI §4 트리거 통합 |
