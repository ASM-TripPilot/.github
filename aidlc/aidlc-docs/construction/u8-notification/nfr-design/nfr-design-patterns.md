# 1. U8 신규 패턴

## PAT-U8-01 서버 스케줄링 정시 발송 + 일정 변경 재계산 `[U8]`

**문제**: 리마인드(D-1·당일·개별 N분 전)를 기기 로컬 알림으로 예약하면 일정이 바뀔 때 재예약이 기기에 종속되고 오프라인·앱 미실행 시 정합이 깨진다. 서버가 발송하되, 일정 변경 시 예약 시각을 정확히 갱신해야 하고, 이벤트 중복 수신에도 리마인드가 이중 생성되면 안 된다.

**적용(설계)**: **서버 스케줄링 단일 경로**(D12·D32)로 발송하고, 일정 변경은 **전체 무효화 후 재계산 + 멱등 키**로 처리한다.

```text
[ItineraryConfirmed] → 리마인드 세트 생성(D-1·당일 8시·개별 N분 전, D32)
[ItineraryChanged]   → 해당 여행 리마인드 전체 무효화 → 재계산(멱등 키: tripId+slotId+type)
[TripEnded]          → 잔여 리마인드 취소
[거점 해제 G97]       → 리마인드 중단 + 재생성 유도 배지
       │
[J4 분 단위 큐] due 도달 스케줄 행 → 발송 파이프라인(PAT-U8-03·02) → SENT/FAILED 기록
```

**설계 원칙**: (a) **서버 스케줄링 단일 경로**(로컬 알림 혼합 금지, G198=A·G166=A), (b) 일정 변경 = **무효화 후 재생성**(부분 갱신 아님 — 정합 보장, D32), (c) **멱등 키(tripId+slotId+type)** 로 중복 이벤트 흡수(at-least-once 구독 전제, services.md §0.2), (d) 리마인드 정밀도는 **J4 분 단위**(NFR-U8-PERF-01 — 초 단위 불요), (e) 이동 거리 포함·**소요시간 미표시**(D25, US-E09-02).

**U8 적용 지점**: M14 이벤트 구독(`ItineraryConfirmed/Changed`·`TripEnded`·거점 해제), 알림 스케줄 엔티티(멱등 키), J4 발송 큐(ShedLock), remote config(리마인드 오프셋).

**검증 기준**: D32 — 일정 변경 시 발송 시각 재계산(무효화 후 재생성) 테스트. PBT (c) — ItineraryChanged 중복 수신에도 멱등 키 기준 동일 결과(NFR-U8-QUAL-02). CP5 소비자 — ItineraryConfirmed/Changed·TripEnded 구독 멱등 계약 테스트. US-E09-02 — 거점 해제 시 중단·배지(G97).

## PAT-U8-02 FCM 발송 + 인앱 선적재 이중 파이프라인 `[U8]`

**문제**: 푸시는 best-effort다 — FCM 지연·실패·토큰 무효·OS 권한 거부로 미도달할 수 있다. 미도달을 그대로 두면 사용자는 알림을 영영 놓치고(침묵 실패), 발송이 조용히 사라진다(ADR-0011 위반).

**적용(설계)**: **인앱 알림함을 발송 파이프라인의 선(先)적재 단계로** 두어 FCM과 독립적으로 보존하고, FCM 실패는 재시도·dead letter·계측한다. "푸시는 실패해도, 알림함은 남는다."

```text
[J4 due 알림] → ① 종류별 토글 확인(off면 종결·적재도 없음 — 사용자 의사)
             → ② 방해금지 창 판정(PAT-U8-03 — 억제면 인앱 적재·푸시 보류)
             → ③ 중복 억제(dedupeKey 10분)
             → ④ TX: 인앱 알림함 적재(90일 D32) + 발송 상태 PENDING→CLAIMED  ← 선적재
             → ⑤ FCM 발송(TX 밖)
                  ├ 성공: SENT 기록
                  └ 실패: 재시도 3회 지수 백오프 → dead letter + 계측(A12)
                          토큰 무효 응답 → 기기 토큰 정리
             → ⑥ 발송 결과 기록(SENT/FAILED)
        ※ 인앱 알림함은 ④에서 이미 적재 → 사용자 관측 보장(침묵 실패 금지)
```

**설계 원칙**: (a) **인앱 선적재(④) = FCM 발송(⑤) 이전 TX** — FCM 실패해도 사용자 관측(D12·NFR-U8-RES-01), (b) FCM 실패는 **재시도 3회→dead letter + 계측**(조용히 사라지지 않음), (c) 토큰 무효 응답은 기기 토큰 정리(다음 발송 제외), (d) OS 푸시 권한 거부 시 푸시 토글 비활성·인앱 지속 누적(US-E09-05), (e) FCM 호출은 **TX 밖**(외부 호출 격리, services.md §0.2).

**U8 적용 지점**: M14 발송 파이프라인(§S5 4~6단계), `FcmPort`(Firebase Admin·토큰 정리·dead letter), 인앱 알림함 엔티티(발송 상태·읽음), 발송 상태 머신.

**검증 기준**: RESILIENCY-10 — FcmPort 명시 타임아웃·재시도·dead letter(코드 리뷰+계약 테스트). fake로 FCM 실패·토큰 무효 주입 시 **인앱 알림함 보존 + dead letter 계측** 검증(NFR-U8-RES-01·02, ADR-0011). D12 — 자체 DB 단일 파이프라인. A12·A10 계측.

## PAT-U8-03 방해금지·중복 억제 — 발송 판정 순수 함수 `[U8]`

**문제**: 방해금지 시간(22~08시)에 푸시가 울리면 사용자 경험을 해치고, 동일 일정에 중복 재계획 알림이 반복되면 알림 피로로 알림 끄기 이탈을 부른다. 판정 로직이 clock·설정에 얽히면 테스트가 불가능하다.

**적용(설계)**: 발송 판정(토글×방해금지×중복 억제)을 **순수 함수로 분리**하고, 방해금지 창·상한을 remote config화한다.

- **방해금지(G100)**: 기본 22~08시(설정 가능). 억제 대상이면 **인앱 알림함에는 적재하고 푸시만 보류**. 예외: 여행 '진행 중' Plan-B 알림(severe)은 통과(services.md §S5 2단계).
- **중복 억제**: 동일 dedupeKey 10분 창 내 재발송 차단(services.md §S5 3단계, US-E09-03).
- **종류별 토글(순수 판정)**: 종류별 푸시/인앱 독립 토글 — off면 미적재(사용자 의사). **시스템/보안 알림은 전 채널 off여도 인앱 표시**(US-E09-05).
- **발송 판정 순수 함수(G116 유사 — 결정성)**: clock·토글·방해금지·dedupe 이력을 주입 가능한 순수 함수로 분리 → 재현 가능한 PBT.

**U8 적용 지점**: M14 발송 판정 순수 함수(토글·방해금지·dedupe), 알림 설정·방해금지 창·dedupe 이력 엔티티, remote config(창·상한, G195 유사).

**검증 기준**: G100 — 방해금지 억제 시 인앱 적재·푸시 보류·Plan-B severe 예외 테스트. PBT (a) — off면 미적재·방해금지면 인앱만·dedupe 10분 억제(NFR-U8-QUAL-02). US-E09-05 — 종류별 독립 토글·시스템 알림 예외. 발송 판정 결정적 테스트(clock 주입).

## PAT-U8-04 계정 삭제 유예·연쇄 배치·법정 보존 분리 `[U8]`

**문제**: 계정 삭제는 개인정보보호법 의무다 — 전 모듈 데이터를 파기해야 하되(D18), 위치 로그·동의 증적·감사 로그 같은 법정 보존 데이터는 남겨야 하며(N2), 부분 삭제·이벤트 중복·재가입 우회가 발생하면 법 위반·데이터 정합 붕괴가 된다.

**적용(설계)**: **소프트 삭제 + 30일 유예 + 아웃박스 구독 연쇄 파기 + 법정 보존 분리 + 재가입 제한**을 결합한다(services.md §S6). U1이 만든 상태 모델·유예 골격 위에 U8이 연쇄를 완성한다.

```text
삭제 요청 → TX: ACTIVE→DEACTIVATED + purgeDueAt(+30일) + 세션/토큰 무효화 + outbox AccountDeletionRequested
   ├ 즉시: 위치 데이터 파기(D34 — 유예 없음, GPS 폴리라인·파생)  ※ 법정 로그는 분리 보관
   └ 유예 중: 동일 식별자 재가입 제한(C4) · 30일 내 재로그인→복구(purge 취소)

[J6 일 1회] purgeDueAt 경과 → TX: PURGING 전이 + outbox AccountDeletionExpired
   → 모듈별 파기 핸들러(구독): M2·M4·M6·M8·M12(사진 S3 포함)·M13·M14(알림함·스케줄·FCM 토큰)
      · at-least-once 재시도 · 기삭제 skip 멱등 · 전 핸들러 완료 후 계정 최종 삭제
   ※ 분리 보관(파기 대상 아님): 위치 법정 로그·동의 증적·감사 로그 (REVOKE+IAM Deny)
```

**설계 원칙**: (a) **소프트 삭제 + 30일 유예**(즉시 비노출·복구 가능, D18), (b) **위치 데이터 즉시 파기·법정 로그 분리 보관**(D34·N2 — 파기와 보존의 분리), (c) 모듈별 파기 = **at-least-once·멱등**(전 핸들러 완료 후 최종 삭제, `PURGING`이 진행 정본), (d) **재가입 제한**(유예 중 동일 식별자, C4), (e) 삭제 사전 고지·재확인(US-E09-09), (f) 상태 전이만 원자적·모듈 파기는 독립 TX(부분 진행 허용, services.md §S6).

**U8 적용 지점**: M1 `requestAccountDeletion`·J6(U8 완성), M14 파기 핸들러(알림함·스케줄·FCM 토큰), `AccountDeletionRequested/Expired` 아웃박스, 계정 삭제 상태 머신, 법정 보존 테이블(U1·U6 append-only 재사용).

**검증 기준**: D37 계정 무결성 100%(머지 차단) — 전 모듈 파기·법정 보존 분리·재가입 제한(NFR-U8-QUAL-01). PBT (b) — 삭제 상태 머신(유예 복구·재가입 제한 불변). SECURITY-14 — 법정 로그·동의 증적·감사 로그 append-only·삭제 권한 없음(REVOKE+IAM Deny 리뷰). fake 이벤트 중복 주입 시 핸들러 멱등(기삭제 skip).

## PAT-U8-05 데이터 내보내기 — 동기 스트리밍(직다운로드) `[U8]`

**문제**: 사용자는 자기 데이터를 내보낼 수 있어야 한다(데이터 주권). 그러나 일부 모듈 수집이 실패하면 부분 데이터를 완전본으로 위장해선 안 되고(침묵 누락 금지), 민감 작업이므로 남이 대신 내보내면 안 된다.

**적용(설계)**: **재인증 + rate-limit + 소유권 + 동기 JSON 스트리밍 직다운로드 + 부분 실패 마커**(services.md §S9). 1차 규모(§6.8)에선 비동기 잡·S3 임시 저장이 불요하다.

- **접근 제어**: 본인 재인증(민감 작업) + rate-limit(계정당 시간당 1회) + accountId 소유권(NFR-U8-SEC-03).
- **읽기 전용 수집(TX 불요)**: M2 프로필·취향, M6 여행·필수 방문지, M8 일정(plan/current), M12 방문 기록·changelog(사진 파일 제외·메타만 G101), M13 회고, M14 알림 설정.
- **동기 스트리밍 직다운로드**: JSON 스트리밍 → 즉시 다운로드 응답(비동기 잡·S3 미도입 — 규모 정합, NFR-U8-SC-03).
- **부분 실패 마커(침묵 누락 금지)**: 특정 모듈 실패 → 해당 섹션 오류 마커 + 재시도 안내(전체 실패 → 오류+재시도). 내보내기 이력 감사 로그.
- **전환 트리거**: 사진 포함 전체 아카이브(후속 G186)·대용량 시 S3 임시 저장 + 비동기 생성 + 서명 URL 재평가.

**U8 적용 지점**: M1 `requestDataExport`(재인증·rate-limit), 각 모듈 퍼사드 읽기 조회, JSON 스트리밍 직렬화, 감사 로그.

**검증 기준**: SECURITY-08 — 소유권·재인증·rate-limit 테스트(IDOR·남용 차단). S9 — 부분 실패 시 오류 마커·완전본 위장 금지 테스트(ADR-0011). PBT (e) — 내보내기 JSON 직렬화 왕복. SECURITY-03 — 내보내기 이력 감사.

## PAT-U8-06 알림함 보존 — 90일·읽음 상태 `[U8]`

**문제**: 인앱 알림함은 무한 누적되면 저장·조회 비용이 늘고, 읽음/안읽음·미읽음 뱃지가 없으면 사용자가 새 알림을 식별할 수 없다. 보존 정책이 없으면 스키마·목록 구현이 불가능하다.

**적용(설계)**: **90일 보존 + 읽음 상태 + 미읽음 뱃지 + J9 정리**(D32·G99).

- **보존(D32·G99)**: 인앱 알림함 90일 보존 — J9(일 1회)가 90일 초과분 삭제(시각 조건 = 자연 멱등, services.md §4 J9).
- **읽음 상태·뱃지**: 읽음/안읽음 상태 관리 + 미읽음 뱃지 카운트(홈 상단 알림 진입점, US-E02-02·US-E09-05).
- **채널 독립 누적**: 특정 종류 푸시 off여도 인앱 알림함에는 계속 누적(반대도 동일). 방해금지 억제분·시스템 알림도 적재.
- **볼륨 정합**: DAU 1천·활성 여행 소량 → 일 수만 행(단일 DB 충분, NFR-U8-SC-02).

**U8 적용 지점**: 인앱 알림함 엔티티(읽음·시각 인덱스), J9 보존 정리 잡(ShedLock), 알림함 목록·뱃지 조회 API, `features/notification`.

**검증 기준**: G99 — 90일 보존·읽음 상태·미읽음 뱃지 테스트. D32 — J9 90일 초과 정리(멱등). PBT (d) — 알림함 보존 90일 경계·읽음 상태 전이(NFR-U8-QUAL-02). NFR-U8-PERF-03 — 조회 300ms(인덱스·페이지네이션).
