# U8 알림·마이페이지·설정 — 기술 스택 결정 (Tech Stack Decisions)

> 2026-07-04 · CONSTRUCTION · **U8 (마감 유닛)** NFR Requirements 산출물
> **전제**: 스택 대분류(RN Expo + Spring Boot Kotlin + PostgreSQL, D02)와 서버·클라이언트 라이브러리 정본(U1 [tech-stack-decisions.md](../../u1-foundation/nfr-requirements/tech-stack-decisions.md) S-1~S-13·C-1~C-8), 복원력 라이브러리(Resilience4j — U3-S1), 스케줄러(ShedLock — SI §1.2), 아웃박스(SI §4), 메일(SES — U1 §4)은 **기확정**이다. U8은 그 위에 **① FCM 서버 SDK, ② 서버 스케줄 폴링 잡, ③ 인앱 알림함(RDS), ④ 계정 삭제 배치** 4개 지점만 구체화하며, **SES(내보내기 안내·시스템 메일)·스케줄러·아웃박스는 재사용(신규 없음)** 한다.
> **재결정 금지**: 클라우드·인프라·서버 프레임워크는 SI·U1 정본 상속. **푸시 인프라는 D12·G198=A 기확정**(FCM 단일, iOS 포함 FCM 경유). **알림 스케줄링은 D32 기확정**(서버 스케줄링 통일·인앱 알림함 자체 DB). 본 문서는 U8의 선택지가 있는 지점만 대안 비교 후 권고한다. **FCM 콘솔·APNs 인증 키는 P9 병렬** — 본 문서는 **어댑터 중립 구조**를 확정한다(nfr-requirements.md §8.1).

## 2. FCM 서버 SDK — Firebase Admin + 토큰 관리 (U8-N1)

D12·G198(=A)이 **"FCM 단일(iOS도 FCM 경유)·모든 알림 서버 스케줄링 발송"** 을 확정했으므로 U8은 **소스 선택이 아니라 어댑터 구조**를 확정한다. FCM 콘솔·APNs 인증 키는 **P9** 병렬이지만(nfr-requirements.md §8.1), **어댑터 추상화·토큰 관리·재시도·무효 토큰 정리는 지금 확정**한다 — 이것이 발송 실패·인증서 미비를 어댑터 내부로 격리하는 방식이다.

### 2.1 어댑터 책임 (`FcmPort` = M14 아웃바운드)

| 책임 | 설계 | 근거 |
|---|---|---|
| **서버 발송** | `FcmPort` 인터페이스 뒤 **Firebase Admin SDK(FCM HTTP v1 API)** 은닉. M14 발송 파이프라인은 알림 페이로드·대상 토큰만 인지 | D12, services.md §S5 |
| **iOS 포함 단일 채널** | iOS도 FCM 경유(APNs는 FCM이 중계 — 콘솔에 APNs 인증 키 등록, P9) — 채널 분기 없음(운영 단순, G198=A) | D12·G198 |
| **토큰 등록·무효 정리** | 기기 FCM 토큰 등록/갱신, 발송 시 무효 응답(UNREGISTERED 등) 토큰 정리(다음 발송 제외) | services.md §S5 5단계 |
| **타임아웃·재시도** | Resilience4j 타임아웃 + **재시도 3회 지수 백오프**(U3·U5·U6 재사용) → 최종 실패 시 **dead letter + 계측**. 인앱 알림함은 발송 전 선적재로 보존 | RESILIENCY-10, NFR-U8-RES-01·02 |
| **멀티캐스트 배치** | 다기기 사용자·묶음 발송은 멀티캐스트(FCM 500토큰/요청 상한 준수) — DAU 1천 규모 배치 소량 | NFR-U8-SC-01 |

- **대안 비교(1줄)**: (A) **`FcmPort` + Firebase Admin SDK(권고)** — 소스 은닉·토큰 관리·재시도·dead letter를 한 곳에 격리(D12 정합). (B) FCM + APNs 직접 이원 연동 — 채널별 세밀 제어 이점이 있으나 iOS 인증서·페이로드 이원 관리로 운영 복잡, G198이 FCM 단일을 확정(비채택). (C) 서드파티 푸시 SaaS(OneSignal 등) — 묶음·세그먼트 기능 위탁 이점이 있으나 개인정보 위탁·비용·인박스 종속으로 D12(자체 DB 알림함)·G142와 배치(비채택).
- **선정 사유**: **인앱 알림함 선적재 보존(D12)** 은 U8 복원력의 핵심이며, FCM 발송은 어댑터 한 곳에서 실패를 감지해 dead letter·계측하고 인앱 알림함이 최후 보루가 되어야 구조적으로 보장된다. **P9 미확정(콘솔·인증서 미비) 상태에서도 어댑터·토큰 관리·재시도는 확정 가능**(fake 발송) — 이것이 P9 비차단의 근거다.

### 2.2 P9 경계 — 지금 확정 vs P9 후

| 지금 확정(콘솔 무관) | P9 후 확정(콘솔 의존) |
|---|---|
| `FcmPort` 인터페이스·토큰 등록/정리·재시도·dead letter·멀티캐스트 구조·`external/fcm` 시크릿 프레임 | Firebase 프로젝트·서비스 계정 키·**APNs 인증 키(iOS)**·FCM 발송 쿼터(A11 임계) |

## 3. 서버 스케줄 폴링 잡 — 기존 ShedLock 스케줄러 재사용 (U8-P1)

모든 알림은 서버 스케줄링 발송(D32)이다. SI §1.2가 "U1~U8 스케줄러 잡은 별도 워커 없이 앱 프로세스 내 `@Scheduled` + ShedLock 분산 락"으로 확정했으므로 U8은 이를 재사용한다.

| 기준 (G142 · 발송량 소량) | **A. 기존 ShedLock 스케줄러 재사용 (권고)** | B. 별도 발송 워커 서비스 + 큐 | C. EventBridge Scheduler + ECS RunTask |
|---|---|---|---|
| 신규 AWS 리소스 | **0** — 앱 프로세스 내 `@Scheduled` + ShedLock(SI §1.2) | +ECS 서비스·SQS·SG (SI §4 브로커 미도입 위반) | +EventBridge·태스크 정의 |
| 주기·부하 | 리마인드 분 단위·삭제 유예 일 1회·알림함 정리 일 1회 — 저부하(발송량 소량) | 과설계 | 과설계 |
| 중복 실행 방지(태스크 2개) | ShedLock 분산 락 + `SKIP LOCKED` 클레임 + dedupeKey(SI §1.2) | 큐 가시성 타임아웃 | 스케줄 단일 트리거 |
| 아웃박스 연계(구독 발송) | **직접**(동일 프로세스 DB 아웃박스, SI §4) | 크로스 프로세스 | 크로스 프로세스 |
| 규모 정합(G142·활성 여행 ≤1/사용자) | **적합** | 과설계 | 과설계 |

- **확정 권고: A(기존 ShedLock 스케줄러 재사용) · 별도 워커 미도입.** 근거: (1) 잡(J4 리마인드 분 단위·J6 삭제 유예 만료 일 1회·J9 알림함 보존 정리 일 1회)은 **전부 저빈도·저부하**이고 발송량이 소량(D21 활성 ≤1/사용자)이라 기존 태스크 2개로 흡수(NFR-U8-SC-01), (2) 구독 발송이 동일 프로세스 DB 아웃박스(SI §4)와 직결, (3) 별도 워커·큐·푸시 SaaS는 SI §4(브로커 없음)·D12(자체 DB 알림함) 결정과 배치. **J4는 `SKIP LOCKED` 클레임 + 발송 상태 전이(`PENDING→CLAIMED→SENT/FAILED`) + dedupeKey**로 이중 발송을 방지(services.md §4 J4).
- **B/C 비채택 사유(1줄)**: 별도 워커·큐·EventBridge는 소량 발송에 과설계이며 SI §4(DB 기반·브로커 없음)와 배치. **전환 트리거**: 분당 발송량·FCM 호출량이 태스크·쿼터를 상시 압박하면 EventBridge + 발송 워커 분리 재평가(SI §1.2 트리거 상속).
- **잡 목록(services.md §4 재사용)**: J4 리마인드 발송 큐(분 단위 — 토글→방해금지→중복 억제→인앱 적재→FCM)·J6 삭제 유예 만료(일 1회 — `PURGING` 전이 + `AccountDeletionExpired`)·J9 알림함 보존 정리(일 1회 — 90일 초과 삭제). J6는 상태 기반 + 파기 핸들러 멱등, J9는 시각 조건 삭제 = 자연 멱등.

## 4. 인앱 알림함 — 자체 RDS 단일 파이프라인 (U8-D1)

D12가 **"인앱 알림함 자체 DB 단일 파이프라인"** 을, D32가 **"90일 보존·읽음 관리"** 를 확정했다. 저장·파이프라인 구조를 구체화한다.

| 기준 (D12·D32) | **A. 자체 RDS 테이블 단일 파이프라인 (권고)** | B. 푸시 SaaS 인박스 위탁 | C. 클라 로컬 저장 |
|---|---|---|---|
| D12 정합(자체 DB) | **정합** — 발송·알림함 단일 파이프라인 | D12 위반(외부 인박스) | D12 위반·기기 종속 |
| 선적재 보존(FCM 실패 시) | **발송 전 TX 적재 — 관측 보장** | 위탁 인박스 실패 시 관측 불가 | 오프라인·기기 변경 시 소실 |
| 90일 보존·읽음·뱃지(G99) | **SQL 조회·상태 관리·J9 정리** | SaaS 정책 종속 | 기기별 상이 |
| 소유권·계정 귀속 | accountId 스코핑(IDOR) | 위탁 경계 | 기기 종속 |
| 규모(G142) | 일 수만 행·단일 DB 충분 | 과잉 위탁 | — |

- **확정 권고: A(자체 RDS 테이블 단일 파이프라인).** 근거: (1) D12가 **자체 DB 단일 파이프라인**을 확정, (2) **인앱 알림함이 발송 파이프라인 선적재 단계**여서 FCM 실패 시 최후 관측 보루가 되려면 발송 상태와 동일 DB에 있어야 함(NFR-U8-RES-01), (3) 90일 보존·읽음/안읽음·미읽음 뱃지·개별 삭제(G99=B 대안 계열)를 SQL로 관리·J9로 정리. 종류별 토글·방해금지·시스템 알림 예외도 동일 DB에서 판정(services.md §S5).
- **B/C 비채택 사유(1줄)**: 푸시 SaaS 인박스(B)는 D12(자체 DB) 위반·개인정보 위탁·관측 종속; 클라 로컬 저장(C)은 기기 변경·오프라인 소실로 알림 이력 신뢰성 붕괴.

## 5. 계정 삭제 배치 · 데이터 내보내기 (U8-X1·X2)

### 5.1 계정 삭제 배치 (U8-X1 — 기존 인프라 재사용)

- **결정**: **기존 스케줄러(J6) + 아웃박스 이벤트 구독 파기 핸들러**(신규 인프라 0). U1이 상태 모델(`ACTIVE→DEACTIVATED`·purgeDueAt·유예 배치 골격)을 생성했고, U8이 **전 모듈 연쇄 파기를 완성**한다(services.md §S6). `AccountDeletionExpired` 구독으로 M2·M4·M6·M8·M12·M13·M14 파기 핸들러 실행 — **at-least-once 재시도·기삭제 skip 멱등·전 핸들러 완료 후 최종 삭제**. 위치 데이터는 삭제 요청 즉시 파기(D34), **법정 보존 3종(위치 로그·동의 증적·감사)은 분리 보관**(REVOKE+IAM Deny).
- **대안 비교(1줄)**: (A) **아웃박스 구독 파기 핸들러(권고)** — 모듈 경계 유지·at-least-once·멱등(services.md §S6). (B) 삭제 시점 동기 연쇄 삭제 — 단일 TX가 전 모듈에 걸쳐 결합도·장애 시 부분 삭제 위험(비채택). **재사용이므로 신규 벤더·인프라 없음.**

### 5.2 데이터 내보내기 (U8-X2 — 동기 직다운로드)

- **결정**: **동기 JSON 스트리밍 직다운로드**(services.md §S9 3단계 — "비동기 잡 없이 동기, 1차 규모 충분"). 읽기 전용 모듈 수집(M2·M6·M8·M12 메타·M13·M14) → JSON 스트리밍 → 즉시 다운로드 응답. **재인증 + rate-limit(계정당 시간당 1회) + 소유권**(NFR-U8-SEC-03). 부분 실패는 오류 마커(침묵 누락 금지).
- **대안 비교(1줄)**: (A) **동기 스트리밍 직다운로드(권고)** — S3 임시 저장·비동기 잡·서명 URL 불요(규모 §6.8, 텍스트만·사진 제외 G101). (B) S3 임시 저장 + 비동기 생성 + 서명 URL 다운로드 — 대용량·사진 포함 전체 아카이브(후속 G186)에 필요하나 1차 텍스트 규모엔 과설계(비채택·전환 트리거로 예약). **신규 S3·잡 미도입.**

**U8 신규 외부 벤더·클라우드 결정 = FCM(P9 병렬) 1건뿐이며, 그마저 본 문서는 `FcmPort` 어댑터 중립 구조·토큰 관리·dead letter로 비차단화한다.** SES(내보내기 안내·시스템 메일)는 U1 재사용, 스케줄러·아웃박스·시크릿은 SI 재사용, 인앱 알림함·삭제 배치·내보내기는 **기존 스택 내 라이브러리·패턴 구체화**(Firebase Admin·ShedLock·RDS·JSON 스트리밍)이며 신규 인프라 벤더가 아니다(과설계 금지·재결정 금지). FCM 키·쿼터는 P9 후 Secrets Manager(`external/fcm`) 주입.
