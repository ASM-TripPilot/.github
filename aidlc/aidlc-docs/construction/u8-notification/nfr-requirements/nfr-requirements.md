## 0. U8 NFR 프로필 — "여행 생명주기를 알림·설정으로 닫는 마감 유닛"의 NFR 본질

U8의 NFR 본질은 **적시에 정확한 알림을 서버가 발송하되(D32), 어떤 채널이 실패해도 인앱 알림함이 사용자를 놓치지 않고(D12), 사용자가 자기 데이터(알림 설정·취향·위치 동의·계정 삭제·내보내기)를 완전하게 통제하되 법정 보존 의무를 어기지 않는 것(D18·D34)** 이다. 세 축이 핵심이다:

1. **신뢰성/정확성(핵심)** — 모든 알림은 **서버 스케줄링 단일 경로**(D32)로 발송하고, **일정 변경 시 발송 시각을 재계산**(전체 무효화 후 재계산·멱등)하며, **중복 발송을 억제**(dedupeKey 10분)한다. 리마인드 정밀도는 J4 분 단위 폴링에 종속된다(services.md §S5).
2. **복원력(핵심)** — **인앱 알림함은 발송 파이프라인의 선(先)적재 단계**여서 FCM이 실패해도 사용자가 관측 가능하다(D12·침묵 실패 금지 ADR-0011). FCM 실패는 재시도 3회→dead letter 계측, 삭제 배치 실패는 이벤트 재전달(at-least-once·핸들러 멱등), 데이터 내보내기 부분 실패는 오류 마커로 노출한다.
3. **보안·개인정보(핵심)** — **계정 삭제 연쇄**(D18)는 전 모듈 데이터를 파기하되 **법정 보존 데이터(위치 로그·동의 증적·감사 로그)는 분리 보관**한다. **데이터 내보내기**(S9)는 재인증·rate-limit·소유권으로 보호하고, **마케팅 동의**(N8·정보통신망법)는 수집·철회만(발송 없음), **위치 3층**(D34·G182)은 설정에서 제어한다.

인프라·무상태·배포의 서버 규약은 **U1 전역 정본을 그대로 상속**하고, U8은 **FCM 발송·서버 폴링 잡·알림함/삭제 스키마·내보내기** 증분만 소유한다. **마감 유닛으로서 전 유닛 통합 관측성(G144)·컴플라이언스 종합**을 §7·§9에 첨부한다.

### 0.1 워크로드 중요도 (requirements.md §6.3)

| 컴포넌트 | 중요도 | 불가용 시 영향·U8 대응 |
|---|---|---|
| 인앱 알림함 저장·조회(M14 저장부·DB) | **High** | 알림 이력 접근 불가 — RDS Multi-AZ 상속(폴백 없음, 오류·재시도). 발송 파이프라인 선적재 단계이므로 발송 실패 관측의 최후 보루 |
| 알림 발송(FCM 아웃바운드) | **Medium 이하** | 푸시 미도달 — **인앱 알림함 선적재로 보존**(D12), 재시도→dead letter 계측. 발송 자체는 폴백 존재(인앱) |
| 계정 삭제 연쇄(M1 소유·M14 등 파기 핸들러) | **High(법정)** | 삭제 미완료 — 개인정보보호법 위반 리스크. at-least-once 재시도·전 핸들러 완료 후 최종 삭제·법정 보존 분리 |
| 데이터 내보내기(M1) | **Medium** | 내보내기 실패 — 부분 실패 마커·재시도(침묵 누락 금지). 온디맨드 저빈도 |
| 마이페이지 집계 조회(BFF성) | **Medium** | 카드 일부 누락 — 가용 카드만 부분 응답(침묵 실패 금지, US-E09-06·07·08) |

**핵심 원칙**: 발송 채널(FCM)은 **폴백 필수 경로**이며(인앱 알림함 선적재), M14 알림함 저장이 High인 근거는 **발송 실패의 관측·복구 지점**이라는 데 있다. 계정 삭제 연쇄는 **법정 완전성**이 요구되어 High이며 부분 진행을 `PURGING` 상태로 추적한다(services.md §S6).

## 1. 성능 (§6.1 · D32 · services.md §S5)

### 1.1 알림 스케줄링·발송 지연

| 대상 | 목표 | 근거·폴백 | 추적 |
|---|---|---|---|
| 리마인드 스케줄 정밀도 | **분 단위**(J4 due 폴링 주기) | D-1·당일 8시·개별 일정 N분 전(D32)은 분 단위 큐 — 초 단위 정밀 불요(사용자 체감 충분). 초과 적체는 A10 알람 | US-E09-02, services.md §4 J4 |
| 발송 파이프라인 인앱 적재 | **FCM 발송 전 완료(TX)** | 인앱 알림함 적재가 발송 상태 `PENDING→CLAIMED` TX 내 → FCM 실패해도 사용자 관측(침묵 실패 금지) | D12, services.md §S5 4~5단계 |
| 알림함 조회 | **p95 300ms** | 계정·시각 인덱스 + 페이지네이션·읽음 상태(§6.1 상속) | US-E09-05, G99 |
| 마이페이지 집계(BFF성) | **가용 카드 부분 응답** | 숙소·여행·스타일 카드 병렬 조회 — 일부 실패해도 가용분 우선(침묵 실패 금지) | US-E09-06·07·08 |
| 설정 토글 반영 | **즉시(별도 저장 버튼 없음)** | 종류별 푸시/인앱 토글·방해금지·마케팅·위치 동의 즉시 저장·다음 알림부터 반영 | US-E09-05·11·14 |

- **NFR-U8-PERF-01 (리마인드 분 단위 정밀도)**: 서버 스케줄링 리마인드는 **J4 분 단위 발송 큐**로 처리한다 — due 도달 알림 행을 분 단위 폴링해 발송 파이프라인(§S5)에 투입한다. 초 단위 정밀 스케줄러(별도 인프라)는 규모(§6.8)·체감상 불요(과설계 금지). 발송 큐 적체(due 대기 age)는 A10 알람(SI §8.2).
- **NFR-U8-PERF-02 (인앱 적재 = 발송 전 완료)**: 알림함 적재는 **FCM 발송 이전 단계에서 TX로 완료**한다(services.md §S5 4단계) — 발송 지연·실패가 발생해도 사용자는 알림함에서 관측 가능(NFR-U8-RES-01과 정합). "발송 지연"의 하한은 인앱 적재 시점이며 푸시는 best-effort.
- **NFR-U8-PERF-03 (알림함 조회 300ms · 마이페이지 부분 응답)**: 알림함은 계정 스코핑·시각 내림차순 인덱스 + 페이지네이션(§6.1 300ms 상속). 마이페이지는 등록 숙소(M4)·여행 목록(M6)·스타일 분석(M13)·설정(M14)을 **병렬 집약**하되 일부 실패해도 가용 카드만 반환한다(부분 응답·침묵 실패 금지, US-E09-06·07·08).
- **NFR-U8-PERF-04 (설정 즉시 반영)**: 종류별 토글·방해금지·마케팅·위치 동의는 **별도 저장 버튼 없이 즉시 저장**되고 다음 알림·다음 생성부터 반영된다(US-E09-05·10·11·14). `updateToggles` 즉시 반영은 발송 1단계에서 확인(off면 적재도 없음, services.md §S5 1단계).

## 2. 정확성·신뢰성 (핵심 · D32 · services.md §S5)

### 2.1 정시 발송 스케줄링 정확도 · 재계산

- **NFR-U8-REL-01 (서버 스케줄링 단일 경로 — D32)**: 모든 알림(리마인드·Plan-B·회고·등록 완료)은 **서버 스케줄링 발송 단일 파이프라인**(D12·D32)으로 처리한다 — 기기 로컬 알림 예약(혼합 방식)을 쓰지 않는다(G198=A·G166=A). 리마인드 세트는 `ItineraryConfirmed` 구독으로 생성(D-1·당일 8시·개별 N분 전, services.md §S5).
- **NFR-U8-REL-02 (일정 변경 재계산 = 전체 무효화 후 재생성·멱등)**: `ItineraryChanged` 수신 시 **해당 여행 리마인드를 전체 무효화 후 재계산**한다(services.md §S5) — 부분 갱신이 아닌 무효화·재생성으로 정합을 보장하고, **멱등 키(tripId+slotId+type)** 로 중복 이벤트를 흡수한다. `TripEnded`는 잔여 리마인드 취소, 거점 해제(G97)는 리마인드 중단 + 재생성 유도 배지.
- **NFR-U8-REL-03 (중복 억제 — dedupeKey 10분)**: 동일 dedupeKey 10분 창 내 재발송을 차단한다(services.md §S5 3단계, US-E09-03 "동일 일정 중복 재계획 알림 10분 내 1회"). Plan-B 자동 트리거 상한·민감도는 U6 소유(M9), U8은 **발송 시점 중복 억제**를 담당한다.
- **NFR-U8-REL-04 (방해금지 = 인앱 적재·푸시 보류 — G100)**: 방해금지 창(기본 22~08시)에 억제 대상이면 **인앱 알림함에는 적재하고 푸시만 보류**한다(G100·US-E09-02·05). 예외: 여행 '진행 중' Plan-B 알림(severe)은 통과(services.md §S5 2단계). 방해금지 창은 설정 가능(US-E09-02 당일 8시 등).
- **NFR-U8-REL-05 (알림 종류 정본 · 종류별 토글 독립)**: 알림 종류는 PRD 12 목록이 정본이며 '체크인 임박'·'예약 링크 리마인드'는 제외(Δ10). 종류별 **푸시/인앱 토글을 각각 독립 제공**하되, **보안/계정 시스템 알림은 전 채널 off여도 인앱 표시**(US-E09-05). 특정 종류 푸시 off여도 인앱 알림함에는 계속 누적(반대도 동일).

## 3. 복원력 (핵심 · §6.2·§6.3 · RESILIENCY-10 · ADR-0011 침묵 실패 금지)

U8 복원력의 정본은 **services.md §0.1 실패 처리 표(FCM 행)**·**U1 PAT-RES-01 4요소 계약**이다. 본 절은 U8 폴백을 설계 정본으로 승격한다. **핵심: 어떤 알림 채널이 죽어도 인앱 알림함은 살아 있고, 삭제·내보내기는 부분 실패를 침묵하지 않는다.**

### 3.1 FCM 실패 → 인앱 알림함 선적재 보존 (D12)

| 의존 단계 | 실패 시 폴백 | 사용자 고지 | 추적 |
|---|---|---|---|
| **FCM 발송(M14 아웃바운드)** | **인앱 알림함은 선적재로 항상 보존**(발송 전 TX), 재시도 3회 지수 백오프 → dead letter + 계측 | 알림함에서 확인 가능(푸시 미도달을 침묵하지 않음) | D12, services.md §0.1·§S5 5단계 |
| **FCM 토큰 무효 응답** | 기기 토큰 정리(다음 발송 대상 제외) | 없음(자동 정리) | services.md §S5 5단계 |
| **OS 푸시 권한 거부** | 푸시 토글 비활성 표시 + 설정 이동 안내, 인앱 알림함 지속 누적 | "기기 설정에서 알림 권한 허용" | US-E09-05 |

- **NFR-U8-RES-01 (FCM 실패 = 인앱 선적재 보존, 침묵 실패 금지)**: FCM 발송 실패(재시도 3회 지수 백오프 최종 실패) 시 **인앱 알림함은 발송 파이프라인 4단계에서 이미 적재**되어 사용자가 관측 가능하다(services.md §S5·§0.1). 최종 실패는 **dead letter + 계측**(A12·커스텀 메트릭)한다 — 발송이 조용히 사라지지 않는다(ADR-0011). FCM 토큰 무효 응답은 기기 토큰을 정리한다.
- **NFR-U8-RES-02 (FCM 어댑터 4요소 계약 — RESILIENCY-10)**: FCM 어댑터(`FcmPort`)는 명시적 타임아웃 + 재시도 3회(지수 백오프) + dead letter 폴백을 가진다(U3·U5·U6 Resilience4j 재사용). 서킷은 발송 특성상 개별 재시도·dead letter로 갈음(대량 배치 아님). 발송 실패율·dead letter는 계측(A12·A10).

### 3.2 삭제 배치 실패 재시도 (D18 · services.md §S6)

- **NFR-U8-RES-03 (모듈별 파기 핸들러 = at-least-once 재시도·멱등)**: 계정 삭제 유예 만료(J6) 후 모듈별 파기 핸들러(M2·M4·M6·M8·M12·M13·M14)는 **이벤트 재전달로 재시도**(at-least-once)하고, **이미 삭제된 리소스는 skip으로 멱등**하다(services.md §S6 5단계). **전 핸들러 완료 확인 후에만 계정 레코드 최종 삭제** — 부분 진행은 `PURGING` 상태가 정본(진행 중 식별). M14는 알림함·스케줄·FCM 토큰 파기 담당.
- **NFR-U8-RES-04 (위치 데이터 즉시 파기 = 유예 없음)**: 계정 삭제 요청 즉시(비활성화 이벤트 구독) GPS 폴리라인·위치 파생 데이터를 파기한다(D34 — 30일 유예와 무관, services.md §S6 2단계). **단 위치정보 법정 로그(append-only)는 분리 보관 유지**(N2·§4 아래).

### 3.3 데이터 내보내기 부분 실패 노출 (S9)

- **NFR-U8-RES-05 (내보내기 부분 실패 = 오류 마커, 침묵 누락 금지)**: 데이터 내보내기(S9)의 특정 모듈 수집 실패 시 **해당 섹션에 오류 마커 포함 + 재시도 안내**한다(services.md §S9 3단계) — 부분 성공을 완전본으로 위장하지 않는다. 전체 실패는 오류 + 재시도. 내보내기 이력은 감사 로그 기록(SECURITY-03).

## 4. 보안·개인정보 (핵심 · §6.4·§6.5 · D18·D34·N8)

U8은 개인정보 통제 기능(삭제·내보내기·마케팅·위치 동의)의 집약 유닛이다 — 전역 보안 설정(deny-by-default·헤더·에러 핸들러·구조화 로깅·시크릿)은 U1 스캐폴드가 소유하고 U8은 상속한다. U8 증분은 **삭제 연쇄 완전성·법정 보존 분리·내보내기 접근 제어·마케팅/위치 동의** 네 지점이다.

- **NFR-U8-SEC-01 (계정 삭제 연쇄 완전성 — D18)**: 계정 삭제는 **소프트 삭제 + 30일 유예**(즉시 비노출·전 세션/리프레시 토큰 무효화·유예 중 동일 식별자 재가입 제한 C4) 후 **J6 만료 시 전 모듈 연쇄 파기**한다(services.md §S6). 파기 대상: M6 여행·필수 방문지, M8 일정, M12 기록·changelog·사진(S3 객체 포함), M13 회고, M14 알림함·스케줄·FCM 토큰, M2 프로필·취향, M4 등록 숙소. (후속) M15 커뮤니티는 '삭제된 사용자' 익명화. **삭제 사전 고지·재확인 필수**(US-E09-09).
- **NFR-U8-SEC-02 (법정 보존 분리 — D18·N2)**: 삭제 연쇄에서 **법정 보존 데이터만 분리 보관 유지** — ① **위치정보 수집·이용 법정 로그**(append-only, U1 생성·U6 기록 개시·앱 삭제 권한 없음 REVOKE+IAM Deny), ② **동의 증적**(약관·위치·마케팅 버전·시각), ③ **감사 로그**(삭제·내보내기 이력). 이들은 파기 핸들러의 접근 대상이 아니다(권한 분리, SECURITY-14, SI §3·§5.3·§7.1).
- **NFR-U8-SEC-03 (데이터 내보내기 접근 제어 — S9)**: 데이터 내보내기는 **본인 재인증(민감 작업) + rate-limit(계정당 시간당 1회) + 소유권 스코핑**으로 보호한다(services.md §S9 1단계). 텍스트 데이터(여행·일정·기록·설정) JSON만·사진 제외(G101/G186). 내보내기 이력 감사 로그 기록(SECURITY-03).
- **NFR-U8-SEC-04 (마케팅 동의 — 정보통신망법·N8)**: 마케팅 수신 동의는 **온보딩 수집(선택) + 설정 철회 토글**만 제공하고 **1차 출시에서 마케팅 알림 발송은 하지 않는다**(N8). 동의·철회 상태·시각은 동의 증적으로 저장(법정 보존, 정보통신망법 제50조 수신 동의·철회 근거). 발송 기능은 후속 — 본 유닛은 상태 수집·관리만.
- **NFR-U8-SEC-05 (위치 3층 동의 설정 제어 — D34·G182)**: 위치 동의는 3층(OS 권한 × 앱 내 위치기반서비스 법정 동의 × GPS 여행 기록 옵트인)으로 **설정에서 각 층 구분 표시·제어**한다(US-E09-11). 철회 직전 영향 기능 고지·재확인, **GPS 옵트인 철회·탈퇴 시 즉시 파기**(D34), 위치 비의존 트리거(예정 일정·날씨·휴무)는 철회 후에도 유지. 3층 모델·법정 로그는 **U1 동의 모델·U6 법정 로그 재사용**(신규 없음).
- **NFR-U8-SEC-06 (알림·설정 소유권 — IDOR 차단)**: 알림함·알림 설정·마이페이지 집계·내보내기·삭제 접근은 **토큰 주체(accountId) 스코핑**(IDOR 차단, NFR-U1-SEC-18 상속·SECURITY-08). 예외: 보안/계정 시스템 알림은 전 채널 off여도 인앱 표시(US-E09-05).
- **NFR-U8-SEC-07 (FCM 서비스 계정 키 = 시크릿)**: FCM 서비스 계정 키(Firebase Admin)는 **소스·이미지·설정 파일에 미포함** — Secrets Manager 주입(`external/fcm`, SI §6 예약 항목 활성화, NFR-U1-SEC-11 상속). 시크릿 스캔 CI 게이트(SECURITY-10). APNs 인증 키는 FCM 콘솔 등록(P9 병렬).

## 5. 규모·확장성 (§6.8 G142 — DAU 1천 · 활성 여행 ≤1/사용자)

- **NFR-U8-SC-01 (리마인드 발송량 = 진행/예정 여행 수 — 과설계 금지)**: 리마인드 세트는 확정 일정(활성 여행 ≤1/사용자 D21)당 D-1·당일·개별 슬롯 N분 전으로, DAU 1천에서 일 발송량은 소량(예정+진행 여행의 부분집합 × 슬롯 수)이다 — **별도 발송 워커·큐·푸시 SaaS는 과설계**(G142·G198). 기존 ShedLock J4 분 단위 큐 + FCM 배치 발송으로 흡수. **전환 트리거**: 분당 발송량이 태스크·FCM 쿼터를 상시 압박하면 EventBridge + 발송 워커 분리 재평가(SI §1.2 상속).
- **NFR-U8-SC-02 (알림함 볼륨 = 90일 보존·J9 정리)**: 인앱 알림함은 **90일 보존**(D32·G99)이고 J9(일 1회)가 90일 초과분을 정리한다 — DAU 1천·활성 여행 소량 기준 일 수만 행 수준(단일 PostgreSQL 충분, SI §3). 읽음/안읽음 상태·미읽음 뱃지 관리(G99).
- **NFR-U8-SC-03 (삭제·내보내기 배치 = 온디맨드 소량)**: 삭제 유예 만료(J6 일 1회)·데이터 내보내기(온디맨드)는 저빈도·소량 — 동기 처리·기존 태스크 흡수로 충분(과설계 금지). 데이터 내보내기는 **동기 스트리밍 직다운로드**(비동기 잡·S3 임시 저장 미도입, services.md §S9·infrastructure-design.md §4).
- **NFR-U8-SC-04 (배치 커넥션 벌크헤드)**: 발송·삭제·정리 잡의 DB 커넥션 풀은 알림함/마이페이지 조회 API 풀과 분리(U1 PAT-PERF-02 벌크헤드 상속) — 배치가 사용자 조회 지연을 유발하지 않게 한다.

## 6. 하드 제약·PBT 요건 (D37 · §6.6)

U8은 하드 제약 4계열 중 **계정 무결성**(삭제 연쇄·재가입 제한)의 마감 지점이다. NFR 관점에서 이는 **품질 게이트(머지 차단)** 요건이다.

- **NFR-U8-QUAL-01 (계정 무결성 100% — 삭제 연쇄, D37 머지 차단)**: 계정 삭제 연쇄의 무결성(전 모듈 파기·법정 보존 분리·유예 중 재가입 제한·핸들러 멱등)은 하드 제약 계정 무결성 계열로 **100% 통과가 머지 차단**이다. 외부(FCM)는 어댑터 fake, 상태 전이·파기 로직은 실코드(D37·§6.6).
- **NFR-U8-QUAL-02 (PBT 대상 — §6.6·G116)**: (a) **알림 발송 판정 순수 함수**(토글×방해금지×중복 억제 — 임의 clock·설정 입력에 대해 off면 미적재·방해금지면 인앱만·dedupe 10분 억제), (b) **계정 삭제 상태 머신**(ACTIVE→DEACTIVATED→PURGING→삭제, 유예 중 복구·재가입 제한 불변식), (c) **리마인드 재계산 멱등**(ItineraryChanged 중복 수신에도 dedupe 키 기준 동일 결과), (d) **알림함 보존**(90일 경계·읽음 상태 전이), (e) **직렬화 왕복**(알림·설정·내보내기 JSON). Kotest(서버)·fast-check(클라)·시드 로깅·수축(PBT-08). 속성 식별은 Functional Design(PBT-01).
- **NFR-U8-QUAL-03 (계약 포인트 — CP5 소비자)**: U8은 **CP5 소비자** — `ItineraryConfirmed`(리마인드 생성)·`ItineraryChanged`(재계산)·`TriggerFired`(severe만 푸시)·`ReflectionReady`·`TripEnded`·`StayRegistered`/`StayLinkedToTrip`·`AccountDeletionExpired`(파기) 구독의 멱등 처리 계약 테스트 통과. **전 유닛 완료 시점 E2E 종단 흐름(숙소 저장→등록→일정 생성→재계획→알림·기록, G118)에 U8 알림·삭제 경로 편입**.

## 7. 관측성 — U8 증분 + 마감 유닛 전 유닛 통합 종합 (§6.7 G144)

### 7.1 U8 관측 증분

- **NFR-U8-OBS-01 (알림 발송 성공/실패 계측)**: 발송 결과(`SENT/FAILED`)·FCM 실패율·dead letter·발송 큐 적체(due 대기 age)를 계측한다 — **A12(FCM 어댑터 실패율)·A10(잡·큐 적체 >15분)·커스텀 메트릭(dead letter 건수)**(SI §8.2). 침묵 실패 금지(발송이 조용히 사라지지 않음, ADR-0011).
- **NFR-U8-OBS-02 (삭제 배치 감사)**: 계정 삭제 요청·유예 만료·모듈별 파기 완료·최종 삭제를 **감사 로그**로 기록(SECURITY-03·14) — 법정 보존 분리 준수 증빙. 데이터 내보내기 이력도 감사 로그(§4 SEC-03). 감사 로그는 앱 역할 삭제 불가(REVOKE+IAM Deny, SI §5.3·§7.1).
- **NFR-U8-OBS-03 (보안 이벤트)**: 삭제·내보내기·재인증 실패·마케팅/위치 동의 변경은 보안 이벤트 메트릭(A14 — 인증 실패·민감 작업)에 편입(SECURITY-14, NFR-U1-SEC-27 상속).

### 7.2 마감 유닛 전 유닛 통합 관측성 종합 (G144 — 가볍게)

**U8은 마감 유닛이므로 전 유닛의 관측 대상이 SI §8.2 알람 프레임(A1~A15)에 수렴함을 확인한다** — 신규 관측 인프라는 없고, U8은 발송/삭제 소스만 편입한다.

| 관측 영역 | 알람(SI §8.2) | 소스 유닛 |
|---|---|---|
| 인프라 기본(5xx·p95·헬시 호스트·ECS·RDS) | A1~A9 | U1 정본(전 유닛 공통) |
| 잡·큐 적체(아웃박스·스케줄) | A10 | U6 폴링 · **U8 리마인드/삭제/정리 잡·dead letter** |
| 외부 API 쿼터 80% | A11 | U3 지도 · U5 LLM · U6 기상청 · **U8 FCM 쿼터** |
| 외부 어댑터 실패율·서킷 | A12 | U3·U5·U6 · **U8 FCM 발송 실패율** |
| SES 바운스·발송 실패 | A13 | U1 SES(내보내기 안내·시스템 메일 재사용) |
| 보안 이벤트(인증·민감 작업) | A14 | U1 · **U8 삭제/내보내기/동의 변경** |
| 비용 예산 | A15 | 전 유닛(U7 사진 스토리지·U5 LLM 비용 등) |

- **종합 소견**: 전 유닛 외부 의존(지도·LLM·기상청·SES·FCM)의 실패율·쿼터·서킷은 **A11·A12 단일 프레임**으로, 침묵 실패 금지 관측(외부 API 실패·LLM 오류·솔버 검증 실패·발송 실패·삭제 배치)은 **구조화 로그 + 커스텀 메트릭 + 대시보드 1장(`trippilot-{env}-ops`)** 으로 종결된다(G144·§6.7). U8은 이 프레임에 **발송·삭제·내보내기 소스만 추가**하며 신규 관측 인프라를 도입하지 않는다.

## 8. 미결·선결 항목

### 8.1 선결 과제 — 스토어·심사 (P9)

U8은 **비개발 선결 1건(P9)** 이 있으나, 본 NFR 산출물은 **어댑터 fake·플레이스홀더로 비차단**이다.

| 과제 | 차단 대상 | 본 문서의 대응 |
|---|---|---|
| **P9 앱스토어·플레이스토어 개발자 계정 및 심사 요건(지원 연락처 등)** | FCM 콘솔·푸시 인증서(APNs 키·Firebase 프로젝트)·스토어 심사(지원 연락처 N5·개인정보 삭제 경로) | `FcmPort` 어댑터를 **fake로 개발·테스트 선행**(D37 계층 분리), 실키는 `external/fcm` 시크릿 병렬 발급. **US-E09-13 고객 지원 연락처·정책 재열람·오픈소스 라이선스(N5)** 는 심사·위치정보법 상시 열람 요건 — 설정 화면에 선결. 삭제(D18)·내보내기(S9) 경로는 스토어 심사 데이터 삭제 요건 충족 |

### 8.2 벤더·인프라 미결 — FCM만 (전역 결정 상속)

**U8의 신규 외부 벤더·인프라 미결은 FCM(P9 병렬) 1건뿐이다.** U1이 PD-1~9를 전부 종결했고, 알림 인프라(D12·G198=A)·스케줄러(ShedLock)·아웃박스·SES·시크릿 프레임은 SI가 확정했으며, U8은 그 결과와 GD-1~7·SI를 상속한다.

| 잠재 결정 항목 | U8 처리 | 근거 |
|---|---|---|
| 클라우드·CI/CD·배포·롤백 | GD-1~7 상속(재질문 없음) | SI 헤더·U1 nfr-design §2 |
| 컴퓨트·DB·로그·스케줄러·아웃박스·SES 스택 | SI 정본 상속(재정의 금지 — ShedLock·아웃박스·NAT·SES) | SI §1.2·§3·§4·§8 |
| **푸시 인프라** | **D12·G198=A 확정 — FCM 단일(iOS 포함), 어댑터 중립·fake 선행** | requirements.md D12·G198 |
| 시크릿 프레임 | U1·SI 상속(FCM 키 = `external/fcm` 예약 활성화) | SI §6 |
| FCM 콘솔·APNs 키 | **P9 병렬 — 어댑터는 벤더 중립·fake 선행** | requirements.md P9·N5 |

## 9. 확장 규칙 컴플라이언스 요약 + 마감 유닛 종합 (NFR Requirements 스테이지 시점)

판정 기준: 본 스테이지 산출물이 U8 관점에서 각 규칙을 요구사항으로 반영/상속했는가.

| 규칙 | 판정 | 근거 |
|---|---|---|
| RESILIENCY-10 의존성 격리 | **준수(핵심)** | §3 — FCM 실패→인앱 선적재·재시도·dead letter, FcmPort 4요소 계약, 삭제 배치 at-least-once |
| RESILIENCY-01 워크로드 분류 | 준수 | §0.1 — 알림함 저장 High·FCM 발송 Medium 이하(인앱 폴백)·삭제 연쇄 High(법정) |
| RESILIENCY-05·07·09 | 준수(증분) | §7 — FCM 실패율·dead letter(A12)·발송 큐 적체(A10)·FCM 쿼터(A11)·비용(A15) |
| RESILIENCY-02·08·11·12 | 상속 | Multi-AZ·백업은 SI·U1 정본 — U8 신규 워크로드는 발송/삭제 잡(기존 스케줄러 흡수) |

| PBT-01~10 | 준수(현 단계분) | §6 — 발송 판정 순수 함수·삭제 상태 머신·재계산 멱등·알림함 보존·직렬화 왕복. 실행은 Code Generation(U8 DoD) |

**마감 유닛 컴플라이언스 종합(가볍게)**: 전 유닛(U1~U8)의 SECURITY-01~15·RESILIENCY-01~15는 **U1 전역 스캐폴드 + SI 인프라 정본**을 공통 상속하고, 각 유닛이 소관 증분(U3 외부 응답 검증·U5 LLM 경계·U6 위치 법정 로그·U7 사진 접근 제어·**U8 삭제 연쇄·법정 보존 분리·발송 관측**)을 더한 구조다. U8 마감 시점에 **법정 보존 3종(위치 로그·동의 증적·감사 로그)의 append-only·삭제 권한 분리**가 삭제 연쇄와 정합함을 확인한다(SECURITY-14 종결).

**차단 소견 없음** — U8의 '차단'은 비개발 선결 **P9(스토어 심사·지원 연락처)** 이며, 이는 출시 게이트 대상이고 본 NFR 산출물(FCM 어댑터 fake·`external/fcm` 시크릿 프레임·설정 정책 재열람·삭제/내보내기 경로)은 비차단이다.
