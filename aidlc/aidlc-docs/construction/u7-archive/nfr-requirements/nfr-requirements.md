# U7 기록·회고 — NFR 요구사항

> 2026-07-04 · CONSTRUCTION · U7 (M12 Travel Archive · M13 AI Reflection + `features/archive`·`shared/storage` 오프라인 로컬 큐)
> 시스템 전역 NFR의 정본은 [requirements.md](../../../inception/requirements/requirements.md) §6이다. 본 문서는 그 정본을 **U7 관점으로 전개**하되, U7이 **사진 저장(S3+CloudFront)이 프로젝트 최초로 등장하는 유닛**이므로 스토리지·미디어·오프라인 축은 유닛 증분을 충실히 확정하고, 인프라·플랫폼·회고 연산은 **전역 정본 상속(재정의 금지)** 한다. 충돌 시 정본이 우선한다.

## 0. U7 NFR 프로필 — "여행의 산출을 자산으로 · 사진 스토리지 첫 등장"

U7은 여행이 끝나도 남는 것(plan/actual/changelog 대조·사진·회고)을 만드는 유닛이다. NFR 관점의 3대 축:

1. **오프라인 우선 입력(핵심)** — 산간·해외 로밍에서도 기록 입력이 유실되지 않아야 한다(US-E8-12). 입력만 보장(조회는 온라인 전제 D24/Δ6), 로컬 큐·재접속 자동 동기화·충돌 무손실 병합(G74)이 본질.
2. **스토리지·미디어(핵심 · 프로젝트 최초)** — 사진 파이프라인은 U7이 처음 도입한다. 클라 압축(5MB/2048px)·서버 썸네일·S3+CloudFront 서빙·EXIF GPS 프라이버시(G75/G145/G168)가 인프라 증분의 실체다. **U1~U6에는 없던 신규 AWS 리소스(S3 버킷·CloudFront)가 실제로 생긴다**(SI §5.2·§5.4 예약분 활성화).
3. **데이터 정합 + 복원력** — changelog diff 누적 재구성=스냅샷 동등성(G132), plan/actual 대조, 업로드 실패 큐, 회고 LLM 실패 기본 카드 폴백.

**회고 연산은 U5 C1 자산 재사용**(신규 LLM 벤더·인프라 표면 0) — M13은 상위 티어 소비자로 호출한다.

### 0.1 워크로드 중요도 (requirements.md §6.3)

| 컴포넌트 | 중요도 | 불가용 시 영향 | U7 대응 |
|---|---|---|---|
| 기록 입력(M12 actual·사진·메모) | **High** | 입력은 로컬 큐로 보존, 조회·생성만 지연(§6.3) | 오프라인 로컬 큐(§2) — 서버 순단에도 입력 무손실 |
| 회고·요약·스타일 분석(M13) | **High** | 생성만 지연(로컬 큐 보존 후 재평가) | 비동기 이벤트 구동 + LLM 실패 시 기본 카드 폴백(§4) |
| 사진 서빙(S3+CloudFront) | Medium | 이미지 로딩 실패 — 썸네일 CDN·재시도, 메모/기록 텍스트는 정상 | CDN 캐시·서명 URL, 이미지 실패가 기록 열람 차단하지 않음 |

- 기록·회고는 **여행 중/후 자산 축적**이므로 조회 지연은 수용 가능하나 **입력 유실은 불가**(High의 본질 — 로컬 큐가 가용성 보완). Critical(여행 중 일정 접근 M18·M6·M8)은 U6 이하 소관.

## 1. 오프라인 우선 입력 (핵심 · Δ6/D24·G74, RESILIENCY-01)

### 1.1 오프라인 보장 경계 (D24/Δ6 — 입력만)

| 대상 | 오프라인 | 근거 |
|---|---|---|
| 방문 체크·수동 체크인·사진 첨부·메모 작성 | **로컬 저장 보장** — 네트워크 무관 기기 로컬 큐 적재 | US-E8-12, RESILIENCY-01 M12 High |
| 일정·기록 **조회** | **온라인 전제** — 실패 시 오류·재시도 안내만(오프라인 조회 미보장) | D24/Δ6, unit-of-work U7 제외 |
| 당일 회고 자동 생성 | 오프라인 중 **보류**, 복구 후 생성 또는 기본 카드 대체 | US-E8-12, §7 |

- **NFR-U7-OFF-01(로컬 큐 영속)**: 오프라인 입력은 앱 재시작·강제 종료에도 살아남는 **기기 영속 저장**(`shared/storage`)에 적재하고 '동기화 대기'로 표시한다. 메모리 큐 금지(전원/크래시 유실).
- **NFR-U7-OFF-02(재접속 자동 동기화)**: 네트워크 복구 감지 시 큐 항목을 순차/재시도 동기화. 사용자 수동 개입 없이 진행하되 진행/실패 상태를 표시(침묵 실패 금지 ADR-0011).
- **NFR-U7-OFF-03(멱등 동기화)**: 각 큐 항목은 클라 생성 ID(멱등 키)를 부여해 **재시도로 중복 레코드가 생기지 않는다**(네트워크 재시도·부분 성공 내성). PBT 대상(직렬화 왕복·임의 재전송 순서 수렴).

### 1.2 동기화 충돌 해소 (G74 — 무손실)

- **NFR-U7-OFF-04(레코드 단위 버전 비교)**: 오프라인 입력과 서버 데이터 충돌 시 **방문 기록 레코드 단위 버전 비교**(레코드 버전 카운터) → 항목별로 사용자가 유지할 버전 선택(US-E8-12).
- **NFR-U7-OFF-05(사진·메모 합집합 병합)**: 사진·메모 **추가**는 충돌 아닌 **합집합 병합**(양쪽 보존 — 어느 경로도 무손실). 사용자 선택은 상충 필드(체류 시각 등)로 최소화(리스크: 충돌 UX 복잡도 — unit-of-work U7).
- **NFR-U7-OFF-06(무손실 불변식 — PBT 1급)**: 임의 순서·임의 중복의 동기화 시퀀스에 대해 **어떤 경로에서도 사용자 데이터가 소실되지 않는다**(합집합·양쪽 보존). 병합 수렴(임의 순서 동일 결과)·데이터 무손실을 Kotest/fast-check 속성 테스트로 강제(§6.6 PBT).

## 2. 스토리지·미디어 (핵심 · 프로젝트 최초 · G75/G145/G168)

### 2.1 사진 정책 (G75/G145 — requirements.md §7.3)

| 항목 | 값 | 근거 |
|---|---|---|
| 장소당 최대 | **20장** | G75/G145, US-E8-02 |
| 클라이언트 압축 | **장당 ≤5MB · 긴 변 2048px** — 업로드 전 클라에서 압축 | G75/G145, US-E8-02 |
| 원본 | **기기 보관**(서버 미전송) — 서버 부담·비용 축소 | G75, unit-of-work U7 리스크 완화 |
| 썸네일 | **서버 생성**(목록·지도 핀 미리보기용) | G75, US-E8-08 지도 핀 미리보기 |
| 스토리지·서빙 | **S3 호환 오브젝트 스토리지 + CDN** | **G168** — 확정: S3 + CloudFront(tech-stack §2·§3, infra §1·§2) |
| 촬영/저장 시각·귀속 | 사진 EXIF 시각·첨부 장소 함께 저장 | US-E8-02 |

- **NFR-U7-MEDIA-01(클라 압축 선행)**: 업로드는 반드시 클라 압축(5MB/2048px) 후 수행 — 원본 업로드 경로가 **구조적으로 존재하지 않는다**(대역폭·스토리지·프라이버시 동시 충족). 압축은 클라 리소스 사용(§3 성능).
- **NFR-U7-MEDIA-02(장당 20장 상한)**: 장소당 사진 수 서버 강제(21번째 거부). 클라도 사전 차단하되 정본은 서버(우회 방지).
- **NFR-U7-MEDIA-03(서버 썸네일)**: 업로드 완료된 압축 이미지로부터 서버가 썸네일을 생성한다(목록·지도 핀). 썸네일 생성 실패는 원본(압축본) 표시 폴백 — 첨부 자체를 차단하지 않음.
- **NFR-U7-MEDIA-04(CDN 서빙)**: 사진·썸네일은 CloudFront 경유 서빙(캐시로 이미지 로딩 지연 최소화·오리진 부하 축소). 버킷 직접 접근 차단(OAC — infra §2).

### 2.2 EXIF GPS 제거 위치 (G168 — 프라이버시 결정, tech-stack §4에서 대안 비교)

- **NFR-U7-MEDIA-05(EXIF GPS 기본 제거)**: 저장 사진의 EXIF GPS는 **기본 자동 제거**(G185 — 기본 자동 제거 원칙). U7은 사진 **저장**에 이 원칙을 적용하며, 커뮤니티 게시 시 EXIF 처리(G185 '정확 위치 공개' 선택)는 U10 소관(unit-of-work U7 제외).
- **NFR-U7-MEDIA-06(원본 GPS 기기 이탈 최소화 — G168 권고)**: G75가 원본을 기기 보관으로 확정했으므로, **EXIF GPS 제거를 클라 전처리(압축 단계)에서 수행**해 원본 위치 정보가 서버에 도달하지 않게 하는 것이 프라이버시 최적이다(tech-stack §4 권고). 서버는 **심층 방어로 재검증·재제거**(클라 우회·잔여 메타 대비) — 트레이드오프는 tech-stack §4.

## 3. 성능 (§6.1)

U7은 실시간 UI 차단 경로가 얕다(사진 업로드=백그라운드, 회고 생성=비동기 이벤트 구동). §6.1이 U7 기능에 부여한 명시 목표는 없으므로 아래는 **제안 목표**(서버 처리 시간 기준, 네트워크 RTT 별도)다.

| # | 대상 | 목표 | 근거 |
|---|---|---|---|
| NFR-U7-PERF-01 | 사진 업로드(presigned URL 발급 API) | p95 **< 300ms** — 업로드 자체는 **비차단 백그라운드 큐**(방문 체크·메모는 업로드와 독립 즉시 저장 US-E8-02) | §6.1 화면 전환 300ms 준용, RESILIENCY-01 |
| NFR-U7-PERF-02 | 서버 썸네일 생성 | 첨부 확정 응답 내 **< 2초/장**(동기) 또는 비동기 — 실패 시 원본 표시 폴백 | NFR-U7-MEDIA-03 |
| NFR-U7-PERF-03 | 회고 생성(당일/전체/스타일) | **비동기**(DayClosed/TripEnded 구동) — 사용자 대기 UI 아님. C1 상위 티어 호출 **타임아웃 예산(권고 25초)**, 초과·실패 시 기본 카드 폴백. 완료 시 인앱 카드/푸시(ReflectionReady) | US-E8-06, §7, D11 |
| NFR-U7-PERF-04 | 기록·요약 조회 | 기록 상세/요약 조회 서버 p95 **< 500ms**, 화면 전환 300ms(§6.1). 이미지는 CDN 캐시 히트로 로딩 최소화 | §6.1, NFR-U7-MEDIA-04 |

- **NFR-U7-PERF-05(사진 로딩 = CDN)**: 기록/회고 화면의 이미지 로딩은 CloudFront 캐시가 담당(목록은 썸네일, 상세는 압축 원본). 오리진(S3) 히트 최소화가 지연·비용 동시 최적.

## 4. 복원력 (핵심 · §6.2·§6.3, RESILIENCY-10, ADR-0011)

### 4.1 사진 업로드 복원력 (침묵 실패 금지)

- **NFR-U7-RES-01(업로드 실패 로컬 큐·재시도)**: 사진 업로드 실패 시 로컬 큐 보존 + **지수 백오프 재시도 3회**(unit-of-work U7 리스크 완화). 최종 실패 시 사용자에게 상태 표시(동기화 대기·재시도 버튼) — 침묵 실패 금지.
- **NFR-U7-RES-02(입력 독립 저장)**: **메모·방문 체크는 사진 업로드 실패와 무관하게 저장**(US-E8-02 수용 기준) — 사진은 사후 재동기화. 사진 첨부가 방문 기록 저장을 차단하지 않는다.
- **NFR-U7-RES-03(원본 기기 보관 = 재시도 소스)**: 원본이 기기에 남아 있으므로(G75) 업로드 실패는 언제든 재시도 가능 — 서버 상태와 무관한 무손실 복구 경로.

### 4.2 회고 LLM 복원력 (기본 카드 폴백)

- **NFR-U7-RES-04(회고 LLM 폴백)**: C1 호출 실패·타임아웃 시 **빈 화면 대신 '방문 N곳·이동 Nkm·사진 N장' 기본 카드**를 제공(US-E8-06·08). C1의 서킷·타임아웃은 U5 자산 상속(RESILIENCY-10 — 인프라는 SI §8.2 A11·A12 알람 제공, U5가 첫 소비자).
- **NFR-U7-RES-05(부분 데이터 명시)**: 사진 0장·GPS 미동의·메모만 등 부분 데이터 상태에서 가용 데이터만으로 회고 생성하고 **누락을 회고에 명시**('사진 없음'·'위치 기록 없음' US-E8-06). 생성 불가 항목(사진 하이라이트·지도 경로)은 제외하되 침묵하지 않음.
- **NFR-U7-RES-06(스타일 분석 게이트)**: 스타일 분석은 방문 10곳 게이트로 빈약 입력 차단(G76·unit-of-work U7) — 미달 시 미제공 안내.

### 4.3 워크로드·가용성

- **NFR-U7-RES-07**: M12·M13 = **High**(§6.3) — 입력은 로컬 큐 보존, 조회·생성만 지연. Multi-AZ·백업·헬스체크는 SI §1~3·§8 상속(신규 워크로드 아님). 사진 스토리지 버저닝은 SI §5.2(이전 버전 30일)·RESILIENCY-12 상속.

## 6. 규모 (§6.8 G142 — DAU 1천)

### 6.1 사진 스토리지·CDN 볼륨·비용

- **NFR-U7-SC-01(스토리지 증가율)**: DAU 1천 · 활성 여행 ≤1/사용자(D21) · 장소당 ≤20장 · 클라 압축 후 장당 ~1–2MB(5MB 상한) · 원본 기기 보관(서버 미저장). MVP 규모에서 **월 누적 수십 GB 수준**(과설계 금지). SI §5.1 버저닝 + 이전 버전 30일 파기(SI §5.2)로 무한 증가 억제.
- **NFR-U7-SC-02(비용 상한)**: 사진 S3 + CloudFront 증분은 **월 수 달러~십수 달러 수준**(infra §6 추정) — SI §10 합계에 소액 가산. **AWS Budgets 알람(SI §8.2 A15)** 이 스토리지·CDN 급증(예: 업로드 폭주·핫링크)을 포함해 감시. 별도 비용 통제 리소스 불요.
- **NFR-U7-SC-03(서버 부담 축소 원칙)**: 원본 기기 보관(G75) + 클라 압축 선행으로 **오리진 저장·전송량을 구조적으로 축소**. 썸네일 생성만 서버 CPU 사용(저빈도 — 기존 ECS 태스크 흡수, 별도 컴퓨트 불요 G142).
- **NFR-U7-SC-04(과설계 금지)**: DAU 1천에서 이미지 처리에 **Lambda·EventBridge·전용 트랜스코딩 파이프라인·ElastiCache 미도입**(tech-stack §5·§6). 전환 트리거: 사진 처리량이 ECS API 지연에 간섭하거나 월 처리 수만 장 초과 시 Lambda 썸네일 분리 재평가(SI §1.2 잡 분리 트리거와 정합).

## 7. 회고 (C1 재사용 · 타임아웃)

- **NFR-U7-REF-01(C1 재사용)**: M13 회고·전체 요약·스타일 분석은 **U5 C1 LLM Gateway를 상위 티어(D11)로 재사용**한다 — 신규 벤더·라우팅·시크릿 없음(component-dependency §2.13 M13→C1). 서버 재조회(D31)·rate-limit·비용 계측·출력 스키마 검증은 U5 자산 상속.
- **NFR-U7-REF-02(비동기 이벤트 구동)**: 당일 회고=DayClosed 트리거, 전체 요약=TripEnded 트리거(component-dependency §40). 사용자 대기 UI가 아니라 **완료 시 알림**(인앱 카드/푸시 — ReflectionReady 이벤트, 발송은 U8 M14 소관. U7은 발행까지 unit-of-work U7).
- **NFR-U7-REF-03(타임아웃·폴백)**: C1 회고 호출 타임아웃(권고 25초 예산) 초과·실패 시 기본 카드 폴백(NFR-U7-RES-04). LLM 비용·품질 편차는 U5 계측 대시보드(SI §8.2 LLM 슬롯) 상속.
- **NFR-U7-REF-04(재생성·덮어쓰기)**: 회고 재생성 시 **수정본 존재하면 덮어쓰기 경고 후 확인 시에만 교체**(G78·US-E8-07). 종료 후 편집·수동 재생성만 갱신(C11 — 자동 갱신 금지).
- **NFR-U7-REF-05(개인화 신호)**: 스타일 분석 결과는 다음 여행 개인화 신호로 취향 7종 축 택소노미에 매핑(G76·US-E8-10) — 스키마는 M2와 공유.

## 8. 미결·선결 항목

### 8.1 선결 과제 — U7 신규 없음

- **P6(LLM 벤더·비용·국외 이전)**: **U5에서 기완료** — 회고는 C1 재사용, 신규 벤더 계약·비용 산정 불요.
- **P1(위치기반서비스사업 신고)·P7(위치기반서비스 약관)**: **U6에서 기완료** — GPS 파기 연동(D34)은 U1 동의 모델·U6 수집 위에서 동작.
- **오브젝트 스토리지·CDN 선정**: **Infrastructure Design 결정 사항**(unit-of-work U7 §9 이관) — SI §5.2·§5.4 예약분(S3·CloudFront)을 활성화로 종결(비외부 선결·비차단).

### 8.2 벤더 종속 미결 — 없음 (전역 결정 상속)

- 클라우드=AWS(GD-1)·스토리지=S3·CDN=CloudFront(SI §5 예약)·LLM=U5 벤더(D11) 전부 기확정. U7은 재확정 없음 — 선택지 있는 지점(EXIF 위치·썸네일 위치·업로드 방식·오프라인 큐)만 tech-stack에서 결정.
