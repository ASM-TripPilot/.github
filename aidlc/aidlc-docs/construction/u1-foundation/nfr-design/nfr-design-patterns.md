# 1. 보안 패턴

## PAT-SEC-01 인증 필터 체인 — deny-by-default `[전역]`

**문제**: 엔드포인트가 늘어날 때마다 "이 핸들러에 인증을 붙였는가"를 사람이 기억해야 하는 구조는 반드시 회귀한다(허용이 기본이면 누락 1건 = 무인증 노출). 로그인 필수 원칙(D22) 아래에서 인증 우회는 곧 전 데이터 노출이다.

**적용(설계)**: Spring Security 6.4(S-3) 단일 필터 체인을 `app` 모듈이 소유하고, 기본 규칙을 `anyRequest().authenticated()`로 고정한다. 공개 엔드포인트는 **명시적 화이트리스트만** 허용하며(NFR-U1-SEC-16의 전수 목록: 가입·로그인·토큰 갱신·이메일 인증/재발송·비밀번호 재설정·부트스트랩 비세션부·shallow 헬스체크), 화이트리스트 추가는 설계 리뷰 필수 항목이다. 필터 순서 정본(tech-stack §2.3):

```text
[요청]
  → 1. 상관 ID 필터 (MDC 주입 — PAT-OBS-01)
  → 2. 요청 크기·컨텐츠 타입 가드 (64KB 상한, NFR-U1-SEC-13)
  → 3. JWT 인증 필터 (Bearer 서명·exp·iss·aud 검증, STATELESS — PAT-PERF-01)
  → 4. 브루트포스/레이트리미터 필터 (공개 인증 엔드포인트 한정 — PAT-SEC-03)
  → 5. 인가 규칙 (anyRequest().authenticated() + 화이트리스트)
  → 6. 보안 헤더 라이터 (HSTS·nosniff·DENY·Referrer-Policy, SECURITY-04)
  → 7. 전역 예외 변환 (일반화 메시지 401/403 — SECURITY-15)
[핸들러]
```

객체 수준 인가는 필터 아래 계층에서 이중화한다: 전 리소스 접근은 토큰 주체(accountId) 기준으로만 조회(IDOR 원천 차단, NFR-U1-SEC-18), 계정 상태별 접근 축소(DELETION_PENDING은 철회·열람 외 거부 — FLOW-6 S4)는 인가 계층에서 일괄 강제. CORS는 전면 거부 기본(NFR-U1-SEC-19). 알고리즘 다운그레이드(none·대칭 전환)는 검증기 레벨에서 거부(NFR-U1-SEC-17).

**U1 적용 지점**: `app` 모듈 SecurityConfig(스캐폴드 산출물 — 전 유닛이 상속), M1 공개 엔드포인트 화이트리스트, M1·M2 전 조회의 accountId 스코핑. 회귀 방지는 아키텍처 테스트(Konsist/ArchUnit — "인증 가드 없는 공개 핸들러" 검출, NFR-U1-SEC-20)를 PR CI 게이트로.

**검증 기준**: SECURITY-08(deny-by-default·매 요청 토큰 검증·객체 수준 인가) — 화이트리스트 외 엔드포인트에 무토큰 요청 시 401, 타 사용자 ID 경로 접근 시 404/403, 아키텍처 테스트 CI 통과. SECURITY-04(헤더) — 전 응답 헤더 존재 통합 테스트. SECURITY-15 — 401/403 응답에 스택·내부 경로 미노출.

## PAT-SEC-02 리프레시 토큰 회전·재사용 감지 `[U1 — 발급·검증은 전역 소비]`

**문제**: 90일 수명 리프레시 토큰(D36)은 탈취 가치가 크다. 회전 없는 장수 토큰은 탈취 시 90일간 유효하고, 회전만 있고 재사용 감지가 없으면 탈취범과 정규 사용자 중 나중에 갱신한 쪽만 살아남아 탈취를 **탐지하지 못한다**.

**적용(설계)**: FLOW-5 상태 머신을 그대로 구현 정본으로 한다.

- **회전**: 갱신마다 구 토큰 `rotatedAt` 마킹 + 동일 `chainId`로 신규 RefreshSession 발급 — 체인당 현행 토큰 정확히 1개(INV-R1).
- **재사용 감지**: `rotatedAt≠NULL`인 토큰 제시 = 탈취 신호 → **해당 체인 전체 즉시 revoke**(현행 토큰 포함, INV-R2) + 보안 이벤트 즉시 알림(PAT-OBS-02 b항) + AuthenticationRequired. 타 기기 체인은 불간섭(INV-R3, 다기기 허용 D36).
- **저장**: 서버는 tokenHash만 저장(원문 금지, NFR-U1-SEC-06). 무효화 매트릭스(NFR-U1-SEC-05): 로그아웃=해당 체인, 비밀번호 변경/재설정·계정 삭제 요청=전 체인, 재사용 감지=해당 체인.
- **클라이언트 정합**: axios 단일 비행(single-flight) 갱신 큐(C-6)로 동시 401 시 회전 1회만 발생 — 정규 사용자의 자기 재사용 오탐을 구조적으로 제거.
- **경합 방어**: 회전은 조건부 UPDATE(현행 토큰 && rotatedAt IS NULL) 원자 연산으로 — 동시 갱신 2건 중 1건만 성공, 패배 건은 재사용 판정 경로가 아닌 신규 토큰 재시도 유도(단일 비행이 1차 방어, 서버 원자성이 2차).

**U1 적용 지점**: M1 `refreshTokens`(FLOW-5), RefreshSession 저장소(logical-components.md LC-4), 클라이언트 shared/api 인터셉터. 정확성은 PBT U1-P2(stateful — 다기기×명령 시퀀스)가 상시 검증.

**검증 기준**: SECURITY-12(세션 무효화·재사용 대응) — U1-P2 속성 4건(체인당 현행 ≤1 / 재사용 시 발급 0+체인 무효 / 타 체인 불간섭 / 90일 만료 후 갱신 불가) PR CI 통과, 재사용 감지 보안 이벤트 발행 통합 테스트. SECURITY-03 — 토큰 원문 로그 미출현(마스킹 테스트).

## PAT-SEC-03 브루트포스 점진 지연·레이트리미터 `[전역 — 정본 U1]`

**문제**: 온라인 추측 공격(단일 계정 사전 공격, 계정 열거, 패스워드 스프레이)과 외부 쿼터 고갈(메일 재발송 남용)은 인증 표면의 상수적 위협이다. 즉시 잠금만 쓰면 공격자가 타인 계정을 고의로 잠그는 DoS가 되고, 무방비면 추측이 무제한이다.

**적용(설계)**: 2축 방어 + 점진 지연(NFR-U1-SEC-07·08 수치를 설계 정본으로):

| 축 | 규칙 | 응답 |
|---|---|---|
| 계정 단위 | 연속 실패 ≤5회 지연 없음 → 6~10회 지수 지연 2·4·8·16·32초 → 10회 초과 15분 일시 잠금 | 잠금 중에도 "로그인할 수 없습니다" 일반화(계정 존재 비노출) |
| 출처(IP/기기) 단위 | 로그인 분당 10·시간당 100 / 가입·재설정·닉네임 확인 동일 축 적용 / 메일 재발송 분당 1·일 5회(G22 — 서버 강제, 외부 쿼터 보호 겸용) | 429 + retryAfter(재발송은 카운트다운 UI, FLOW-2 E5) |

카운터 정본은 **PostgreSQL**(인스턴스 로컬 금지 — NFR-U1-SEC-09·AV-02, 초기 규모 0.5 RPS에서 DB로 충분, 공유 캐시 승격은 PD-9 대기). 지연 수치·상한은 외부화 설정(NFR-U1-MT-04)으로 재빌드 없이 조정. 성공 시 카운터 리셋. CAPTCHA는 미도입 — 도입 트리거(자동화 공격 관측 지표)만 정의. 이 레이트리미터 필터는 전역 재사용 컴포넌트(logical-components.md LC-3)로, 후속 유닛의 공개·고비용 엔드포인트(예: U5 LLM 호출 사용자별 상한 — requirements §6.4)가 동일 메커니즘을 재사용한다.

**U1 적용 지점**: 필터 체인 4단(PAT-SEC-01), 로그인(FLOW-1·2)·재발송(FLOW-2 S7)·닉네임 확인(FLOW-7). 단조성·리셋·창 상한은 PBT U1-P13(재발송 60초/24h 창)·U1-P14(지연 비감소·리셋)가 검증.

**검증 기준**: SECURITY-12(브루트포스 방어) — U1-P13·P14 통과, 잠금 응답 일반화 테스트. SECURITY-11(공개 엔드포인트 rate-limit·오용 시나리오 명시) — 본 절이 오용 시나리오 정본. RESILIENCY-09(외부 쿼터 보호) — 재발송 상한 서버 강제 테스트. 지연 발동·잠금은 감사 이벤트(PAT-OBS-02) 기록 확인.

## PAT-SEC-04 비밀번호 해시 전략 — argon2id + 투명 재해시 `[U1]`

**문제**: 오프라인 크래킹(DB 유출 시)은 해시 강도가 유일한 방어다. 동시에 해시 비용은 로그인 지연(p95 1,000ms 예산, §1.2)과 서버 자원(메모리·CPU)을 직접 소비하며, 하드웨어 발전에 따라 파라미터를 **운영 중 상향**할 수 있어야 한다.

**적용(설계)**:

- **알고리즘**: argon2id(메모리 하드 — GPU/ASIC 내성, PHC 우승·RFC 9106·OWASP 1순위). 초기 파라미터 `m=19456KiB, t=2, p=1`(OWASP 최소 권고선)에서 시작해 검증 지연 실측 100–300ms로 캘리브레이션(NFR-U1-SEC-02 (c)).
- **자기 기술·업그레이드 경로**: `DelegatingPasswordEncoder`(`{argon2}` 프리픽스)로 감싸 해시 문자열에 알고리즘·파라미터 자기 기술 — 파라미터 상향 시 **로그인 성공 시점에 투명 재해시**(사용자 무감지 마이그레이션, NFR-U1-SEC-02 (a)). 파라미터는 외부화 설정(NFR-U1-MT-04).
- **용량 계획 연동**: 해시 1건당 ~19MiB 오프힙 메모리 × 동시 로그인 병렬도가 인스턴스 사이징 입력(NFR-U1-SC-01 — Infrastructure Design 전달). 해시는 **서버 전용**(클라이언트 키 스트레칭 금지, §1.4).
- **경계 조건**: 최대 128자 입력 검증(NFR-U1-SEC-01 — argon2는 절단 없음), 소셜 계정은 passwordHash NULL + 비밀번호 로그인 경로 차단(NFR-U1-SEC-03), 가입 시 유출 목록 2단 검증(로컬 상위 목록 필수 + k-익명성 온라인 1초 타임박스 fail-open, NFR-U1-SEC-04 — 대상 서비스는 PD-6 대기).

**U1 적용 지점**: M1 가입(FLOW-2 S1·S5)·로그인·비밀번호 변경/재설정. 재설정 완료 시 전 체인 revoke(PAT-SEC-02 매트릭스).

**검증 기준**: SECURITY-12(적응형 해시·유출 목록) — 해시 문자열 프리픽스 자기 기술 테스트, 구 파라미터 해시 로그인 시 재해시 발생 테스트, 검증 지연 실측 100–300ms 캘리브레이션 기록, 유출 목록 1차(로컬) 차단 테스트 + 온라인 실패 주입 시 fail-open·계측 확인(RESILIENCY-10).

## PAT-SEC-05 시크릿 관리 — 외부 주입·하드코딩 금지 `[전역]`

**문제**: U1은 시크릿 밀도가 가장 높은 유닛이다 — 소셜 클라이언트 시크릿 3종+Apple p8 서명 키, JWT 서명 키, 메일 API 키, DB 자격 증명. 소스·설정 파일 포함은 유출 시 회수 불가(git 이력 영구)이고, 로테이션 불가 구조는 사고 대응을 마비시킨다.

**적용(설계)**:

- **주입 인터페이스**: 전 시크릿은 기동 시 환경 주입(외부화 설정)으로만 수취 — 코드·리포지토리·이미지에 미포함(NFR-U1-SEC-11). 설정 키 계약(이름·형식)만 U1이 소유하고, 공급자는 어댑터 뒤(logical-components.md LC-6).
- **로테이션 가능 구조**: JWT 서명 키는 **JWK 셋 + `kid` 롤오버**(Nimbus 지원, S-5) — 신 키 발급·구 키 검증 병행 창을 두어 무중단 키 교체. Apple client_secret은 p8로 서명한 ES256 JWT(수명 ≤6개월)라 **주기 재생성이 구조적으로 강제**됨(tech-stack §2.4) — 만료 전 갱신을 운영 체크리스트(OPS-01 변경 기록 대상)에 포함.
- **AWS 매핑(참조 — 확정은 Infrastructure Design, PD-2·3)**: 시크릿 저장 = **AWS Secrets Manager**(로테이션 스케줄) 또는 SSM Parameter Store(SecureString), 키 계층 = **AWS KMS**(저장 암호화 SECURITY-01·JWT 서명 키 보호). GitHub Actions(GD-3)에서는 OIDC 연동 임시 자격 증명으로 시크릿 조회(장수 액세스 키 금지).
- **개발·CI**: 로컬/CI는 fake 어댑터·더미 시크릿(D37 — 실 IdP·메일 호출 없음), 시크릿 스캔(gitleaks류)을 PR CI 게이트에 포함(SECURITY-10 공급망 게이트와 동일 워크플로).

**U1 적용 지점**: `app` 모듈 설정 로더(스캐폴드), SocialOAuthPort 어댑터 4종·MailDeliveryPort·JWT 서명기·DataSource.

**검증 기준**: SECURITY-12(하드코딩 금지) — 리포지토리 시크릿 스캔 CI 통과(0건), 설정 키 부재 시 기동 실패(fail-fast — 묵시 기본값 금지). PD-2·3 해소 시 로테이션 절차가 OPS-01 변경 기록에 연결되는지 확인.

# 2. 복원력 패턴

## PAT-RES-01 외부 의존 격리 계약 — 타임아웃+재시도+서킷 브레이커+폴백 `[전역]`

**문제**: U1의 외부 의존(소셜 IdP 4종, 메일 발송, 유출 목록 온라인 조회)은 우리가 제어할 수 없는 가용성을 가진다. 무한 대기는 스레드 고갈→전면 장애로 증폭되고(Critical 워크로드 M1, §6.3), 침묵 실패는 사용자를 원인 불명 상태에 방치한다(ADR-0011).

**적용(설계)**: 전 외부 호출은 Port/Adapter 뒤에서 다음 **4요소 계약**을 의무화한다(계약 누락은 설계 리뷰 차단 항목). 이 계약 표는 전역 정본이며, 후속 유닛의 외부 의존(지도·기상청·LLM·FCM — services.md §0.1)도 동일 형식으로 자기 표를 작성한다.

| 의존 (Port) | 타임아웃 | 재시도 | 서킷 브레이커 | 폴백·사용자 고지 |
|---|---|---|---|---|
| 소셜 IdP 4종 (SocialOAuthPort) | 어댑터별 3초 (콜백 전체 p95 3,000ms 예산 내, §1.2) | 없음 — 토큰 교환은 비멱등(코드 1회용), 재시도는 사용자 버튼으로 | 제공자별 독립 서킷 (실패율 임계 초과 시 open → 즉시 실패) | 해당 제공자 신규 로그인만 "잠시 후 재시도"(FLOW-1 E2, 계정 미생성 보장) — 타 제공자·이메일 로그인·기존 세션 무영향(PAT-RES-03) |
| 메일 발송 (MailDeliveryPort) | 발송 시도당 5초 | 비동기 잡 재시도 — 지수 백오프+지터, 최대 N회 후 실패 확정 | 발송 서비스 단위 서킷 (open 시 잡 대기·조기 실패) | 접수만 동기 확인(§1.2), 실패 확정 시 미인증 유지+재발송 UI(FLOW-2 E4) — 가입 트랜잭션은 성공 유지, 실패율 계측 |
| 유출 목록 온라인 조회 (k-익명성) | 1초 타임박스 | 없음 (가입 지연 직결) | 서비스 단위 서킷 | **fail-open** — 로컬 목록 통과만으로 진행 + 실패 계측(NFR-U1-SEC-04) |
| PostgreSQL | 커넥션 획득·쿼리 타임아웃 명시(PAT-PERF-02) | 트랜잭션 단위 — 멱등 연산만 제한 재시도 | 해당 없음(서킷 대신 풀 상한이 벌크헤드) | Multi-AZ 페일오버(관리형) 대기 — 열화 시 동작은 PAT-RES-03 표 |

공통 규칙: (a) 재시도는 **멱등 연산에만**, 지수 백오프+지터, 총 재시도 시간이 상위 타임아웃 예산을 넘지 않게(재시도 폭풍 금지), (b) 서킷 open/half-open 전이는 계측 이벤트(PAT-OBS-03), (c) 폴백은 반드시 사용자 고지 문구·계측과 짝(침묵 실패 금지), (d) 외부 호출은 DB 트랜잭션 밖(services.md §0.2 — 예: 가입 TX 커밋 후 메일 잡 적재).

**U1 적용 지점**: SocialOAuthPort 어댑터 4종(FLOW-1 S1·S2), 메일 발송 큐(logical-components.md LC-5, FLOW-2 S6·S7), 유출 조회 어댑터(FLOW-2 S1). 구현 라이브러리(Resilience4j 상당)는 Code Generation에서 확정하되 계약 수치는 본 표가 정본.

**검증 기준**: RESILIENCY-10 — 전 외부 호출 명시 타임아웃 존재(코드 리뷰+계약 테스트), 서킷 구현 또는 N/A 문서화, 우아한 저하 문서화(본 표+PAT-RES-03), 풀 상한 구성(PAT-PERF-02). D37 — 어댑터 fake로 실패 주입 테스트(타임아웃·5xx·서킷 open) PR CI 포함.

## PAT-RES-02 헬스체크 — shallow `/health` + deep `/health/deep` `[전역]`

**문제**: 헬스체크가 없으면 LB가 죽은 인스턴스로 트래픽을 보낸다. 반대로 **deep 체크를 LB 판정에 쓰면** DB 순단 1회가 전 인스턴스 동시 unhealthy → 대량 축출 → 전면 장애로 **증폭**된다(순단이 정전이 되는 안티패턴).

**적용(설계)**: 2계층 분리(NFR-U1-AV-06)를 전역 표준으로 한다.

| 엔드포인트 | 검사 내용 | 소비자 | 판정 용도 |
|---|---|---|---|
| `GET /health` (shallow) | 프로세스 생존·HTTP 서빙 가능 (외부 의존 미검사) | **LB 대상 등록·롤링 배포 게이트(GD-5)** | 인스턴스 교체·트래픽 라우팅 |
| `GET /health/deep` (deep) | DB 연결(SELECT 1 수준)·필수 의존 상태 + 선택적으로 서킷 상태 요약 | **관측·알람 전용**(PAT-OBS-03) — LB 판정 금지 | 열화 감지·인시던트 트리거(OPS-02) |

설계 규칙: (a) shallow는 무인증 공개(화이트리스트, PAT-SEC-01), deep은 내부 접근으로 제한(정보 노출 최소화 — SECURITY-09), (b) deep의 의존 검사는 자체 짧은 타임아웃(예: 2초)을 가져 헬스체크 스레드 고갈 방지, (c) 롤링 배포(GD-5)의 인스턴스 교체 판정은 shallow + 기동 완료(readiness) 신호로 — 신규 인스턴스가 준비 전 트래픽을 받지 않게, (d) 공개 API에 대한 합성(synthetic) 모니터링은 Operations 이연(OPS-03 시나리오와 함께 — RESILIENCY-06의 SHOULD 항목).

**U1 적용 지점**: `app` 모듈 스캐폴드 산출물(U1 DoD). AWS 매핑(참조): ALB 대상 그룹 헬스체크=shallow, CloudWatch 캔어리/알람=deep.

**검증 기준**: RESILIENCY-06 — 양 엔드포인트 존재·응답 통합 테스트, deep의 DB 단절 주입 시 열화 보고(Testcontainers), LB 통합 구성은 Infrastructure Design에서 shallow 지정 확인, "deep을 LB에 연결하지 않음"이 인프라 리뷰 체크 항목.

## PAT-RES-03 우아한 성능 저하 — 부분 장애 격리 `[전역 — 정본 U1]`

**문제**: "인증 불가 = 전 기능 마비"(M1 Critical, §6.3)를 뒤집으면, 인증 **주변부**(특정 IdP, 메일, 금칙어 사전)의 장애가 인증 코어(로그인·세션 유지)로 전파되지 않아야 시스템 가용성 바닥이 유지된다.

**적용(설계)**: 장애 모드별 저하 계약(NFR-U1-AV-04를 설계 정본으로 승격):

| 장애 | 유지되는 기능 (저하 모드) | 상실되는 기능 | 고지 |
|---|---|---|---|
| 소셜 IdP 일부 장애 | **이메일 로그인·타 제공자 로그인·기발급 세션 전부 정상** | 해당 제공자 신규 로그인만 | "잠시 후 다시 시도" + 재시도 버튼(FLOW-1 E2) |
| 소셜 IdP 4종 전체 장애 | 이메일 로그인·기존 세션 정상 | 소셜 신규 로그인 | 동일 |
| 메일 발송 장애 | 가입 완결(계정 생성 유지)·기존 사용자 전 기능 | 신규 이메일 가입의 인증 완료만 지연 | 미인증 안내+재발송 UI(FLOW-2 E4) |
| C3 금칙어 사전 로딩 실패 | 로그인·세션·조회 전부 정상 | 닉네임·제목 **저장만** 보류(fail-closed — Pass 위조 금지, INV-B2·U1-P15) | 재시도 안내(FLOW-7 E1) |
| PostgreSQL 열화 | 액세스 토큰 검증 무상태(PAT-PERF-01)라 **기존 세션의 조회성 기능은 각 모듈 판단으로 지속 가능** | 로그인·토큰 갱신·가입·저장 (Critical 본질) | Multi-AZ 페일오버 대기 — 알람은 OPS-02 라우팅 |
| 유출 목록 온라인 조회 실패 | 가입 전 과정 (로컬 목록 검증으로 진행 — fail-open) | 보강 검증만 | 무고지(내부 계측만) |

설계 원칙: **보안 게이트는 fail-closed**(C3 사전, IdP id_token 검증 실패 — NFR-U1-SEC-15), **가용성 보조 검증은 fail-open**(유출 온라인 조회) — 방향을 항목별로 명시해 애매함을 남기지 않는다. 저하 모드 진입·이탈은 계측 이벤트(PAT-OBS-03)로 관측한다.

**U1 적용 지점**: M1 로그인 경로 분기, C3 어댑터, 메일 큐. 후속 유닛은 services.md §0.1 폴백 표(LLM→규칙 점수, 지도→추정치 등)를 동일 형식으로 소유 — 본 패턴이 그 형식 정본.

**검증 기준**: RESILIENCY-10(graceful degradation 문서화) — 본 표. 각 행은 fake 실패 주입 통합 테스트로 검증(IdP 1종 다운 시 이메일 로그인 성공, 사전 로드 실패 시 저장 보류+로그인 성공 등, D37). OPS-03 시나리오 SC-3·SC-4와 연동.

## PAT-PERF-01 토큰 검증 무상태 — 서명 검증·DB 비조회 `[전역]`

**문제**: 전 모듈 전 요청이 인증을 통과한다(D22). 요청마다 세션 DB 조회가 있으면 (a) DB가 전 기능의 동기 결합점이 되어 열화가 즉시 전면 장애로 전파되고, (b) 인증 오버헤드가 모든 API의 300ms 예산(D38)을 상시 잠식하며, (c) 수평 확장·롤링 재시작(GD-5)에 세션 지속성 문제가 생긴다.

**적용(설계)**:

- **액세스 토큰(1h, D36) 검증은 서명·만료·클레임(iss·aud)만으로 완결** — 요청 경로에 DB·캐시 조회 없음(NFR-U1-AV-01). JWT 서명 키는 JWK 셋 캐시(인메모리)로 검증(PAT-SEC-05의 kid 롤오버와 정합).
- **상태 접근은 갱신 시점으로 집중**: 세션 정본(RefreshSession)은 1시간에 1회의 갱신 호출에서만 조회 — DB 부하가 요청 수가 아닌 세션 수에 비례.
- **트레이드오프 명시**: 무효화 지연 최대 1시간은 D36이 수용한 결정. **즉시 무효화가 필요한 경우(계정 삭제 요청·비밀번호 재설정·제재)의 보완 수단(§8 이관분)을 다음과 같이 확정한다**: 전 체인 revoke(PAT-SEC-02)로 갱신을 즉시 차단하고, 상태 민감 연산(삭제 철회, 프로필 변경, 이후 유닛의 결제성·공유성 연산)은 핸들러 레벨에서 **계정 상태 재확인**을 명시적으로 수행한다(무상태 원칙의 국소 예외 — 예외 지점은 코드에 어노테이션/명명으로 식별 가능하게). 전면 토큰 블랙리스트는 초기 규모에서 미도입(도입 시 PD-9 캐시와 함께 재평가).
- **무상태의 배포 내성**: 롤링 재시작 중 어떤 인스턴스로 라우팅되어도 검증 동일(NFR-U1-AV-05, GD-5 전제) — 인스턴스 로컬 세션 상태 금지(NFR-U1-AV-02)와 한 쌍.

**U1 적용 지점**: JWT 인증 필터(PAT-SEC-01 3단), M1 토큰 발급기(클레임 설계), 부트스트랩 확인 API(p95 500ms — 무상태 검증+단일 질의, §1.3).

**검증 기준**: NFR-U1-AV-01 — 인증 필터 경로에 리포지토리 의존 부재(아키텍처 테스트), 부하 테스트에서 인증 오버헤드가 DB 연결 수와 무상관. NFR-U1-AV-05 — 롤링 재시작 중 기존 토큰 요청 성공(OPS-03 SC-5 리허설 항목). 상태 재확인 예외 지점 목록이 코드 리뷰 체크 항목.

## PAT-PERF-02 커넥션 풀 원칙 `[전역]`

**문제**: 커넥션 풀은 성능 장치이자 벌크헤드다. 무계획 풀은 두 방향으로 실패한다 — 과소(대기 급증, p95 붕괴)와 과대(DB 최대 연결 고갈 — 특히 롤링 배포 중 구·신 인스턴스 동시 존재 시 합산 초과).

**적용(설계)**:

- **풀 사이징 공식**: `인스턴스 수(롤링 중 최대치 포함) × 풀 최대 크기 < DB max_connections × 안전율(80%)` — Multi-AZ 2+ 인스턴스(GD-5)와 오토스케일 max(NFR-U1-SC-02)를 입력으로 Infrastructure Design에서 수치 확정. 초기 규모(피크 0.5 RPS, 헤드룸 10 RPS — §2)에서는 인스턴스당 소형 풀(10 내외)로 충분.
- **타임아웃 3종 명시**: 커넥션 획득 타임아웃(무한 대기 금지 — 획득 실패는 빠른 5xx+계측), 쿼리 타임아웃, 유휴/수명 타임아웃(페일오버 후 죽은 연결 정리 — Multi-AZ 전환 시 재연결 지연 최소화).
- **벌크헤드 분리**: 요청 처리 풀과 **백그라운드 잡(메일 발송·아웃박스 릴레이·정리 배치) 풀을 분리** — 잡 폭주가 로그인 경로의 커넥션을 고갈시키지 않게(RESILIENCY-10 bulkhead). argon2 해시(CPU·메모리 점유, PAT-SEC-04)는 DB 커넥션을 잡지 않은 상태에서 수행(해시 후 커넥션 획득 순서).
- **관측**: 풀 사용률·대기 시간·획득 실패를 메트릭 노출(PAT-OBS-03) — 사용률 80% 알람.

**U1 적용 지점**: `app` 모듈 DataSource 구성(스캐폴드 — HikariCP 상당, 수치는 외부화 설정), 잡 실행기 분리 구성.

**검증 기준**: RESILIENCY-10(풀·자원 상한 구성) — 획득 타임아웃 존재, 풀 분리 구성 확인. RESILIENCY-06과 연동 — deep 헬스체크의 DB 검사가 별도 짧은 타임아웃 사용. 풀 메트릭이 대시보드(PAT-OBS-03)에 존재.

## PAT-OBS-01 구조화 로깅 — 상관 ID·PII 마스킹 `[전역]`

**문제**: 인시던트 대응(OPS-02)의 속도는 "요청 하나의 전 구간을 이어 볼 수 있는가"에 달려 있고, 동시에 인증 서버 로그는 자격 증명·PII가 흘러들기 가장 쉬운 곳이다(로그 유출 = 2차 사고).

**적용(설계)**:

- **형식**: 전 서버 로그 stdout JSON(SLF4J+Logback+logstash-encoder, S-13) — 타임스탬프(ISO 8601)·레벨·모듈 태그(auth/profile/moderation/core)·**상관 ID** 필수 필드. 상관 ID는 필터 체인 1단(PAT-SEC-01)에서 요청당 생성·MDC 주입, 응답 헤더로 반환(클라이언트 오류 리포트와 연결), 비동기 잡·아웃박스 이벤트에 전파(이벤트 봉투에 correlation ID 포함 — LC-10).
- **PII·자격 증명 구조적 마스킹**: 금지 목록(NFR-U1-SEC-22 — 비밀번호·전 토큰류·소셜 코드/id_token·이메일 주소·생년월일·Authorization 헤더)은 **로깅 프레임워크 레벨 마스킹 컨버터**로 차단(개별 로그문 재량 금지 — 사람이 안 지키는 규칙은 규칙이 아니다). 계정 식별은 내부 accountId로만.
- **클라이언트 대칭**: 토큰을 크래시 리포트·로그에 미포함(NFR-U1-SEC-28).
- **중앙 집중**: U1은 stdout JSON까지 보장 — 수집 스택은 PD-4 대기. AWS 매핑(참조): CloudWatch Logs(수집·보존 90일+, SECURITY-14) + 메트릭 필터.

**U1 적용 지점**: 스캐폴드 로깅 구성(전 유닛 상속), 상관 ID 필터, 마스킹 컨버터+마스킹 회귀 테스트(금지 항목이 포함된 로그 이벤트를 생성해 출력에 미출현 검증).

**검증 기준**: SECURITY-03 — JSON 구조·필수 필드·마스킹 테스트 PR CI 통과, 중앙 라우팅은 Infrastructure Design 검증. RESILIENCY-05(logs 축) — 구조화·중앙 집중 요건의 애플리케이션 절반 충족.

## PAT-OBS-02 보안 이벤트 감사 로그 `[전역 — 이벤트 목록 U1]`

**문제**: "무슨 일이 있었는가"를 사후 재구성할 수 없는 인증 시스템은 침해 대응·법정 대응(N2·D18)·이상 탐지가 전부 불가능하다. 감사 로그는 일반 로그와 요건이 다르다 — 누락 불가·변조 불가·보존 기간 법정.

**적용(설계)**:

- **이벤트 전수(SECURITY-14 — NFR-U1-SEC-26 목록을 설계 정본으로)**: 행위자(accountId)·시각·결과·사유 코드·상관 ID를 구조화 기록. ① 계정 생명주기(가입 경로별·이메일 인증 완료·연령 차단 N1·삭제 요청/철회/파기 D18·미인증 정리 G22) ② 인증(로그인 성공/실패 사유·지연 발동/잠금·토큰 회전·**리프레시 재사용 감지**·로그아웃·비밀번호 변경/재설정) ③ 동의(약관 동의/재동의 N3·철회, 위치 약관·GPS 옵트인 N2, 마케팅 N8) ④ 권한(401/403 위반·삭제 유예 계정 접근 시도) ⑤ 프로필(닉네임 변경·금칙어 차단 C3).
- **기록 경로**: 감사 이벤트는 도메인 이벤트와 동일하게 **아웃박스 경유**(LC-10) — 업무 트랜잭션과 원자적으로 적재되어 "상태는 바뀌었는데 감사 기록이 없는" 창을 제거(SECURITY-13 무결성). 파이프라인(LC-8)이 저장·알림 라우팅.
- **알림 임계(NFR-U1-SEC-27)**: (a) 인증 실패 급증 (b) 리프레시 재사용 감지 — **건당 즉시** (c) 403 급증 (d) 잠금 급증 (e) 메일 발송 실패율. 알림 라우팅은 OPS-02 프로세스로 접속.
- **보존**: 감사 로그 90일+(SECURITY-14), 동의 증적·위치 법정 로그는 별도 저장소(LC-7 — append-only·6개월+/계정 생명주기, 감사 로그와 구분).

**U1 적용 지점**: M1·M2·C3 각 유스케이스의 이벤트 발행 지점(FLOW-1~7 전 분기), 감사 파이프라인 계약(common/core).

**검증 기준**: SECURITY-14 — 이벤트 전수 표 대비 발행 커버리지 테스트(각 FLOW 분기에서 기대 이벤트 발행 확인), 재사용 감지 즉시 알림 경로 통합 테스트, 보존 구성은 Infrastructure Design(PD-4) 검증. SECURITY-03 — 감사 레코드에도 PII 마스킹 규칙 동일 적용(이메일 대신 accountId).

# 5. 운영 프로세스 — 전역 정본 (U1에서 1회 수립, 전 유닛·Operations 재사용)

## OPS-01 경량 변경 관리 프로세스 (AI-DLC 제안 — 사용자 채택 GD-2) `[전역 정본]`

**문제**: 조직에 공식 변경 관리가 없다(GD-2=A — B안 '기존 조직 프로세스' 아님). 프로세스 부재는 "누가 언제 무엇을 왜 바꿨고 어떻게 되돌리는가"를 답할 수 없게 하며, 이는 변경 유발 장애(장애의 최다 원인)를 통제 불능으로 만든다. RESILIENCY-03은 프로세스의 명시적 식별을 요구한다.

**적용(제안 프로세스 — 채택으로 확정)**: GitHub(GD-3) 위에서 도구 추가 없이 성립하는 3요소:

1. **PR 승인 게이트 (승인 없는 프로덕션 변경 없음)**
   - `main` 브랜치 보호: 직접 push 금지, PR 필수, **승인 1인 이상**, CI 게이트 전부 녹색(테스트·PBT·아키텍처 테스트·시크릿 스캔·의존성 스캔 — SECURITY-10·NFR-U1-MT-01~03) 필수.
   - 프로덕션 배포는 `main` 머지 커밋 + 버전 태그에서만 GitHub Actions(GD-3)로 실행 — 수동 배포 경로 없음(RESILIENCY-04 자동화).
   - 고위험 변경 라벨(`risk:high` — 인증 필터·마이그레이션·시크릿·인프라)은 승인 2인 또는 셀프머지 금지 강화 규칙.
2. **변경 기록 (change record)**
   - PR 템플릿이 곧 변경 기록: 변경 요약 / 영향 범위(모듈·엔드포인트) / 리스크와 검증 방법 / **롤백 노트(3항 참조)** / 관련 추적 ID(SECURITY·RESILIENCY·D·G). 머지된 PR + 배포 태그 + Actions 실행 이력이 변경 이력 정본(별도 도구 불요).
   - DB 마이그레이션 포함 PR은 **forward-only·호환 규칙 준수 확인**(GD-4)과 append-only 권한 DDL 리뷰(NFR-U1-SEC-24)를 체크 항목으로 강제.
3. **롤백 노트 (rollback note — 모든 배포 PR 필수 기입)**
   - 정본 메커니즘(GD-4): **직전 버전 아티팩트 재배포(버전 고정 롤백)** — Actions 워크플로에 "태그 지정 재배포" 잡을 상비.
   - **DB는 롤백하지 않는다**: 마이그레이션은 forward-only. 따라서 모든 스키마 변경은 **N-1 호환 규칙**을 지킨다 — expand(추가·병행) → 코드 전환 → contract(제거)의 2단계 배포. 컬럼 삭제·NOT NULL 강화·타입 변경은 반드시 별도 후행 배포로 분리. 이 규칙 위반은 PR 차단 소견.
   - 롤백 노트에는 "이 배포를 되돌릴 때 추가로 필요한 조치"(설정 되돌림, 캐시 무효화, 없음 등)를 명기 — "없음"도 명시적 기입.

**U1 적용 지점**: U1 CI 파이프라인 산출물(브랜치 보호·PR 템플릿·배포/롤백 워크플로 — unit-of-work.md U1 포함 범위)이 본 프로세스의 구현체다. 이후 전 유닛의 전 변경이 동일 프로세스를 따른다.

**검증 기준**: RESILIENCY-03 — 프로세스가 명시적으로 제안·채택됨(GD-2), 변경 기록 메커니즘 식별(PR+태그+Actions 이력), 승인 게이트 존재. RESILIENCY-04 — 롤백 메커니즘이 사용자 선택으로 문서화(GD-4), forward-only 마이그레이션의 호환 규칙 문서화(DB-aware 롤백 부재의 보상 설계). Infrastructure Design·Code Generation에서 브랜치 보호·워크플로 실물 검증.

## OPS-02 경량 인시던트 대응 프로세스 (AI-DLC 제안 — 사용자 채택 GD-6) `[전역 정본]`

**문제**: 조직에 공식 온콜/IR 프로세스가 없다(GD-6=A). M1은 Critical(불가용 = 전 기능 마비, §6.3)인데 알람이 울려도 "누가 받고, 무엇부터 보고, 끝나면 무엇을 남기는가"가 없으면 알람 정의(PAT-OBS-03)는 소음이다. RESILIENCY-15는 알림이 식별된 대응 프로세스로 라우팅될 것을 요구한다.

**적용(제안 프로세스 — 채택으로 확정)**: 소규모 팀 전제의 3단 구성.

1. **알람 라우팅 (심각도 3단)**

| 심각도 | 정의 (U1 예시) | 라우팅 | 초동 목표 |
|---|---|---|---|
| P1 | Critical 경로 불능 — 로그인/갱신 실패율 급증, DB 불능, 전 AZ 인스턴스 미가용 | 즉시 호출 채널(전화/푸시 — 채널 제품은 PD-4와 함께 확정, AWS 참조: CloudWatch Alarm→SNS) | 15분 내 인지·대응 개시 |
| P2 | 저하 운전 — IdP 서킷 open 지속, 메일 실패율 초과, 재사용 감지 급증, 단일 AZ 운전 | 팀 메신저 채널 | 업무 시간 내 대응 |
| P3 | 주의 — 쿼터/풀 80%, 배치 실패 1회, p95 목표 초과 | 메신저(비긴급) + 주간 리뷰 | 다음 계획 주기 |

   - 보안 이벤트 특칙: **리프레시 재사용 감지는 건당 P2 이상**(다계정 동시면 P1 승격), 인증 실패 급증+잠금 급증 동시 발생은 공격 의심으로 P1(PAT-OBS-02 임계와 접속).
2. **대응 절차 (인시던트 런북 골격 — 상세 런북은 Infrastructure Design·Operations에서 채움)**
   - 선언: P1/P2 판단 → 대응자 지정(소규모 팀 — 발견자가 초기 대응자) → 인시던트 스레드 개설(타임라인 기록 시작).
   - 진단 순서(U1): deep 헬스(`/health/deep`) → 대시보드 골든 시그널 → 상관 ID로 로그 추적(PAT-OBS-01) → 최근 배포 대조(OPS-01 변경 기록 — "최근에 무엇이 바뀌었나"가 첫 질문).
   - 완화 우선: 원인 규명보다 복구 우선 — **직전 버전 롤백(OPS-01 3항)**, 서킷/설정 조정(외부화 설정 NFR-U1-MT-04 — 재빌드 없이), Multi-AZ 페일오버 확인. 보안 인시던트(탈취 의심)는 해당 계정 전 체인 revoke(PAT-SEC-02 매트릭스)가 1차 조치.
   - 종결: 사용자 영향 종료 확인(메트릭 정상 복귀) + 타임라인 마감.
3. **사후 회고 (COE — 비난 없는 포스트모템, P1·P2 필수)** — 템플릿:

```markdown
# 인시던트 회고 — [YYYY-MM-DD] [제목]
- 심각도/영향: (P1/P2, 영향 사용자·기간·기능)
- 타임라인: (감지 → 선언 → 완화 → 종결, 시각 명시)
- 근본 원인: (5 Whys — 사람 아닌 시스템·프로세스 결함으로)
- 잘 된 것 / 운이 좋았던 것:
- 시정 조치(Corrective Actions): (각 항목 = GitHub Issue로 발행·담당·기한 — 추적 메커니즘)
- 재발 감지 개선: (이 인시던트를 더 빨리 알 수 있었을 알람·메트릭)
```

   - 시정 조치는 GitHub Issues로 추적(도구 추가 없음) — 미종결 시정 조치는 주간 리뷰 상시 항목.

**U1 적용 지점**: 알람 임계·심각도 매핑(PAT-OBS-03 표의 라우팅 열), 회고 템플릿 파일(리포지토리 `docs/incident/` 상당 — Code Generation 산출), 채널 실물 연결은 Infrastructure Design(PD-4).

**검증 기준**: RESILIENCY-15 — 프로세스 제안·채택 명시(GD-6), COE 메커니즘 식별(템플릿+Issues), RESILIENCY-05 알림이 본 라우팅 표로 접속, 시정 조치 추적 메커니즘 식별. 실 채널 구성은 Infrastructure Design 검증.

## OPS-03 복원력 테스트 시나리오 (Operations 이연 — 사용자 결정 GD-7) `[전역 정본]`

**문제**: 검증되지 않은 페일오버는 존재하지 않는 페일오버다. 단, MVP 규모(DAU 1천)에서 카오스 실험 상시 운영은 과설계다 — 사용자 결정(GD-7=A)에 따라 **실행은 Operations 단계로 이연하고, 본 문서는 시나리오 정의까지만** 확보한다(RESILIENCY-14의 Note와 정합: 설계 시점에 시나리오를 캡처해 Operations의 시작점을 제공).

**적용(시나리오 정의 — 실행 이연 명시)**: DR 전략 = Backup & Restore, RTO/RPO = 시간 단위(CQ4=A, RESILIENCY-02·11) 기준.

| ID | 시나리오 | 절차 골격 | 합격 기준 | 실행 시점 |
|---|---|---|---|---|
| SC-1 | **Multi-AZ 컴퓨트 페일오버** | 1개 AZ의 인스턴스 강제 종료 → LB(shallow 헬스) 축출·재분배 관측 | 로그인·갱신·조회 무중단(기존 토큰 유효 — PAT-PERF-01), 오류율 순간 상승 후 1분 내 정상, 알람 P2 발화 확인 | Operations (이연) |
| SC-2 | **관리형 PostgreSQL Multi-AZ 페일오버** | DB 강제 페일오버 트리거 → 재연결(PAT-PERF-02 수명 타임아웃) 관측 | 전환 중 로그인·저장 실패는 명시 오류(침묵 실패 금지), 전환 후 자동 회복, 데이터 손실 0(동의 증적·법정 로그 무결) | Operations (이연) |
| SC-3 | **DB 백업 복원 리허설 (DR 본체)** | 최신 자동 백업/PITR로 신규 인스턴스 복원 → 계정·동의 증적·법정 로그·세션 정합 검증 → IaC 재배포 접속 | **RTO 시간 단위 내 복원 완료·RPO 손실 창 확인**(RESILIENCY-02 목표 실측), append-only 권한 모델 복원 확인 | Operations (이연 — 최초 1회는 출시 전 권장) |

| SC-5 | **롤백 리허설** | 무해 변경 배포 → OPS-01 3항 절차로 직전 버전 태그 재배포 → forward-only 스키마 위에서 N-1 코드 정상 동작 확인 | 롤백 소요 시간 측정·기존 세션 유지(NFR-U1-AV-05), 호환 규칙(expand/contract) 실증 | Operations (이연 — 첫 프로덕션 배포 직후 1회 권장) |
| SC-6 | **롤링 배포 무중단 검증** | 배포 중 로그인·갱신·부트스트랩 연속 호출 부하 | 오류 0(재시도 포함), 구·신 인스턴스 혼재 구간 정상(무상태 검증) | Operations (배포 파이프라인 정착 시) |

- **결과 추적 메커니즘**: 각 실행 결과는 OPS-02 회고 템플릿의 축약형(시나리오 ID·실측 RTO/RPO·발견 사항·시정 조치 Issue)으로 리포지토리에 기록 — RESILIENCY-14 "test results tracking" 충족.
- **이연 경계 명시**: PR CI의 fake 실패 주입 테스트(D37 — PAT-RES-01·03 검증)는 **이연 대상이 아니며** U1 Code Generation부터 상시 실행한다. 이연되는 것은 실 인프라 대상 실험(SC-1~6)뿐이다.

**U1 적용 지점**: 본 시나리오 표가 산출물 그 자체(문서 확보 — GD-7 요구 충족). Operations 단계 진입 시 본 표를 시작점으로 일정 수립.

**검증 기준**: RESILIENCY-14 — 접근 방식이 사용자 답변으로 식별(GD-7=C상당의 'Operations 이연'), DR 시나리오가 선택된 DR 전략(Backup & Restore, RESILIENCY-11)에 대해 문서화(SC-3), 결과 추적 메커니즘 식별. RESILIENCY-13(런북) — SC-2·3의 절차 골격이 페일오버/복원 런북의 씨앗이며 상세화는 Infrastructure Design·Operations.
