## 1. 성능 (D38, §6.1)

### 1.1 원칙 — "일반 화면 전환 300ms"의 U1 해석

§6.1의 성능 목표 중 U1에 직접 해당하는 항목은 **일반 화면 전환 300ms**뿐이다(AI 생성·Plan-B는 U5·U6 소관). U1의 온보딩 여정(가입→약관→닉네임→취향 7단계)은 화면 전환이 연속되는 흐름이므로, **API 지연이 화면 전환 예산을 잠식하지 않는 것**이 U1 성능 요구의 본질이다. 이를 위해 U1 API를 두 부류로 나눈다:

- **전환 차단(blocking) API** — 응답이 와야 다음 화면으로 넘어가는 호출(닉네임 검증, 취향 저장, 동의 기록). 서버 p95가 300ms 안에 들어와야 체감 전환이 300ms대에 머문다.
- **전환 비차단 API** — 낙관적 UI·백그라운드 처리로 전환과 분리 가능한 호출(인증 메일 발송, 백그라운드 세션 재검증).

### 1.2 U1 API 응답 목표 (제안 — 서버 처리 시간 기준, 네트워크 RTT 별도)

| API (M1/M2 인터페이스 그룹) | p50 | p95 | 상한(타임아웃) | 비고 · 추적 |
|---|---|---|---|---|
| 부트스트랩 확인(세션 유효성·재동의·최소 버전) | 100ms | 500ms | 클라이언트 3초(G5) | §1.3 참조. 단일 질의·무상태 검증으로 구성 |
| 토큰 갱신(리프레시 회전) | 100ms | 300ms | 3초 | 회전은 UPDATE 1건+발급 — 전환 차단 최소화(D36) |
| 이메일 로그인 | 400ms | 1,000ms | 5초 | 적응형 해시 검증 비용(의도적 ~200ms 내외, SECURITY-12)을 예산에 포함 — 300ms 목표에서 **의도적 예외** |
| 이메일 가입(해시 생성+유출 목록 검증) | 500ms | 1,500ms | 5초 | 유출 검증 외부 조회는 1초 타임박스(§4.1.2), 메일 발송은 비동기 분리 |
| 소셜 로그인 콜백 처리(IdP 토큰 교환·검증 포함) | 800ms | 3,000ms | 5초(IdP 어댑터 자체 타임아웃 3초) | 외부 IdP 의존 — RESILIENCY-10 타임아웃·서킷 필수 |
| 취향 저장(축별 부분 저장, M2) | 50ms | 200ms | 3초 | 온보딩 7단계 연속 전환의 핵심 — **낙관적 UI 병행 권장** |
| 닉네임 검증(금칙어 C3+중복)·자동 생성(재추첨 포함) | 100ms | 300ms | 3초 | 금칙어 사전은 인메모리 매칭(§6), 중복은 유니크 인덱스 조회 |
| 동의 기록(약관 3종+선택 동의, append-only) | 100ms | 300ms | 3초 | 증적 INSERT — 트랜잭션 내 완결(N2·N3) |
| 인증 메일 발송 요청 | 100ms | 300ms | 3초 | **접수 확인만 동기 응답**, 실제 발송은 비동기 잡. 메일 도달 SLO는 별도: 5분 내 도달 95%(발송 실패 시 재발송 UI — ADR-0011) |
| 로그아웃(세션 무효화) | 50ms | 200ms | 3초 | revoke UPDATE 1건 |

수치의 성격: D38(§6.1)이 확정한 것은 "일반 화면 전환 300ms"이며, 위 표의 API별 세부 수치는 그 지원을 위한 **U1 제안값**이다. NFR Design에서 측정 방법(APM p95 대시보드, §6.7)과 함께 확정한다.

### 1.3 스플래시 세션 검증 3초 타임아웃(G5)과의 관계

- G5 확정: 스플래시에서 세션 검증(부트스트랩 확인) **3초 타임아웃** — 로컬 토큰이 미만료면 **홈 진입 + 백그라운드 재검증**, 만료면 로그인 분기.
- 따라서 3초는 **서버 성능 목표가 아니라 클라이언트의 최종 대기 한계(fail-policy 경계)**다. 서버 목표는 p95 500ms(위 표)로 두어, 3초 타임아웃이 발동하는 경우가 비정상 상황(콜드 스타트·네트워크 열화·부분 장애)에 한정되게 한다.
- U1 서버 요구사항: 부트스트랩 확인 API는 **멱등**해야 하며(components.md M1 실패·폴백 책임), 백그라운드 재검증에서 세션 무효가 판명되면 클라이언트가 로그인으로 유도할 수 있는 명확한 응답 계약(401 + 사유 코드)을 가진다.
- U1 클라이언트 요구사항: 타임아웃 발동 시 침묵 실패 금지(ADR-0011) — 재검증 실패는 계측 이벤트로 보고한다(§6.7). 실제 스플래시 분기 UI는 U2 소관이나, **API 계약과 타임아웃 정책의 서버측 절반은 U1이 소유**한다.

### 1.4 클라이언트 성능 요구

- 토큰 읽기(expo-secure-store)는 앱 기동 크리티컬 패스에 있다 — 저장 항목을 최소화(액세스·리프레시 토큰 2건)하고 기동 시 1회만 읽어 메모리 보관한다(SECURITY-12 저장 요건은 §4.5).
- 취향 7단계는 단계별 저장 + 낙관적 UI로, 서버 왕복이 화면 전환(300ms)을 차단하지 않게 한다. 저장 실패 시 해당 단계로 복귀 가능한 오류 처리(침묵 실패 금지).
- 적응형 해시는 **서버 전용** — 클라이언트에서 비밀번호 해시·키 스트레칭을 수행하지 않는다(전송은 TLS 보호, SECURITY-01).

## 2. 규모 (§6.8 G142 — DAU 1천 기준 인증 처리량 산정)

### 2.1 산정 전제

§6.8 확정: 초기 출시 **DAU 1,000**(국내 한정 MVP 검증 규모). 인증 계열 트래픽 산정:

| 트래픽원 | 산정 | 일간 | 피크 시간 집중(일량의 20%가 1시간) |
|---|---|---|---|
| 부트스트랩 확인(앱 실행마다 1회) | DAU 1,000 × 일 평균 실행 5회 | 5,000건 | ~0.28 RPS |
| 토큰 갱신(액세스 1h 만료, D36) | 활성 사용 1,000명 × 일 평균 2회 | 2,000건 | ~0.11 RPS |
| 로그인(신규 세션 — 리프레시 90일이므로 저빈도) | DAU의 ~5% | 50건 | ~0.003 RPS |
| 가입(온보딩 전체 여정 API 약 15콜/인) | 신규 일 50명 가정 × 15 | 750건 | ~0.04 RPS |
| 취향·닉네임·동의 등 프로필 계열 | 가입 포함 + 기존 사용자 수정 | ~1,000건 | ~0.06 RPS |

**정상 피크 합산 ≈ 0.5 RPS.** 절대 규모는 작으나, 아래 스파이크 요구가 설계를 지배한다.

### 2.2 처리량 요구사항

- **NFR-U1-SC-01 (설계 처리량)**: 인증 계열 API는 정상 피크의 **최소 20배 헤드룸 = 10 RPS**를 단일 인스턴스 기준으로 소화한다(출시 직후 마케팅·스토어 피처링 가입 스파이크 대비). 적응형 해시의 CPU 비용(§4.1.1)을 포함해 산정한다 — 해시 1건당 코어 점유 ~100–200ms이므로 로그인/가입 동시 처리 병렬도가 실질 상한이며, 이 계산이 인스턴스 사이징의 입력이다(Infrastructure Design 전달).
- **NFR-U1-SC-02 (오토스케일 연동)**: 오토스케일 min/max 한도와 비용 상한 알람(RESILIENCY-09, §6.8)은 Infrastructure Design에서 확정한다 — U1은 무상태 수평 확장이 가능한 구조(§3.2)를 보장하는 책임까지.
- **NFR-U1-SC-03 (데이터 볼륨)**: 첫해 가정 계정 ~2만, 동의 증적(append-only) 계정당 평균 8행, RefreshSession 계정×기기(다기기 허용, D36) — 90일 만료분 정리 배치 필요. 위치정보 법정 로그(N2)는 U6 기능 활성 후 증가 — 테이블 설계만 U1(파티셔닝 여지 확보). 전 테이블이 단일 PostgreSQL로 충분한 규모이며 샤딩 불요.
- **NFR-U1-SC-04 (외부 쿼터)**: 소셜 IdP 4종·메일 발송의 rate-limit·쿼터를 문서화하고 사용량 80% 알람을 설정한다(RESILIENCY-09). 특히 메일 재발송 상한(분당 1회·일 5회, G22)은 서버 강제 — 외부 쿼터 보호를 겸한다.

## 3. 가용성·복원력 (§6.2·§6.3, RESILIENCY-01·08·10)

### 3.1 중요도 — M1은 Critical

§6.3 워크로드 분류 확정: **Auth·세션(M1) = Critical — 불가용 시 전 기능 접근 불가.** 로그인 필수 원칙(D22) 때문에 인증 실패는 곧 앱 전체 마비다. 모든 Critical 경로가 PostgreSQL·Auth에 의존하므로(§6.3), U1의 가용성 설계가 시스템 가용성의 바닥을 결정한다.

| 항목 | 요구 | 추적 |
|---|---|---|
| 토폴로지 | 단일 리전 + **Multi-AZ 전제** — 컴퓨트 2개 이상 AZ 분산, 관리형 PostgreSQL Multi-AZ, LB 다중 AZ | RESILIENCY-08, §6.2 |
| RTO/RPO | 시간 단위 / DR = Backup & Restore(IaC 재배포+백업 복원) — 계정·동의 증적·법정 로그는 백업 최우선 대상 | CQ4=A, RESILIENCY-02·11·12 |
| M2·C3 | M2(취향)·C3(금칙어)는 M1과 동일 배포 단위(D04)이므로 별도 분류 불요 — 단 C3 사전 로딩 실패는 fail-closed(components.md C3)로 가입·제목 저장만 보류, 로그인은 무영향이어야 함 | §6.3 |

### 3.2 무상태성 요구 — "인증 불가 = 전 기능 마비"의 구조적 차단

**NFR-U1-AV-01 (액세스 토큰 검증 무상태)**: 액세스 토큰(1h, D36) 검증은 **서명·만료·클레임 검증만으로 완결**되어야 하며, 요청마다 DB·캐시 조회를 요구하지 않는다. 근거: (a) 타 모듈 전 요청의 인증 경로가 M1 저장소 상태에 동기 결합되면 DB 열화가 즉시 전 기능 마비로 전파된다, (b) 인스턴스 수평 확장·롤링 재시작 시 세션 지속성 문제가 없어야 한다(Multi-AZ·오토스케일 전제). 트레이드오프로 발생하는 "무효화 지연 최대 1시간"은 액세스 토큰 수명(1h)이 이미 수용한 설계 결정(D36)이다. 즉시 무효화가 법적으로 필요한 경우(계정 삭제·제재)는 보완 수단을 NFR Design에서 결정한다(§8).

**NFR-U1-AV-02 (인스턴스 로컬 세션 상태 금지)**: 리프레시 세션·브루트포스 카운터·rate-limit 상태를 **인스턴스 메모리에 정본으로 두지 않는다**. 정본은 PostgreSQL(RefreshSession 회전 체인, D36)이며, 어느 인스턴스로 라우팅되어도 동일하게 동작해야 한다(무상태 수평 확장 — NFR-U1-SC-02의 전제).

**NFR-U1-AV-03 (세션 캐시는 최적화일 뿐)**: 성능 목적의 세션·프로필 캐시를 도입하는 경우에도 **캐시 소실이 인증 실패로 이어져서는 안 된다** — 캐시 미스는 항상 DB 폴백으로 충족한다(캐시는 가용성 요소가 아니라 지연 최적화). 초기 규모(§2)에서는 별도 캐시 계층 없이 DB 직행으로 충분하며, 캐시 인프라 추가 여부는 Infrastructure Design 대기(§8).

**NFR-U1-AV-04 (부분 장애 격리 — ADR-0011·RESILIENCY-10)**:

| 의존 장애 | 요구 동작 |
|---|---|
| 소셜 IdP(4종 중 일부) 장애 | 해당 제공자 **신규 로그인만** 실패("잠시 후 재시도") — 기발급 토큰 세션·이메일 로그인·타 제공자는 무영향. 어댑터별 타임아웃 3초+서킷 브레이커(Port/Adapter, components.md §2-2) |
| 메일 발송 실패 | 미인증 상태 유지 + 재발송 UI 제공, 가입 트랜잭션은 성공 유지(발송 비동기 분리, §1.2) — 침묵 실패 금지: 실패율 계측 |
| PostgreSQL 열화 | 액세스 토큰 검증은 무상태라 기존 세션의 **조회성 기능은 각 모듈 판단으로 지속 가능** — 단 로그인·갱신·가입은 불가(Critical 본질). Multi-AZ 페일오버(관리형)로 복구 |
| C3 사전 로딩 실패 | fail-closed(닉네임·제목 저장 보류+재시도 안내) — 인증·로그인 경로 무영향 |

**NFR-U1-AV-05 (무중단 배포 내성)**: 토큰 검증 무상태(AV-01) + 세션 정본 DB(AV-02) 덕분에 롤링 재시작 중에도 로그인 상태가 유지되어야 한다. 배포 스타일 자체는 NFR Design 질문 대기(§8).

**NFR-U1-AV-06 (헬스체크)**: shallow(프로세스 생존)·deep(DB 연결 포함) 헬스체크 엔드포인트는 U1 스캐폴드 산출물이다(§6.7 — 설계 단계 요건). deep은 LB 판단이 아닌 관측 용도로 분리해 DB 순단이 인스턴스 대량 축출로 증폭되지 않게 한다.

## 4. 보안 (Security Baseline 전체 강제 — CQ1=A, §6.4)

U1은 SECURITY-03·04·05·08·10·11·12·14·15의 **1차 구현 지점**이다(unit-of-work.md U1 DoD). SECURITY-01·02·06·07·09·13은 인프라·코드 생성 단계 검증 항목으로 §7 요약에 처리한다.

### 4.1 SECURITY-12 — 인증·자격 증명 관리 (U1 구체화)

#### 4.1.1 비밀번호 정책·적응형 해시

- **NFR-U1-SEC-01**: 비밀번호 8자 이상 + 영문/숫자 조합(G22) + 유출 목록 검증(4.1.2). 최대 길이 128자(비정상 입력 방어, SECURITY-05).
- **NFR-U1-SEC-02 (적응형 해시 요건)**: 메모리 하드 적응형 알고리즘을 사용한다. 요건: (a) 알고리즘·파라미터가 해시 문자열에 자기 기술(self-describing)되어 **파라미터 상향 시 로그인 성공 시점에 투명 재해시(업그레이드 경로)** 가능, (b) 파라미터는 운영 설정으로 조정 가능, (c) 검증 1건당 목표 지연 100–300ms(브루트포스 비용과 로그인 p95 1,000ms 예산의 균형 — §1.2). **알고리즘 권고는 argon2id** — bcrypt 비교·파라미터 확정은 [tech-stack-decisions.md](./tech-stack-decisions.md) §2.5.
- **NFR-U1-SEC-03**: 소셜 계정(sub 식별)은 비밀번호 부재 — passwordHash nullable, 비밀번호 로그인 경로 자체가 차단되어야 한다(components.md M1 Account).

#### 4.1.2 유출 비밀번호 검증 방식

- **NFR-U1-SEC-04**: 2단 구성 — (1) **로컬 상위 유출 목록**(오프라인 사전, 상위 10만 수준)을 필수 1차 검증으로 두고, (2) **k-익명성 방식 온라인 조회**(해시 프리픽스 5자만 외부 전송 — 평문·전체 해시 미전송)를 보강 검증으로 둔다. 온라인 조회는 타임아웃 1초, 실패 시 **로컬 목록 통과만으로 진행 허용(가용성 우선 fail-open)** 하되 실패를 계측한다(침묵 실패 금지 — 외부 서비스가 가입 가용성을 좌우하지 않게 함, RESILIENCY-10). 온라인 조회 대상 서비스 확정은 Infrastructure Design 대기(§8 — 외부 아웃바운드 정책과 함께).

#### 4.1.3 세션 관리·무효화

- **NFR-U1-SEC-05 (세션 무효화 매트릭스)**:

| 이벤트 | 무효화 범위 |
|---|---|
| 로그아웃 | 해당 기기 RefreshSession revoke (다기기 유지 — D36) |
| 비밀번호 변경·재설정 완료 | **전 기기 RefreshSession revoke** (탈취 대응) |
| 리프레시 토큰 **재사용 감지**(회전 체인에서 이미 회전된 토큰 제시) | 해당 **회전 체인 전체 revoke** + 보안 이벤트 로그·알림(§4.4) — 탈취 표준 대응 |
| 계정 삭제 요청(ACTIVE→DELETION_PENDING) | 전 세션 revoke (재로그인=철회 경로만 허용, D18) |
| 유예 만료 파기 | 전 인증 수단·세션 소멸 |

- **NFR-U1-SEC-06**: 리프레시 토큰은 서버에 **해시로만 저장**(tokenHash — 원문 저장 금지), 회전 시 이전 토큰 즉시 무효(D36 회전). 액세스 토큰 즉시 무효화 한계는 NFR-U1-AV-01의 트레이드오프 — 수명 1h로 수용. 모바일 API이므로 세션 쿠키 미사용 — SECURITY-12의 쿠키 속성 요건은 N/A(근거: 토큰은 헤더 전송, 저장은 §4.5), 서버측 만료·무효화 요건은 위 매트릭스로 충족.

#### 4.1.4 브루트포스 방어 — 점진 지연 수치 (제안, remote config로 운영 조정)

- **NFR-U1-SEC-07 (계정 단위)**: 연속 실패 5회까지 지연 없음 → 6회부터 지수 지연 **2·4·8·16·32초**(6~10회) → **10회 초과 시 15분 일시 잠금** + 인증 실패 급증 보안 이벤트(§4.4). 성공 시 카운터 리셋. 잠금 중에도 응답 메시지는 일반화("로그인할 수 없습니다") — 계정 존재 여부 비노출.
- **NFR-U1-SEC-08 (IP/기기 단위)**: 동일 출처 로그인 시도 **분당 10회·시간당 100회** 상한(SECURITY-11 rate-limit과 결합) — 계정 열거(enumeration)·스프레이 공격 방어. 가입·재설정·인증 메일 재발송(분당 1회·일 5회, G22)·닉네임 중복 확인도 출처 단위 상한 적용.
- **NFR-U1-SEC-09**: 카운터 상태는 인스턴스 로컬 금지(NFR-U1-AV-02) — 초기 규모에서는 DB 기반으로 충분. 비밀번호 재설정 링크·이메일 인증 링크는 1회용·24h 유효(G22)·사용 즉시 소진.
- CAPTCHA는 1차 미도입(위 2단 방어로 시작) — 도입 트리거(자동화 공격 관측)를 운영 지표로 정의.

#### 4.1.5 MFA·시크릿

- **NFR-U1-SEC-10 (MFA)**: SECURITY-12의 MFA 요건은 **어드민 계정 대상**이며, 어드민 도구는 후속 단계(D35)다. U1에서는 (a) 일반 사용자 MFA 미제공을 명시하고, (b) 계정 모델에 인증 수단 확장 여지(SocialIdentity와 동열의 확장 가능한 자격 증명 구조)를 두어 후속 어드민 MFA를 차단하지 않는다. → 컴플라이언스 판정: **후속 단계 적용 예약**(1차 어드민 부재로 현 시점 N/A, §7).
- **NFR-U1-SEC-11 (하드코딩 금지)**: 소셜 클라이언트 시크릿·Apple 서명 키(p8)·JWT 서명 키·메일 API 키·DB 자격 증명은 소스·설정 파일에 미포함 — 시크릿 매니저 주입. **시크릿 매니저 제품 선정은 클라우드 벤더 결정 대기(§8)**, U1 코드는 환경 주입 인터페이스(외부화된 설정)로 벤더 중립 구성.

### 4.2 SECURITY-05 — 입력 검증 (U1 구체화)

- **NFR-U1-SEC-12**: 전 API DTO에 선언적 스키마 검증(Jakarta Validation — tech-stack §2.6). 타입·형식(이메일 RFC 5321 254자 상한, 생년월일 형식·범위, provider 열거형)·**전 문자열 명시적 최대 길이**(닉네임 20자 G23, 비밀번호 128자, 여행 취향 열거형) 강제.
- **NFR-U1-SEC-13**: 요청 본문 크기 상한 — U1 API는 전부 소형 JSON이므로 **64KB** 전역 상한(프레임워크 레벨) 제안. 파일 업로드 없음.
- **NFR-U1-SEC-14**: 전 DB 접근 파라미터라이즈드(JPA/JDBC 바인딩 — 문자열 결합 금지), 사용자 문자열(닉네임)은 저장 전 C3 금칙어 + 출력 시 이스케이프(XSS — 클라이언트 RN 렌더링은 기본 이스케이프이나 서버는 신뢰하지 않음).
- **NFR-U1-SEC-15**: 소셜 IdP 콜백 입력(코드·id_token)도 외부 입력이다 — id_token은 서명·iss·aud·exp·nonce 검증(어댑터 책임), 검증 실패는 구조화 오류로 fail-closed(SECURITY-15).

### 4.3 SECURITY-08 — 인증 미들웨어 deny-by-default (U1 구체화)

- **NFR-U1-SEC-16 (deny-by-default)**: Spring Security 필터 체인의 기본 규칙을 `anyRequest → 인증 필수`로 두고, **공개 엔드포인트는 명시적 화이트리스트만** 허용한다. U1의 공개 엔드포인트 전수(이외 추가는 설계 리뷰 필수):
  - 가입(이메일·소셜)·로그인·토큰 갱신 — 갱신은 리프레시 토큰 자체가 자격 증명
  - 이메일 인증 링크 확인·재발송, 비밀번호 재설정 요청·확인
  - 부트스트랩 확인 중 **비세션 부분**(최소 지원 버전) — 세션 유효성 부분은 토큰 제시 시 검증
  - shallow 헬스체크
- **NFR-U1-SEC-17 (토큰 검증)**: 서명·만료·발급자(iss)·대상(aud) 검증을 **매 요청 서버측 수행**(무상태, NFR-U1-AV-01). 알고리즘 고정(none·대칭 다운그레이드 거부).
- **NFR-U1-SEC-18 (객체 수준 권한)**: 프로필·취향·동의·세션 목록 등 리소스 접근은 **토큰 주체(accountId) 기준으로만 조회** — 경로/본문의 타 사용자 ID를 신뢰하지 않는다(IDOR 원천 차단, components.md §2-5). 삭제 유예·재동의 상태 등 계정 상태별 접근 축소(DELETION_PENDING은 철회·열람 외 거부)는 미들웨어 레벨에서 일괄 강제.
- **NFR-U1-SEC-19 (CORS)**: 모바일 앱 전용 API — 브라우저 오리진 불요이므로 CORS는 **기본 전면 거부**(와일드카드 금지). 후속 어드민 웹(D35) 도입 시 명시 오리진만 추가.
- **NFR-U1-SEC-20**: 아키텍처 테스트로 "인증 어노테이션/가드 없는 공개 핸들러" 존재 자체를 CI에서 검출(deny-by-default의 회귀 방지 — §6).

### 4.4 SECURITY-03·14 — 로깅·증적·알림 (U1 구체화)

#### 4.4.1 구조화 로깅·PII 금지 (SECURITY-03)

- **NFR-U1-SEC-21**: 전 서버 로그는 구조화(JSON) — 타임스탬프·상관 ID(요청 단위 전파, MDC)·레벨·모듈 태그 필수. 중앙 집중 로그 서비스로 라우팅(제품 선정은 클라우드 대기, §8 — U1은 stdout JSON까지 보장).
- **NFR-U1-SEC-22 (로그 금지 항목 — U1 열거)**: 비밀번호(평문·해시 모두)·액세스/리프레시 토큰·소셜 코드/id_token·이메일 인증/재설정 토큰·**이메일 주소·생년월일(PII)**·Authorization 헤더. 계정 식별은 내부 accountId로만. 마스킹은 로깅 프레임워크 레벨에서 구조적으로 강제(개별 로그문 재량 금지).

#### 4.4.2 동의 증적·법정 로그 (N2·N3·SECURITY-14)

- **NFR-U1-SEC-23 (동의 증적)**: ConsentRecord는 **append-only** — 항목·약관 버전·동의/철회 일시를 행 추가로만 기록(UPDATE/DELETE 금지). 철회도 revokedAt 신규 행 이력으로 남긴다. 보존: 계정 생명주기 + 분쟁 대비 법정 보존(D18 — 파기 시에도 법정 보존 데이터로 분리 보관).
- **NFR-U1-SEC-24 (위치정보 법정 로그, N2)**: 위치정보 수집·이용·제공 사실 확인자료 테이블 — **append-only·최소 6개월 보존·애플리케이션 DB 역할에 DELETE/UPDATE 권한 없음(DB 권한 분리로 강제)**. U1은 테이블·권한 모델·기록 인터페이스를 소유하고, 실제 기록 발생은 U6(GPS 수집)부터다. append-only는 마이그레이션 리뷰로 검증(U1 DoD).
- **NFR-U1-SEC-25 (보존 기간)**: 일반 애플리케이션·감사 로그 90일+(SECURITY-14 기본), 위치 법정 로그 6개월+(N2), 동의 증적은 4.4.2 기준 — 보존·아카이브 저장소 구성은 Infrastructure Design 대기(§8).

#### 4.4.3 감사 이벤트 목록·보안 알림 (SECURITY-14)

- **NFR-U1-SEC-26 (U1 감사 이벤트 전수)** — 행위자·시각·결과·상관 ID 포함 구조화 기록:

| 분류 | 이벤트 |
|---|---|
| 계정 생명주기 | 가입(경로별), 이메일 인증 완료, 연령 확인 통과/차단(N1), 계정 삭제 요청/철회/파기 실행(D18), 미인증 정리(G22) |
| 인증 | 로그인 성공/실패(사유 코드), 브루트포스 지연 발동/잠금, 토큰 회전, **리프레시 재사용 감지**, 로그아웃, 비밀번호 변경/재설정 요청·완료 |
| 동의 | 약관 동의/재동의(N3)·철회, 위치 약관·GPS 옵트인 동의/철회(N2), 마케팅 동의/철회(N8) |
| 권한 | 401/403 권한 위반(엔드포인트·주체), 삭제 유예 계정 접근 시도 |
| 프로필 | 닉네임 변경, 금칙어 차단(C3) |

- **NFR-U1-SEC-27 (알림 대상)**: (a) 인증 실패 급증(임계 초과), (b) 리프레시 재사용 감지(건당 즉시), (c) 권한 위반(403) 급증, (d) 브루트포스 잠금 급증, (e) 메일 발송 실패율. 알림 채널·대시보드 제품은 Infrastructure Design 대기(§8) — U1은 이벤트 발행 지점과 임계 정의까지.

### 4.5 토큰 클라이언트 저장 (SECURITY-12)

- **NFR-U1-SEC-28**: 액세스·리프레시 토큰은 **OS 보안 저장소만** 사용 — iOS Keychain / Android Keystore(expo-secure-store 경유, tech-stack §3.3). AsyncStorage·일반 파일·MMKV 평문 저장 금지. 로그아웃·401 확정 시 저장 토큰 즉시 삭제. 토큰을 로그·오류 리포트(크래시 리포팅 포함)에 포함하지 않는다(§4.4.1과 대칭 — 클라이언트측).

### 4.6 기타 U1 적용 규칙

- **SECURITY-04 (보안 헤더)**: U1 스캐폴드의 전역 응답 인터셉터로 HSTS·X-Content-Type-Options: nosniff·X-Frame-Options: DENY·Referrer-Policy 설정. CSP는 HTML 서빙 엔드포인트 부재(순수 JSON API)로 **실효 N/A이나 방어적 기본값(default-src 'none') 설정을 권고** — 오류 페이지 등 우발적 HTML 응답 대비.
- **SECURITY-11 (보안 설계 원칙)**: 인증·인가 로직은 M1 전용 모듈에 격리(모놀리스 내 분산 금지 — components.md §2-1과 정합), 공개 엔드포인트 rate-limit(§4.1.4), 오용 시나리오 명시(본 문서: 브루트포스·토큰 재사용·계정 열거·IDOR — 4.1~4.3).
- **SECURITY-15 (예외 처리)**: 전역 에러 핸들러(U1 스캐폴드 산출) — fail-closed, 사용자 응답은 일반화 메시지+오류 코드(스택·내부 경로 비노출, SECURITY-09와 결합), 전 외부 호출(IdP·메일·DB) 명시적 오류 처리, 오류 경로 자원 정리(트랜잭션 롤백).
- **SECURITY-10 (공급망)**: 잠금 파일 커밋(Gradle versions.lock/verification-metadata, pnpm/yarn lock), 의존성 취약점 스캔 CI 게이트, SBOM 생성, 프로덕션 이미지 태그 고정 — U1 CI 파이프라인 산출물 요건(unit-of-work.md U1 포함 범위).

## 6. 유지보수성 (D37 + 공통 설계 규약)

- **NFR-U1-MT-01 (모듈 경계 컴파일 타임 강제)**: 서버 모노레포는 Gradle 멀티모듈로 컴포넌트 경계(M1=`modules/auth`, M2=`modules/profile`, C3=`common/moderation`, `common/core`, `app`)를 **컴파일 의존성으로 강제**한다 — 타 모듈 내부 접근은 컴파일 에러. 공개 퍼사드·도메인 이벤트만 모듈 간 계약(components.md §2-1, D04). 보조로 아키텍처 테스트(ArchUnit/Konsist)를 CI에 두어 퍼사드 우회·순환 의존을 검출한다(unit-of-work.md U1 리스크 완화와 정합).
- **NFR-U1-MT-02 (테스트 계층 분리, D37)**: PR CI는 외부 의존(소셜 IdP 4종·메일) 전부 **Port에 대한 fake**로 실행 — 실 IdP 호출 없음. DB는 Testcontainers PostgreSQL(실 엔진 — 마이그레이션·append-only 권한 검증 포함). U1은 LLM 미사용이므로 실 LLM 회귀 계층 해당 없음.
- **NFR-U1-MT-03 (PBT 상시 실행)**: U1 PBT 대상(계정 상태 머신·토큰 회전·닉네임 생성기·직렬화 왕복 — unit-of-work.md U1 DoD)은 PR CI 포함, 시드 로깅·수축 필수(PBT-08). 프레임워크 요건 충족 확인은 tech-stack §2.9(PBT-09).
- **NFR-U1-MT-04 (설정 외부화)**: 브루트포스 수치(§4.1.4)·토큰 수명·메일 재발송 상한·해시 파라미터는 코드 재빌드 없이 조정 가능한 설정으로 외부화(remote config 관행 — G58·G106과 동일 패턴).
- **NFR-U1-MT-05 (관측성, §6.7)**: APM·크래시 리포팅 연동 훅, 외부 어댑터(IdP·메일) 실패율 계측, 헬스체크 shallow/deep(NFR-U1-AV-06) — 제품 선정은 클라우드 대기(§8).
