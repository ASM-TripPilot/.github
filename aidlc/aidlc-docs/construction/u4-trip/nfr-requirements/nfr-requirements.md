## 0. U4 NFR 프로필 — "솔버 입력 무결성"의 NFR 본질

U4에는 신규 인프라·외부 의존이 없다. U4 NFR의 본질은 **U5 솔버가 소비할 여행 컨텍스트(날짜·거점·필수 방문지·시간창·예산)의 무결성을 CRUD·검증 계층에서 보증하는 것**이다:

1. **데이터 무결성(핵심)** — 세 하드 제약(D37)이 U5 솔버의 "숙소 기준점·충돌 무배치" 입력 무결성이다: **날짜 겹침 차단(D21)·거점 구간 비중첩(D15)·필수 방문지 한도(G40)**. 이 셋의 위반은 곧 솔버 입력 오염이므로 100% 테스트가 머지를 차단한다(unit-of-work.md U4 DoD).
2. **파생 일관성** — 예산은 전체 총액 단일 정본(D26)이고 1인·1일은 파생값이다. 파생 계산의 왕복·반올림 손실을 한계 내로 강제한다.
3. **파괴적 변경 안전성** — 기간 축소·거점 삭제·필수 방문지 변경(G39·G43)은 하위 일정·기록을 손상할 수 있으므로 **차단형 확인·미리보기**로 무단 반영을 막는다.
4. **소유권** — 여행·거점·필수 방문지는 전부 계정 귀속(D15·D22)이므로 IDOR을 원천 차단한다(SECURITY-08).

보안 기본·무상태·관측성·복원력·배포의 서버 규약은 **U1 전역 정본을 그대로 상속**하고, U4는 CRUD·검증 계층의 무결성 증분만 소유한다.

### 0.1 워크로드 중요도 (requirements.md §6.3)

| 컴포넌트 | 중요도 | 불가용 시 영향·U4 대응 |
|---|---|---|
| 여행 저장/조회 (M6, DB) | **Critical** | §6.3 "여행·일정 조회/저장 Critical — 여행 중 접근 불가 시 현장 피해 최대". U4는 자체 데이터(외부 의존 0)이므로 가용성 바닥은 **RDS·Auth**(SI §3·U1 M1) 상속 — U4 신규 워크로드 없음 |
| 거점 연결·검증 (M6, U3 M4 데이터 소비) | Critical 경로 상속 | 등록 숙소·저장 POI 스키마는 U3 소유(CP1 소비), 여행 컨텍스트 검증·연결은 U4. 외부 의존 없음 |
| 필수 방문지 (M6) | Critical 경로 상속 | 저장 POI **사본**(G129 — 원본과 독립). 원본 소실도 사본은 유지(§2.4) |

**핵심 원칙**: U4는 **전부 자체 데이터 위에서 동작**(unit-of-work.md U4 — "외부 연동 신규 없음, 장소 검색은 M7 재사용")하므로 외부 순단·서킷 브레이커 대상이 없다. 가용성은 RDS·Auth(Critical) 상속이며, U4의 NFR 초점은 **가용성이 아니라 저장 데이터의 무결성**이다.

## 1. 성능 (D38, §6.1)

§6.1이 U4에 직접 부과하는 목표는 **일반 화면 전환 300ms**뿐이며(AI 생성은 U5·U6), U4 API는 전부 자체 데이터에 대한 **단일/소량 질의 CRUD**라 U1 §1.2의 CRUD 성능 규약을 상속한다(재산정 없음).

### 1.1 U4 API 응답 목표 (제안 — 서버 처리 시간 기준, 네트워크 RTT 별도 · U1 §1.2 상속)

| API (M6 인터페이스 그룹) | p50 | p95 | 상한(타임아웃) | 비고 · 추적 |
|---|---|---|---|---|
| 여행 생성(겹침 검증·시간창·속성 저장) | 100ms | 300ms | 3초 | 겹침 검증 = 인덱스 조회 + DB EXCLUDE 제약(§2.1). 전환 차단 API — 300ms 예산 준수 |
| 여행 조회/목록 | 50ms | 200ms | 3초 | 계정 스코핑 인덱스 조회(활성 ≤1·이력 소량, §3) |
| 여행 수정(날짜·인원·예산·제목) | 100ms | 300ms | 3초 | 재검증(겹침·기간·금칙어) 포함. 파괴적 변경은 §5 차단형 확인 선행 |
| 거점 연결·구간 편집(비중첩 검증) | 100ms | 300ms | 3초 | 동일 여행 거점 daterange 비중첩 검증(§2.2) — 트랜잭션 내 완결 |
| 필수 방문지 추가/삭제(사본 복제·한도 검증) | 50ms | 200ms | 3초 | 사본 INSERT + 일수 비례 한도 카운트(§2.4) |
| 필수 방문지 변경 미리보기(전/후 비교 산출) | 100ms | 300ms | 3초 | 미리보기 = 영향 범위 산출(§6, G43). 실제 재계산 적용은 U5 |
| 여행 컨텍스트 계약 조회(CP2 → U5 공급) | 100ms | 300ms | 3초 | 시간창·거점 목록·필수 방문지·속성 집약(§7.2, CP2) |

- **NFR-U4-PERF-01 (CRUD 300ms 규약 상속)**: 전 U4 API는 전환 차단 CRUD이므로 서버 p95 ≤ 300ms(U1 §1.2 규약)를 목표로 한다 — 자체 데이터·단일 질의라 외부 의존 지연이 없다. 클라이언트는 저장 성공 확인이 필요한 생성·검증 흐름을 낙관적 UI로 과도하게 감추지 않는다(하드 제약 위반은 즉시 오류 노출 — 침묵 실패 금지, ADR-0011).
- **NFR-U4-PERF-02 (검증 비용 국소화)**: 겹침 검증(D21)·거점 비중첩(D15)·한도(G40)는 **DB 제약·인덱스 조회 + 트랜잭션 내 검증**으로 국소화한다 — 전체 여행 스캔·N+1 조회를 금지(계정당 여행 이력 소량, §3). 겹침 검증은 daterange 인덱스(GiST)로 O(log n) 조회.
- **NFR-U4-PERF-03 (클라이언트 폼 전환)**: 여행 생성 폼(제목·날짜·인원·예산·시간창·취향 카드)·거점 편집·필수 방문지 선택 화면 전환은 300ms 예산 내(D38). 폼 검증(RHF+Zod)은 클라이언트 로컬 즉시 검증으로 왕복과 분리하되, **하드 제약(겹침·한도)은 서버가 정본**이므로 제출 시 서버 검증 결과를 신뢰한다(클라이언트 검증은 UX 선반영일 뿐).

## 2. 데이터 무결성 (핵심 · §6.6 D37 하드 제약, PBT-01)

U4 NFR의 척추다. 세 하드 제약은 **U5 솔버 입력 무결성**이며(unit-of-work.md U4 DoD — "테스트 100%·머지 차단"), 파생 일관성·사본 독립은 이를 보강한다.

### 2.1 여행 날짜 겹침 차단 — 동시성 (D21/Δ3, PBT)

- **NFR-U4-DI-01 (겹침 차단 하드 제약)**: 동일 계정의 여행은 **날짜 구간이 겹칠 수 없다**(D21 — "활성 여행은 항상 최대 1개"). 생성·수정·숙소 날짜 자동 반영(US-E4-05)·기간 확장(US-E4-03) 전 경로가 동일 검증을 통과한다. 위반 시 오류로 차단하고 **겹치는 기존 여행을 명시 + 편집·삭제 바로가기**를 제공한다(unit-of-work.md U4 리스크 완화 — UX 마찰 대응).
- **NFR-U4-DI-02 (동시성 경합 — 동일 계정 여행 생성 경합)**: 동일 계정이 **동시에 두 여행을 생성**하면 앱 레벨 사전 검증(SELECT)만으로는 둘 다 통과할 수 있다(TOCTOU 경합). 따라서 겹침 불변식은 **DB 레벨 제약이 정본 가드**여야 한다 — 앱 검증은 UX 오류 메시지용 사전 점검이고, 최종 무결성은 DB가 강제한다(설계는 nfr-design PAT-U4-01·tech-stack U4-S3). 경합 패배 트랜잭션은 제약 위반을 친절 오류로 변환한다(침묵 실패·중복 저장 금지).
- **불변식(PBT)**: 임의 여행 집합·임의 추가 순서에 대해 **겹침 차단 후 임의 시점(오늘 포함)을 커버하는 여행 ≤ 1**(unit-of-work.md U4 PBT). 시작일 오늘 이후·최대 30일(G42)도 동일 검증 경로(US-E4-01·05).

### 2.2 거점 구간 비중첩 — 트랜잭션 경계 (D15, PBT)

- **NFR-U4-DI-03 (거점 비중첩 하드 제약)**: **같은 여행에 연결된 거점끼리 날짜 구간이 겹치지 않는다**(D15·US-E4-06). 다중 숙소 구간별 거점·다박 연속 숙박(US-E4-07)은 daterange로 표현하고, 연결·구간 편집을 **단일 트랜잭션 경계** 안에서 검증한다(부분 반영 금지 — 검증 실패 시 전체 롤백).
- **NFR-U4-DI-04 (거점 공백·기본 거점 — 검증이 아닌 채움)**: 거점 공백일은 차단이 아니라 **스마트 기본 거점**으로 채운다 — 공백일=직전 숙소(US-E4-06), 첫날 등 직전 숙소 없으면 **여행지 중심 좌표 기본 거점**(G41). 이는 비차단 안내("이 날은 ○○ 기준 — 바꾸기")이지 하드 제약이 아니다(모달 남발 금지, US-E4-06).
- **NFR-U4-DI-05 (숙소 날짜↔여행 기간 정합)**: 숙소 체크인/아웃이 여행 기간을 벗어나면 **기간 자동 확장 여부를 사용자에게 확인**(US-E4-03)하고, 자동 반영된 기간도 겹침·기간·시작일 검증(D21·G42)을 재통과한다(US-E4-05). 여행 기간과 숙소 날짜 충돌 시 두 값을 나란히 제시해 선택하게 한다(무단 덮어쓰기 금지).
- **불변식(PBT)**: 거점 구간 집합의 **비중첩 속성 + 기간 커버리지 속성**(unit-of-work.md U4 PBT — 전환일·첫날 공백 거점 엣지 케이스 포함, CP2 계약).

### 2.3 예산 파생 일관성 (D26/Δ2, PBT)

- **NFR-U4-DI-06 (예산 단일 정본)**: 예산은 **여행 전체 총액(항공 제외) 단일 정본**으로 입력·저장하고, 1인·1일 값은 인원·일수로 나눈 **파생값**이다(D26 — 저장하지 않고 조회 시 계산, 또는 저장 시 정본과 정합 강제). 온보딩 러프 예산은 기본값 제시일 뿐 수정 가능(US-E4-01). 1박 가격대 환산(G26)은 일수 확보 후 파생.
- **NFR-U4-DI-07 (파생 왕복·반올림 한계)**: 총액 → (1인·1일) 파생과 역환산의 **왕복 손실이 반올림 한계 내**여야 한다(unit-of-work.md U4 PBT — "예산 총액↔1인·1일 파생값 왕복, 반올림 손실 한계"). 파생은 정밀 산술(BigDecimal, tech-stack U4-S4)로 계산하고 표시 반올림 정책을 명문화한다. 인원·일수 변경 시 파생값은 자동 재계산(정본 총액 불변).

### 2.4 필수 방문지 사본·한도 (G129·G40·G158, PBT)

- **NFR-U4-DI-08 (사본 복제 독립 — G129)**: 저장 POI를 필수 방문지로 투입하면 **사본 복제**되어 원본 저장 목록과 독립한다(원본 삭제와 무관). 사본은 명칭·좌표·카테고리 등 시드에 필요한 필드를 확정 시점 스냅샷으로 보유한다. 원본 소실 시에도 사본은 유지되며, 외부 출처 확인 불가는 **'확인 불가' 배지**(G8)로 표기(unit-of-work.md U4 리스크 완화 — "사본" 의미를 UI·설계 양쪽 명시).
- **NFR-U4-DI-09 (일수 비례 한도 하드 제약 — G40)**: 필수 방문지는 **하루 3곳 × 여행 일수**를 상한으로 하며 초과 시 추가를 차단하고 사유를 안내한다(US-E4-08). 이는 D37 하드 제약(테스트 100%)이다.
- **NFR-U4-DI-10 (권역 밖 경고·좌표 검증)**: 여행지 권역 밖 POI는 **경고 배지 + 기본 해제 상태**로 표시(G158), 투입은 사용자 선택. 지정 필수 방문지의 좌표 미확정은 "지도에서 위치 확인" 강제(US-E4-08), 등록·핀 지정 좌표는 **국내 좌표 범위 검증**(G120 — bounding box/역지오코딩, 범위 밖 차단·사유 안내).
- **불변식(PBT)**: 임의 일수·임의 추가 순서에 대해 **한도 초과 불가**, 사본이 원본 변경·삭제에 불변(unit-of-work.md U4 PBT).

### 2.5 여행 제목 결정성 (N6, PBT)

- **NFR-U4-DI-11 (제목 자동 생성·금칙어)**: 제목 미입력 시 **'{여행지} N박M일'** 형식으로 자동 생성하고, 입력·수정 시 **금칙어 검증**(C3 재사용, SECURITY-05)을 적용해 위반 시 저장을 막고 사유를 안내한다(N6·US-E4-11). 자동 생성은 결정적 순수 함수(동일 여행지·기간 → 동일 제목, unit-of-work.md U4 PBT).

## 3. 규모 (§6.8 G142 — DAU 1천)

- **NFR-U4-SC-01 (계정당 여행 볼륨)**: 겹침 차단(D21)으로 **활성 여행은 계정당 최대 1개**이며, 과거·미래 여행 이력도 개인 여행 빈도상 계정당 수십 건 이하로 가정한다. DAU 1천 × 이력 소량 = 여행 테이블 **수만 행** 수준 — 단일 PostgreSQL로 충분(SI §3, 샤딩·파티셔닝 불요). 거점·필수 방문지·시간창은 여행당 소수 행(거점 수 개·필수 방문지 ≤ 3×일수·시간창 = 일수)로 종속 볼륨도 작다.
- **NFR-U4-SC-02 (트래픽)**: 여행 생성·편집은 계획 단계 저빈도 쓰기(사용자당 여행 착수 시 수 회)이며, 조회는 홈 여행 카드(U2)·상세 진입 시. DAU 1천 기준 정상 피크 수 RPS 미만으로 **SI §1.2 태스크 2개가 흡수**(추가 컴퓨트·오토스케일 산정 불요, G142).
- **NFR-U4-SC-03 (신규 인프라 규모 결정 0)**: U4는 외부 API·캐시·스케줄러·비동기 잡이 없어 쿼터·오토스케일·큐 산정 대상이 없다(unit-of-work.md U4 — 스케줄러 잡 없음).

## 4. 보안 (Security Baseline 상속 — §6.4)

U4는 SECURITY 규칙의 신규 1차 구현 지점이 아니다 — 전역 보안 설정(deny-by-default·헤더·에러 핸들러·구조화 로깅·PII 마스킹)은 U1 스캐폴드가 소유하고 U4는 상속한다. U4 증분은 **소유권·입력 검증** 두 지점이다.

- **NFR-U4-SEC-01 (소유권 — SECURITY-08 IDOR)**: 여행·거점 연결·필수 방문지·여행 속성의 접근은 **토큰 주체(accountId) 기준으로만 조회·수정**한다(IDOR 원천 차단, NFR-U1-SEC-18 상속). 여행은 생성 계정 귀속, 거점 숙소는 **계정 레벨 풀**(D15) — 타 계정 여행에 타인 숙소·POI를 연결할 수 없다(소유권·IDOR 차단 D15·D22·SECURITY-08의 U4 입력 측). 리소스 ID 참조 시 서버가 소유권을 재검증한다(클라이언트 신뢰 금지).
- **NFR-U4-SEC-02 (전 입력 스키마 검증 — SECURITY-05)**: 여행 생성·수정·거점·필수 방문지·시간창의 전 파라미터는 스키마 검증(Jakarta Validation, U1 관례 상속) — 날짜(오늘 이후·≤30일 G42)·인원·예산(음수·과대 상한)·좌표(국내 범위 G120)·시간창(시작<종료)·문자열 최대 길이. 제목·메모 등 UGC는 **금칙어 + 새니타이즈**(C3 재사용) 후 저장, 출력 이스케이프(XSS 방어, NFR-U1-SEC-14 상속).
- **NFR-U4-SEC-03 (파괴적 변경 승인 경계)**: 기간 축소·거점 삭제·확정 일정의 필수 방문지 변경은 **차단형 확인·미리보기 승인 없이 반영 불가**(G39·G43, §5·§6) — 무단 파괴적 쓰기를 서버가 거부한다(멱등·확인 토큰 기반, nfr-design PAT-U4-05·06).

---

## 5. 파괴적 변경 차단형 확인 (G39, ADR-0011)

- **NFR-U4-CHG-01 (기간 축소·거점 삭제 차단형 확인 — G39)**: 여행 기간 축소·숙소(거점) 삭제는 하위 일정·기록을 손상할 수 있으므로 **서버가 영향 블록을 나열**하고 사용자 확인 없이 반영하지 않는다(G39·US-E4 리스크 완화). 영향 산출은 서버 정본(클라이언트 추정 금지) — 삭제될 거점·재계산될 필수 방문지 배치·손상 일정 블록을 구조화 응답으로 반환(설계는 nfr-design PAT-U4-05).
- **NFR-U4-CHG-02 (침묵 실패 금지)**: 파괴적 변경의 영향은 빈 화면·무통보가 아니라 **명시적 영향 목록 + 확인 게이트**로 노출한다(ADR-0011). 확인 후에만 트랜잭션 반영.

## 6. 필수 방문지 변경 재계산 미리보기 (G43, U5 경계)

- **NFR-U4-CHG-03 (변경 미리보기 — G43)**: 필수 방문지를 추가·삭제·변경하면 **변경안 미리보기(전/후 비교)를 제시하고 사용자 승인 후에만 적용**한다(G43·US-E4-08). U4는 생성 전 여행 컨텍스트의 필수 방문지 CRUD와 **미리보기 산출·승인 계약**을 소유한다.
- **NFR-U4-CHG-04 (재계산 실행 경계 — U5)**: **확정된 일정의 실제 재계산**(추천 POI 배치 재산출)은 **U5 솔버 소관**이다(US-E4-08 — "생성 후에도 추가·삭제, 변경 시 해당 범위 재계산"의 재계산 본체는 C2/U5). U4는 필수 방문지 변경 이벤트·미리보기 요청을 발생시키고, U5가 재계산 결과를 반환한다(CP2 계약 경계). 고정 블록(시각 고정 필수 방문지·숙소)은 재생성에도 불변(US-E4-09) — U4는 이 불변 속성을 스키마로 표현하고 U5가 warm-start로 보존.

## 7. 미결·계약 항목

### 7.1 미결 항목 — 없음 (전역 결정 상속)

**U4는 신규 인프라·벤더 종속·비개발 선결 미결 항목이 없다.** U1이 PD-1~9를 전부 종결했고, U4는 그 종결 결과와 GD-1~7·SI를 상속한다. U4는 **전부 자체 데이터 위에서 동작**(외부 연동 0)하므로 U3의 P2·P4 같은 벤더·약관 선결도 무관하다(unit-of-work.md U4 §8 — "직접 선결 없음, U3의 P2/P4 완료를 전제").

| 잠재 결정 항목 | U4 처리 | 근거 |
|---|---|---|
| 클라우드·CI/CD·배포·롤백 | GD-1~7 상속(재질문 없음) | u1-foundation-nfr-design-plan.md §2 |
| 컴퓨트·DB·시크릿·로그·알림 스택 | SI 정본 상속(재정의 금지) — SI §0 "U4 = 추가 없음" | SI §1·3·6·8 |
| 서버 프레임워크·인증·DB 접근·마이그레이션 | U1 S-2·S-3·S-8·S-9 상속(신규 0) | tech-stack-decisions.md |
| 날짜 겹침 동시성 제어 | 기존 PostgreSQL 기능(EXCLUDE·daterange) 내 선택 — 신규 인프라 아님 | tech-stack U4-S2·S3 |

### 7.2 계약 포인트 (unit-of-work.md U4)

- **CP1 소비자 측(U3 → U4)**: 등록 숙소·저장 POI 스키마 계약 테스트 — U4는 U3(M4·M7)이 공급한 스키마를 소비(거점 연결·필수 방문지 사본 시드). U3 CP1 공급자 측과 대칭.
- **CP2 공급자 측(U4 → U5)**: **여행 컨텍스트 계약**(시간창·거점 목록·필수 방문지·여행 속성) 계약 테스트 — U5 솔버 문제 정의의 입력 스키마를 U4가 고정한다. 엣지 케이스(전환일·첫날 공백 거점·시각 고정 충돌)를 계약 테스트에 상비(unit-of-work.md U4 리스크 완화).
