# 1. U4 신규 패턴

## PAT-U4-01 여행 날짜 겹침 차단 동시성 제어 — 활성 여행 ≤1 `[U4]`

**문제**: 동일 계정의 여행은 날짜 구간이 겹칠 수 없다(D21 — 활성 여행 항상 최대 1개). 그런데 앱 레벨 "SELECT로 겹침 확인 → INSERT"만 두면, **동일 계정이 동시에 두 여행을 생성**할 때 두 트랜잭션이 모두 겹침 없음으로 판정하고 둘 다 저장하는 경합(TOCTOU)이 발생해 불변식이 깨진다. 이는 U5 솔버 입력 오염(어느 여행이 기준인지 모호)으로 직결된다.

**적용(설계)**: 겹침 불변식을 **DB 레벨 배제 제약이 정본 가드**로 강제하고, 앱 검증은 UX 오류 품질용 사전 점검으로 둔다(2중):

```text
[여행 생성/수정/숙소 날짜 반영 요청]
  → ① 앱 사전 검증(UX): 계정 여행 daterange 겹침 SELECT → 겹치면 친절 오류
        "○○ 여행과 날짜가 겹쳐요" + [편집]/[삭제] 바로가기
  → ② INSERT/UPDATE (트랜잭션)
        DB EXCLUDE 제약: gist(account_id WITH =, date_range WITH &&)
        └ 동시 경합 패배 트랜잭션 → 배제 위반 예외(SQLState 23P01)
              → 도메인 예외로 변환 → ①과 동일 친절 오류(중복 저장 0)
  → 시작일 오늘 이후·최대 30일(G42) 검증도 동일 경로
```

- **설계 원칙**: (a) **DB 제약이 최종 무결성 정본** — 앱 검증 통과 여부와 무관하게 겹침 저장이 물리적으로 불가, (b) 경합·제약 위반은 침묵·중복 저장이 아니라 **친절 오류 + 겹치는 여행 명시**(unit-of-work.md U4 리스크 완화 — UX 마찰 대응), (c) 숙소 날짜 자동 반영(US-E4-05)·기간 자동 확장(US-E4-03)도 이 검증 경로를 재통과, (d) daterange GiST 인덱스로 겹침 조회 O(log n)(NFR-U4-PERF-02).

**U4 적용 지점**: `modules/trip` 여행 애그리게이트·생성/수정 유스케이스, Flyway 마이그레이션의 EXCLUDE 제약(infrastructure-design.md §2), 배제 위반 → 도메인 예외 변환 핸들러, 겹침 오류 UI(겹치는 여행 바로가기).

**검증 기준**: D21/PBT — **임의 여행 집합·임의 추가 순서에 대해 겹침 차단 후 임의 시점 커버 여행 ≤1** 불변식(unit-of-work.md U4 PBT), **동시 생성 경합 주입 시 하나만 성공·중복 저장 0**(제약 위반 → 친절 오류) 통합 테스트, G42 시작일·최대 30일 경계 테스트. 하드 제약이므로 테스트 100%·머지 차단(D37, §6.6 G114).

## PAT-U4-02 거점 구간 비중첩 검증 트랜잭션 `[U4]`

**문제**: 하나의 여행에 다중 숙소를 구간별 거점으로 연결할 때(US-E4-06), 거점끼리 날짜 구간이 겹치면 "그날의 기준 거점"이 모호해져 솔버가 숙소 기준점을 잡을 수 없다(D15). 또 다중 거점 연결·구간 편집을 여러 쿼리로 나눠 부분 반영하면 중간 상태에서 불변식이 깨진다.

**적용(설계)**: 거점 구간을 daterange로 표현하고, **연결·편집을 단일 트랜잭션 경계**에서 비중첩 검증한다:

- **DB 비중첩 제약**: `EXCLUDE USING gist (trip_id WITH =, stay_range WITH &&)` → 동일 여행 거점 구간 중첩을 원자 차단(D15). 다박 연속 숙박(US-E4-07)은 단일 daterange 행으로 표현(매일 반복 아님).
- **트랜잭션 경계**: 다중 거점 일괄 연결·구간 조정은 `@Transactional` 단일 커밋 — 검증 실패 시 전체 롤백(부분 반영 금지). 숙소 날짜↔여행 기간 정합(NFR-U4-DI-05)은 기간 확장 확인 후 동일 트랜잭션.
- **거점 공백 = 차단 아닌 채움**: 공백일은 모달 남발이 아니라 **스마트 기본 거점**(공백일=직전 숙소, 첫날 없으면 여행지 중심 좌표 G41)으로 채우고 "이 날은 ○○ 기준 — 바꾸기" 비차단 안내(US-E4-06). 이는 UX 채움이지 하드 제약이 아니다(NFR-U4-DI-04).

**U4 적용 지점**: `modules/trip` 여행-거점 연결 애그리게이트(조인·날짜 구간), 거점 구간 편집 유스케이스, EXCLUDE 제약 마이그레이션, 기본 거점 채움 로직(G41), 겹침 오류 UI(겹치는 날짜 표시·거점 선택).

**검증 기준**: D15/PBT — **거점 구간 집합의 비중첩 속성 + 기간 커버리지 속성**(전환일·첫날 공백 거점 엣지 케이스, unit-of-work.md U4 PBT·CP2 계약), 다중 거점 일괄 연결의 트랜잭션 원자성(부분 실패 시 롤백) 테스트. 하드 제약 100%·머지 차단(D37).

## PAT-U4-03 예산 단일 정본(전체 총액)→파생 일관성 `[U4]`

**문제**: 예산을 전체 총액·1인·1일로 각각 저장하면 인원·일수 변경 시 세 값이 어긋나 사용자를 오도한다(README·온보딩의 '1인 기준' 서술을 D26이 전체 총액으로 개정한 이유). 부동소수점 나눗셈은 반올림 오차가 누적돼 왕복(총액→파생→총액)이 원값과 벌어진다.

**적용(설계)**: **전체 총액을 단일 정본(SSOT)**으로 두고 파생은 계산으로만 존재:

| 값 | 저장 | 계산 |
|---|---|---|
| 전체 총액(항공 제외) | **정본(저장)** | 입력·저장 모두 총액(D26) |
| 1인 금액 | 파생(비저장) | 총액 ÷ 인원 |
| 1일 금액 | 파생(비저장) | 총액 ÷ 일수 |
| 1박 가격대 환산(G26) | 파생 | 일수 확보 후 환산 |

- **설계 원칙**: (a) 파생값은 저장하지 않고 조회 시 `BigDecimal` 정밀 계산(U4-S4) — 저장한다면 정본 총액과 정합을 강제(비저장이 기본), (b) **표시 반올림 정책 명문화**(스케일·`HALF_UP` 등)로 왕복 손실을 예측 가능하게, (c) 인원·일수 변경 시 파생값 자동 재계산(정본 총액 불변), (d) 온보딩 러프 예산은 기본값 제시일 뿐 수정 가능(US-E4-01).

**U4 적용 지점**: `modules/trip` 예산 값 객체(총액 정본 + 파생 계산 메서드), 예산 표시 DTO(파생 반올림), 클라이언트 예산 입력·파생 표시(RHF, US-E4-01).

**검증 기준**: D26/PBT — **총액↔1인·1일 파생 왕복 손실이 반올림 한계 내**(unit-of-work.md U4 PBT), 인원·일수 변경 시 파생 재계산·정본 불변 테스트, 음수·과대 예산 입력 검증(SECURITY-05).

## PAT-U4-04 필수 방문지 사본 복제·일수 비례 한도 불변식 `[U4]`

**문제**: 저장 POI를 필수 방문지로 투입할 때 참조(원본 공유)로 두면 원본 삭제·변경이 여행 시드를 오염시킨다(G129 — 사본이어야 독립). 또 필수 방문지를 무제한 추가하면 솔버 제약이 과잉이 돼 해가 없어진다(G40). 사용자가 "사본"을 원본 동기화로 오해하면 데이터 정합 혼란이 생긴다(unit-of-work.md U4 리스크).

**적용(설계)**:

- **사본 복제(G129)**: 투입 시 저장 POI를 **사본으로 복제**(명칭·좌표·카테고리 등 시드 필드 확정 스냅샷) — 원본 저장 목록과 독립. 원본 삭제·소실에도 사본 유지, 외부 출처 확인 불가는 **'확인 불가' 배지**(G8). "사본" 의미를 UI 카피·설계 양쪽에 명시(리스크 완화).
- **일수 비례 한도(G40 하드 제약)**: 상한 = **하루 3곳 × 여행 일수**. 추가 시 현재 카운트 검증, 초과 시 차단 + 사유 안내(US-E4-08). 여행 일수 변경 시 상한도 변동(축소로 초과 시 §PAT-U4-05 차단형 확인과 연동).
- **권역·좌표(G158·G120)**: 여행지 권역 밖 POI는 경고 배지 + 기본 해제(G158), 좌표 미확정은 "지도에서 위치 확인" 강제(US-E4-08), 국내 좌표 범위 검증(G120 — 범위 밖 차단).
- **고정/포함 구분(US-E4-09)**: '시간 정해두기'(시각 고정) vs '아무 때나 꼭 가기'(포함만) 토글 — 시각 고정 필수 방문지·숙소는 재생성에도 불변(U5 warm-start 대상), 포함형은 시각·순서 재계산 가능하나 누락 불가. U4는 이 구분을 스키마 속성으로 표현.

**U4 적용 지점**: `modules/trip` 필수 방문지 애그리게이트(POI 사본·시각 고정·포함/고정 플래그), 사본 복제 유스케이스, 한도 검증기, 체크박스 선택 화면(권역 경고 G158)·시각 고정 편집 UI.

**검증 기준**: G40/PBT — **임의 일수·임의 추가 순서에 대해 한도 초과 불가**(하드 제약 100%·머지 차단, D37), G129 — **사본이 원본 변경·삭제에 불변** 속성 테스트 + 원본 소실 시 '확인 불가' 배지 시나리오(리스크 완화), G120 국내 좌표 범위 차단 테스트.

## PAT-U4-05 파괴적 변경 차단형 확인 — 영향 산출 `[U4]`

**문제**: 여행 기간 축소·거점(숙소) 삭제는 하위 일정·기록·필수 방문지 배치를 손상할 수 있다(unit-of-work.md U4 리스크). 확인 없이 반영하면 사용자가 의도치 않게 일정을 파괴하고, 영향 범위를 클라이언트가 추정하면 불일치가 생긴다.

**적용(설계)**: **서버가 영향을 산출하고 확인 없이 반영하지 않는다**(G39):

```text
[기간 축소 / 거점 삭제 요청]
  → 서버가 영향 블록 산출(정본): 삭제될 거점·범위 밖으로 밀리는 일정 블록·
        재계산 대상 필수 방문지·초과되는 한도(G40)
  → 구조화 영향 응답 반환(빈 화면·무통보 금지 — ADR-0011)
  → 사용자 확인 게이트 통과 시에만 트랜잭션 반영(멱등·확인 토큰)
  → 미확인 상태의 파괴적 쓰기는 서버가 거부(NFR-U4-SEC-03)
```

- **설계 원칙**: (a) 영향 산출은 **서버 정본**(클라이언트 추정 금지), (b) 확인 게이트 없는 파괴적 쓰기 거부(2단계 요청 또는 확인 플래그), (c) 침묵 실패 금지 — 영향 목록을 명시 노출(§PAT-RES-03 상속).

**U4 적용 지점**: `modules/trip` 파괴적 변경 영향 산출 서비스(기간·거점), 확인 게이트 API(2단계/확인 토큰), 기간 변경 영향 확인 다이얼로그(클라이언트).

**검증 기준**: G39 — 기간 축소·거점 삭제 시 영향 블록 산출 정확성 테스트, **미확인 파괴적 쓰기 거부** 테스트, 확인 후 반영의 트랜잭션 정합. 침묵 실패 없음 통합 테스트(ADR-0011).

## PAT-U4-06 필수 방문지 변경 재계산 미리보기 핸드오프 `[U4 — U5 경계]`

**문제**: 필수 방문지를 추가·삭제·변경하면 추천 POI 배치가 달라진다(US-E4-08). 사용자 확인 없이 확정 일정을 바꾸면 신뢰가 깨지고(G43 — 미리보기·승인 필요), 재계산 로직을 U4가 중복 구현하면 U5 솔버와 이원화된다.

**적용(설계)**: **U4는 변경·미리보기 요청·승인 계약을 소유하고, 재계산 본체는 U5로 핸드오프**:

```text
[필수 방문지 추가/삭제/변경]
  → U4: 필수 방문지 CRUD(사본·시각 고정·한도 PAT-U4-04) + 변경 이벤트
  → 확정 일정 대상이면 → U5 솔버에 재계산 미리보기 요청(CP2 계약)
        └ 고정 블록(시각 고정 필수 방문지·숙소)은 불변(warm-start, US-E4-09)
  → U5: 변경안(전/후 비교) 반환 → U4/클라이언트가 미리보기 제시
  → 사용자 승인 후에만 적용(G43) — 미승인 시 원상 유지
```

- **설계 원칙**: (a) **재계산 실행 경계 = U5**(NFR-U4-CHG-04) — U4는 필수 방문지 상태·변경 이벤트·미리보기 승인 흐름만 소유, (b) 고정/포함 구분(PAT-U4-04)이 warm-start 보존 대상을 규정(U5가 소비), (c) 미리보기 승인 없이 확정 일정 무단 변경 금지(G43).

**U4 적용 지점**: `modules/trip` 필수 방문지 변경 이벤트·미리보기 요청(CP2 → U5), 승인 적용 유스케이스, 변경안 미리보기 UI(전/후 비교). 재계산 결과 생성은 U5 C2 솔버.

**검증 기준**: G43 — 필수 방문지 변경 시 미리보기 제시·**미승인 시 미적용** 테스트, CP2 계약(여행 컨텍스트·필수 방문지·고정 블록 불변 속성) 계약 테스트, 고정 블록이 재계산에 불변(US-E4-09) 검증(실행은 U5, U4는 계약·불변 속성 고정).
