# U6 여행 중 실행·Plan-B — 기술 스택 결정 (Tech Stack Decisions)

> 2026-07-04 · CONSTRUCTION · U6 NFR Requirements 산출물
> **전제**: 스택 대분류(RN Expo + Spring Boot Kotlin + PostgreSQL, D02)와 서버·클라이언트 라이브러리 정본(U1 [tech-stack-decisions.md](../../u1-foundation/nfr-requirements/tech-stack-decisions.md) S-1~S-13·C-1~C-8), 복원력 라이브러리(Resilience4j — U3-S1), 스케줄러(ShedLock — SI §1.2)는 **기확정**이다. U6는 그 위에 **① 기상청 API 클라이언트, ② 서버 배치 폴링, ③ 클라이언트 지오펜스/포그라운드 위치, ④ GPS 폴리라인 단순화** 4개 지점만 구체화하며, **재계획 연산(C1 LLM Gateway·C2 Solver Engine)은 U5 확정분을 재사용(신규 없음)** 한다.
> **재결정 금지**: 클라우드·인프라·서버 프레임워크는 SI·U1 정본 상속. **날씨 소스는 D10 기확정**(기상청 공공데이터포털 단기예보+특보). **하이브리드 트리거는 D27 기확정**(날씨·휴무=서버 폴링, 위치=클라 포그라운드). 본 문서는 U6의 선택지가 있는 지점만 대안 비교 후 권고한다. **기상청 API 활용 신청·쿼터는 P3 확정** — 본 문서는 **어댑터 중립 구조**를 확정한다(nfr-requirements.md §8.1).

## 2. 기상청 API 클라이언트 — 격자 변환 + 트리거 침묵 (U6-W1)

D10이 **"기상청 공공데이터포털 단기예보 + 기상특보"** 를 확정했으므로 U6는 **소스 선택이 아니라 어댑터 구조**를 확정한다. 기상청 API 활용 신청·쿼터는 **P3** 확정이지만(nfr-requirements.md §8.1), **어댑터 추상화·격자 변환·타임아웃·서킷·트리거 침묵 폴백은 지금 확정**한다 — 이것이 소스 교체·무응답을 어댑터 내부로 격리하는 방식이다.

### 2.1 어댑터 책임 (`WeatherPort` = M11, 트리거 모듈 무의존)

| 책임 | 설계 | 근거 |
|---|---|---|
| **단기예보·특보 조회** | `WeatherPort` 인터페이스 뒤 기상청 REST 클라이언트(U1 Spring `RestClient` 상속) 은닉. M9 트리거는 강수확률·특보 여부만 인지 | D10, services.md §S2.1 |
| **좌표→격자 변환** | 기상청 단기예보는 격자(nx·ny) 단위 — **LCC DFS 격자 변환 알고리즘**(기상청 공식)으로 방문 예정지 좌표를 격자로 변환·중복 제거(호출 억제) | services.md J1, NFR-U6-SC-02 |
| **타임아웃·서킷** | Resilience4j 타임아웃 + 독립 서킷(U3·U5 재사용). 즉시 1회 재시도(services.md J1) 후 실패 | RESILIENCY-10, NFR-U6-RES-08 |
| **트리거 침묵 폴백** | 무응답·서킷 open 시 마지막 성공 데이터 TTL 내 사용 → 만료 시 **자동 알림 미발화**(허위 알림 금지) + 실패율 계측 | D27, NFR-U6-RES-01·02 |
| **격자 캐시** | 격자 단위 인메모리 TTL 캐시(Caffeine 재사용) — 폴링 주기(1h) 내 동일 격자 재조회 억제 | U3-S4, NFR-U6-SC-02 |

- **대안 비교(1줄)**: (A) **`WeatherPort` 어댑터 추상화(권고)** — 소스 교체·격자 변환·타임아웃·트리거 침묵을 한 곳에 격리(D27 정합). (B) 트리거 모듈이 기상청 직접 호출 — 소스 종속·침묵 로직 분산으로 허위 알림 금지 강제가 약해짐(비채택). (C) 유료 날씨 SaaS(OpenWeather 등) — D10이 기상청 공공데이터포털을 확정(무료·국내 특보 직접), 대안 불필요·비용 증가(비채택).
- **선정 사유**: **트리거 침묵(허위 알림 금지 D27)** 은 U6 복원력의 핵심이며, 이는 어댑터 한 곳에서 무응답을 감지해 발화를 억제해야 구조적으로 보장된다. 격자 변환·캐시·서킷도 어댑터 내부에 격리해야 트리거 순수 함수(G116)가 외부 상태에 오염되지 않는다. **P3 미확정(쿼터 미상) 상태에서도 어댑터·격자 변환·침묵 폴백은 확정 가능**(소스 무관) — 이것이 P3 비차단의 근거다.

### 2.2 P3 경계 — 지금 확정 vs P3 후

| 지금 확정(소스 무관) | P3 후 확정(소스 의존) |
|---|---|
| `WeatherPort` 인터페이스·LCC DFS 격자 변환·타임아웃·서킷·트리거 침묵·격자 캐시 TTL·즉시 1회 재시도 | 기상청 서비스 키·문서화 쿼터(폴링 주기 상한 G195 최종값)·엔드포인트 버전 |

## 3. 서버 배치 폴링 — 기존 ShedLock 스케줄러 재사용 (U6-P1)

날씨·휴무 트리거는 **서버 배치 폴링**(D27)이다. SI §1.2가 "U1~U8 스케줄러 잡은 별도 워커 없이 앱 프로세스 내 `@Scheduled` + ShedLock 분산 락"으로 확정했으므로 U6는 이를 재사용한다.

| 기준 (G142 · 진행 중 여행 소량) | **A. 기존 ShedLock 스케줄러 재사용 (권고)** | B. 별도 폴링 워커 서비스 + 큐 | C. EventBridge Scheduler + ECS RunTask |
|---|---|---|---|
| 신규 AWS 리소스 | **0** — 앱 프로세스 내 `@Scheduled` + ShedLock(SI §1.2) | +ECS 서비스·SQS·SG (SI §4 브로커 미도입 위반) | +EventBridge·태스크 정의 |
| 주기·부하 | 날씨 1h·휴무 아침 1회·자정 종료 — 저부하(진행 중 여행 소량) | 과설계 | 과설계 |
| 중복 실행 방지(태스크 2개) | ShedLock 분산 락(SI §1.2) | 큐 가시성 타임아웃 | 스케줄 단일 트리거 |
| 아웃박스 연계(TriggerFired) | **직접**(동일 프로세스 DB 아웃박스, SI §4) | 크로스 프로세스 | 크로스 프로세스 |
| 규모 정합(G142·활성 여행 ≤1/사용자) | **적합** | 과설계 | 과설계 |

- **확정 권고: A(기존 ShedLock 스케줄러 재사용) · 별도 워커 미도입.** 근거: (1) 폴링 잡(J1 날씨 1h·J2 휴무 아침·J3 자동 종료 자정)은 **전부 저빈도·저부하**이고 진행 중 여행 수가 소량(D21 활성 ≤1/사용자)이라 기존 태스크 2개로 흡수(NFR-U6-SC-01), (2) TriggerFired 아웃박스 발행이 동일 프로세스 DB 아웃박스(SI §4)와 직결, (3) 별도 워커·큐는 SI §4(브로커 없음) 결정과 배치. **주기·상한은 전부 원격 구성**(G58/G195, NFR-U6-PRIV-04) — 코드 재배포 없이 조정.
- **B/C 비채택 사유(1줄)**: 별도 워커·큐·EventBridge는 진행 중 여행 소량·저빈도 폴링에 과설계이며 SI §4(DB 기반·브로커 없음)와 배치. **전환 트리거**: 폴링 대상·기상청 호출량이 태스크·쿼터를 상시 압박하면 EventBridge + ECS RunTask 분리 재평가(SI §1.2 트리거 상속).
- **잡 목록(services.md §4 재사용)**: J1 날씨·특보 폴링(M9 via M11, 1h)·J2 휴무·영업시간 재조회(M9 via M7, 아침 07:00)·J3 여행 자동 종료·일자 경계(M18, 자정 KST). 전부 상태 기반 조회 + 조건부 전이로 자연 멱등(services.md §4).

## 4. 클라이언트 지오펜스·포그라운드 위치 — expo-location (U6-C1)

도착 감지·이동 지연·체류 초과는 **클라이언트 포그라운드 한정 감지**(D27)다. RN Expo(D02·U1 C-1) 위치 모듈을 구체화한다.

| 기준 (D27 포그라운드 한정·배터리) | **A. expo-location 포그라운드 한정 (권고)** | B. 백그라운드 지오펜스(expo-location background) | C. OS 네이티브 지오펜스 저전력 API |
|---|---|---|---|
| D27 정합(포그라운드 한정) | **정합** — 포그라운드 권한만 요청 | **D27 위반**(백그라운드 위치 권한) | 백그라운드 상주(D27 위반) |
| 배터리 | **저소모** — 포그라운드·저빈도(1~5분) | 상시 백그라운드 추적(고소모) | 저전력이나 백그라운드 |
| 커버리지 | 앱 사용 중만(한계 수용 D27) | 넓음(그러나 금지) | 넓음(그러나 금지) |
| 프라이버시·법정 | 최소 수집(위치정보법 정합) | 과잉 수집 리스크 | 과잉 수집 리스크 |
| Expo 정합 | **Expo SDK 표준 모듈** | 지원하나 정책 위반 | config plugin·네이티브 복잡 |

- **확정 권고: A(expo-location 포그라운드 한정) · 백그라운드 위치 미도입.** 근거: (1) D27·G62가 **포그라운드 한정·백그라운드 위치 미요청**을 확정, (2) 저빈도(1~5분) 포그라운드 수집으로 배터리 소모 억제(NFR-U6-PRIV-02), (3) 지오펜스 진입 평가는 **클라이언트 로컬 연산**(무네트워크 — 도착 프롬프트 즉시, NFR-U6-PERF-05). 커버리지 한계(백그라운드 미발화)는 **제품 동작으로 수용**하고 앱 재진입 시 일괄 재평가(unit-of-work.md U6). **OS 지오펜스 저전력 도착 감지는 후속 검토**(D27·G62 — 지금 미도입).
- **B/C 비채택 사유(1줄)**: 백그라운드 지오펜스·OS 저전력 지오펜스는 커버리지가 넓으나 **D27 포그라운드 한정 결정 위반**이자 위치정보법·배터리 리스크로 1차 비채택(후속 검토).
- **네이티브 델타 주석**: expo-location은 iOS `NSLocationWhenInUseUsageDescription`·Android 위치 권한 문자열을 **config plugin**으로 주입하는 네이티브 모듈이다 — U3(카카오 지도 SDK)·U1('내 주변' 프리프롬프트) 시점에 도입됐는지 확인하고, 신규 권한 문자열 추가 시 **EAS Build 재빌드 델타**(OTA 불가), 기존 권한 재사용이면 순수 JS OTA(infrastructure-design.md §5 확인).

## 5. GPS 폴리라인 단순화 — 원시 좌표 파기 (U6-C2)

GPS 발자취는 **단순화 폴리라인만 서버 보존·원시 좌표 파기**(G73·D34)다. 단순화 방식을 구체화한다.

| 기준 | **A. Douglas-Peucker 단순화 (권고)** | B. 원시 좌표 전량 저장 | C. 서버 전용 단순화(원시 업로드 후) |
|---|---|---|---|
| 원시 파기(G73·D34) | **클라 1차 단순화 → 원시 즉시 파기** | **G73 위반**(원시 보존) | 원시가 서버에 잠시 상주(파기 지연·리스크) |
| 저장 볼륨 | 소(위상 보존 포인트만) | 대(고빈도 좌표) | 소(단, 업로드 볼륨 대) |
| 프라이버시·법정 | 최소 보존(위치정보법 정합) | 과잉 보존 | 원시 전송·상주 리스크 |
| 네트워크 | 소(단순화 후 전송) | 대 | 대(원시 전송) |

- **확정 권고: A(Douglas-Peucker 클라 1차 단순화 + 서버 정규화) · 원시 좌표 클라 즉시 파기.** 근거: (1) **원시 좌표를 클라에서 단순화 후 즉시 파기**해 서버·네트워크에 원시가 상주하지 않게(G73·D34·NFR-U6-PRIV-06), (2) 단순화 폴리라인은 위상(경로 형태)을 보존하되 원시 재구성 불가(§6 PBT 대상), (3) 저장·네트워크 볼륨 최소(규모 정합). 서버는 수신 폴리라인을 추가 정규화·저장(법정 로그와 동일 TX, NFR-U6-SEC-01).
