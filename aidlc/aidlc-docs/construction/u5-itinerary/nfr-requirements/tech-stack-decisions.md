# U5 AI 일정 생성·확정 — 기술 스택 결정 (Tech Stack Decisions)

> 2026-07-04 · CONSTRUCTION · U5 NFR Requirements 산출물
> **전제**: 스택 대분류(RN Expo + Spring Boot Kotlin + PostgreSQL, D02)와 서버·클라이언트 라이브러리 정본(U1 [tech-stack-decisions.md](../../u1-foundation/nfr-requirements/tech-stack-decisions.md) S-1~S-13·C-1~C-8)은 **기확정**이다. U5는 그 위에 **① C1 LLM Gateway, ② C2 Solver Engine, ③ 점진 응답 스트리밍, ④ 백그라운드 잔여 일자 잡** 4개 지점만 구체화하며, 나머지(프레임워크·인증·DB 접근·마이그레이션·상태 관리·HTTP·PBT)는 U1 결정을 상속한다.
> **재결정 금지**: 클라우드·인프라·서버 프레임워크는 SI·U1 정본 상속. **LLM 아키텍처는 D11 기확정**(단일 벤더 관리형·서버 경유·기능별 티어). **지도·거리 벤더는 D08 기확정**(TMap 도로거리). 본 문서는 U5의 선택지가 있는 지점만 대안 비교 후 권고한다. **LLM 벤더·모델·단가는 P6 확정** — 본 문서는 **벤더 중립 추상화 구조**만 지금 확정한다(nfr-requirements.md §7.1).

## 2. C1 LLM Gateway — 벤더 중립 추상화 + 티어 라우팅 (U5-C1)

D11이 **"단일 벤더 관리형 API·서버 경유·기능별 모델 티어 분리"** 를 확정했으므로 U5는 **벤더 선택이 아니라 게이트웨이 구조**를 확정한다. 벤더·모델·단가는 **P6 비용 산정 후** 결정되지만(nfr-requirements.md §7.1), **추상화 인터페이스·티어 라우팅·출력 검증·rate-limit·비용 계측은 지금 확정**한다 — 이것이 벤더 교체를 어댑터 구현 교체로 한정하는 격리다(component-dependency §6 "LLM 벤더 전환은 어댑터 구현 교체로 한정").

### 2.1 게이트웨이 책임 (C1 = `common/llm-gateway`, 기능 모듈 무의존)

| 책임 | 설계 | 근거 |
|---|---|---|
| **단일 벤더 클라이언트** | `LlmPort` 인터페이스 뒤 벤더 SDK/HTTP 클라이언트 은닉(U5-S3). 프롬프트·모델·벤더는 C1 내부 — M8은 점수 스키마(poi_id→score)만 인지 | D11, component-dependency §2.2 |
| **기능별 티어 라우팅** | `scoreCandidates(tier=경량)` · `explainPlacement(tier=상위)` · (U7)`generate(회고, tier=상위)`. 티어→모델 매핑은 remote config | D11·G194, NFR-U5-COST-02 |
| **타임아웃(예산 < )** | 점수 2.5초(NFR-U5-PERF-02), 설명·회고는 상위 티어 별도 예산. 초과→폴백 트리거(§0.3 원칙) | D38, services.md §0.3 |
| **출력 스키마 검증** | JSON Schema 검증 + **closed-set ID 화이트리스트 교차 검증**(C1 출구). 위반 1회 재시도 후 규칙 점수 폴백 | G115, NFR-U5-SEC-02 |
| **서버 재조회 컨텍스트** | 호출자가 전달한 조회 콜백/참조 해석기로 재조회(D31) — C1→기능 모듈 역의존 방지, 타 계정·내부 지표 미주입 | D31, SECURITY-11 |
| **rate-limit·비용 계측** | 사용자별 rate-limiter + 호출별 티어·토큰·추정 비용 로그·메트릭 | SECURITY-11, NFR-U5-COST-03·05 |

- **대안 비교(1줄)**: (A) **자체 게이트웨이 추상화(권고)** — 벤더 교체·티어 라우팅·출력 검증·비용 계측을 한 곳에 격리(D11 정합). (B) 벤더 SDK 직접 호출 — 벤더 종속·출력 검증·rate-limit 분산으로 SECURITY-11·G115 강제가 약해짐(비채택). (C) LangChain류 오케스트레이션 프레임워크 — 단일 벤더·단일 호출 패턴(전 일자 1회)에는 과설계(G142, 추상화 이득 없음, 비채택).
- **선정 사유**: closed-set 그라운딩(G115)·서버 재조회 경계(D31)·비용 통제(전 일자 1회·티어·rate-limit)는 **LLM을 신뢰하지 않는 구조적 강제**가 핵심이며, 이는 게이트웨이 한 곳에 격리해야 검증·교체·계측이 가능하다. **P6 미확정 상태에서도 인터페이스·검증·라우팅은 확정 가능**(벤더 무관) — 이것이 P6 비차단의 근거다.

### 2.2 P6 경계 — 지금 확정 vs P6 후

| 지금 확정(벤더 무관) | P6 후 확정(벤더 의존) |
|---|---|
| `LlmPort` 인터페이스·티어 라우팅 구조·출력 스키마 검증·closed-set 교차 검증·rate-limit·비용 계측·전 일자 1회 호출 전략·후보 프루닝 | 벤더·모델(경량/상위 매핑)·단가·토큰 상한·후보 풀 크기 최종값·국외/국내 판별(G181 고지 요부) |

## 3. C2 Solver Engine — OPTW/TOPTW 구현 (U5-C2)

C2는 **하드 제약(영업시간·이동 부등식·고정 블록)을 만족하는 배치를 산출·검증**하는 CPU 집약 결정론 엔진이다. 외부 무의존 인프로세스 코드(component-dependency §2.3 "C2는 인프로세스 결정론 코드 — 외부 다운 없음"). 구현 방식을 비교한다.

| 기준 (U5 요구) | **A. 검증 솔버 라이브러리 + 도메인 모델 (권고)** | B. 자체 휴리스틱(그리디+로컬 서치) | C. 범용 MIP/CP 솔버(OR-Tools 등) |
|---|---|---|---|
| 하드 제약 표현(영업시간·이동 부등식·고정) | 제약·점수 DSL로 선언적 표현 | 코드로 직접 구현(검증 부담↑) | 강력하나 모델링·튜닝 비용 큼 |
| warm-start(고정 블록 보존 G46) | 지원(초기 해 고정) | 직접 구현 | 지원하나 복잡 |
| 결정론 폴백 모드(NFR-U5-RES-04) | 시드·타임박스 고정으로 결정성 확보 | **가장 단순한 결정성** | 결정성 보장 튜닝 필요 |
| 시간 예산(day1 0.8초·잔여 0.5초) | 타임박스 종료 후 best-so-far | 빠름(작은 문제) | 소규모엔 오버헤드 |
| 규모 정합(G142 · 후보 5천·동시 10) | **적합** — 문제 크기 소·JVM 내장 | 적합하나 품질 상한 | 과설계 위험 |
| JVM/Kotlin 정합 | **Timefold(구 OptaPlanner) — JVM 네이티브** | 순수 Kotlin | 네이티브 바인딩·별도 프로세스 |

- **확정 권고: A(Timefold/OptaPlanner류 JVM 제약 솔버 + 도메인 모델), 결정론 폴백은 A의 타임박스·시드 고정 모드로 실현.** 근거: (1) 하드 제약(영업시간·이동 부등식·고정 블록)의 **선언적 표현·검증**이 D37 하드 제약 테스트·PBT 1급 대상과 직결, (2) warm-start(G46)·타임박스(0.5~0.8초)·best-so-far 반환이 성능 예산·부분해 폴백(G161)과 정합, (3) JVM 네이티브라 U1 스택 내 신규 프로세스 0. **결정론 폴백 모드**(NFR-U5-RES-04)는 동일 엔진을 **시드·타임박스·규칙 점수 고정**으로 돌려 "동일 입력→동일 출력"을 보장(PBT (d)).
- **B 비채택 사유(1줄)**: 자체 휴리스틱은 초기 속도 이점이 있으나 하드 제약 정확성·warm-start·품질을 전부 자체 검증해야 해 D37 리스크가 크다 — 단, **결정론 폴백의 최소 배치**(숙소+시각 고정만)는 자체 그리디로 충분(A와 병존).
- **C 비채택 사유(1줄)**: 범용 MIP/CP는 문제 크기(후보 5천·동시 10)에 과설계이며 네이티브 바인딩·별도 프로세스가 SI 컴퓨트 모델(단일 컨테이너)과 불일치(G142).
- **오라클 검증 분리(NFR-U5-QUAL-01)**: 솔버가 산출한 배치는 **독립 검증기(오라클)** 로 하드 제약을 재확인한다 — 솔버 최적화 로직과 검증 로직을 분리해 "솔버 버그가 곧 제약 위반"이 되지 않게 한다(확정 시점 `validate` 재검증 G56은 U6 재사용).
- **클라 경량 검증기 규칙 공유(D28)**: 편집 재검증의 클라이언트 경량 검증기(`shared/validation`)와 서버 확정 검증(C2)은 **규칙 명세를 단일 원천에서 파생**(TypeScript·Kotlin 양쪽 생성/파생) — 통과→저장 시 거부 UX 불일치 차단(unit-of-work.md U5 리스크). 이중 실행 계약 테스트로 동치 검증.
- **라이브러리 등록**: 솔버 라이브러리·commons-math(거리·기하)는 U1 Gradle version catalog에 추가, 잠금 파일 커밋(SECURITY-10 상속).

## 4. 점진 응답 스트리밍 — SSE (U5-S1)

첫 1일 5초 게이트 후 잔여 일자를 백그라운드로 채우며 **단계 텍스트·진행률·경과 시간·일자 완료를 실시간 노출**해야 한다(US-E5-09, NFR-U5-PERF-04). 채널을 비교한다.

| 기준 | **A. SSE (권고)** | B. WebSocket | C. HTTP 폴링 |
|---|---|---|---|
| 방향성(서버→클라 단방향) | **정합** — 진행 스트림은 단방향 | 양방향(과잉 — 생성은 요청 후 수신만) | 단방향(요청 반복) |
| ALB 정합(SI §2.3) | HTTP 롱리브드 — **스티키니스 불요**, 유휴 60초 > 20초 예산 | ALB 지원하나 **스티키니스 필요**(U11 WebSocket 인프라 이연 전까지 미도입) | 표준 HTTP |
| 구현 복잡도 | 낮음(Spring `SseEmitter`/`Flux`) | 높음(핸드셰이크·세션 관리) | 낮으나 폴링 낭비 |
| 재연결·체감 | 표준 재연결(Last-Event-ID) | 수동 | 지연·부하 |
| 규모 정합(G142·동시 10) | **적합** — 생성당 1 스트림·20초 수명 | 과설계 | 폴링 부하 |

- **확정 권고: A(SSE) · 폴백 = 폴링.** 근거: (1) 진행 스트림은 **단방향**이라 WebSocket 양방향이 불필요, (2) **ALB 스티키니스 불요**(SI §2.3 — WebSocket 인프라는 U11 이연이므로 지금 도입 회피), (3) SSE 스트림 수명(≤20초)이 ALB 유휴 타임아웃 60초 내라 인프라 변경 0, (4) Spring `SseEmitter`로 U1 스택 내 구현. **폴백**: SSE 미지원 클라이언트·프록시 환경은 **생성 세션 상태 폴링**(GenerationSession 상태가 진행 정본, services.md §1.3)으로 대체.
- **B(WebSocket) 비채택 사유(1줄)**: 양방향·스티키니스가 불필요하고, WebSocket 인프라(ALB 스티키니스·다중 인스턴스 팬아웃)는 **U11로 이연된 결정**(SI §0 U9~U11)이라 U5에서 선도입은 규모 대비 과설계(G142).

## 5. 백그라운드 잔여 일자 잡 — 요청 단위 비동기 (U5-S2)

잔여 일자 생성은 **요청 단위 비동기 작업(워커 스레드, 모듈 경계 유지)** 이다(unit-of-work.md U5: "백그라운드 생성은 요청 단위 비동기 작업 — 워커 스레드, 스케줄러 잡 없음").

| 기준 | **A. 요청 단위 비동기 워커 (권고 — 가상 스레드/`@Async`)** | B. 별도 워커 서비스 + 큐(SQS 등) | C. ShedLock 스케줄 잡 |
|---|---|---|---|
| 수명 | 생성 요청 수명 내(≤20초) — SSE 스트림과 동일 프로세스 | 요청 분리·비동기 큐 | 주기 폴링(생성엔 부적합) |
| 신규 인프라(G142) | **0** — JDK 21 가상 스레드/Executor | +큐·워커 서비스(SI §4 브로커 미도입 정합 위반) | 스케줄러 프레임 있으나 용도 불일치 |
| 진행 스트림 연계 | **직접**(동일 프로세스 → SSE) | 크로스 프로세스 팬아웃 필요 | 부적합 |
| 규모 정합(동시 10) | 적합 — 태스크 내 CPU 벌크헤드(NFR-U5-SC-03) | 과설계 | — |

- **확정 권고: A(요청 단위 비동기 워커 — JDK 21 가상 스레드 또는 `@Async` Executor).** 근거: 생성은 **요청 수명 내 완결**(≤20초)이라 큐·별도 워커가 불필요하고(SI §4 "크로스-인스턴스 브로커 미도입" 정합), 진행 스트림(SSE)이 동일 프로세스에 있어야 직접 연계된다. **솔버 CPU 벌크헤드**(NFR-U5-SC-03)로 무거운 solve가 편집 재검증·CRUD를 지연시키지 않게 스레드 풀 분리.
- **B/C 비채택 사유(1줄)**: 큐·별도 워커는 SI §4(브로커 없음·DB 기반) 결정과 배치되고 동시 10 규모에 과설계(G142); ShedLock은 주기 잡용이라 요청 구동 생성에 부적합.
- **전환 트리거**: 동시 생성 실측이 태스크 CPU를 상시 포화시키거나 생성이 API p95를 열화시키면 솔버 워커 분리(EventBridge·큐) 재평가(SI §1.2 스케줄 잡 전환 트리거 상속).
