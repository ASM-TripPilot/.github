# PRD 요구사항 추출 결과 (requirements.md 작성용 중간 산출물)

## docs/PRD/01-앱셸.md

**요약**: 01-앱셸 문서는 TripPilot의 골격 화면을 정의한다: 스플래시(세션·약관 상태 확인 후 로그인/온보딩/홈으로 분기), 홈 대시보드(여행 카드·빠른 액션·인기 장소·취향 매칭 커뮤니티 기록·추억 다시 보기), 홈·탐색·일정·기록·마이 5탭 바텀 네비게이션, 몰입 화면에서의 탭바 숨김 규칙, 그리고 장소 우선 진입(Case A) 온램프로서 POI 검색·저장→여행 생성 시 필수 방문지 시드 투입 흐름이다. 여행 중에는 홈 활성 카드와 일정 탭이 단일 '여행 중 활성 일정 허브'로 수렴하며, 도착 체크·Plan-B·경로 토글 등이 맥락 액션으로 붙는다. 상세 정본은 각 섹션(03/04/05/06/07/09/10/12)에 위임하고 본 문서는 진입점·네비게이션 통합 규칙을 소유한다.

**기능 요구사항 (29건)**:
- 콜드 스타트 시 로고·서비스명 스플래시를 표시하고 저장된 세션 토큰과 약관 동의 버전을 확인해야 한다 (스토리 1)
- 세션 없음→로그인, 세션 있으나 약관 미동의/온보딩 미완료→해당 단계, 완료 사용자→홈 대시보드로 분기해야 한다 (스토리 1)
- 스플래시 화면에서는 바텀 탭바를 노출하지 않아야 한다 (스토리 1)
- 세션·약관 확인 네트워크 지연 시 진행 표시를 유지하고, 시간 초과 시 마지막 로컬 세션 상태로 분기하되 온라인 복구 시 재검증해야 한다 (스토리 1 예외, ADR-0011)
- 홈 대시보드는 진행 중/예정 여행 카드(목적지·기간·D-day·등록 숙소 수·일정 유무·진행률), '여행 만들기' 빠른 액션, '지금 인기 있는 장소'(POI), '지금 뜨는·내 취향 여행 기록', '여행 추억 다시 보기'를 한 화면에 표시해야 한다 (스토리 2)
- 여행 예정/진행중/종료 분류는 '알림/마이페이지/설정' 스토리 7의 정본을 따라야 한다 (스토리 2)
- 진행 중 여행('Plan-B' 스토리 1의 여행 중 정의)이 있으면 홈 상단에 그날의 활성 일정 진입 카드를 D-day·진행률과 함께 강조 노출해야 한다 (스토리 2)
- 숙소·여행이 없는 첫 사용자에게는 빈 대시보드 대신 '여행 만들기', '예약 없이 AI로 먼저 일정 받아보기', '가고 싶은 곳을 먼저 저장해 두기'(Case A 온램프) 진입을 노출해야 한다 (스토리 2)
- 홈 추천·커뮤니티 카드 로딩 중 로딩 상태를 표시하고, 일부 실패 시 침묵 실패 없이 가용 카드만 우선 표시해야 한다 (스토리 2, ADR-0011)
- 홈 상단에 서비스 워드마크와 알림 진입점(읽지 않은 알림 표시 포함)을 노출해야 한다 (스토리 2)
- 종료된 과거 여행이 있으면 '여행 추억 다시 보기' 진입점으로 여행 캘린더·회고에 연결해야 한다 (스토리 2)
- 취향+지금 인기 신호를 결합해 다른 이용자의 공개 여행 기록 카드(닉네임·스타일 요약·핵심 통계·인기/취향 일치 신호)를 노출하되 팔로우·구독·스크랩·평판 요소는 두지 않아야 한다 (스토리 2)
- 취향·기록 데이터 부족 시 추천 영역을 비우지 않고 취향 설정 안내·바로가기와 미설정 취향 점진 설정 카드를 노출해야 한다 (스토리 2 예외)
- 매칭되는 공개 일정이 없으면 빈 영역 대신 안내 문구와 탐색 랜딩→여행자 일정 진입점을 노출해야 한다 (스토리 2 예외)
- 화면 하단에 홈·탐색·일정·기록·마이 5개 고정 탭의 글로벌 바텀 탭바를 제공해야 한다 (스토리 3)
- 탐색 탭 진입 시 '무엇을 찾을까요?' 탐색 랜딩을 노출하고 숙소·장소·여행자 일정 3개 큰 카드로 각 상세 탐색에 진입시키며, 전용 커뮤니티 탭은 두지 않아야 한다 (스토리 3)
- 일정 탭은 진행 중 여행이 있으면 활성 일정 실행 화면, 없으면 최근 여행 일정으로 진입하고, 일정이 없으면 숙소 등록·여행 생성 유도를 노출해야 한다 (스토리 3)
- 여행 중에는 홈 활성 카드와 일정 탭이 같은 '여행 중 활성 일정 화면'(단일 허브)으로 수렴하고, 도착 체크·재계획·계획↔실제 경로 토글·다음 이동 길찾기가 맥락 액션으로 붙어야 한다 (스토리 3)
- 탭 전환 시 각 탭의 이전 화면 상태(스크롤 위치·하위 화면)를 보존하고 현재 선택 탭을 시각적으로 구분해야 한다 (스토리 3)
- 바텀 탭바는 단일 공용 컴포넌트로 구현해 모든 탭 루트 화면에서 동일 형태로 노출해야 한다 (스토리 3)
- 탭 루트가 데이터 0건이면 빈 탭 대신 각 섹션 정본의 빈 상태 안내를 노출해야 한다 (스토리 3 예외)
- 탭 루트와 직계 콘텐츠 화면(숙소 상세, 장소 검색·저장, 공유 일정 둘러보기/상세, 다른 사용자 프로필, 회고 등)에서는 탭바를 노출해야 한다 (스토리 4)
- 스플래시, 온보딩 전체, 입력 폼(숙소 검색·등록, 여행 생성, 필수 방문지 지정), 일정 생성 진행·편집, 여행 중 실행·Plan-B 플로우, 바텀시트/다이얼로그(권한 프리프롬프트·재계획 사유·위치 동의·제휴 고지·AI 어시스턴트)에서는 탭바를 숨기고 전체 폭 단일 CTA 또는 자체 하단 액션을 사용해야 한다 (스토리 4)
- 탭바 숨김 화면에서도 시스템 뒤로가기 또는 닫기/취소로 직전 탭 컨텍스트로 복귀할 수 있어야 한다 (스토리 4 예외)
- 탐색 랜딩 '장소' 진입에서 장소(명소·맛집·카페 등)를 검색해 저장/해제할 수 있고, 저장 POI는 이름·카테고리·지역과 함께 목록으로 표시해야 한다 (스토리 5)
- 저장 POI는 위시리스트('숙소 탐색' 스토리 4)·필수 방문지 지정('여행 생성' 스토리 8)의 입력으로 재사용되어 AI 일정 생성의 시드('꼭 갈 곳')로 투입되어야 한다 (스토리 5)
- '이 장소들로 여행 만들기' 진입을 제공해 저장 POI를 가진 채 바로 여행 생성으로 이어갈 수 있어야 한다 (스토리 5)
- 저장 POI가 0곳이면 빈 화면 대신 안내 문구와 장소 검색 진입점을 노출해야 한다 (스토리 5 예외)
- 저장 POI 집합으로부터 숙소 권역을 무게중심 기반으로 역추천하는 기능은 제공하지 않아야 한다 (스토리 5 비고)

**비기능 요구사항 (8건)**:
- 침묵 실패 금지(ADR-0011): 스플래시 세션 확인·홈 카드 로딩 등 외부 호출 실패 시 항상 진행 표시·부분 표시·폴백 분기로 대응해야 한다
- 오프라인 내성: 세션 검증 시간 초과 시 마지막 로컬 세션 상태로 분기하고 온라인 복구 시 재검증하는 로컬 우선 동작이 요구된다
- 홈 대시보드 카드들은 병렬 로딩·부분 실패 허용 구조여야 한다(일부 실패 시 가용 카드만 표시)
- 탭 전환 시 상태(스크롤·하위 스택) 보존 — 탭별 독립 네비게이션 스택과 상태 유지가 요구된다
- 바텀 탭바 단일 공용 컴포넌트 — UI 일관성과 재사용성 요구
- UI 레이아웃 가이드 §2 준수: 엄지 영역·전체 폭 CTA 원칙(탭바 숨김 화면)
- 저장 POI 표시 데이터는 국내 지도 API 약관(영구 캐싱 금지·실시간 호출·출처 표기)을 준수하는 데이터 모델이어야 한다(README 선결 과제의 암묵적 제약)
- 홈의 인기/취향 추천은 다른 이용자의 비공개 데이터를 노출하지 않는 개인정보 경계를 지켜야 한다(공개 기록만 카드화)

**문서가 미정으로 남긴 결정 (7건)**:
- 여행 예정/진행중/종료 분류 기준의 정본은 '알림/마이페이지/설정' 스토리 7에 위임 — 본 문서 단독으로는 미확정
- '여행 중' 상태 정의는 'Plan-B' 스토리 1 정본에 위임(등록 숙소 유무에 따라 체크인/아웃 구간 vs 여행 시작일·현위치 기준)
- 숙소 미등록 시 일정 생성 정책은 'AI 일정 생성' 스토리 1 정본에 위임
- 홈 커뮤니티 카드·공개 기록 노출의 가드레일 정본은 '여행자 커뮤니티' 섹션에 위임
- 지도·장소 API 선택(카카오맵/네이버/TMap)은 README 차원의 운영 결정으로 미확정 — 장소 검색·저장(스토리 5)의 구현 기반이 이에 종속
- 스플래시 세션 확인의 '시간 초과' 임계값은 문서가 수치를 정하지 않음
- '지금 인기 있는 장소'·'지금 뜨는' 인기 신호의 집계 방식(기간·표본·지역 범위)은 문서가 정의하지 않음

## docs/PRD/02-AI어시스턴트.md

**요약**: 앱 전반 AI 어시스턴트는 숙소 탐색·여행 생성·일정 생성·Plan-B·여행 중 실행 화면에서 FAB로 호출되는 대화형 오케스트레이션 계층으로, 자연어 요청을 해석해 기존 모듈(숙소 탐색·AI 일정 생성·솔버)을 호출·조립하고 결과를 설명하며 재질의를 받는다. 어시스턴트는 결정권자가 아닌 통역·중개자로, 실현가능성(영업시간·이동시간·시각) 판단과 일정 확정은 항상 솔버가 소유하며(ADR-0008·0015), 예약·결제는 외부 OTA로 위임한다(ADR-0003·0012). 총 8개 스토리로 호출/컨텍스트, 티키타카 재질의, 진행 검토, 액션 위임, 대화 이력/세션, 가드레일, 범위 한계, 폴백을 규정한다. 모든 응답에 다음 행동 1개를 동봉하는 dead-end 금지 원칙과, 타 사용자 비공개 데이터를 LLM 컨텍스트에 애초에 넣지 않는 서버측 권한 경계가 핵심 제약이다.

**기능 요구사항 (36건)**:
- [스토리 1] 숙소 탐색·여행 생성·AI 일정 생성·Plan-B·여행 중 현장 이용 등 주요 흐름에서 어시스턴트 진입점(우하단 FAB ✦)을 제공하고, 호출 시 현재 화면 컨텍스트(현재 여행·등록 숙소·일정 상태·진행 단계·직전 추천 결과)를 자동 전달해야 한다.
- [스토리 1] 어시스턴트는 현재 작업 위에 바텀시트/모달로 열리며, 이때 하단 네비게이션 탭바를 숨겨야 한다('앱 셸' 스토리 4와 일관).
- [스토리 1] 어시스턴트는 예시 프롬프트(quick reply)를 제시해 빈 입력창에서 막히지 않게 하고, 사용 중인 컨텍스트(예: '현재 여행: 부산 2박3일')를 사용자에게 표기해야 한다.
- [스토리 1] 모든 응답은 거절·외부 위임·폴백 상황을 포함해 항상 바로 누를 수 있는 다음 행동 1개를 동봉해야 하며(dead-end 금지), quick reply에 '내가 할 수 있는 것 보기'를 상시 노출해야 한다.
- [스토리 1 예외] 여행·일정 미생성 상태에서는 일반 도움 모드로 동작하고 일정 관련 요청에는 여행 만들기/숙소 등록 진입점을 안내해야 하며, 비로그인 상태에서는 일반 도움 범위로만 동작하고 계정 귀속 기능은 로그인 안내로 연결해야 한다.
- [스토리 1 예외] 어시스턴트 호출 자체가 네트워크/LLM 오류로 실패하면 오류와 재시도 진입점을 표시하고 기존 비대화형 동작으로 계속 진행할 수 있어야 한다(ADR-0011).
- [스토리 2] 사용자의 자연어 재질의를 해당 모듈의 검색/추천 조건(지역·예산·평점·취향·반경·이동 수단 등)으로 변환해 그 모듈을 재호출하고 갱신된 결과를 보여줘야 한다(LLM 해석 계층은 'AI 일정 생성' 스토리 2 재사용).
- [스토리 2] 직전 대화 맥락(이전 요청·결과)을 이어받아 누적 조건으로 좁혀 가고, 바뀐 조건을 명시적으로 요약 표시해야 한다.
- [스토리 2] 어시스턴트가 변환한 조건은 실제 모듈의 필터/입력값에 반영되어, 어시스턴트를 닫은 뒤에도 탐색/일정 화면이 그 조건으로 이어져야 한다(대화↔화면 상태 일관).
- [스토리 2] 숙소·장소 결과는 외부 데이터에 그라운딩된 실재 항목만 노출하고, 확인되지 않은 항목은 제시하지 않아야 한다.
- [스토리 2 예외] 결과가 0건이면 어떤 조건 조합이 0건을 만들었는지 설명하고 완화안을 제안해야 하며, 잘못된 해석 지적 시 직전 변환을 되돌리고 조건 직접 선택/재발화를 유도해야 한다.
- [스토리 3] '지금까지 한 거 검토해줘' 요청에 대해 현재 여행 상태(등록 숙소·기간·고정/필수 블록·일정 유무·미해결 충돌)를 요약하고 다음 할 일을 짚어야 한다.
- [스토리 3] 지적하는 일정 문제(이동시간 부족·영업시간 외·고정 블록 충돌)는 솔버가 이미 검증·표기한 위반 정보만 가져와 설명하며, 검증되지 않은 시각·이동시간을 새로 만들지 않아야 한다(ADR-0008·0015).
- [스토리 3] 진단에 그치지 않고 각 문제의 해결 동선(숙소 등록·장소 이동·AI 재추천 등)을 실제 화면/모듈 진입점으로 연결해야 한다.
- [스토리 3] 제시하는 모든 시각·이동 거리·체류 시간 수치는 솔버 소유·검증 값이어야 하며(이동 소요시간은 표시하지 않음 — ADR-0009), LLM 임의 수치를 노출하지 않아야 한다.
- [스토리 3 예외] 검토 대상 일정이 없으면 '아직 검토할 일정이 없어요'와 진입점을 안내하고, 검증값을 가져올 수 없으면 추측 단정 없이 '일부 정보 확인 불가'를 표시하고 가용 범위만 요약해야 한다.
- [스토리 4] 어시스턴트의 일정 변경 제안(추가·교체·삭제·재정렬·시간 조정)은 사용자가 적용을 선택하는 순간 'AI 일정 생성' 스토리 7 또는 Plan-B 재계산 흐름에 넘겨 솔버 재검증 후에만 일정에 반영되어야 한다.
- [스토리 4] 제안이 솔버 재검증에서 제약을 위반하면 조용히 적용하지 않고 위반 사유를 표시한 뒤 대안(다른 시간대·다른 날·장소 제외)을 제시해야 한다.
- [스토리 4] 어시스턴트는 사용자의 명시적 확인 없이 일정을 자동 확정·저장하지 않으며, 변경 전후 비교를 보여준 뒤 '적용' 선택 시에만 반영해야 한다.
- [스토리 4 예외] 제안 변경을 어느 시간대에도 넣을 수 없으면 불가 사유와 함께 '다른 날짜 이동/고정 해제/인접 일정 조정' 선택지를 제시하고, 솔버 재검증 불가(외부 API 실패) 시 자동 적용 없이 수동 편집 모드로 폴백하며 미검증 상태를 표기해야 한다.
- [스토리 5] 한 작업 세션 동안 대화 맥락·누적 조건을 유지하고, 어시스턴트를 닫았다 같은 화면에서 다시 열면 직전 대화 내역을 이어서 보여줘야 한다.
- [스토리 5] 대화 이력은 로그인 계정에 귀속되고, 비로그인 대화는 세션 내 임시 유지 후 로그인 시 이어붙임/폐기 여부를 안내해야 한다.
- [스토리 5] 어시스턴트가 어떤 모듈을 호출했고 무엇을 바꿨는지 대화 내역에 남기고, 대화에서 비롯된 실제 변경은 여행/일정의 change log(ADR-0013)와 일관되게 추적하되 대화 자체를 확정 데이터로 취급하지 않아야 한다.
- [스토리 5 예외] 다른 여행/작업으로 전환하면 새 컨텍스트로 시작하되 이전 대화로 돌아갈 수 있어야 하며, 이력 동기화 실패 시 임시 보관 상태를 표시하고 재연결 시 동기화해야 한다.
- [스토리 6] 역할·범위 변경 요청(시스템 지침 무시·페르소나 강요·내부 프롬프트 유출 요구)에 응하지 않고 가능한 도움 범위로 되돌려 안내해야 한다.
- [스토리 6] 유해·차별·욕설 등 정책 위반 요청은 거절하고 사유를 간단히 안내해야 한다.
- [스토리 6] LLM 호출에 빈도 제한(rate-limit)을 두고 과도한 연속 요청 시 잠시 후 재시도 안내를 표시해야 한다.
- [스토리 6] 타 사용자 비공개 데이터·내부 운영 지표(제휴 수수료 등)의 비노출은 LLM 프롬프트 거절이 아니라 컨텍스트 주입 단계의 서버측 권한 경계(애초에 LLM 입력에 미포함)로 구조적으로 강제해야 한다.
- [스토리 6 예외] 정책 위반 요청이 반복되면 같은 거절을 되풀이하지 않고 사용 안내(가능한 도움 범위)로 전환해야 한다.
- [스토리 7] 예약·결제·외부 실거래 요청은 직접 처리하지 않고 외부 OTA 딥링크로 위임하며 '예약·결제는 외부 사이트에서 진행돼요'를 안내해야 한다(ADR-0003·0012).
- [스토리 7] '이 시간에 갈 수 있어?'류 실현가능성 질문에는 솔버 검증 결과로 답하거나 검증 흐름으로 연결해야 하며, 턴바이턴 내비·실시간 가격 보증 등 미제공 기능을 약속하지 않고 외부 지도앱/외부 OTA로 연결해야 한다.
- [스토리 8] LLM 해석/응답 실패 시 '지금 도우미가 응답하지 못했어요'와 재시도를 표시하고 어시스턴트 없이 기존 비대화형 동작으로 계속 진행할 수 있어야 한다(핵심 흐름은 어시스턴트 없이도 완결).
- [스토리 8] 의도가 모호하면 임의로 단정·실행하지 않고 되묻거나 선택지를 제시해야 한다.
- [스토리 8] 솔버 비가용 시 시각·실현가능성을 추측해 채우지 않고 '지금은 시간·동선 검증을 할 수 없어요'를 표시한 뒤 결정론적 솔버 폴백('AI 일정 생성' 스토리 9) 또는 수동 편집 폴백으로 연결해야 한다.
- [스토리 8] 응답이 일정 시간 내 오지 않으면 진행 표시를 유지하고 사용자가 대기를 취소해 기존 흐름으로 돌아갈 수 있어야 한다.
- [스토리 8 예외] 어시스턴트 문구와 솔버 검증 결과가 어긋나면 항상 솔버 검증 결과를 기준으로 표시해야 한다(검증값 우선).

**비기능 요구사항 (9건)**:
- 침묵 실패 금지(ADR-0011): 모든 실패(호출 실패·LLM 실패·솔버 비가용·이력 동기화 실패)는 사용자 관찰 가능하게 표시하고 수동 폴백을 제공해야 한다.
- 가용성/독립성: 어시스턴트는 보조 계층이므로 앱의 핵심 흐름(탐색·일정 편집)은 어시스턴트 없이도 완결되어야 한다.
- 보안: 프롬프트 주입 방어는 서버측 컨텍스트 필터링(권한 경계)으로 구조적으로 강제하며, 타 사용자 비공개 데이터·내부 운영 지표가 LLM 입력에 포함되지 않아야 한다.
- 정확성/신뢰성: 사용자에게 노출되는 모든 시각·거리·체류시간은 솔버 검증값이어야 하며 LLM 환각(비그라운딩 장소·임의 수치)은 구조적으로 차단되어야 한다(ADR-0008·0009·0015).
- 비용/남용 방지: LLM 호출 rate-limit(사용자별 빈도 상한·과도 연속 호출 차단)을 두어야 한다(README 선결 과제와 공유).
- 응답성: 응답 지연 시 진행 표시 유지와 대기 취소를 지원해야 한다(구체 타임아웃 값은 미정).
- UI 일관성: 모달/바텀시트 표시 중 탭바 숨김, FAB가 탭바와 겹치지 않는 배치 등 앱 셸 규칙과 일관해야 한다.
- 감사 가능성: 어시스턴트발 변경은 change log(ADR-0013)와 일관되게 추적 가능해야 한다.
- 데이터 지속성: 대화 이력은 계정 귀속 저장, 비로그인은 세션 임시 저장, 동기화 실패 시 임시 보관 후 재동기화.

**문서가 미정으로 남긴 결정 (5건)**:
- LLM 호출 비용·rate-limit의 구체 수치(사용자별 빈도 상한·과도 연속 호출 차단 임계) — README 'AI 어시스턴트 운영' 선결 과제에서 운영 결정으로 남김.
- 가드레일 임계(프롬프트 주입·유해 요청 거절 판정 기준의 민감도) — 운영 결정.
- 대화 이력 보관·정리 정책(보관 기간·용량·삭제 규칙) — 운영 기준으로 정해야 함.
- 어시스턴트가 사용할 LLM 모델/제공자 선택(비용·캐싱 정책 포함) — README 외부 API 비용·캐싱 선결 과제와 공유.
- 응답 타임아웃('일정 시간 내')의 구체 값 — 문서에 미정.

## docs/PRD/03-온보딩.md

**요약**: 계정 및 온보딩 문서는 소셜(Google·Apple·카카오·네이버) 4종과 이메일+비밀번호 기반의 회원가입/로그인, 최초 1회 약관 동의, 자동 생성 닉네임, just-in-time 위치 권한 고지, 그리고 7종 취향(여행 스타일·페이스·예산·동행·선호 활동·이동 방식·음식) 온보딩을 15개 유저스토리로 정의한다. 모든 취향 단계는 건너뛰기 가능하며 '나중에 설정하고 시작' 일괄 탈출구를 제공하고, 미설정 항목은 스토리 14의 중립 기본값(이동 수단은 대중교통 보수 추정, 예산은 필터 미적용)으로 동작해 일정 생성이 실패하지 않도록 한다. 동일 이메일 소셜/이메일 계정 충돌은 기존 로그인 수단 유도로 처리하되, 이메일 대조 불가 시 별도 계정 생성 후 수동 연결을 허용한다. 수집된 취향은 숙소 탐색 필터, AI 일정 생성, 장소 추천, 여행 스타일 분석에 활용되며 설정 화면에서 언제든 수정 가능하고 즉시 반영된다.

**기능 요구사항 (31건)**:
- [스토리 1] 소셜 4종(Google·Apple·카카오·네이버) 또는 이메일+비밀번호로 회원가입/로그인할 수 있으며, 소셜은 제공자 고유 ID(sub)로 계정을 식별해 신규 생성 또는 기존 로그인 처리한다.
- [스토리 1] 로그인/회원가입 화면에 4개 소셜 버튼과 인증 수단 비종속 라벨 '회원가입' 진입을 노출하고, '이미 계정이 있으신가요? 로그인' 링크로 화면 전환을 제공한다.
- [스토리 1] 이메일 가입 시 인증 메일(링크 또는 코드)을 발송하고 인증 완료 전에는 계정을 '미인증' 상태로 두며, 재발송 동작을 제공하되 미인증 상태로는 약관 동의 이후 온보딩 단계로 진행시키지 않는다.
- [스토리 1] 이메일 로그인 계정에 '비밀번호 재설정'(등록 이메일로 재설정 링크 발송) 경로를 제공한다.
- [스토리 1] 모든 계정은 '여행자' 단일 유형으로 생성하며 호스트/숙박업주 등 다른 계정 유형 선택지를 제공하지 않는다.
- [스토리 1 예외] 소셜 인증 거부/취소 시 로그인 화면 복귀 및 '로그인이 취소되었습니다' 안내, 제공자 서버 오류 시 재시도 버튼을 표시한다.
- [스토리 1 예외] 이미 가입된 이메일 재가입 시도 시 중복 계정을 생성하지 않고 로그인/비밀번호 재설정으로 유도한다.
- [스토리 1 예외] 동일 이메일이 소셜/이메일 양쪽에 확인되면 기존 로그인 수단으로 유도하고, 이메일 대조 불가(예: Apple 비공개 릴레이) 시 별도 계정으로 생성한 뒤 사용자가 나중에 수동 연결할 수 있게 한다.
- [스토리 2] 최초 로그인 직후 온보딩 진입 전에 약관 동의 화면을 1회 표시하고, 필수 약관(이용약관·개인정보 처리방침) 전체 동의 시에만 '다음'을 활성화하며, 마케팅 수신 동의는 선택 항목으로 둔다.
- [스토리 2] 동의 완료 시 동의 항목·동의 일시·약관 버전을 계정에 저장한다.
- [스토리 3] 회원가입 시 중복되지 않는 닉네임을 자동 생성해 기본값으로 채우고, 자동 생성값만으로 온보딩을 통과시키며 '나중에 설정에서 변경 가능'을 명시한다.
- [스토리 3] 닉네임은 2~20자 제한이며 길이 위반·금칙어·중복 시 인라인 오류로 진행을 막고, 중복 시 사용 가능한 대체 닉네임을 원탭 적용으로 추천한다.
- [스토리 4] 온보딩에서 OS 위치 권한 다이얼로그를 바로 호출하지 않고, 위치가 실제로 쓰이는 첫 맥락 직전에 just-in-time 프리프롬프트(사용 목적 고지)를 1회 보인 뒤 OS 다이얼로그를 호출한다.
- [스토리 4 예외] 위치 권한 거부 시에도 등록된 숙소 위치 기준으로 동작하며 '설정에서 켤 수 있음'을 안내하고, 권한 거부만으로 온보딩을 중단하지 않는다.
- [스토리 5] 여행 스타일 7종(휴양·관광·액티비티·미식·쇼핑·자연·문화예술)을 복수 선택으로 설정할 수 있다.
- [스토리 6] 예산은 전체 여행 예산(1인·항공 제외·숙소+식비+활동+교통 합산) 기준 4단계 구간(저가/중간/고급/럭셔리) 선택 또는 직접 총액 입력으로 설정하며, 온보딩 값은 러프 선호 값으로 저장하고 정확 총액은 여행 생성 시 입력한다.
- [스토리 6] 예산 값은 (a) 1박 숙소 가격대로 환산해 숙소 탐색 가격 필터 기본값에, (b) 1인 1일 지출 수준으로 일정 생성 비용 가중치에 반영한다.
- [스토리 6] 온보딩 예산 화면 카피는 '대략적인 여행 씀씀이'로 부담을 낮추고, 정확 정의(항공 제외 등)는 직접 총액 입력 시에만 노출한다.
- [스토리 7] 동행 유형(혼자·커플·친구·가족(아동 동반)·부모님·반려동물 동반)을 설정할 수 있다.
- [스토리 8] 선호 활동 9종(자연·역사문화·테마파크·맛집투어·카페·전시·야경·쇼핑·스포츠)을 복수 선택하며 장소 추천 우선순위에 반영한다.
- [스토리 9] 이동 방식(도보·대중교통·렌터카·택시·자전거)을 복수 선택하며 일정 생성의 기본 이동 수단·이동 시간 계산에 반영한다.
- [스토리 10] 음식 취향(한식·양식·일식·중식·아시안 등)을 복수 선택하며 맛집·음식 장소 추천에 반영한다.
- [스토리 15] 여행 페이스(느긋하게 1~2곳/균형있게 3~4곳/빡빡하게 5곳 이상)를 단일 선택하며 AI 일정 생성의 일정 밀도·예상 체류 시간 기본값에 반영한다.
- [스토리 11] 모든 취향 단계(5~10·15)에 '건너뛰기'를 제공하고, 각 단계에 진행률 표시와 '이전' 버튼을 제공해 응답 수정 후 재진행할 수 있게 한다.
- [스토리 11] 취향 설정 단계 어디에서나 '나중에 설정하고 시작' 일괄 탈출구를 제공해 남은 취향 단계를 한 번에 건너뛰고 홈으로 진입시킨다(약관 동의·닉네임은 대상 제외).
- [스토리 11] 건너뛴 취향은 홈 대시보드와 첫 여행 생성 직전의 점진 설정 카드로 미설정 항목을 한두 개씩 재권유한다.
- [스토리 11] 온보딩 완료 또는 일괄 탈출 시 곧바로 홈 대시보드로 진입시키고 빈 홈에서 첫 행동(여행 만들기·숙소 탐색·예약 없이 일정 받아보기)을 강조 노출한다.
- [스토리 12] 설정 화면에서 닉네임과 7종 취향 전체를 온보딩과 동일한 선택지로 언제든 수정·해제·신규 설정할 수 있으며, 저장 즉시 숙소 탐색·일정 생성·장소 추천에 반영한다(닉네임은 스토리 3 규칙 동일 적용).
- [스토리 13] 수집한 취향은 숙소 탐색(예산·동행 유형 정렬/필터), 일정 생성(스타일·활동·이동·음식 입력값), 장소 추천(활동·음식 우선순위), 여행 스타일 분석에 활용한다.
- [스토리 14] 미설정 취향은 가중치 없는 '중립' 기본값으로 처리해 일정 생성이 실패하지 않게 하고, 이동 방식 미설정 시 기본 이동 수단은 대중교통(안전계수 적용 보수 추정), 예산 미설정 시 가격 필터 미적용 전체 노출로 동작한다.
- [스토리 14] 취향 다수 미설정으로 추천 근거가 부족하면 일정 생성 결과 상단에 설정 유도 안내와 설정 화면 바로가기를 표시한다.

**비기능 요구사항 (8건)**:
- 비밀번호는 평문으로 저장하지 않고 해시로 저장해야 하며, 길이·복잡도 기준을 충족해야 한다(구체 기준은 미정의).
- 계정 식별은 소셜 제공자 고유 ID(sub) 또는 이메일 계정 식별자를 기준으로 하여 중복 계정 생성을 방지해야 한다.
- 약관 동의 증적(동의 항목·일시·약관 버전)을 계정에 보존해야 한다(컴플라이언스 감사 대응).
- 위치 권한은 위치정보법(ADR-0010/0017 계승)에 따라 사용 목적 고지·동의·거부 허용을 준수하고, OS 다이얼로그는 실제 사용 맥락 직전 just-in-time으로만 호출해야 한다(권한 수락률·규제 준수).
- 온보딩은 첫 가치 도달 시간을 최소화해야 한다: 필수 단계는 약관·닉네임 유효성뿐이며, 나머지는 건너뛰기/일괄 탈출구로 마찰을 제거한다(UX 성능 요구).
- 취향 수정은 저장 즉시 이후의 탐색·생성·추천에 반영되어야 한다(일관성/반영 지연 없음).
- 미설정 취향만으로 일정 생성이 실패해서는 안 된다(견고성 — 침묵 실패 금지 원칙 ADR-0011과 일관).
- 닉네임 금칙어 검증이 필요하므로 금칙어 사전(README: 닉네임·일정 제목·캡션·댓글 공통)이 가입 시점부터 동작해야 한다.

**문서가 미정으로 남긴 결정 (7건)**:
- 예산 4구간의 금액 경계(저가 50만 미만 등)는 초기 국내 지역 기준 '운영 기본값'이며 지역·시즌에 따라 조정 가능하다고 명시 — 조정 메커니즘은 운영 결정.
- 동행 유형의 단일/복수 선택 여부를 '항목 정의에 따른다'로 문서 스스로 미정으로 남김.
- 온보딩 단계에서 위치 관련 고지를 '두거나 생략한다'로 두 가지 가능성을 열어 둠.
- 음식 취향 선택지(한식·양식·일식·중식·아시안 '등')와 여행 페이스 선택지('등 일정 밀도 수준')의 최종 목록이 예시 수준으로만 제시됨.
- 이메일 인증 방식이 '인증 링크 또는 인증 코드'로 양자택일 미확정.
- 비밀번호 길이·복잡도의 구체 기준 미명시.
- 닉네임 자동 생성 규칙(형식·중복 회피 알고리즘) 미명시.

## docs/PRD/04-숙소탐색.md

**요약**: 숙소 탐색/저장/외부 예약 연동 문서는 "탐색·저장은 앱, 상세 가격·예약·결제는 외부 OTA 딥링크"라는 teaser+딥링크 모델을 정본으로 삼는다. 검색은 여행지만 입력받고(날짜·인원 미입력), 목록은 정적 대표 가격대를 표시하며 정확 가격은 사용자가 명시적으로 '가격 보기'/딥링크로 나갈 때만 라이브 호출한다(가격 캐싱 금지). 숙소 유형 필터는 전면 제공, 편의시설 필터는 데이터 채움률에 따라 조건부 확대하고, 리뷰·평점은 앱 내부에 표시하지 않고 외부 OTA로 위임한다. 숙소 '등록'(위치+체크인/아웃+인원)이 AI 일정 생성의 출발점이 되며, 등록 경로는 제휴 링크 예약 확인 시 1탭 자동 후보 제시와 수동 입력(지도 장소 검색·URL 붙여넣기·지도 핀 직접 지정) 두 가지다. 다도시 여행을 위한 일자별 복수 거점 등록(날짜 구간 비중첩 검증)과 부분 장애 시 침묵 실패 금지·폴백 흐름을 규정한다.

**기능 요구사항 (20건)**:
- [S1] 여행지(또는 '내 주변')만 입력받아 숙소를 탐색하며, 탐색 단계에서 체크인/체크아웃 날짜·인원은 입력받지 않고 여행지가 비면 탐색 버튼이 비활성화된다.
- [S1] 숙소 데이터 소스는 글로벌 OTA 인벤토리(Booking·Agoda·Expedia 등)를 기본으로 하고, 국내 숙소는 글로벌 OTA 한국 인벤토리 + 지도/장소 데이터(Google Places·공공 TourAPI)로 보강하며, 국내 OTA는 딥링크 연결만 하고 데이터 크롤링을 하지 않는다.
- [S1] 검색 결과 카드는 숙소명·대표 가격대(정적 가격 구간·등급)·위치를 표기하고, 정확한 1박 가격은 사용자가 '가격 보기' 또는 딥링크 이동 시에만 라이브 호출하며 목록 전체 일괄 라이브 호출을 금지한다.
- [S1] 검색 결과가 N건 이상이면 페이지네이션 또는 무한스크롤로 추가 결과를 노출하고, 위치 권한 거부 상태의 '내 주변' 검색은 등록 숙소 또는 여행지 중심 좌표로 대체 조회한다.
- [S2] 검색 결과를 가격대(구간 기준)·숙소 유형(전면 제공, 한국형 유형은 TourAPI 정본, 소스별 코드는 Place Data 모듈에서 표준화)·편의시설(TourAPI 채움률 검증 후 노출, 미확보 시 핵심 3종 주차·조식·와이파이 폴백)·거점 후보까지의 거리로 필터링·정렬할 수 있으며, 가격순 정렬은 제공하지 않고 '대표 가격대순'으로 대체한다.
- [S2] 필터 적용 결과가 0건이면 어떤 필터가 0건을 만들었는지 표기하고 해당 필터의 완화를 제안한다.
- [S3] 숙소 상세 화면은 위치(지도 핀)·사진·가격대·편의시설·체크인/체크아웃 시각 등 정적 콘텐츠 위주로 구성하고, 리뷰·평점은 내부 표시 없이 '외부 OTA에서 확인' 딥링크로 위임하며, 누락 데이터는 빈칸 대신 '미확인'으로 표기한다.
- [S4] 숙소를 위시리스트에 저장할 수 있고, 저장은 로그인 계정에 귀속되어 기기 변경에도 유지되며, 항목별 카드(숙소명·대표 사진·가격대·위치) 표시·삭제·메모 추가가 가능하고, 가격은 '외부 OTA 기준, 변동 가능'으로 표기한다.
- [S5] 숙소 상세·저장 화면에서 [외부 OTA에서 예약하기]로 OTA 딥링크(외부 브라우저/인앱 웹뷰)에 이동하되, 이동 직전 제휴(어필리에이트) 수수료 고지를 표시하고, 동일 숙소에 복수 OTA가 연결되면 OTA명·표시 가격을 함께 보여주고 사용자가 선택하게 한다.
- [S5] 사용자가 외부 OTA 이탈 후 앱으로 포그라운드 복귀하면 '방금 본 [숙소명], 예약하셨나요? → 거점으로 등록하기' 복귀 핸드오프 카드를 노출하되 무시하고 계속 탐색할 수 있어야 하며, OTA 링크 장애(404·만료·무응답) 시 안내와 함께 다른 OTA 링크 또는 장소 검색 우회 경로를 제시한다.
- [S6] 외부에서 예약한 숙소를 앱에 '등록'할 수 있고(필수: 숙소 장소·체크인/아웃 날짜·인원, 선택: OTA 예약번호·결제 금액), 등록된 숙소 좌표와 날짜가 AI 일정 생성의 기준점(거점·기간)이 된다.
- [S6] 등록 경로는 (a) 앱 제휴 링크 예약이 포스트백으로 확인되면 자동 등록 후보 제시 후 1탭 등록, (b) 그 외는 위치·날짜 수동 입력이며, 예약번호·메일 연동 자동 인식은 제공하지 않는다.
- [S6] 검색 결과·저장 목록에서 [예약했어요/등록]으로 곧바로 등록을 시작하면 숙소명·위치·가격대가 자동 채워지고, 등록 완료 시 [AI 일정 생성하기] 진입점을 즉시 노출하며, 등록 후 날짜·인원·예약번호·금액 수정과 등록 취소가 가능하다.
- [S6] 체크아웃이 체크인보다 빠르거나 날짜가 비는 등 불완전 등록은 저장을 막고 누락 항목을 인라인 표시하며, 좌표 미확정 등록은 '지도에서 위치 확인' 단계를 요구한다.
- [S7] 일자별로 거점 숙소를 다르게 등록할 수 있으며(다도시 여행), 복수 숙소의 날짜 구간이 겹치지 않게 검증하고 겹치면 '같은 날짜에 거점이 둘이에요'를 표시해 구간 조정을 요구한다.
- [S8] OTA 검색 결과와 독립적으로 지도 장소 검색(숙소명·주소)·OTA 예약 링크 붙여넣기(가능 범위 내 숙소명·위치 추출 후 폼 미리 채움)·지도 핀 직접 지정으로 숙소를 직접 등록할 수 있어야 하며, 장소 선택 시 숙소명·주소·좌표가 자동 채워져 날짜·인원만 입력하면 등록이 완료된다.
- [S8] 동명·유사 숙소 다건 반환 시 후보 목록과 지도 핀을 함께 보여 사용자가 한 곳을 선택해 좌표를 확정하게 하고, 링크 추출 실패 시 수동 핀 지정으로 안내하며, 지도/장소 API 무응답 시 침묵 실패 없이 지도 핀 직접 지정으로 폴백한다(ADR-0011).
- [S9] 등록·저장 숙소 목록에서 숙소명·위치·체크인/아웃·인원·(입력 시) OTA명·예약번호·금액·거점 사용 여부를 확인할 수 있으며, 앱은 예약 진행/확정 상태를 추적하지 않고 사용자 입력 기록과 거점 지정 상태만 보여주고, 선택 항목 미입력 시 '미입력'으로 표시한다.
- [S10] 검색 조건에 맞는 숙소가 없으면 빈 화면 대신 '조건에 맞는 숙소가 없어요'와 함께 날짜·지역·필터 완화 등 다음 행동을 제안하고, OTA 제휴 미지원 지역은 수동 등록(스토리 8) 경로로 연결한다.
- [S11] OTA 목록 수집 중 '숙소를 모으는 중' 로딩 상태를 표시하고, 일부 OTA 장애 시 '일부 가격이 빠졌을 수 있어요' 표기와 함께 가능한 결과를 즉시 보여주며, 전체 실패 시 재시도 버튼과 수동 등록 우회 경로를 제공한다.

**비기능 요구사항 (12건)**:
- 정확 가격·재고는 캐싱 금지, 표시 시점 라이브 호출만 허용하며 결과 목록 일괄 라이브 호출 금지(OTA 약관·비용·지연 대응).
- 정적 콘텐츠(이름·위치·설명·시설·사진 URL·체크인아웃)는 OTA 콘텐츠 API 갱신주기를 준수하는 범위 내에서 캐싱 적재 가능.
- 좌표는 지도 API 약관 허용 범위 내 캐싱하되, 국내 지도 API(카카오·네이버·TMap)는 영구 캐싱 금지·실시간 호출·출처 표기 약관을 준수해야 한다.
- place_id·OTA 숙소 ID 같은 식별자는 영구 보존 가능.
- 국내 OTA 데이터를 크롤링하지 않는다(부정경쟁 민사 리스크 회피) — 딥링크 연결만 허용.
- 외부 OTA 리뷰·평점을 앱 내부에 표시하지 않는다(원문·점수 모두) — 법적/약관 준수.
- 편의시설 표시는 가격 비교 화면이 아닌 곳에 배치한다(Booking 약관).
- 침묵 실패 금지(ADR-0011): 부분 공급자 장애·API 타임아웃 시 사용자 고지 + 폴백 경로(수동 등록·핀 지정) 제공.
- 제휴 링크 이동 전 어필리에이트 수수료 고지 의무 표시(광고·제휴 고지 규제 준수).
- 위시리스트는 계정 귀속으로 기기 간 동기화되어야 한다(서버 저장 필수).
- 편의시설 필터는 필드 채움률을 검증해 실제로 채워지는 항목만 노출한다(데이터 품질 게이트).
- 앱은 자체 예약·객실 선택·인앱 결제를 제공하지 않으며 모든 결제 흐름을 외부 OTA로 위임한다(범위 제약).

**문서가 미정으로 남긴 결정 (7건)**:
- 외부 OTA 제휴 파트너 선정(Booking/Agoda/Expedia/야놀자/여기어때 등) — README에서 운영 결정으로 명시.
- OTA 딥링크 연결 방식과 어필리에이트 수수료 트래킹·정산 구조(포스트백 제공 여부 포함) — 운영 결정.
- 지도·장소 API 벤더 선택(국내 지도 API 우선 원칙만 확정, 카카오맵/네이버/TMap 중 미정; 스토리 8은 예시로 Google Places 언급).
- 정적 콘텐츠 캐싱의 구체적 갱신주기 값(문서는 '갱신주기 준수'만 명시).
- 페이지네이션/무한스크롤 임계값 N과 UI 방식 선택.
- 글로벌 숙소 편의시설 필터의 풀 Content API 확대 시점(파트너 계약 체결 조건부).
- TourAPI 편의시설 필드 채움률 검증 기준값(어느 채움률 이상이면 노출할지).

## docs/PRD/05-여행생성.md

**요약**: 여행 생성 문서는 여행지·날짜(필수)와 인원·예산(선택)으로 새 여행을 만드는 흐름을 정의하며, 숙소 미등록 상태의 여행 생성도 1급 온램프로 지원한다. 외부 OTA 예약 숙소 또는 앱에 저장한 숙소를 여행에 '등록'하면 위치+체크인/아웃이 일정 생성의 기준 거점이 되고, 다중 숙소(도시 이동)·다박(연속 숙박)·공백/겹침 구간의 스마트 기본 거점 규칙을 다룬다. 또한 AI 일정 생성 전 '필수 방문지'를 '아무 때나 꼭 가기'(포함 고정형) 기본 + '시간 정해두기'(시각 고정형) 선택으로 지정해 일정의 앵커로 투입하고, 고정/필수 블록이 재생성 후에도 유지되도록 규정한다. 숙소 미등록 시 일정 생성 정책과 필수 방문지 충돌 처리는 'AI 일정 생성' 섹션을, 등록 폼 화면과 OTA 이동·수수료 고지는 '숙소 탐색' 섹션을 정본으로 위임한다.

**기능 요구사항 (26건)**:
- FR-1 (스토리 1): 사용자는 여행지, 날짜 범위(시작일·종료일), 인원, 예산을 입력해 새 여행을 생성할 수 있으며, 여행지·날짜는 필수이고 둘 중 하나라도 비면 '여행 생성' 버튼이 비활성화되어야 한다.
- FR-2 (스토리 1): 인원·예산은 선택 입력이며 미입력 시 인원 1명, 예산 미설정 상태로 생성되고, 예산은 항공 제외 전체 여행 총액(숙소+식비+활동+교통)으로서 온보딩 러프 예산 단계가 있으면 기본값으로 채우되 사용자가 수정할 수 있어야 한다.
- FR-3 (스토리 1): 입력된 전체 여행 예산은 여행 일수·인원으로 분배되어 숙소 가격 필터와 일정 비용 가중치('AI 일정 생성' 스토리 2)에 반영되어야 한다.
- FR-4 (스토리 1): 종료일이 시작일보다 빠르면 '종료일은 시작일 이후여야 합니다' 오류를 표시하고 생성을 막아야 한다.
- FR-5 (스토리 1): 등록된 숙소가 있으면 그 체크인/아웃을 여행 날짜로 채우는 '등록 숙소에서 날짜 가져오기' 단일 버튼을 제공하고, 없으면 '숙소 등록하기 / 날짜 직접 입력' 대안을 노출해야 한다.
- FR-6 (스토리 1): 온보딩에서 건너뛴 취향이 있으면 첫 여행 생성 흐름에서 미설정 취향을 한두 개씩 다시 권유하는 점진 설정 카드를 노출하되, 사용자는 건너뛸 수 있고 미설정 항목은 중립 기본값('계정 및 온보딩' 스토리 14)으로 동작해야 한다.
- FR-7 (스토리 2): 사용자는 숙소를 등록하지 않은 상태에서도 여행지와 날짜만으로 여행을 먼저 생성할 수 있으며, 해당 여행은 '숙소 미등록' 상태로 표시되고 '숙소를 등록하면 그 위치를 기준으로 일정을 만들 수 있습니다' 안내가 표시되어야 한다.
- FR-8 (스토리 2): 사용자는 여행 생성 후 언제든 숙소를 추가 등록할 수 있어야 한다.
- FR-9 (스토리 2): 숙소 미등록 상태로도 여행지 중심으로 일정을 먼저 만들고 완성 동선 기준 숙소 추천을 받을 수 있으며(정본: 'AI 일정 생성' 섹션), 이 경로는 숙소 우선 진입과 동등한 1급 온램프로 노출해 '결제·예약부터 해야 일정이 나온다'는 인상을 주지 않아야 한다.
- FR-10 (스토리 2 예외): 숙소가 한 개도 등록되지 않은 상태에서 AI 일정 생성을 요청하면 일정 생성 버튼을 비활성화하고 '일정을 생성하려면 숙소를 먼저 등록하세요' 안내와 숙소 등록 진입점을 표시해야 한다(정본: 'AI 일정 생성' 스토리 1).
- FR-11 (스토리 3): 사용자는 외부 OTA 예약 숙소 또는 앱 저장 숙소를 여행에 '등록'할 수 있으며, 등록 숙소는 숙소명·위치(좌표/주소)·체크인·체크아웃 날짜를 필수 속성으로 가지고 해당 체류 기간 동안 일정 생성의 기준 거점으로 사용되어야 한다.
- FR-12 (스토리 3): 사용자는 등록한 숙소를 수정·삭제할 수 있어야 하며, 등록 폼은 '숙소 탐색' 섹션의 직접 등록 화면(지도 검색·예약 링크 붙여넣기·지도 핀 지정 3경로)을 정본으로 하고 여행 생성·거점 화면에서는 등록 진입점만 제공한다.
- FR-13 (스토리 3 예외): 등록 시 위치 좌표를 확인할 수 없으면 지도에서 위치를 직접 지정하도록 요청하고, 체크인/아웃이 여행 기간을 벗어나면 경고를 표시하고 여행 기간 자동 확장 여부를 사용자에게 물어야 한다.
- FR-14 (스토리 4): 저장된 숙소를 등록할 때 저장 시점의 숙소명·위치를 자동으로 채워 사용자는 체크인/아웃 날짜만 입력하면 등록이 완료되어야 하며, 외부 출처에서 더 이상 조회되지 않으면 캐시 정보로 등록을 허용하되 '최신 정보 확인 불가'를 표시해야 한다.
- FR-15 (스토리 5): 숙소 등록 시 '이 숙소의 체크인/아웃 날짜를 여행 기간으로 가져올까요?'를 제안하고, 수락 시 여행 시작일·종료일을 숙소 날짜로 설정하되 사용자가 다시 수정할 수 있어야 하며, 기존 여행 기간과 충돌하면 두 값을 나란히 보여주고 선택하게 해야 한다.
- FR-16 (스토리 6): 하나의 여행에 여러 숙소를 등록해 도시 이동·구간별 거점을 표현할 수 있어야 하며, 다중 숙소는 체크인 날짜 순 정렬로 표시하고 각 날짜에 해당하는 숙소를 그날의 기준 거점으로 사용해야 한다.
- FR-17 (스토리 6): 두 숙소의 체류 기간이 겹치면 겹치는 날짜를 표시하고 어느 숙소를 그날의 거점으로 쓸지 선택하게 하되, 매번 모달로 막지 않고 스마트 기본 거점(공백일=직전 숙소, 겹침=체크인 우선 숙소 등)을 미리 채워 일정을 일단 생성한 뒤 '이 날은 ○○ 기준으로 잡았어요 — 바꾸기' 비차단 안내로 사후 수정하게 해야 한다.
- FR-18 (스토리 6 예외): 어떤 숙소도 배정되지 않은 날짜(공백 구간)가 있으면 해당 날짜를 표시하고 직전 숙소 기준 또는 여행지 중심 중 어느 쪽을 쓸지 사용자에게 물어야 한다.
- FR-19 (스토리 7): 같은 숙소 다박은 하나의 등록으로 연속 숙박 기간을 지정하며, 체크인~체크아웃 사이 모든 숙박일에 동일 숙소를 거점으로 적용하고 일정 화면에서 매일 반복 블록이 아닌 'N박 체류'로 묶어 표시해야 한다.
- FR-20 (스토리 8): 사용자는 AI 일정 생성 전에 장소 검색(POI명·주소, Place Data 모듈 기반)으로 필수 방문지를 여러 곳 지정할 수 있으며, 앱은 이를 등록 숙소와 함께 일정 생성의 앵커로 사용해야 한다.
- FR-21 (스토리 8): 필수 방문지는 기본 '아무 때나 꼭 가기'(포함만 보장, 시각·순서는 AI 배치)로 추가되고, 사용자가 원할 때만 '시간 정해두기' 토글로 날짜·시각을 고정할 수 있어야 한다(반영 규칙 정본: 'AI 일정 생성' 스토리 4).
- FR-22 (스토리 8): 필수 방문지는 일정 생성 전뿐 아니라 생성 후에도 추가·삭제할 수 있어야 하며, 변경 시 해당 날짜(날짜 미지정 시 여행 전체)의 추천 POI 배치를 다시 계산해야 한다.
- FR-23 (스토리 8): 필수 방문지 지정은 완전 AI·같이 고르기 경로에서 선택적으로 제공하고 직접 작성 경로에서는 사용자의 직접 장소 추가로 대체하며, 완전 AI 경로에서 미지정 시 생성된 추천안에서 특정 장소를 사후 고정(LOCK)해 필수화할 수 있어야 하고, 같이 고르기 경로에서는 지정 필수 방문지를 후보 추천의 시드로 사용해야 한다.
- FR-24 (스토리 8 예외): 필수 방문지 좌표를 확인할 수 없으면 지도에서 직접 지정하도록 요청하고, 시각 고정형인데 날짜·시작 시각·체류 시간 미입력이면 영업시간·동선에 맞춰 자동 배치하며(화면 d07), 기존 고정 블록과의 충돌 처리는 'AI 일정 생성' 스토리 4 예외를 정본으로 따라야 한다.
- FR-25 (스토리 9): 앱은 등록 숙소와 필수 방문지를 고정/필수 블록으로 처리해 시각 고정 블록은 재생성 후에도 날짜·시간이 변경되지 않고, 포함 고정형은 시각·순서 재계산은 가능하되 일정에서 누락되지 않아야 하며, 고정/필수 블록과 AI 추천 블록을 시각적으로 구분 표시해야 한다.
- FR-26 (스토리 10): 외부 OTA 이동·제휴 수수료 고지는 '숙소 탐색' 섹션(us_stay) 스토리 5를 정본으로 하며, 본 섹션은 외부 예약 완료 숙소를 스토리 3 흐름으로 여행에 등록한다는 연결만 정의한다.

**비기능 요구사항 (7건)**:
- 결정 피로 완화(UX): 다중 숙소의 공백·겹침 구간마다 모달로 흐름을 차단하지 않고 스마트 기본값을 채워 일단 진행시키는 비차단(non-blocking) 인터랙션 원칙을 지켜야 한다.
- 진입 장벽 최소화(UX/비즈니스): 숙소 미등록 일정 생성 경로를 숙소 우선 진입과 동등한 1급 온램프로 노출해 '예약·결제가 선행 조건'이라는 인상을 주지 않아야 한다(전환 심리 장벽 제거).
- 데이터 신선도 투명성: 외부 출처에서 조회 불가한 저장 숙소는 캐시 정보로 등록을 허용하되 '최신 정보 확인 불가'를 명시해 침묵 실패를 금지한다(ADR-0011 원칙과 일관).
- 외부 지도 API 제약 준수(암묵): 장소 검색(Place Data 모듈)·좌표 확인은 국내 지도 API를 사용하므로 응답 데이터 영구 캐싱 금지·실시간 호출·출처 표기 약관을 데이터 모델에 반영해야 한다(README 선결 과제 계승). 단, 저장 숙소의 캐시 등록 허용(스토리 4)과의 정합은 확인 필요.
- 입력 검증 일관성: 날짜 역전, 좌표 미확인, 기간 이탈 등 모든 입력 오류는 명시적 오류 메시지·사용자 선택 요청으로 처리하며 자동 무시하지 않는다.
- 사용자 언어 원칙: '시각 고정형/포함 고정형' 같은 내부 용어를 화면에 노출하지 않고 '시간 정해두기'/'아무 때나 꼭 가기' 같은 사용자 언어로 라벨링해야 한다.
- 점진적 개인화: 온보딩 미설정 취향은 여행 생성 흐름에서 점진적으로 보완하되 항상 건너뛰기 가능해야 하며, 미설정은 중립 기본값으로 안전하게 동작해야 한다.

**문서가 미정으로 남긴 결정 (8건)**:
- 숙소 미등록 시 AI 일정 생성 허용 정책의 정본은 'AI 일정 생성' 섹션 스토리 1에 위임되어 있으며, 본 문서 스토리 2 예외(버튼 비활성화)는 그 정본과 일치시켰다고만 명시한다.
- 시각 고정형/포함 고정형 필수 방문지의 일정 반영 규칙과 고정 블록 충돌 처리는 'AI 일정 생성' 섹션 스토리 4에 위임.
- 숙소 나중 등록·완성 동선 기준 숙소 추천의 상세는 'AI 일정 생성' 섹션의 해당 스토리에 위임.
- 숙소 직접 등록 폼(지도 검색·예약 링크 붙여넣기·지도 핀 지정 3경로)의 상세는 '숙소 탐색' 섹션에 위임.
- 외부 OTA 이동·제휴 수수료 고지 동작은 '숙소 탐색' 섹션 스토리 5에 위임(중복 정의 금지).
- 생성 방식 분기(완전 AI/같이 고르기/직접 작성)의 정의는 'AI 일정 생성' 스토리 10에 위임.
- 스마트 기본 거점 규칙이 '공백일=직전 숙소, 겹침=체크인 우선 숙소 등'으로 예시 수준으로만 제시되어 있어 '등'에 포함되는 나머지 규칙은 미확정.
- 예산이 숙소 가격 필터·일정 비용 가중치로 분배되는 구체 산식은 'AI 일정 생성' 스토리 2에 위임되어 있으나 분배 규칙 자체는 어느 문서에도 확정되어 있지 않음.

## docs/PRD/06-AI일정생성.md

**요약**: 등록된 숙소(위치+체크인/아웃)를 출발점으로 여행 기간 전체의 날짜별 일정을 LLM+솔버 하이브리드(LLM=취향 해석·점수·설명, OPTW/TOPTW 솔버=선택·순서·시간 보장)로 생성하는 기능을 정의한다. 영업시간·이동시간·체류시간을 하드 제약으로 하고, 필수 방문지를 시각 고정형/포함 고정형 두 유형으로 반영하며, 시간표/지도 2보기와 수정 시 재검증, 위반 시 경고 배지+수동 보존을 규정한다. 생성 방식은 완전 AI 자동/같이 고르기/직접 만들기 3분기이며 도중 전환이 가능하고, 숙소 나중 등록(동선 무게중심 기반 숙소 권역 추천) 온램프와 일정 '확정'(읽기 전용 최종본 잠금) 흐름을 포함한다. 모든 외부 API·LLM 실패에 대해 결정론적 솔버 폴백과 관찰 가능한 실패 고지(ADR-0011)를 요구한다.

**기능 요구사항 (51건)**:
- 등록 숙소의 위경도와 체크인/아웃 날짜를 고정 입력값으로 삼아 체크인 당일~체크아웃 당일까지 날짜별 1개씩 일정을 생성해야 한다 (스토리 1)
- 등록 숙소가 없으면 일정 생성 버튼을 비활성화하고 안내 문구를 표시하되, 스토리 11의 숙소 나중 등록 온램프는 예외로 허용해야 한다 (스토리 1)
- 같은 여행 기간에 위치가 다른 숙소가 여러 개 등록되면 각 숙소의 체크인/아웃 날짜에 맞춰 날짜별 기준점을 자동 전환해야 한다 (스토리 1)
- 숙소 지오코딩 실패 시 지도에서 위치를 직접 지정하게 하고 지정 전까지 일정 생성을 보류해야 한다 (스토리 1 예외)
- 체크인/아웃 날짜 누락·역전 시 일정을 생성하지 않고 날짜 수정 화면으로 이동시켜야 한다 (스토리 1 예외)
- 위치 권한 거부 시에도 등록 숙소 위치를 출발점으로 일정을 생성하고, 현재 위치 필요 기능은 숙소 위치로 대체하며 그 가정을 화면에 표기해야 한다 (스토리 1 예외)
- LLM이 사용자 취향(자유 입력 포함)을 해석해 후보 POI별 선호 점수를 매기고 그 점수를 OPTW/TOPTW 목적함수 보상값으로 사용해야 한다 (스토리 2)
- 추천 POI를 외부 POI 데이터에 그라운딩(RAG)해 실재 장소만 노출하고, 미확인 POI는 일정에서 제외해야 한다 (스토리 2)
- 이동 방식(도보/자차 등)에 따라 직선/도로 거리 기반 이동시간을 다르게 계산해 추천에 반영해야 한다 (스토리 2)
- 예산으로 채울 POI가 없으면 안내와 함께 무료/저비용 대안 POI를 우선 제시하고, 후보 0건이면 어떤 조건이 결과를 0으로 만들었는지 표시하고 완화를 제안해야 한다 (스토리 2 예외)
- 어떤 POI도 영업시간 밖(휴무 포함)에 배치하지 않아야 한다 (스토리 3)
- 직전 일정 종료 시각 + 이동시간(안전 마진 포함) ≤ 다음 일정 시작 시각을 항상 만족시키고 위반 배치는 생성 결과에서 제외해야 한다 (스토리 3)
- 사용자에게 보이는 모든 시각은 솔버가 소유·검증한 값만 표시하고 LLM 생성 시각을 그대로 노출하지 않아야 한다 (스토리 3, 10)
- POI 체류 시간을 카테고리·데이터 기반 기본값에 일정 밀도·여행 스타일을 반영해 채우고 사용자가 직접 수정할 수 있어야 한다 (스토리 3)
- 체류 시간을 최소·권장·최대 범위로 관리해 솔버가 권장→최소 안에서 줄여 배치할 수 있고, 사용자가 고정하면 고정 제약으로 사용해야 한다 (스토리 3)
- 솔버가 체류 시간을 권장보다 줄여 배치하면 그 사실을 해당 POI에 표시해야 한다 (스토리 3)
- 영업시간 데이터가 없는 POI는 '영업시간 미확인'으로 표시하고 확정 배치 대신 사용자 확인 필요 후보로 분리해야 한다 (스토리 3, 4)
- 이동시간 제약상 하루 일정이 1개 이하이면 억지로 늘리지 않고 사유를 안내해야 한다 (스토리 3 예외)
- 숙소 체크인/아웃 시각과 시각 고정형 필수 방문지를 변경 불가 고정 블록으로 놓고 나머지를 충돌 없이 배치해야 한다 (스토리 4)
- 필수 방문지는 시각 고정형과 포함 고정형(기본값) 두 유형을 지원하고, 화면 라벨은 사용자 언어('시간 정해두기'/'아무 때나 꼭 가기')로 표기해야 한다 (스토리 4)
- 포함 고정형 필수 방문지는 솔버가 반드시 포함(필수 노드)하되 시각·순서는 하드 제약 안에서 자동 결정해야 한다 (스토리 4)
- 필수 방문지(특히 포함 고정형)를 주변 추천 POI가 모이는 공간적 앵커로 활용해야 한다 (스토리 4)
- 고정/필수 일정을 추가·삭제하면 해당 날짜(날짜 미지정 시 여행 전체)의 추천 POI 배치를 재계산해야 한다 (스토리 4)
- 두 고정 일정이 충돌하면 자동 변경 없이 충돌을 강조 표시하고 어느 쪽을 옮길지 사용자에게 물어야 한다 (스토리 4 예외)
- 필수 방문지를 어떤 시간대에도 넣을 수 없으면 불가 사유를 표시하고 다른 날짜 이동/시각 고정 해제/인접 일정 조정 중 하나를 제안해야 한다 (스토리 4 예외)
- 장소별 추천 이유(반영된 선호)와 배치 이유(솔버 제약 근거)를 함께 제공하되, 설명은 표시용이며 설명·배치 불일치 시 검증된 시각을 기준으로 해야 한다 (스토리 5)
- 동일 일정 데이터를 시간표형/지도형으로 전환해 보여주고 한 보기의 수정이 다른 보기에 즉시 반영되어야 한다 (스토리 6)
- 지도형은 순서 번호 핀과 이동 동선을 표시해야 한다 (스토리 6)
- 연속 일정 구간마다 예상 소요 시간·이동 수단·거리 요약을 범위·추정치로 표시하고, 대중교통 실시간 경로 안내는 제공하지 않아야 한다 (스토리 6)
- 각 이동 구간에 '길찾기' 동작을 제공해 외부 지도앱으로 좌표를 전달해 길안내를 위임하고, 지도앱 없으면 웹 지도 URL, 그것도 불가하면 앱 내 경로 요약으로 폴백해야 한다 (스토리 6)
- 지도 API 무응답 시 시간표형으로 폴백하고, 라우팅 API 무응답 시 직선거리 추정 또는 수동 입력 모드로 폴백하며 추정/수동임을 표기해야 한다 (스토리 6 예외)
- POI 추가/삭제/재정렬/시간 변경마다 영업시간·이동시간·고정 일정 충돌을 즉시 재검증해야 한다 (스토리 7)
- 제약 위반 수정은 차단하지 않되 경고 배지와 사유를 표시하고, POI 추가 시 삽입 가능한 시간대만 후보로 제시해야 한다 (스토리 7)
- 위반 상태로 저장 시 위반을 보존·지속 가시화하고 자동 시각 변경 없이 'AI로 자동 보정' 또는 '그대로 저장'을 선택하게 해야 한다 (스토리 7 예외)
- 저장 시 모든 시각·순서·고정 블록을 보존하고, 실행 화면에서 다음 일정과 이동 거리(소요시간 미표시)를 표시하며 저장 일정을 Plan-B 입력 기준선으로 사용해야 한다 (스토리 8)
- 저장 중 네트워크 오류 시 로컬 임시 보관 후 재연결 시 자동 동기화하고 '저장 대기 중'을 표시해야 한다 (스토리 8 예외)
- 외부 API 실패 시에도 내장 결정론적 솔버(규칙 점수+TOPTW+제약 검증)로 일정을 생성하고 기본 모드 생성 사실을 고지해야 한다 (스토리 9)
- LLM 실패 시 일정은 솔버 결과로 제공하고 설명 문구만 생략 또는 실패 고지해야 한다 (스토리 9)
- 첫 1일 일정을 수 초 내 먼저 노출하고 나머지는 백그라운드로 채우며, 생성 중 단계 텍스트·진행률·경과 시간과 취소/백그라운드 계속 선택을 제공해야 한다 (스토리 9)
- 모든 생성 경로 실패 시 숙소+시각 고정형 필수 방문지만 배치된 최소 일정과 재시도 동작을 제공해야 한다 (스토리 9 예외)
- 완전 AI 자동/같이 고르기/직접 만들기 3방식을 선택할 수 있어야 하며, 분기 화면 카드에 한 줄 결과 예고와 예상 소요·인터랙션 양을 표시해야 한다 (스토리 10)
- 완전 AI 자동은 전체 초안을 한 번에 생성하고 사용자가 개별 슬롯을 LOCK 또는 교체할 수 있어야 한다 (스토리 10)
- 같이 고르기는 슬롯마다 컨셉/테마와 반경 내 후보를 제시하고, 반경 확대 시 거리 트레이드오프 표기와 함께 후보를 포함하며 반경 밖 후보는 비활성 처리해야 한다 (스토리 10)
- 직접 만들기는 빈 일정에서 장소 검색·추가로 구성하되 추가마다 스토리 7과 동일하게 재검증해야 한다 (스토리 10)
- 분기 도중 방식 전환 시 이미 채운 슬롯을 보존한 채 나머지만 새 방식으로 이어가야 한다(진행분 손실 없음) (스토리 10)
- 같이 고르기 반경 내 후보 0건이면 반경 확대(다음 이동 수단) 또는 컨셉 변경을 제안해야 한다 (스토리 10 예외)
- 숙소 없이 장소 기준으로 일정을 먼저 구성한 뒤, 동선 무게중심·평균 이동 거리 기반 숙소 권역을 지도로 추천받아 나중에 등록할 수 있어야 한다 (스토리 11)
- 추천 숙소 후보를 평균 이동 거리 순으로 제시하고 before/after 이동 거리 개선을 '추정'으로 표기해야 하며, 등록 시 그 숙소를 기준점으로 스토리 1의 생성·재정렬을 수행해야 한다 (스토리 11)
- 숙소를 끝까지 등록하지 않은 날(당일치기·이동일)은 숙소 없이 동선만으로 일정을 유지해야 한다 (스토리 11 예외)
- 완성 일정을 '확정'해 읽기 전용 최종본으로 잠그고, 확정 화면에 D-day 등 여행 시작 맥락과 확정의 의미·이점을 한 줄로 설명해야 한다 (스토리 12)
- 확정 후에도 '일정 수정'으로 자유 편집에 진입할 수 있어야 하며, 확정 일정은 여행 중 실행·Plan-B의 입력 기준선이 되어야 한다 (스토리 12)

**비기능 요구사항 (10건)**:
- 성능: 첫 1일 일정을 수 초 내 노출, 전체 생성 완료 한계 20초, 나머지 날짜는 백그라운드 생성 (스토리 9)
- 신뢰성/정확성: 사용자에게 보이는 모든 시각·순서는 솔버 검증값만 사용, LLM 생성 시각·환각 POI(그라운딩 미확인) 노출 금지 (스토리 2, 3, 5)
- 가용성/폴백: LLM·POI·지도·라우팅 API 실패 시에도 결정론적 솔버로 항상 결과 제공, 침묵 실패 금지(모든 폴백을 화면에 고지) (스토리 6, 9, ADR-0011)
- 정확도 고지: 이동시간은 거리 기반 추정이므로 단일값이 아닌 범위·추정으로 표시하고 시간창에 안전 마진 적용 (스토리 3, 6, ADR-0009)
- 오프라인 내구성: 저장 실패 시 로컬 임시 보관 + 재연결 자동 동기화 (스토리 8)
- 벤더 독립성: 거리·좌표는 Place Data 모듈 경유로 받아 단일 지도 벤더에 종속되지 않음 (스토리 6, ADR-0009)
- 관찰가능성/UX: 생성 중 단계 텍스트·진행률·경과 시간 표시, 취소·백그라운드 계속 제공 (스토리 9)
- 규제 준수: 국내 지도 API 약관(영구 캐싱 금지·실시간 호출·출처 표기)을 데이터 모델에 반영 (README 선결 과제, 암묵적 전제)
- 비용: LLM·외부 API 반복 호출 비용·rate-limit 상한 설계 필요 (README 선결 과제, 암묵적)
- 사용자 통제: 자동 시각 변경 금지 — 위반 시에도 사용자 선택(자동 보정/그대로 저장)을 거쳐야 함 (스토리 7)

## docs/PRD/07-PlanB재계획.md

**요약**: 여행 중(여행 날짜 구간 진입) 변수 발생 시 일정을 재구성하는 Plan-B 재계획 기능의 PRD로, 수동 트리거(사유 선택/사유 없음)와 자동 트리거(날씨·휴무·이동지연·체류초과 감지) 두 진입을 정의한다. 재계획은 LLM이 사유·취향을 해석하고 솔버(OPTW/TOPTW)가 고정 제약(숙소 체크인/아웃, 시각 고정 필수방문지, 영업시간, 이동시간)을 검증한 대안 2~3개를 제시하며, 사용자는 AI 맡기기/직접 수정을 선택하고 확정 전 전/후 비교 화면에서 승인한다. 확정 시 변경 이력(사유·스냅샷·트리거 유형)을 저장해 회고에 반영하고, GPS 불가 시 수동 위치 입력, 외부 API 오류 시 수동 수정 폴백을 제공한다. 숙소 예약 자체 변경은 범위 밖(ADR-0006)이며, 이동시간은 거리 기반 추정(ADR-0009), 실제 길안내는 외부 지도 앱 위임(ADR-0012)이다. 계획 동선과 실제 GPS 이동 경로의 지도 토글 비교(스토리 13)도 포함한다.

**기능 요구사항 (23건)**:
- [S1] 사용자는 진행 중인 여행의 활성 일정 화면에서 '재계획' 버튼으로 사유(날씨 악화·임시 휴무·이동 지연·POI 취소/예약 마감·체력 저하)를 선택하거나 '사유 없음'으로 수동 재계획을 요청할 수 있어야 한다.
- [S1] '여행 중(진행 중)' 상태는 여행 날짜 구간 진입으로 정의하며(숙소 있으면 체크인/아웃 구간, 없으면 시작일+현재 위치/첫 일정 기준), 등록 숙소가 0개여도 날짜 구간 안이면 재계획이 가능해야 한다.
- [S1] 선택된 사유는 LLM 해석 입력으로 전달하되 최종 후보는 솔버(OPTW/TOPTW) 검증을 통과한 결과만 노출해야 한다.
- [S2] 앱은 (a)다음 방문지 도착 시간대 강수확률 60% 이상/기상특보, (b)당일 임시휴무·영업시간 변경, (c)예상 이동시간 계획 대비 임계(기본 15분, 사용자 조절) 초과, (d)실제 체류시간 임계(기본 20분, 사용자 설정) 초과로 고정 일정 도착 위협 시 비차단 배너로 재계획을 선제 제안해야 한다.
- [S2] 자동 트리거는 제안일 뿐이며 사용자가 '대안 보기'를 명시적으로 눌러야 재계획 흐름이 시작되고, 자동으로 일정을 변경하지 않아야 한다.
- [S2] 동일 사유 알림은 동일 방문지당 1회만 노출하고 닫으면 재노출하지 않으며, 전역 빈도 상한(시간당/하루 최대 N회, 초과 시 묶음 제시)을 두고, 체류 초과 알림은 연속 무시 시 해당 일정에 대해 억제(학습)해야 한다.
- [S2] 푸시는 심각도 높은 사유(기상특보·고정 일정 도착 위협)에 한정하고 경미한 건은 인앱 칩만 띄우며, 앱 밖에서는 비차단 푸시로 전달하고 푸시 탭 시 '대안 보기' 흐름으로 진입해야 한다.
- [S2] 자동 트리거 알림에 사용한 외부 데이터(날씨/휴무) 출처와 감지 시각을 표기해야 하며, 외부 API 응답이 없으면 자동 알림을 띄우지 않고 수동 경로만 유지해야 한다(허위 알림 금지).
- [S2] 자동 트리거 민감도(적게/보통/많이)는 설정에서 조절 가능해야 한다.
- [S3] 영향 분석은 현재 위치(GPS)·현재 시각·남은 방문지·고정 제약(숙소 체크인/아웃 일시·위치)·시각 고정 필수방문지 확정 시간·당일 영업시간·거리 기반 추정 이동시간을 기준 입력으로 사용해야 한다.
- [S3] 등록 숙소 체크인/아웃과 시각 고정형 필수 방문지는 변경 불가 고정 제약으로 취급하고, 영향 분석은 현재 시각 이후 남은 일정에만 적용하며 완료된 방문지는 변경하지 않아야 한다.
- [S4] 앱은 고정 제약과 충돌하지 않는 대안 후보를 2~3개 추천하되, 각 후보는 솔버 검증(숙소 시간창·영업시간·이동시간 오버랩 없음·고정 시각 준수)을 통과해야 한다.
- [S4] 후보 POI는 RAG로 그라운딩한 실재 장소여야 하며 환각·폐업·좌표 불명 POI는 제외하고, 후보는 사유에 부합(날씨→실내 우선, 체력→이동/방문 수 축소)해야 한다.
- [S4] 대안 0개 시 빈 화면 대신 '방문지 1개 건너뛰기 / 휴식 모드 전환 / 수동 수정 진입' 폴백과 사유 한 줄 설명을 제시해야 한다.
- [S5] 각 후보마다 추천 이유(LLM 자연어, 모호어 금지·구체 근거), 예상 이동 거리·수단, 예상 체류 시간, 다음 고정 제약까지 여유 시간을 표시하되, 추정치임과 산출 기준(거리 기반)을 명시하고 차량/도보 소요시간은 표시하지 않아야 한다.
- [S6] 사용자는 대안 선택 / 기존 유지 / 휴식 모드 전환 중 택일할 수 있으며, 기존 유지는 무변경 복귀, 휴식 전환은 재개 시각 입력 또는 즉시 재개를 지원하고 휴식 종료 후 남은 일정을 재계산해야 한다.
- [S7] 대안 선택 시 남은 일정을 자동 재정렬하되 솔버 검증을 통과해야 하고, warm-start로 변경 부분만 갱신하며 고정 항목·완료 방문지는 보존하고, 제외·이월되는 방문지는 명시적으로 보여주고 확정 전 동의를 받아야 한다.
- [S8] 확정 직전 전/후 일정을 나란히 비교(추가/삭제/시간이동 시각 구분, 총 이동거리·방문지 수·숙소 복귀 시각 변화 요약)하고 최종 '확정' 또는 '취소(원상복구)'를 선택하게 해야 한다.
- [S9] 확정 시 변경 이력(사유, 전/후 스냅샷, 확정 시각, 트리거 유형 자동/수동)을 저장하고 여행 기록에 반영하며, 회고 화면에서 타임라인으로 열람 가능해야 하고, 이력 저장이 등록 숙소 정보를 수정하지 않아야 한다.
- [S10] 위치 권한 거부/GPS 불가 시 재계획을 차단하지 않고 수동 위치 입력(지도 핀/장소 검색)으로 진행하며, 수동 위치는 추정 출발지임을 표기하고, 입력도 생략하면 마지막 완료 방문지 또는 숙소 위치를 기준점으로 사용하며 가정을 명시해야 한다.
- [S11] 외부 API 오류(날씨·영업시간·길찾기·POI 검색 실패/타임아웃) 시 수동 일정 수정 화면(순서 재배열·삭제·시각 직접 입력)으로 전환하되, 숙소 고정 제약 위반 입력은 경고 후 저장 차단하고, 누락 데이터를 화면에 표기하며, API 복구 시 수동 결과를 유지한 채 자동 기능을 재활성화해야 한다.
- [S12] 사용자는 사유 선택 후 'AI에게 맡기기(솔버 검증 후보 2~3개 택일)' 또는 '직접 수정(스토리 11과 동일 편집)' 방식을 선택할 수 있어야 하며, 어느 방식에서도 숙소 고정 제약은 위반 불가이고 AI 후보 0건 시 스토리 4의 폴백을 동일하게 제시해야 한다.
- [S13] 사용자는 활성 일정 화면 지도에서 계획 동선(흐리게)과 실제 GPS 이동 경로(점선·점)를 토글/중첩 비교하고 누적 실제 이동 거리·걸음 수(추정치 표기)를 볼 수 있어야 하며, 위치 불가 시 실제 경로 레이어를 비활성화하고 사유를 표기해야 한다.

**비기능 요구사항 (10건)**:
- 허위 알림 금지: 외부 데이터 미확인 시 자동 트리거를 발화하지 않고 침묵 실패 대신 수동 경로 유지·사유 표기(ADR-0011 계승).
- 알림 폭주 방지: 전역 빈도 상한(시간당/하루 N회), 동일 사유·동일 방문지 1회 노출, 다중 사유 묶음 제시, 연속 무시 시 억제 학습 — 사용자 피로도와 배터리·비용 보호.
- 자동 트리거 폴링 주기·LLM 호출 빈도에 상한을 두어 비용·배터리·rate-limit을 통제해야 한다(README 선결 과제).
- 추정치 투명성: 이동 거리·시간·걸음 수는 항상 추정치임과 산출 기준을 표기하고, 자동 트리거 임계는 거리 추정 오차 마진 위로 설정(ADR-0009).
- 설명 가능성: LLM 추천 이유는 모호어 없이 구체적 근거(거리·시간 수치)를 포함해야 한다.
- 일관성/무결성: 어떤 경로(AI/수동/폴백)에서도 숙소 체크인/아웃 고정 제약은 위반 불가이며, 솔버 검증 미통과 재정렬안은 노출 금지.
- 감사 가능성: 모든 확정 변경은 사유·스냅샷·시각·트리거 유형을 포함한 변경 이력으로 보존.
- 개인위치정보 처리: GPS 발자취 수집·표시(스토리 13)는 위치정보법상 동의·파기·로그 보존 의무(README 선결, ADR-0017)와 연동되어야 한다(암묵적).
- 국내 지도 API 약관 준수: 응답 데이터 영구 캐싱 금지·실시간 호출·출처 표기(암묵적, README 계승).
- 프라이버시/권한 저항성: 위치 권한 거부 시에도 기능이 차단되지 않는 수동 폴백 제공.

**문서가 미정으로 남긴 결정 (7건)**:
- 자동 트리거 전역 빈도 상한의 구체 값(시간당/하루 최대 N회) — 문서가 N으로 남김.
- 자동 트리거(날씨·휴무·이동지연 감지)의 폴링 주기와 LLM 호출 빈도 상한 — README 선결 과제로 명시, 미정.
- 날씨 API 선택(공공데이터포털 vs OpenWeather) — 강수확률 60%·기상특보 감지의 데이터 소스 미확정.
- 지도·장소 API 선택(카카오맵/네이버/TMap) — 이동시간 추정·POI 영업시간 감지의 소스 미확정.
- 자동 휴무·영업시간 감지의 실제 커버리지 — 데이터 커버리지 정량 게이트(영업시간 필드 채움률 등) 확인 후 출시, 미커버 POI는 수동 트리거로만 대응(README 선결).
- 체류 초과 억제(학습)의 '연속 무시' 판정 기준 — 횟수·억제 지속 범위 미정.
- 이동 수단별 안전계수 값(거리 기반 이동시간 추정 파라미터, ADR-0009 참조) — 구체 수치 미정.

## docs/PRD/08-여행중현장이용.md

**요약**: 본 문서는 여행 중 사용자가 일정을 정상 실행할 때의 현장 화면을 정의하며, Plan-B 재계획과 명확히 구분된다. 핵심은 3가지 흐름이다: (1) 장소 도착 감지(자동/수동)와 방문 시작·완료 체크 및 사진·메모 진입점 노출, (2) 현재 장소의 상세 정보(영업시간·예상 체류·혼잡도·주변 추천·다음 일정까지 여유 시간) 현장 확인, (3) 다음 예정지까지의 거리(거리 기반 추정, 소요시간 미표시) 인라인 표시와 외부 지도 앱으로의 길찾기 위임 및 복귀 핸드오프. 방문 체크·기록의 저장 규칙은 '여행 기록/회고'를, 거리 산출 기준은 'AI 일정 생성'(ADR-0009)을 정본으로 따르며, 본 문서는 현장 진입점(UI 흐름)만 소유한다.

**기능 요구사항 (14건)**:
- [스토리 1] 앱은 현재 위치가 다음 예정지 인근임을 감지하거나 사용자가 수동으로 '도착'을 표시하면, 방문 시작/완료 체크·사진·메모 진입점을 활성 일정 위에 노출해야 한다.
- [스토리 1] 방문 완료 표시는 활성 일정의 진행 상태(완료/진행 중/다음)에 반영되어야 한다.
- [스토리 1] 방문 시작·완료 시각은 실제 체류 시간 측정의 기준이 되어 Plan-B 스토리 2의 자동 트리거(체류 초과·이동 지연) 입력으로 사용되어야 한다.
- [스토리 1] 방문 완료 체크·사진·메모의 저장·기록 규칙은 '여행 기록/회고' 스토리 1·2를 정본으로 따르고, 본 화면은 현장 진입점만 제공한다.
- [스토리 1 예외] 위치 권한이 없거나 GPS 정확도가 낮으면 자동 도착 감지를 보류하고 수동 체크인으로 대체해야 한다('여행 기록/회고' 스토리 3과 일관).
- [스토리 2] 앱은 현재 방문 중(또는 예정) 장소의 영업시간, 예상 체류 시간, (가용 시) 혼잡도, 주변 추천을 현장에서 보여줘야 한다.
- [스토리 2] 현재 장소 상세에 다음 일정까지의 여유 시간을 함께 표기해야 한다.
- [스토리 2] 예상 체류·혼잡도 등은 추정치임을 명시하고 산출 방식·데이터 출처를 표기해야 한다.
- [스토리 2 예외] 장소 상세 데이터(영업시간·혼잡도)가 누락되면 빈칸 대신 '미확인'으로 표기해야 한다.
- [스토리 3] 앱은 별도 길찾기 전용 화면 없이 활성 일정 화면의 '다음' 행/맥락 액션에 현재 위치→다음 예정지 거리(거리 기반 추정)를 인라인으로 표시해야 한다.
- [스토리 3] 차량/도보 예상 소요시간은 표시하지 않으며, 실제 단계별 길안내·소요시간(대중교통 노선·환승 포함)은 '외부 지도 앱으로 열기'로 위임해야 한다(ADR-0012).
- [스토리 3] 외부 지도 앱으로 이탈했다가 복귀하면 현재 활성 일정·다음 행동(도착 체크인 등)으로 매끄럽게 복귀시켜 맥락을 유지해야 한다('숙소 탐색' 스토리 5와 동일 핸드오프 패턴).
- [스토리 3] 이동 거리는 추정치임을 명시하고 'AI 일정 생성' 스토리 3·6의 산출 기준(장소 간 거리 기반 추정, ADR-0009)과 일관되게 표기해야 한다.
- [스토리 3 예외] 연결 가능한 외부 지도 앱이 없으면 웹 지도 길찾기로 폴백하고, 그것도 불가하면 앱이 산출한 거리 요약(직선/도로 거리·추정)을 최종 안내로 표시해야 한다(ADR-0011과 일관).

**비기능 요구사항 (8건)**:
- 위치 권한 미보유·GPS 저정확도 상황에서 기능이 침묵 실패하지 않고 수동 체크인으로 graceful degradation 되어야 한다(ADR-0011 '미확인 분리·수동 폴백·침묵 실패 금지' 원칙).
- 모든 추정치(체류 시간·혼잡도·이동 거리)는 단정값이 아닌 추정임을 UI에 명시하고 산출 방식·데이터 출처를 표기해야 한다(투명성).
- 데이터 누락 시 빈칸이 아닌 '미확인' 명시 표기(데이터 신뢰성 커뮤니케이션).
- 자체 길찾기·소요시간 계산을 구현하지 않고 외부 지도 앱에 위임한다(ADR-0012, 시스템 경계·유지비 최소화).
- 외부 앱 이탈→복귀 시 앱 상태(활성 일정·다음 행동)가 보존되어야 한다(상태 복원성·UX 연속성).
- 국내 지도 API 약관 제약(응답 데이터 영구 캐싱 금지·실시간 호출·출처 표기)이 현재 장소 상세·거리 표시 데이터 모델에 반영되어야 한다(README 법적·규제 선결 과제).
- 개인위치정보(GPS) 상시 이용에 따른 위치정보법 준수(동의·철회·목적달성 후 파기·로그 보존, ADR-0017)가 도착 감지 기능의 전제다.
- 여행 중 위치 감지·거리 갱신의 폴링 빈도는 배터리·API 호출 비용·rate-limit 상한 내에서 설계되어야 한다(README '외부 API 호출 비용·빈도·캐싱' 선결 과제 암시).

**문서가 미정으로 남긴 결정 (4건)**:
- 채택할 국내 지도·장소 API(카카오맵/네이버 지도/TMap) 선택 — 도착 감지 좌표·거리 산출·장소 상세 데이터의 공급원이 이에 종속된다(README 운영 결정).
- 혼잡도 데이터의 공급원과 가용 범위 — 문서는 '(가용 시)'로만 조건화하고 어떤 데이터로 혼잡도를 산출하는지 미정.
- 여행 중 위치 폴링 주기·자동 감지 호출 빈도 상한 — README가 출시 전 설계 과제로 남긴 비용·배터리·rate-limit 산정에 포함.
- 초기 출시 지역의 POI 데이터 커버리지 정량 게이트(영업시간 채움률·좌표 확보율) — 현재 장소 상세의 '미확인' 비율을 좌우한다(README 선결 과제).

## docs/PRD/09-여행기록회고.md

**요약**: 여행 기록/회고 섹션은 여행 중 방문 체크(완료/미방문/즉석 방문)와 사진·메모 첨부, GPS 동의 기반 자동 기록을 다루며, 동일 여행에 대해 계획(plan)·실제(actual)·변경 이력(changelog) 3종 데이터를 구분 저장하는 것을 핵심 데이터 원칙(ADR-0007/0013)으로 삼는다. 당일 회고 초안을 AI로 자동 생성하되 실패·부분 데이터 시 기본 카드로 폴백하고, 여행 종료 후 지도 기반 전체 요약과 누적 방문 10곳 이상 시 여행 스타일 분석을 제공한다. 누적 기록은 다음 여행 개인화 입력으로 활용(동의 기반)되고, 마이페이지·월 캘린더에서 지난 여행을 재열람하며, SNS 공유용 정적 카드 이미지(9:16/1:1/4:5)로 외부 공유할 수 있다. 오프라인에서도 기록을 로컬 저장 후 동기화하며, 충돌 시 사용자가 버전을 선택한다.

**기능 요구사항 (27건)**:
- [S1] 여행 중 일정 실행 화면에서 각 장소를 '방문 완료' 또는 '방문 안 함/취소'로 표시할 수 있어야 한다.
- [S1] 방문 완료 체크 시 기기 시각 기준으로 실제(actual) 방문 시각을 자동 기록하고, 사용자가 이를 직접 수정할 수 있어야 한다.
- [S1] 도착(방문 시작)과 다음 이동(방문 종료) 시각으로 실제 체류 시간을 산출해 계획 체류 시간과 함께 보관해야 한다(plan/actual 구분, ADR-0013).
- [S1] 실제 체류 시간이 계획을 크게 초과해 남은 일정이 밀릴 것으로 예상되면 Plan-B 재계획 트리거(이동 지연·체력 저하)로 연결해 선제 제안해야 한다.
- [S1] 계획에 없던 장소도 '즉석 방문'으로 추가해 방문 기록에 저장할 수 있어야 한다.
- [S2] 방문한 장소에 사진 여러 장과 자유 텍스트 메모를 첨부할 수 있고, 사진 촬영/저장 시각과 장소를 함께 저장해야 한다.
- [S2] 사진 업로드 실패 시 로컬 보관 후 '업로드 대기'로 표시하고 네트워크 복구 시 자동 재시도(기본 3회), 초과 시 수동 재시도 버튼을 노출하며 메모·방문 체크는 사진 실패와 무관하게 저장되어야 한다.
- [S3] 위치정보 동의 시에만 GPS 기반 방문 기록(방문 좌표·이동 경로) 자동 저장을 선택할 수 있고, 미동의/거부 상태에서는 GPS를 수집하지 않고 좌표 없는 기록을 생성해야 한다.
- [S3] 위치 권한 거부 시 수동 체크인을 기본 기록 수단으로 제공하고, 설정에서 위치 동의 OFF 전환 시 즉시 GPS 수집을 중단해야 한다.
- [S3] GPS 신호 미수신·저정확도 시 자동 방문 판정을 보류하고 수동 체크인으로 대체해야 한다.
- [S4] 동일 여행에 대해 계획(plan)·실제(actual)·변경 이력(changelog) 3종을 구분 저장하고, 기록 화면에서 시각적으로 구분 표시하며 계획-실제 차이(미방문·추가 방문·순서 변경)를 비교 표시해야 한다(ADR-0007).
- [S4] 각 변경 이력 항목은 변경 시각, 변경 전/후 장소, 변경 사유(휴무·날씨·이동 지연 등)를 포함해야 한다.
- [S5] 등록된 숙소의 위치·체크인/아웃 날짜를 기준점으로 그날의 방문 기록을 해당 숙소·날짜에 귀속시키고, 다중 숙소(이동 숙박)는 날짜별 기준 숙소로 구분하며, 숙소 없는 날은 날짜만 기준으로 저장해야 한다(ADR-0002 계승).
- [S6] 방문 장소·사진·메모·변경 이력 기반으로 당일 회고 초안을 자동 생성하고(방문 장소 수·이동 거리·사진 수·주요 변경 포함), 생성 완료 시 인앱 카드 또는 푸시로 알려야 한다.
- [S6] 회고 생성(AI 호출) 실패·시간 초과 시 '방문 N곳·이동 Nkm·사진 N장' 기본 카드를 제공하고, 부분 데이터 상태에서는 가용 데이터만으로 생성하되 누락 항목('사진 없음' 등)을 명시하며, 당일 기록이 전무하면 안내 문구와 수동 기록 추가 버튼을 노출해야 한다.
- [S7] 사용자는 AI 회고 초안을 수정·삭제·덧붙일 수 있고, 수정본은 원본 초안과 별도 저장되어 최종 표시본으로 사용되며, 생성 실패 시에도 기본 카드 위에 직접 회고를 작성할 수 있어야 한다.
- [S8] 여행 종료 후 날짜별 기록·지도 기반 이동 경로·전체 요약(총 방문 수·총 이동 거리·총 사진 수·날짜별 하이라이트)을 생성하고, 지도 히어로(방문 순서 핀·날짜별 동선·기준 숙소 마커·핀별 사진 미리보기)를 중심으로 구성해야 한다.
- [S8] GPS 동의 구간은 GPS 경로로, 미동의 구간은 수동 체크인 장소를 순서대로 연결한 경로로 표시하며, 위치 데이터가 전혀 없으면 방문 목록으로, 요약 생성 실패 시 날짜별 기본 카드 모음으로 폴백해야 한다.
- [S9] 누적 방문 장소 수 ≥ 10곳이면 방문 패턴(카테고리 분포·평균 체류 시간·이동 반경·활동 밀도) 기반 여행 스타일 분석을 생성하고, 구체적 수치·근거 데이터와 함께 막대 차트·지도 분포점/반경 원으로 시각화해야 한다.
- [S9] 임계치 미달 시 분석을 보류하되 온보딩 취향 기반 '임시 스타일 미리보기'(정식 분석 아님 명시)와 진행 게이지(N/10곳)를 노출해야 한다.
- [S10] 누적 여행 기록(방문 카테고리·선호 지역·체류 패턴)을 다음 여행 일정 생성·장소 추천 개인화 입력으로 활용하되, 사용 항목·목적을 설정에서 안내하고 미동의 시 기록을 추천 입력에서 제외해 기본 추천만 제공해야 한다.
- [S11] 마이페이지에서 지난 여행별 날짜별 기록·사진·메모·회고·이동 경로·스타일 분석을 여행 단위 목록으로 재확인할 수 있고, 각 여행에서 3종 데이터를 구분해 볼 수 있으며, 기록이 없으면 빈 목록 대신 안내와 새 여행 생성 진입점을 노출해야 한다.
- [S12] 오프라인 상태에서도 방문 체크·사진 첨부·메모·수동 체크인을 로컬 저장하고 '동기화 대기'로 표시 후 네트워크 복구 시 자동 동기화하며, 서버 데이터와 충돌 시 사용자가 유지할 버전을 선택하게 해야 한다.
- [S12] 오프라인 구간에서는 당일 회고 자동 생성을 보류하고, 복구 후 생성하거나 기본 카드로 대체해야 한다.
- [S13] 전체 요약 또는 당일 회고 화면의 '공유'에서 대표 사진·여행 제목·기간·핵심 통계·주요 동선·TripPilot 워터마크를 담은 SNS 공유 카드를 자동 구성하고, 포맷(9:16/1:1/4:5) 선택과 캡션·해시태그 편집 후 기기 저장 또는 OS 공유 시트로 외부 공유할 수 있어야 한다.
- [S13] 대표 사진이 없으면 지도 경로·통계 기반 '사진 없는 카드' 레이아웃으로 대체하고, 여행 종료/요약 생성 전에는 공유 진입점을 비활성화·비노출해야 한다.
- [S14] 지난 여행을 월 캘린더에 기간 마킹(같은 달 다중 여행 모두 표시)으로 보여주고, 마킹 또는 하단 '지난 여행' 목록에서 특정 여행의 기록·회고로 진입하며, 월 이동 탐색이 가능하고 '기록' 탭과 마이페이지 양쪽에서 진입할 수 있어야 한다.

**비기능 요구사항 (9건)**:
- 오프라인 우선(offline-first): 방문 체크·사진·메모·수동 체크인은 네트워크 없이 기기 로컬에 저장 가능해야 하며, 복구 시 자동 동기화되어야 한다(S12).
- 침묵 실패 금지(ADR-0011): 회고/요약 생성 실패·부분 데이터 시 빈 화면 대신 기본 카드·누락 명시·대체 표시로 항상 폴백해야 한다.
- 업로드 신뢰성: 사진 업로드는 자동 재시도(기본 3회) 후 수동 재시도 경로를 제공하고, 사진 실패가 메모·방문 체크 저장을 막지 않아야 한다(부분 실패 격리).
- 개인위치정보 보호(위치정보법·ADR-0010/0017 계승): GPS 수집은 명시적 동의 기반이며, 동의 OFF 시 즉시 수집 중단, 미동의 시 좌표 없는 기록으로 동작해야 한다.
- 개인화 데이터 투명성: 개인화에 쓰이는 기록 항목·목적을 사용자에게 고지하고 옵트아웃을 지원해야 한다(S10).
- 데이터 무결성: plan/actual/changelog 3종은 상호 오염 없이 구분 저장되어야 하며, 변경 이력은 시각·전후·사유를 갖춘 감사 가능(auditable) 구조여야 한다(ADR-0007).
- 국내 지도 API 약관 제약(README 선결 과제 계승): 지도 경로·핀 표시에 쓰는 외부 지도 API 응답은 영구 캐싱 금지·실시간 호출·출처 표기 제약을 데이터 모델에 반영해야 한다.
- GPS 품질 강건성: 저정확도·신호 미수신 시 자동 판정을 보류하는 등 부정확한 측위에 대한 오판 방지 동작이 필요하다(암묵적으로 배터리·수집 빈도 관리도 요구됨).
- AI 호출 시간 제한: 회고 생성은 시간 초과 처리 경로가 있어야 하며(타임아웃 존재 전제), LLM 호출 비용·빈도 상한은 README 선결 과제와 공유된다.

**문서가 미정으로 남긴 결정 (7건)**:
- 재계획 선제 제안 트리거의 '실제 체류 시간이 계획을 크게 초과' 판정 임계값(정량 기준)이 미정 — Plan-B 섹션 트리거와의 연동 조건만 명시됨(S1).
- 회고 생성 완료 알림을 인앱 카드로 할지 푸시로 할지(또는 둘 다/조건 분기)의 선택 기준이 미정(S6).
- 회고 생성 AI 호출의 시간 초과(타임아웃) 임계값이 미정(S6 예외).
- 사진 업로드 재시도 횟수는 '기본 3회'로 명시되어 운영 조정 가능한 기본값으로 남겨짐(S2).
- 개인화에 사용하는 기록 데이터 항목의 구체 목록(방문 카테고리·선호 지역·체류 패턴 외 세부 필드)은 설정 화면 안내 요건만 있고 확정 목록은 미정(S10).
- 여행 스타일 분석의 카테고리 체계(어떤 분류 택소노미를 쓸지)와 분석 갱신 시점은 문서에 정의되지 않음(S9).
- 지도·장소 API 선정(카카오맵/네이버/TMap)은 README 차원의 운영 선결 과제로, 기록의 지도 경로·핀 구현이 이에 종속됨.

## docs/PRD/10-여행자커뮤니티.md

**요약**: 여행자 커뮤니티 섹션은 다른 여행자가 공개한 일정을 탐색 랜딩의 '여행자 일정' 진입에서 게시물형(인스타 피드 스타일)으로 둘러보고, 읽기전용 상세(시간표·지도 2뷰 재사용)를 보며, '내 여행으로 가져오기'(원본과 완전 분리된 스냅샷 복제)로 자기 날짜·숙소에 맞춰 편집하는 기능을 정의한다. 모든 콘텐츠는 기본 비공개이며 명시적 옵트인 공개(캡션·해시태그 게시물형) 시 정확 숙소 위치 권역화·절대 날짜 상대화·닉네임만 표기 등 마스킹을 적용하고, actual·GPS·개인 메모·변경 이력은 공개·복제에서 제외한다(사진은 게시 옵트인만 공개, 복제 제외). 소셜 인터랙션은 좋아요(토글·집계만, 좋아요한 사람 리스트 비노출)와 평면 댓글(대댓글·수정 없음, 본인 삭제만)로 한정하고 팔로우·구독·스크랩·평판 등 소셜 그래프는 제외한다. 신고·숨기기(양방향 차단)·금칙어·'검토 중' 노출 보류·신고 큐는 커뮤니티 출시 선결 인프라로 요구하며, SLA·자동 필터 임계 등 조정값은 운영 결정으로 남긴다.

**기능 요구사항 (36건)**:
- [스토리 1] 탐색 랜딩 '여행자 일정' 진입에서 공개 일정을 카드 목록(제목·목적지·N박M일·닉네임·핵심 통계·대표 사진·캡션 요약·해시태그·좋아요/댓글 수)의 게시물형 피드로 둘러볼 수 있어야 한다.
- [스토리 1] 목록은 여행지·동행 유형·이동 수단·예산대·기간 길이·상대 여행 시기(계절·요일 패턴)로 필터·정렬할 수 있고, 사용자 취향·여행지·저장 POI와 관련도 높은 일정을 우선 노출할 수 있어야 한다.
- [스토리 1] 본인 공개 일정은 타인 콘텐츠와 동일 비중으로 추천하지 않고, 이미 가져온 일정에는 '가져옴' 표시로 중복 복제를 방지해야 한다.
- [스토리 1] 필터 결과 0건이면 '조건을 넓혀 보세요'와 'AI로 직접 만들기' 폴백을 제시하고, 일부 로드 실패 시 가용 카드 우선 표시·누락 표기를 해야 한다(ADR-0011).
- [스토리 2] 공개 일정 상세를 시간표·지도 2뷰 읽기전용으로 재사용해 날짜별 방문지·구간 이동·체류 시간을 보여주고, 작성자 미니 프로필·프로필 보기·가져오기·신고/숨기기 진입점과 캡션·해시태그·좋아요·댓글을 노출해야 한다.
- [스토리 2] 상세에서 작성자의 actual·GPS 경로·개인 메모·정확 숙소 핀·미공개 사진은 노출하지 않고, 작성자가 비공개 전환·삭제 시(열람 중 전환 포함) '비공개/삭제된 일정' 안내를 표시해야 한다.
- [스토리 3] 공개 프로필은 닉네임·여행 스타일 요약·공개 일정 목록·집계 지표(공개 일정 수·받은 좋아요 총합)만 읽기전용으로 노출하고, 이메일·실명·연락처·비공개 일정·개인 메모와 팔로우/팔로잉·DM 진입점은 두지 않아야 한다.
- [스토리 3] 프로필 전체 비공개·계정 정지 시 '비공개 사용자' 처리하고 공개 일정 목록을 노출하지 않아야 한다.
- [스토리 4] 모든 공개 일정·프로필·댓글에 사유 선택(스팸·욕설·음란·개인정보·저작권·기타) 신고 진입점을 제공하고 신고자에게 접수 사실을 최소 1회 안내해야 한다.
- [스토리 4] 숨기기(차단)는 양방향으로 적용해 상호 둘러보기·검색·추천·복제에서 비노출하며, 숨김 목록에서 해제할 수 있어야 한다(이미 복제한 사본은 삭제하지 않음).
- [스토리 4] 일정 제목·캡션·공개 메모·닉네임·댓글에 온보딩 스토리 3과 동일한 금칙어 기준을 적용하고, 신고 누적·자동 필터 감지 시 침묵 삭제 없이 '검토 중' 노출 보류 후 작성자에게 사유를 고지해야 한다(보류 중에도 작성자 본인은 사본 조회 가능).
- [스토리 4] 신고 접수·큐, '검토 중' 노출 보류 상태, 금칙어 기본 사전, 양방향 차단의 4개 모더레이션 최소 인프라를 커뮤니티 출시 전에 구현해야 한다(출시 선결).
- [스토리 5] 모든 여행·일정·기록은 기본 비공개로 생성·저장하고, 콘텐츠 1건 단위의 명시적 '커뮤니티에 공개' 선택 전까지 타인에게 노출하지 않아야 한다(계정 일괄 공개 없음).
- [스토리 5] 공개 시 닉네임만 표기, 숙소 정확 위치의 권역(동/지역) 마스킹, 절대 날짜의 상대 표현(N박M일·계절·요일 패턴) 변환을 적용하고, 정확 위치·절대 날짜 노출은 명시적 선택 시에만 허용해야 한다(권역 마스킹 수준은 본 스토리가 정본).
- [스토리 5] 사진·메모 공개 포함 시 제3자 동의 확인 안내를 게시 전 1회 노출하고, EXIF GPS는 자동 제거(또는 제거 여부 확인)하되 자유 텍스트 PII·얼굴·위치 역추론은 best-effort 한계와 사용자 책임을 고지해야 한다.
- [스토리 5] 공개(게시) 시 캡션·해시태그를 입력할 수 있어야 하며 금칙어·마스킹 원칙을 동일 적용해야 한다.
- [스토리 5] 확정/종료 일정만 공개할 수 있으며, 공개 가능 일정이 없으면 '지금 확정하기'로 1탭 연결해 사유와 해결 경로를 함께 제시해야 한다.
- [스토리 5] 진행 중 여행의 실시간 위치·당일 동선은 자동 공개하지 않고, 여행 종료 후 또는 명시적 선택 시에만 공개를 허용해야 한다.
- [스토리 5] 공개 철회는 즉시 둘러보기·검색·홈 추천·프로필에서 제거되고, 철회 시점에 진행 중인 타인의 복제는 스냅샷 기준 완료를 허용하되 신규 진입은 차단하며, 재공개 시 이전 공개 식별자(URL)를 재사용하지 않아야 한다.
- [스토리 5] 계정 정지 시 공개 일정·프로필을 비노출 처리하고, 계정 삭제·데이터 내보내기 요청 시 공개 커뮤니티 콘텐츠도 비공개·삭제 대상에 포함해야 한다.
- [스토리 6] '내 여행으로 가져오기'는 공개 일정의 스냅샷을 사용자 소유의 새 여행으로 완전 분리(decouple) 복제하며, 원본과 사본은 상호 영향이 없어야 한다.
- [스토리 6] 복제 대상은 공개 계획(plan)·장소 목록으로 한정하고 actual·GPS·사진·개인 메모·변경 이력은 제외하며, 숙소 예약/결제는 가져오지 않는다.
- [스토리 6] 복제본에 원작자 출처(닉네임·'○○님의 일정에서 가져옴')와 복제 시점을 부착하고, 사본 재공개 시에도 최초 원작자 표기를 복제 체인 전체에 보존·전파해야 한다.
- [스토리 6] 복제 직후 가져온 사용자의 날짜·인원·등록 숙소·취향 기준으로 재정렬·재검증하도록 여행 생성·자유 편집 흐름으로 연결하고, 숙소 미등록 상태 시작도 허용해야 한다.
- [스토리 6] 비로그인 사용자의 가져오기·신고 시도는 로그인 유도 후 원래 동작을 이어가고, 동일 일정 재복제 시 기존 사본 존재를 안내하고 확인받아야 한다.
- [스토리 6] 복제 시 폐업·영업정보 없음·날짜/거점/예산 충돌 항목은 복제를 막지 않고 '확인 필요'로 분리 표기(conflict 상태)하며, 라우팅 실패 시 직선거리 추정치임을 표기해야 한다.
- [스토리 6] 원본이 복제 후 비공개·삭제되어도 사본은 유지하되 출처 표시를 '비공개로 전환된/삭제된 일정'으로 갱신해야 한다.
- [스토리 7] 마이페이지 '내가 공유한 일정' 목록(제목·공개 일시·공개 범위)에서 각 일정의 공개를 즉시 해제하고, 공개 범위 설정으로 재진입할 수 있어야 한다.
- [스토리 8] 공개 일정에 좋아요를 토글(1인 1회, 중복 카운트 금지)할 수 있고, 상세·둘러보기 카드에 누적 좋아요 수와 본인 좋아요 상태를 표시해야 한다.
- [스토리 8] 좋아요는 낙관적 업데이트로 즉시 반영하되 서버 실패 시 롤백·재시도 안내하고, 연타는 마지막 상태 1건으로 정규화해야 한다.
- [스토리 8] 좋아요한 사람 리스트는 노출하지 않고 집계 수만 표시하며, 좋아요 사용자의 계정 삭제 시 반응을 철회·집계 차감해야 한다.
- [스토리 8] 좋아요 발생 시 작성자에게 알림을 보내되 다량은 묶음('○○ 외 N명') 발송하고 self-notification·차단 관계 알림은 제외해야 한다.
- [스토리 9] 공개 일정에 단일 레벨 평면 댓글(시간순, 대댓글·수정 없음)을 작성·조회하고 본인 댓글만 삭제할 수 있으며, 글자 수 상한과 금칙어 검사를 적용해 위반 시 인라인 오류로 게시를 막아야 한다.
- [스토리 9] 댓글 목록은 '더보기'/페이지네이션으로 점진 로드하고 빈·로딩·에러 상태를 구분 표시하며, 게시·삭제 실패 시 입력 보존·재시도 진입점을 제공해야 한다.
- [스토리 9] 댓글 신고는 스토리 4 흐름을 공유하고, 도배(단시간 동일·유사 댓글 반복)는 빈도 제한·스팸 신호로 처리하며, 일정 비공개·삭제 시 댓글도 함께 비노출·정리하고 재공개 시 임의 복원하지 않아야 한다.
- [스토리 9] 댓글 작성자의 계정 삭제 시 타인 일정에 남긴 댓글도 삭제·익명화 대상에 포함해야 한다.

**비기능 요구사항 (11건)**:
- 침묵 실패 금지(ADR-0011): 모든 로딩은 스켈레톤 표시, 부분 실패는 가용 카드 우선+누락 분리 표기, 오류는 재시도 진입점과 함께 노출.
- 개인정보 보호: 기본 비공개 옵트인, 닉네임만 표기, 숙소 위치 권역화, 절대 날짜 상대화, EXIF GPS 자동 제거, 실시간 소재지 비노출, 좋아요 사용자 리스트 비노출.
- 자동 개인정보 통제의 한계 고지: 자유 텍스트 PII·사진 픽셀 얼굴·핀 분포 위치 역추론은 best-effort 처리, 게시 전 제3자 동의·책임 고지 필수.
- 데이터 무결성: 복제본은 원본과 완전 분리(스냅샷), 좋아요는 사용자당 1회 정규화(연타·어뷰징 방지), 비공개 전환 시 이전 공개 URL 재사용 금지(stale 링크 방지).
- 동시성/경쟁 상태 처리: 열람 중 비공개 전환, 복제 진행 중 철회(스냅샷 완료 허용) 등 경쟁 상태의 일관된 처리.
- 낙관적 UI 업데이트와 실패 롤백(좋아요), 입력 보존 재시도(댓글).
- 알림 폭주 방지: 좋아요·댓글 알림 묶음 발송, self-notification 제외, 차단 관계 알림 차단.
- 모더레이션 안전성: UGC 노출 전 신고 큐·검토 중 보류·금칙어 사전·양방향 차단 인프라 필수(출시 선결), 침묵 삭제 금지·작성자 사유 고지.
- GDPR류 데이터 권리: 계정 삭제·데이터 내보내기 시 공개 콘텐츠·타인 콘텐츠에 남긴 좋아요/댓글까지 삭제·익명화·집계 차감.
- 확장성 경계: 소셜 그래프(팔로우·구독·스크랩·평판) 미도입으로 시스템 복잡도 제한, 예약·결제·재고 미보유(ADR-0003·0012).
- 댓글 목록 점진 로드(페이지네이션)로 대량 댓글 성능 대응.

**문서가 미정으로 남긴 결정 (8건)**:
- 신고 처리 SLA(운영 결정 — README '여행자 커뮤니티 운영 정책').
- 자동 필터 민감도 임계값(어떤 신호·몇 건에서 '검토 중' 보류로 전환하는지).
- 스팸/도배 빈도 임계값(댓글 도배·좋아요 연타 판정 기준).
- 좋아요·댓글 알림 묶기(rate-limit) 기준(몇 건·몇 분 단위로 묶는지).
- 운영 검토 콘솔의 도구 수준(검토·보류·삭제 UI의 범위).
- 금칙어 기본 사전의 구체 목록(온보딩 스토리 3 기준 공유, 사전 내용은 별도).
- 댓글 글자 수 상한의 구체 값.
- 둘러보기 관련도 우선 노출(추천)의 구체 랭킹 로직.

## docs/PRD/11-동행공동편집.md

**요약**: 동행 공동 편집 섹션(모듈 17, ADR-0016)은 여행 소유자가 동행자를 초대해 하나의 일정을 함께 편집하는 비공개 협업 기능을 8개 유저스토리로 정의한다. MVP는 문자 단위 CRDT가 아닌 항목 단위 잠금(soft lock)·교대 편집으로 온라인 충돌을 예방하고, 잠금으로 못 막는 충돌(주로 오프라인 재연결)은 두 버전 제시 후 사용자 선택으로 해소한다. 권한은 소유자/편집자/뷰어 3단계이며, 모든 편집은 'AI 일정 생성' 스토리 7과 동일한 솔버 재검증과 변경 이력(ADR-0013)을 거친다. 여행 중 멀티유저 현장 실행(도착 체크·기록·Plan-B)은 MVP 제외로 소유자 단일 사용자 기준 동작하고 동행자는 공유 뷰로 따라가며, 본 기능 전체가 핵심 여정과 분리된 출시 게이트로 다뤄진다.

**기능 요구사항 (34건)**:
- 소유자는 여행/일정 화면의 '함께 편집하기' 진입점에서 초대 링크 또는 계정(닉네임/이메일) 지정으로 동행자를 초대할 수 있어야 한다 (스토리 1)
- 초대 시 소유자는 동행자별 권한을 편집자(편집 가능)/뷰어(읽기 전용) 중에서 지정할 수 있어야 한다 (스토리 1)
- 초대 링크에는 만료 시점이 있어야 하며, 소유자는 발급한 링크를 만료·재발급할 수 있어야 한다 (스토리 1)
- 초대 수락 시 해당 여행이 수락자의 '함께하는 여행' 목록에 나타나고 권한 범위 안에서 보기/편집이 가능해야 한다 (스토리 1)
- 비로그인 사용자가 초대 링크로 진입하면 로그인/회원가입('계정 및 온보딩' 스토리 1)으로 유도한 뒤 인증 완료 시 초대 수락을 이어가야 한다 (스토리 1 예외)
- 만료·회수된 초대 링크 진입 시 빈 화면 대신 '초대가 만료되었어요' 안내를 표시해야 한다 (스토리 1 예외, ADR-0011)
- 소유자는 편집·확정·공개·초대/권한변경/제거·공유 종료를, 편집자는 일정 편집(장소·시각·순서)만, 뷰어는 읽기만 할 수 있어야 한다 (스토리 2)
- 소유자는 동행자 권한을 편집자↔뷰어로 변경하거나 참여에서 제거할 수 있고, 변경은 해당 동행자 화면에 즉시 반영되어야 한다 (스토리 2)
- 여행 확정·커뮤니티 공개·외부 OTA 예약 등록 등 소유자 전용 동작은 편집자/뷰어에게 비활성 또는 숨김 처리해야 한다 (스토리 2)
- 권한 없는 동작 시도 시 침묵 거부 대신 '소유자만 할 수 있어요' 안내와 소유자에게 요청하는 경로를 제시해야 한다 (스토리 2 예외)
- 편집 중 뷰어로 강등되거나 제거되면 미저장 편집의 처리(저장 시도/폐기)를 명시하고 읽기 전용으로 전환해야 한다 (스토리 2 예외)
- 한 참여자가 항목(슬롯·장소·시각) 편집을 시작하면 해당 항목은 편집 중 잠금되어 타 참여자가 동시 수정할 수 없고, 저장 시 타 참여자 화면에 반영되어야 한다 (스토리 3)
- 현재 누가 일정을 보고 있는지·어떤 항목을 누가 편집 중인지 프레즌스('○○ 편집 중')를 표시해야 한다 (스토리 3)
- 동기화 지연 시 '동기화 중' 상태를 표시하고, 최신 상태 수렴 시 알려야 한다 (스토리 3)
- 실시간 채널 연결 끊김 시 '실시간 반영되지 않을 수 있어요'를 표시하고 로컬 편집을 허용하되 재연결 시 동기화로 정리해야 한다 (스토리 3 예외)
- 잠금으로 예방하지 못한 충돌(오프라인 편집의 재연결 충돌 등)은 임의 덮어쓰기 없이 두 버전을 보여주고 사용자가 선택하게 하며, 선택 후에도 직전 상태로 되돌릴 수 있어야 한다 (스토리 4)
- 서로 다른 항목에 대한 편집은 충돌 없이 병합되어야 한다 (스토리 4)
- 충돌 해소 결과는 모든 참여자에게 동일하게 수렴하고, 적용/보류된 변경을 변경 이력에 남겨야 한다 (스토리 4)
- 충돌 해소 중 추가 변경이 들어오면 마지막 확정 상태 기준으로 재해소하고 데이터를 유실하지 않아야 한다 (스토리 4 예외)
- 어떤 참여자의 장소 추가·이동·시각 변경도 영업시간·이동시간·체크인/아웃·시각 고정형 필수 방문지 충돌을 즉시 솔버로 재검증하고 위반에 경고 배지·사유를 표시해야 한다 (스토리 5)
- 사용자에게 보이는 모든 시각·이동 거리는 솔버가 검증한 값이어야 하며 공동 편집이라도 검증을 건너뛰지 않아야 한다 (스토리 5, ADR-0008·0009)
- 위반이 있는 채로 저장하려는 참여자에게 'AI 자동 보정/그대로 저장' 선택을 제시하고 그 결정과 결과를 모든 참여자가 볼 수 있어야 한다 (스토리 5)
- 라우팅·지오코딩 등 외부 API 실패 시 직선거리 추정·수동 입력으로 폴백하고 추정치임을 표기해야 한다 (스토리 5 예외, ADR-0011)
- 오프라인 중 편집은 로컬에 '동기화 대기'로 보관하고 네트워크 복구 시 공유 일정과 자동 동기화해야 한다 (스토리 6)
- 오프라인 편집이 그 사이 타 참여자 변경과 충돌하면 스토리 4의 충돌 해소로 처리하고 데이터를 임의 폐기하지 않아야 한다 (스토리 6)
- 동기화 최종 실패 시 로컬 편집을 보존한 채 '동기화 실패, 다시 시도' 진입점을 제공해야 한다 (스토리 6 예외)
- 변경 이력 항목은 변경 시각·변경자(닉네임)·변경 전/후 항목을 포함하고 여행 변경 이력(ADR-0013)에 통합되어 Plan-B 변경·회고와 함께 열람 가능해야 한다 (스토리 7)
- 소유자(또는 권한 있는 편집자)는 특정 변경을 직전 상태로 되돌릴 수 있어야 하며 되돌리기도 솔버 재검증을 거쳐야 한다 (스토리 7)
- 변경 이력 조회 실패 시 재시도 진입점과 함께 오류를 표시해야 한다 (스토리 7 예외)
- 소유자는 '공유 종료'를 할 수 있고, 종료 시 여행은 소유자 단독 소유로 돌아가며 동행자는 편집·열람 불가여야 한다 (스토리 8)
- 동행자는 스스로 '나가기'할 수 있고, 나간 뒤 그 여행이 '함께하는 여행' 목록에서 사라져야 한다 (스토리 8)
- 원본 여행은 항상 소유자에게 귀속되며 동행자에게 별도 사본을 자동 생성하지 않는다 (스토리 8)
- 소유자 계정 삭제·데이터 내보내기 시 해당 여행과 공동 편집 권한·이력도 삭제·정리 대상에 포함하고 동행자에게 공유 종료를 안내해야 한다 (스토리 8 예외)
- 여행 중 현장 실행(도착 체크·기록·사진·Plan-B)은 소유자(활성 단일 사용자) 기준으로 동작하고 동행자는 공유 뷰로 따라가며, Plan-B 확정이 타 참여자 미저장 편집과 충돌하면 스토리 4 충돌 해소·스토리 5 재검증을 적용해야 한다 (스토리 8 예외)

**비기능 요구사항 (11건)**:
- 실시간성: 한 참여자의 저장된 변경·권한 변경·프레즌스 상태가 다른 참여자 화면에 즉시(또는 준실시간) 반영되어야 한다
- 일관성/수렴성: 충돌 해소 결과가 모든 참여자에게 동일하게 수렴해야 한다 (최종 일관성 보장)
- 데이터 무손실: 충돌 해소·재해소·오프라인 동기화 과정에서 어떤 참여자의 편집도 임의 폐기되지 않아야 한다
- 침묵 실패 금지(ADR-0011): 연결 끊김·만료 링크·동기화 실패·이력 조회 실패 등 모든 실패 상태를 사용자에게 고지하고 재시도/폴백 경로를 제공해야 한다
- 오프라인 가용성: 네트워크 없이도 로컬 편집이 가능하고 재연결 시 자동 동기화되어야 한다
- 무결성: 공동 편집 경로에서도 솔버 검증을 우회할 수 없어야 하며, 사용자에게 표시되는 시각·거리는 항상 솔버 검증값이어야 한다 (ADR-0008·0009)
- 보안/접근 제어: 권한(소유자/편집자/뷰어)별 동작 제한이 서버 측에서 강제되어야 하고, 권한 회수·공유 종료가 즉시 접근 차단으로 이어져야 한다
- 초대 링크 보안: 링크에 만료 시점이 있어야 하고 소유자가 만료·재발급으로 회수할 수 있어야 한다
- 감사 가능성: 누가·언제·무엇을 바꿨는지 변경 이력이 빠짐없이 기록되어야 한다 (ADR-0013)
- 개인정보: 계정 삭제 시 공동 편집 권한·이력을 포함한 데이터 정리가 이뤄져야 한다
- 리스크 격리: 본 기능은 핵심 여정과 분리된 출시 게이트로 배포되어야 한다 (ADR-0016)

**문서가 미정으로 남긴 결정 (5건)**:
- 문자 단위 실시간 동시 타이핑·CRDT 자동 병합은 MVP 제외, 후속 과제로 남김
- 여행 중 멀티유저 동시 현장 실행(도착 체크·기록·재계획)은 MVP 제외, 후속 과제로 남김
- 공동 편집 기능의 출시 시점: 핵심 여정과 분리된 별도 출시 게이트(ADR-0016)로 다루며 구체 출시 시점은 미정
- 초대 링크의 구체적 만료 시점 값(문서는 '만료 시점이 있어야 한다'만 요구)
- '권한 있는 편집자'의 되돌리기 권한 범위(스토리 7이 소유자 외 편집자 되돌리기 허용 여부를 괄호로 열어둠)

## docs/PRD/12-알림마이설정.md

**요약**: 이 문서는 TripPilot의 알림 체계(숙소 등록/저장 완료, 여행 단계별 리마인드, Plan-B 재계획, 회고 완료, 커뮤니티 인터랙션)와 채널(푸시/인앱)·종류별 토글, 마이페이지(등록 숙소·예약 기록, 여행 목록 예정/진행/종료 구분, 여행 스타일 분석 카드), 설정(계정 정보 관리·계정 삭제, 취향 직접 수정, 위치정보 동의 부여/철회), 외부 OTA 제휴 링크 고지를 12개 유저스토리로 정의한다. 알림에는 소요시간이 아닌 이동 거리를 표시하고(ADR-0009), 외부 API 실패 시 침묵·재평가, 위치동의 철회 시 위치 비의존 트리거만 평가하는 등 폴백 원칙(ADR-0011)을 따른다. 사용자가 직접 설정한 취향이 자동 분석 스타일보다 우선하며, 스타일 분석 트리거 정본은 누적 방문 장소 10곳('여행 기록/회고' 스토리 9)이다. 제휴 링크 이동 전 수익 고지와 재표시 설정, 위치정보법상 동의·철회·기능 영향 고지(ADR-0010/0017 계승)도 포함한다.

**기능 요구사항 (26건)**:
- FR-1: 외부 OTA 예약 등록 완료 시 숙소명·위치·체크인/아웃 날짜와 "일정 생성의 출발점으로 설정" 문구를 포함한 알림을 발송해야 한다(스토리 1).
- FR-2: 단순 저장(찜) 완료 알림에는 등록과 구분하여 "일정에 반영하려면 숙소를 등록하세요" 행동 유도를 포함해야 한다(스토리 1).
- FR-3: 체크인/아웃 날짜 없는 등록에는 완료 알림 대신 "날짜를 입력해야 일정 생성이 가능합니다" 알림을 발송해야 한다(스토리 1 예외).
- FR-4: 여행 시작 D-1에 "여행 시작 전" 알림을 발송해야 한다(스토리 2).
- FR-5: 각 여행일 첫 일정 시작 전(기본 당일 오전 8시, 사용자 설정 가능)에 "당일 일정" 요약 알림을 발송해야 한다(스토리 2).
- FR-6: 개별 일정 시작 전(기본 30분, 0/15/30/60분 중 선택)에 장소명·시작 시각·직전 지점으로부터의 예상 이동 거리(소요시간 미표시, ADR-0009)를 포함한 "다음 일정" 알림을 발송해야 한다(스토리 2).
- FR-7: 등록 숙소가 없어 일정이 없으면 일정 단계 알림 대신 숙소 등록 안내를 1회만 발송해야 한다(스토리 2 예외).
- FR-8: Plan-B 트리거 감지 시 트리거 유형과 영향받는 일정을 본문에 명시하고 "대안 일정 보기" 액션으로 바로 진입할 수 있는 알림을 발송해야 한다(스토리 3).
- FR-9: 동일 일정에 대한 중복 재계획 알림을 10분 내 1회로 제한하고, 자동 트리거의 전역 빈도 상한과 민감도(적게/보통/많이)를 설정에서 조절할 수 있어야 한다(스토리 3, 정본은 'Plan-B 재계획' 스토리 2).
- FR-10: 외부 날씨/지도/휴무 API 실패 시 잘못된 알림 없이 침묵하고 다음 정상 응답 시 재평가하며, 위치동의 철회 상태에서는 이동 지연 알림을 중단하고 위치 비의존 트리거(날씨·휴무)만 평가해야 한다(스토리 3 예외).
- FR-11: 여행 종료 후 회고 자동 생성 완료 시 회고 화면 직접 진입 액션이 있는 알림을 발송하고, 기록 데이터가 전무하면 "회고를 생성하지 못했습니다" 알림으로 대체해야 한다(스토리 4).
- FR-12: 알림 종류(숙소 등록/저장, 여행 시작 전, 당일 일정, 일정 시작 전, Plan-B, 회고 완료, 커뮤니티 좋아요·댓글)별로 푸시/인앱 토글을 각각 독립 제공해야 한다(스토리 5).
- FR-13: 특정 종류의 푸시를 꺼도 인앱 알림함에는 동일 알림을 계속 누적할 수 있어야 한다(반대도 동일, 스토리 5).
- FR-14: OS 푸시 권한 거부 시 푸시 토글을 비활성 표시하고 기기 설정 이동 링크를 노출해야 하며, 알림 설정 변경은 별도 저장 버튼 없이 즉시 저장·다음 알림부터 반영해야 한다(스토리 5).
- FR-15: 모든 채널을 꺼도 보안/계정 관련 시스템 알림은 인앱으로 표시할 수 있어야 한다(스토리 5 예외).
- FR-16: 마이페이지에서 등록 숙소·외부 예약 기록을 숙소명·위치·날짜·등록 출처(OTA 예약 등록/앱 내 저장)·연결된 여행과 함께 한 곳에서 표시해야 한다(스토리 6).
- FR-17: 숙소 항목별 "일정 출발점으로 등록/해제" 전환을 제공하고, 출발점 변경 시 해당 여행 일정 재생성 여부 확인을 받아야 한다(스토리 6).
- FR-18: 등록/저장 숙소가 없으면 빈 목록 대신 숙소 탐색 진입 버튼과 안내를 노출해야 한다(스토리 6 예외).
- FR-19: 마이페이지에서 여행을 '예정/진행 중/종료'로 분류하고 대표 정보(목적지·기간·등록 숙소 수·일정 수)를 표시하며, 지난 여행에서 회고로 바로 진입할 수 있어야 한다(스토리 7).
- FR-20: 마이페이지에 여행 스타일 요약 카드(디스크립터 한 줄+선호 카테고리 칩+dot 게이지)와 분석 사용 여행 수·마지막 갱신 시점을 노출하고 상세 분석으로 진입할 수 있어야 하며, 누적 방문 장소 10곳 미만이면 현재 N곳 안내를 표시해야 한다(스토리 8).
- FR-21: 설정에서 닉네임/이메일 조회·수정, 데이터 내보내기 요청, 계정 삭제 진입을 제공해야 한다(스토리 9).
- FR-22: 계정 삭제 시 등록 숙소·여행 기록·회고·커뮤니티 공개 콘텐츠·타인 일정에 남긴 좋아요·댓글이 함께 삭제됨을 사전 고지하고 재확인을 받아야 한다(스토리 9).
- FR-23: 설정에서 수정한 여행 취향을 다음 AI 일정 생성·재계획의 입력값으로 반영하고, 직접 설정 취향이 자동 분석 스타일과 충돌하면 직접 설정 값을 우선해야 한다(스토리 10).
- FR-24: 위치정보 동의 화면에 용도(이동 지연 감지·실시간 Plan-B·주변 추천)를 명시하고, 철회 직전 영향 기능을 구체 고지·재확인해야 하며, 철회 후에도 비실시간 기능(예정 일정 알림, 날씨·휴무 트리거)은 계속 동작해야 한다(스토리 11).
- FR-25: OS 위치 권한 거부 시 앱 내 위치 동의 토글을 비활성 표시하고 기기 설정 안내를 노출해야 한다(스토리 11 예외).
- FR-26: 외부 OTA 링크 이동 직전 외부 사이트 결제 고지와 제휴 수수료 고지를 표시하고, '다시 보지 않기'를 제공하되 설정에서 언제든 다시 켤 수 있어야 하며, URL 확인 불가/연결 실패 시 이동을 중단하고 재시도 안내를 표시해야 한다(스토리 12).

**비기능 요구사항 (10건)**:
- 알림 중복 억제: 동일 일정 재계획 알림 10분 내 1회 제한 및 전역 빈도 상한 — 알림 피로 억제(스토리 3).
- 침묵 실패 금지의 역방향 적용: 외부 API 실패 시 오정보 알림을 보내지 않고 침묵 후 재평가(ADR-0011 계승, 스토리 3).
- 즉시성: 알림 설정 변경은 저장 버튼 없이 즉시 저장·즉시 반영(스토리 5).
- 법규 준수: 위치정보법상 동의 용도 명시·철회 절차·철회 후 기능 영향 고지(제16·18·19조, ADR-0010/0017 계승, 스토리 11).
- 제휴 투명성: 어필리에이트 링크 수익 고지 의무(스토리 12) — 광고/제휴 표시 규제 대응.
- 개인정보 권리 보장: 데이터 내보내기·계정 삭제 및 연쇄 삭제 고지(스토리 9) — 개인정보보호법 대응.
- 폴백 UX: OS 권한(푸시/위치) 거부 시 토글 비활성 + 기기 설정 이동 안내로 상태 불일치 방지(스토리 5·11).
- 이동 시간 미표시 원칙: 알림에 소요시간 대신 거리만 표시해 거리 기반 추정의 부정확성을 노출하지 않음(ADR-0009).
- 암묵: 리마인드 알림은 정확한 시각(오전 8시, N분 전)에 발송되어야 하므로 백그라운드/서버 스케줄링의 시간 정확도가 요구됨.
- 암묵: 자동 트리거 폴링은 비용·배터리·rate-limit 상한 내에서 동작해야 함(README 선결 과제 공유).

**문서가 미정으로 남긴 결정 (5건)**:
- 자동 재계획 트리거의 민감도(적게/보통/많이) 단계별 구체 임계값 매핑 — 정본은 'Plan-B 재계획' 스토리 2이며 운영 조정값으로 남김.
- 커뮤니티 좋아요·댓글 알림 묶기(rate-limit) 기준 — README가 운영 결정으로 명시, 정본은 '여행자 커뮤니티' 스토리 8·9.
- 제휴 링크 여부 판별의 전제인 OTA 제휴 파트너 선정 및 딥링크/수수료 트래킹 방식 — README가 운영 결정으로 명시(스토리 12의 '제휴 링크인 경우' 분기가 이에 종속).
- Plan-B 자동 트리거(날씨·휴무·이동지연) 폴링 주기와 호출 비용 상한 — README '외부 API 호출 비용·빈도·캐싱' 선결 과제로 남김.
- '당일 일정' 알림 시각의 사용자 설정 가능 범위·단위(기본 오전 8시만 명시, 선택지 미정의).

## docs/PRD/13-구현결정.md

**요약**: 이 문서는 TripPilot의 구현 결정 정본으로, B2B2C에서 B2C 여행자 전용으로 피벗한 결과인 17개 모듈(인증, 프로필, 숙소 탐색, 저장 숙소, 제휴 링크, 여행 생성, 장소 데이터, AI 일정 생성, Plan-B 감지, 일정 재계산, 날씨·맥락, 여행 기록, AI 회고, 알림, 커뮤니티, 대화형 어시스턴트, 동행 공동 편집)의 책임 경계와 17개 ADR을 정의한다. 핵심 원칙은 '등록된 숙소=일정 출발점'(ADR-0002/0004), '외부 OTA 제휴 링크 단일 공급 모델'(ADR-0003/0012), 'LLM+알고리즘 하이브리드로 솔버가 실현가능성 소유'(ADR-0008/0009/0015), '침묵 실패 금지·수동 폴백'(ADR-0011), 'plan/actual/changelog 3종 분리 저장'(ADR-0013)이다. 이동시간은 실시간 경로가 아닌 거리 기반 추정(안전계수 적용)으로 산출하며 소요시간은 UI에 노출하지 않고, 위치정보법 신고 및 국내 지도 API 약관 준수를 출시 선결로 둔다(ADR-0017). 동행 공동 편집은 항목 단위 잠금 MVP로 한정하고 별도 출시 게이트로 리스크를 격리한다(ADR-0016).

**기능 요구사항 (22건)**:
- [M1 Auth] 여행자 단일 사용자 유형에 대해 이메일·소셜 로그인, 토큰 갱신, 로그아웃, 계정 삭제를 제공한다(호스트·사용자 유형 분기 없음).
- [M2 User Profile] 여행 스타일·관심 카테고리·이동 강도·동행 유형 등 취향과 기본 정보를 저장·수정하고 AI 일정 생성의 개인화 입력값으로 제공한다.
- [M3 Accommodation Search] 지역·예산 조건으로 외부 OTA 숙소를 탐색하고 관심 숙소를 저장 목록에 담으며, 숙소 카드에 외부 OTA 예약 링크를 UI 노출한다(딥링크 생성·추적은 M5에 위임).
- [M4 Saved Accommodation] 외부 예약/앱 저장 숙소를 사용자가 직접 등록·관리하고, 위치와 체크인/아웃 날짜를 보관해 AI 일정 생성의 출발점으로 제공한다(내부 예약 상태·재고 미보유, ADR-0002).
- [M5 Affiliate Link] OTA 제휴 딥링크 생성, 아웃바운드 클릭 추적, 전환 기록·집계를 담당하며 모든 실거래를 외부 OTA로 위임한다(클릭/전환·수수료는 내부 운영 지표로만 수집하고 사용자 비노출).
- [M6 Trip Creation] 여행 단위(목적지·기간·동행·예산)를 생성·편집하고 등록/저장 숙소를 여행에 연결한다.
- [M7 Place Data] 장소의 위치·영업시간·카테고리 메타데이터를 외부 소스에서 수집·정규화해 표준화 스키마로 일정 생성 후보 풀을 제공한다(단일 외부 API 비종속, ADR-0009).
- [M8 Itinerary Generation] 등록 숙소 위치·체크인/아웃, 취향, 고정 블록, 장소 데이터·영업시간·이동시간을 입력으로 day별 일정(장소·시작/종료 시각·체류·이동시간·추천 이유)을 출력하고, 후보 부족·이동 불가 시 대체 장소 또는 일정 축소 fallback을 제공한다(LLM 점수+TOPTW/최단경로 결합).
- [M9 Plan-B Detection] 진행 중 일정과 실시간 맥락(날씨·교통·운영 상태)·휴무/지연 신호·사용자 위치/이탈 이벤트를 입력으로 트리거 발생 여부·유형(날씨 악화/휴무/지연/이탈)·영향 항목·재계획 권고를 출력하고, 감지 시 M10을 호출한다.
- [M10 Itinerary Recalculation] 남은 일정 항목·트리거 유형·숙소 기준점·현재 시각/위치·장소 후보를 입력으로 재최적화된 잔여 일정(대체 장소·갱신 시각·변경 사유)을 출력하고, 변경 불가 시 일정 보류 또는 사용자 확인 요청 fallback을 제공한다.
- [M11 Weather & Context] 여행 지역의 날씨·교통·운영 상태 등 외부 맥락 데이터를 수집해 M8·M9에 공급한다.
- [M12 Travel Archive] 완료된 여행의 방문 장소·동선·사진·메모를 타임라인으로 보관하고 조회·편집을 지원하며, plan/actual/changelog 3종을 구분 저장한다(ADR-0013).
- [M13 AI Reflection] 여행 기록 기반 회고 요약·하이라이트·다음 여행 제안을 생성하며, 마지막 등록 숙소 체크아웃 이후를 자동 생성 트리거로 삼는다(ADR-0007).
- [M14 Notification] 일정 변경, Plan-B 제안, 체크인 임박, 외부 예약 링크 리마인드를 푸시·인앱으로 전달한다.
- [M15 Community] 옵트인 공개 일정에 민감필드 마스킹(숙소 위치 권역화·절대 날짜 상대화·EXIF 제거)을 적용해 둘러보기·읽기전용 상세·공개 프로필·좋아요(토글·집계)·평면 댓글(작성/본인삭제/신고·금칙어·도배 방지·작성자 알림)을 제공하고, '내 여행으로 가져오기'는 plan 스냅샷을 원본과 분리된 사본으로 복제(출처 전파)해 M6·M8 재검증으로 넘긴다(팔로우·구독·스크랩·평판 미제공, ADR-0014).
- [M15 Community] 신고 접수, 양방향 차단(숨김), 운영 검토 보류 상태를 관리하며 침묵 삭제하지 않는다.
- [M16 Conversational Assistant] 자연어 요청과 화면 컨텍스트·대화 이력·누적 조건을 입력으로 의도 해석 후 M3·M8·M10 등을 호출하고, 솔버 검증값만으로 자연어 설명·재질의·화면 진입점 연결을 제공하며, 제안 변경은 사용자 명시적 적용+솔버 재검증을 거쳐서만 반영되고 자동 확정·저장하지 않는다(ADR-0015).
- [M16] 어시스턴트 발 일정 변경은 변경 이력(change log)으로 추적하고, LLM·솔버·외부 데이터 실패 시 비대화형·수동 폴백으로 연결한다.
- [M17 Collaborative Editing] 소유자가 동행자를 링크/계정으로 초대해 소유자/편집자/뷰어 권한으로 하나의 일정을 항목 단위 잠금(soft lock) 기반 교대 편집하게 하고, 편집 결과는 솔버 재검증과 변경 이력을 거치며, 오프라인 충돌은 사용자 선택으로 해소한다(문자 단위 CRDT 동시편집·여행 중 멀티유저 현장 실행은 MVP 제외, ADR-0016).
- [ADR-0009] 사용자에게 표시하는 이동 지표는 거리로 한정하고 차량/도보 예상 소요시간을 화면에 표시하지 않으며, 지도 이동 경로는 핀 간 직선으로 표시한다(도착 시각·체류 시간은 유지, 실제 길안내는 외부 지도 앱 위임 — ADR-0012).
- [ADR-0010] 위치정보는 사용자 동의 시에만 수집·사용하고, 미동의 시 위치 기반 기능을 비활성화하며 수동 입력 경로로 대체한다.
- [ADR-0005] 액티비티(투어·체험) 탐색·예약·일정 고정 입력은 제외하고, 고정 블록은 등록 숙소와 사용자 지정 필수 방문지로만 구성한다.

**비기능 요구사항 (11건)**:
- 침묵 실패 금지: 외부 API(지도/장소/날씨/OTA)·AI 자동 산출 실패 시 반드시 수동 수정 fallback을 제공한다(ADR-0011).
- 지도 API 폴백: 거리·좌표 산출용 외부 지도 API 응답 실패 시 직선거리 추정·수동 입력으로 폴백하고, 단일 벤더에 종속되지 않게 추상화 계층을 둔다(ADR-0009).
- 국내 지도 API 약관 준수: 응답 데이터 영구 캐싱 금지·실시간 호출·출처 표기를 데이터 모델 차원에서 반영한다(ADR-0017).
- 이동시간 추정 정확도 보정: 솔버는 거리 기반 추정 시간에 안전 마진(버퍼)을 두어 빡빡한 배치를 피하고, 자동 재계획 트리거 임계는 추정 오차 위로 설정한다(ADR-0009).
- 위치정보법 준수(출시 선결): 위치기반서비스사업 신고(제9조), 개인위치정보 약관 명시·동의·철회/일시중지·목적 달성 후 파기·수집이용 로그 보존을 구현한다(ADR-0017).
- AI 어시스턴트 보안 가드레일: 프롬프트 주입·역할 변경·시스템 지시 유출·유해/차별 요청 거절, LLM 호출 빈도 제한을 두고, 타 사용자 비공개 데이터·내부 운영 지표 비노출은 LLM 거절이 아닌 컨텍스트 주입 단계의 서버측 권한 경계로 강제한다(ADR-0015).
- 감사성/추적성: 모든 일정 변경(어시스턴트 발·공동 편집 포함)은 변경 이력(change log)으로 추적되고, plan/actual/changelog 3종이 구분 보존된다(ADR-0013).
- 테스트 커버리지: 핵심 8개 모듈(Auth, Accommodation Search, Affiliate Link, Itinerary Recalculation, Plan-B Detection, Itinerary Generation, Saved Accommodation, AI Reflection)을 테스트 대상으로 지정한다.
- 리스크 격리: 동행 공동 편집은 핵심 여정(혼자 계획→일정→Plan-B→기록)과 분리된 출시 게이트로 다룬다(ADR-0016).
- 프라이버시: 공개 콘텐츠의 정확 숙소 위치 권역화·절대 날짜 상대화·EXIF/제3자 PII 통제, 제휴 클릭/전환·수수료 지표의 사용자 비노출.
- 비용·자원 상한: LLM·외부 API 반복 호출(일정 생성·재계획·자동 트리거·어시스턴트)에 호출 빈도 상한과 캐싱 정책을 두어 비용·배터리·rate-limit을 통제한다(README 선결 과제 참조).

**문서가 미정으로 남긴 결정 (11건)**:
- 외부 OTA 제휴 파트너 선정(Booking.com/Agoda/야놀자/여기어때 등) — 운영 결정(README 참조, M5 설계의 전제).
- 딥링크 연결 방식과 제휴 수수료(어필리에이트) 트래킹·정산 구조 — 운영 결정.
- 국내 지도·장소 API 벤더 선택(카카오맵/네이버 지도/TMap 중) — '국내 우선'만 확정, 구체 벤더 미정.
- 날씨 API 선택(공공데이터포털/OpenWeather) — 미정.
- 이동 수단별 안전계수의 확정값(문서는 대중교통 ×1.5·도보 ×1.4를 '예'로만 제시)과 직선거리 우회계수 값.
- Plan-B 자동 트리거의 폴링 주기·감지 임계값·LLM 호출 빈도 상한 — 운영 결정으로 남김.
- AI 어시스턴트의 LLM rate-limit 수치·가드레일 임계·대화 이력 보관/정리 정책 — 운영 결정.
- 커뮤니티 조정값: 신고 처리 SLA, 자동 필터 민감도, 스팸/도배 빈도 임계, 알림 묶기 기준, 운영 콘솔 도구 수준 — 운영 결정.
- 위치정보사업 진입규제(제5조) 관련 현행 법령 명칭 확인 및 출시 전 위치정보 전문 법무 자문·방송미디어통신위 사전상담.
- 초기 지역 데이터 커버리지 정량 게이트의 구체 임계(POI 영업시간 필드 채움률·좌표 확보율) — 출시 전 산정 필요.
- 문자 단위 실시간 동시 편집(CRDT)·여행 중 멀티유저 현장 실행 — 후속 범위로 유보.

## docs/PRD/14-테스트.md

**요약**: 14-테스트.md는 TripPilot의 테스트 철학(좋은 테스트의 기준), 테스트 대상 11개 모듈, 테스트 불포함 범위를 정의한 테스트 결정 문서다. 핵심 기조는 LLM 문장의 표현 품질이 아니라 사용자에게 도달한 결과의 필수 조건 충족(숙소 기준점 준수, 하드 제약 무충돌, POI 그라운딩)을 검증하는 것이며, 단일 기능이 아닌 '숙소 저장→등록→AI 일정 생성→재계획'의 종단 흐름 연결성을 검증 대상으로 삼는다. 외부 API 실패 시 fallback 복구, 계정 중복 생성 방지, 알림 토글·중복 억제 등도 검증 기준에 포함된다. 불포함 범위는 항공·액티비티·인앱 결제·호스트 기능·OTA 내부 처리 정확성·LLM 문장 품질 절대 평가 등으로 명시했다. 단, 테스트 레벨/도구/환경/합격 기준 등 실행 체계는 전혀 정의하지 않았다.

**기능 요구사항 (19건)**:
- 생성된 일정이 등록 숙소의 위치·체크인/아웃 날짜 기준점을 벗어나지 않는지 검증하는 테스트가 있어야 한다 (기준 1, 모듈 3).
- '숙소 저장 → 외부 예약/저장 숙소 등록 → AI 일정 생성 → 여행 중 재계획'이 하나의 흐름으로 연결되고 각 단계 출력이 다음 단계 입력으로 정상 전달되는지 검증하는 종단(E2E) 테스트가 있어야 한다 (기준 2).
- 등록 숙소 변경(수정) 시 일정의 지리적·시간적 기준점이 갱신되는지 검증해야 한다 (기준 3, 모듈 1).
- 탐색·저장한 숙소에서 올바른 숙소·날짜 파라미터가 담긴 외부 OTA 제휴 링크로 정확히 연결되는지 검증해야 한다 (기준 4, 모듈 6).
- 재계획 결과가 남은 일정·이동 시간·고정 항목(등록 숙소 체크인/아웃, 외부 예약 항목)과 충돌하지 않는지 검증해야 한다 (기준 5, 모듈 4).
- 추천 POI가 외부 장소 데이터에 그라운딩(RAG)되어 환각 POI·폐업 장소가 일정에서 제외되는지 검증해야 한다 (기준 6, 모듈 9).
- 외부 장소 데이터가 없거나 조회 실패한 POI는 일정에 포함되지 않는지 검증해야 한다 (모듈 9).
- 소셜/이메일 가입·로그인 시 동일 이메일 계정이 중복 생성되지 않고(충돌 시 단일 계정 통합), 이메일 미인증 계정이 활성 계정으로 취급되지 않는지 검증해야 한다 (기준 7).
- 외부 API(OTA 링크/숙소 데이터/지도·경로/LLM) 실패·빈 결과 시 앱이 fallback 흐름으로 복구하고 빈 화면·오류만 남지 않는지 검증해야 한다 (기준 8).
- 지역·필터 조건 숙소 탐색이 동작하고 결과마다 OTA 제휴 링크가 노출·연결되며, 빈 결과·필터 무결과 시 안내가 노출되는지 검증해야 한다 (모듈 2).
- 재계획 필요 상황(지연·휴무·악천후 등)을 감지해 재계획 흐름으로 연결하는지 검증해야 한다 (모듈 5 Plan-B Detection).
- OTA 링크 생성 실패 시 fallback이 동작하는지 검증해야 한다 (모듈 6).
- 여행한 일정·등록 숙소·방문 항목이 기록으로 정상 보존·조회되는지 검증해야 한다 (모듈 7 Travel Archive).
- AI 회고·요약이 실제 기록 데이터에 근거해 생성되고 없는 내용을 지어내지 않는지 검증해야 한다 (모듈 8).
- Plan-B/일정/회고 알림이 채널별 토글 설정대로 발송되고 동일 트리거 반복 알림이 중복 억제 규칙대로 억제되는지 검증해야 한다 (모듈 10).
- 알림 권한 거부 시 인앱 표시로 대체되는지 검증해야 한다 (모듈 10).
- 숙소 미등록 상태에서도 여행 생성·일정 생성이 실패하지 않는지 검증해야 한다 (모듈 11, us_tripcreate 스토리 2).
- 다도시 거점 여행에서 거점별 기준점이 올바르게 처리되는지 검증해야 한다 (모듈 11).
- 취향 미설정 사용자에게 중립 기본값으로 일정이 생성되는지 검증해야 한다 (모듈 11, us_account 스토리 14).

**비기능 요구사항 (6건)**:
- 테스트는 LLM 출력의 표현 품질이 아닌 결과의 필수 조건 충족(하드 제약 준수)을 검증한다는 검증 철학 — LLM 비결정성에 견고한 속성 기반(assertion) 테스트 설계가 암묵적으로 요구된다.
- 외부 의존성(OTA·지도·경로·LLM·장소 데이터) 실패에 대한 회복탄력성(graceful degradation)이 테스트 가능해야 한다 — ADR-0011 '침묵 실패 금지' 원칙의 검증 가능성.
- 계정 무결성(동일 이메일 단일 계정) 및 인증 상태 보안(미인증 계정 비활성) 보장.
- 알림 발송의 정확성(사용자 설정 준수)과 중복 억제(스팸 방지) — 사용자 신뢰·피로도 관련 품질 속성.
- 데이터 영속성: 여행 기록이 손실 없이 보존·조회 가능해야 한다.
- AI 출력의 사실 근거성(grounding): 일정 POI와 회고 내용 모두 외부/내부 실데이터에 근거해야 하며 환각이 사용자에게 도달하지 않아야 한다.

**문서가 미정으로 남긴 결정 (7건)**:
- 테스트 레벨 구분(단위/통합/E2E)과 각 모듈별 어느 레벨에서 검증할지 미정.
- 테스트 프레임워크·도구·자동화 스택 미정.
- 외부 API(OTA·지도·LLM·날씨) 테스트 환경 전략(모킹/스텁/샌드박스/실호출) 미정.
- LLM 비결정적 출력에 대한 테스트 방법론(고정 시드·모킹·속성 검증·LLM-judge 등) 미정.
- 합격 기준·커버리지 임계·CI 게이트 조건 미정.
- Plan-B 감지 테스트의 시나리오 재현 방법(악천후·휴무 등 외부 이벤트 시뮬레이션) 미정.
- README의 '데이터 커버리지 정량 게이트(POI 영업시간 채움률·좌표 확보율 임계)' 검증을 테스트 체계에 어떻게 편입할지 미정.

## docs/PRD/15-범위제외.md

**요약**: 15-범위제외.md는 TripPilot의 Out of Scope를 확정하는 경계 문서다. 자체 예약·인앱 결제·객실 재고·호스트(B2B) 기능·항공·액티비티 전용 기능·PMS/채널매니저·환불 분쟁 자동화를 전부 제외하고, 예약·결제·재고 정합성 책임을 외부 OTA에 귀속시킨다. 커뮤니티에서는 소셜 그래프(팔로우·구독·스크랩·평판)와 독립 게시판을 제외하되 공개 일정에 대한 좋아요(토글·집계)와 평면 댓글(대댓글·수정 없음)은 포함하고, 타인 공개 일정은 원본 편집 불가·사본 복제(가져오기)로만 활용한다. 초기 출시는 한국어 단일 언어·국내(대한민국) 한정이며, 항공 지연 자동 반영·AI 자동 예약 변경·여행 영상 자동 생성도 제외한다. 일정의 고정 블록은 등록 숙소와 사용자 지정 필수 방문지로만 구성한다(ADR-0005).

**기능 요구사항 (14건)**:
- 숙소는 앱 내에서 탐색·저장만 지원하며, 실제 예약·확정은 외부 OTA 딥링크로 이동해 처리한다(앱 내 객실 점유·확정 없음).
- 숙소 결제는 전부 외부 OTA에서 이루어지며, 앱은 인앱 결제 흐름·결제 수단 등록·정산을 일절 제공하지 않는다.
- 잔여 객실 수·가격은 외부 OTA의 표기를 그대로 따르며, 앱은 객실 재고를 보유·동기화하지 않는다(재고 정합성은 OTA 책임).
- 호스트 입점·숙소/객실 등록·AI 리스팅 빌더·호스트 예약 관리·재무 대시보드·AI 경영 에이전트 등 B2B(숙박업주) 기능을 제공하지 않는다.
- 항공권 검색·예약·발권·요금 비교 기능을 제공하지 않는다.
- 액티비티(투어·체험) 탐색·상세·외부 예약 연동·일정 고정 입력 등 액티비티 전용 기능을 제공하지 않으며, 여행 일정의 고정 블록은 등록 숙소와 사용자 지정 필수 방문지로만 구성한다(ADR-0005).
- OTA 실시간 재고 동기화·정산/수수료 자동 정산 파이프라인·PMS 연동·채널매니저를 구축하지 않는다.
- 예약 환불·취소·분쟁 처리는 외부 OTA에서 처리하며 앱은 이를 중개·대행하지 않고, 숙소 품질 AI 심사·승인 자동화도 두지 않는다.
- 커뮤니티에서 팔로우·구독·스크랩 컬렉션·평판 점수 등 소셜 그래프 기능과 일정과 무관한 독립 게시판·포럼형 토론을 제공하지 않는다.
- 공개 일정에 대한 좋아요는 토글·집계 방식으로, 댓글은 공개 일정에 종속된 평면 반응(대댓글·수정 없음)으로 한정해 제공하며 신고·숨김·운영 검토 인프라를 커뮤니티와 공유한다('여행자 커뮤니티' 스토리 8·9).
- 타인이 공개한 일정의 원본 직접 편집은 불가하며 '내 여행으로 가져오기'(원본과 분리된 사본 복제)로만 활용한다.
- 내 여행에 동행자를 초대해 함께 실시간 편집하는 공동 편집은 범위에 포함된다('동행 공동 편집' 섹션).
- 항공 지연을 일정에 자동 반영하거나 AI가 사용자 대신 예약을 자동 변경·취소하는 기능을 제공하지 않는다.
- 여행 기록·회고를 영상으로 자동 편집·생성하는 기능을 제공하지 않는다.

**비기능 요구사항 (5건)**:
- 초기 출시는 국내(대한민국) 커버리지로 한정하며 해외 도시는 다루지 않는다(이후 단계적 확장) — 검색·지도·POI·일정 생성 전반에 지역 제약이 적용된다.
