### Question 2 — 재작성 수준
PRD 스토리를 어느 수준으로 재작성할까요?

A) 표준 형식 전면 재작성 (권장) — 모든 스토리를 "As a / I want / So that" + 수용 기준 형식으로 통일 재작성, requirements.md 결정 반영. 이후 단계가 참조할 단일 정본이 됨 (분량 큼)

B) 델타 반영 이관 — PRD 문구를 최대한 유지하며 옮기고, 결정(Δ1~Δ10 등)이 닿는 스토리만 수정. 작업량 적으나 형식 불균일

C) 요약 + 참조 — 스토리별 한 줄 요약과 PRD 원문 링크·결정 참조만 담은 인덱스형 문서 (최소 분량, 정본은 PRD에 분산 유지)

D) 기타 (아래 [Answer]: 태그 뒤에 직접 서술)

[Answer]: A

### Question 3 — 구성 방식
스토리를 어떤 구조로 조직할까요?

A) 사용자 여정 기반 (권장) — 핵심 여정 순서(가입·온보딩 → 탐색·저장 → 여행 생성 → 일정 생성 → 여행 전 준비 → 여행 중 실행·Plan-B → 기록·회고 → 재방문)로 에픽을 배열. 흐름 연결성 검증(14-테스트 기준 2)과 정합. 단점: 횡단 기능(알림·설정)은 별도 묶음 필요

B) 기능(모듈) 기반 — PRD 문서 구조(01~12) 그대로 유지. PRD와 1:1 대응이 쉬우나 여정 연결성이 안 보임

C) 에픽 기반 하이브리드 — 여정 순서로 에픽을 세우되 각 에픽 아래 스토리는 PRD 문서·스토리 번호를 보존해 양방향 추적

D) 기타 (아래 [Answer]: 태그 뒤에 직접 서술)

[Answer]: A

### Question 4 — 페르소나 구성
personas.md의 사용자 원형은 어떻게 구성할까요?

A) 취향 축 기반 3종 (권장) — 온보딩 취향·페이스 축으로 여행자 3원형(예: 꼼꼼 계획형 / 균형 실속형 / 즉흥 유연형)을 정의해 개인화·Plan-B·기록 기능의 관점 차이를 드러냄

B) 여정 단계 기반 2종 — 첫 여행 준비자(온보딩·탐색 중심) / 여행 중 실행자(현장·Plan-B 중심)

### Question 5 — 수용 기준 형식
각 스토리의 수용 기준(acceptance criteria)은 어떤 형식으로 작성할까요?

A) 체크리스트형 (권장) — PRD 수용 기준·예외 문구를 검증 가능한 불릿으로 정리 (PRD 원문과 대응 쉬움, 이후 테스트 지침의 입력)

C) 혼합 — 상태 전이·검증 로직이 복잡한 스토리(일정 생성·Plan-B·상태 머신)만 Given-When-Then, 나머지는 체크리스트

D) 기타 (아래 [Answer]: 태그 뒤에 직접 서술)

## 3. 구성 방식별 트레이드오프 요약 (Question 3 참고)

| 방식 | 장점 | 단점 |
|---|---|---|
| 사용자 여정 기반 | 종단 흐름(저장→등록→생성→재계획) 연결성이 구조에 드러남, E2E 테스트 시나리오 도출 용이 | 횡단 기능(알림·설정·앱셸) 배치 애매 |
| 기능(모듈) 기반 | PRD·모듈 경계와 1:1, 추적 단순 | 여정 단절, 스토리 간 의존이 안 보임 |
| 에픽 하이브리드 | 여정 구조 + PRD 번호 보존 양립 | 문서 구조 한 단계 깊어짐 |
| 페르소나 기반 | 사용자 관점 대비 명확 | 단일 사용자 유형(여행자) 앱이라 중복 다수 |
