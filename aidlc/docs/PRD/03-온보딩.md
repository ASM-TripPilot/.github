# 계정 및 온보딩

> 상위 문서: [PRD 인덱스](./README.md)

1. 사용자는 소셜 계정(Google·Apple·카카오·네이버) 또는 이메일로 회원가입 및 로그인할 수 있어야 한다.
   - 수용 기준:
     - 앱은 로그인/회원가입 화면에 4개 소셜 로그인 버튼과 함께 '회원가입'(자체 계정 가입) 진입을 노출해야 한다. 이 진입은 회원가입 화면으로 연결되며, 기존 사용자는 '이미 계정이 있으신가요? 로그인' 링크로 로그인 화면으로 전환할 수 있어야 한다. **진입 라벨은 인증 수단에 종속되지 않는 '회원가입'으로 두고, 구체 인증 수단(현재 이메일+비밀번호)은 가입 화면 단계에서 다룬다** — 향후 전화번호 등 수단이 바뀌어도 진입 라벨은 그대로 유지한다.
     - (소셜) 각 소셜 버튼 선택 시 해당 제공자의 OAuth 인증 화면으로 이동하고, 인증 성공 시 앱은 제공자가 반환한 고유 ID(sub)로 계정을 식별해 신규면 계정을 생성하고 기존이면 해당 계정으로 로그인 처리한다.
     - (이메일 회원가입) 사용자는 이메일과 비밀번호를 입력해 가입하며, 앱은 입력한 이메일로 인증 메일(인증 링크 또는 인증 코드)을 발송하고 사용자가 이메일 인증을 완료해야 계정을 활성화해야 한다. 비밀번호는 길이·복잡도 기준을 충족해야 하며, 앱은 비밀번호를 평문으로 저장하지 않고 해시로 저장해야 한다.
     - (이메일 로그인) 이미 인증된 이메일 계정은 이메일과 비밀번호로 로그인하며, 앱은 '비밀번호 재설정'(등록 이메일로 재설정 링크 발송) 경로를 제공해야 한다.
     - 모든 계정은 '여행자' 단일 유형으로 생성된다. 호스트/숙박업주 등 다른 계정 유형 선택지는 제공하지 않는다.
   - 예외:
     - (소셜) 사용자가 제공자 인증 화면에서 동의를 거부하거나 인증을 취소하면, 앱은 로그인 화면으로 복귀하고 "로그인이 취소되었습니다" 안내를 표시한다. 계정은 생성되지 않는다.
     - (소셜) 소셜 제공자 인증 서버가 오류(타임아웃·5xx)를 반환하면, 앱은 "잠시 후 다시 시도해 주세요" 안내와 재시도 버튼을 표시한다.
     - (이메일) 이미 가입된 이메일로 다시 회원가입을 시도하면, 앱은 "이미 가입된 이메일입니다. 로그인하거나 비밀번호를 재설정하세요" 안내를 표시하고 중복 계정을 생성하지 않는다.
     - (이메일) 인증 메일 발송이 실패하거나 사용자가 이메일 인증을 완료하지 않으면, 앱은 계정을 '미인증' 상태로 두고 인증 메일 재발송 동작을 제공해야 하며, 미인증 상태로는 약관 동의(스토리 2) 이후 온보딩 단계로 진행시키지 않아야 한다.
     - (계정 충돌) 동일 이메일이 소셜 가입과 이메일 가입 양쪽에 존재할 수 있는 경우, 앱은 계정을 제공자 고유 ID(sub) 또는 이메일 계정 식별자 기준으로 식별하되, 동일 이메일이 확인되면 "이 이메일은 이미 OO 로그인으로 가입되어 있습니다. OO 로그인으로 계속하시겠어요?" 안내로 기존 로그인 수단으로 유도하고 중복 계정을 신규 생성하지 않는다. 단, 제공자가 이메일을 제공하지 않거나 비공개(예: Apple 비공개 이메일 릴레이)인 경우 이메일 대조 병합은 best-effort로만 시도하고, 대조 불가 시 별도 계정으로 생성한 뒤 사용자가 나중에 수동 연결할 수 있게 해야 한다.

2. 앱은 최초 1회에 한해 서비스 이용약관·개인정보 처리방침 동의를 받아야 한다.
   - 수용 기준:
     - 최초 로그인 직후, 온보딩 진입 전에 약관 동의 화면을 1회 표시한다.
     - 필수 약관(이용약관, 개인정보 처리방침)에 모두 동의해야 '다음' 버튼이 활성화된다.
     - 마케팅 수신 동의는 선택 항목으로 표시되며, 미동의 상태로도 진행할 수 있다.
     - 동의 완료 시 앱은 동의 항목·동의 일시·약관 버전을 계정에 저장한다.
   - 예외:
     - 필수 약관 중 하나라도 미동의 상태면 '다음' 버튼은 비활성화되며, 미동의 항목을 안내한다.

3. 앱은 회원가입 시 닉네임을 자동 생성하고, 사용자는 이를 수정할 수 있어야 한다.
   - 수용 기준:
     - 계정 생성 시 앱은 중복되지 않는 닉네임을 자동 생성하여 기본값으로 채운다.
     - 사용자는 온보딩 닉네임 입력란에서 자동 생성된 값을 그대로 사용하거나 직접 수정할 수 있다. 닉네임 수정은 선택이며, 앱은 자동 생성값만으로 온보딩을 통과시키고 "닉네임은 나중에 설정에서 바꿀 수 있어요"를 명시해, 가입 직후 닉네임 유효성에서 막히지 않게 해야 한다.
     - 닉네임은 2~20자로 제한하며, 길이 위반·금칙어·중복 시 인라인 오류 메시지를 표시하고 진행을 막는다.
   - 예외:
     - 입력한 닉네임이 이미 사용 중이면 "이미 사용 중인 닉네임입니다" 메시지를 표시하고, 사용 가능한 대체 닉네임을 추천해 원탭으로 적용할 수 있게 한다(연속 중복 입력 마찰 최소화).

4. 앱은 위치 권한 사용 목적을 고지하고 권한을 요청해야 하며, 사용자는 권한을 거부할 수 있어야 한다.
   - 수용 기준:
     - 앱은 온보딩 단계에서 OS 위치 권한 다이얼로그를 바로 호출하지 않고, 위치가 실제로 쓰이는 첫 맥락('내 주변 숙소 탐색', '여행 중 실행' 첫 진입) 직전에 just-in-time으로 "현재 위치 기반 숙소 탐색·동선 안내에 사용됩니다" 사전 안내(프리프롬프트)를 1회 보인 뒤 OS 권한 다이얼로그를 호출해야 한다. 온보딩에서는 권한을 강제하지 않고 "위치는 나중에 필요할 때 물어볼게요" 정도의 고지만 두거나 생략한다.
     - 권한 허용 시 앱은 현재 위치를 숙소 탐색·동선 기준점에 활용한다.
   - 예외:
     - 사용자가 권한을 거부하면, 앱은 위치 권한 없이도 등록된 숙소 위치를 기준으로 동작하며 "현재 위치 기능은 설정에서 켤 수 있습니다" 안내를 표시한다. 위치 권한 거부만으로 온보딩을 중단하지 않는다.

5. 사용자는 온보딩에서 여행 스타일을 설정할 수 있어야 한다.
   - 수용 기준:
     - 여행 스타일 선택지를 제시한다: 휴양·관광·액티비티·미식·쇼핑·자연·문화예술 (복수 선택 가능).
     - 1개 이상 선택 시 '다음'으로 진행할 수 있다.
   - 예외:
     - 아무것도 선택하지 않고 '건너뛰기'를 누르면 해당 항목을 미설정 상태로 저장하고 다음 단계로 진행한다.

6. 사용자는 온보딩에서 예산 수준(전체 여행 예산 기준)을 설정할 수 있어야 한다.
   - 수용 기준:
     - 예산은 **전체 여행 예산(1인 기준 · 항공 제외 · 숙소+식비+활동+교통 합산)** 을 기준으로 하며, 사용자는 다음 4단계 구간 중 하나를 선택하거나 직접 총액을 입력할 수 있어야 한다. 아래 금액은 초기 국내 지역(예: 서울·부산) 기준 운영 기본값이며 지역·시즌에 따라 조정할 수 있다.
       - **저가**: 1인 약 50만 원 미만
       - **중간**: 1인 약 50만 ~ 150만 원
       - **고급**: 1인 약 150만 ~ 300만 원
       - **럭셔리**: 1인 약 300만 원 이상
     - 온보딩 시점에는 여행 기간·인원이 아직 없으므로 선택한 단계는 **러프 선호 값**으로 저장하고, 사용자가 여행을 생성할 때 정확한 전체 여행 예산 총액을 입력·수정할 수 있어야 한다. (정확한 총액 입력의 정본은 '여행 생성' 스토리 1)
     - 온보딩 화면 카피는 '대략적인 여행 씀씀이'로 부담을 낮추고(구체 금액은 작게 보조 표기), '여행을 만들 때마다 정확히 정할 수 있어요'를 명시해야 한다. '항공 제외·숙소+식비+활동+교통 합산' 같은 정확 정의는 사용자가 직접 총액을 입력할 때만 노출하고, 4구간 선택 단계에서는 강요하지 않는다.
     - 선택·입력한 예산은 (a) 전체 예산을 여행 일수로 분배해 산출한 **1박 숙소 가격대**를 숙소 탐색 결과의 가격 필터 기본값으로, (b) **1인 1일 지출 수준**을 일정 생성의 비용 가중치('AI 일정 생성' 스토리 2)로 반영해야 한다.
   - 예외:
     - '건너뛰기' 시 예산을 미설정 상태로 저장한다. (미설정 시 동작은 스토리 14 기본값 정책을 따른다 — 숙소 탐색에 가격 필터를 적용하지 않고 전체 가격대 노출)

7. 사용자는 온보딩에서 동행 유형을 설정할 수 있어야 한다.
   - 수용 기준:
     - 동행 유형 선택지를 제시한다: 혼자·커플·친구·가족(아동 동반)·부모님·반려동물 동반 (단일 또는 복수 선택은 항목 정의에 따른다).
   - 예외:
     - '건너뛰기' 시 동행 유형을 미설정 상태로 저장한다.

8. 사용자는 온보딩에서 선호 활동을 설정할 수 있어야 한다.
   - 수용 기준:
     - 선호 활동 선택지를 제시한다: 자연·역사문화·테마파크·맛집투어·카페·전시·야경·쇼핑·스포츠 (복수 선택 가능).
     - 선택 결과는 장소 추천의 우선순위에 반영된다.
   - 예외:
     - '건너뛰기' 시 선호 활동을 미설정 상태로 저장한다.

9. 사용자는 온보딩에서 이동 방식을 설정할 수 있어야 한다.
   - 수용 기준:
     - 이동 방식 선택지를 제시한다: 도보·대중교통·렌터카·택시·자전거 (복수 선택 가능).
     - 선택 결과는 일정 생성 시 장소 간 동선·이동 시간 계산의 기본 이동 수단에 반영된다.
   - 예외:
     - '건너뛰기' 시 이동 방식을 미설정 상태로 저장한다.

10. 사용자는 온보딩에서 음식 취향을 설정할 수 있어야 한다.
    - 수용 기준:
      - 음식 취향 선택지를 제시한다: 선호 음식 종류(한식·양식·일식·중식·아시안 등)를 복수 선택할 수 있다.
      - 선택한 선호 음식 종류는 맛집·음식 장소 추천에 반영된다.
    - 예외:
      - '건너뛰기' 시 음식 취향을 미설정 상태로 저장한다.

11. 사용자는 온보딩 각 단계를 건너뛰거나 이전 단계로 돌아갈 수 있어야 한다.
    - 수용 기준:
      - 모든 취향 항목(스토리 5~10 및 15 여행 페이스)에는 '건너뛰기'가 제공되며, 건너뛴 항목은 미설정 상태로 저장된다.
      - 각 단계에 진행률 표시와 '이전' 버튼을 제공하여, 이전 단계로 돌아가 응답을 수정한 뒤 다시 진행할 수 있다.
      - 약관 동의(스토리 2)와 닉네임 유효성(스토리 3)을 제외한 모든 항목은 건너뛰기로 온보딩을 완료할 수 있다.
      - 앱은 취향 설정 단계(스토리 5~10·15) 어디에서나 '나중에 설정하고 시작'(취향 일괄 건너뛰기) 단일 탈출구를 제공해, 남은 모든 취향 단계를 한 번에 건너뛰고 곧바로 첫 가치(홈·맞춤 일정)로 진입할 수 있게 해야 한다. 일괄 건너뛴 취향 항목은 미설정 상태로 저장되며, 일정 생성·추천은 스토리 14의 중립 기본값으로 동작한다. (필수 단계인 약관 동의(스토리 2)·닉네임(스토리 3)은 일괄 탈출구 대상이 아니다.)
      - 일괄 또는 단계별로 건너뛴 취향은 영구 미설정으로 방치하지 않고, 앱은 홈 대시보드와 첫 여행 생성 직전에 '취향을 알려주면 더 맞춤화돼요' 점진 설정 카드로 미설정 항목을 한두 개씩 다시 권유해, 첫 가치를 늦추지 않으면서 개인화를 점진적으로 채워야 한다. (홈 점진 카드는 '앱 셸' 스토리 2, 첫 여행 생성 직전 카드는 '여행 생성' 스토리 1, 미설정 기본값은 스토리 14와 일관)
      - 온보딩(약관·닉네임 + 취향)을 마치거나 '나중에 설정하고 시작'으로 빠져나오면, 앱은 곧바로 홈 대시보드('앱 셸' 스토리 1·2)로 진입시키고 빈 홈에서 첫 행동(여행 만들기·숙소 탐색·예약 없이 일정 받아보기)을 강조 노출해, 사용자가 다음 행동을 스스로 찾지 않게 해야 한다(온보딩→첫 가치 이음매).

12. 사용자는 설정 화면에서 닉네임과 모든 여행 취향(여행 스타일·여행 페이스·예산·동행 유형·선호 활동·이동 방식·음식 취향)을 언제든지 수정할 수 있어야 한다.
    - 수용 기준:
      - 설정 화면에서 온보딩과 동일한 선택지로 각 취향 항목을 다시 선택·해제할 수 있다.
      - 미설정 상태였던 항목도 설정 화면에서 새로 설정할 수 있다.
      - 수정·저장 즉시 변경값이 이후의 숙소 탐색 필터, 일정 생성, 장소 추천에 반영된다.
    - 예외:
      - 닉네임 수정 시 길이·금칙어·중복 규칙(스토리 3)을 동일하게 적용하며, 위반 시 저장하지 않고 인라인 오류를 표시한다.

13. 앱은 온보딩에서 수집한 취향 정보를 숙소 탐색 결과, 일정 생성, 장소 추천, 여행 스타일 분석에 활용해야 한다.
    - 수용 기준:
      - 숙소 탐색 결과는 예산(가격 필터)과 동행 유형을 기본 정렬·필터 기준으로 반영한다. (예약·결제는 외부 OTA 링크로 연결되며 앱은 탐색·저장만 제공한다.)
      - 일정 생성은 여행 스타일·선호 활동·이동 방식·음식 취향을 입력값으로 사용한다.
      - 장소 추천은 선호 활동·음식 취향을 우선순위에 반영한다.

14. 앱은 취향이 미설정된 항목에 대해 일정 생성·추천 시 기본값으로 동작해야 한다.
    - 수용 기준:
      - 미설정 취향 항목은 해당 차원에 가중치를 두지 않는 '중립' 기본값으로 처리하여, 미설정만으로 일정 생성이 실패하지 않는다.
      - 이동 방식 미설정 시 동선 계산의 기본 이동 수단은 '대중교통'으로 하되, 이동시간은 안전계수를 적용한 보수적 추정 범위로 산출한다(가장 보수적인 수단을 기본으로 둬 현장 지각을 완화 — 산출 기준 'AI 일정 생성' 섹션·ADR-0009).
      - 예산 미설정 시 숙소 탐색에 가격 필터를 적용하지 않고 전체 가격대를 노출한다.
      - 모든 취향이 미설정인 상태에서도 등록된 숙소(위치+체크인/아웃 날짜)를 출발점으로 일정 생성이 가능하다.
    - 예외:
      - 취향이 다수 미설정되어 추천 근거가 부족하면, 앱은 일정 생성 결과 상단에 "온보딩 취향을 설정하면 더 맞춤화된 일정을 받을 수 있습니다" 안내와 설정 화면 바로가기를 표시한다.

15. 사용자는 온보딩에서 여행 페이스(일정 밀도)를 설정할 수 있어야 한다.
    - 수용 기준:
      - 여행 페이스 선택지를 제시한다: 느긋하게(하루 1~2곳)·균형있게(하루 3~4곳)·빡빡하게(하루 5곳 이상) 등 일정 밀도 수준 (단일 선택).
      - 선택한 여행 페이스는 'AI 일정 생성' 스토리 3의 일정 밀도(여유형/빡빡형)·예상 체류 시간 기본값에 반영된다.
    - 예외:
      - '건너뛰기' 시 여행 페이스를 미설정 상태로 저장하며, 미설정 시 'AI 일정 생성' 기본값(중립)으로 동작한다. (스토리 14)
