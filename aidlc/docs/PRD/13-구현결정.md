# Implementation Decisions (구현 결정)

> 상위 문서: [PRD 인덱스](./README.md)

### Modules

본 모듈 목록은 피벗(B2B2C → B2C 여행자 전용) 결과를 반영한다. 원본 19개 모듈 중 5개 모듈 — 호스트 사이드 3개(Host Onboarding / AI Listing Builder / Host Reservation Management)와 외부 OTA 모델로 불필요해진 2개(Room Inventory / Payment) — 를 제거하여 14개로 축소되었고, 이후 여행자 커뮤니티 도입으로 Community 모듈(15)을, 앱 전반 AI 어시스턴트 도입으로 Conversational Assistant 모듈(16)을, 동행 공동 편집 도입으로 Collaborative Editing 모듈(17)을 추가해 현재 17개다. 핵심 일정 모듈(Itinerary Generation, Itinerary Recalculation, Plan-B Detection)은 (입력)→(출력) 계약 형식으로 기술한다.

1. **Auth (인증)** — 여행자 단일 사용자 유형의 가입/로그인/세션을 처리한다. 호스트·사용자 유형 분기 없이 이메일·소셜 로그인, 토큰 갱신, 로그아웃, 계정 삭제만 제공한다.
2. **User Profile (사용자 프로필)** — 여행자의 선호(여행 스타일·관심 카테고리·이동 강도·동행 유형)와 기본 정보를 저장·수정하고, AI 일정 생성의 개인화 입력값으로 제공한다.
3. **Accommodation Search (숙소 탐색·저장 — 링크 노출 UI)** — 지역·예산 조건으로 외부 OTA 숙소를 탐색하고, 관심 숙소를 저장 목록에 담는다. 각 숙소 카드에 외부 OTA 예약 링크를 **표시(UI 노출)**하는 책임만 가진다. 실제 딥링크 생성·추적·전환은 모듈 5(Affiliate Link)에 위임하며, 자체 예약·인앱 결제·내부 객실 재고는 제공하지 않는다.
4. **Saved Accommodation (외부 예약/저장 숙소 기록)** — 외부 OTA에서 예약했거나 앱에서 저장한 숙소를 사용자가 직접 등록·관리한다. 숙소 위치와 체크인/아웃 날짜를 보관하여 AI 일정 생성의 기준점(출발점)으로 제공한다. 내부 예약 상태나 객실 재고는 보유하지 않는다. (ADR-0002 '예약=출발점'을 '등록된 숙소=출발점'으로 계승)
5. **Affiliate Link (외부 예약 경로 — 딥링크 생성·추적·전환)** — 숙소의 실제 예약/결제로 나가는 외부 경로를 담당한다. 모듈 3이 노출한 링크에 대해 OTA 제휴 **딥링크 생성**, 아웃바운드 클릭 **추적**, **전환** 기록·집계를 관리한다(링크 UI 노출 자체는 모듈 3 책임). 앱은 예약·결제를 직접 수행하지 않고 모든 실거래를 이 모듈을 통해 외부 OTA로 위임한다.
6. **Trip Creation (여행 생성)** — 여행 단위(목적지·기간·동행·예산)를 생성·편집하고, 등록된 숙소·저장 숙소를 해당 여행에 연결한다.
7. **Place Data (장소 데이터)** — 관광지·식당·카페 등 장소의 위치·영업시간·카테고리 메타데이터를 외부 소스에서 수집·정규화하여 일정 생성 후보 풀로 제공한다.
8. **Itinerary Generation (AI 일정 생성)** — (입력) 등록 숙소 위치·체크인/아웃 날짜, 사용자 취향(모듈 2), 고정 블록(사용자 지정 필수 일정), Place Data(모듈 7), 장소 영업시간·이동시간 → (출력) day별 일정(각 항목: 장소·시작/종료 시각·체류 시간·이동 시간·추천 이유) + 단계별 fallback(후보 부족·이동 불가 시 대체 장소 또는 일정 축소). LLM 점수·설명과 특화 알고리즘(TOPTW/최단경로)을 결합한다.
9. **Plan-B Detection (재계획 트리거 감지)** — (입력) 진행 중 일정(모듈 8 출력), 실시간 맥락(모듈 11: 날씨·교통·운영 상태), 장소 휴무/지연 신호, 사용자 위치·이탈 이벤트 → (출력) 트리거 발생 여부(true/false)·트리거 유형(날씨 악화/휴무/지연/이탈)·영향받는 일정 항목·재계획 권고. 트리거 감지 시 모듈 10을 호출한다.
10. **Itinerary Recalculation (일정 재계산)** — (입력) 현재 일정의 남은 항목, Plan-B 트리거(모듈 9)·유형, 등록 숙소 기준점, 현재 시각·위치, Place Data 후보 → (출력) 재최적화된 잔여 day 일정(대체 장소·갱신된 시각·이동 시간·변경 사유) + 변경 불가 시 fallback(일정 보류 또는 사용자 확인 요청). 남은 일정을 등록된 숙소 기준점에 맞춰 재최적화한다.
11. **Weather & Context (날씨·맥락)** — 여행 지역의 날씨·교통·운영 상태 등 외부 맥락 데이터를 수집하여 일정 생성(모듈 8)과 Plan-B 감지(모듈 9)에 공급한다.
12. **Travel Archive (여행 기록)** — 완료된 여행의 방문 장소·동선·사진·메모를 타임라인으로 보관하고 사용자가 조회·편집할 수 있게 한다.
13. **AI Reflection/Summary (AI 회고·요약)** — 여행 기록을 바탕으로 회고 요약·하이라이트·다음 여행 제안을 생성한다.
14. **Notification (알림)** — 일정 변경, Plan-B 재계획 제안, 체크인 임박, 외부 예약 링크 리마인드 등을 푸시·인앱으로 전달한다.
15. **Community (여행자 커뮤니티 — 일정 공유·발견·소셜 반응)** — 사용자가 명시적으로 공개(opt-in, 기본 비공개)한 일정에 공개 범위·민감필드 마스킹(정확 숙소 위치 권역화·절대 날짜 상대화·EXIF 제거)을 적용해 다른 여행자에게 노출하고, 공개 일정 둘러보기·읽기전용 상세·작성자 공개 프로필을 제공한다. '내 여행으로 가져오기'는 공개 일정의 계획(plan) 스냅샷을 원본과 분리된 사용자 소유 사본으로 복제(원작자 출처 전파)해 모듈 6(Trip Creation)과 모듈 8(Itinerary Generation 재검증)로 넘긴다. 신고·숨김(양방향 차단)·운영 검토 보류 상태를 관리하고, 공개 일정에 대한 좋아요(토글·집계)·댓글(평면·작성/본인삭제/신고·금칙어·도배 방지·작성자 알림 트리거)과 둘러보기 피드(좋아요/댓글 수·캡션·해시태그 노출)를 제공한다. 팔로우·구독·스크랩 컬렉션·평판 점수 등 소셜 그래프는 포함하지 않으며, 여행자 간 일정 실시간 공동 편집은 모듈 17(Collaborative Editing)이 별도 담당한다. (ADR-0014)
16. **Conversational Assistant (대화형 AI 어시스턴트)** — 앱 전반에서 호출되는 대화형 오케스트레이션 계층. (입력) 사용자의 자연어 요청, 현재 화면 컨텍스트(현재 여행·등록 숙소·현재 일정 상태·진행 단계·직전 추천 결과), 대화 이력·누적 조건 → (출력) 해석된 의도에 따른 모듈 호출(Accommodation Search 모듈 3·Itinerary Generation 모듈 8·Itinerary Recalculation 모듈 10 등), 솔버가 검증한 결과의 자연어 설명, 후속 재질의 처리, 화면/모듈 진입점 연결. 자체적으로 일정 시각·실현가능성을 산출·확정하지 않고 솔버(모듈 8·10)에 위임하며, 사용자에게 보이는 시각·이동시간은 솔버 검증값만 사용한다(ADR-0008·0009·ADR-0015). 예약·결제·외부 실거래는 보유하지 않고 Affiliate Link(모듈 5)·외부 OTA로 위임한다(ADR-0003·0012). LLM·솔버·외부 데이터 실패 시 비대화형·수동 폴백으로 연결한다(ADR-0011). 어시스턴트 입력의 오·남용(프롬프트 주입·역할 변경·유해/차별 요청·과도 반복)에는 가드레일을 두며, 다른 사용자의 비공개 데이터·내부 운영 지표(제휴 수수료 등 본 섹션 비고)는 노출하지 않는다. 어시스턴트 발(發) 일정 변경은 변경 이력(change log, ADR-0013)으로 추적한다. (상세: '앱 전반 AI 어시스턴트' 섹션.)
17. **Collaborative Editing (동행 공동 편집)** — (입력) 한 여행의 소유자가 초대한 동행자(편집자/뷰어), 공유 일정의 현재 상태(모듈 6·8 출력), 각 참여자의 편집 작업·프레즌스 → (출력) **항목 단위 잠금**으로 동기화된 공유 일정, (잠금으로 못 막는) 오프라인 충돌 해소 결과, 참여자별 권한 적용. 동행자를 링크/계정으로 초대해 하나의 여행 일정을 **항목 단위 교대 편집**하게 하며(**MVP** — 문자 단위 실시간 동시편집·CRDT 자동 병합은 후속), 편집 결과는 모듈 8(Itinerary Generation 재검증)·솔버 검증을 거치고 변경 이력(ADR-0013)으로 추적한다. 권한은 소유자/편집자/뷰어로 구분하고, 오프라인 충돌은 침묵 실패 없이 사용자 선택으로 해소한다(ADR-0011). **여행 중 멀티유저 현장 실행은 본 MVP 제외**(소유자/활성 단일 사용자 기준, 동행자는 공유 뷰). 다른 사용자의 *공개* 커뮤니티 일정 원본 편집은 다루지 않는다(그건 '내 여행으로 가져오기' 복제 — 모듈 15). (ADR-0016)

**테스트 대상 모듈(핵심 8개)** — 아래 항목은 자체 나열 번호가 아니라 위 모듈 목록과 1:1로 매핑된다. 각 항목에 대응 모듈 번호를 병기한다.

1. Auth(모듈 1)
2. Accommodation Search(모듈 3) — 링크 노출(UI) 책임 범위까지
3. Affiliate Link(모듈 5) — 딥링크 생성·추적·전환 책임 범위까지
4. Itinerary Recalculation(모듈 10)
5. Plan-B Detection(모듈 9)
6. Itinerary Generation(모듈 8)
7. Saved Accommodation(모듈 4)
8. AI Reflection/Summary(모듈 13)

### 핵심 아키텍처 결정

- ADR-0001: TripPilot의 최종 구조는 **B2C 여행자 전용 플랫폼**이다. 숙박업주(호스트) 사이드 제품(입점/숙소 등록/AI 리스팅 빌더/호스트 예약관리/호스트 재무 대시보드/AI 경영 에이전트)은 제공하지 않으며, 앱의 모든 기능은 여행자 단일 사용자군을 대상으로 한다.
- ADR-0002: AI 일정 생성의 출발점은 **사용자가 외부에서 예약했거나 앱에서 저장한 뒤 '등록'한 숙소**다. 등록된 숙소의 위치와 체크인/체크아웃 날짜를 일정 생성의 기준 입력값으로 삼는다. (기존 '예약=출발점'을 '등록된 숙소=출발점'으로 계승)
- ADR-0003: 숙소 예약 경로는 **외부 OTA 제휴 링크**다. 앱은 숙소를 탐색·저장하는 기능만 제공하고, 실제 예약·결제는 외부 OTA로 이동(딥링크)하여 진행한다.
- ADR-0004: 등록된 숙소의 위치를 일정 생성의 **공간적 기준점(앵커)** 으로 사용한다. 일자별 동선·이동 시간·장소 추천은 등록 숙소 위치를 중심으로 산출한다.
- ADR-0005: **액티비티(투어·체험) 탐색·예약·일정 고정 입력 기능은 범위에서 제외한다.** 여행 일정의 고정 블록은 등록 숙소와 사용자 지정 필수 방문지로 구성하며, 액티비티 전용 탐색·외부 예약 연동·고정 입력은 두지 않는다.
- ADR-0006: Plan-B는 외부 OTA 링크 예약의 변경 행위가 아니라 **여행 중 일정 실행 보조**다. 일정 차질(지연·휴무·날씨 등) 발생 시 등록 숙소 위치와 남은 일정을 기준으로 대체 동선·장소를 재계획한다.
- ADR-0007: 여행 기록은 **방문 체크·사진·메모·위치 기반**으로 구성한다. 사용자는 일정상의 장소에 대해 방문 여부를 체크하고 사진/메모를 위치와 함께 남길 수 있다. 여행 종료 시점(마지막 등록 숙소의 체크아웃 이후)을 회고 자동 생성의 트리거로 삼는다.
- ADR-0008: 일정 생성은 **LLM 단독이 아니다**. 지도/장소/날씨 API와 규칙 기반 로직(이동 시간·운영 시간·동선 제약)을 선연산한 뒤, 그 결과를 바탕으로 일정을 구성한다. (LLM·알고리즘 하이브리드)
- ADR-0009: 장소 데이터는 **Place Data Module에서 표준화**하여 단일 외부 API에 종속되지 않는다. 공급원이 바뀌어도 상위 일정 생성 로직은 표준화된 스키마만 의존한다. **장소 간 이동시간은 거리 기반 추정으로 산출한다 — 도로 거리(가용 시) 우선, 직선거리만 있으면 우회계수로 보정하고, 이동 수단별 평균 속도에 대기·환승·신호를 흡수하는 안전계수(예: 대중교통 ×1.5·도보 ×1.4)를 적용한다. 실시간 대중교통 노선·환승·시간표·턴바이턴 경로는 제공하지 않는다.** 거리 기반 추정은 본질적으로 부정확하므로 **(i) 사용자에게 표시하는 이동 지표는 거리만으로 한정하고 차량/도보 예상 소요시간은 화면에 표시하지 않으며(실제 경로·길안내를 구현하지 않아 시간 추정이 오인을 부르므로), 지도의 이동 경로는 실제 도로가 아닌 핀 간 직선으로 표시한다 — 일정의 도착 시각·체류 시간은 유지한다, (ii) 솔버는 내부적으로 거리 기반 추정 시간으로 이동 시간창에 안전 마진(버퍼)을 두어 빡빡한 배치를 피하되 그 소요시간 값을 UI에 노출하지 않으며, (iii) 자동 재계획 트리거 임계는 추정 오차 위로 설정한다.** 실제 단계별 길안내·소요시간(대중교통 포함)이 필요하면 외부 지도 앱으로 위임한다(ADR-0012). 거리·좌표 산출용 외부 지도 API(국내 출시는 카카오·네이버·TMap 우선)는 단일 벤더에 종속되지 않게 추상화하고, 응답 실패 시 직선거리 추정·수동 입력으로 폴백한다(ADR-0011). 지도 API 약관(영구 캐싱 금지·실시간 호출·출처 표기)은 데이터 모델에 반영한다(ADR-0017).
- ADR-0010: 위치정보는 **사용자 동의 시에만** 수집·사용한다. 동의가 없으면 위치 기반 기능은 비활성화하고 수동 입력 경로로 대체한다.
- ADR-0011: 외부 API(지도/장소/날씨/OTA 링크) 실패 또는 AI 자동 산출 실패 시 **수동 수정 fallback**을 제공한다(침묵 실패 금지). 자동 산출이 불가하면 사용자가 일정/숙소/장소를 직접 편집할 수 있도록 한다.
- ADR-0012: **외부 제휴 링크화를 숙소 공급의 단일 모델**로 채택한다(ADR-0003의 상위 원칙). TripPilot은 객실 재고·예약 상태·결제를 보유하지 않고, 탐색·저장 후 외부 OTA로의 전환만 담당한다. 외부 길안내(지도 앱)·외부 예약 등 외부 위임 전반의 상위 원칙이기도 하다.
- ADR-0013: 여행 기록은 **계획(plan)·실제(actual)·변경 이력(change log) 3종을 구분 저장**한다. 등록 시점의 계획 일정, 여행 중 실제 실행 내역, 그리고 Plan-B 재계획을 포함한 변경 이력을 각각 보존해 회고·재방문 시 대조할 수 있게 한다.
- ADR-0014: **여행자 커뮤니티는 콘텐츠 공유·발견과 그에 대한 좋아요·댓글(게시물형 소셜 반응)까지 다룬다.** 다른 여행자의 공개 일정을 읽기전용으로 둘러보고(탐색 랜딩 '여행자 일정' 진입 — 전용 커뮤니티 탭 미신설, '앱 셸' 스토리 3), 사용자가 명시적으로 공개(기본 비공개·옵트인)한 일정을 '내 여행으로 가져오기'(원본과 분리된 계획(plan) 스냅샷 복제·원작자 출처 전파, ADR-0013 plan/actual/changelog 구분 계승)하며, 작성자 공개 프로필을 조회한다. 공개 일정에 대한 좋아요(토글·집계)·댓글(평면·작성/본인삭제/신고·숨김·운영 검토 인프라 공유·작성자 알림 트리거)은 커뮤니티에 포함하되, 팔로우·구독·스크랩 컬렉션·평판 점수 등 소셜 그래프는 제공하지 않으며, 여행자 간 일정 실시간 공동 편집은 별도(ADR-0016 동행 공동 편집)로 다룬다(기존 '커뮤니티 전체 제외'를 이 범위로 정밀화 — Out of Scope 갱신). 공개 콘텐츠는 정확 숙소 위치를 권역으로·절대 여행 날짜를 상대 표현으로 마스킹하고 EXIF·제3자 PII를 통제하며, 신고·숨김과 운영 검토 경로를 두되 침묵 삭제하지 않는다(ADR-0011 계승). 외부 SNS 정적 카드 공유(여행 기록/회고 섹션)와 앱 내부 동선 공개는 병렬 채널이다. (상세 정본: '여행자 커뮤니티' 섹션.)
- ADR-0015: **AI 어시스턴트는 대화형 오케스트레이션 계층이며, 실현가능성 결정은 솔버에 위임한다.** 앱 전반 AI 어시스턴트(모듈 16)는 사용자의 자연어 요청을 해석해 기존 모듈(숙소 탐색·AI 일정 생성·Plan-B 재계획)과 알고리즘 솔버(모듈 8·10)를 호출·조립·설명하는 대화 인터페이스다. 어시스턴트는 장소의 영업시간·이동시간·배치 시각 같은 실현가능성 판단과 일정의 최종 시각·순서 확정을 **단독으로 내리지 않는다** — 모든 실현가능성 검증과 사용자에게 보이는 시각·이동시간은 솔버가 소유·검증한 값이어야 하며(ADR-0008 'LLM 단독 아님'을 주로 계승·ADR-0009 Place Data 표준화를 보조로 계승, 'AI 일정 생성' 섹션 스토리 3·5·7 계승), 어시스턴트는 그 검증된 결과만 전달한다. 어시스턴트가 제안한 변경은 사용자의 명시적 적용 시 솔버 재검증을 거쳐서만 일정에 반영되며, 어시스턴트가 일정을 자동 확정·저장하지 않는다. 예약·결제·외부 실거래는 직접 수행하지 않고 외부 OTA/제휴처로 위임한다(ADR-0003·0005·0012). LLM 해석·솔버·외부 데이터 실패 시 어시스턴트는 추측으로 채우지 않고 수동/비대화형 폴백으로 연결한다(침묵 실패 금지 — ADR-0011). 어시스턴트는 자유 텍스트 LLM 입력면이므로 역할 변경·시스템 지시 유출·유해/차별 요청 거절, LLM 호출 빈도 제한, 현재 사용자 컨텍스트 외 데이터(타 사용자 비공개 데이터·내부 운영 지표) 비노출의 가드레일을 둔다. 이 비노출은 LLM 거절이 아니라 **컨텍스트 주입 단계의 권한 경계(서버측 필터)로 강제**한다 — 타 사용자 비공개 필드·내부 운영 지표를 애초에 LLM 입력에 포함하지 않아 프롬프트 주입으로 우회할 수 없게 한다('앱 전반 AI 어시스턴트' 스토리 6). (상세 정본: '앱 전반 AI 어시스턴트' 섹션.)
- ADR-0016: **여행자 간 일정 실시간 공동 편집(동행 공동 편집)을 범위에 포함한다.** 한 여행의 소유자가 동행자를 링크/계정으로 초대해 하나의 일정을 함께 실시간 편집할 수 있으며, 권한은 소유자/편집자/뷰어로 구분한다. 모든 편집은 솔버 재검증(ADR-0008·0009 — 사용자에게 보이는 시각·이동 거리는 솔버 검증값)과 변경 이력(ADR-0013)을 거치고, 동시 편집 충돌·오프라인은 침묵 실패 없이 사용자 선택으로 해소한다(ADR-0011). 이는 커뮤니티의 *공개* 일정 복제('내 여행으로 가져오기')와 구별되는, 사용자 본인 여행에 대한 비공개 협업이다(모듈 17, '동행 공동 편집' 섹션). 기존 '실시간 공동 편집 제외'(Out of Scope)를 이 범위로 갱신한다. **MVP 범위(모바일 실현 가능): 항목 단위 잠금(soft lock) 기반 계획 협업으로 한정** — 한 참여자가 한 항목을 편집하는 동안 다른 참여자는 그 항목을 동시에 못 바꾸게 해 충돌을 *예방*한다. 문자 단위 실시간 동시 편집(CRDT 자동 병합)·여행 중 멀티유저 현장 실행은 후속이며, 본 기능은 핵심 여정(혼자 계획→일정→Plan-B→기록)과 **분리된 출시 게이트**로 다뤄 리스크를 격리한다.
- ADR-0017: **위치 기반 서비스 규제 준수를 출시 선결로 둔다(국내 출시).** TripPilot은 이용자 본인의 GPS(개인위치정보)로 근처 추천·여행 중 안내·거점 추천을 제공하므로 위치정보법상 **위치기반서비스사업 신고(제9조)** 의무가 있으며(사업자등록 전제), '내 위치만 사용' 또는 '외부 API 위임'으로 면제되지 않는다(위치정보사업 진입규제(제5조 허가·등록)는 자체 측위 인프라 미보유로 불요 — 명칭은 현행 법령 확인). 개인위치정보 이용약관·동의·철회/일시중지·목적 달성 후 파기·로그 보존(ADR-0010 계승)을 구현하고, 지도는 국내 API(카카오·네이버·TMap) 우선 채택으로 국외반출 규제를 회피하되 영구 캐싱 금지·실시간 호출·출처 표기 약관을 데이터 모델에 반영한다. (상세: README '법적·규제 선결 과제'. 출시 전 위치정보 전문 법무 자문으로 최종 확인.)

> 비고: 외부 제휴 링크의 클릭/전환·수수료는 **TripPilot 내부 운영 지표로만 수집**하며 사용자에게 노출하지 않는다. 호스트 사이드(입점·숙소 등록·AI 리스팅 빌더·호스트 예약관리·호스트 재무 대시보드), 내부 객실 재고·예약 상태 관리, 직판 매출/수수료 마진 추적, 호스트용 AI 경영 에이전트 관련 결정은 B2C 피벗으로 모두 제거되었다.
