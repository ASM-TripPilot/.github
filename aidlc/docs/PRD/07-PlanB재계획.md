# 여행 중 일정 재계획 (Plan-B)

> 상위 문서: [PRD 인덱스](./README.md)

> ↔ 여행 중 **정상 실행**(장소 도착·방문 체크·현재 장소 상세·다음 이동 길찾기)은 [여행 중 현장 이용](./08-여행중현장이용.md)을 정본으로 한다. 본 섹션은 변수 발생 시 **재계획**만 다룬다.

1. 사용자는 여행 중 활성 일정 화면에서 "재계획" 버튼을 눌러 다음 사유 중 하나를 선택해 수동으로 재계획을 요청할 수 있어야 한다: 날씨 악화, 방문지 임시 휴무, 도로/교통 이동 지연, 방문지(POI) 일정 취소·예약 마감, 체력 저하/휴식 필요.
   - 수용 기준:
     - 사용자는 사유를 선택하지 않고도 "사유 없음(그냥 바꾸고 싶음)"으로 재계획을 요청할 수 있어야 한다.
     - 앱은 선택된 사유를 LLM 취향·맥락 해석 입력값으로 전달하되, 최종 후보는 솔버(OPTW/TOPTW)가 검증한 결과만 노출해야 한다.
     - 재계획 요청은 현재 진행 중인 여행에서만 가능하다. '여행 중(진행 중)'은 **여행 날짜 구간(시작일~종료일)에 진입한 상태**로 정의하며 — 등록 숙소가 있으면 그 체크인/체크아웃 구간을, 등록 숙소가 없는 여행(숙소 나중 등록·당일치기·이동일)은 여행 시작일과 현재 위치(또는 첫 일정)를 기준으로 한다. 따라서 등록 숙소가 0이어도 여행 날짜 구간 안이면 재계획을 요청할 수 있어야 한다. (ADR-0002의 '등록된 숙소=일정 출발점'은 *일정 생성 기준*으로 유지하되 '여행 중 상태 진입'과는 분리한다. 'AI 일정 생성' 스토리 11 숙소 나중 등록·'여행 기록/회고' 스토리 5 숙소 미등록 경로와 일관.)

2. 앱은 자동 트리거로 재계획을 선제 제안해야 한다 — 다음 조건을 감지하면 일정 화면 상단에 비차단(non-blocking) 알림 배너를 띄운다: (a) 다음 방문 예정지 도착 시간대 강수확률 60% 이상 또는 기상특보, (b) 다음 방문 예정지의 당일 임시 휴무/영업시간 변경 감지, (c) 현재 위치 기준 다음 방문지까지 예상 이동시간이 계획 대비 임계(기본 15분, 거리 추정 오차 마진 위로 설정·사용자 민감도 조절 — ADR-0009) 이상 초과, (d) 현재 방문 중인 POI의 실제 체류 시간이 계획 체류 시간을 임계(기본 20분, 사용자 설정 가능) 이상 초과해 다음 고정 일정(등록 숙소 복귀·시각 고정형 필수 방문지) 도착이 위협받는 경우.
   - 수용 기준:
     - 자동 트리거 알림은 제안일 뿐이며, 사용자가 명시적으로 "대안 보기"를 눌러야 재계획 흐름이 시작되어야 한다(자동으로 일정을 바꾸지 않는다).
     - 동일 사유의 알림은 동일 방문지에 대해 1회만 노출하고, 사용자가 닫으면 해당 방문지로 재노출하지 않아야 한다.
     - 앱은 자동 트리거에 **전역 빈도 상한(사유 무관, 시간당/하루 최대 N회 — 초과 시 묶어서 1회로 정리)** 을 두고, 서로 다른 사유가 동시 발화하면 개별 폭주 대신 묶어 제시해야 한다. '체류 초과'(d)는 정상 상황(여유 관람)에서 오발화할 수 있으므로, 사용자가 연속 무시하면 해당 일정에 대해 억제(학습)해야 한다. 푸시는 심각도 높은 사유(기상특보·고정 일정 도착 위협)에 한정하고 경미한 건 인앱 칩만 띄운다. 자동 트리거 민감도(적게/보통/많이)는 설정에서 조절할 수 있어야 한다('알림/마이페이지/설정' 스토리 3).
     - 자동 트리거가 사용한 외부 데이터(날씨/휴무) 출처와 감지 시각을 알림에 표기해야 한다.
     - 앱은 자동 트리거를 활성 일정 화면 내 인앱 배너/칩으로 띄우되, 사용자가 앱 밖(잠금화면·백그라운드)일 때는 비차단 푸시 알림으로도 전달할 수 있어야 하며, 푸시를 탭하면 해당 트리거의 '대안 보기' 흐름으로 진입해야 한다.
   - 예외: 자동 트리거용 외부 API(날씨/영업시간) 응답이 없으면 앱은 자동 알림을 띄우지 않고, 수동 재계획 경로만 유지해야 한다(허위 알림 금지).

3. 앱은 재계획 영향 분석 시 다음을 기준 입력값으로 사용해야 한다: 현재 위치(GPS), 현재 시각, 남은 방문 예정지 목록, 고정 제약(등록 숙소의 체크인/체크아웃 일시 및 위치), 시각 고정형 필수 방문지의 확정 시간, 각 POI의 당일 영업시간, POI 간 이동시간(장소 간 직선/도로 거리 기반 추정 — 거리×이동 수단별 속도, 실시간 대중교통 경로 미제공·ADR-0009).
   - 수용 기준:
     - 등록 숙소의 체크인/체크아웃 일시·위치는 변경 불가 고정 제약으로 취급해야 한다. Plan-B는 숙소 예약(OTA 예약) 자체를 바꾸지 않으며, 숙소를 출발·복귀 기준점으로 고정한 채 그날의 방문 일정만 재구성해야 한다. (ADR-0006: Plan-B = 여행 중 일정 실행 보조이지 숙박 예약 변경이 아니다)
     - 시각 고정형 필수 방문지(시작 시각이 확정된 항목)는 고정 제약으로 취급하고 재배치 대상에서 제외해야 한다.
     - 영향 분석은 "현재 시각 이후 남은 일정"에만 적용하고, 이미 완료된 방문지는 변경하지 않아야 한다.

4. 앱은 기존 남은 일정 및 모든 고정 제약과 충돌하지 않는 대안 후보를 2~3개 추천해야 한다.
   - 수용 기준:
     - 각 후보는 솔버 검증을 통과해야 한다: (a) 등록 숙소 체크인/체크아웃 시간 창을 위반하지 않음, (b) 모든 방문지를 해당 POI 영업시간 내에 방문, (c) 산출된 POI 간 이동시간을 반영해 시간 겹침(오버랩)이 없음, (d) 시각 고정형 필수 방문지 시작 시각 준수.
     - 후보 POI는 RAG로 그라운딩한 실재 장소여야 하며, 알고리즘 검증 레이어를 통과하지 못한(환각·폐업·좌표 불명) POI는 후보에서 제외해야 한다.
     - 후보는 사유에 부합해야 한다(예: 날씨 사유면 실내 대안 우선, 체력 저하 사유면 이동거리·방문 수가 적은 대안 우선).
   - 예외(대안 0개): 솔버가 제약을 만족하는 대안을 0개 산출하면, 앱은 빈 화면 대신 다음 폴백을 제시해야 한다 — (a) "남은 방문지 중 1개 건너뛰기" 옵션, (b) "휴식 모드로 전환" 옵션, (c) "수동 일정 수정" 진입. 이때 왜 대안이 없는지(예: 남은 시간이 부족함 / 모든 후보가 영업 종료)를 한 줄로 설명해야 한다.

5. 앱은 각 대안 후보마다 다음을 함께 표시해야 한다: 추천 이유(LLM 생성, 선택 사유와 연결된 자연어 설명), 현재 위치로부터의 예상 이동 거리와 이동 수단, 예상 체류 시간, 다음 고정 제약(숙소 복귀 또는 시각 고정형 필수 방문지)까지 여유 시간.
   - 수용 기준:
     - 예상 이동 거리·체류 시간은 추정치임을 명시하고 산출 기준(거리 기반 추정)을 표기해야 한다. 차량/도보 예상 소요시간은 표시하지 않는다(실제 경로·소요시간은 외부 지도 앱 — ADR-0009).
     - 추천 이유는 "적절히/잘" 같은 모호어 없이 구체적 근거를 포함해야 한다(예: "비 예보가 있어 도보권(약 350m)의 실내 미술관을 제안합니다. 17:30 숙소 체크인까지 90분 여유").

6. 사용자는 제시된 대안 중 하나를 선택하거나, 기존 일정을 그대로 유지하거나, 휴식 모드로 전환하는 것 중 하나를 선택할 수 있어야 한다.
   - 수용 기준:
     - "기존 유지" 선택 시 어떤 변경도 저장하지 않고 활성 일정 화면으로 복귀해야 한다.
     - "휴식 전환" 선택 시 사용자가 휴식 종료(재개) 시각을 입력하거나 즉시 재개를 선택할 수 있어야 하며, 휴식 종료 후 남은 일정을 재정렬 대상으로 다시 계산해야 한다.

7. 앱은 사용자가 대안을 선택하면 남은 일정을 자동 재정렬해야 한다.
   - 수용 기준:
     - 재정렬 결과는 솔버 검증을 통과해야 하며, 등록 숙소 체크인/체크아웃 고정 제약·각 POI 영업시간·POI 간 이동시간 어느 것과도 충돌하지 않아야 한다. 하나라도 위반하면 그 재정렬안을 확정 후보로 제시하지 않아야 한다.
     - 재정렬은 변경된 부분만 갱신하고 시각 고정형 필수 방문지 및 완료된 방문지는 그대로 보존해야 한다(warm-start 방식).
     - 재정렬로 인해 일부 방문지를 더 이상 넣을 수 없게 되면, 어떤 방문지가 제외·이월되는지 사용자에게 명시적으로 보여주고 확정 전 동의를 받아야 한다.

8. 앱은 확정 직전 변경 전/후 일정을 나란히 비교해 보여줘야 한다.
   - 수용 기준:
     - 비교 화면은 시간대별로 추가/삭제/시간이동된 항목을 시각적으로 구분 표시해야 한다.
     - 비교 화면은 변경으로 영향받는 지표(총 이동 거리 증감, 방문지 수 증감, 숙소 복귀 예정 시각 변화)를 요약해야 한다.
     - 사용자는 비교 화면에서 최종 "확정" 또는 "취소(원상복구)"를 선택할 수 있어야 한다.

9. 앱은 변경이 확정되면 변경 이력(사유, 변경 전/후 일정 스냅샷, 확정 시각, 트리거 유형=자동/수동)을 저장하고 여행 기록에 반영해야 한다.
   - 수용 기준:
     - 저장된 변경 이력은 여행 종료 후 회고 화면에서 "이날 무엇을 왜 바꿨는지" 타임라인으로 열람 가능해야 한다.
     - 변경 이력 저장은 등록 숙소 정보를 수정하지 않으며, 숙소는 고정 기준점으로 유지된 채 일정 변경분만 기록해야 한다.

10. 사용자는 위치 권한이 거부되었거나 GPS를 사용할 수 없을 때 현재 위치를 수동으로 입력(지도 핀 지정 또는 장소 검색)하여 재계획을 계속 진행할 수 있어야 한다.
    - 수용 기준:
      - 위치 권한 거부 감지 시 앱은 재계획을 차단하지 않고 즉시 수동 위치 입력 화면으로 안내해야 한다.
      - 수동 입력된 위치는 영향 분석·이동시간 산출의 현재 위치 기준점으로 사용되며, 추정 출발지임을 화면에 표기해야 한다.
      - 사용자가 위치 입력도 거부·생략하면, 앱은 마지막 완료 방문지 또는 등록 숙소 위치를 기준점으로 사용하고 그 가정을 명시해야 한다.

11. 앱은 외부 API 오류(날씨·영업시간·길찾기·POI 검색 응답 실패 또는 시간 초과) 발생 시 수동 일정 수정 화면으로 전환해야 한다.
    - 수용 기준:
      - 폴백 화면에서 사용자는 남은 방문지의 순서를 직접 재배열하고, 방문지를 삭제하거나, 방문 예정 시각을 직접 입력할 수 있어야 한다.
      - 수동 수정 중에도 등록 숙소 체크인/체크아웃 고정 제약은 위반 불가로 유지되며, 사용자가 이를 침범하는 입력을 하면 경고하고 저장을 막아야 한다.
      - 어떤 외부 데이터가 누락된 채 수동 모드로 전환되었는지(예: 이동시간 자동 계산 불가)를 화면에 표기해야 한다.
      - 외부 API가 복구되면 앱은 수동 수정 결과를 유지한 채 자동 검증·자동 재계획 기능을 다시 활성화할 수 있어야 한다.

12. 사용자는 재계획 사유를 선택한 뒤, 대안을 얻는 방식을 **AI에게 맡기기(완전 AI) / 직접 수정** 중에서 선택할 수 있어야 한다.
    - 수용 기준:
      - **AI에게 맡기기**: 앱은 문제 시점 이후 남은 일정 전체에 대한 대안을 솔버 검증을 거쳐 **여러 후보(2~3개)로 제시**하고, 사용자가 그중 하나를 택일하게 해야 한다(스토리 4와 일관 — AI가 선택지를 충분히 제시).
      - **직접 수정**: 앱은 외부 API 오류가 아니어도 사용자가 원할 때 수동 일정 수정(스토리 11과 동일한 편집 기능: 순서 재배열·삭제·시각 직접 입력)으로 진입할 수 있게 해야 한다.
    - 예외:
      - 어느 방식에서도 등록 숙소 체크인/체크아웃 고정 제약은 위반 불가로 유지되어야 한다(스토리 3·11과 일관).
      - 'AI에게 맡기기'에서 제시할 후보가 0건이거나 모든 후보가 솔버 검증을 통과하지 못하면, 앱은 빈 화면 대신 스토리 4 예외(대안 0개)의 폴백 — "남은 방문지 1개 건너뛰기 / 휴식 모드로 전환 / 수동 일정 수정" — 을 제시하고 사유를 한 줄로 설명해야 한다(두 분기 모두 0건·예외 경로 대칭).

13. 사용자는 활성 일정 화면에서 계획된 동선과 실제 이동 경로(GPS 발자취)를 지도 위에서 토글해 비교할 수 있어야 한다.
    - 수용 기준:
      - 앱은 지도에 계획 동선(흐리게)과 실제 이동 경로(지나온 GPS 좌표 점선·점)를 함께 또는 토글로 보여주고, 누적 실제 이동 거리·걸음 수를 표시할 수 있어야 한다.
      - 이동 거리·걸음 수는 추정치임을 표기해야 한다(GPS 정확도·실내 구간 한계).
    - 예외:
      - 위치 권한이 없거나 GPS를 쓸 수 없으면 실제 경로 레이어를 비활성화하고 사유를 표기해야 한다(스토리 10과 일관). 이때 지도는 계획 동선만 표시한다.
