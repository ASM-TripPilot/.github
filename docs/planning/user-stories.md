# TripPilot 유저스토리 (128개)

> 출처: aidlc/aidlc-docs/inception/user-stories/stories.md · aidlc/aidlc-docs/inception/application-design/unit-of-work-story-map.md · aidlc-docs에서 2026-07-05 추출 · 이후 본 문서가 정본이다.

TripPilot의 전체 유저스토리 128개(1차 출시 범위 102 + 후속 26)를 에픽(E1~E12)·유닛(U1~U11)별로 재구성한 단일 정본이다. 각 스토리는 INVEST 원칙을 따르며 체크리스트형 수용 기준을 가진다. 스토리마다 원 PRD 스토리(`원 스토리`)와 확정 결정(`관련 결정` — D01~D38 결정, Δ1~Δ10 PRD 델타, N1~N8 신규 요구사항, G### 게이트/규칙, ADR-000x, C1~C13 계약)의 양방향 추적을 보존한다. PRD 원문과 충돌 시 본 문서가 우선한다.

페르소나는 [personas.md](./personas.md)를 정본으로 한다 — P1 지유(꼼꼼 계획형) · P2 민준(균형 실속형) · P3 하람(즉흥 유연형) · 운영자. 에픽 개요는 [epics.md](./epics.md), 개발 순서(유닛)는 [units.md](./units.md), 유닛 상세는 units/ 하위 문서를, 용어는 [glossary.md](./glossary.md)를 참조한다.

## Epic 1 — 가입·온보딩 (E1 / U1, 18개)

여행자가 계정을 만들고 약관에 동의하며 취향 7종을 (선택적으로) 설정해, 첫 가치(홈·맞춤 일정)까지 막힘없이 도달하게 하는 에픽. 전 스토리 `[1차]`·유닛 U1 배정.

### US-E1-01 소셜·이메일 회원가입 및 로그인 `[1차]` · U1
**요지**: 여행자로서 소셜(Google·Apple·카카오·네이버)/이메일로 가입·로그인하고 싶다 — 익숙한 수단으로 빠르게 시작하려고.
**원 스토리·결정**: PRD 03 스토리 1 · D22, G22
**수용 기준**:
- 로그인/회원가입 화면에 소셜 4개 버튼과 '회원가입'(수단 비종속 라벨) 진입을 노출하고, '이미 계정이 있으신가요? 로그인' 링크로 전환할 수 있다. 앱은 로그인 필수이며 비로그인 진입은 초대·공유 딥링크 수신에 한정한다 [D22/Δ5].
- 소셜 인증 성공 시 제공자 고유 ID(sub)로 신규 생성/기존 로그인을 분기하고, 동의 거부·취소 시 "로그인이 취소되었습니다" 안내로 로그인 화면 복귀, 제공자 오류 시 재시도 버튼을 표시한다.
- 이메일 가입은 인증 링크 방식(유효 24시간·재발송 분당 1회/일 5회 제한)이며, 인증 완료 전 '미인증' 상태로 약관 동의 이후 온보딩으로 진행시키지 않는다 [G22].
- 비밀번호는 8자 이상+영문/숫자 포함+유출 목록 검증을 통과해야 하며 해시로 저장한다 [G22].
- 이미 가입된 이메일 재가입 시 중복 계정을 만들지 않고 로그인/비밀번호 재설정을 유도하며 재설정 링크 경로를 제공한다.
- 동일 이메일이 소셜/이메일 양쪽에 확인되면 기존 수단으로 유도하고, 이메일 대조 불가(Apple 비공개 릴레이 등)에만 별도 계정 생성(1차 수동 연결 미제공, CS 처리 [G20]).
- 모든 계정은 '여행자' 단일 유형으로 생성되며 다른 유형 선택지는 없다.
**유닛 비고**: 소셜 4종 어댑터+이메일 인증(G22). 계정 수동 연결은 1차 미제공(G20 — CS 처리).

### US-E1-02 최초 1회 필수 약관 동의 `[1차]` · U1
**요지**: 여행자로서 최초 로그인 직후 약관에 한 번만 동의하고 싶다 — 이후 반복 동의 없이 이용하려고.
**원 스토리·결정**: PRD 03 스토리 2 · N8
**수용 기준**:
- 최초 로그인 직후, 온보딩 진입 전에 약관 동의 화면을 1회 표시한다.
- 필수 약관(이용약관·개인정보 처리방침·위치기반서비스 이용약관 — US-E1-17)에 모두 동의해야 '다음'이 활성화되며, 미동의 시 미동의 항목을 안내한다.
- 마케팅 수신 동의는 선택 항목으로 표시되고 미동의로도 진행 가능하며, 이후 설정에서 철회할 수 있다 [N8].
- 동의 완료 시 동의 항목·일시·약관 버전을 계정에 저장한다(동의 증적).
**유닛 비고**: 동의 증적(항목·일시·버전) 저장. 마케팅 선택 동의는 US-E09-14(U8 철회 토글)와 연계.

### US-E1-03 닉네임 자동 생성과 수정 `[1차]` · U1
**요지**: 여행자로서 자동 생성 닉네임을 그대로 쓰거나 원할 때만 고치고 싶다 — 가입 직후 닉네임 때문에 막히지 않으려고.
**원 스토리·결정**: PRD 03 스토리 3 · G23
**수용 기준**:
- 계정 생성 시 '형용사+여행명사+2자리 숫자' 패턴으로 중복 없는 닉네임을 자동 생성해 기본값으로 채우고 충돌 시 재추첨한다 [G23].
- 자동 생성값만으로 온보딩을 통과할 수 있으며 "닉네임은 나중에 설정에서 바꿀 수 있어요"를 명시한다.
- 닉네임은 2~20자로 제한하고 길이 위반·금칙어·중복 시 인라인 오류로 진행을 막는다. 금칙어 기본 사전은 가입 시점 검증부터 적용 [G23].
- 입력 닉네임이 중복이면 "이미 사용 중인 닉네임입니다"와 함께 대체 닉네임을 추천해 원탭 적용한다.
**유닛 비고**: 금칙어 검증은 C3(U1 소유) — P8 사전 확보 전 임시 최소 사전.

### US-E1-04 위치 권한 just-in-time 고지·요청 `[1차]` · U1
**요지**: 여행자로서 위치 권한을 실제 필요한 순간에 이유와 함께 요청받고 싶다 — 온보딩에서 불필요한 권한 요구 없이 진행하려고.
**원 스토리·결정**: PRD 03 스토리 4 · N2
**수용 기준**:
- 온보딩 단계에서는 OS 위치 다이얼로그를 호출하지 않고, 위치가 실제 쓰이는 첫 맥락('내 주변 숙소 탐색'·'여행 중 실행' 첫 진입) 직전에 목적 프리프롬프트를 1회 보인 뒤 OS 다이얼로그를 호출한다.
- 허용 시 현재 위치를 숙소 탐색·동선 기준점에 활용한다.
- 거부 시 등록 숙소 위치 기준으로 동작하고 "현재 위치 기능은 설정에서 켤 수 있습니다"를 안내하며 거부만으로 온보딩을 중단하지 않는다.
- OS 위치 권한은 앱 내 위치기반서비스 법정 동의·GPS 기록 옵트인(US-E1-17)과 독립된 층으로 관리한다 [N2/D34].
**유닛 비고**: U1은 프리프롬프트 프레임·권한 상태 관리까지. OS 다이얼로그 실제 발화 지점은 U3('내 주변')·U6(여행 중 첫 진입).

### US-E1-05 여행 스타일 설정 `[1차]` · U1
**요지**: P1 지유(꼼꼼 계획형)로서 온보딩에서 내 여행 스타일을 고르고 싶다 — 이후 일정·추천이 내 스타일에 맞게 시작되도록.
**원 스토리·결정**: PRD 03 스토리 5
**수용 기준**:
- 여행 스타일 선택지(휴양·관광·액티비티·미식·쇼핑·자연·문화예술)를 복수 선택으로 제시한다.
- 1개 이상 선택 시 '다음'으로 진행할 수 있다.
- 선택 없이 '건너뛰기' 시 미설정으로 저장하고 다음 단계로 진행한다.
**유닛 비고**: 취향 7종 중 1축(M2).

### US-E1-06 예산 수준 설정 (여행 전체 총액 기준) `[1차]` · U1
**요지**: P2 민준(균형 실속형)으로서 부담 없이 대략적인 씀씀이를 알려주고 싶다 — 예산대에 맞는 숙소·일정을 추천받으려고.
**원 스토리·결정**: PRD 03 스토리 6 · D26, Δ2, G26
**수용 기준**:
- 예산은 **여행 전체 총액(항공 제외 · 숙소+식비+활동+교통 합산)** 기준이며 4단계 구간(저가/중간/고급/럭셔리) 선택 또는 직접 총액 입력을 제공한다. 1인·1일 값은 인원·일수로 나눈 파생 표기로만 다룬다 [D26/Δ2].
- 온보딩 값은 러프 선호로 저장하고 정확한 총액은 여행 생성 시 입력·수정한다. 직접 입력 시 총액 원값+매핑 구간을 함께 저장하며 1박 가격대 환산은 여행 생성 시점(일수 확보 후) 수행 [G26].
- 화면 카피는 '대략적인 여행 씀씀이'로 부담을 낮추고, '항공 제외·합산' 등 정확 정의는 직접 총액 입력 시에만 노출한다.
- 저장 예산은 (a) 숙소 탐색 가격 필터 기본값, (b) 일정 생성 비용 가중치에 반영된다.
- '건너뛰기' 시 미설정 저장하며 숙소 탐색에 가격 필터를 적용하지 않고 전체 가격대를 노출한다.
**유닛 비고**: D26/Δ2 전체 총액 기준. 1박 가격대 환산은 여행 생성 시점(U4, G26).

### US-E1-07 동행 유형 설정 `[1차]` · U1
**요지**: 여행자로서 누구와 여행하는지 알려주고 싶다 — 동행에 맞는 숙소·장소를 추천받으려고.
**원 스토리·결정**: PRD 03 스토리 7 · G19
**수용 기준**:
- 기본 동행 유형(혼자·커플·친구·가족(아동 동반)·부모님)은 **1개 단일 선택**으로 제시한다 [G19].
- '반려동물 동반'은 기본 유형과 분리된 별도 불리언(토글)으로 제공한다 [G19].
- '건너뛰기' 시 동행 유형을 미설정으로 저장한다.
**유닛 비고**: 단일 선택+반려동물 보조 불리언(G19).

### US-E1-08 선호 활동 설정 `[1차]` · U1
**요지**: 여행자로서 좋아하는 활동을 고르고 싶다 — 장소 추천이 내 관심사 우선으로 정렬되도록.
**원 스토리·결정**: PRD 03 스토리 8
**수용 기준**:
- 선호 활동 선택지(자연·역사문화·테마파크·맛집투어·카페·전시·야경·쇼핑·스포츠)를 복수 선택으로 제시한다.

## Epic 2 — 앱 셸·홈·내비게이션 (E2 / U2, 6개)

스플래시 분기·홈 대시보드·하단 5탭·탭바 노출 규칙·장소 우선 진입으로, 여행자가 앱 어디서든 자신의 여행 상태와 다음 행동에 한 번에 도달하게 하는 에픽.

### US-E2-01 스플래시 진입 분기 `[1차]` · U2
**요지**: 여행자로서 앱을 켜면 내 상태에 맞는 화면으로 바로 이동하고 싶다 — 매번 로그인·설정을 다시 거치지 않으려고.
**원 스토리·결정**: PRD 01 스토리 1 · D22, G5, N3
**수용 기준**:
- 콜드 스타트 시 로고·서비스명 스플래시를 표시하며 저장 세션 토큰·약관 동의 버전을 확인한다. 스플래시에는 하단 탭바를 노출하지 않는다.
- 유효 세션이 없으면 로그인, 세션은 있으나 약관 미동의/온보딩 미완료면 해당 단계, 모두 완료면 홈 대시보드로 진입시킨다. 로그인 필수, 비로그인 진입은 초대·공유 딥링크 수신에 한정 [D22/Δ5].
- 중대 개정으로 '재동의 필요' 상태면 재동의 화면으로 분기한다 [N3] (버전 게이트는 US-E2-06).
- 세션 검증 네트워크 호출 타임아웃 3초, 초과 시 로컬 토큰 미만료면 홈 진입+백그라운드 재검증, 만료면 로그인 분기 [G5].
- 검증 지연 중 스플래시에 진행 표시를 유지하고 온라인 복구 시 재검증한다(침묵 실패 금지).
**유닛 비고**: U1 세션 검증·재동의 플래그 API 소비(5분기×3초 타임아웃 G5) — U1 공급 계약의 첫 소비자.

### US-E2-02 홈 대시보드 `[1차]` · U2
**요지**: 여행자로서 홈에서 내 여행 상태와 다음 행동을 한눈에 보고 싶다 — 무엇을 해야 할지 스스로 찾지 않으려고.
**원 스토리·결정**: PRD 01 스토리 2 · D21, G2, G3
**수용 기준**:
- 홈은 진행 중/예정 여행 카드(목적지·기간·D-day·등록 숙소 수·일정 유무·진행률), 빠른 액션('여행 만들기' 단일), '지금 인기 있는 장소', '여행 추억 다시 보기'를 한 화면에 표시하고, 상단에 워드마크·알림 진입점(읽지 않은 알림 표시)을 노출한다.
- 여행 카드 진행률은 여행 중=방문 체크 완료 비율, 여행 전=진행률 없이 D-day만 표시한다 [G3].
- 진행 중 여행은 날짜 겹침 차단으로 **항상 최대 1개**이며 [D21/Δ3], 있으면 홈 상단에 그날 활성 일정으로 진입하는 카드를 D-day·진행률과 함께 강조 노출한다(복수 활성 전환 UI 없음).
- '지금 인기 있는 장소'는 최근 7일 저장+방문 가중합을 일 1회 배치 집계한 결과다 [G2]. '지금 뜨는·내 취향 여행 기록' 카드는 그 전까지 미노출(후속: 커뮤니티).
- 여행·숙소 없는 첫 사용자에게는 빈 대시보드 대신 '여행 만들기'와 '예약 없이 AI로 먼저 일정 받아보기', '가고 싶은 곳 먼저 저장'(US-E2-05) 진입을 노출한다.
- 외부 데이터 로딩 중 로딩 상태를 표시하고 일부 실패해도 가용 카드만 우선 표시한다(침묵 실패 금지).
- 취향·기록 부족 시 추천 영역을 비우지 않고 설정 유도 안내·바로가기를 표시하며, 건너뛴 취향이 있으면 미설정 항목을 한두 개씩 점진 설정 카드로 권유한다 [G157].
**유닛 비고**: U2는 레이아웃·빈 상태까지. 카드 데이터는 후행 유닛이 점진 공급(인기 장소=U3, 여행 카드=U4, 활성 일정=U6, 추억=U7), 최종 통합 검증=U8. 커뮤니티 카드는 U10 출시 전 미노출.

### US-E2-03 하단 5탭 글로벌 내비게이션 `[1차]` · U2
**요지**: 여행자로서 하단 탭으로 주요 기능군에 한 번에 이동하고 싶다 — 어디에 있든 원하는 기능으로 바로 가려고.
**원 스토리·결정**: PRD 01 스토리 3 · D21, D24, G6, G7
**수용 기준**:
- 탭은 홈·탐색·일정·기록·마이 5개 고정. 홈=홈 대시보드, 탐색=탐색 랜딩, 일정=생성된 일정, 기록=여행 기록·요약, 마이=마이페이지. 바텀 탭바는 단일 공용 컴포넌트로 모든 탭 루트에서 동일 형태, 선택 탭을 시각 구분한다.
- 탐색 탭은 '무엇을 찾을까요?' 랜딩에서 숙소·장소·여행자 일정(커뮤니티 둘러보기) 3개 카드로 진입시키며, 전용 커뮤니티 탭은 두지 않는다.
- 일정 탭은 진행 중 여행(항상 최대 1개 [D21])이 있으면 '여행 중 활성 일정 화면'(홈 활성 카드와 수렴하는 단일 허브)으로, 없으면 가장 최근 여행 일정으로 진입하고, 일정 없으면 숙소 등록·여행 생성 유도를 노출한다.
- 여행 중 일정 조회는 온라인 전제이며 오프라인 조회는 보장하지 않고 실패 시 오류·재시도 안내를 표시한다 [D24/Δ6].
- 탭 전환 시 각 탭의 이전 화면 상태(스크롤·하위 화면)는 세션 내 보존(앱 재시작 시 초기화), 현재 탭 재탭 시 스크롤 탑 이동 [G6].
- 알림·딥링크 진입은 대상 화면이 속한 탭을 활성화하고 그 탭 스택에 화면을 푸시한다 [G7].
- 탭 루트가 데이터 0건이면 빈 탭 대신 각 섹션 정본의 빈 상태 안내를 노출한다.
**유닛 비고**: 탭 상태 보존(G6)·딥링크 탭 스택 푸시(G7) — U3~U8 전 화면의 진입 틀. 딥링크 라우팅 마감은 U8(shared/api).

### US-E2-04 몰입 화면의 탭바 숨김 `[1차]` · U2
**요지**: 여행자로서 입력·실행에 집중할 화면에서 탭바가 사라지길 원한다 — 그 화면의 단 하나의 다음 행동에 집중하려고.
**원 스토리·결정**: PRD 01 스토리 4
**수용 기준**:
- 탭 루트 화면과 직계 콘텐츠 화면(탐색 랜딩, 숙소 상세, 장소 검색·저장, 공유 일정 둘러보기·상세, 기록 탭 회고 등)에서는 탭바를 노출한다.
- 스플래시, 온보딩 전체(로그인·약관·취향), 입력 폼(숙소 검색·등록, 여행 생성, 필수 방문지 지정), 일정 생성 진행·편집, 여행 중 실행·Plan-B 플로우, 바텀시트/다이얼로그에서는 탭바를 숨기고 전체 폭 단일 CTA 또는 자체 하단 액션을 사용한다.
- 탭바 숨긴 화면에서도 시스템 뒤로가기 또는 화면 내 닫기/취소로 직전 탭 컨텍스트로 복귀할 수 있다.
**유닛 비고**: shared/ui 단일 컴포넌트로 강제 — 후행 유닛 파편화 방지.

### US-E2-05 장소 우선 저장과 여행 시드 연결 (Case A 온램프) `[1차]` · U2
**요지**: P1 지유로서 가고 싶은 장소를 먼저 모아 두고 여행 만들 때 '꼭 갈 곳'으로 가져오고 싶다 — 평소 찜한 장소가 그대로 일정 시드가 되도록.
**원 스토리·결정**: PRD 01 스토리 5 · G8
**수용 기준**:
- 탐색 랜딩 '장소' 진입에서 장소(명소·맛집·카페 등)를 검색해 저장/해제할 수 있고, 저장 POI는 이름·카테고리·지역과 함께 목록 표시한다.
- 저장 POI 개수 상한은 없으며, 원본 소실 POI는 '확인 불가' 배지를 유지하고 시드 투입 시 제외 안내를 표시한다 [G8].
- 저장 POI는 위시리스트·필수 방문지 지정의 입력으로 재사용되어 여행 생성 시 AI 일정 시드('꼭 갈 곳')로 투입된다.
- '이 장소들로 여행 만들기' 진입으로 저장 POI를 가진 채 바로 여행 생성으로 이어간다.
- 저장 POI가 0곳이면 빈 화면 대신 "저장한 장소는 여행 만들 때 '꼭 갈 곳'으로 담겨요" 안내와 장소 검색 진입점을 노출한다.
- 저장 POI 집합에서 숙소 권역을 **직접** 역추천하지는 않는다 — 숙소 권역 추천은 저장 POI를 시드로 만든 **무숙소 초안 일정의 완성 동선(무게중심)**을 거쳐 제공한다(US-E5-11, 장소 우선). 추천 권역에서 숙소를 등록하면 그 숙소를 앵커로 재정렬되어 '등록 숙소=공간 앵커'로 수렴한다.
**유닛 비고**: 온램프 UI 셸은 U2, POI 검색·저장 백엔드(M7) 연결·전 기능 활성화는 U3. 시드 투입(사본·G8 소실 배지)은 U4.

### US-E2-06 강제 업데이트 게이트 `[1차]` · U2 · 신규(N4)
**요지**: 여행자로서 지원이 끝난 구버전 앱에서는 업데이트로 안내받고 싶다 — 깨진 화면·오류 대신 정상 동작하는 앱을 쓰려고.
**원 스토리·결정**: 신규 (요구사항 §5.3-N4, C6)
**수용 기준**:
- 스플래시에서 서버의 최소 지원 버전을 확인하며 서버 응답 계약에 최소 버전 필드를 포함한다 [N4/C6].
- 현재 앱 버전이 최소 지원 버전 미만이면 강제 업데이트 화면으로 분기하고 스토어 이동 액션을 제공한다.
- 강제 업데이트 화면에서는 서비스 진입(홈·로그인 포함)을 차단한다.
- 버전 확인은 세션 검증과 함께 스플래시 분기 흐름 안에서 수행하며 확인 불가 시 스플래시 타임아웃 정책(US-E2-01)을 따른다.
**유닛 비고**: 부트스트랩 API 최소 버전 필드, 미달 시 전면 차단.

## Epic 3 — 숙소 탐색·저장·등록 (E3 / U3, 11개)

여행자가 숙소를 탐색·저장하고, 외부에서 예약한 숙소를 앱에 등록해 AI 일정 생성의 거점(출발점)으로 삼는 여정. 탐색·저장은 앱, 상세·예약·결제는 외부 OTA 딥링크로 위임한다.

### US-E3-01 여행지 기반 숙소 탐색 `[1차]` · U3
**요지**: 여행자로서 여행지만 입력해 숙소를 탐색하고 싶다 — 날짜·인원·예약 부담 없이 거점 후보를 먼저 둘러보려고.
**원 스토리·결정**: PRD 04 스토리 1 · D09, G33
**수용 기준**:
- 여행지(또는 '내 주변')만 입력받으며 체크인/아웃 날짜·인원은 탐색 단계에서 받지 않는다. 여행지가 비면 탐색 버튼은 비활성.
- 탐색 결과는 TourAPI + 지도 장소 검색 데이터로 구성하고 외부 OTA는 숙소명 기반 검색 딥링크로만 연결한다(OTA 크롤링 없음) [D09].
- 결과 카드에 숙소명·위치·(확보 가능 시) 대표 가격대를 표시하고 정확 가격·재고는 외부 OTA 확정으로 안내한다. 결과 전체 정확 가격 일괄 조회는 안 하며 정확 가격 값은 캐싱하지 않는다.
- '가격 보기' 탭 시 날짜·인원 바텀시트 입력 후 정확 가격을 라이브 조회하고 입력값은 세션 내 재사용한다 [G33].
- 결과 N건 이상이면 페이지네이션 또는 무한스크롤로 추가 노출한다.
- 위치 권한 거부 상태의 '내 주변' 검색은 등록 숙소 또는 여행지 중심 좌표로 대체 조회한다.
**유닛 비고**: 날짜·인원 없이 탐색(D09), '가격 보기' 라이브 조회(G33).

### US-E3-02 검색 결과 필터·정렬 `[1차]` · U3
**요지**: P2 민준으로서 가격대·유형·편의시설·거리로 결과를 좁히고 싶다 — 조건·예산에 맞는 숙소만 빠르게 비교하려고.
**원 스토리·결정**: PRD 04 스토리 2 · D09, D25, G34
**수용 기준**:
- 숙소 유형 필터(호텔·게스트하우스·펜션·모텔·호스텔·아파트·B&B·한옥 등)를 전면 제공하며 한국형 유형 분류는 TourAPI를 정본으로 표준화한다 [D09].
- 편의시설 필터는 TourAPI 필드 채움률 검증 항목만 노출(핵심 3종 주차·조식·와이파이부터 확대), 편의시설 표시는 가격 비교 화면이 아닌 곳에 배치한다.
- 가격대 필터는 대표 가격대(구간) 기준이며 정확 가격 의존 가격순 정렬은 미제공, '대표 가격대순'으로 대체한다.
- 거리 필터·정렬은 검색 여행지 중심 좌표 기준 직선거리이며 차량/도보 예상 소요시간은 표시하지 않는다 [G34, D25].
- 필터 결과가 0건이면 어떤 필터가 0건을 만들었는지 표기하고 완화를 제안한다.
**유닛 비고**: 취향 예산이 가격 필터 기본값(U1 연계). 직선거리 기준(G34)·소요시간 미표시(D25).

### US-E3-03 숙소 상세 확인 `[1차]` · U3
**요지**: 여행자로서 숙소의 위치·사진·가격대·시설을 한 화면에서 확인하고 싶다 — 거점 후보로 적합한지 판단하려고.
**원 스토리·결정**: PRD 04 스토리 3
**수용 기준**:
- 상세 화면에 위치(지도 핀)·사진·대표 가격대·편의시설·체크인/아웃 시각 등 정적 콘텐츠를 표시한다.
- 리뷰·평점은 앱 내부에 표시하지 않고 "리뷰·평점 보기 → 외부 OTA" 딥링크로 위임한다(원문·점수 모두).
- 평가 정보가 필요하면 "외부 OTA에서 확인" 진입점을 노출한다.
- 누락 필드는 빈칸 대신 "미확인"으로 표기한다.
**유닛 비고**: 리뷰·평점은 OTA 위임(범위 제외 정본).

### US-E3-04 숙소 위시리스트 저장 `[1차]` · U3
**요지**: P1 지유로서 마음에 드는 숙소를 저장하고 메모를 남기고 싶다 — 후보를 비교한 뒤 등록·예약으로 이어가려고.
**원 스토리·결정**: PRD 04 스토리 4 · D22
**수용 기준**:
- 저장은 로그인 계정에 귀속되어 기기를 바꿔도 유지된다(로그인 필수 전제) [D22].
- 저장 목록 각 항목은 숙소명·대표 사진·가격대·위치를 카드로 표시한다.
- 저장 항목을 삭제하거나 메모(예: "역에서 가까움")를 덧붙일 수 있다.
- 저장/조회 시점 가격·재고 차이를 안내하고 가격은 "외부 OTA 기준, 변동 가능"으로 표기한다.
**유닛 비고**: 계정 귀속(U1)·소유권 검증(SECURITY-08).

### US-E3-05 외부 OTA 딥링크 예약 이동 `[1차]` · U3
**요지**: 여행자로서 [외부 OTA에서 예약하기]로 이동해 실제 예약·결제를 진행하고 싶다 — 신뢰할 수 있는 OTA에서 확정하려고.
**원 스토리·결정**: PRD 04 스토리 5 · D09, D17, G32
**수용 기준**:
- 상세·저장 화면의 예약 이동은 숙소명 기반 OTA 검색 딥링크로 연결한다(외부 브라우저 또는 인앱 웹뷰) [D09].
- 외부 이동 직전 "예약·결제는 외부 OTA에서 진행되며 TripPilot은 제휴 수수료를 받을 수 있다"는 제휴 링크·수수료 고지를 표시한다.
- 앱은 자체 예약·객실 선택·예약 확정·인앱 결제를 제공하지 않고 모든 결제를 외부 OTA로 위임한다.
- 동일 숙소에 OTA가 여러 개면 내부 숙소 ID 기준으로 묶어 OTA명 목록을 보여주고 사용자가 링크를 선택한다 [D17].
- 딥링크 이탈 후 24시간 내 첫 복귀 시 "방금 본 [숙소명], 예약하셨나요? → 거점으로 등록하기" 복귀 핸드오프 카드를 1회 노출한다(최근 이탈 숙소 1건). 무시하면 재노출 없이 계속 탐색 [G32].
- OTA 링크가 깨졌거나 무응답이면 "지금 이 OTA로 연결할 수 없어요"를 표시하고 같은 숙소의 다른 OTA 링크 또는 숙소 장소 검색으로 우회 경로를 제시한다.
**유닛 비고**: 제휴 고지·다중 OTA 선택(D17)·복귀 핸드오프 카드(G32). P5 제휴 계약은 출시 전(개발은 검색 딥링크로 무차단).

### US-E3-06 예약한 숙소 등록 `[1차]` · U3 · 이연(포스트백 자동 등록 후속)
**요지**: 여행자로서 외부에서 이미 예약한 숙소를 앱에 등록하고 싶다 — 그 숙소가 AI 일정 생성의 거점·기간 기준점이 되도록.
**원 스토리·결정**: PRD 04 스토리 6 · D09, D15
**수용 기준**:
- 등록 정보는 숙소(장소)·체크인/아웃 날짜·인원이 필수, OTA 예약번호·결제 금액은 선택 입력이다.
- 1차 등록 경로는 수동 입력 단일 경로 — 검색 결과·저장 목록 [예약했어요/등록]에서 시작하면 숙소명·위치·가격대가 자동 채워지고 날짜·인원만 입력하면 완료된다. 예약번호·메일 연동 자동 인식은 미제공 [D09].
- 등록 숙소는 계정 레벨 풀에 저장되고 여행에 연결(거점 지정)되어 해당 체류 기간 일정 생성의 기준점(거점·기간)이 된다 [D15].
- 등록 완료 시 해당 숙소를 거점으로 한 [AI 일정 생성하기] 진입점을 즉시 노출한다.
- 등록 숙소의 날짜·인원·예약번호·금액은 나중에 수정할 수 있고 등록 자체를 취소할 수 있다.
- 체크아웃이 체크인보다 빠르거나 날짜가 비면 저장을 막고 누락을 인라인 표시하며, 좌표 미확정 등록은 "지도에서 위치 확인" 단계를 요구한다.
**유닛 비고**: 이연 — 포스트백 1탭 자동 등록은 후속(G29/G108, 집계용 설계 여지만). 1차는 복귀 카드+수동 빠른 등록으로 충족.

### US-E3-07 일자별 다중 거점 등록 `[1차]` · U3
**요지**: 여행자로서 일자별로 거점 숙소를 다르게 등록하고 싶다 — 다도시 이동 여행에서도 구간별 거점 기반 일정을 받으려고.
**원 스토리·결정**: PRD 04 스토리 7 · D15
**수용 기준**:
- 하나의 여행에 일자별로 서로 다른 거점 숙소를 등록할 수 있다.
- 날짜 비중첩 검증은 같은 여행에 연결된 거점끼리 수행한다(계정 풀 미연결 숙소는 검증 대상 아님) [D15].
- 거점 날짜 구간이 겹치면 "같은 날짜에 거점이 둘이에요"를 표시하고 구간 조정을 요구한다.
- 각 구간 일정은 해당 구간 거점 기준으로 생성된다.
**유닛 비고**: 등록 데이터·API는 U3(M4·D15) 소유, 여행 연결·날짜 비중첩 검증·UI 완성은 U4(US-E4-06) — CP1 경계.

### US-E3-08 숙소 직접 등록 3경로 `[1차]` · U3
**요지**: 여행자로서 지도 검색·링크 붙여넣기·지도 핀으로 숙소를 직접 등록하고 싶다 — OTA 탐색 결과에 없는 숙소도 빠르게 거점으로 지정하려고.
**원 스토리·결정**: PRD 04 스토리 8 · G31
**수용 기준**:
- 지도/장소 검색(숙소명·주소)으로 실제 숙소를 찾아 위치·좌표를 자동 확보하고 거점 지정할 수 있으며, 앱 내 숙소 검색 결과에 없어도 독립 동작한다.
- 검색 결과에서 숙소 선택 시 숙소명·주소·좌표가 자동 채워지고 체크인/아웃 날짜·인원만 입력하면 완료된다.
- OTA 예약 링크 붙여넣기는 지원 도메인 화이트리스트 내 URL 패턴 파싱만으로 숙소명·위치를 추출해 폼에 채운다(페이지 fetch 없음). 실패 시 수동 핀 지정으로 안내 [G31].
- 지도 검색에서 동명·유사 숙소가 여러 건이면 후보 목록+각 후보 지도 핀을 보여주고 사용자가 정확한 한 곳을 선택해 좌표를 확정한다.
- 붙여넣은 링크가 숙소 페이지가 아니거나 위치 특정 불가면 "위치를 찾지 못했어요. 지도에서 직접 지정해 주세요"로 수동 핀 지정으로 넘긴다.
- 지도/장소 검색 API 무응답 시 침묵 실패 없이 지도 핀 직접 지정으로 폴백한다.
**유닛 비고**: 지도 검색·링크 파싱(G31 화이트리스트)·핀 지정. 좌표 미확정 시 지도 확인 강제(CP1 시나리오 2).

### US-E3-09 등록·저장 숙소 목록 확인 `[1차]` · U3
**요지**: 여행자로서 등록·저장한 숙소 목록을 한눈에 확인하고 싶다 — 예약 기록과 거점 지정 상태를 관리하려고.
**원 스토리·결정**: PRD 04 스토리 9 · D15
**수용 기준**:
- 목록에서 숙소명·위치·체크인/아웃 일정·인원·(입력한 경우) OTA명·예약번호·금액·거점 사용 여부를 확인할 수 있다.
- 등록 숙소는 계정 레벨 풀로 관리되며 어느 여행 거점으로 연결되어 있는지 표시한다 [D15].
- 앱은 예약 진행/확정 상태를 추적하지 않고, 사용자 입력 외부 예약 기록과 거점 지정 상태만 보여준다.
- 금액·예약번호 등 선택 항목이 비면 빈칸 대신 "미입력"으로 표시하고 입력을 강요하지 않는다.
**유닛 비고**: 마이페이지 통합 목록(출처 라벨 G103)의 최종 화면은 U8(US-E09-06).

### US-E3-10 검색 결과 없음 안내 `[1차]` · U3
**요지**: 여행자로서 조건에 맞는 숙소가 없을 때 다음 행동을 안내받고 싶다 — 빈 화면에서 막히지 않고 계획을 이어가려고.
**원 스토리·결정**: PRD 04 스토리 10 · D09
**수용 기준**:
- 조건에 맞는 숙소가 없으면 빈 화면 대신 "조건에 맞는 숙소가 없어요"와 지역·필터 완화 같은 다음 행동을 제안한다.
- 탐색 데이터가 없는 지역이면 "이 지역은 아직 숙소 정보가 부족해요. 숙소를 직접 등록하면 일정을 만들어 드려요"로 직접 등록(US-E3-08) 경로로 연결한다 [D09].
- 완화 제안 탭 시 해당 필터가 조정된 상태로 재검색된다.
**유닛 비고**: 데이터 부족 지역 흐름 포함.

### US-E3-11 탐색 로딩·부분 실패 처리 `[1차]` · U3
**요지**: 여행자로서 숙소 정보를 모으는 동안 진행 상태를 보고 싶다 — 일부 소스가 실패해도 가능한 결과로 탐색을 계속하려고.
**원 스토리·결정**: PRD 04 스토리 11 · D09
**수용 기준**:
- 숙소 목록을 모으는 동안 "숙소를 모으는 중" 로딩 상태를 표시한다.
- 일부 소스(TourAPI·지도 장소 검색)가 무응답이어도 나머지 결과로 목록을 구성하고 "일부 정보가 빠졌을 수 있어요"를 표기하며 침묵 실패하지 않는다 [D09].
- 모든 조회 실패 시 "지금 숙소 정보를 불러올 수 없어요"와 재시도 버튼을 제공하고 원하면 숙소 직접 등록(US-E3-08)으로 우회하게 한다.
**유닛 비고**: RESILIENCY-10(카카오→네이버 폴백, 전체 실패 시 수동 등록 우회) — 침묵 실패 금지.

## Epic 4 — 여행 생성·거점·필수 방문지 (E4 / U4, 11개)

여행자가 여행지·날짜·인원·예산으로 여행을 만들고, 숙소를 거점으로 연결하며, 필수 방문지를 지정해 AI 일정 생성의 기준점을 준비하는 여정.

### US-E4-01 새 여행 생성 `[1차]` · U4
**요지**: 여행자로서 여행지·날짜·인원·예산을 입력해 새 여행을 만들고 싶다 — 일정 생성의 기본 틀을 확보하려고.
**원 스토리·결정**: PRD 05 스토리 1 · D21, D26, D29, G42, G119
**수용 기준**:
- 여행지·날짜 범위는 필수, 둘 중 하나라도 비면 "여행 생성" 비활성. 인원·예산은 선택이며 미입력 시 인원 1명·예산 미설정으로 생성된다.
- 예산은 여행 전체 총액(항공 제외 · 숙소+식비+활동+교통) 기준으로 입력·저장하고 1인·1일 값은 파생 표시. 온보딩 러프 예산이 있으면 기본값으로 제시하되 수정 가능 [D26].
- 시작일은 오늘 이후만 허용, 여행 기간 최대 30일, 종료일이 시작일보다 빠르면 오류 표시·생성 차단 [G42].
- 기존 여행과 날짜 구간이 겹치면 오류 표시·생성 차단(활성 여행은 항상 최대 1개) [D21].
- 날짜별 '이용 가능 시작/종료 시각'을 선택 입력받고(기본 09:00~21:00) 첫날 도착·마지막날 출발 시각을 일정 생성 시간 예산에 반영한다 [G119, D29].
- 등록 숙소가 있으면 '등록 숙소에서 날짜 가져오기' 단일 버튼을, 없으면 '숙소 등록하기 / 날짜 직접 입력' 대안을 노출한다.
- 온보딩에서 건너뛴 취향이 있으면 첫 여행 생성 흐름에서 점진 설정 카드로 권유하되 건너뛸 수 있고 미설정은 중립 기본값으로 동작한다.
**유닛 비고**: 날짜 겹침 차단(D21/Δ3)·예산 총액(D26)·시간창(D29/G119)·최대 30일(G42) — CP2 무결성의 본체.

### US-E4-02 숙소 없이 여행 먼저 생성 `[1차]` · U4
**요지**: P3 하람으로서 숙소를 정하기 전에 여행지·날짜만으로 여행을 만들고 싶다 — 예약 부담 없이 일정부터 그려보려고.
**원 스토리·결정**: PRD 05 스토리 2
**수용 기준**:
- 숙소 미등록 상태로 생성된 여행은 "숙소 미등록"으로 표시하고 "숙소를 등록하면 그 위치를 기준으로 일정을 만들 수 있습니다" 안내를 노출한다.
- 여행 생성 후 언제든 숙소를 추가 등록할 수 있다.
- '예약·숙소 없이 일정부터 받아보기' 경로를 숙소 우선 진입과 동등한 1급 온램프로 노출하며 '결제·예약부터 해야 일정이 나온다'는 인상을 주지 않는다(상세는 E5 정본).
- 숙소가 한 개도 없는 상태에서 AI 일정 생성 요청 시 생성 버튼을 비활성화하고 "일정을 생성하려면 숙소를 먼저 등록하세요" 안내·진입점을 표시한다(E5 스토리 1 정본).
**유닛 비고**: Case B 온램프 — 숙소 나중 등록 시 권역 추천은 U5(US-E5-11).

### US-E4-03 숙소를 여행에 등록(거점 지정) `[1차]` · U4
**요지**: 여행자로서 예약했거나 저장해 둔 숙소를 여행에 등록하고 싶다 — 그 위치가 일정 생성의 기준 거점이 되도록.
**원 스토리·결정**: PRD 05 스토리 3 · D15
**수용 기준**:
- 등록 숙소는 숙소명·위치(좌표/주소)·체크인·체크아웃 날짜를 필수 속성으로 가진다.
- 숙소는 계정 레벨 풀에 존재하고 여행에 연결(거점 지정)하는 구조이며, 앱은 연결 숙소 위치를 해당 체류 기간 동안 일정 생성 기준 거점으로 사용한다 [D15].
- 등록 숙소를 수정하거나 삭제할 수 있다.
- 등록 폼 화면은 숙소 직접 등록 3경로(US-E3-08)를 정본으로 사용하고 여행 생성·거점 화면에서는 등록 진입점만 제공한다.
- 위치 좌표를 확인할 수 없으면 지도에서 위치를 직접 지정하도록 요청한다.
- 체크인/아웃 날짜가 여행 기간을 벗어나면 경고를 표시하고 여행 기간 자동 확장 여부를 사용자에게 묻는다.
**유닛 비고**: CP1 소비자 — U3 등록 숙소 풀(D15)에서 거점 배정.

### US-E4-04 저장 숙소에서 여행 등록 `[1차]` · U4
**요지**: 여행자로서 저장해 둔 숙소를 골라 바로 여행에 등록하고 싶다 — 정보를 다시 입력하지 않고 거점을 지정하려고.
**원 스토리·결정**: PRD 05 스토리 4
**수용 기준**:
- 저장 목록에서 숙소 선택 시 저장 시점 숙소명·위치가 자동으로 채워지고 체크인/아웃 날짜만 입력하면 완료된다.
- 저장 숙소 정보가 외부 출처에서 더 이상 조회되지 않으면 저장 당시 캐시 정보로 등록을 허용하되 "최신 정보 확인 불가"를 표시한다.
- 등록 숙소는 US-E4-03과 동일한 거점 지정·검증 규칙을 따른다.
**유닛 비고**: 위시리스트(U3)→등록→거점 경로.

### US-E4-05 숙소 날짜를 여행 기간으로 자동 반영 `[1차]` · U4

### US-E4-08 필수 방문지 지정 `[1차]` · U4
**요지**: P1 지유로서 꼭 가고 싶은 장소를 미리 지정하고 싶다 — AI 일정이 내 필수 장소를 반드시 포함해 만들어지도록.
**원 스토리·결정**: PRD 05 스토리 8 · G40, G43, G158, G129
**수용 기준**:

- 저장 POI를 체크박스 화면에서 골라 필수 방문지로 투입할 수 있고, 여행지 권역 밖 항목은 경고 배지+기본 해제로 표시하며 투입은 사본 복제로 원본 저장 목록과 독립이다 [G158, G129].
- 필수 방문지는 하루 3곳 × 여행 일수를 상한으로 하며 초과 시 추가 차단·사유 안내 [G40].
- 필수 방문지는 등록 숙소와 함께 일정 생성 기준점(앵커)으로 사용되고 생성 전·후 추가·삭제할 수 있으며 변경 시 해당 범위 추천 POI 배치를 다시 계산한다.
- 확정 일정에서 필수 방문지 변경 시 변경안 미리보기(전/후 비교)를 제시하고 사용자 승인 후에만 적용한다 [G43].
- 완전 AI 경로에서 필수 방문지 미지정 시 생성 추천안에서 특정 장소를 사후 고정(LOCK)해 필수화할 수 있고, 같이 고르기 경로에서는 지정 필수 방문지를 후보 추천 앵커(시드)로 사용한다.
- 좌표 확인 불가 필수 방문지는 지도에서 위치를 직접 지정하도록 요청한다(시각 미입력·고정 블록 충돌 처리는 E5 스토리 4 정본).
**유닛 비고**: 저장 POI 사본 복제(G129)·한도(G40)·권역 밖 경고(G158)·변경 미리보기(G43).

## Epic 5 — AI 일정 생성·확정 (E5 / U5, 12개)

등록 숙소를 출발점으로 LLM(취향 해석·설명)과 솔버(선택·순서·시간 보장)가 협업해 실행 가능한 날짜별 일정을 생성하고, 편집·저장을 거쳐 여행 전 최종본으로 확정하는 핵심 여정(M7 Place Data, M8 Itinerary Generation).

### US-E5-01 숙소 기준 날짜별 일정 생성 `[1차]` · U5
**요지**: 여행자로서 등록 숙소 위치·체크인/아웃 날짜를 기준으로 여행 기간 전체의 날짜별 AI 일정을 생성하고 싶다 — 매일 어디서 출발·복귀하는지가 맞는 현실적 일정을 받으려고.
**원 스토리·결정**: PRD 06 스토리 1 · D28, G50
**수용 기준**:
- 등록 숙소 위경도·체크인/아웃 날짜를 일정 생성 고정 입력값(출발점)으로 사용하고 체크인 당일~체크아웃 당일 각 날짜에 일자 일정을 1개씩 생성한다. 생성·재계획 연산은 서버에서 수행한다 [D28].
- 숙소 우선 생성은 등록 숙소가 없으면 버튼을 비활성화하고 "숙소를 먼저 등록하면 일정을 만들 수 있어요"를 안내한다. 단 **장소 우선(스토리 11 '숙소 나중 등록') 온램프는 1급 경로로 여행지 중심 무숙소 생성을 허용**한다.
- 같은 기간에 위치 다른 숙소가 여러 개면 각 숙소 체크인/아웃 날짜에 맞춰 날짜별 기준점을 자동 전환하고, 숙소 전환일은 출발=A숙소·복귀=B숙소 편도 동선으로 모델링한다 [G50].
- 숙소 지오코딩 실패 시 지도에서 위치를 직접 지정하도록 요청하고 지정 전까지 해당 숙소 기준 생성을 보류한다.
- 체크인/아웃 날짜 누락·역전 시 일정을 생성하지 않고 날짜 수정 화면으로 이동시킨다.
- 위치 권한 거부여도 등록 숙소 위치를 출발점으로 생성하며 현재 위치 필요 기능은 등록 숙소 위치로 대체하고 그 가정을 화면에 표기한다.
**유닛 비고**: 하드 제약 1계열(숙소 기준점)의 본체. 거점 전환 편도 모델링(G50).

### US-E5-02 취향 기반 POI 선별·추천 `[1차]` · U5
**요지**: P2 민준으로서 여행 스타일·예산·동행·선호 활동·이동 방식·음식 취향을 입력하면 그에 맞는 장소를 추천받고 싶다 — 내 취향·형편에 맞는 곳들로 일정이 채워지도록.
**원 스토리·결정**: PRD 06 스토리 2 · G47, G115
**수용 기준**:
- LLM이 사용자 자유 입력(취향 문장 포함)을 해석해 각 후보 POI에 선호 점수를 매기고 그 점수를 OPTW/TOPTW 최적화의 목적함수 보상값으로 사용한다.
- LLM은 그라운딩된 후보 POI ID 목록에서만 선택(closed-set)해 실재 장소만 노출하고, 후보 목록 밖 POI는 구조적으로 포함될 수 없다 [G115].
- 예산은 솔버 하드 제약이 아니라 LLM 추천 소프트 가중치로 반영한다 — 카테고리별 무료/저/중/고 비용 추정으로 예산대에 맞는 POI를 우선 추천 [G47].
- 이동 방식(도보/자차 등)에 따라 장소 간 거리 기반 이동시간을 다르게 계산해 추천에 반영한다.
- 예산으로 채울 POI가 없으면 "예산을 높이거나 활동 수를 줄여보세요"를 안내하고 빈 일정 대신 무료/저비용 대안 POI를 우선 제시한다.
- 선호 조건이 너무 좁아 후보 0건이면 결과를 0으로 만든 조건을 표시하고 완화를 제안한다.
**유닛 비고**: C1 closed-set 강제(G115 — POI 그라운딩)·U1 취향 소비. 예산은 소프트 가중치(G47).

### US-E5-03 시간 하드 제약 기반 시간표 구성 `[1차]` · U5
**요지**: 여행자로서 영업시간·체류 시간·이동시간이 맞는 시간표를 받고 싶다 — 현장에서 문 닫힌 가게 앞에 서거나 시간이 안 맞아 일정이 무너지지 않도록.
**원 스토리·결정**: PRD 06 스토리 3 · D29, G51
**수용 기준**:
- 어떤 POI도 영업시간 밖(종료 후·휴무일 포함)에 배치하지 않고, 직전 종료 시각 + 이동시간(안전 마진 포함) ≤ 다음 시작 시각을 항상 만족하며 위반 배치는 결과에서 제외한다.
- 사용자에게 보이는 모든 시각(시작/종료/이동)은 솔버가 소유·검증한 값만 표시하고 LLM 생성 임의 시각은 노출하지 않는다.
- 하루 시간창은 기본 09:00~21:00에 여행별 조정을 허용하고, 점심·저녁은 취향 기반 솔버 자동 배치, 자정 넘는 활동은 시작한 날짜에 귀속 [D29].
- 체류 시간 기본값은 운영 정의 정적 테이블(카테고리 20~30종)에서 시작해 [G51] 일정 밀도·여행 스타일을 반영하며 사용자가 수정할 수 있다.
- 체류 시간은 최소·권장·최대 범위로 관리해 솔버가 권장→최소 범위에서 조정하고 사용자 고정값은 고정 제약이며, 권장보다 줄인 배치는 해당 POI에 표시한다.
- 영업시간 데이터 없는 POI는 "영업시간 미확인"으로 표시하고 확정 시간대 배치 대신 사용자 확인 필요 후보로 분리한다.
- 이동시간 제약상 하루 채울 일정이 1개 이하이면 억지로 늘리지 않고 "이동 거리가 멀어 하루에 한 곳만 추천돼요"를 안내한다.
**유닛 비고**: 충돌 무배치 계열 — 영업시간·이동 부등식·시간창(D29). 체류 정적 테이블(G51)·이동 추정 파라미터(G106).

### US-E5-04 숙소·필수 방문지 고정 반영 `[1차]` · U5
**요지**: P1 지유로서 꼭 가야 하는 장소를 미리 지정하면 일정이 그것을 중심으로 짜이길 원한다 — AI 추천 속에서도 내가 정한 우선순위가 절대 밀리지 않도록.
**원 스토리·결정**: PRD 06 스토리 4
**수용 기준**:
- 등록 숙소 체크인/아웃 시각과 시각 고정형 필수 방문지를 변경 불가 고정 블록으로 놓고 나머지 추천 POI를 충돌 없이 배치한다.
- 필수 방문지는 생성 전·후 어느 시점에든 지정할 수 있고 기본은 포함 고정형(반드시 포함, 시각·순서는 솔버 위임)이며 시각 지정은 선택이다. 화면 라벨은 사용자 언어('시간 정해두기'/'아무 때나 꼭 가기').
- 시각 고정형 지정 시 방문 날짜·시작 시각·예상 체류 시간을 입력받고 시각 미지정 시 "영업시간·동선에 맞춰 자동 배치됨"을 명시한다.
- 필수 방문지(특히 포함 고정형)를 주변 추천 POI가 모이는 공간적 앵커로 활용하고, 고정 일정과 시간 겹침·이동시간 부족 추천 POI는 자동 배제한다.
- 고정·필수 일정을 추가·삭제하면 해당 날짜(미지정 시 여행 전체)의 추천 배치를 다시 계산한다.
- 두 고정 일정이 충돌하면 자동 변경하지 않고 충돌 항목을 강조한 뒤 어느 쪽을 옮길지 사용자에게 묻는다.
- 필수 방문지를 어떤 시간대에도 넣을 수 없으면 빈 일정 대신 불가 사유를 표시하고 다른 날짜 이동 / 시각 고정 해제 / 인접 일정 조정 중 하나를 제안한다.
**유닛 비고**: CP2 소비자 — 고정 블록 불변(PBT 1급).

### US-E5-05 추천·배치 이유 설명 `[1차]` · U5
**요지**: 여행자로서 각 장소가 왜 추천됐고 왜 그 시간·순서에 배치됐는지 설명을 보고 싶다 — AI 일정을 납득하고 믿고 따르려고.
**원 스토리·결정**: PRD 06 스토리 5
**수용 기준**:
- 장소별로 사용자의 어떤 선호(예: "한식 선호", "도보 이동", "예산대")가 반영됐는지 LLM 설명 문구로 제시한다.
- 배치 이유(예: "영업시간이 오전만 → 오전 배치", "숙소에서 가까워 첫 일정")를 솔버가 실제 사용한 제약 근거에 맞춰 표시한다.
- 설명 문구는 시간표를 바꾸지 않는 표시용이며 설명과 실제 배치 시각이 불일치하면 검증된 시각을 기준으로 표시한다.
**유닛 비고**: 추천 이유 메타 — CP3 슬롯 필드.

### US-E5-06 시간표/지도 2보기·이동 구간 정보 `[1차]` · U5
**요지**: 여행자로서 일정을 시간표·지도로 전환해 보고 장소 사이 이동 정보를 함께 확인하고 싶다 — 하루 흐름과 동선을 한눈에 파악하려고.
**원 스토리·결정**: PRD 06 스토리 6 · D25, Δ1
**수용 기준**:
- 동일 일정 데이터를 시간표형(시간순 타임라인·시각/체류 블록)과 지도형(순서 번호 핀+이동 동선)으로 보여주고 상단 세그먼트로 전환하며 한 보기의 수정이 다른 보기에 즉시 반영된다.
- 연속 두 일정(A→B) 사이 각 이동 구간에 **이동 수단·거리 요약만** 범위·추정으로 표시한다(예: "약 850m · 도보, 추정") — 차량/도보 예상 소요 시간은 표시하지 않는다 [D25, Δ1].
- 대중교통 노선·환승 실시간 경로 안내와 자체 턴바이턴 내비게이션은 미제공하며 각 이동 구간 '길찾기' 동작으로 외부 지도앱에 출발·도착 좌표를 전달해 위임한다.
- 지도 타일/경로 API 무응답 시 지도형 대신 시간표형으로 폴백하고 "지도를 불러오지 못했어요"를 표시하되 일정 데이터는 정상 제공한다.
- 외부 지도앱이 없으면 웹 지도 길찾기로 폴백, 그것도 불가하면 앱 산출 이동 수단·거리 요약(소요 시간 없이)을 최종 안내로 표시한다 [D25].
- 라우팅 API 무응답 시 직선거리 기반 추정 또는 수동 입력 모드로 폴백하고 추정치·수동 입력임을 표기한다.
**유닛 비고**: 거리·수단만 표시(D25/Δ1 — 소요시간 삭제 델타 반영).

### US-E5-07 일정 편집과 제약 재검증 `[1차]` · U5
**요지**: 여행자로서 생성 일정에서 장소를 추가·삭제·재정렬하고 시간을 조정하면 매번 시간 제약이 검증되길 원한다 — 내 마음대로 고치면서도 실행 불가능한 일정이 되는 걸 즉시 알려고.
**원 스토리·결정**: PRD 06 스토리 7 · D28, G49
**수용 기준**:
- POI 추가/삭제/재정렬/시간 변경 시 클라이언트 경량 검증기가 영업시간·이동시간·고정 일정 충돌을 즉시 재검증하고 저장 시 서버가 확정 검증한다 [D28].
- 수정 결과가 제약을 위반해도 변경을 차단하지 않되 위반 항목에 경고 배지("이동시간 부족"/"영업시간 외")·사유를 표시한다.
- POI 추가 시 현재 동선 기준 영업시간·이동시간상 삽입 가능한 시간대만 후보로 제시한다.
- 위반이 있는 채로 저장하려 하면 "이대로 두면 ○곳에서 시간이 안 맞아요"를 요약하고 'AI로 자동 보정' 또는 '그대로 저장'을 선택하게 한다.
- 'AI로 자동 보정'은 위반 구간의 시각·순서만 조정하며 POI를 추가·삭제하지 않는다 [G49].
- '그대로 저장' 시 위반 상태를 자동 변경 없이 보존하고 저장 후 활성 일정 화면에서도 위반 블록을 지속 가시화한다.
**유닛 비고**: 클라 경량 검증기+서버 확정 검증(D28), AI 보정 최소 변경(G49) — U11 공동편집이 재사용.

### US-E5-08 일정 저장과 여행 중 실행 연계 `[1차]` · U5

**원 스토리·결정**: PRD 06 스토리 8 · D25
**수용 기준**:

- 저장 일정을 여행 중 Plan-B 재계획의 입력 기준선으로 사용할 수 있다.

**유닛 비고**: U5는 확정본·current 기준선 공급(CP3 공급자)까지 — 실행 허브 수신은 U6.

### US-E5-09 생성 실패·지연 폴백 `[1차]` · U5
**요지**: 여행자로서 일정 생성이 실패하거나 오래 걸려도 빈 화면 대신 쓸 수 있는 결과·진행 상황을 보고 싶다 — 기다림·실패 속에서도 여행 준비를 멈추지 않으려고.
**원 스토리·결정**: PRD 06 스토리 9 · G161, D38
**수용 기준**:
- 외부 POI/영업시간/이동시간/지도 API가 실패해도 내장 결정론적 솔버(규칙 점수 + TOPTW + 제약 검증)로 시간 제약을 만족하는 일정을 생성하고 "일부 추천이 기본 모드로 생성되었어요"를 표시한다.
- LLM 취향 해석/설명 단계 실패 시 일정은 결정론적 솔버 결과로 제공하고 설명 문구만 생략하거나 "추천 이유를 불러오지 못했어요"로 표시한다.
- 완료된 첫 1일 일정을 수 초 내에 먼저 노출하고 나머지 날짜를 백그라운드로 이어 채우며(20초는 전체 완료 한계) 생성 중 단계 텍스트·진행률·경과 시간을 표시한다 [D38].
- 생성 중 '취소' 또는 '백그라운드로 계속'을 선택할 수 있고 취소 시 만들어진 부분 일정을 초안으로 저장하고 '이어서 생성' 진입점을 제공한다 [G161].
- 모든 생성 경로 실패 시 등록 숙소·시각 고정형 필수 방문지만 배치된 최소 일정을 제공하고 "추천을 다시 시도" 동작을 노출한다.
**유닛 비고**: 5초/20초(D38)·취소 시 부분 초안+이어서 생성(G161)·결정론 폴백.

### US-E5-10 생성 방식 선택(완전 AI/같이 고르기/직접 만들기) `[1차]` · U5
**요지**: P3 하람으로서 일정을 AI에 통째로 맡기거나, 장소별로 같이 고르거나, 직접 만들고 싶다 — 내가 원하는 만큼만 관여하며 일정을 완성하려고.
**원 스토리·결정**: PRD 06 스토리 10 · D25, G46, G48, G136
**수용 기준**:
- 완전 AI 자동: 등록 숙소·취향·필수 방문지를 입력으로 전체 일정을 한 번에 생성하고 초안 개별 슬롯을 고정(LOCK)하거나 교체할 수 있다.
- 재생성 시 LOCK 슬롯·체류 시간 고정값·수동 추가 POI를 모두 고정 블록으로 보존(warm-start)하고 나머지만 재생성하며, 여행당 활성 일정은 1개로 유지하고 이전 상태는 변경 이력으로 추적한다 [G46, G136].
- 같이 고르기: 슬롯마다 사용자가 고른 컨셉/테마와 반경 안 후보를 제시하고 선택하면 다음 슬롯으로 진행한다. 반경 기준점은 직전 확정 슬롯 위치이며 첫 슬롯은 등록 숙소 [G48]. 반경을 넓히면 더 먼 후보가 예상 이동 거리 표기와 함께 포함되고 반경 밖 후보는 비활성·회색 처리한다.
- 직접 만들기: 빈 일정에서 장소를 검색·추가해 구성하되 추가 시점마다 스토리 7과 동일하게 제약을 재검증한다.
- 분기 선택 화면 각 방식 카드에 한 줄 결과 예고·예상 소요·인터랙션 양을 표시해 고르기 전에 무엇을 얼마나 해야 하는지 알 수 있다.
- 분기 도중 방식 전환이 가능하며(예: 같이 고르기 → 나머지 완전 AI) 이미 고른 슬롯을 보존한 채 남은 일정만 생성한다.
- 세 방식 모두 최종 시각·순서·이동 거리는 솔버 검증값이며 이동 소요시간은 표시하지 않는다 [D25]. 같이 고르기에서 반경 내 후보 0건이면 반경 확대 또는 컨셉 변경을 제안한다.
**유닛 비고**: 같이 고르기 기준점(G48)·재생성 보존(G46/G136).

### US-E5-11 숙소 나중 등록(동선 기반 숙소 권역 추천) `[1차]` · U5
**요지**: P3 하람으로서 숙소 없이 가고 싶은 장소부터 일정을 짠 뒤 그 동선에 맞는 숙소를 추천받고 싶다 — 숙소부터 정하라는 강요 없이 내 방식대로 준비하려고.
**원 스토리·결정**: PRD 06 스토리 11 · D25
**수용 기준**:
- 완성 일정의 방문지 분포(동선 무게중심)·방문지 간 평균 이동 거리를 기준으로 '이 동선에 묵으면 이동이 최단'인 숙소 권역을 지도로 추천한다.
- 추천 숙소 후보를 전체 일정 평균 이동 거리 순으로 제시하고 특정 숙소 기준 재정렬 시 이동 거리 개선(before/after)을 '추정 이동 거리'로 표시한다 — 소요시간은 표시하지 않는다 [D25].
- 무게중심 기반 추천은 추정이며 개별 동선의 전역 최적을 보장하지 않음을 표기한다.
- 추천 숙소를 등록하면 그 숙소를 출발·복귀 기준점으로 스토리 1의 일정 생성·재정렬을 정상 수행한다.
- 숙소를 끝까지 등록하지 않은 날(당일치기·이동일)은 숙소 없이 동선만으로 일정을 유지한다.
**유닛 비고**: M8 `recommendStayZone` 소유 — U4 Case B 온램프의 완결.

### US-E5-12 일정 확정과 확정본 열람 `[1차]` · U5
**요지**: 여행자로서 완성한 일정을 확정해 여행 시작 전 최종본으로 잠그고 다시 확인하고 싶다 — 출발 직전까지 흔들리지 않는 기준 일정을 갖으려고.
**원 스토리·결정**: PRD 06 스토리 12 · D14, D20
**수용 기준**:
- '일정 확정' 시 plan 스냅샷을 동결(불변)하고 여행 중 편집·Plan-B 변경은 현재본(current)에만 반영한다 [D14].
- 확정 후 확정 일정 화면(시간표/지도 토글·읽기 전용)으로 진입시키고 D-day·출발일 등 여행 시작 맥락을 표시한다.
- 확정의 의미·이점(여행 시작 전 최종본 잠금 등)을 확정 화면 또는 확정 직전에 한 줄로 설명한다.
- 확정 후 '일정 수정'으로 편집 진입 시 '편집 중' 상태로 전환되고 저장 후 재확정해야 확정 상태로 복귀한다 [D20].
- 확정 일정은 여행 중 실행·Plan-B의 입력 기준선이 된다.
**유닛 비고**: plan 동결(D14)·확정 해제→재확정(D20). `ItineraryConfirmed` 발행(CP5 — U8 리마인드 소스).

## Epic 6 — 여행 중 현장 실행 (E6 / U6, 3개)

여행 중 확정 일정을 현장에서 정상 실행하는 흐름(도착 확인·방문 상태 전이·장소 상세·다음 예정지 이동). 활성 일정 상태 머신·도착 확인 프롬프트·실제 체류 시간 측정을 소유하는 **신설 M18 Trip Execution 모듈이 담당한다(Δ7)**. Plan-B 재계획과 구분되는 실행 흐름이며 기록 저장 규칙은 E8을, 거리 산출 기준은 E5를 정본으로 따른다. E6+E7은 단일 유닛 U6에 통합 배정된다.

### US-E6-01 도착 확인과 방문 시작·완료 처리 `[1차]` · U6
**요지**: 여행자로서 일정상 장소에 도착하면 도착을 확인하고 방문을 시작·완료 처리하고 싶다 — 여행 진행이 일정에 반영되고 다음 행동으로 자연스럽게 이어지도록.
**원 스토리·결정**: PRD 08 스토리 1 · D23, D27, G62
**수용 기준**:
- 포그라운드 상태에서 지오펜스 진입 감지 시 '도착하셨나요?' 확인 프롬프트를 노출하되 도착 확정·방문 완료는 항상 사용자 탭으로만 처리한다(자동 확정 없음) [D23].
- 위치 감지는 포그라운드 한정이며 백그라운드 위치 권한은 요청하지 않는다 [D27, G62].
- 사용자가 수동으로 '도착'을 표시하는 경로를 항상 제공하고 도착 시 방문 시작/완료 체크·사진·메모 진입점을 활성 일정 위에 노출한다.
- 방문 완료는 활성 일정 진행 상태(완료/진행 중/다음)에 반영되고 실제 체류 시간 측정 기준이 되어 Plan-B 자동 트리거(체류 초과·이동 지연)의 입력이 된다. 방문 종료 시각은 다음 장소 체크 시각으로 추정한다 [D23].
- 장소 스킵과 방문 완료 취소를 지원한다 [D23].
- 위치 권한이 없거나 GPS 정확도가 낮으면 자동 도착 감지를 보류하고 수동 체크인으로 대체한다.
- 방문 체크·사진·메모의 저장·기록 규칙은 E8을 정본으로 따른다(본 화면은 현장 진입점만 제공).
**유닛 비고**: 확인 프롬프트 방식(D23 — 확정은 항상 사용자 탭)·실제 체류 측정. `VisitChecked` 발행(CP4 — U7 actual 소스).

### US-E6-02 현장 장소 상세 확인 `[1차]` · U6
**요지**: 여행자로서 지금 방문 중(또는 예정)인 장소 정보를 현장에서 바로 확인하고 싶다 — 얼마나 머물지와 다음 일정까지 여유를 그 자리에서 판단하려고.
**원 스토리·결정**: PRD 08 스토리 2 · G67, G199, D25
**수용 기준**:
- 현재 장소의 영업시간·예상 체류 시간·주변 추천을 보여주고 다음 일정까지 여유 시간을 함께 표기한다.
- '다음 일정까지 여유 시간'은 다음 장소 계획 시작 시각까지 남은 시간(이동시간 미반영 단순 차이)으로 계산한다 [G67, D25].
- 예상 체류 시간 등 추정치는 추정임을 명시하고 산출·데이터 출처를 표기한다.
- 혼잡도는 1차 범위에서 제외하며 화면에는 '미확인'으로 통일 표기한다 [G199].
- 장소 상세 데이터(영업시간 등) 누락 시 빈칸 대신 "미확인"으로 표기한다.
**유닛 비고**: 여유 시간(G67)·혼잡도 '미확인'(G199)·소요시간 미표시(D25).

### US-E6-03 다음 예정지 거리 확인과 외부 길찾기 `[1차]` · U6
**요지**: 여행자로서 활성 일정 화면에서 다음 예정지까지 거리를 바로 보고 외부 지도 앱으로 길찾기를 이어가고 싶다 — 별도 화면을 오가지 않고 다음 장소로 바로 이동하려고.
**원 스토리·결정**: PRD 08 스토리 3 · D25, G65, G66, D23
**수용 기준**:
- 별도 길찾기 전용 화면 없이 활성 일정 화면 '다음' 행/맥락 액션에 현재 위치→다음 예정지 거리(거리 기반 추정)를 인라인 표시하고 추정치임을 명시하며 차량/도보 예상 소요시간은 표시하지 않는다 [D25].
- 거리는 화면 진입/포커스 시 1회 산출하고 수동 새로고침을 제공한다 [G65].
- 실제 단계별 길안내·소요시간(대중교통 노선·환승 포함)은 '외부 지도 앱으로 열기'로 위임하며 기기 설치 지도 앱 목록 시트에서 선택하게 한다 [G66].
- 외부 지도앱에서 앱으로 복귀하면 현재 활성 일정·다음 행동(도착 체크인 등)으로 매끄럽게 복귀시키고 복귀 시 다음 예정지에 근접해 있으면 도착 확인 프롬프트를 노출한다 [G66, D23].
- 연결 가능한 외부 지도 앱이 없으면 웹 지도 길찾기로 폴백, 그것도 불가하면 앱 산출 거리 요약(직선/도로 거리·추정)을 최종 안내로 표시한다 — 소요시간·수단별 경로는 표시하지 않는다 [D25].
**유닛 비고**: 직선거리(G65)·외부 지도 앱 시트(G66)·복귀 시 도착 프롬프트.

## Epic 7 — Plan-B 재계획 (E7 / U6, 13개)

여행 중 날씨·휴무·이동 지연·체력 저하 같은 변수가 발생했을 때, 고정 제약(등록 숙소·시각 고정 일정)을 지키면서 남은 일정을 안전하게 재구성한다. E6과 함께 유닛 U6에 배정된다.

### US-E7-01 수동 재계획 요청 `[1차]` · U6
**요지**: 여행자로서 여행 중 활성 일정 화면에서 사유(날씨 악화·임시 휴무·이동 지연·일정 취소/예약 마감·체력 저하)를 골라 재계획을 요청하고 싶다 — 변수가 생겼을 때 즉시 일정을 다시 짜려고.
**원 스토리·결정**: PRD 07 스토리 1 · D19, D21, D22
**수용 기준**:
- 5가지 사유 중 하나를 선택하거나 "사유 없음(그냥 바꾸고 싶음)"으로도 재계획을 요청할 수 있다.
- 선택 사유는 LLM 취향·맥락 해석 입력으로 전달되되 최종 후보는 솔버(OPTW/TOPTW) 검증을 통과한 결과만 노출된다.
- 재계획은 '여행 중' 상태에서만 가능하다. '여행 중'은 여행 날짜 구간(시작일~종료일) 진입으로 판정하며 종료는 종료일 다음날 00:00 자동 종료와 수동 '여행 종료' 버튼을 병행한다 [D19].
- 등록 숙소가 0개여도 여행 날짜 구간 안이면 재계획을 요청할 수 있다(숙소=일정 생성 기준일 뿐, 여행 중 상태 진입 조건 아님).
- 여행 날짜 겹침 생성이 차단되므로 진행 중 여행은 항상 최대 1개이며 재계획 대상 여행의 모호성이 없다 [D21].
**유닛 비고**: CP3 소비자 — 확정/current 기준선(D14) 위에서 동작.

### US-E7-02 자동 트리거 감지와 재계획 제안 알림 `[1차]` · U6
**요지**: 여행자로서 앱이 날씨·휴무·이동 지연·체류 초과를 감지해 재계획을 선제 제안하길 원한다 — 문제를 뒤늦게 알아채 일정이 무너지는 일을 막으려고.
**원 스토리·결정**: PRD 07 스토리 2 · D27, D10, G58, G195
**수용 기준**:
- 감지 조건 4종: (a) 다음 방문지 도착 시간대 강수확률 60% 이상 또는 기상특보 — 기상청 공공데이터포털 기반 [D10], (b) 다음 방문지 당일 임시 휴무/영업시간 변경, (c) 다음 방문지까지 예상 이동시간의 계획 대비 임계 초과, (d) 현재 POI 체류 초과로 다음 고정 일정(숙소 복귀·시각 고정 방문지) 도착 위협.
- 감지 방식은 하이브리드: 날씨·휴무는 서버 배치 폴링+푸시, 위치 의존 트리거(이동 지연·체류 초과)는 클라이언트 포그라운드 한정 감지 [D27].
- 알림은 비차단 배너/인앱 칩이며 제안일 뿐 — 사용자가 명시적으로 "대안 보기"를 눌러야 재계획 흐름이 시작되고 자동으로 일정을 바꾸지 않는다. 푸시는 심각 사유(기상특보·고정 일정 도착 위협)에 한정하고 푸시 탭 시 해당 트리거의 '대안 보기'로 진입한다.
- 동일 사유 알림은 동일 방문지에 1회만 노출하고 닫으면 재노출하지 않는다. 전역 빈도 상한은 초기값 시간당 2회/하루 8회이며 초과·동시 발화는 묶어 1회로 제시한다. 민감도 3단계(적게/보통/많이)가 상한을 ±50% 조정하고 2회 연속 무시하면 당일 동일 사유를 억제한다 [G58, G195].
- 알림에 자동 트리거가 사용한 외부 데이터(날씨/휴무) 출처·감지 시각을 표기한다.
- 외부 API(날씨/영업시간) 무응답 시 자동 알림을 띄우지 않고 수동 재계획 경로만 유지한다(허위 알림 금지).
**유닛 비고**: 하이브리드 감지(D27)·기상청(D10, P3 선결)·상한/민감도/억제(G58). `TriggerFired` 발행(CP5). 푸시 발송 자체는 U8(US-E09-03).

### US-E7-03 재계획 영향 분석 기준 입력 `[1차]` · U6
**요지**: 여행자로서 재계획이 내 현재 위치·시각과 고정 제약을 기준으로 분석되길 원한다 — 실제 실행 가능한 대안만 받아보려고.
**원 스토리·결정**: PRD 07 스토리 3 · D25
**수용 기준**:
- 영향 분석 입력: 현재 위치(GPS)·현재 시각·남은 방문 예정지·등록 숙소의 체크인/아웃 일시·위치·시각 고정형 필수 방문지 확정 시간·각 POI 당일 영업시간·POI 간 이동시간(거리×이동 수단별 속도의 거리 기반 추정 — 내부 계산용이며 화면에 소요시간 미표시 [D25]).
- 등록 숙소의 체크인/아웃 일시·위치는 변경 불가 고정 제약이다. Plan-B는 숙소 예약(OTA 예약)을 바꾸지 않으며 숙소를 출발·복귀 기준점으로 고정한 채 그날 방문 일정만 재구성한다.
- 시각 고정형 필수 방문지는 고정 제약으로 취급하고 재배치 대상에서 제외한다.
- 영향 분석은 현재 시각 이후 남은 일정에만 적용하고 이미 완료된 방문지는 변경하지 않는다.

### US-E7-04 대안 후보 2~3개 추천 `[1차]` · U6
**요지**: 여행자로서 남은 일정·고정 제약과 충돌하지 않는 대안 후보 2~3개를 받아보고 싶다 — 상황에 맞는 대안을 비교해 고르려고.
**원 스토리·결정**: PRD 07 스토리 4 · G53
**수용 기준**:
- 각 후보는 솔버 검증을 통과한다: 숙소 체크인/아웃 시간 창 준수, 모든 방문지 영업시간 내 방문, POI 간 이동시간 반영 시 시간 겹침 없음, 시각 고정형 필수 방문지 시작 시각 준수.
- 후보 POI는 RAG로 그라운딩한 실재 장소이며 검증 레이어를 통과하지 못한(환각·폐업·좌표 불명) POI는 제외한다.
- 대안 후보는 사용자 저장 장소를 우선 소싱하고 부족하면 주변 신규 POI로 확장한다 [G53].
- 후보는 선택 사유에 부합한다(날씨 사유면 실내 우선, 체력 저하면 이동거리·방문 수 적은 대안 우선).
- 솔버가 대안 0개를 산출하면 빈 화면 대신 폴백 3종 — "남은 방문지 1개 건너뛰기 / 휴식 모드로 전환 / 수동 일정 수정" — 을 제시하고 대안이 없는 이유를 한 줄로 설명한다.
**유닛 비고**: 저장 장소 우선 소싱(G53 — U3 M7 재사용), 10초 예산(D38).

### US-E7-05 대안 후보별 정보 표시 `[1차]` · U6
**요지**: 여행자로서 각 대안의 추천 이유·거리·체류 시간·여유 시간을 함께 보고 싶다 — 근거를 이해하고 대안을 판단하려고.
**원 스토리·결정**: PRD 07 스토리 5 · D25, Δ1
**수용 기준**:

**수용 기준**:

- 위치 권한이 없거나 미동의 상태에서는 GPS를 수집·저장하지 않고 좌표 없는 방문 기록을 생성한다 [D34].
- 위치 권한 거부 시 '수동 체크인'을 기본 기록 수단으로 제공하고 사용자가 장소를 직접 선택해 방문 처리할 수 있다.
- 설정에서 위치정보 동의를 ON/OFF할 수 있고 OFF 전환 시 그 시점 이후 GPS 수집을 즉시 중단한다.
- GPS 신호가 없거나 정확도가 낮으면 자동 방문 판정을 보류하고 수동 체크인으로 대체 — 자동 판정 시에도 확정·완료는 항상 사용자 탭(확인 프롬프트 방식) [D23].
**유닛 비고**: 옵트인 동의 모델은 U1(N2/D34), 수집은 U6 — U7은 기록 귀속·철회 시 파기 연동.

### US-E8-04 계획·실제·변경 이력 구분 저장 `[1차]` · U7
**요지**: 여행자로서 같은 여행의 계획·실제 기록·변경 이력이 구분되어 보이길 원한다 — 계획과 실제가 어떻게 달랐는지 정확히 알려고.
**원 스토리·결정**: PRD 09 스토리 4 · D14, G57
**수용 기준**:
- 동일 여행에 대해 계획(plan, 확정 시점 불변 스냅샷)·현재본(current)·실제 방문 기록(actual)·변경 이력(changelog)을 구분해 저장하고 회고·비교는 plan과 actual을 대조한다 [D14].
- 기록 화면에서 계획·실제·변경 3종 데이터를 라벨·색상·아이콘 등으로 시각 구분 표시한다.
- 계획과 실제가 다른 장소(미방문·추가 방문·순서 변경)를 한눈에 비교할 수 있게 표시한다.
- 각 변경 이력 항목은 변경 시각·변경 전/후 장소·변경 사유(휴무·날씨·이동 지연 등)를 포함하며 항목 단위 diff 통일 스키마(행위자·출처·사유·전/후 값)를 따른다 [G57].
**유닛 비고**: plan/actual/changelog 3계열(D14·G57) — CP3 plan 불변성과 CP4 diff의 수렴점.

### US-E8-05 숙소·날짜 기준 방문 기록 귀속 `[1차]` · U7
**요지**: 여행자로서 그날의 방문 기록이 그날의 기준 숙소·날짜에 묶이길 원한다 — 여행 기록을 날짜 흐름대로 자연스럽게 돌아보려고.
**원 스토리·결정**: PRD 09 스토리 5
**수용 기준**:
- 등록 숙소(외부 예약 또는 앱 저장 후 등록) 위치·체크인/아웃 날짜를 기준점으로 그날 방문 기록을 해당 숙소·날짜에 귀속시킨다.
- 등록 숙소가 여러 곳(이동 숙박)이면 날짜별로 기준 숙소를 구분해 귀속시킨다.
- 등록 숙소가 없는 날(당일치기·이동일)은 숙소 없이 날짜만을 기준으로 방문 기록을 저장한다.

### US-E8-06 당일 회고 초안 자동 생성 `[1차]` · U7
**요지**: 여행자로서 하루의 방문·사진·메모·변경 이력로부터 회고 초안이 자동으로 만들어지길 원한다 — 힘들이지 않고 하루를 정리하려고.
**원 스토리·결정**: PRD 09 스토리 6
**수용 기준**:
- 방문 장소·사진·메모·변경 이력을 기반으로 당일 회고 초안을 자동 생성하며 방문 장소 수·이동 거리·첨부 사진 수·주요 변경 이력을 포함한다.
- 회고 생성 완료 시 인앱 카드 또는 푸시로 알린다.
- 회고 생성(AI 호출) 실패·시간 초과 시 빈 화면 대신 '방문 N곳·이동 Nkm·사진 N장' 기본 카드를 제공한다.
- 부분 데이터(예: 사진 0장·GPS 미동의·메모만 존재)에서는 가용 데이터만으로 생성하고 생성 불가 항목(사진 하이라이트·지도 경로)은 제외하되 '사진 없음', '위치 기록 없음'처럼 누락을 명시한다.
- 당일 기록이 전혀 없으면 '오늘 기록된 활동이 없습니다' 안내와 수동 기록 추가 버튼을 노출한다.
**유닛 비고**: U6 `DayClosed` 트리거(CP4)·C1 상위 티어 경유·실패 시 기본 카드 폴백.

### US-E8-07 회고 초안 수정·재생성 `[1차]` · U7
**요지**: 여행자로서 AI 회고 초안을 내 말로 고치고 덧붙이고 싶다 — 회고가 남의 글이 아니라 내 기록이 되도록.
**원 스토리·결정**: PRD 09 스토리 7 · G78
**수용 기준**:

- 수정 회고는 원본 초안과 별도로 저장하고 수정본을 최종 표시본으로 사용한다.
- 회고 생성 실패 시에도 기본 카드 위에 직접 회고를 작성할 수 있다.
- 회고 재생성을 제공하되 수정본이 있으면 덮어쓰기 경고를 표시하고 사용자 확인 시에만 교체한다 [G78].

### US-E8-08 여행 종료 후 전체 여행 요약 생성 `[1차]` · U7
**요지**: 여행자로서 여행이 끝나면 날짜별 기록·지도 동선·전체 요약이 만들어지길 원한다 — 여행 전체를 한 편의 기록으로 간직하려고.
**원 스토리·결정**: PRD 09 스토리 8 · D19, Δ4, G72, D34
**수용 기준**:
- 여행 종료·요약 생성 트리거는 여행 종료일 다음날 00:00 자동 종료와 '여행 종료' 수동 버튼 병행이다(숙소 유무 무관 단일 규칙) [D19, Δ4].
- 전체 요약에 총 방문 장소 수·총 이동 거리·총 사진 수·날짜별 하이라이트를 포함한다.
- 총 이동 거리는 GPS 동의 구간의 실측 경로 합산과 미동의 구간의 장소 간 거리 추정을 혼합 합산한다 [G72].
- 전체 요약은 지도 기반 히어로(방문 순서 핀·날짜별 동선·기준 숙소 마커) 중심으로 구성하고 방문지마다 첨부 사진을 지도 핀 미리보기로 노출한다.
- 위치정보 동의 구간은 GPS 경로로, 미동의 구간은 수동 체크인 장소를 순서대로 연결한 경로로 지도에 표시한다 [D34].
- 위치 데이터가 전혀 없으면 지도 대신 방문 장소 목록(순서·날짜 포함)으로 대체하고 요약 생성 실패 시 날짜별 기본 카드(방문 N곳·이동 Nkm·사진 N장) 모음 폴백을 제공한다.
**유닛 비고**: U6 `TripEnded` 트리거(D19/Δ4 — CP4 시나리오 2)·이동 거리 혼합 합산(G72).

### US-E8-09 여행 스타일 분석 `[1차]` · U7

**원 스토리·결정**: PRD 09 스토리 9 · G76
**수용 기준**:
- 방문 카테고리 분포·평균 체류 시간·이동 반경·활동 밀도를 기반으로 여행 스타일 분석을 생성한다.

- 분석 결과는 구체적 수치·카테고리로 제시하고(예: '카페 비중 40%', '하루 평균 4곳 방문') 근거가 된 방문 데이터를 함께 표시한다.
- 카테고리 분포는 막대 차트로, 이동 반경·방문 분포는 지도 위 분포점·평균 이동 반경 원으로 시각화한다.
- 분석 가능 임계치는 누적 방문 장소 수 10곳 단일 기준이다. 미달 시 '분석에 필요한 기록이 부족합니다(현재 N곳 / 필요 10곳)' 안내와 온보딩 취향 기반 '임시 스타일 미리보기'(정식 분석 아님 명시)·진행 게이지('N/10곳')를 노출한다.
**유닛 비고**: 방문 10곳 게이트·취향 7종 축 택소노미(G76) — U1 취향 스키마 공유.

**유닛 비고**: 입력만 오프라인(D24/Δ6 — 조회 미보장), 레코드 버전+항목 선택·합집합 병합(G74).

### US-E8-13 SNS 공유 카드 만들기 `[1차]` · U7 · 이연(커뮤니티 게시·EXIF는 U10)

### US-E8-14 캘린더로 지난 여행 탐색 `[1차]` · U7
**요지**: 여행자로서 지난 여행을 월 캘린더에서 한눈에 보고 특정 여행으로 들어가고 싶다 — 시간 축으로 여행 기록을 쉽게 되짚으려고.
**원 스토리·결정**: PRD 09 스토리 14 · D21
**수용 기준**:

## Epic 9 — 알림·마이페이지·설정 (E9 / U8, 14개)

### US-E09-01 숙소 등록·저장 완료 알림 `[1차]` · U8
**요지**: 여행자로서 외부 OTA 예약 등록 또는 숙소 저장을 완료했을 때 성공 알림을 받고 싶다 — 등록이 성공했고 일정 생성 출발점이 설정됐음을 바로 확신하려고.
**원 스토리·결정**: PRD 12 스토리 1 · D32, D12
**수용 기준**:
- 외부 예약 등록 저장 직후 숙소명·위치·체크인/아웃 날짜와 "이 숙소가 일정 생성의 출발점으로 설정되었습니다" 문구를 포함한 알림을 서버가 발송한다(D32).
- 단순 '저장'(찜)과 '등록'(일정 기준점)을 구분하고 저장 완료 알림에는 "일정에 반영하려면 숙소를 등록하세요" 행동 유도를 포함한다.
- 체크인/아웃 날짜가 없는 등록에는 등록 완료 알림 대신 "날짜를 입력해야 일정 생성이 가능합니다" 알림을 발송한다.
- 푸시는 FCM 단일 채널로 발송하고(D12) 동일 알림을 인앱 알림함에도 적재한다(D32).
**유닛 비고**: CP5 소비자 — 소스는 U3 `StayRegistered`/U4 `StayLinkedToTrip` 이벤트.

### US-E09-02 여행 단계별 리마인드 알림 `[1차]` · U8
**요지**: 여행자로서 여행 시작 전·당일·개별 일정 전 단계별 리마인드 알림을 받고 싶다 — 일정을 놓치지 않고 여행을 실행하려고.
**원 스토리·결정**: PRD 12 스토리 2 · D32, D25, G97, G100
**수용 기준**:
- 여행 시작 D-1 "여행 시작 전" 알림, 각 여행일 첫 일정 전 "당일 일정" 요약 알림(기본 오전 8시, 설정 가능), 각 개별 일정 시작 전 "다음 일정" 알림(기본 30분 전, 0/15/30/60분 선택)을 발송한다.
- 알림에 장소명·시작 시각·직전 지점(등록 숙소 또는 직전 일정)으로부터 예상 이동 거리를 포함하고 소요시간은 표시하지 않는다(D25).
- 모든 리마인드는 서버 스케줄링으로 발송하며 일정이 변경되면 발송 시각을 재계산한다(D32).
- 등록 숙소가 없어 일정이 없는 경우 "숙소를 등록하면 일정 알림을 받을 수 있습니다" 안내만 1회 발송한다.
- 출발점 숙소 해제/변경 후 사용자가 일정 재생성을 거부하면 기존 일정은 유지하되 리마인드 발송을 중단하고 일정 화면에 재생성 유도 배지를 표시한다(G97).
- 방해금지 시간(기본 22~08시)에는 리마인드를 억제하고 억제된 알림은 인앱 알림함에 적재한다(G100).
**유닛 비고**: 소스는 U5 `ItineraryConfirmed/Changed`(D32 변경 시 재계산). 출발점 해제 시 중단·재생성 배지(G97).

### US-E09-03 Plan-B 재계획 알림 `[1차]` · U8
**요지**: 여행자로서 여행 중 재계획이 필요한 상황(날씨·휴무·이동 지연)을 알림으로 받고 싶다 — 변수에 즉시 대응해 일정을 살리려고.
**원 스토리·결정**: PRD 12 스토리 3 · D32, D34, G100

- 모든 채널을 꺼도 보안/계정 관련 시스템 알림은 인앱으로 표시한다.
**유닛 비고**: 90일 보존·읽음 관리(D32). '체크인 임박' 등은 종류 목록 제외(Δ10). 커뮤니티 알림 종류는 U10 출시와 함께 활성화.

**수용 기준**:

- 재생성을 거부하면 기존 일정을 유지하되 리마인드를 중단하고 재생성 유도 배지를 표시한다(G97).

**유닛 비고**: U3 데이터 소비 — 저장/등록 통합+출처 라벨(G103), '등록하기' 액션.

### US-E09-08 여행 스타일 분석 확인 `[1차]` · U8

- 철회 후에도 등록 숙소 위치·일정 시각 기반 비실시간 기능(예정 일정 알림·날씨·휴무 트리거)은 계속 동작한다 — 위치 비의존 트리거만 유지(D34).

## Epic 10 — AI 어시스턴트 (E10 / U9, 8개) `[후속 게이트]`

앱 전반에서 호출되는 대화형 오케스트레이션 계층(통역·중개자 — 실현가능성 판단·확정은 항상 솔버 소유). 1차 출시 이후 별도 출시 게이트로 다루며 LLM 호출 아키텍처(D11)·컨텍스트 권한 경계(D31)만 1차 설계에 아키텍처 여지로 반영한다. 로그인 필수(D22/Δ5)에 따라 PRD의 비로그인 어시스턴트 조항은 삭제한다. 전 스토리 `[후속: 어시스턴트]`·유닛 U9.

### US-E10-01 어시스턴트 호출·컨텍스트 전달 `[후속: 어시스턴트]` · U9
**요지**: 여행자로서 주요 화면 어디서든 어시스턴트를 모달/바텀시트로 열어 자연어로 도움받고 싶다 — 막힌 지점에서 화면을 떠나지 않고 다음 행동을 찾으려고.
**원 스토리·결정**: PRD 02 스토리 1 · D22/Δ5
**수용 기준**:
- 숙소 탐색·일정·여행 중 실행 화면의 우하단 FAB(✦)로 진입하고 호출 시 현재 화면 컨텍스트(현재 여행·등록 숙소·일정 상태·직전 추천 결과)를 자동 전달하며 사용한 컨텍스트를 사용자에게 표기한다.
- 예시 프롬프트(quick reply)를 제시하고 거절·위임·폴백을 포함한 모든 응답에 바로 누를 수 있는 다음 행동 1개를 항상 동봉한다(dead-end 금지).
- 컨텍스트 없이(여행·일정 미생성) 열면 일반 도움 모드로 동작하고 일정 관련 요청에는 여행 만들기·숙소 등록 진입점을 안내한다. 비로그인 동작 조항은 제외 — 로그인 필수(D22/Δ5).
- 호출이 네트워크/LLM 오류로 실패하면 오류·재시도를 표시하고 기존(비대화형) 동작으로 계속 진행하게 한다(ADR-0011).
**유닛 비고**: U5 C1 재사용·D31 서버 재조회 — 신규 도메인 로직 없음(퍼사드 소비).

### US-E10-02 대화형 재질의(티키타카) `[후속: 어시스턴트]` · U9
**요지**: P2 민준으로서 대화로 조건을 바꿔 가며("더 조용한 동네로", "예산 낮춰서") 탐색·추천을 다시 받고 싶다 — 필터를 일일이 조작하지 않고 원하는 결과에 도달하려고.
**원 스토리·결정**: PRD 02 스토리 2 · Δ9, D11
**수용 기준**:
- 자연어 재질의를 해당 모듈 조건으로 변환해 재호출한다 — 변환 대상 필터는 04 필터 축(가격대·유형·편의시설·거리)이 정본이며 평점 요청은 외부 OTA 위임을 안내한다(Δ9). 해석은 경량 모델 티어를 사용한다(D11).
- 직전 대화 맥락을 이어받아 누적 조건으로 좁히고 바꾼 조건을 명시적으로 요약해 보여주며 변환 조건은 실제 화면 필터에도 반영된다(대화 ↔ 화면 상태 일관).
- 결과는 외부 데이터에 그라운딩된 실재 항목만 노출하고 0건이면 원인 조건을 설명하고 완화안을 제안한다.
- 잘못 해석했다고 지적받으면 직전 변환을 되돌리고 직접 선택 또는 재입력을 유도한다(ADR-0011).
**유닛 비고**: 경량 티어(D11). 평점 필터 요청은 OTA 위임(Δ9).

### US-E10-03 진행 검토 요청 `[후속: 어시스턴트]` · U9
**요지**: 여행자로서 "지금까지 한 거 검토해줘"로 현재 여행·일정 상태와 문제점을 진단받고 싶다 — 빠뜨린 것과 다음 할 일을 한눈에 파악하려고.
**원 스토리·결정**: PRD 02 스토리 3 · D25
**수용 기준**:
- 현재 여행 상태(등록 숙소·기간·고정 블록·일정 유무·미해결 충돌)를 요약하고 다음 할 일을 짚어 주되 일정상 문제는 솔버가 이미 검증·표기한 위반 정보만 가져와 설명한다(ADR-0008·0015 — 미검증 수치 생성 금지).
- 각 문제에 해결 동선(숙소 등록·다른 날로 이동·AI 재추천)을 실제 화면/모듈 진입점으로 연결한다.
- 제시하는 모든 수치는 솔버 검증값이며 이동 소요시간은 표시하지 않고 거리만 표시한다(D25).
- 검토 대상이 없거나 검증값을 가져올 수 없으면 추측으로 단정하지 않고 가용 범위에서만 요약하며 진입점/안내를 제공한다(ADR-0011).
**유닛 비고**: M6·M8 퍼사드 조회.

### US-E10-04 액션 위임(어시스턴트 단독 확정 금지) `[후속: 어시스턴트]` · U9
**요지**: 여행자로서 어시스턴트 제안을 수락하면 실제 변경은 기존 모듈/솔버가 검증·실행하길 원한다 — 대화로 바꾼 일정도 항상 실행 가능함이 보장되도록.
**원 스토리·결정**: PRD 02 스토리 4 · D20, G18
**수용 기준**:
- 제안 적용 선택 시 자유 편집·재검증(E5 스토리 7) 또는 Plan-B 재계산 흐름으로 넘겨 솔버 재검증 후에만 일정에 반영하고 위반 시 조용히 적용하지 않고 사유·대안을 제시한다.
- 명시적 '적용' 확인 없이 자동 확정·저장하지 않으며 변경 전후를 비교해 보여준다. 최종 시각·순서는 항상 솔버 소유 값이다(ADR-0008·0015).
- 확정 일정 변경 요청은 여행 전이면 확정 해제→재확정 흐름을 안내하고(D20) 여행 중이면 Plan-B 재계획으로 라우팅한다(G18).
- 솔버 재검증 불가(외부 API 실패) 시 자동 적용하지 않고 수동 편집 모드로 폴백하며 미검증 상태를 표기한다(ADR-0011).
**유닛 비고**: 확정은 항상 사용자 UI 액션(D20 상태 머신 준수 — PBT 대상).

### US-E10-05 대화 이력·세션 유지 `[후속: 어시스턴트]` · U9
**요지**: 여행자로서 대화 맥락이 작업 흐름 안에서 이어지고 닫았다 열어도 직전 대화를 다시 보고 싶다 — 같은 맥락을 반복 설명하지 않으려고.
**원 스토리·결정**: PRD 02 스토리 5 · G135, G197, D22
**수용 기준**:
- 대화 이력은 여행 단위 스레드 1개로 유지하고 여행 생명주기에 연동해 보관한다(G135, G197). 이력은 로그인 계정에 귀속된다 — 비로그인 임시 세션 조항은 제외(D22/Δ5).
- 다른 여행/작업으로 전환하면 새 컨텍스트 기준으로 시작하되 이전 대화로 돌아갈 수 있다(맥락 혼선 방지).
- 어시스턴트가 호출한 모듈·반영 변경을 대화 내역에 남기고 실제 변경은 여행 변경 이력(ADR-0013)과 일관되게 추적한다 — 대화 자체는 확정 데이터가 아니다.
- 이력 저장 실패 시 침묵 없이 임시 보관 상태를 표시하고 재연결 시 동기화한다(ADR-0011).
**유닛 비고**: 여행 단위 스레드 1개(G135/G197).

### US-E10-06 가드레일(오·남용 방지) `[후속: 어시스턴트]` · U9
**요지**: 여행자로서 어시스턴트가 주입·유출·유해 요청에 안전하게 대응하길 원한다 — 내 비공개 데이터가 다른 사용자에게 노출되지 않음을 신뢰하려고.
**원 스토리·결정**: PRD 02 스토리 6 · D31, D11
**수용 기준**:
- 역할 변경·시스템 지침 유출 요구에 응하지 않고 가능한 도움 범위로 되돌리며 유해·차별 요청은 사유와 함께 거절한다(반복 시 같은 거절 대신 사용 안내로 전환).
- 타 사용자 비공개 데이터·내부 운영 지표 비노출은 프롬프트 거절이 아니라 서버 재조회 권한 경계로 강제한다 — 클라이언트는 ID만 전달하고 서버가 요청자 권한으로 재조회해 주입하므로 권한 밖 데이터는 LLM 입력에 구조적으로 미포함(D31).
- LLM 호출에 사용자별 빈도 제한(rate-limit)을 두고 과도한 연속 요청 시 재시도 안내를 표시한다(D11 비용·남용 방지).
**유닛 비고**: C1 출구 스키마 검증 재사용·사용자별 rate-limit(SECURITY-11).

### US-E10-07 범위 한계 안내·외부 위임 `[후속: 어시스턴트]` · U9
**요지**: 여행자로서 어시스턴트가 할 수 없는 일(예약·결제·미검증 확정)을 명확히 안내받고 대안으로 연결되길 원한다 — 헛된 기대 없이 올바른 경로로 목적을 달성하려고.
**원 스토리·결정**: PRD 02 스토리 7 · Δ9
**수용 기준**:
- 예약·결제·실거래 요청은 직접 처리하지 않고 외부 OTA 딥링크로 위임하며 "예약·결제는 외부 사이트에서 진행돼요"를 안내한다(ADR-0003·0012).
- "이 시간에 갈 수 있어?"류 실현가능성 질문에는 솔버 검증 결과로 답하거나 검증 흐름으로 연결한다(ADR-0015 — LLM 임의 단정 금지).
- 자체 내비게이션·실시간 가격 보증 등 미제공 기능을 약속하지 않고 외부 지도앱·외부 OTA 라이브 확인으로 연결하며 평점 기반 요청도 외부 OTA 위임을 안내한다(Δ9).
- 범위 밖 요청 반복 시 같은 거절을 되풀이하지 않고 가장 가까운 가능한 대안을 제시한다.
**유닛 비고**: 미제공 기능 위임 원칙(Δ9).

### US-E10-08 어시스턴트 폴백 `[후속: 어시스턴트]` · U9
**요지**: 여행자로서 LLM·솔버·외부 데이터가 실패해도 관찰 가능한 폴백을 받고 싶다 — 어시스턴트 없이도 하던 작업을 끝까지 완료하려고.
**원 스토리·결정**: PRD 02 스토리 8 · D38
**수용 기준**:
- LLM 실패 시 "지금 도우미가 응답하지 못했어요"·재시도를 표시하고 기존(비대화형) 탐색·편집으로 계속 진행할 수 있다 — 핵심 흐름은 어시스턴트 없이도 완결(ADR-0011).
- 의도가 모호하면 임의 실행하지 않고 되묻거나 선택지를 제시하고, 솔버 비가용 시 시각·실현가능성을 추측하지 않고 결정론적 솔버 폴백 또는 수동 편집으로 연결한다(ADR-0008·0015).
- 응답 지연 시 진행 표시를 유지하고 대기 취소 후 기존 흐름 복귀를 허용한다(첫 응답 3초·전체 15초 타임아웃 목표 — D38).
- 어시스턴트 문구와 솔버 검증 결과가 어긋나면 항상 솔버 검증 결과를 기준으로 표시한다(검증값 우선).
**유닛 비고**: 첫 응답 3초·전체 15초(D38).

## Epic 11 — 여행자 커뮤니티 (E11 / U10, 10개) `[후속 게이트]`

다른 여행자의 공개 일정을 둘러보고(읽기전용), 내 일정을 명시적으로 공개하며, 가져오기(복제)·프로필 조회·좋아요·댓글로 반응하는 에픽. 1차 출시 이후 별도 출시 게이트로 다루며 공개 스냅샷 모델(D16)·기본 비공개 원칙·모더레이션 인프라 요건(D35)만 1차 데이터 모델에 선반영한다. 소셜 그래프(팔로우 등)·독립 게시판은 범위 제외. 유닛 U10.

### US-E11-01 공개 일정 둘러보기 `[후속: 커뮤니티]` · U10
**요지**: P3 하람으로서 탐색 랜딩 '여행자 일정'에서 다른 여행자의 공개 일정을 피드형 카드로 둘러보고 싶다 — 직접 짜지 않고도 검증된 동선에서 영감을 얻으려고.
**원 스토리·결정**: PRD 10 스토리 1 · G84, C13, D22
**수용 기준**:
- 공개 일정을 게시물형 카드(제목·목적지·N박 M일·작성자 닉네임·대표 사진·캡션 요약·해시태그·좋아요/댓글 수)로 보여주고 로그인 사용자 대상 읽기전용으로 제공한다(D22).
- 여행지·동행 유형·이동 수단·예산대·기간·여행 시기(상대)로 필터·정렬한다 — 여행 시기는 계절+주중/주말+N박M일 상대 스키마로만 동작하고(G84) 키워드 검색은 제공하지 않는다(C13 — PRD의 '검색'은 필터·정렬 지칭).
- 카드에는 작성자가 공개로 설정한 정보만 사용하고 정확 숙소 위치·절대 날짜·개인 메모를 노출하지 않으며 이미 가져온 일정에는 '가져옴' 표시로 중복 복제를 방지한다.
- 0건이면 필터 완화·'AI로 직접 만들기' 폴백을 제시하고 일부 로딩 실패 시 가용 카드 우선 표시+누락 표기한다(ADR-0011).
**유닛 비고**: 상대 시기 3필드(G84), 검색 없음 — 필터·정렬만(C13).

### US-E11-02 공개 일정 상세 읽기전용 보기 `[후속: 커뮤니티]` · U10
**요지**: 여행자로서 공개 일정을 열어 날짜별 동선·지도·방문지를 자세히 보고 싶다 — 가져올 가치가 있는 일정인지 판단하려고.
**원 스토리·결정**: PRD 10 스토리 2 · D16
**수용 기준**:
- 시간표·지도 2뷰(E5 스토리 6)를 읽기전용으로 재사용해 날짜별 방문지·구간 이동 정보·체류 시간을 표시하고 작성자 미니 프로필·'내 여행으로 가져오기'·신고/숨기기 진입점을 노출한다.
- 표시 내용은 게시 시점 스냅샷으로 고정된 공개본이다(D16). 공개 계획(plan)과 작성자가 공개 선택한 사진·캡션·해시태그만 보여주고 실제 방문 기록(actual)·GPS 경로·개인 메모·정확 숙소 핀은 노출하지 않는다(ADR-0013).
- 상세에 좋아요 버튼·수(US-E11-08)와 댓글 진입·수·스레드(US-E11-09)를 노출한다.
- 작성자가 비공개 전환·삭제했으면(열람 중 전환 포함) 빈 화면 대신 해당 상태 안내를 표시한다(ADR-0011).
**유닛 비고**: U5 확정 스냅샷 소비(D16 게시 시점 고정).

### US-E11-03 작성자 공개 프로필 보기 `[후속: 커뮤니티]` · U10
**요지**: 여행자로서 공개 일정 작성자 프로필에서 여행 스타일·다른 공개 일정을 보고 싶다 — 취향이 비슷한 여행자의 일정을 더 찾으려고.
**원 스토리·결정**: PRD 10 스토리 3
**수용 기준**:
- 프로필에 닉네임·여행 스타일 요약(마이페이지 스타일 카드 공개용 미러)·공개 일정 목록·집계 지표(공개 일정 수·받은 좋아요 총합)를 표시한다.
- 공개 설정 정보만 노출하고 이메일·실명·연락처·비공개 일정·개인 메모는 절대 노출하지 않으며 팔로우·구독·DM·팔로워 수는 두지 않는다(소셜 그래프 제외).
- 프로필에 신고·숨기기 진입점을 노출한다.
- 프로필 비공개·계정 정지 시 '비공개 사용자'로 처리하고 공개 일정 목록을 노출하지 않으며 로딩 실패 시 재시도와 함께 오류를 표시한다(ADR-0011).
**유닛 비고**: 닉네임 라이브 참조(C3 기본값).

### US-E11-04 신고·숨기기(안전 통제) `[후속: 커뮤니티]` · U10
**요지**: 여행자로서 부적절한 공개 콘텐츠·프로필·댓글을 신고하고 특정 작성자를 내 화면에서 숨기고 싶다 — 안전하고 불쾌하지 않은 환경에서 커뮤니티를 이용하려고.
**원 스토리·결정**: PRD 10 스토리 4 · G86, G178
**수용 기준**:
- 모든 공개 일정·프로필·댓글에 신고 진입점·사유 선택(스팸·욕설·음란·개인정보 노출·저작권·기타)을 제공하고 신고자에게 접수 사실을 최소 1회 안내한다.
- 숨기기(차단)는 양방향 적용된다 — 숨긴 콘텐츠·작성자는 둘러보기·검색·추천에서 비노출, 차단 상대도 내 공개 콘텐츠를 볼 수 없으며 숨김은 설정에서 해제할 수 있다. 이미 복제한 사본은 삭제하지 않는다.
- 자동 필터는 텍스트 금칙어 매칭만 적용하고(계정·온보딩 닉네임 금칙어와 동일 기준) 이미지·스팸은 신고 기반으로 처리한다(G86).
- 신고 N회 누적 시 자동으로 '검토 중' 노출 보류 후 운영자가 확정하며(G178) 침묵 삭제 없이 작성자에게 사유를 고지하고 작성자 본인은 영구 삭제 전까지 자기 사본을 조회할 수 있다(ADR-0011).
**유닛 비고**: 자동 금칙어 필터(G86 — U1 C3 확장)·신고 누적 자동 보류(G178).

### US-E11-05 내 일정 공개(게시) `[후속: 커뮤니티]` · U10
**요지**: 여행자로서 내 일정을 커뮤니티에 공개할지 콘텐츠 단위로 직접 선택하고 싶다 — 원하는 범위만 안전하게 공유하려고.
**원 스토리·결정**: PRD 10 스토리 5 · D16, G84, G85, G185
**수용 기준**:
- 모든 콘텐츠는 기본 비공개이며 확정/종료 일정만 콘텐츠 1건 단위로 명시적 공개할 수 있다(미확정이면 '지금 확정하기' 1탭 연결). 공개 전 노출/비노출 항목을 확인·선택한다.
- 공개본은 게시 시점 스냅샷으로 고정되고 원본 변경 반영은 명시적 '공개본 업데이트'로만 수행한다(D16). 언제든 철회(비공개 전환)하면 즉시 전 노출면에서 제거되고 재공개 시 이전 공개 식별자를 재사용하지 않는다.
- 공개 시 닉네임만 표기, 숙소 위치는 권역(동/지역) 마스킹, 날짜는 상대 표현(계절+주중/주말+N박M일 — G84)으로 변환한다. 정확 위치·절대 날짜는 명시적 선택 시에만 노출한다.
- 캡션(최대 1,000자)·해시태그(최대 10개, 표시 전용 — G82/G85)를 입력할 수 있고 금칙어 기준을 적용한다. 사진은 옵트인 공개이며 EXIF GPS는 기본 자동 제거하고 '정확 위치 공개'를 명시 선택한 게시물만 유지를 확인받는다(G185). 제3자 개인정보·자동 마스킹 한계는 게시 전 고지한다.
**유닛 비고**: 마스킹 화이트리스트(C1 공유 뷰 범위)·EXIF 기본 제거(G185 — US-E8-13 이연분의 완결).

### US-E11-06 내 여행으로 가져오기(복제) `[후속: 커뮤니티]` · U10
**요지**: 여행자로서 마음에 드는 공개 일정을 내 여행으로 복제해 편집하고 싶다 — 처음부터 짜지 않고 검증된 동선 위에서 내 여행을 시작하려고.
**원 스토리·결정**: PRD 10 스토리 6 · G156, D22/Δ5
**수용 기준**:
- 복제는 공개 일정 스냅샷을 사용자 소유 새 여행으로 독립 복제한다(원본과 완전 분리 — 상호 영향·자동 반영 없음). 대상은 공개 계획(plan)·장소 목록만이며 actual·GPS·사진·개인 메모·변경 이력은 제외한다(ADR-0013).
- 복제 플로우에 내 여행 날짜 입력을 필수 스텝으로 포함하고(G156) 복제 직후 내 날짜·인원·등록 숙소·취향 기준 재정렬·재검증으로 연결한다(숙소 미등록 시작 허용, 예약은 가져오지 않음 — ADR-0002·0003).
- 원작자 출처 표시(닉네임·'○○님의 일정에서 가져옴')와 복제 시점을 부착하고 사본 재공개 시에도 최초 원작자 표기를 보존·전파한다. 원본이 이후 비공개·삭제되어도 사본은 유지하고 출처 표시만 갱신한다.
- 같은 일정 재복제 시 기존 사본을 안내하고 확인받으며 비로그인 딥링크 진입 시 로그인 게이트 후 원래 동작을 이어간다(D22/Δ5). 일부 POI 불일치는 '확인 필요' 분리 표기로 복제를 막지 않는다(ADR-0011).
**유닛 비고**: 날짜 입력 필수(G156) — U4 여행 생성 검증(D21 등) 경유.

### US-E11-07 내가 공유한 일정 관리 `[후속: 커뮤니티]` · U10
**요지**: 여행자로서 내가 공개한 일정 목록을 한곳에서 보고 각각 해제하고 싶다 — 내 공개 상태를 항상 통제하려고.
**원 스토리·결정**: PRD 10 스토리 7 · D16
**수용 기준**:
- 마이페이지에서 '내가 공유한 일정' 목록(제목·공개 일시·공개 범위)으로 진입하고 각 일정 공개 범위(노출 항목·정확 위치/절대 날짜 여부)를 표기하며 그 설정으로 재진입할 수 있다.
- 각 일정 공개를 즉시 해제할 수 있고 해제 즉시 둘러보기·검색·추천·프로필에서 제거된다. 공개본 갱신은 명시적 '공개본 업데이트'로만 수행한다(D16).
- 이미 복제된 사본은 해제해도 소급 삭제하지 않고 출처 표시만 '비공개로 전환된 일정'으로 갱신한다.
- 공개 일정이 0건이면 빈 목록 대신 공개 가능한 확정/종료 일정으로의 진입 안내를 노출한다.
**유닛 비고**: 공개본 업데이트는 명시적 액션(D16).

### US-E11-08 좋아요 `[후속: 커뮤니티]` · U10
**요지**: 여행자로서 공개 일정에 좋아요를 누르거나 취소하고 좋아요 수를 보고 싶다 — 좋은 일정에 가볍게 반응하고 인기 일정을 가늠하려고.
**원 스토리·결정**: PRD 10 스토리 8 · D18
**수용 기준**:
- 좋아요는 토글(1인 1일정 1회)이며 낙관적 업데이트 후 서버 실패 시 되돌리고 재시도를 안내한다(ADR-0011). 연타는 마지막 상태 1건으로 정규화한다.
- 좋아요한 사람 리스트는 노출하지 않고 집계 수만 표시한다(마스킹 원칙·소셜 그래프 제외).
- 작성자 알림은 묶음 발송('○○ 외 N명이 좋아합니다')하고 self-notification은 제외하며 채널 토글은 알림 설정(US-E09-05)을 따른다. 차단 관계 좋아요는 알림으로 가지 않는다.
- 일정 비공개 전환·삭제 시 집계도 함께 비노출하고 좋아요 남긴 사용자 계정 삭제 시 그 반응도 철회·차감한다(D18).
**유닛 비고**: 삭제 사용자 익명화(D18) 정합.

### US-E11-09 댓글 `[후속: 커뮤니티]` · U10
**요지**: 여행자로서 공개 일정에 댓글을 쓰고 내 댓글을 지우고 부적절한 댓글을 신고하고 싶다 — 일정에 대해 질문·감상을 나누려고.
**원 스토리·결정**: PRD 10 스토리 9 · G85, D18
**수용 기준**:
- 댓글은 단일 레벨 평면(대댓글·수정 없음)이며 최대 300자·오래된순 정렬로 표시하고(G85) '더보기'로 점진 로드한다. 작성자는 닉네임만 표기한다.
- 댓글 본문에 금칙어 기준을 적용해 위반 시 인라인 오류로 게시를 막고 본인 삭제 + 신고(US-E11-04 흐름)로 관리한다. 도배는 빈도 제한·스팸 신호로 처리한다.
- 작성자 알림은 묶음 발송·self-notification 제외를 적용하고 채널 토글은 알림 설정(US-E09-05)을 따른다. 차단한 작성자 댓글은 내 화면에서 비노출(양방향)한다.
- 게시·삭제 실패 시 입력 내용을 보존한 채 재시도를 제공하고(ADR-0011) 댓글 작성자 계정 삭제 시 그 댓글은 삭제·익명화 대상에 포함한다(D18).
**유닛 비고**: 한도(G85)·익명화(D18).

### US-E11-10 신고 큐 운영(최소 내부 도구) `[후속: 커뮤니티]` · U10 · 신규(D35, 운영자)
**요지**: 운영자로서 신고된 콘텐츠를 큐에서 조회하고 보류/복원/삭제 처리하고 싶다 — UGC를 다른 사용자에게 노출하기 전에 최소한의 모더레이션 인프라를 갖추려고.
**원 스토리·결정**: 신규 — requirements.md D35 (커뮤니티 출시 선결) · D35, G178, G179
**수용 기준**:
- 신고 큐 조회 + 보류/복원/삭제 API·웹 화면을 최소 내부 도구로 제공하고 운영자 인증은 수동 관리한다(D35). 이 인프라는 커뮤니티 기능 출시의 선결 조건이다.
- '검토 중' 노출 보류는 신고 N회 누적 시 자동 전이(N은 운영 설정)하고 운영자가 확정한다(G178). 처리 시 작성자에게 사유를 고지하고 침묵 삭제하지 않는다(ADR-0011).
- 계정 제재는 단계적(경고 → 커뮤니티 정지 → 전체 정지)으로 적용한다(G179). 정지 계정 공개 콘텐츠·프로필은 '비공개 사용자'에 준해 비노출 처리한다.
- 운영자 처리 이력(누가·언제·무엇을)이 기록되어 추적 가능해야 한다.
**유닛 비고**: 운영자 페르소나, 웹 어드민(보류/복원/삭제)·제재 상태(G179).

## Epic 12 — 동행 공동 편집 (E12 / U11, 8개) `[후속 게이트]`

여행 소유자가 동행자를 초대해 하나의 일정을 항목 단위 잠금(soft lock)으로 함께 편집하는 비공개 협업 에픽(커뮤니티 복제와 구분). 1차 출시 이후 별도 출시 게이트(ADR-0016)로 다루며 동기화·잠금 아키텍처(D30 — 서버 권위+낙관적 잠금, WebSocket, 최대 10명, 수렴 목표 3초)만 1차 설계에 확장 여지로 반영한다. 유닛 U11.

### US-E12-01 동행자 초대 `[후속: 공동편집]` · U11
**요지**: 여행자로서 내 여행에 동행자를 초대해 일정을 함께 편집하게 하고 싶다 — 동행과 하나의 일정을 같이 완성하려고.
**원 스토리·결정**: PRD 11 스토리 1 · G91, D30, D22/Δ5
**수용 기준**:
- '함께 편집하기'에서 초대 링크 또는 계정(닉네임/이메일) 지정으로 초대하고 초대 링크는 다회용·만료 7일이며 발급 시 권한(편집자/뷰어)을 지정한다(G91). 소유자는 링크를 만료·재발급할 수 있다.
- 수락한 동행자 '함께하는 여행' 목록에 여행이 나타나고 권한 범위 안에서 보거나 편집할 수 있으며 참여 인원은 최대 10명이다(D30).
- 비로그인 사용자가 초대 링크로 진입하면 로그인 게이트 후 초대 수락을 이어간다(D22/Δ5 — 비로그인 예외는 초대 딥링크 진입 한정).
- 만료·회수된 링크 진입 시 빈 화면 대신 '초대가 만료되었어요' 안내를 표시한다(ADR-0011).
**유닛 비고**: 다회용 링크·만료 7일·권한 지정(G91). 비로그인 진입은 딥링크 수신 한정(D22/Δ5).

### US-E12-02 권한별 동작 제어 `[후속: 공동편집]` · U11
**요지**: 여행자로서 참여자가 권한(소유자/편집자/뷰어)에 따라 허용된 동작만 하게 하고 싶다 — 내 여행의 확정·공개·예약 정보가 통제 밖에서 바뀌지 않도록.
**원 스토리·결정**: PRD 11 스토리 2 · C1
**수용 기준**:
- 소유자는 편집·확정·공개·초대/권한 변경/제거·공유 종료 전부, 편집자는 일정 편집만, 뷰어는 읽기만 가능하며 권한은 서버 측에서 강제한다. 소유자 전용 동작은 편집자·뷰어에게 비활성/숨김 처리한다.
- 공유 데이터 범위는 일정(plan)과 숙소 위치·날짜만이며 예약번호·금액·여행 기록·사진·회고는 소유자 전용으로 참여자에게 노출하지 않는다(C1).
- 소유자 권한 변경(편집자↔뷰어)·제거는 대상 화면에 즉시 반영되고 편집 중 강등·제거 시 미저장 편집 처리(저장 시도/폐기)를 명시한 뒤 읽기 전용으로 전환한다.
- 권한 없는 동작 시도에는 침묵 거부 대신 '소유자만 할 수 있어요' 안내·요청 경로를 제시한다.
**유닛 비고**: 소유자/편집자/뷰어 서버 강제(SECURITY-08 — 권한 매트릭스 PBT 전수).

### US-E12-03 항목 단위 잠금 동시 편집·동기화 `[후속: 공동편집]` · U11
**요지**: 여행자로서 동행과 같은 일정을 항목 단위로 번갈아 편집하고 변경이 서로 화면에 동기화되길 원한다 — 충돌 걱정 없이 함께 일정을 다듬으려고.
**원 스토리·결정**: PRD 11 스토리 3 · D30, G89
**수용 기준**:
- 한 참여자가 항목(슬롯·장소·시각) 편집을 시작하면 그 항목은 잠금되어 타인이 동시에 못 바꾸고('○○ 편집 중' 프레즌스) 저장 시 다른 참여자 화면에 반영된다 — 동기화는 서버 권위+낙관적 잠금, WebSocket, 수렴 목표 3초(D30).
- 잠금은 TTL 60~120초+하트비트로 유지되고 소유자가 강제 해제할 수 있다(D30).
- 여러 일자에 걸치는 광역 편집(재정렬·재생성 등)은 영향 일자(day) 전체 잠금으로 승격한다(G89).
- 동기화 지연 시 '동기화 중' 상태를 표시하고 실시간 채널이 끊기면 침묵하지 않고 안내 후 로컬 편집을 허용하며 재연결 시 동기화로 정리한다(ADR-0011).
**유닛 비고**: WebSocket 신규 인프라(D30)·잠금 TTL+하트비트(G89) — U5 슬롯 ID 식별 체계 의존(CP3).

### US-E12-04 편집 충돌 해소 `[후속: 공동편집]` · U11
**요지**: 여행자로서 같은 항목의 편집 충돌이 침묵 덮어쓰기 없이 해소되길 원한다 — 누구의 변경도 모르게 사라지지 않도록.
**원 스토리·결정**: PRD 11 스토리 4 · G93
**수용 기준**:
- 온라인 동시 충돌은 항목 잠금(US-E12-03)으로 예방하고 잠금으로 못 막는 충돌(주로 오프라인 편집 재연결)은 두 버전을 보여주고 충돌을 감지한 당사자가 선택하며 서버는 직렬화 선착순으로 적용한다(G93). 선택 후에도 직전 상태로 되돌릴 수 있다.
- 서로 다른 항목의 편집은 충돌 없이 병합된다.
- 해소 결과는 모든 참여자에게 동일하게 수렴하고 적용/보류된 변경은 변경 이력(US-E12-07)에 남긴다.
- 해소 중 추가 변경이 들어오면 마지막 확정 상태 기준으로 재해소하고 데이터를 유실하지 않는다.
**유닛 비고**: 감지 당사자 선택+서버 직렬화(G93).

### US-E12-05 공동 편집 솔버 재검증 `[후속: 공동편집]` · U11
**요지**: 여행자로서 누가 편집하든 변경이 솔버 재검증을 거치길 원한다 — 여럿이 고쳐도 일정 실행 가능성이 유지되도록.
**원 스토리·결정**: PRD 11 스토리 5 · G90, D25
**수용 기준**:
- 어떤 참여자의 장소 추가·이동·시각 변경도 영업시간·이동시간·체크인/아웃·고정 방문지 충돌을 즉시 재검증하고 위반 항목에 경고 배지·사유를 표시한다(E5 스토리 3·7 계승). 표시 시각·이동 거리는 솔버 검증값이며 소요시간은 표시하지 않는다(D25).
- 위반 상태 저장 시 'AI 자동 보정 / 그대로 저장' 선택을 제시하고 결정·결과를 모든 참여자가 볼 수 있다.
- 편집자는 위반 시 AI 자동 보정만 사용할 수 있고 일정 재생성·어시스턴트 편집은 소유자 전용이다(G90).
- 라우팅·지오코딩 실패 시 직선거리 추정·수동 입력으로 폴백하고 추정치임을 표기한다(ADR-0011).
**유닛 비고**: U5 C2·편집 재검증(D28) 재사용 — 위반 시 AI 최소 보정, 재생성은 소유자 전용(G90).

### US-E12-06 오프라인 편집·재연결 동기화 `[후속: 공동편집]` · U11
**요지**: 여행자로서 오프라인에서도 일정을 편집하고 재연결 시 동기화되길 원한다 — 네트워크가 불안정해도 편집 내용을 잃지 않도록.
**원 스토리·결정**: PRD 11 스토리 6
**수용 기준**:
- 오프라인 중 편집을 로컬에 '동기화 대기'로 보관하고 네트워크 복구 시 공유 일정과 자동 동기화한다.
- 오프라인 편집이 그 사이 다른 참여자 변경과 같은 항목에서 충돌하면 US-E12-04의 충돌 해소로 처리하고 데이터를 임의 폐기하지 않는다.
- 동기화가 끝까지 실패하면 로컬 편집을 보존한 채 '동기화 실패, 다시 시도' 진입점을 제공한다(ADR-0011).
**유닛 비고**: 항목별 버전(낙관적 잠금 D30).

### US-E12-07 공동 편집 변경 이력 `[후속: 공동편집]` · U11
**요지**: 여행자로서 누가·언제·무엇을 바꿨는지 이력을 보고 되돌리고 싶다 — 동행의 변경을 신뢰하고 잘못된 변경을 복구하려고.
**원 스토리·결정**: PRD 11 스토리 7
**수용 기준**:
- 각 이력 항목은 변경 시각·변경자(닉네임)·변경 전/후 값을 포함하고 여행 변경 이력(ADR-0013 change log)에 통합되어 Plan-B 변경·회고와 함께 열람 가능하다.
- 소유자(또는 권한 있는 편집자)는 특정 변경을 직전 상태로 되돌릴 수 있고 되돌리기도 솔버 재검증(US-E12-05)을 거친다.
- 이력 조회 실패 시 침묵 없이 재시도 진입점과 함께 오류를 표시한다(ADR-0011).
**유닛 비고**: U7 changelog 통합 스키마 재사용(G132 — 출처 유형 '공동편집' 기예약).

### US-E12-08 공유 종료·이탈과 소유권 `[후속: 공동편집]` · U11
**요지**: 여행자로서 공동 편집을 종료하거나 스스로 나갈 수 있고 종료 후 소유권이 명확하길 원한다 — 협업이 끝나도 내 여행의 귀속이 흔들리지 않도록.
**원 스토리·결정**: PRD 11 스토리 8 · G94, G160, G95, C8, D18
**수용 기준**:
- 소유자 '공유 종료' 시 여행은 소유자 단독 소유로 돌아가고 동행자는 편집·열람 불가가 되며 동행자는 스스로 '나가기' 할 수 있다. 원본은 항상 소유자 귀속이고 동행자 사본을 자동 생성하지 않는다(커뮤니티 복제와 구분).
- 푸시 알림은 초대·권한 제거·공유 종료에만 발송하고 여행 리마인드·Plan-B 알림은 소유자에게만 발송한다(G95, C8).
- 확정 후에도 편집자는 이력·재검증을 경유해 편집할 수 있으나 여행 중 현장 실행 액션(도착 체크·기록·Plan-B 확정)은 소유자 기준으로만 동작하고 동행자는 공유 뷰로 따라간다(G94, G160). 소유자 주도 Plan-B 확정이 타 참여자 미저장 편집과 충돌하면 US-E12-04·05를 적용한다.
- 소유자가 계정을 삭제하면 해당 여행과 공동 편집 권한·이력도 삭제·정리 대상에 포함하고(D18 연쇄와 일관) 동행자에게 공유 종료를 안내한다.
**유닛 비고**: 현장 액션 소유자만(G94)·탈퇴 연쇄(D18) 정합.
