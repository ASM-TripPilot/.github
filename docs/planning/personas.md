# TripPilot 페르소나

## 페르소나의 성격 — 계정 유형이 아니라 '취향 조합' 대표점

TripPilot의 계정 유형은 **'여행자' 단일**이다(PRD 03, ADR). 가입 시 다른 계정 유형 선택지는 없다. 따라서 아래 3원형(P1·P2·P3)은 서로 다른 계정 종류가 아니다. 이들은 **온보딩 취향 7종(여행 스타일·페이스·예산·동행·활동·이동·음식)의 대표 조합**이며, 개인화·Plan-B·기록 기능이 사용자에 따라 어떻게 달리 동작해야 하는지의 관점 차이를 드러내기 위한 **설계·테스트 기준점**이다.

- 3원형은 '취향 축 기반 3원형' 구성 방식으로 정의된다(story-generation-plan Question 4 = A).
- 유저스토리의 "As a 여행자"는 3원형 모두에 적용된다. 특정 원형 관점이 중요한 스토리만 페르소나 이름(P1 지유·P2 민준·P3 하람)을 명시해 그 관점을 강조한다.
- 3원형과 별개로 보조 액터 **운영자(Admin)**를 정의한다(D35). 운영자는 UGC 모더레이션을 담당하는 어드민으로, 커뮤니티(후속) 출시의 선결 인프라와 연결된다.

각 페르소나의 상세 스토리는 [유저스토리](./user-stories.md), 사용자 여정·시나리오는 [시나리오](./scenarios.md), 에픽 정의는 [에픽](./epics.md)을 정본으로 참조한다.

## P1 지유 — 꼼꼼 계획형

한 줄 정의: 시간 낭비 없는 검증된 동선을 원하고, 일정의 하드 제약을 직접 통제하려는 계획 주도형 여행자.

| 항목 | 내용 |
|---|---|
| 프로필 | 29세 직장인, 연 3\~4회 국내 여행, 여행 2\~3주 전부터 준비 |
| 취향 축 | 페이스: 빡빡하게(5곳+) · 스타일: 관광+미식 · 이동: 대중교통 · 예산: 중간(전체 총액 직접 입력) |
| 목표 | 시간 낭비 없는 검증된 동선 — 영업시간·이동시간이 어긋나지 않는 일정 |
| 불편점 | 블로그·지도·예약 앱을 오가며 직접 검증하는 피로, 계획이 어긋났을 때의 연쇄 붕괴 |
| TripPilot 사용 방식 | 숙소를 먼저 등록하고 필수 방문지를 여러 곳 지정, '같이 고르기'로 슬롯을 직접 통제, 시각 고정형 필수 방문지 활용, 일정 확정 후 D-day 알림 의존 |
| 대표 검증 관점 | 하드 제약(영업시간·이동시간·고정 블록) 무위반, 확정 해제→재확정 흐름(D20), 필수 방문지 한도(G40), 시각 고정 충돌 처리 |
| 기술 숙련도·태도 | 앱·도구 활용에 능숙하나 자동 추천을 그대로 믿지 않고 근거·통제권을 요구하는 편 |

## P2 민준 — 균형 실속형

한 줄 정의: 준비 시간이 부족해 AI에 맡기되, 가족 조건과 예산에 맞는 신뢰 가능한 결과를 원하는 실속형 여행자.

| 항목 | 내용 |
|---|---|
| 프로필 | 35세, 가족(아동 동반) 여행 중심, 연 1~2회, 준비 시간 부족 |
| 취향 축 | 페이스: 균형있게(3~4곳) · 스타일: 휴양+자연 · 이동: 렌터카 · 동행: 가족 · 예산: 구간 선택(중간) |
| 목표 | 최소 입력으로 무난하고 실패 없는 일정 — AI에게 맡기되 결과는 신뢰 가능해야 |
| 불편점 | 가족 조건(아이 동반·이동 최소화)을 반영한 추천 부재, 예산 초과 |
| TripPilot 사용 방식 | 온보딩 취향 일부 건너뛰기, '완전 AI 자동'으로 초안 생성 후 마음에 안 드는 슬롯만 교체(LOCK), 예산은 소프트 가중치로 반영(G47), 숙소 미등록 상태에서 일정 먼저 생성하는 온램프 사용 |
| 대표 검증 관점 | 중립 기본값 동작(취향 미설정 시 일정 생성 무실패), 재생성 시 LOCK 보존(G46), 전체 총액 예산 파생 계산(D26), 다중 숙소·가족 동행 필터 |
| 기술 숙련도·태도 | 최소한의 조작으로 '적당히 좋은' 결과를 얻길 원함 — 세밀 조정보다 위임 선호 |

## P3 하람 — 즉흥 유연형

한 줄 정의: 계획은 뼈대만 세우고 현장 상황(날씨·기분·체력)에 따라 즉시 재구성하려는 유연형 여행자.

| 항목 | 내용 |
|---|---|
| 프로필 | 24세 대학원생, 혼자 또는 친구와 즉흥 주말 여행, 당일 변경 잦음 |
| 취향 축 | 페이스: 느긋하게(1~2곳) · 스타일: 카페+야경 · 이동: 도보+대중교통 · 예산: 저가 |
| 목표 | 현장에서의 유연함 — 계획은 뼈대만, 상황(날씨·기분·체력)에 따라 바로 재구성 |
| 불편점 | 갑작스런 비·휴무에 대안을 급히 검색하는 스트레스, 기록을 남기고 싶지만 정리가 귀찮음 |
| TripPilot 사용 방식 | 여행 중 Plan-B 수동/자동 트리거의 주 사용자, 휴식 모드·즉석 방문 추가, GPS 기록 옵트인으로 자동 발자취, AI 회고 초안 의존, 자정 초과 야경 활동(D29) |
| 대표 검증 관점 | Plan-B 10초 응답(D38)·당일 잔여 재정렬(C10), 트리거 알림 피로 억제(G58), 오프라인 기록 입력 큐(Δ6), 방문 확인 프롬프트(D23) |
| 기술 숙련도·태도 | 앱을 가볍게 쓰며 설정에 시간 쓰기 싫어함 — 건너뛰기·일괄 탈출구·자동화 선호 |

## 보조 액터: 운영자 (Admin)

한 줄 정의: 커뮤니티(후속)의 UGC 안전을 지키는 어드민. 3원형과 달리 여행자 앱이 아니라 최소 내부 웹 도구를 사용한다.

| 항목 | 내용 |
|---|---|
| 역할 | UGC 신고 큐 처리(보류/복원/삭제), 계정 제재, 금칙어 사전 관리 |
| 도구 | 최소 내부 웹 도구(D35) — 커뮤니티 출시 선결 인프라 |
| 관련 범위 | [후속: 커뮤니티] 에픽의 모더레이션 스토리 |
| 주요 규칙 | 신고 N회 누적 시 '검토 중' 자동 전이 후 운영자 확정(G178), 단계적 제재(경고→커뮤니티 정지→전체 정지, G179), 처리 이력(누가·언제·무엇을) 추적 가능 |

운영자의 유일한 직접 스토리는 US-E11-10(신고 큐 운영, 최소 내부 도구)이며, 이 인프라는 커뮤니티 기능 출시의 선결 조건이다.

## 페르소나 ↔ 유저스토리 매핑

유저스토리 128개 중 대부분은 "As a 여행자"로 3원형 공통에 해당한다. 아래 표는 **특정 페르소나 관점이 명시된 스토리**만 모은 것으로, 각 페르소나가 어떤 스토리의 대표 관점으로 지정되었는지 보여준다. 스토리 본문·수용 기준은 [유저스토리](./user-stories.md)를 정본으로 참조한다.
