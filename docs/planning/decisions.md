# 핵심 결정 (ADR & 결정 레지스터)

> 출처: aidlc/docs/PRD/13-구현결정.md · aidlc/aidlc-docs/inception/requirements/requirements.md · aidlc-docs에서 2026-07-05 추출 · 이후 본 문서가 정본이다.

이 문서는 TripPilot의 **'왜 그렇게 정했나'의 정본**이다. 아키텍처 결정 기록(ADR-0001~0017), 모듈 분해 결정(M1~M18), 확정 결정 레지스터(D01~D38), 결정에 따른 PRD 수정 델타(Δ1~Δ10), 신규 요구사항(N1~N8), 확정 기본값(Gap 레지스터), 후속 이연 결정, 비개발 선결 과제(P1~P9)를 한곳에 모은다. 다른 기획·설계 문서(아키텍처, 도메인 모델, NFR, 인프라, 유닛 상세)는 개별 결정의 근거를 물을 때 이 문서의 ID를 참조한다.

## 2. 아키텍처 결정 기록 (ADR-0001 ~ ADR-0017)

각 ADR을 **맥락 / 결정 / 근거 / 영향·연관**으로 정리한다. ADR은 상위 원칙이고, 이후 D-레지스터 결정이 구현 방식·개정 사항을 확정한다(개정된 ADR은 해당 항에 명시).

### ADR-0001 — B2C 여행자 전용 플랫폼

- **맥락**: 초기 구상은 B2B2C(숙박업주 + 여행자)였다. 피벗으로 사용자군을 단일화한다.
- **결정**: TripPilot의 최종 구조는 **B2C 여행자 전용 플랫폼**이다. 숙박업주(호스트) 사이드 제품(입점/숙소 등록/AI 리스팅 빌더/호스트 예약관리/호스트 재무 대시보드/AI 경영 에이전트)은 제공하지 않는다. 모든 기능은 여행자 단일 사용자군을 대상으로 한다.
- **근거**: 호스트 사이드 제거로 범위·복잡도 축소, 여행자 경험에 집중.
- **영향·연관**: 원본 19개 모듈 중 호스트 사이드 3개 + OTA 모델로 불필요해진 2개(객실 재고·결제)를 제거 → 모듈 재편(§3). Auth는 사용자 유형 분기 없이 여행자 단일 유형만 처리(M1).

### ADR-0002 — 일정 생성의 출발점 = 등록된 숙소

- **맥락**: 자체 예약을 하지 않으므로 '예약=출발점' 원칙을 유지할 수 없다.
- **결정**: AI 일정 생성의 출발점은 **사용자가 외부에서 예약했거나 앱에서 저장한 뒤 '등록'한 숙소**다. 등록 숙소의 **위치와 체크인/체크아웃 날짜**를 일정 생성의 기준 입력값으로 삼는다.
- **근거**: 자체 예약 부재 상황에서 일정의 공간·시간 기준점을 확보하기 위함. 기존 '예약=출발점'을 '등록된 숙소=출발점'으로 계승.
- **영향·연관**: M4(Saved Accommodation)가 등록 숙소를 관리, 내부 예약 상태·객실 재고는 보유하지 않음. ADR-0004(공간 앵커)로 이어짐.

### ADR-0003 — 숙소 예약 경로 = 외부 OTA 제휴 링크

- **맥락**: 인앱 결제·자체 예약을 배제한다.
- **결정**: 숙소 예약 경로는 **외부 OTA 제휴 링크**다. 앱은 탐색·저장만 제공하고, 실제 예약·결제는 외부 OTA로 이동(딥링크)하여 진행한다.
- **근거**: 결제·재고·정산 부담 제거, 실거래 위험 외부화.
- **영향·연관**: 링크 UI 노출은 M3, 딥링크 생성·클릭 추적·전환 집계는 M5(Affiliate Link)로 분리. ADR-0012(단일 공급 모델)의 하위.

### ADR-0004 — 등록 숙소 위치 = 공간적 기준점(앵커)

- **맥락**: 일자별 동선을 어디를 중심으로 짤 것인가.
- **결정**: 등록된 숙소의 위치를 일정 생성의 **공간적 기준점(앵커)** 으로 사용한다. 일자별 동선·이동 시간·장소 추천은 등록 숙소 위치를 중심으로 산출한다.
- **근거**: 숙소가 매일의 출발·복귀 거점이라는 여행 현실 반영.
- **영향·연관**: 솔버(M8·M10)의 기준점. 숙소 전환일은 출발점=A숙소·복귀점=B숙소 편도 동선으로 모델링(G50).

### ADR-0005 — 액티비티(투어·체험) 제외

- **맥락**: 여행 상품 유형 중 어디까지 다룰지.
- **결정**: **액티비티(투어·체험) 탐색·예약·일정 고정 입력 기능은 범위에서 제외한다.** 여행 일정의 고정 블록은 등록 숙소 + 사용자 지정 필수 방문지로 구성한다. 액티비티 전용 탐색·외부 예약 연동·고정 입력은 두지 않는다.
- **근거**: 범위 집중, 외부 예약 연동 복잡도 회피.
- **영향·연관**: 고정 블록 정의(M6·M8), [scope.md](./scope.md) 제외 항목.

### ADR-0006 — Plan-B = 여행 중 일정 실행 보조 (예약 변경 아님)

- **맥락**: Plan-B가 예약을 바꾸는 기능이라는 오해를 차단한다.
- **결정**: Plan-B는 외부 OTA 링크 예약의 변경 행위가 아니라 **여행 중 일정 실행 보조**다. 일정 차질(지연·휴무·날씨 등) 발생 시 등록 숙소 위치와 남은 일정을 기준으로 대체 동선·장소를 재계획한다.
- **근거**: 앱이 예약을 보유하지 않으므로 예약 변경은 불가·범위 밖. 실행 단계의 회복력에 집중.
- **영향·연관**: M9(트리거 감지)·M10(재계산). 실행 이벤트는 신규 M18(Trip Execution)이 공급.

### ADR-0007 — 여행 기록 = 방문 체크·사진·메모·위치 기반  *(D19/Δ4로 트리거 개정됨)*

- **맥락**: 여행 기록의 구성 요소와 회고 자동 생성 시점.
- **결정**: 여행 기록은 **방문 체크·사진·메모·위치 기반**으로 구성한다. 사용자는 일정상의 장소에 대해 방문 여부를 체크하고 사진/메모를 위치와 함께 남긴다. 원안의 회고 자동 생성 트리거는 '여행 종료 시점(마지막 등록 숙소의 체크아웃 이후)'이었다.
- **근거**: 방문 실적 기반 회고 자동화.
- **영향·연관·개정**: 회고 트리거는 **D19/Δ4로 '여행 종료일 다음날 00:00 자동 종료 + 수동 종료 버튼'으로 개정**되었다(숙소 유무 무관 단일 규칙). ADR-0013(plan/actual/changelog)와 함께 M12·M13이 소유.

### ADR-0008 — 일정 생성은 LLM 단독이 아니다 (LLM·알고리즘 하이브리드)

- **맥락**: LLM만으로 실현 가능한 일정을 만들 수 있는가.
- **결정**: 일정 생성은 **LLM 단독이 아니다.** 지도/장소/날씨 API와 규칙 기반 로직(이동 시간·운영 시간·동선 제약)을 **선연산**한 뒤, 그 결과를 바탕으로 일정을 구성한다.
- **근거**: LLM 환각·실현 불가 배치 방지, 하드 제약은 결정론적 솔버가 보장.
- **영향·연관**: 사용자에게 보이는 시각·이동시간은 솔버 검증값만 사용(ADR-0015 계승). 성능 초과 시 결정론적 솔버 폴백(NFR). LLM은 후보 ID closed-set에서만 선택(POI 그라운딩, [nfr.md](./nfr.md)).

### ADR-0009 — Place Data 표준화 + 거리 기반 이동시간 추정

- **맥락**: 장소 데이터 벤더 종속과, 실시간 경로 API 없이 이동시간을 어떻게 다룰지.
- **결정**:
  - 장소 데이터는 **Place Data 모듈(M7)에서 표준화**하여 단일 외부 API에 종속되지 않는다. 공급원이 바뀌어도 상위 일정 로직은 표준화 스키마만 의존한다.
  - **장소 간 이동시간은 거리 기반 추정**으로 산출한다 — 도로 거리(가용 시) 우선, 직선거리만 있으면 **우회계수**로 보정, 이동 수단별 평균 속도에 대기·환승·신호를 흡수하는 **안전계수**(예: 대중교통 ×1.5·도보 ×1.4) 적용. 실시간 대중교통 노선·환승·시간표·턴바이턴 경로는 제공하지 않는다.
  - 추정은 본질적으로 부정확하므로: **(i)** 사용자에게 표시하는 이동 지표는 **거리만**으로 한정하고 차량/도보 예상 소요시간은 화면에 표시하지 않으며, 지도의 이동 경로는 실제 도로가 아닌 **핀 간 직선**으로 표시한다(일정의 도착 시각·체류 시간은 유지). **(ii)** 솔버는 내부적으로 추정 시간에 **안전 마진(버퍼)** 을 두어 빡빡한 배치를 피하되 그 소요시간 값을 UI에 노출하지 않는다. **(iii)** 자동 재계획 트리거 임계는 추정 오차 위로 설정한다.
  - 실제 단계별 길안내·소요시간(대중교통 포함)은 외부 지도 앱으로 위임한다(ADR-0012).
  - 거리·좌표 산출용 외부 지도 API(국내: 카카오·네이버·TMap 우선)는 단일 벤더에 종속되지 않게 추상화하고, 응답 실패 시 직선거리 추정·수동 입력으로 폴백한다(ADR-0011).
  - 지도 API 약관(영구 캐싱 금지·실시간 호출·출처 표기)은 데이터 모델에 반영한다(ADR-0017).
- **근거**: 실제 경로·길안내를 구현하지 않으므로 소요시간 추정을 UI에 노출하면 오인을 부른다. 거리만 노출 + 솔버 내부 버퍼로 신뢰성과 정확성 사이를 절충.
- **영향·연관**: D08(카카오+TMap+네이버), D25/Δ1(소요시간 미표시 통일), 이동 파라미터 초기값 G106.

### ADR-0010 — 위치정보 동의 시에만 수집·사용

- **맥락**: 위치 기반 기능의 개인정보 처리 원칙.
- **결정**: 위치정보는 **사용자 동의 시에만** 수집·사용한다. 동의가 없으면 위치 기반 기능은 비활성화하고 **수동 입력 경로로 대체**한다.
- **근거**: 개인정보 최소 수집·동의 기반 처리.
- **영향·연관**: ADR-0017(위치정보법)로 확장, D34/N2(3층 동의 모델·법정 로그).

### ADR-0011 — 외부 API/AI 실패 시 수동 수정 fallback (침묵 실패 금지)

- **맥락**: 다수 외부 의존성과 AI 산출의 실패 가능성.
- **결정**: 외부 API(지도/장소/날씨/OTA 링크) 실패 또는 AI 자동 산출 실패 시 **수동 수정 fallback을 제공**한다(침묵 실패 금지). 자동 산출이 불가하면 사용자가 일정/숙소/장소를 직접 편집할 수 있게 한다.
- **근거**: 복원력·사용자 신뢰. 오류를 숨기지 않고 항상 대안 경로 제시.
- **영향·연관**: 전 외부 호출에 타임아웃 + 서킷 브레이커 + 우아한 성능 저하(RESILIENCY-10, [nfr.md](./nfr.md)). 오프라인 충돌·공동편집 충돌도 사용자 선택으로 해소(ADR-0016).

### ADR-0012 — 외부 제휴 링크화 = 숙소 공급의 단일 모델

- **맥락**: 숙소 공급·외부 위임 전반의 상위 원칙 확립.
- **결정**: **외부 제휴 링크화를 숙소 공급의 단일 모델**로 채택한다(ADR-0003의 상위 원칙). TripPilot은 객실 재고·예약 상태·결제를 보유하지 않고, 탐색·저장 후 외부 OTA로의 전환만 담당한다. 외부 길안내(지도 앱)·외부 예약 등 **외부 위임 전반의 상위 원칙**이다.
- **근거**: 실거래·재고·정산·길안내를 일관되게 외부화하여 앱은 오케스트레이션에 집중.
- **영향·연관**: ADR-0003·0005·0009(길안내 위임)·0015(예약 위임)와 정합.

### ADR-0013 — 여행 기록 = plan / actual / change log 3종 구분 저장

- **맥락**: 계획과 실제, 변경 이력을 어떻게 보존할지.
- **결정**: 여행 기록은 **계획(plan)·실제(actual)·변경 이력(change log) 3종을 구분 저장**한다. 등록 시점의 계획 일정, 여행 중 실제 실행 내역, Plan-B 재계획을 포함한 변경 이력을 각각 보존해 회고·재방문 시 대조할 수 있게 한다.
- **근거**: 회고·대조·감사가능성 확보.
- **영향·연관**: D14(불변 plan 스냅샷 + 가변 current)로 구체화. changelog는 Plan-B·공동편집·어시스턴트 공용 스키마(G57/G132). M12·M18이 actual 기록.

### ADR-0014 — 여행자 커뮤니티 = 콘텐츠 공유·발견 + 좋아요·댓글

- **맥락**: 기존 'Out of Scope: 커뮤니티 전체 제외'를 정밀화한다.
- **결정**: 커뮤니티는 **콘텐츠 공유·발견과 그에 대한 좋아요·댓글(게시물형 소셜 반응)까지** 다룬다.
  - 다른 여행자의 공개 일정을 **읽기전용 둘러보기**(탐색 랜딩 '여행자 일정' 진입 — 전용 탭 미신설).
  - 사용자가 명시적으로 공개(**기본 비공개·옵트인**)한 일정을 **'내 여행으로 가져오기'**(원본과 분리된 plan 스냅샷 복제·원작자 출처 전파, ADR-0013 계승).
  - 작성자 **공개 프로필** 조회, 공개 일정에 대한 **좋아요(토글·집계)·댓글**(평면·작성/본인삭제/신고·숨김·운영 검토 인프라 공유·작성자 알림 트리거).
  - **소셜 그래프(팔로우·구독·스크랩 컬렉션·평판 점수)는 제외**. 실시간 공동 편집은 별도(ADR-0016).
  - 공개 콘텐츠는 **정확 숙소 위치를 권역으로·절대 날짜를 상대 표현으로 마스킹**하고 EXIF·제3자 PII를 통제하며, 신고·숨김·운영 검토 경로를 두되 **침묵 삭제하지 않는다**(ADR-0011 계승).
  - 외부 SNS 정적 카드 공유(기록·회고)와 앱 내부 동선 공개는 병렬 채널.
- **근거**: 발견·공유의 제품 가치는 취하되, 소셜 그래프 운영 부담과 프라이버시 위험은 통제.
- **영향·연관**: M15가 담당(후속 단계). 공개 스냅샷 모델 D16, 상대 시기 스키마 G84, EXIF 정책 G185, 공유 뷰 범위 C1은 데이터 모델에 선반영.

### ADR-0015 — AI 어시스턴트 = 대화형 오케스트레이션 (실현가능성은 솔버 위임)

- **맥락**: 앱 전반 AI 어시스턴트가 일정의 시각·실현가능성을 스스로 확정하면 신뢰성이 무너진다.
- **결정**: 앱 전반 AI 어시스턴트(M16)는 자연어 요청을 해석해 기존 모듈(숙소 탐색·AI 일정 생성·Plan-B 재계획)과 솔버(M8·M10)를 **호출·조립·설명하는 대화 인터페이스**다.
  - 장소 영업시간·이동시간·배치 시각 같은 **실현가능성 판단과 최종 시각·순서 확정을 단독으로 내리지 않는다.** 모든 실현가능성 검증과 사용자에게 보이는 시각·이동시간은 **솔버가 소유·검증한 값**이어야 하며(ADR-0008 주 계승·ADR-0009 보조 계승), 어시스턴트는 검증된 결과만 전달한다.
  - 어시스턴트 제안 변경은 사용자의 **명시적 적용 시 솔버 재검증을 거쳐서만** 반영되며, 어시스턴트가 자동 확정·저장하지 않는다.
  - 예약·결제·외부 실거래는 외부 OTA/제휴처로 위임(ADR-0003·0005·0012).
  - LLM 해석·솔버·외부 데이터 실패 시 추측으로 채우지 않고 수동/비대화형 폴백으로 연결(침묵 실패 금지 — ADR-0011).
  - **가드레일**: 역할 변경·시스템 지시 유출·유해/차별 요청 거절, LLM 호출 빈도 제한, 현재 사용자 컨텍스트 외 데이터(타 사용자 비공개 데이터·내부 운영 지표) 비노출.
  - 이 비노출은 LLM 거절이 아니라 **컨텍스트 주입 단계의 권한 경계(서버측 필터)로 강제**한다 — 타 사용자 비공개 필드·내부 지표를 애초에 LLM 입력에 포함하지 않아 프롬프트 주입으로 우회할 수 없게 한다.
- **근거**: LLM은 해석·설명에만 쓰고 사실·실현가능성은 결정론 솔버가 소유해야 신뢰성·안전성이 확보된다.
- **영향·연관**: M16(후속). 구현 방식은 **D31(서버 재조회 — 클라이언트는 ID만 전달, 서버가 요청자 권한으로 재조회해 주입)**. 어시스턴트 발 변경은 changelog로 추적(ADR-0013).

### ADR-0016 — 동행 공동 편집 범위 포함 (MVP = 항목 단위 잠금)

- **맥락**: 기존 'Out of Scope: 실시간 공동 편집 제외'를 갱신한다.
- **결정**: **여행자 간 일정 실시간 공동 편집(동행 공동 편집)을 범위에 포함**한다.
  - 여행 소유자가 동행자를 링크/계정으로 초대해 하나의 일정을 함께 편집하며, 권한은 **소유자/편집자/뷰어**로 구분한다.
  - 모든 편집은 **솔버 재검증**(ADR-0008·0009)과 **변경 이력**(ADR-0013)을 거치고, 동시 편집 충돌·오프라인은 침묵 실패 없이 사용자 선택으로 해소(ADR-0011).
  - 커뮤니티의 *공개* 일정 복제('내 여행으로 가져오기')와 구별되는, **사용자 본인 여행에 대한 비공개 협업**이다.
  - **MVP 범위(모바일 실현 가능): 항목 단위 잠금(soft lock) 기반 계획 협업으로 한정** — 한 참여자가 한 항목을 편집하는 동안 다른 참여자는 그 항목을 동시에 못 바꾸게 해 충돌을 *예방*한다.
  - 문자 단위 실시간 동시 편집(CRDT 자동 병합)·여행 중 멀티유저 현장 실행은 **후속**이며, 본 기능은 핵심 여정과 **분리된 출시 게이트**로 다뤄 리스크를 격리한다.
- **근거**: 동행 협업 수요는 크지만 CRDT·현장 멀티유저는 모바일 리스크가 커서 MVP는 잠금 기반으로 축소.
- **영향·연관**: M17(후속). 동기화 아키텍처 D30(서버 권위 + 낙관적 잠금, WebSocket, 잠금 TTL 60~120초+하트비트+소유자 강제 해제, 최대 10명, 지연 목표 3초)은 1차 설계에 여지로 반영.

### ADR-0017 — 위치기반서비스 규제 준수를 출시 선결로 (국내 출시)

- **맥락**: 국내 출시 시 위치정보 관련 법적 의무.
- **결정**: 위치 기반 서비스 규제 준수를 **출시 선결**로 둔다.
  - TripPilot은 이용자 본인의 GPS(개인위치정보)로 근처 추천·여행 중 안내·거점 추천을 제공하므로 위치정보법상 **위치기반서비스사업 신고(제9조)** 의무가 있으며(사업자등록 전제), '내 위치만 사용' 또는 '외부 API 위임'으로 **면제되지 않는다**. 위치정보사업 진입규제(제5조 허가·등록)는 자체 측위 인프라 미보유로 불요(명칭은 현행 법령 확인).
  - 개인위치정보 이용약관·동의·철회/일시중지·목적 달성 후 파기·로그 보존(ADR-0010 계승)을 구현한다.
  - 지도는 국내 API(카카오·네이버·TMap) 우선 채택으로 **국외반출 규제를 회피**하되, 영구 캐싱 금지·실시간 호출·출처 표기 약관을 데이터 모델에 반영한다.
- **근거**: 규제 미준수 시 출시 자체가 불가. 위치 서비스가 제품 핵심이므로 선결.
- **영향·연관**: N2(법정 의무 구현), P1(신고+법무 자문), P2(지도 약관·계약), D13(스냅샷 저장 약관 적합성). 출시 전 위치정보 전문 법무 자문으로 최종 확인.

### 2.1 ADR 비고 (내부 지표·제거 항목)

- 외부 제휴 링크의 클릭/전환·수수료는 **TripPilot 내부 운영 지표로만 수집**하며 사용자에게 노출하지 않는다(ADR-0015 비노출 대상과 정합).
- 호스트 사이드(입점·숙소 등록·AI 리스팅 빌더·호스트 예약관리·호스트 재무 대시보드), 내부 객실 재고·예약 상태 관리, 직판 매출/수수료 마진 추적, 호스트용 AI 경영 에이전트 관련 결정은 **B2C 피벗으로 모두 제거**되었다(ADR-0001).

## 3. 모듈 분해 결정 (M1 ~ M18)

### 3.1 모듈 재편 이력 (분해 결정)

원본 19개 모듈에서 **5개 제거**(호스트 사이드 3개 = Host Onboarding / AI Listing Builder / Host Reservation Management + 외부 OTA 모델로 불필요해진 2개 = Room Inventory / Payment) → **14개**로 축소. 이후 **3개 추가**(여행자 커뮤니티 도입으로 Community, 앱 전반 AI 어시스턴트 도입으로 Conversational Assistant, 동행 공동 편집 도입으로 Collaborative Editing) → **17개**. 요구사항 단계에서 **M18 Trip Execution 신설**(Δ7/N7) → 현재 **18개**.

핵심 일정 모듈(Itinerary Generation·Recalculation·Plan-B Detection)은 (입력)→(출력) 계약 형식으로 기술한다. 모듈 경계·의존의 상세는 [architecture.md](./architecture.md)·[domain.md](./domain.md)가 정본이며, 여기서는 각 모듈의 존재 결정과 책임 경계만 등록한다.

## 9. 확장 규칙·기준선 결정 (Security / Resiliency / PBT)

D05~D07의 확정에 따라 세 개 기준선이 전 단계의 **차단 제약**으로 적용된다. 세부 항목·구현은 [nfr.md](./nfr.md)가 정본이며, 여기서는 결정과 근거만 등록한다.

| 확장 | 결정 | 근거 |
|---|---|---|
| Security Baseline (SECURITY-01~15) | **전체 강제(blocking)** | CQ1=A |
| Resiliency Baseline (RESILIENCY-01~15) | **적용** — 설계 시점 방향성 모범 사례 | Q6=A |
| Property-Based Testing (PBT-01~10) | **전체 강제(blocking)** | Q7=A |

**RTO/RPO·DR 전략(CQ4=A)**: RTO/RPO 목표 **시간 단위**, DR 전략 **Backup & Restore**. 자동 백업으로 데이터 보호, 재해 시 IaC 재배포 + 백업 복원으로 복구. 리전 토폴로지는 **단일 리전 + 다중 가용영역(Multi-AZ)** (RESILIENCY-08). 이후 모든 단계는 적용 확장 규칙의 검증 기준을 차단 제약으로 평가한다. (성능 목표·보안 세부·워크로드 중요도 분류·테스트 계층 분리는 [nfr.md](./nfr.md) 참조.)
