# TripPilot Jira 백로그: E9~E12

> 이 문서의 E9~E12, US-*, ST-* 식별자는 Jira가 발급하는 issue key가 아니라 기획 문서와 Jira를 연결하기 위한 **외부 추적 ID**다. Jira 생성 후 실제 key를 별도 매핑하며, 외부 추적 ID는 변경하지 않는다.
>
> Story 제목·요지·수용 기준의 정본은 [user-stories.md](../user-stories.md)다. 아래 수용 기준은 정본을 누락 없이 옮겼다. Story Point는 초기 상대 추정치이며 refinement 후 확정한다.

Subtask의 나열 순서는 실행 순서가 아니다. 각 Story는 테스트 Subtask에서 실패 케이스를 먼저 작성한 뒤 최소 구현과 리팩터링을 진행하며, 공통 완료 기준(수용 기준 전수 통과, 관련 하드 제약 100% 통과, 단위·통합·E2E, PBT, 전체 커버리지 80% 이상, 보안·복원력 검증)을 충족해야 Done으로 전환한다.

### US-E09-01 — 숙소 등록·저장 완료 알림

- **유형**: Story
- **상위 Epic**: E9
- **Priority**: P1
- **목표 Sprint**: S8
- **초기 SP**: 5
- **의존**: US-E3-04, US-E3-06, U3 StayRegistered, U4 StayLinkedToTrip, E9 알림 공통 파이프라인
- **Source**: [user-stories.md](../user-stories.md) US-E09-01; [u8-notification.md](../units/u8-notification.md) CP5·NoticeType

**요지**

여행자로서 외부 OTA 예약 등록 또는 숙소 저장을 완료했을 때 성공 알림을 받고 싶다 — 등록이 성공했고 일정 생성 출발점이 설정됐음을 바로 확신하려고.

**원문 수용 기준**:

- [ ] 외부 예약 등록 저장 직후 숙소명·위치·체크인/아웃 날짜와 "이 숙소가 일정 생성의 출발점으로 설정되었습니다" 문구를 포함한 알림을 서버가 발송한다.
- [ ] 단순 '저장'(찜)과 '등록'(일정 기준점)을 구분하고 저장 완료 알림에는 "일정에 반영하려면 숙소를 등록하세요" 행동 유도를 포함한다.
- [ ] 체크인/아웃 날짜가 없는 등록에는 등록 완료 알림 대신 "날짜를 입력해야 일정 생성이 가능합니다" 알림을 발송한다.
- [ ] 푸시는 단일 채널로 발송하고 동일 알림을 인앱 알림함에도 적재한다.

**Subtasks**

- [ ] **ST-E09-01-01 [BE] 이벤트 소비·분기 구현** — 저장/등록/날짜 누락 이벤트를 서로 다른 NoticeType·문구로 변환하고 동일 eventId 재수신 시 알림이 중복 생성되지 않는다.
- [ ] **ST-E09-01-02 [BE] 푸시·인앱 이중 적재 구현** — 단일 발송 요청으로 FCM 전송과 Notice 영속화가 실행되며 한 채널 실패가 다른 채널 기록을 유실시키지 않는다.
- [ ] **ST-E09-01-03 [Mobile] 알림 표시·딥링크 구현** — 알림 카드가 숙소명·위치·날짜·행동 유도를 표시하고 등록 또는 일정 생성 화면으로 이동한다.
- [ ] **ST-E09-01-04 [QA] 계약·분기 테스트** — 저장/등록/날짜 누락/중복 이벤트 4개 시나리오와 FCM 실패 폴백을 자동 테스트로 통과한다.

### US-E09-02 — 여행 단계별 리마인드 알림

- **유형**: Story
- **상위 Epic**: E9
- **Priority**: P0
- **목표 Sprint**: S8
- **초기 SP**: 8
- **의존**: US-E5-12, US-E09-05, U5 ItineraryConfirmed/Changed, U3/U4 거점 상태, 서버 스케줄러
- **Source**: [user-stories.md](../user-stories.md) US-E09-02; D25·D32; [u8-notification.md](../units/u8-notification.md) NotificationSchedule

**요지**

여행자로서 여행 시작 전·당일·개별 일정 전 단계별 리마인드 알림을 받고 싶다 — 일정을 놓치지 않고 여행을 실행하려고.

**원문 수용 기준**:

- [ ] 여행 시작 D-1 "여행 시작 전" 알림, 각 여행일 첫 일정 전 "당일 일정" 요약 알림(기본 오전 8시, 설정 가능), 각 개별 일정 시작 전 "다음 일정" 알림(기본 30분 전, 0/15/30/60분 선택)을 발송한다.
- [ ] 알림에 장소명·시작 시각·직전 지점(등록 숙소 또는 직전 일정)으로부터 예상 이동 거리를 포함하고 소요시간은 표시하지 않는다.
- [ ] 모든 리마인드는 서버 스케줄링으로 발송하며 일정이 변경되면 발송 시각을 재계산한다.
- [ ] 등록 숙소가 없어 일정이 없는 경우 "숙소를 등록하면 일정 알림을 받을 수 있습니다" 안내만 1회 발송한다.
- [ ] 출발점 숙소 해제/변경 후 사용자가 일정 재생성을 거부하면 기존 일정은 유지하되 리마인드 발송을 중단하고 일정 화면에 재생성 유도 배지를 표시한다.
- [ ] 방해금지 시간(기본 22~08시)에는 리마인드를 억제하고 억제된 알림은 인앱 알림함에 적재한다.

**Subtasks**

- [ ] **ST-E09-02-01 [BE] 리마인드 스케줄 생성기 구현** — D-1·당일·슬롯별 스케줄과 0/15/30/60분 옵션을 생성하며 동일 일정 재처리는 멱등이다.
- [ ] **ST-E09-02-02 [BE] 일정 변경·거점 해제 재계산 구현** — ItineraryChanged 수신 시 기존 스케줄을 전면 무효화 후 재생성하고 재생성 거부 상태에서는 발송 0건을 보장한다.
- [ ] **ST-E09-02-03 [Mobile] 리마인드 설정·배지 구현** — 당일 시각과 슬롯 사전 알림 값을 변경할 수 있고 거점 변경 후 재생성 유도 배지가 일정 화면에 표시된다.
- [ ] **ST-E09-02-04 [QA] 시간 경계·표시 테스트** — D-1/08시/슬롯 경계, 방해금지, 숙소 없음, 거리 표시·소요시간 비노출을 fake clock 기반 자동 테스트로 검증한다.
- [ ] **ST-E09-02-05 [Observability] 스케줄 관측 추가** — 생성·취소·보류·중복 억제 수와 지연 발송을 메트릭으로 기록하고 실패 알람을 재현한다.

### US-E09-03 — Plan-B 재계획 알림

- **유형**: Story
- **상위 Epic**: E9
- **Priority**: P0
- **목표 Sprint**: S8
- **초기 SP**: 8

**요지**

여행자로서 여행 중 재계획이 필요한 상황(날씨·휴무·이동 지연)을 알림으로 받고 싶다 — 변수에 즉시 대응해 일정을 살리려고.

**원문 수용 기준**:

- [ ] 알림 본문에 트리거 유형(예: "비 예보", "장소 휴무", "이동 지연 25분")·영향받는 일정을 명시하고 "대안 일정 보기" 액션으로 바로 진입할 수 있다.

- [ ] 외부 날씨/지도/휴무 API 호출 실패 시 잘못된 정보로 알림을 보내지 않고 침묵하며 다음 정상 응답 시점에 재평가한다.
- [ ] 위치정보 동의 철회 상태에서는 이동 지연 기반 알림을 발송하지 않고 날씨·휴무 등 위치 비의존 트리거만 평가한다.
- [ ] 여행 '진행 중'의 Plan-B 알림만 방해금지 시간(기본 22~08시)의 예외로 발송한다.

**Subtasks**

- [ ] **ST-E09-03-01 [BE] TriggerFired 발송 판정 구현** — 유형·심각도·진행 상태·위치 동의를 입력으로 발송/보류/폐기 결정을 순수 함수로 산출한다.
- [ ] **ST-E09-03-02 [BE] 중복·빈도 억제 구현** — 일정별 10분 dedupe와 E7 전역 상한을 함께 적용하고 동일 키 경쟁 상황에서도 1건만 발송한다.
- [ ] **ST-E09-03-03 [Mobile] Plan-B 알림 랜딩 구현** — 트리거 유형과 영향 일정이 보이며 액션 탭 시 해당 재계획 세션으로 정확히 이동한다.
- [ ] **ST-E09-03-04 [QA/PBT] 판정 행렬 검증** — 위치 동의·방해금지·진행 상태·심각도·외부 데이터 유효성의 임의 조합에서 금지 알림 0건 속성을 검증한다.

- **Source**: [user-stories.md](../user-stories.md) US-E10-03; ADR-0015·D25·D31

**요지**

여행자로서 "지금까지 한 거 검토해줘"로 현재 여행·일정 상태와 문제점을 진단받고 싶다 — 빠뜨린 것과 다음 할 일을 한눈에 파악하려고.

**원문 수용 기준**:

- [ ] 현재 여행 상태(등록 숙소·기간·고정 블록·일정 유무·미해결 충돌)를 요약하고 다음 할 일을 짚어 주되 일정상 문제는 솔버가 이미 검증·표기한 위반 정보만 가져와 설명한다(미검증 수치 생성 금지).
- [ ] 각 문제에 해결 동선(숙소 등록·다른 날로 이동·AI 재추천)을 실제 화면/모듈 진입점으로 연결한다.
- [ ] 제시하는 모든 수치는 솔버 검증값이며 이동 소요시간은 표시하지 않고 거리만 표시한다.
- [ ] 검토 대상이 없거나 검증값을 가져올 수 없으면 추측으로 단정하지 않고 가용 범위에서만 요약하며 진입점/안내를 제공한다.

**Subtasks**

- [ ] **ST-E10-03-02 [BE] 검토 응답 제약 구현** — 검증되지 않은 수치와 이동 소요시간 필드가 응답 스키마에 들어갈 수 없고 각 문제에 유효한 진입점이 연결된다.
- [ ] **ST-E10-03-03 [Mobile] 검토 결과·해결 액션 구현** — 상태 요약과 문제별 해결 버튼을 표시하고 대상 화면의 정확한 리소스로 이동한다.
- [ ] **ST-E10-03-04 [QA] 불완전 데이터 테스트** — 상태 없음·일부 조회 실패·솔버 값 없음에서 추측 문구 0건과 가용 범위 안내를 검증한다.

### US-E10-04 — 액션 위임(어시스턴트 단독 확정 금지)

- **유형**: Story
- **상위 Epic**: E10
- **Priority**: P2
- **목표 Sprint**: Unscheduled
- **초기 SP**: 8
- **의존**: US-E10-01, US-E5-07, US-E7-01, C2 재검증, D20 상태 머신
- **Source**: [user-stories.md](../user-stories.md) US-E10-04; ADR-0015·D14·D20

**요지**

여행자로서 어시스턴트 제안을 수락하면 실제 변경은 기존 모듈/솔버가 검증·실행하길 원한다 — 대화로 바꾼 일정도 항상 실행 가능함이 보장되도록.

**원문 수용 기준**:

- [ ] 제안 적용 선택 시 자유 편집·재검증(E5 스토리 7) 또는 Plan-B 재계산 흐름으로 넘겨 솔버 재검증 후에만 일정에 반영하고 위반 시 조용히 적용하지 않고 사유·대안을 제시한다.
- [ ] 명시적 '적용' 확인 없이 자동 확정·저장하지 않으며 변경 전후를 비교해 보여준다. 최종 시각·순서는 항상 솔버 소유 값이다.
- [ ] 확정 일정 변경 요청은 여행 전이면 확정 해제→재확정 흐름을 안내하고 여행 중이면 Plan-B 재계획으로 라우팅한다.
- [ ] 솔버 재검증 불가(외부 API 실패) 시 자동 적용하지 않고 수동 편집 모드로 폴백하며 미검증 상태를 표기한다.

**Subtasks**

- [ ] **ST-E10-04-01 [BE] 위임 명령 어댑터 구현** — 어시스턴트 제안을 기존 편집/Plan-B 명령으로만 변환하고 직접 일정 쓰기 경로를 구조적으로 두지 않는다.
- [ ] **ST-E10-04-02 [BE] 상태별 라우팅·검증 구현** — 여행 전/여행 중 상태에 맞는 흐름을 선택하고 C2 검증 성공 후에만 변경 커밋이 가능하다.
- [ ] **ST-E10-04-03 [Mobile] 적용 확인·전후 비교 구현** — 사용자 명시 확인, 전후 차이, 위반 사유·대안, 미검증 폴백 상태를 표시한다.
- [ ] **ST-E10-04-04 [PBT/Security] 단독 확정 금지 검증** — 임의 대화·상태·실패 입력에서 사용자 확인 없는 plan/current 변경과 확정 전이가 0건임을 검증한다.

### US-E10-05 — 대화 이력·세션 유지

- **유형**: Story
- **상위 Epic**: E10
- **Priority**: P2
- **목표 Sprint**: Unscheduled
- **초기 SP**: 5
- **의존**: US-E10-01, U4 여행 생명주기, EN-10(G132) changelog
- **Source**: [user-stories.md](../user-stories.md) US-E10-05; G135·D22·ADR-0013

**요지**

여행자로서 대화 맥락이 작업 흐름 안에서 이어지고 닫았다 열어도 직전 대화를 다시 보고 싶다 — 같은 맥락을 반복 설명하지 않으려고.

**원문 수용 기준**:

- [ ] 대화 이력은 여행 단위 스레드 1개로 유지하고 여행 생명주기에 연동해 보관한다. 이력은 로그인 계정에 귀속된다 — 비로그인 임시 세션 조항은 제외.
- [ ] 다른 여행/작업으로 전환하면 새 컨텍스트 기준으로 시작하되 이전 대화로 돌아갈 수 있다(맥락 혼선 방지).
- [ ] 어시스턴트가 호출한 모듈·반영 변경을 대화 내역에 남기고 실제 변경은 여행 변경 이력과 일관되게 추적한다 — 대화 자체는 확정 데이터가 아니다.
- [ ] 이력 저장 실패 시 침묵 없이 임시 보관 상태를 표시하고 재연결 시 동기화한다.

**Subtasks**

- [ ] **ST-E10-05-01 [BE] 여행 단위 스레드 모델 구현** — 계정+여행당 활성 스레드 1개를 강제하고 여행 전환 시 컨텍스트를 분리한다.
- [ ] **ST-E10-05-02 [BE] 도구 호출·변경 참조 기록 구현** — 메시지가 모듈 호출과 changelog ID를 참조하되 대화 본문을 일정 정본으로 사용하지 않는다.
- [ ] **ST-E10-05-03 [Mobile] 이력 전환·임시 상태 구현** — 여행별 대화를 탐색하고 저장 실패 메시지는 동기화 대기로 표시한 뒤 재연결 시 갱신한다.
- [ ] **ST-E10-05-04 [QA] 계정·여행 격리 테스트** — 다계정·다여행·재연결 시 이력 혼입 0건과 임시 메시지 멱등 동기화를 검증한다.

### US-E10-06 — 가드레일(오·남용 방지)

- **유형**: Story
- **상위 Epic**: E10
- **Priority**: P2
- **목표 Sprint**: Unscheduled
- **초기 SP**: 8
- **의존**: EN-10(D31), C1 가드레일, 사용자별 rate-limit, 보안 로깅
- **Source**: [user-stories.md](../user-stories.md) US-E10-06; SECURITY-05·08·11·D31

**요지**

여행자로서 어시스턴트가 주입·유출·유해 요청에 안전하게 대응하길 원한다 — 내 비공개 데이터가 다른 사용자에게 노출되지 않음을 신뢰하려고.

**원문 수용 기준**:

- [ ] 역할 변경·시스템 지침 유출 요구에 응하지 않고 가능한 도움 범위로 되돌리며 유해·차별 요청은 사유와 함께 거절한다(반복 시 같은 거절 대신 사용 안내로 전환).
- [ ] 타 사용자 비공개 데이터·내부 운영 지표 비노출은 프롬프트 거절이 아니라 서버 재조회 권한 경계로 강제한다 — 클라이언트는 ID만 전달하고 서버가 요청자 권한으로 재조회해 주입하므로 권한 밖 데이터는 LLM 입력에 구조적으로 미포함.
- [ ] LLM 호출에 사용자별 빈도 제한(rate-limit)을 두고 과도한 연속 요청 시 재시도 안내를 표시한다(비용·남용 방지).

**Subtasks**

- [ ] **ST-E10-06-01 [Security/BE] 서버 재조회 권한 경계 구현** — 변조 ID와 교차 계정 ID를 포함한 요청에서도 비가시 데이터가 프롬프트·로그·응답에 들어가지 않는다.
- [ ] **ST-E10-06-02 [BE] 입력·출력 가드레일 구현** — 역할 변경·지침 유출·유해 요청을 분류하고 허용 범위 안내 또는 안전한 거절로 반환한다.
- [ ] **ST-E10-06-03 [BE] 사용자별 rate-limit 구현** — 호출 상한과 retryAfter를 서버에서 강제하고 차단 이벤트·비용을 계측한다.
- [ ] **ST-E10-06-04 [Security QA] 공격 코퍼스 회귀 테스트** — prompt injection·IDOR·운영 지표 요구·반복 유해 요청 코퍼스에서 누출 0건과 기대 분기를 검증한다.
- [ ] **ST-E10-06-05 [Mobile] 거절·재시도 UX 구현** — 안전 거절, 가능한 도움 안내, rate-limit 재시도 시각을 dead-end 없이 표시한다.

### US-E10-07 — 범위 한계 안내·외부 위임

- [ ] **ST-E10-07-02 [BE] 미검증 단정 방지 구현** — 솔버 결과가 없는 실현가능성 질문에는 확정형 응답 대신 검증 액션만 반환한다.
- [ ] **ST-E10-07-03 [Mobile] 외부 위임 액션 구현** — OTA·지도 앱 이동 전 고지와 실패 폴백을 제공하고 반복 범위 밖 요청에는 대체 행동을 표시한다.
- [ ] **ST-E10-07-04 [QA] 범위 코퍼스 테스트** — 지원/미지원/반복 요청별 응답과 목적지, 예약 직접 처리 0건, 미검증 수치 0건을 검증한다.

- [ ] **ST-E10-08-04 [QA/Resilience] 장애 매트릭스 테스트** — 각 의존성 단독·복합 장애에서 핵심 흐름 완결, 추측 값 0건, 솔버 값 우선 표시를 검증한다.

- [ ] 복제는 공개 일정 스냅샷을 사용자 소유 새 여행으로 독립 복제한다(원본과 완전 분리 — 상호 영향·자동 반영 없음). 대상은 공개 계획(plan)·장소 목록만이며 actual·GPS·사진·개인 메모·변경 이력은 제외한다.
- [ ] 복제 플로우에 내 여행 날짜 입력을 필수 스텝으로 포함하고 복제 직후 내 날짜·인원·등록 숙소·취향 기준 재정렬·재검증으로 연결한다(숙소 미등록 시작 허용, 예약은 가져오지 않음).
- [ ] 원작자 출처 표시(닉네임·'○○님의 일정에서 가져옴')와 복제 시점을 부착하고 사본 재공개 시에도 최초 원작자 표기를 보존·전파한다. 원본이 이후 비공개·삭제되어도 사본은 유지하고 출처 표시만 갱신한다.
- [ ] 같은 일정 재복제 시 기존 사본을 안내하고 확인받으며 비로그인 딥링크 진입 시 로그인 게이트 후 원래 동작을 이어간다. 일부 POI 불일치는 '확인 필요' 분리 표기로 복제를 막지 않는다.

**Subtasks**

- [ ] **ST-E11-06-01 [BE] 독립 복제 서비스 구현** — plan·장소만 새 소유자 ID로 복제하고 원본·actual·GPS·미디어·메모·changelog 참조가 사본에 남지 않는다.
- [ ] **ST-E11-06-02 [BE] 출처·중복·POI 상태 구현** — 최초 원작자 계보와 복제 시점을 보존하고 중복 복제·원본 비공개·POI 불일치를 명시 상태로 반환한다.
- [ ] **ST-E11-06-03 [Mobile] 날짜 필수 복제 플로우 구현** — 로그인 복귀, 날짜 입력, 중복 확인, POI 확인 필요 표시 후 U4/U5 재검증으로 연결한다.
- [ ] **ST-E11-06-04 [Security/PBT] 소유권 분리 검증** — 원본/사본 양쪽 임의 변경·삭제에서 상호 데이터 불변과 최초 출처 전파 속성을 검증한다.
- [ ] **ST-E11-06-05 [QA] 복제 E2E 검증** — 숙소 있음/없음, 비로그인 딥링크, 원본 비공개, 일부 POI 불일치 시나리오를 통과한다.

### US-E11-08 — 좋아요

**요지**

- [ ] **ST-E11-08-04 [QA/PBT] 동시성·수명주기 테스트** — 임의 연타·순서에서도 최종 상태 1건, 계정 삭제 시 차감, 비공개 시 비노출 속성을 검증한다.

- [ ] **ST-E12-02-04 [Mobile] 역할별 UI·권한 변경 UX 구현** — 금지 액션 숨김/비활성, 강등 안내, 저장 시도/폐기, 소유자 요청 경로를 제공한다.
- [ ] **ST-E12-02-05 [Security/QA] 권한 전수 테스트** — 역할×명령 매트릭스와 직접 API 호출, 강등 경쟁 조건에서 권한 밖 변경·민감 필드 노출 0건을 검증한다.

### US-E12-03 — 항목 단위 잠금 동시 편집·동기화

- **목표 Sprint**: Unscheduled
- **초기 SP**: 8

**요지**

여행자로서 동행과 같은 일정을 항목 단위로 번갈아 편집하고 변경이 서로 화면에 동기화되길 원한다 — 충돌 걱정 없이 함께 일정을 다듬으려고.

**원문 수용 기준**:

- [ ] 한 참여자가 항목(슬롯·장소·시각) 편집을 시작하면 그 항목은 잠금되어 타인이 동시에 못 바꾸고('○○ 편집 중' 프레즌스) 저장 시 다른 참여자 화면에 반영된다 — 동기화는 서버가 최종 권위를 가지며 수렴 목표는 3초.
- [ ] 잠금은 TTL 60~120초+하트비트로 유지되고 소유자가 강제 해제할 수 있다.
- [ ] 여러 일자에 걸치는 광역 편집(재정렬·재생성 등)은 영향 일자(day) 전체 잠금으로 승격한다.
- [ ] 동기화 지연 시 '동기화 중' 상태를 표시하고 실시간 채널이 끊기면 침묵하지 않고 안내 후 로컬 편집을 허용하며 재연결 시 동기화로 정리한다.

**Subtasks**

- [ ] **ST-E12-03-01 [BE] 잠금 상태 머신 구현** — slot/day 잠금, TTL 60~120초, 하트비트, 소유자 강제 해제에서 동일 자원 이중 획득을 방지한다.
- [ ] **ST-E12-03-02 [Realtime] WebSocket 프레즌스·변경 전파 구현** — 인증 채널에서 잠금 소유자와 저장 버전을 전파하고 정상 조건 p95 3초 이내 수렴을 계측한다.
- [ ] **ST-E12-03-03 [Mobile] 프레즌스·동기화 상태 구현** — 편집 중 사용자, 잠긴 컨트롤, 동기화 중, 채널 단절, 로컬 대기 상태를 표시한다.
- [ ] **ST-E12-03-04 [PBT/Load] 잠금·수렴 검증** — 최대 10명의 임의 획득/만료/하트비트/강제 해제 순서에서 이중 잠금 0건과 최종 버전 수렴을 검증한다.
- [ ] **ST-E12-03-05 [Observability] 실시간 운영 지표 구현** — 연결 수·재연결·전파 지연·잠금 만료·강제 해제·버전 거부를 대시보드와 알람으로 제공한다.

### US-E12-04 — 편집 충돌 해소

- **유형**: Story
- **상위 Epic**: E12
- **Priority**: P2
- **목표 Sprint**: Unscheduled
- **초기 SP**: 8
- **의존**: US-E12-03, US-E12-06, US-E12-07, 항목 version·changelog
- **Source**: [user-stories.md](../user-stories.md) US-E12-04; D30·G93·ADR-0011

**요지**

여행자로서 같은 항목의 편집 충돌이 침묵 덮어쓰기 없이 해소되길 원한다 — 누구의 변경도 모르게 사라지지 않도록.

**원문 수용 기준**:

- [ ] 온라인 동시 충돌은 항목 잠금(US-E12-03)으로 예방하고 잠금으로 못 막는 충돌(주로 오프라인 편집 재연결)은 두 버전을 보여주고 충돌을 감지한 당사자가 선택하며 서버는 직렬화 선착순으로 적용한다. 선택 후에도 직전 상태로 되돌릴 수 있다.
- [ ] 서로 다른 항목의 편집은 충돌 없이 병합된다.
- [ ] 해소 결과는 모든 참여자에게 동일하게 수렴하고 적용/보류된 변경은 변경 이력(US-E12-07)에 남긴다.
- [ ] 해소 중 추가 변경이 들어오면 마지막 확정 상태 기준으로 재해소하고 데이터를 유실하지 않는다.

**Subtasks**

- [ ] **ST-E12-04-01 [BE] 버전 충돌 감지·병합 구현** — 같은 항목은 충돌 객체를 생성하고 다른 항목은 자동 병합하며 서버 직렬화 순서를 기록한다.
- [ ] **ST-E12-04-02 [BE] 해소·재해소·되돌리기 구현** — 사용자 선택을 최신 확정 버전에 조건부 적용하고 추가 변경 시 새 충돌로 재평가하며 양 버전을 보존한다.
- [ ] **ST-E12-04-03 [Mobile] 두 버전 비교·선택 UX 구현** — 변경자·시각·전후 값을 표시하고 적용/보류/되돌리기·재해소 상태를 제공한다.
- [ ] **ST-E12-04-04 [PBT] 무손실 수렴 속성 검증** — 임의 동일/상이 항목 변경 순서에서 모든 입력 버전이 이력에 남고 참여자 최종 상태가 동일함을 검증한다.
- [ ] **ST-E12-04-05 [QA] 재연결 충돌 E2E 검증** — 오프라인 A와 온라인 B의 경쟁, 해소 중 C 변경, 되돌리기를 다기기 시나리오로 통과한다.

### US-E12-08 — 공유 종료·이탈과 소유권

- **유형**: Story
- **상위 Epic**: E12
- **Priority**: P2
- **목표 Sprint**: Unscheduled
- **초기 SP**: 8
- **의존**: US-E12-01~07, US-E09-09, U6 현장 실행, U8 알림 확장
- **Source**: [user-stories.md](../user-stories.md) US-E12-08; D18·D30·G94

**요지**

여행자로서 공동 편집을 종료하거나 스스로 나갈 수 있고 종료 후 소유권이 명확하길 원한다 — 협업이 끝나도 내 여행의 귀속이 흔들리지 않도록.

**원문 수용 기준**:

- [ ] 소유자 '공유 종료' 시 여행은 소유자 단독 소유로 돌아가고 동행자는 편집·열람 불가가 되며 동행자는 스스로 '나가기' 할 수 있다. 원본은 항상 소유자 귀속이고 동행자 사본을 자동 생성하지 않는다(커뮤니티 복제와 구분).
- [ ] 푸시 알림은 초대·권한 제거·공유 종료에만 발송하고 여행 리마인드·Plan-B 알림은 소유자에게만 발송한다.
- [ ] 확정 후에도 편집자는 이력·재검증을 경유해 편집할 수 있으나 여행 중 현장 실행 액션(도착 체크·기록·Plan-B 확정)은 소유자 기준으로만 동작하고 동행자는 공유 뷰로 따라간다. 소유자 주도 Plan-B 확정이 타 참여자 미저장 편집과 충돌하면 US-E12-04·05를 적용한다.
- [ ] 소유자가 계정을 삭제하면 해당 여행과 공동 편집 권한·이력도 삭제·정리 대상에 포함하고(계정 삭제 연쇄와 일관) 동행자에게 공유 종료를 안내한다.

**Subtasks**

- [ ] **ST-E12-08-01 [BE] 공유 종료·이탈 상태 전이 구현** — 종료/나가기에서 원본 소유자는 불변이며 참여 권한·잠금·채널 접근을 즉시 철회한다.
- [ ] **ST-E12-08-02 [BE] 알림·현장 액션 권한 구현** — 세 공동 편집 알림만 참여자에게 보내고 리마인드·Plan-B·도착·기록·확정은 소유자만 실행한다.
- [ ] **ST-E12-08-03 [BE] 계정 삭제 연쇄 구현** — 소유자 삭제 시 여행·참여·잠금·이력을 정리하고 동행자에게 종료 이벤트를 발행한다.
- [ ] **ST-E12-08-04 [Mobile] 종료·나가기·공유 뷰 UX 구현** — 영향 고지·확인, 종료 후 접근 차단, 동행자 나가기, 여행 중 소유자 액션의 읽기전용 미러를 제공한다.
- [ ] **ST-E12-08-05 [QA] 소유권·종료 E2E 검증** — 공유 종료, 동행 이탈, 확정 후 편집, Plan-B 충돌, 소유자 삭제에서 자동 사본 0건과 권한 정리를 검증한다.
