# TripPilot NFR 기준 (보안·복원력·성능·테스트)

> 출처: aidlc/aidlc-docs/inception/requirements/requirements.md(§6) · aidlc/aidlc-docs/construction/u1-foundation/nfr-requirements/nfr-requirements.md · aidlc/aidlc-docs/construction/u1-foundation/nfr-design/nfr-design-patterns.md · aidlc/docs/PRD/14-테스트.md · aidlc-docs에서 2026-07-05 추출 · 이후 본 문서가 정본이다.

## 이 문서의 위치 — 크로스커팅 NFR 기준선

이 문서는 TripPilot 전 유닛에 공통으로 적용되는 **비기능 요구사항(NFR) 기준선**을 정본으로 정리한다. 성능·복원력·보안·관측성·운영 프로세스·테스트의 전역 기준을 담는다.

- **정본 관계**: 시스템 전역 NFR의 최상위 정본은 요구사항 정의서 §6이며, 본 문서가 그 실질 내용을 재구성한 정본이다. 유닛별로 이 기준선이 구체화되는 세부(예: 인증 API별 지연 목표의 U1 산정, 유닛별 속성 식별)는 [유닛 상세 문서](./units.md)의 각 `units/uN-*.md`에 있으며, 필요 지점에서 링크로 안내한다.
- **전역/유닛 표기**: 아래 설계 패턴 중 `전역`으로 표기된 것은 **U1(기반·계정·온보딩)이 정본을 수립하고 U2~U8 전 유닛이 재사용(재정의 금지)** 한다는 의미다. `U1`로 표기된 것은 U1 코드가 구현을 소유하되 발급·검증 계약은 전 유닛이 소비한다. 이 관계 때문에 U1의 NFR 산출물이 사실상 시스템 NFR 기준선의 본체가 된다.
- **확장 규칙 강제**: `SECURITY-01~15`(보안 기준선), `RESILIENCY-01~15`(복원력 기준선), `PBT-01~10`(속성 기반 테스트)은 AI-DLC 확장 규칙으로 **전량 강제(하드 제약)** 된다. 비준수는 스테이지 완료를 차단한다. 각 규칙이 현 단계에서 적용 대상이 아니면 근거와 함께 N/A로 판정한다(N/A는 차단 소견이 아니다).
- **미결 처리 원칙**: 클라우드 벤더·CI/CD·배포·인시던트 대응 도구 등 벤더 종속 결정은 인프라 설계 단계로 이연한다. 이 문서는 **벤더 중립 요건**까지만 확정하고, 이연 항목은 §11과 [인프라 설계](./infrastructure.md)로 넘긴다. 이연은 계획된 지연이며 비준수가 아니다.

관련 정본: 아키텍처 경계는 [아키텍처](./architecture.md), 도메인 상태 머신은 [도메인 모델](./domain.md), 각 흐름의 실패 분기는 [주요 흐름](./flows.md), 결정·ADR은 [핵심 결정](./decisions.md), 용어는 [용어집](./glossary.md)을 참조한다.

## 1. 성능 (D38)

### 1.1 시스템 성능 목표 (§6.1)

D38이 확정한 기능별 응답 시간 목표다. 초과 시 폴백 고지가 원칙(침묵 실패 금지 — ADR-0011)이다.

| 기능 | 목표 | 초과 시 |
|---|---|---|
| AI 일정 생성 | 첫 1일 5초 내 노출 · 전체 20초 한계 | 결정론적 솔버 폴백 고지 |
| Plan-B 대안 제시 | 10초 | 수동 수정 폴백 제안 |
| AI 어시스턴트(후속) | 첫 응답 3초 · 전체 15초 타임아웃 | 타임아웃 처리 |
| 일반 화면 전환 | 300ms | — |
| 공동 편집 동기화(후속, D30) | 3초 이내 수렴 | — |

측정 방법: API별 지연 p50/p95를 APM 대시보드로 상시 관측하며, p95가 목표를 5분 지속 초과하면 알람(§4.3 골든 시그널).

### 1.2 "일반 화면 전환 300ms"의 해석 — 전환 차단/비차단 분리

300ms는 온보딩처럼 화면이 연속되는 흐름에서 **API 지연이 화면 전환 예산을 잠식하지 않는 것**을 요구한다. API를 두 부류로 나눈다.

- **전환 차단(blocking) API** — 응답이 와야 다음 화면으로 넘어가는 호출(닉네임 검증, 취향 저장, 동의 기록). 서버 p95가 300ms 안에 들어와야 체감 전환이 300ms대에 머문다.
- **전환 비차단 API** — 낙관적 UI·백그라운드 처리로 전환과 분리 가능한 호출(인증 메일 발송, 백그라운드 세션 재검증).

낙관적 UI 병행이 원칙이며, 저장 실패 시 해당 단계로 복귀 가능한 오류 처리(침묵 실패 금지)를 둔다.

### 1.3 인증 계열 API 응답 목표 (U1 기준선 — 서버 처리 시간, 네트워크 RTT 별도)

D38이 확정한 것은 "일반 화면 전환 300ms"이며, 아래 API별 세부는 그 지원을 위한 U1 제안값이다. 인증 계열은 시스템 진입 경로이므로 전역 체감 성능의 바닥을 결정한다. 유닛별 전체 산정은 [units/u1-foundation.md](./units/u1-foundation.md).

| API | p50 | p95 | 상한(타임아웃) | 비고 |
|---|---|---|---|---|
| 부트스트랩 확인(세션 유효성·재동의·최소 버전) | 100ms | 500ms | 클라이언트 3초(G5) | 단일 질의·무상태 검증 |
| 토큰 갱신(리프레시 회전) | 100ms | 300ms | 3초 | UPDATE 1건+발급 — 전환 차단 최소화(D36) |
| 이메일 로그인 | 400ms | 1,000ms | 5초 | 적응형 해시 검증 비용(의도적 ~200ms)을 예산에 포함 — 300ms의 **의도적 예외** |
| 이메일 가입(해시 생성+유출 목록 검증) | 500ms | 1,500ms | 5초 | 유출 온라인 조회 1초 타임박스, 메일 발송 비동기 분리 |
| 소셜 로그인 콜백(IdP 토큰 교환·검증) | 800ms | 3,000ms | 5초(IdP 어댑터 3초) | 외부 IdP 의존 — 타임아웃·서킷 필수(RESILIENCY-10) |
| 취향 저장(축별 부분 저장) | 50ms | 200ms | 3초 | 온보딩 7단계 연속 전환 핵심 — 낙관적 UI 병행 |
| 닉네임 검증(금칙어 C3+중복)·자동 생성 | 100ms | 300ms | 3초 | 금칙어 사전 인메모리 매칭, 중복은 유니크 인덱스 |
| 동의 기록(약관 3종+선택 동의, append-only) | 100ms | 300ms | 3초 | 증적 INSERT — 트랜잭션 내 완결(N2·N3) |
| 인증 메일 발송 요청 | 100ms | 300ms | 3초 | 접수 확인만 동기, 발송은 비동기 잡. 도달 SLO 5분 내 95%(ADR-0011) |
| 로그아웃(세션 무효화) | 50ms | 200ms | 3초 | revoke UPDATE 1건 |

**스플래시 세션 검증 3초 타임아웃(G5)**: 3초는 서버 성능 목표가 아니라 **클라이언트 최종 대기 한계(fail-policy 경계)** 다. 로컬 토큰 미만료면 홈 진입+백그라운드 재검증, 만료면 로그인 분기. 서버 목표는 p95 500ms로 두어 3초 타임아웃 발동이 비정상 상황(콜드 스타트·네트워크 열화·부분 장애)에 한정되게 한다. 부트스트랩 확인 API는 멱등이어야 하며, 재검증에서 세션 무효 판명 시 401+사유 코드 계약을 가진다.

**클라이언트 성능**: 토큰 읽기(OS 보안 저장소)는 기동 크리티컬 패스이므로 저장 항목을 액세스·리프레시 토큰 2건으로 최소화하고 기동 시 1회만 읽어 메모리 보관한다. 적응형 해시는 **서버 전용** — 클라이언트에서 비밀번호 해시·키 스트레칭을 수행하지 않는다(전송은 TLS 보호).

## 2. 가용성·복원력 (RESILIENCY-*)

### 2.1 워크로드 중요도 분류 (RESILIENCY-01, §6.3)

| 컴포넌트 | 중요도 | 불가용 시 영향 |
|---|---|---|
| Auth·세션 (M1) | Critical | 전 기능 접근 불가 |
| 여행·일정 조회/저장 (M6·M8 저장부, DB) | Critical | 여행 중 사용자 일정 접근 불가 — 현장 피해 최대 |
| Plan-B 감지·재계산 (M9·M10) | High | 여행 중 대응 불가(수동 수정 폴백 존재) |
| AI 일정 생성 LLM 경로 (M8) | Medium | 결정론적 솔버 폴백으로 지속(품질 저하 고지) |
| 숙소 탐색 (M3·M5) | Medium | 계획 단계 기능 — 수동 등록 우회 존재 |
| 기록·회고 (M12·M13, M18) | High | 입력은 로컬 큐로 보존, 조회·생성만 지연 |
| 알림 (M14) | Medium | 리마인드 미발송 — 인앱 재평가로 보완 |
| 어드민 도구 | Low | 운영 지연만 |

의존성: 모든 Critical 경로는 PostgreSQL·Auth에 의존한다. 외부 의존성(카카오·TMap·기상청·LLM·FCM)은 전부 High 이하 경로이며 폴백 필수(ADR-0011). **로그인 필수 원칙(D22)** 때문에 인증 실패는 곧 앱 전체 마비이므로, M1(Auth)의 가용성 설계가 시스템 가용성의 바닥을 결정한다. M2(취향)·C3(금칙어)는 M1과 동일 배포 단위(D04)이므로 별도 분류하지 않되, C3 사전 로딩 실패는 fail-closed로 가입·제목 저장만 보류하고 로그인은 무영향이어야 한다.

### 2.2 RTO/RPO·토폴로지·백업 (§6.2)

| 항목 | 요구 | 추적 |
|---|---|---|
| RTO/RPO | **시간 단위** / DR 전략 = **Backup & Restore**(IaC 재배포+백업 복원) — 계정·동의 증적·법정 로그는 백업 최우선 대상 | CQ4=A, RESILIENCY-02·11 |
| 토폴로지 | **단일 리전 + Multi-AZ** — 컴퓨트 2개 이상 AZ 분산, 관리형 PostgreSQL Multi-AZ, 로드밸런서 다중 AZ 분산 | RESILIENCY-08 |
| 백업 | 관리형 DB 자동 백업(+PITR 활성화 권장 — 목표 상회), 사진 오브젝트 스토리지 버저닝, 백업 암호화. 보존 기간·복원 검증 절차는 인프라 설계 확정 | RESILIENCY-12, G147 |
| 오토스케일·쿼터 | 컴퓨트 오토스케일 min/max 한도, 외부 API rate-limit·쿼터 문서화 및 **80% 알람** | RESILIENCY-09 |

### 2.3 무상태성 요구 — "인증 불가 = 전 기능 마비"의 구조적 차단

Critical 경로가 특정 인스턴스·저장소 상태에 동기 결합되면 국소 열화가 즉시 전면 장애로 전파된다. 이를 구조적으로 차단하는 무상태 요구다.

- **액세스 토큰 검증 무상태(NFR-U1-AV-01)**: 액세스 토큰(1h, D36) 검증은 **서명·만료·클레임(iss·aud) 검증만으로 완결** — 요청마다 DB·캐시 조회를 요구하지 않는다. 트레이드오프인 "무효화 지연 최대 1시간"은 액세스 토큰 수명이 이미 수용한 결정(D36)이다.
- **인스턴스 로컬 세션 상태 금지(NFR-U1-AV-02)**: 리프레시 세션·브루트포스 카운터·rate-limit 상태의 정본을 **인스턴스 메모리에 두지 않는다**. 정본은 PostgreSQL이며, 어느 인스턴스로 라우팅되어도 동일하게 동작(무상태 수평 확장).
- **세션 캐시는 최적화일 뿐(NFR-U1-AV-03)**: 성능용 캐시를 도입해도 **캐시 소실이 인증 실패로 이어져서는 안 된다** — 캐시 미스는 항상 DB 폴백. 초기 규모에서는 별도 캐시 없이 DB 직행으로 충분하다.
- **무중단 배포 내성(NFR-U1-AV-05)**: 토큰 검증 무상태+세션 정본 DB 덕분에 롤링 재시작 중에도 로그인 상태가 유지되어야 한다.

### 2.4 외부 의존 격리 4요소 계약 (PAT-RES-01, RESILIENCY-10) `[전역]`

모든 외부 호출은 Port/Adapter 뒤에서 **타임아웃 + 재시도 + 서킷 브레이커 + 폴백·사용자 고지**의 4요소 계약을 의무화한다(계약 누락은 설계 리뷰 차단 항목). 이 표는 전역 정본이며, 후속 유닛의 외부 의존(지도·기상청·LLM·FCM)도 동일 형식으로 자기 표를 작성한다.

| 의존 (Port) | 타임아웃 | 재시도 | 서킷 브레이커 | 폴백·사용자 고지 |
|---|---|---|---|---|
| 소셜 IdP 4종 (SocialOAuthPort) | 어댑터별 3초 | 없음 — 토큰 교환은 비멱등(코드 1회용), 재시도는 사용자 버튼 | 제공자별 독립 서킷(실패율 임계 초과 시 open→즉시 실패) | 해당 제공자 신규 로그인만 "잠시 후 재시도"(계정 미생성 보장) — 타 제공자·이메일·기존 세션 무영향 |
| 메일 발송 (MailDeliveryPort) | 발송 시도당 5초 | 비동기 잡 재시도(지수 백오프+지터, 최대 N회 후 실패 확정) | 발송 서비스 단위 서킷 | 접수만 동기 확인, 실패 확정 시 미인증 유지+재발송 UI — 가입 트랜잭션은 성공 유지, 실패율 계측 |
| 유출 목록 온라인 조회 (k-익명성) | 1초 타임박스 | 없음(가입 지연 직결) | 서비스 단위 서킷 | **fail-open** — 로컬 목록 통과만으로 진행 + 실패 계측 |
| PostgreSQL | 커넥션 획득·쿼리 타임아웃 명시 | 트랜잭션 단위 — 멱등 연산만 제한 재시도 | 해당 없음(서킷 대신 풀 상한이 벌크헤드) | Multi-AZ 페일오버(관리형) — 열화 시 동작은 §2.5 표 |

공통 규칙: (a) 재시도는 **멱등 연산에만**, 지수 백오프+지터, 총 재시도 시간이 상위 타임아웃 예산을 넘지 않게(재시도 폭풍 금지), (b) 서킷 open/half-open 전이는 계측 이벤트, (c) 폴백은 반드시 **사용자 고지 문구·계측과 짝**(침묵 실패 금지), (d) 외부 호출은 DB 트랜잭션 밖에서 수행(예: 가입 TX 커밋 후 메일 잡 적재). 즉 외부 의존 격리의 3요소는 **타임아웃 + 폴백 + 사용자 고지**이며, 여기에 서킷 브레이커·멱등 재시도가 결합한다.

### 2.5 우아한 성능 저하 — 부분 장애 격리 (PAT-RES-03, NFR-U1-AV-04) `[전역 — 정본 U1]`

인증 **주변부**(특정 IdP, 메일, 금칙어 사전)의 장애가 인증 코어(로그인·세션 유지)로 전파되지 않아야 시스템 가용성 바닥이 유지된다. 장애 모드별 저하 계약이다.

| 장애 | 유지되는 기능(저하 모드) | 상실되는 기능 | 고지 |
|---|---|---|---|
| 소셜 IdP 일부 장애 | 이메일 로그인·타 제공자 로그인·기발급 세션 전부 정상 | 해당 제공자 신규 로그인만 | "잠시 후 다시 시도"+재시도 버튼 |
| 소셜 IdP 4종 전체 장애 | 이메일 로그인·기존 세션 정상 | 소셜 신규 로그인 | 동일 |
| 메일 발송 장애 | 가입 완결(계정 생성 유지)·기존 사용자 전 기능 | 신규 이메일 가입의 인증 완료만 지연 | 미인증 안내+재발송 UI |
| C3 금칙어 사전 로딩 실패 | 로그인·세션·조회 전부 정상 | 닉네임·제목 **저장만** 보류(fail-closed — Pass 위조 금지) | 재시도 안내 |
| PostgreSQL 열화 | 토큰 검증 무상태라 **기존 세션의 조회성 기능은 각 모듈 판단으로 지속 가능** | 로그인·토큰 갱신·가입·저장(Critical 본질) | Multi-AZ 페일오버 대기 |
| 유출 목록 온라인 조회 실패 | 가입 전 과정(로컬 목록 검증으로 진행 — fail-open) | 보강 검증만 | 무고지(내부 계측만) |

설계 원칙: **보안 게이트는 fail-closed**(C3 사전, IdP id_token 검증 실패), **가용성 보조 검증은 fail-open**(유출 온라인 조회) — 방향을 항목별로 명시해 애매함을 남기지 않는다. 저하 모드 진입·이탈은 계측 이벤트로 관측한다.

### 2.6 헬스체크 — shallow/deep 분리 (PAT-RES-02, RESILIENCY-06, NFR-U1-AV-06) `[전역]`

deep 체크를 LB 판정에 쓰면 DB 순단 1회가 전 인스턴스 동시 unhealthy → 대량 축출 → 전면 장애로 **증폭**된다(순단이 정전이 되는 안티패턴). 2계층 분리를 전역 표준으로 둔다.

| 엔드포인트 | 검사 내용 | 소비자 | 판정 용도 |
|---|---|---|---|
| `GET /health` (shallow) | 프로세스 생존·HTTP 서빙 가능(외부 의존 미검사) | **LB 대상 등록·롤링 배포 게이트** | 인스턴스 교체·트래픽 라우팅 |
| `GET /health/deep` (deep) | DB 연결(SELECT 1)·필수 의존 상태 + 선택적 서킷 상태 요약 | **관측·알람 전용** — LB 판정 금지 | 열화 감지·인시던트 트리거 |

규칙: shallow는 무인증 공개(화이트리스트), deep은 내부 접근 제한(정보 노출 최소화). deep의 의존 검사는 자체 짧은 타임아웃(예: 2초)을 가져 헬스체크 스레드 고갈을 방지한다. 신규 인스턴스는 준비(readiness) 완료 전 트래픽을 받지 않는다.

## 3. 보안 기준선 (SECURITY-*)

Security Baseline(SECURITY-01~15)은 **전량 강제(CQ1=A)** 다. 요구사항 §6.4가 정본이고, U1 설계 패턴(PAT-SEC-*)이 구현 정본이다.

### 3.1 인증 필터 체인 — deny-by-default (PAT-SEC-01, SECURITY-08) `[전역]`

허용이 기본이면 누락 1건이 곧 무인증 노출이다. `app` 모듈이 단일 필터 체인을 소유하고 기본 규칙을 `anyRequest().authenticated()`로 고정한다. 공개 엔드포인트는 **명시적 화이트리스트만** 허용하며, 화이트리스트 추가는 설계 리뷰 필수다.

필터 순서(정본):

```text
[요청]
  → 1. 상관 ID 필터 (MDC 주입)
  → 2. 요청 크기·컨텐츠 타입 가드 (64KB 상한)
  → 3. JWT 인증 필터 (Bearer 서명·exp·iss·aud 검증, STATELESS)
  → 4. 브루트포스/레이트리미터 필터 (공개 인증 엔드포인트 한정)
  → 5. 인가 규칙 (anyRequest().authenticated() + 화이트리스트)
  → 6. 보안 헤더 라이터 (HSTS·nosniff·DENY·Referrer-Policy)
  → 7. 전역 예외 변환 (일반화 메시지 401/403)
[핸들러]
```

U1 공개 엔드포인트 전수(이외 추가는 설계 리뷰 필수): 가입(이메일·소셜)·로그인·토큰 갱신, 이메일 인증 링크 확인·재발송, 비밀번호 재설정 요청·확인, 부트스트랩 확인의 **비세션 부분**(최소 지원 버전), shallow 헬스체크.

- **전 퍼사드 인증(SECURITY-08)**: 전 엔드포인트 deny-by-default 인증 필수(D22 로그인 필수와 정합). 서명·만료·발급자(iss)·대상(aud) 검증을 **매 요청 서버측 수행**(무상태). 알고리즘 다운그레이드(`none`·대칭 전환)는 검증기 레벨에서 거부(NFR-U1-SEC-17).
- **소유권 검증 = 객체 수준 인가(NFR-U1-SEC-18)**: 프로필·취향·동의·세션 목록 등 전 리소스 접근은 **토큰 주체(accountId) 기준으로만 조회** — 경로·본문의 타 사용자 ID를 신뢰하지 않는다(IDOR 원천 차단). 공동 편집 권한(소유자/편집자/뷰어)은 서버 측 강제. 계정 상태별 접근 축소(DELETION_PENDING은 철회·열람 외 거부)는 미들웨어 레벨 일괄 강제.
- **CORS(NFR-U1-SEC-19)**: 모바일 앱 전용 API — 브라우저 오리진 불요이므로 CORS **기본 전면 거부**(와일드카드 금지). 후속 어드민 웹(D35) 도입 시 명시 오리진만 추가.
- **회귀 방지(NFR-U1-SEC-20)**: 아키텍처 테스트(Konsist/ArchUnit)로 "인증 가드 없는 공개 핸들러" 존재 자체를 CI에서 검출.

### 3.2 인증·자격 증명 관리 (SECURITY-12, D36) `[U1 — 전역 소비]`

#### 토큰 수명·저장·회전

- 액세스 토큰 **1시간** + 리프레시 **90일 회전**, **다기기 허용**(D36). 토큰은 OS 보안 저장소(iOS Keychain / Android Keystore, expo-secure-store)에만 저장 — AsyncStorage·일반 파일·평문 저장 금지. 로그아웃·401 확정 시 저장 토큰 즉시 삭제, 토큰을 로그·크래시 리포트에 미포함(NFR-U1-SEC-28).
- 리프레시 토큰은 서버에 **tokenHash로만 저장**(원문 저장 금지, NFR-U1-SEC-06). 모바일 API이므로 세션 쿠키 미사용 — 쿠키 속성 요건은 N/A(토큰은 헤더 전송).

#### 리프레시 토큰 회전·재사용 감지 (PAT-SEC-02)

90일 수명 리프레시 토큰은 탈취 가치가 크다. 회전만 있고 재사용 감지가 없으면 탈취를 **탐지하지 못한다**.

- **회전**: 갱신마다 구 토큰 `rotatedAt` 마킹 + 동일 `chainId`로 신규 세션 발급 — 체인당 현행 토큰 정확히 1개.
- **재사용 감지**: `rotatedAt≠NULL`인 토큰 제시 = 탈취 신호 → **해당 체인 전체 즉시 revoke**(현행 포함) + 보안 이벤트 즉시 알림 + 재인증 요구. 타 기기 체인은 불간섭(다기기 허용).
- **경합 방어**: 회전은 조건부 UPDATE(현행 토큰 && rotatedAt IS NULL) 원자 연산. 클라이언트는 axios 단일 비행(single-flight) 갱신 큐로 동시 401 시 회전 1회만 발생시켜 자기 재사용 오탐을 제거한다.
- 정확성은 PBT U1-P2(stateful — 다기기×명령 시퀀스)가 상시 검증: 체인당 현행 ≤1 / 재사용 시 발급 0+체인 무효 / 타 체인 불간섭 / 90일 만료 후 갱신 불가.

#### 비밀번호 정책·적응형 해시 (PAT-SEC-04)

- **정책(NFR-U1-SEC-01, G22)**: 8자 이상 + 영문/숫자 조합, 최대 128자(비정상 입력 방어). 유출 비밀번호 목록 검증 필수.
- **적응형 해시(NFR-U1-SEC-02)**: 메모리 하드 알고리즘 — **argon2id 권고**(GPU/ASIC 내성, RFC 9106·OWASP 1순위). 초기 파라미터 `m=19456KiB, t=2, p=1`에서 시작해 검증 지연 실측 **100–300ms**로 캘리브레이션. `DelegatingPasswordEncoder`(`{argon2}` 프리픽스)로 감싸 알고리즘·파라미터를 **해시 문자열에 자기 기술** — 파라미터 상향 시 **로그인 성공 시점에 투명 재해시**(사용자 무감지 업그레이드). 파라미터는 외부화 설정으로 재빌드 없이 조정. bcrypt 비교·argon2 확정은 기술 스택 결정 참조.
- **유출 검증 2단(NFR-U1-SEC-04)**: (1) **로컬 상위 유출 목록**(오프라인 사전, 상위 10만 수준) 필수 1차 검증, (2) **k-익명성 온라인 조회**(해시 프리픽스 5자만 외부 전송 — 평문·전체 해시 미전송) 보강 검증. 온라인은 1초 타임아웃, 실패 시 **로컬 통과만으로 진행(fail-open)** 하되 실패 계측(외부 서비스가 가입 가용성을 좌우하지 않게).
- **소셜 계정(NFR-U1-SEC-03)**: sub 식별, passwordHash nullable(NULL), 비밀번호 로그인 경로 자체 차단.
- 해시는 **서버 전용** — 클라이언트 키 스트레칭 금지. 해시 1건당 ~19MiB 오프힙 메모리 × 동시 로그인 병렬도가 인스턴스 사이징 입력이다. argon2 해시는 DB 커넥션을 잡지 않은 상태에서 수행(해시 후 커넥션 획득 순서).

#### 세션 무효화 매트릭스 (NFR-U1-SEC-05)

| 이벤트 | 무효화 범위 |
|---|---|
| 로그아웃 | 해당 기기 RefreshSession revoke(다기기 유지) |
| 비밀번호 변경·재설정 완료 | **전 기기 RefreshSession revoke**(탈취 대응) |
| 리프레시 토큰 재사용 감지 | 해당 **회전 체인 전체 revoke** + 보안 이벤트·알림 |
| 계정 삭제 요청(ACTIVE→DELETION_PENDING) | 전 세션 revoke(재로그인=철회 경로만 허용, D18) |
| 유예 만료 파기 | 전 인증 수단·세션 소멸 |

#### 브루트포스 방어 — 2축 점진 지연 (PAT-SEC-03) `[전역 — 정본 U1]`

즉시 잠금만 쓰면 공격자가 타인 계정을 고의로 잠그는 DoS가 되고, 무방비면 추측이 무제한이다. 2축 방어 + 점진 지연(수치는 외부화 설정, 운영 조정):

| 축 | 규칙 | 응답 |
|---|---|---|
| 계정 단위(NFR-U1-SEC-07) | 연속 실패 ≤5회 지연 없음 → 6~10회 지수 지연 **2·4·8·16·32초** → 10회 초과 **15분 일시 잠금** + 실패 급증 보안 이벤트. 성공 시 카운터 리셋 | 잠금 중에도 "로그인할 수 없습니다" **일반화**(계정 존재 비노출) |
| 출처(IP/기기) 단위(NFR-U1-SEC-08) | 로그인 **분당 10·시간당 100** / 가입·재설정·닉네임 확인 동일 축 / 메일 재발송 **분당 1·일 5회**(G22 — 서버 강제, 외부 쿼터 보호 겸용) | 429 + retryAfter(재발송은 카운트다운 UI) |

카운터 정본은 **PostgreSQL**(인스턴스 로컬 금지, 초기 규모 0.5 RPS에서 DB로 충분). 비밀번호 재설정·이메일 인증 링크는 **1회용·24h 유효·사용 즉시 소진**(NFR-U1-SEC-09). CAPTCHA는 1차 미도입 — 도입 트리거(자동화 공격 관측)를 운영 지표로 정의. 이 레이트리미터는 전역 재사용 컴포넌트로, 후속 유닛의 고비용 엔드포인트(예: U5 LLM 호출 사용자별 상한)가 동일 메커니즘을 재사용한다. 단조성·리셋·창 상한은 PBT U1-P13·P14가 검증.

#### MFA (NFR-U1-SEC-10)

SECURITY-12의 MFA 요건은 **어드민 계정 대상**이며, 어드민 도구는 후속 단계(D35)다. U1에서는 일반 사용자 MFA 미제공을 명시하고, 계정 모델에 인증 수단 확장 여지(SocialIdentity와 동열의 확장 가능한 자격 구조)를 두어 후속 어드민 MFA를 차단하지 않는다. → 판정: **후속 단계 적용 예약(현 시점 N/A)**.

### 3.3 시크릿 관리 — 외부 주입·하드코딩 금지 (PAT-SEC-05, NFR-U1-SEC-11) `[전역]`

U1은 시크릿 밀도가 가장 높다(소셜 클라이언트 시크릿 3종+Apple p8 서명 키, JWT 서명 키, 메일 API 키, DB 자격 증명). 소스·설정 파일 포함은 유출 시 회수 불가(git 이력 영구)이고 로테이션 불가는 사고 대응을 마비시킨다.

- **주입 인터페이스**: 전 시크릿은 기동 시 환경 주입(외부화 설정)으로만 수취 — 코드·리포지토리·이미지에 미포함. 설정 키 계약(이름·형식)만 코드가 소유하고 공급자는 어댑터 뒤. 설정 키 부재 시 **기동 실패(fail-fast — 묵시 기본값 금지)**.
- **로테이션 가능 구조**: JWT 서명 키는 **JWK 셋 + `kid` 롤오버**로 신 키 발급·구 키 검증 병행 창을 두어 무중단 교체. Apple client_secret은 p8로 서명한 ES256 JWT(수명 ≤6개월)라 주기 재생성이 구조적으로 강제 — 만료 전 갱신을 운영 체크리스트(OPS-01)에 포함.
- **개발·CI**: 로컬/CI는 fake 어댑터·더미 시크릿(실 IdP·메일 호출 없음), 시크릿 스캔(gitleaks류)을 PR CI 게이트에 포함(SECURITY-10 공급망 게이트와 동일 워크플로).
- 시크릿 매니저·KMS 제품 선정은 벤더 결정 대기(§11 PD-2·3) — 코드는 벤더 중립 환경 주입 인터페이스로 구성.

### 3.4 입력 검증 (SECURITY-05, NFR-U1-SEC-12~15)

- 전 API DTO에 **선언적 스키마 검증**(Jakarta Validation). 타입·형식(이메일 RFC 5321 254자 상한, 생년월일 형식·범위, provider 열거형)·**전 문자열 명시적 최대 길이**(닉네임 20자 G23, 비밀번호 128자, 취향 열거형) 강제.
- 요청 본문 크기 상한 **64KB**(프레임워크 레벨) — U1은 파일 업로드 없음.
- 전 DB 접근 **파라미터라이즈드**(JPA/JDBC 바인딩 — 문자열 결합 금지). 사용자 문자열(닉네임)은 저장 전 C3 금칙어+새니타이즈, 출력 시 이스케이프(XSS — 서버는 클라이언트 렌더링을 신뢰하지 않음).
- 소셜 IdP 콜백 입력(코드·id_token)도 외부 입력 — id_token은 서명·iss·aud·exp·nonce 검증(어댑터 책임), 검증 실패는 구조화 오류로 **fail-closed**(SECURITY-15).

### 3.5 LLM 경계 (SECURITY-11, D31) `[전역]`

- **서버 재조회 방식**: LLM 컨텍스트 주입은 서버가 필요한 데이터를 서버측에서 재조회해 구성한다 — **타 사용자 비공개 데이터·내부 지표(제휴 수수료 등)가 구조적으로 LLM 입력에 미포함**되게 한다. 클라이언트가 보낸 컨텍스트를 그대로 신뢰하지 않는다.
- **POI 그라운딩 closed-set(G115)**: LLM은 후보 ID 목록에서만 선택 — 그라운딩 실패(환각 POI)가 구조적으로 불가능하게 설계한다.
- **rate-limit**: LLM 호출은 사용자별 상한 적용(§3.2 레이트리미터 재사용).
- **국외 이전(G181)**: 국외 LLM 사용 시 개인정보처리방침에 국외 이전·처리위탁 고지, LLM 전송 필드는 목적 최소화. 상세는 U5·U6 유닛 문서.

### 3.6 로깅·PII 금지·증적 (SECURITY-03·14) `[전역]`

#### 구조화 로깅·PII 마스킹 (PAT-OBS-01)

- 전 서버 로그 **stdout JSON**(SLF4J+Logback+logstash-encoder) — 타임스탬프(ISO 8601)·레벨·모듈 태그(auth/profile/moderation/core)·**상관 ID** 필수. 상관 ID는 필터 체인 1단에서 요청당 생성·MDC 주입, 응답 헤더로 반환, 비동기 잡·아웃박스 이벤트에 전파.
- **로그 금지 항목(NFR-U1-SEC-22)**: 비밀번호(평문·해시 모두)·액세스/리프레시 토큰·소셜 코드/id_token·이메일 인증/재설정 토큰·**이메일 주소·생년월일(PII)**·Authorization 헤더. 계정 식별은 내부 accountId로만. 마스킹은 **로깅 프레임워크 레벨 마스킹 컨버터로 구조적 강제**(개별 로그문 재량 금지 — 사람이 안 지키는 규칙은 규칙이 아니다). 마스킹 회귀 테스트로 검증.
- 중앙 집중 로그 라우팅 제품은 벤더 대기 — 애플리케이션은 stdout JSON까지 보장.

#### 동의 증적·법정 로그 append-only 분리 (N2·N3, SECURITY-14)

- **동의 증적(NFR-U1-SEC-23)**: ConsentRecord는 **append-only** — 항목·약관 버전·동의/철회 일시를 행 추가로만 기록(UPDATE/DELETE 금지). 철회도 `revokedAt` 신규 행으로 남긴다. 보존은 계정 생명주기 + 분쟁 대비 법정 보존(파기 시에도 법정 보존 데이터로 분리 보관).
- **위치정보 법정 로그(NFR-U1-SEC-24, N2)**: 위치정보 수집·이용·제공 사실 확인자료 테이블 — **append-only · 최소 6개월 보존 · 애플리케이션 DB 역할에 DELETE/UPDATE 권한 없음(DB 권한 분리로 강제)**. U1은 테이블·권한 모델·기록 인터페이스를 소유하고, 실제 기록 발생은 U6(GPS 수집)부터다. append-only는 마이그레이션 리뷰로 검증.
- **보존 기간(NFR-U1-SEC-25)**: 일반 애플리케이션·감사 로그 **90일+**, 위치 법정 로그 **6개월+**, 동의 증적은 위 기준. 보존·아카이브 저장소 구성은 인프라 설계 대기.

#### 감사 이벤트·보안 알림 (PAT-OBS-02, NFR-U1-SEC-26·27)

행위자(accountId)·시각·결과·사유 코드·상관 ID를 구조화 기록하며, **아웃박스 경유**로 업무 트랜잭션과 원자 적재해 "상태는 바뀌었는데 감사 기록이 없는" 창을 제거한다(SECURITY-13 무결성).

| 분류 | 이벤트 |
|---|---|
| 계정 생명주기 | 가입(경로별), 이메일 인증 완료, 연령 확인 통과/차단(N1), 계정 삭제 요청/철회/파기 실행(D18), 미인증 정리(G22) |
| 인증 | 로그인 성공/실패(사유 코드), 브루트포스 지연 발동/잠금, 토큰 회전, **리프레시 재사용 감지**, 로그아웃, 비밀번호 변경/재설정 요청·완료 |
| 동의 | 약관 동의/재동의(N3)·철회, 위치 약관·GPS 옵트인 동의/철회(N2), 마케팅 동의/철회(N8) |
| 권한 | 401/403 권한 위반(엔드포인트·주체), 삭제 유예 계정 접근 시도 |
| 프로필 | 닉네임 변경, 금칙어 차단(C3) |

**보안 알림 대상**: (a) 인증 실패 급증, (b) 리프레시 재사용 감지 — **건당 즉시**, (c) 권한 위반(403) 급증, (d) 브루트포스 잠금 급증, (e) 메일 발송 실패율. 알림 채널·대시보드 제품은 인프라 대기 — 애플리케이션은 이벤트 발행 지점과 임계 정의까지 소유. 감사 로그(90일+)와 동의 증적·위치 법정 로그(별도 저장소, append-only·6개월+)는 저장소를 구분한다.

### 3.7 기타 보안 규칙 (SECURITY-04·09·10·15)

- **보안 헤더(SECURITY-04)**: 전역 응답 인터셉터로 HSTS·`X-Content-Type-Options: nosniff`·`X-Frame-Options: DENY`·Referrer-Policy. 순수 JSON API라 CSP는 실효 N/A이나 방어적 기본값(`default-src 'none'`) 권고(우발적 HTML 응답 대비).
- **하드닝(SECURITY-09)**: 오류 응답 일반화 — 스택·내부 경로 비노출. 이미지·스토리지 하드닝은 후속 스테이지.
- **공급망(SECURITY-10)**: 잠금 파일 커밋(Gradle versions.lock/verification-metadata, pnpm/yarn lock), 의존성 취약점 스캔 CI 게이트, SBOM 생성, 프로덕션 이미지 태그 고정.
- **예외 처리(SECURITY-15)**: 전역 에러 핸들러 — fail-closed, 사용자 응답은 일반화 메시지+오류 코드, 전 외부 호출(IdP·메일·DB) 명시적 오류 처리, 오류 경로 자원 정리(트랜잭션 롤백).

## 4. 관측성 (RESILIENCY-05·SECURITY-14) `[전역]`

구조화 로그 + APM/크래시 리포팅 + 외부 API 실패율 대시보드가 기본 구성이다(G144). 침묵 실패 금지(ADR-0011)의 서버 관측: 외부 API 실패·LLM 오류·솔버 검증 실패는 전부 계측 대상이다. 헬스체크(shallow+deep)는 설계 단계 요건이다.

### 4.1 3축 관측성

- **로그**: 구조화 stdout JSON·상관 ID·PII 마스킹(§3.6, PAT-OBS-01).
- **메트릭**: 골든 시그널·보안·외부 의존·자원·복원력 자세(아래 표, PAT-OBS-03).
- **트레이스**: 모듈러 모놀리스 단일 서비스(D04)이므로 분산 트레이싱은 **N/A(단일 서비스)** — 상관 ID가 요청 추적을 담당. 후속 워커 분리 시 재평가.

### 4.2 감사 로그 (PAT-OBS-02)

감사 로그는 일반 로그와 요건이 다르다 — 누락 불가·변조 불가·보존 기간 법정. 이벤트 전수·발행 경로(아웃박스)·알림 임계·보존은 §3.6에 정리했다.

### 4.3 메트릭·알람 대상 (PAT-OBS-03)

제품 선정은 인프라 대기(§11 PD-4). 알람 임계는 초안(운영 조정)이며 전 알람은 OPS-02 라우팅 표(P1/P2/P3)로 접속한다.

| 분류 | 메트릭 | 알람 임계(초안) |
|---|---|---|
| 골든 시그널 | API별 지연 p50/p95·오류율·처리량·포화(CPU/메모리) | p95가 §1 목표 5분 지속 초과 / 5xx율 >1%(5분) |
| 인증 보안 | 로그인 실패율·잠금 발동·재사용 감지·403 수 | §3.6 알림 임계 (a)~(d) |
| 외부 의존 | IdP별 실패율·지연, 메일 성공률·큐 적체, 유출 조회 실패율 | 서킷 open 즉시 / 발송 실패율 >5% / 큐 적체 임계 |
| 자원·용량 | 커넥션 풀 사용률·획득 대기, argon2 동시 실행 수, 외부 쿼터 사용률 | 풀·쿼터 **80%**(RESILIENCY-09) |
| 복원력 자세 | deep 헬스 상태, AZ별 인스턴스 수, DB 복제·백업 성공 | 단일 AZ 운전 감지 / 백업 실패 즉시(RESILIENCY-07) |
| 배치·잡 | 미인증 정리(G22)·유예 만료 파기·세션 만료 정리·아웃박스 릴레이 지연 | 잡 실패/미실행 24h / 릴레이 지연 임계 |
| 클라이언트 | 부트스트랩 3초 타임아웃 발동률(G5)·크래시율 | 타임아웃 발동률 급증 |

위 표를 1개 운영 대시보드(U1 헬스) 정의로 두며, 인프라 설계 산출물에 정의를 포함한다.

## 5. 성능 설계 패턴 (PAT-PERF-*) `[전역]`

### 5.1 토큰 검증 무상태 (PAT-PERF-01, NFR-U1-AV-01)

전 모듈 전 요청이 인증을 통과한다(D22). 요청마다 세션 DB 조회가 있으면 DB가 전 기능의 동기 결합점이 되어 열화가 즉시 전면 장애로 전파되고, 인증 오버헤드가 모든 API의 300ms 예산을 상시 잠식한다.

- 액세스 토큰(1h) 검증은 서명·만료·클레임(iss·aud)만으로 완결 — 요청 경로에 DB·캐시 조회 없음. JWT 서명 키는 JWK 셋 인메모리 캐시로 검증(kid 롤오버 정합).
- 상태 접근은 **갱신 시점으로 집중** — 세션 정본(RefreshSession)은 1시간에 1회의 갱신에서만 조회. DB 부하가 요청 수가 아닌 세션 수에 비례한다.
- **즉시 무효화 보완 수단**: 즉시 무효화가 필요한 경우(계정 삭제·비밀번호 재설정·제재)는 전 체인 revoke로 갱신을 즉시 차단하고, 상태 민감 연산(삭제 철회, 프로필 변경, 후속 유닛의 결제성·공유성 연산)은 핸들러 레벨에서 **계정 상태 재확인**을 명시 수행한다(무상태 원칙의 국소 예외 — 예외 지점은 코드에 어노테이션/명명으로 식별 가능하게). 전면 토큰 블랙리스트는 초기 규모 미도입.

### 5.2 커넥션 풀 원칙 (PAT-PERF-02)

커넥션 풀은 성능 장치이자 벌크헤드다. 과소(대기 급증)·과대(DB 최대 연결 고갈 — 특히 롤링 배포 중 구·신 인스턴스 합산 초과) 양방향으로 실패한다.

- **풀 사이징 공식**: `인스턴스 수(롤링 중 최대치 포함) × 풀 최대 크기 < DB max_connections × 안전율(80%)`. 초기 규모(피크 0.5 RPS, 헤드룸 10 RPS)에서는 인스턴스당 소형 풀(10 내외)로 충분.
- **타임아웃 3종**: 커넥션 획득(무한 대기 금지 — 획득 실패는 빠른 5xx+계측), 쿼리, 유휴/수명(페일오버 후 죽은 연결 정리).
- **벌크헤드 분리**: 요청 처리 풀과 **백그라운드 잡(메일 발송·아웃박스 릴레이·정리 배치) 풀을 분리** — 잡 폭주가 로그인 경로 커넥션을 고갈시키지 않게. argon2 해시는 DB 커넥션을 잡지 않은 상태에서 수행.
- 풀 사용률·대기·획득 실패 메트릭 노출, 사용률 80% 알람.

## 6. 규모 가정 (G142)

초기 출시(국내 한정): **DAU 1천 / 동시 일정 생성 10건 / 지역당 POI 후보 5천**(MVP 검증 규모). 오토스케일 한도·비용 상한 알람과 함께 인프라 설계의 입력으로 사용한다.

인증 계열 트래픽 산정(DAU 1,000 기준, 일량의 20%가 1시간에 집중):

| 트래픽원 | 산정 | 피크 |
|---|---|---|
| 부트스트랩 확인(앱 실행마다 1회) | 1,000 × 일 5회 = 5,000건 | ~0.28 RPS |
| 토큰 갱신(액세스 1h 만료) | 활성 1,000명 × 일 2회 = 2,000건 | ~0.11 RPS |
| 로그인(리프레시 90일이라 저빈도) | DAU의 ~5% = 50건 | ~0.003 RPS |
| 가입(온보딩 여정 ~15콜/인) | 신규 일 50명 × 15 = 750건 | ~0.04 RPS |
| 취향·닉네임·동의 등 프로필 | ~1,000건 | ~0.06 RPS |

**정상 피크 합산 ≈ 0.5 RPS.** 절대 규모는 작으나 아래 요구가 설계를 지배한다.

- **설계 처리량(NFR-U1-SC-01)**: 인증 계열 API는 정상 피크의 **최소 20배 헤드룸 = 10 RPS**를 단일 인스턴스 기준으로 소화(출시 직후 마케팅·스토어 피처링 스파이크 대비). 적응형 해시 CPU 비용(해시 1건당 코어 점유 ~100–200ms)이 로그인/가입 병렬도의 실질 상한이며 인스턴스 사이징 입력이다.
- **오토스케일 연동(NFR-U1-SC-02)**: min/max 한도·비용 상한 알람은 인프라 설계 확정 — 애플리케이션은 무상태 수평 확장 가능 구조를 보장.
- **데이터 볼륨(NFR-U1-SC-03)**: 첫해 가정 계정 ~2만, 동의 증적 계정당 평균 8행, RefreshSession 계정×기기 — 90일 만료분 정리 배치 필요. 위치 법정 로그는 U6 활성 후 증가(테이블 파티셔닝 여지 확보). 전 테이블이 단일 PostgreSQL로 충분하며 **샤딩 불요**.
- **외부 쿼터(NFR-U1-SC-04)**: 소셜 IdP 4종·메일 발송의 rate-limit·쿼터 문서화, 사용량 80% 알람. 메일 재발송 상한(분당 1·일 5회, G22)은 서버 강제로 외부 쿼터 보호를 겸한다.

## 7. 테스트 전략 (D37 + PBT 전체 강제)

### 7.1 계층 분리 (D37)

- PR CI에서 **LLM·외부 API는 모킹**(어댑터 계약 인터페이스 + Port fake) — 실 IdP·메일 호출 없음. 솔버·하드 제약 검증은 실코드 테스트. **실 LLM 회귀 평가는 릴리스 파이프라인에서만**.
- DB는 Testcontainers PostgreSQL(실 엔진 — 마이그레이션·append-only 권한 검증 포함). 외부 어댑터 fake로 실패 주입(타임아웃·5xx·서킷 open)을 PR CI에 포함해 §2.4·§2.5 계약을 검증한다.
- 모듈 경계는 컴파일 타임 강제(Gradle 멀티모듈 — 타 모듈 내부 접근 시 컴파일 에러) + 아키텍처 테스트(ArchUnit/Konsist)로 퍼사드 우회·순환 의존 검출(NFR-U1-MT-01).

### 7.2 CI 게이트 — 하드 제약 100% (G114)

**하드 제약 검증 테스트는 100% 통과 필수(머지 차단)**, 그 외는 비차단 리포트로 시작한다. 하드 제약 대상:

- 숙소 기준점 — 등록 숙소 위치·체크인/아웃 날짜가 일정 생성 기준점(출발점)으로 반영.
- 충돌 무배치 — 재계획 결과가 남은 일정·이동 시간·고정 항목과 충돌하지 않음.
- POI 그라운딩(G115) — LLM은 closed-set 후보 ID에서만 선택, 환각·폐업 POI 배제.
- 계정 무결성 — 동일 이메일 중복 계정 미생성(충돌 시 단일 계정 통합), 이메일 미인증 계정이 활성으로 취급되지 않음.

CI 파이프라인 자체가 변경 관리(OPS-01)의 구현체다: 브랜치 보호는 테스트·PBT·아키텍처 테스트·시크릿 스캔·의존성 스캔이 전부 녹색이어야 머지를 허용한다.

### 7.3 PBT 전체 강제 (Kotest · fast-check)

속성 기반 테스트(Property-Based Testing)를 전 유닛 강제한다.

- **프레임워크**: **Kotest**(백엔드) · **fast-check**(클라이언트). 시드 로깅·수축(shrinking) 필수(PBT-08).
- **1급 대상**: 솔버 하드 제약(영업시간 내 배치·이동시간 부등식·고정 블록 불변), 직렬화 왕복, 상태 머신(여행·일정·방문 상태). 각 유닛의 속성 식별은 Functional Design에서 수행(PBT-01).
- **트리거 재현(G116)**: Plan-B 판정 로직은 clock·외부 데이터 주입 가능한 순수 함수로 분리해 결정적 테스트.
- **U1 PBT 예시**: 계정 상태 머신, 토큰 회전 재사용 감지(U1-P2), 닉네임 생성기, 직렬화 왕복, 재발송 창(U1-P13), 브루트포스 지연 비감소·리셋(U1-P14), 금칙어 Pass 불위조(U1-P15). PR CI 포함, 시드 로깅·수축 필수.

### 7.4 좋은 테스트의 기준 (PRD 14)

테스트는 LLM이 만든 문장의 표현 품질이 아니라 **사용자에게 도달한 결과가 필수 조건을 충족했는지**를 검증한다.

- 단일 기능이 아니라 **'숙소 저장 → 외부 예약/저장 숙소 등록 → AI 일정 생성 → 여행 중 재계획'이 하나의 흐름으로 연결되는지** — 각 단계 출력이 다음 단계 입력으로 정상 전달되는지 검증한다.
- **등록 숙소의 위치·체크인/아웃 날짜가 AI 일정 생성의 기준점으로 반영되는지** — 등록 숙소가 바뀌면 일정의 지리적·시간적 기준점도 갱신되는지 검증한다.
- **외부 OTA 제휴 링크가 핵심 예약 경로로 정상 동작하는지** — 올바른 숙소·날짜 파라미터 전달 포함.
- **재계획 결과가 남은 일정·이동 시간·고정 항목(등록 숙소 체크인/아웃, 외부 예약 항목)과 충돌하지 않는지**.
- **추천 POI가 외부 장소 데이터에 그라운딩(RAG)되어, 환각 POI·폐업 장소가 일정에서 제외되는지**.
- **소셜/이메일 가입·로그인 시 동일 이메일 중복 계정이 생성되지 않고(충돌 시 단일 계정 통합), 이메일 미인증 계정이 활성으로 취급되지 않는지**.
- **외부 API(OTA 링크/숙소 데이터/지도·경로/LLM)가 실패하거나 빈 결과를 반환할 때 앱이 fallback 흐름으로 복구하는지** — 사용자에게 빈 화면·오류만 남지 않아야 한다.

## 8. 운영 프로세스 (OPS-*) `[전역 정본]`

조직에 공식 변경 관리·온콜/IR 프로세스가 없어(GD-2·GD-6=A), U1에서 1회 수립해 전 유닛·Operations가 재사용하는 경량 프로세스다. GitHub 위에서 도구 추가 없이 성립한다.

### 8.1 OPS-01 경량 변경 관리 (RESILIENCY-03·04, GD-2·3·4·5)

프로세스 부재는 변경 유발 장애(장애 최다 원인)를 통제 불능으로 만든다. 3요소:

1. **PR 승인 게이트** — `main` 브랜치 보호: 직접 push 금지, PR 필수, **승인 1인 이상**, CI 게이트 전부 녹색. 프로덕션 배포는 `main` 머지 커밋 + 버전 태그에서만 실행(수동 배포 경로 없음). 고위험 라벨(`risk:high` — 인증 필터·마이그레이션·시크릿·인프라)은 승인 2인 강화.
2. **변경 기록** — PR 템플릿이 곧 변경 기록: 변경 요약 / 영향 범위(모듈·엔드포인트) / 리스크·검증 방법 / **롤백 노트** / 관련 추적 ID. 머지된 PR + 배포 태그 + Actions 실행 이력이 변경 이력 정본. DB 마이그레이션 PR은 forward-only·호환 규칙과 append-only 권한 DDL 리뷰를 체크 항목으로 강제.
3. **롤백 노트(모든 배포 PR 필수)** — 정본 메커니즘은 **직전 버전 아티팩트 재배포(버전 고정 롤백)**. **DB는 롤백하지 않는다** — 마이그레이션은 forward-only이므로 모든 스키마 변경은 **N-1 호환 규칙**을 지킨다: expand(추가·병행) → 코드 전환 → contract(제거)의 2단계 배포. 컬럼 삭제·NOT NULL 강화·타입 변경은 별도 후행 배포로 분리(위반은 PR 차단 소견). 롤백 시 추가 조치("없음" 포함)를 명시 기입.

### 8.2 OPS-02 경량 인시던트 대응 (RESILIENCY-15, GD-6)

M1은 Critical이라 알람이 울려도 "누가 받고, 무엇부터 보고, 끝나면 무엇을 남기는가"가 없으면 알람 정의는 소음이다. 3단 구성:

**1) 알람 라우팅 (심각도 3단)**

| 심각도 | 정의(예시) | 라우팅 | 초동 목표 |
|---|---|---|---|
| P1 | Critical 경로 불능 — 로그인/갱신 실패율 급증, DB 불능, 전 AZ 인스턴스 미가용 | 즉시 호출 채널(전화/푸시 — 제품은 PD-4 확정) | 15분 내 인지·대응 개시 |
| P2 | 저하 운전 — IdP 서킷 open 지속, 메일 실패율 초과, 재사용 감지 급증, 단일 AZ 운전 | 팀 메신저 | 업무 시간 내 대응 |
| P3 | 주의 — 쿼터/풀 80%, 배치 실패 1회, p95 목표 초과 | 메신저(비긴급)+주간 리뷰 | 다음 계획 주기 |

보안 특칙: **리프레시 재사용 감지는 건당 P2 이상**(다계정 동시면 P1 승격), 인증 실패 급증+잠금 급증 동시 발생은 공격 의심으로 P1.

**2) 대응 절차(런북 골격)** — 선언(대응자 지정, 발견자가 초기 대응자, 인시던트 스레드·타임라인 시작) → 진단 순서(deep 헬스 → 골든 시그널 대시보드 → 상관 ID 로그 추적 → **최근 배포 대조**("최근에 무엇이 바뀌었나"가 첫 질문)) → 완화 우선(원인 규명보다 복구 — 직전 버전 롤백, 외부화 설정 조정, Multi-AZ 페일오버 확인, 보안 인시던트는 해당 계정 전 체인 revoke가 1차 조치) → 종결(메트릭 정상 복귀 확인 + 타임라인 마감).

**3) 사후 회고 (COE — 비난 없는 포스트모템, P1·P2 필수)** — 심각도/영향, 타임라인(감지→선언→완화→종결), 근본 원인(5 Whys — 사람 아닌 시스템·프로세스 결함), 잘 된 것/운이 좋았던 것, 시정 조치(각 항목 = GitHub Issue로 발행·담당·기한), 재발 감지 개선. 미종결 시정 조치는 주간 리뷰 상시 항목.

### 8.3 OPS-03 복원력 테스트 시나리오 (RESILIENCY-14, GD-7)

검증되지 않은 페일오버는 존재하지 않는 페일오버다. MVP 규모(DAU 1천)에서 카오스 상시 운영은 과설계이므로, **실행은 Operations로 이연하고 시나리오 정의까지 확보**한다. DR 전략 = Backup & Restore, RTO/RPO = 시간 단위 기준.

| ID | 시나리오 | 합격 기준 | 실행 시점 |
|---|---|---|---|
| SC-1 | Multi-AZ 컴퓨트 페일오버(1개 AZ 인스턴스 강제 종료) | 로그인·갱신·조회 무중단(기존 토큰 유효), 오류율 순간 상승 후 1분 내 정상, 알람 P2 발화 | Operations 이연 |
| SC-2 | 관리형 PostgreSQL Multi-AZ 페일오버 | 전환 중 실패는 명시 오류(침묵 실패 금지), 전환 후 자동 회복, 데이터 손실 0(동의 증적·법정 로그 무결) | Operations 이연 |
| SC-3 | DB 백업 복원 리허설(DR 본체 — PITR로 신규 인스턴스 복원) | **RTO 시간 단위 내 복원·RPO 손실 창 확인**, append-only 권한 모델 복원 확인 | Operations 이연(최초 1회는 출시 전 권장) |
| SC-4 | 외부 의존 장애 주입(IdP 1종/메일 차단) | §2.5 "유지되는 기능" 전부 정상, 서킷 open·알람 발화, 고지 문구 노출 | 스테이징(CI fake 주입이 선행 커버) |
| SC-5 | 롤백 리허설(직전 버전 태그 재배포) | 롤백 소요 측정·기존 세션 유지, 호환 규칙(expand/contract) 실증 | Operations 이연(첫 배포 직후 1회 권장) |
| SC-6 | 롤링 배포 무중단 검증(배포 중 연속 호출 부하) | 오류 0(재시도 포함), 구·신 인스턴스 혼재 구간 정상(무상태 검증) | Operations(파이프라인 정착 시) |

이연 경계 명시: PR CI의 fake 실패 주입 테스트(§7.1)는 **이연 대상이 아니며** Code Generation부터 상시 실행한다. 이연되는 것은 실 인프라 대상 실험(SC-1~6)뿐이다. 각 실행 결과는 OPS-02 회고 템플릿 축약형(시나리오 ID·실측 RTO/RPO·발견·시정 조치 Issue)으로 리포지토리에 기록한다.
