## 1. 개요

### 1.1 유닛 정의

| 항목 | 내용 |
|---|---|
| 유닛 | U3 숙소·장소 데이터 |
| 에픽·스토리 | E3 — US-E3-01~11 (11개) |
| 담당 서버 모듈 | M3 Accommodation Search, M4 Saved Accommodation, M5 Affiliate Link, M7 Place Data(기반) |
| 클라이언트 features | `features/stay`(탐색·상세·위시리스트·등록), `shared/map`(카카오 지도 SDK 브리지 — config plugin 네이티브 모듈) |
| 신규 엔티티 | 약 8~9종 |
| 외부 연동 | 4종(TourAPI·카카오·네이버 폴백·OTA 아웃바운드) + 클라이언트 카카오 지도 SDK — 1차 유닛 중 외부 의존 최다 |
| 계약 포인트 | CP1(U3→U4) 공급자 측 |

**목적**: TripPilot의 데이터 척추 — POI 정본 파이프라인(M7)과 숙소 도메인(M3·M4·M5) — 를 세운다. "숙소 탐색 → 위시리스트 → OTA 딥링크 예약 → 등록(거점 후보화)"와 "장소 검색 → 저장 POI"가 완성되어, U4 이후의 모든 유닛이 이 데이터 위에서 동작한다.

### 1.2 범위 — 포함

- **M7 Place Data 기반**: TourAPI·카카오 장소 검색 수집·정규화, canonical POI ID(좌표 50m 근접+명칭 유사도 매칭, G133/G148), 하이브리드 캐싱(사용자 확정 데이터 스냅샷 영구 + 탐색 풀 TTL 캐시, D13), 영업시간·카테고리·체류 기본값 공급, 인기 장소 일 1회 배치 집계(G2).
- **M3 숙소 탐색**: 여행지 기반 탐색(날짜·인원 없이, D09), 필터·정렬(유형·편의시설·대표 가격대·직선거리 G34 — 소요시간 미표시 D25), 상세(정적 콘텐츠·리뷰는 OTA 위임), '가격 보기' 라이브 조회(G33), 부분 실패 처리.
- **M4 등록 숙소**: 계정 레벨 풀(D15), 수동 등록 단일 경로+직접 등록 3경로(지도 검색·링크 파싱 G31·핀 지정), 내부 숙소 ID·소스별 외부 ID N:1 매핑(D17), 등록·저장 목록.
- **M5 OTA 딥링크**: 숙소명 검색 딥링크, 제휴 고지, 다중 OTA 선택(D17), 복귀 핸드오프 카드(G32), 아웃바운드 클릭 로그.
- **U2 온램프 셸에 장소 검색·저장 POI 백엔드 연결**(US-E2-05 전 기능 활성화).

### 1.3 범위 — 명시적 제외

- 포스트백 1탭 자동 등록 — 후속(D09/G29·G108), 집계용 설계 여지만.
- 거점의 **여행 연결·날짜 비중첩 검증** — 데이터·API는 U3(M4)이 소유하되 여행 컨텍스트가 필요한 검증·UI는 U4에서 완성.
- 정확 가격 일괄 조회·캐싱(하지 않음), OTA 크롤링(금지), 리뷰·평점 표시(OTA 위임).
- 혼잡도(1차 제외, '미확인' 표기 G199).

### 1.6 완료 기준(DoD)

- **기능**: E3 11개 스토리 수용 기준 충족(부분 실패·0건·데이터 부족 지역 흐름 포함). 좌표 미확정 등록의 "지도에서 위치 확인" 강제.
- **하드 제약(D37)**: POI 그라운딩의 전제인 canonical ID 무결성(동일 실세계 장소 1 canonical), 등록 숙소 필수 필드·날짜 순서 검증(체크아웃>체크인) 테스트 100%.
- **PBT**: POI 매칭 멱등성·대칭성(같은 입력 재수집 시 canonical 불변), 숙소 ID N:1 매핑 불변식(외부 ID는 정확히 1개 내부 ID), TTL 캐시 만료 속성(만료 후 재조회 강제), OTA 링크 파서(화이트리스트 밖 URL 전부 거부·파싱 실패 시 폴백 경로), 등록 폼 검증 함수.
- **확장 규칙**: RESILIENCY-10(TourAPI·카카오·네이버 전 어댑터 타임아웃+서킷+폴백 — 카카오 실패 시 네이버, 전체 실패 시 수동 등록 우회), SECURITY-05(외부 응답 스키마 검증·링크 파싱 입력 방어), SECURITY-08(위시리스트·등록 숙소 소유권).
- **계약 포인트**: **CP1(U3→U4) 공급자 측** — 등록 숙소·저장 POI 스키마 계약 테스트. StayRegistered 이벤트 발행 검증(U8 소비 대비).

### 1.8 선결 과제 (비개발 — 설계 차단)

| 과제 | 시점 | 내용 | 중립 대응 |
|---|---|---|---|
| **P2 지도 API 약관 검토·계약** | **U3 착수 전(설계 차단)** | 카카오·카카오모빌리티·네이버 — 특히 D13 스냅샷 적법성 법무 확인 | 지도 소스 영구 저장 표면 0(인메모리 TTL), 스냅샷 필드 최소화·정책 컬럼, P2 확정 전 지도 소스 스냅샷 보류 |
| **P4 TourAPI 활용 신청·캐싱 조건 확인** | **U3 착수 전(설계 차단)** | 캐싱 허용 범위가 M7 저장 정책을 좌우 | `TourContentPort` 격리, 허용분만 정본 영구·보존 정책 컬럼으로 소급 축소 |
| P5 OTA 제휴 계약 | U3 출시 전(개발은 검색 딥링크로 무차단) | 파트너별 딥링크 정책 확인, 수수료 고지 문안 확정 | 딥링크 템플릿·화이트리스트 remote config화 |

**차단 경계**: P2·P4는 비개발(법무·제휴) 선결이며 NFR·Infra 산출물(중립 요건)은 차단되지 않는다. 차단되는 것은 **U3 Functional Design의 스냅샷·정본 테이블 확정 스키마**뿐이고, 본 설계는 소스별 캐싱 매트릭스(§2.11)를 데이터 모델에 컬럼 수준으로 반영해 법무 확정 결과를 셀 교체로 흡수하도록 구성했다.

### 1.9 Functional Design 확정 설계 결정 (FD-U3-xx)

| # | 결정 | 내용 | 근거 |
|---|---|---|---|
| FD-U3-01 | canonical 키 이원화 | POI는 `canonicalPoiId`, 숙소는 `canonicalStayId`를 각각 정본 키로 분리 — 소스별 외부 ID는 양쪽 모두 N:1 매핑(동일 패턴). POI/숙소는 별개 정체성(숙소도 장소지만 SavedStay는 예약 컨텍스트 보유) | D17, G133/G148 |
| FD-U3-02 | 소스별 캐싱 매트릭스 총함수화 | `cachingPolicy(source, field) → {PERMANENT, TTL_CACHE, LIVE_ONLY, NOT_HELD}`를 전 (소스×필드) 셀 정의 총함수로 확정(§2.11). 지도 API 비식별자=TTL, TourAPI·사용자 확정 스냅샷=영구, 정확 가격=LIVE_ONLY. 법무(P2·P4) 결과는 셀 교체로 흡수 | D13, ADR-0017, G70/G105/G124/G169/G193 |
| FD-U3-03 | 사용자 확정 스냅샷 영구화 축 | 지도 API 영구 캐싱 금지의 정책적 예외를 'PoiSnapshot(사용자 입력 데이터)'로 한정 확정 — 탐색·추천 풀은 스냅샷화 금지(TTL만). 원본 갱신은 스냅샷에 역류하지 않음(시점 고정) | D13, P2, G124 |
| FD-U3-04 | dataStatus 4상태 | Poi 생애를 `ACTIVE / UNVERIFIED / LOST / CLOSED` 4상태로 확정. UNVERIFIED는 자동 확정 안 함(사용자 확인 후보 분리), LOST/CLOSED는 후보 풀·필수 방문지 시드 투입 제외(스냅샷은 잔존) | G8, G192, G199 |
| FD-U3-05 | 매칭 보수 정책 | 좌표(50m)+명칭 유사도 매칭은 임계 미달 시 신규 canonical 생성(과병합보다 과분리 우선) + 운영자 보정 여지 — 오매칭이 U5 그라운딩을 오염시키는 리스크 최소화 | D17, G133/G148, U3 리스크 |
| FD-U3-06 | 좌표 확정 게이트 | coordConfirmed=false SavedStay는 거점 자격·일정 생성 기준점 투입 불가 — "지도에서 위치 확인" 강제. CP1 소비(U4)에서도 동일 차단(이중 방어) | D15, PRD 04-6·8 예외, CP1 |
| FD-U3-07 | 거점 비중첩 분업 | BaseAssignment 스키마·비중첩 판정 순수 함수는 U3 공급, 여행 컨텍스트 검증 실행·거점 UI는 U4 소비 — 계정 풀 미연결 숙소는 검증 대상 아님 | D15, unit-of-work U3 §제외, CP1 |

## 2. 도메인 엔티티

소유 모듈: M3 Accommodation Search, M4 Saved Accommodation, M5 Affiliate Link, M7 Place Data. 표기 규약: `필수`=NULL 불가, `선택`=NULL 허용(NULL의 의미를 반드시 명기), `유니크`=제약 범위 내 유일. 기술 중립 — 저장 기술·컬럼 타입·인덱스·캐시 구현체는 §6·§7(NFR/Infra)이 소유한다. ### 2.1 Poi — 장소 정본 (M7)

관광지·식당·카페·숙소 장소의 canonical 정본. **canonical POI ID가 전 유닛 장소 참조의 단일 키**이며, 소스별 place_id는 이 아래에 N:1로 매달린다(G133/G148, D17 동일 패턴). U5 POI 그라운딩 하드 제약(출력 POI ⊆ 후보 풀)의 전제 무결성을 소유한다.

| 속성 | 타입 | 제약 | 의미·근거 |
|---|---|---|---|
| canonicalPoiId | 식별자 | 필수 · 유니크 · 불변 | **정본 키** — 동일 실세계 장소는 정확히 1개(INV-POI1). 앱 내부 생성 ID(외부 place_id와 독립) — 영구 저장 허용(식별자는 캐싱 금지 대상 아님) [G133] |
| nameKo | 문자열 | 필수 | **한국어명 정본**. API 응답의 한국어 필드 우선, 없으면 원문 노출 [G123/G148] |
| aliasEn | 문자열 | 선택 | 영문 alias(대조·검색 보조). NULL=영문 미확보 [G148] |
| coord | 구조체 {lat, lng} | 선택 | 좌표(위도·경도). NULL=지오코딩 미확보(=`UNVERIFIED`/`LOST` 사유). 지도 API 유래 좌표는 영구 저장 아닌 TTL 캐시, TourAPI 유래는 정본 저장 [D08, D13, G124] |
| category | 열거(표준 카테고리) | 필수 · 기본 `UNKNOWN` | 소스별 카테고리 코드를 단일 표준 스키마로 정규화(ADR-0009). 숙소 유형(호텔·게스트하우스·펜션·모텔·호스텔·아파트·B&B·한옥 등)의 한국형 분류는 TourAPI cat3 정본 [PRD 04-2] |
| openingHours | 구조체(요일별 구간) | 선택 | 영업시간. NULL=미확인 → dataStatus `UNVERIFIED`(빈칸 금지·"미확인" 표기의 원천) [PRD 04-3 예외, G192] |
| stayRange | 구조체 {min, rec, max} | 선택 | 카테고리 체류 기본값의 조인 참조(정본은 StayTimeTable §2.12). NULL=테이블 미매핑 [G51] |
| dataStatus | 열거 {ACTIVE, UNVERIFIED, LOST, CLOSED} | 필수 · 기본 `UNVERIFIED` | POI 생애 상태 — 아래 §2.1.2 상태 규칙 |
| sourceAttribution | 문자열 | 필수 | 출처 표기(약관 요건 — 지도 API 응답 표기 의무) [ADR-0017] |

**PoiSourceRef**(Poi 서브엔티티): `{source: 열거(KAKAO, NAVER, TOURAPI, OTA), placeId: 문자열}` — 소스별 외부 식별자. `(source, placeId)`는 canonicalPoiId에 N:1로 귀속. 식별자 자체는 영구 저장 허용.

#### 2.1.1 dataStatus 값 의미

- `ACTIVE`(정상) · `UNVERIFIED`(영업시간·좌표 미확인) · `LOST`(정본 동기화에서 확인 불가 → '확인 불가' 배지·시드 투입 제외, G8) · `CLOSED`(폐업 확인).

#### 2.1.2 dataStatus 상태 규칙 (정본)

```text
   [수집·정규화]
       │
       ▼
   UNVERIFIED ──(좌표·영업시간 확보)──▶ ACTIVE
       │                                  │
       │                                  │ (정본 동기화 배치 — 소스에서 확인 불가)
       │◀─(재조회 성공)────────────────────┤
       ▼                                  ▼
     LOST ◀──(동기화 확인 불가)──────────── LOST
       │
       │ (폐업 확인 신호)
       ▼
    CLOSED
```

- 확정 배치는 `UNVERIFIED`를 자동 확정하지 않는다 — 사용자 확인 후보로 분리 판정 데이터만 공급(M7이 소비자 M8·M9에 상태 전파) [PRD 06-3 예외].
- `LOST`·`CLOSED` POI는 후보 풀·필수 방문지 시드 투입에서 제외(G8) — 단 이미 저장된 PoiSnapshot(사용자 확정 사본)은 마지막 스냅샷으로 잔존 표시(허위 최신성 금지).

#### 2.1.3 불변식 (Invariants)

- **INV-POI1 (canonical 유일성 — 하드 제약)**: 동일 실세계 장소(좌표 50m 근접 ∧ 명칭 유사도 임계 이상)는 정확히 1개 canonicalPoiId에 매핑된다 — 소스별 place_id는 N:1. 이 무결성은 D37 하드 제약 '숙소 기준점·POI 그라운딩' 계열의 전제이며 머지 차단 테스트 대상 [G133/G148, U3 DoD, PBT U3-P1].
- **INV-POI2 (매핑 단조·비파괴)**: PoiSourceRef 추가로 canonicalPoiId는 변경되지 않는다(같은 입력 재수집 시 canonical 불변 — 멱등·대칭) [PBT U3-P1].
- **INV-POI3 (미확인 명시)**: coord·openingHours가 NULL이면 dataStatus는 `ACTIVE`일 수 없다(`UNVERIFIED` 이상) — "미확인"은 빈칸이 아니라 명시 상태값으로 전파 [PRD 04-3 예외, G199].
- **INV-POI4 (한국어명 존재)**: nameKo는 항상 값을 가진다(한국어 필드 없으면 원문으로 채움) — 표시 공백 금지 [G123].
- **INV-POI5 (캐싱 게이트)**: 지도 API(KAKAO·NAVER) 유래 좌표·영업시간·명칭 필드는 §2.11 매트릭스의 '영구 금지' 셀에 해당하면 TTL 캐시로만 보관하며 만료 후 재조회를 강제한다. 정본 영구 저장은 캐싱 허용 소스(TourAPI) 또는 사용자 확정 스냅샷(§2.2)에 한한다 [D13, ADR-0017, G193, PBT U3-P3].

### 2.2 PoiSnapshot — 사용자 확정 스냅샷 (M7, 영구)

사용자가 **확정한** 데이터(등록 숙소·방문 체크·필수 방문지·저장 POI)의 확정 시점 사본. '사용자 입력 데이터'로 취급해 **영구 저장**하며(D13, P2 법무 확인 전제), 원본 Poi가 `LOST`/`CLOSED`가 되어도 잔존한다.

| 속성 | 타입 | 제약 | 의미·근거 |
|---|---|---|---|
| snapshotId | 식별자 | 필수 · 유니크 · 불변 | — |
| canonicalPoiId | 식별자(Poi 참조) | 필수 | 원본 참조(라이브 대조용) — 원본 소실 시에도 스냅샷은 독립 잔존 |
| purpose | 열거 {SAVED_PLACE, MUST_VISIT, VISIT_CHECK, REGISTERED_STAY} | 필수 | 확정 맥락. MUST_VISIT·VISIT_CHECK는 U4·U6 소비(U3은 SAVED_PLACE·REGISTERED_STAY 생산이 주) [D13] |
| nameKo | 문자열 | 필수 | 확정 시점 명칭 사본(영구) |
| coord | 구조체 {lat, lng} | 선택 | 확정 시점 좌표 사본 — **사용자 입력 데이터로 영구 저장 허용**(지도 API 실시간 캐싱 금지의 예외 축, D13) [G124, G70] |
| category | 열거 | 필수 | 확정 시점 카테고리 사본 |
| openingHours | 구조체 | 선택 | 확정 시점 영업시간 사본. NULL=확정 시점 미확인 |
| sourceAttribution | 문자열 | 필수 | 확정 시점 출처 표기(약관 요건) |
| snapshotAt | 시각(UTC) | 필수 · 불변 | 확정 시각 |

**불변식**
- **INV-SNAP1 (불변)**: 스냅샷은 생성 후 수정되지 않는다 — 원본 갱신이 스냅샷에 역류하지 않는다(대조를 위해 시점 고정). 재확정은 새 스냅샷 생성 [D13, G129 사본 원칙과 동형].
- **INV-SNAP2 (소실 독립)**: 원본 Poi.dataStatus=`LOST`여도 스냅샷은 파기되지 않고 마지막 사본으로 표시된다(허위 최신성 금지·확인 불가 배지 병기) [G8, G70].
- **INV-SNAP3 (영구 게이트)**: 영구 저장은 purpose가 사용자 확정 4종에 한하며, 탐색·추천 풀 데이터(StaySearchCache·후보 풀)는 스냅샷화 대상이 아니다(TTL 캐시로만 존재) [D13, §2.11].

### 2.3 SavedPlace — 계정 저장 POI (M7, Case A 온램프)

'장소 우선' 진입(US-E2-05)에서 사용자가 저장한 POI. 상한 없음(G8). 저장 실체는 PoiSnapshot(purpose=SAVED_PLACE) 참조이며, CP1의 '저장 POI' 계약 공급원이다(필수 방문지 시드 — 여행 투입 시 **사본 복제**는 U4 M6 소유 G129).

| 속성 | 타입 | 제약 | 의미·근거 |
|---|---|---|---|
| savedPlaceId | 식별자 | 필수 · 유니크 | — |
| accountId | 식별자(Account FK) | 필수 | 계정 귀속 — 기기 전환 유지 [D22] |
| snapshotRef | 식별자(PoiSnapshot 참조) | 필수 | 저장 시점 스냅샷(영구) [D13] |
| savedAt | 시각(UTC) | 필수 | 저장 일시 |
| staleFlag | 불리언(파생) | 필수 · 기본 false | 원본 Poi가 `LOST`면 true → '확인 불가' 배지, 필수 방문지 시드 투입 제외 신호 [G8] |

**불변식**
- **INV-SP1 (계정 유니크)**: `(accountId, canonicalPoiId)`는 유니크 — 동일 계정의 동일 장소 중복 저장 금지(재저장은 멱등).
- **INV-SP2 (소실 표기)**: staleFlag=true 항목은 목록에 '확인 불가' 배지로 표시되되 삭제되지 않는다(사용자 삭제만 파기) [G8].

### 2.4 StaySearchCache — 숙소 탐색 정적 콘텐츠 TTL 캐시 (M3)

숙소 탐색 결과 카드·상세의 **정적 콘텐츠**(명칭·위치·유형·편의시설·대표 가격대·사진 URL·체크인/아웃 시각) TTL 캐시. **정확 가격·재고는 캐싱하지 않으며**(OTA 약관), 라이브 호출은 사용자가 '가격 보기'/딥링크 시에만 발생한다 [D09, D13, G196].

| 속성 | 타입 | 제약 | 의미·근거 |
|---|---|---|---|
| stayRef | 식별자(StayIdentity 참조) | 필수 | 내부 숙소 ID 기준 캐시 키 |
| nameKo | 문자열 | 필수 | 숙소명(한국어 정본) |
| coord | 구조체 {lat, lng} | 선택 | 위치(지도 핀) — §2.11 매트릭스: 지도 API 유래는 TTL, TourAPI 유래는 정본 |
| stayType | 열거 | 선택 | 숙소 유형(TourAPI cat3 정본) |
| amenities | 목록<열거> | 선택 | 편의시설 — **채움률 검증된 항목만**(핵심 3종 주차·조식·와이파이부터 확대) [PRD 04-2] |
| priceBand | 열거 {저가, 중간, 고급, 럭셔리} | 선택 | **대표 가격대(정적 분류)** — 콘텐츠 메타로 정적 표시·캐싱 가능(정확 1박 요금이 아니므로 가격 캐싱 금지 대상 아님) [D09, PRD 04 註 ②] |
| photos | 목록<URL> | 선택 | 사진 URL(정적 콘텐츠 — 약관 갱신주기 준수 캐싱) |
| checkInOutTime | 구조체 {checkIn, checkOut} | 선택 | 체크인/아웃 **시각**(정적 콘텐츠, 날짜 아님) |
| fetchedAt | 시각(UTC) | 필수 | 수집 시각 |
| ttlExpiresAt | 시각(UTC) | 필수 | `= fetchedAt + 약관 허용 TTL` — 만료 후 재조회 강제(영구 저장 금지) [D13, ADR-0017] |

**불변식**
- **INV-CACHE1 (영구 금지)**: StaySearchCache에는 정본 개념이 없다 — 모든 행은 ttlExpiresAt를 가지며 만료분은 무효로 취급(재조회 없이 최신으로 표시 금지) [D13, ADR-0017, PBT U3-P3].
- **INV-CACHE2 (가격 비보관)**: 정확 1박 가격·재고는 어떤 필드에도 저장되지 않는다 — priceBand(정적 구간)만 허용, 정확 가격은 라이브 결과로만 존재하고 캐싱 0 [D09, PRD 04 註 ②, PBT U3-P3].
- **INV-CACHE3 (갱신 실패 표기)**: TTL 만료 후 재조회 실패 시 stale 데이터에 '갱신 실패' 표기를 병기한다(허위 최신성 금지) [ADR-0011, M7 실패 책임].

### 2.5 StayIdentity + StayIdentityMapping — 내부 숙소 ID·소스 매핑 (M4)

숙소 식별 통합. **내부 canonical 숙소 ID**가 정본이며, 소스별 외부 ID(OTA·지도)는 N:1로 매달린다(POI와 동일 패턴, D17). 동일 숙소에 연결된 복수 OTA를 묶는 기반(US-E3-05 OTA 선택 목록).

**StayIdentity 속성**

| 속성 | 타입 | 제약 | 의미·근거 |
|---|---|---|---|
| canonicalStayId | 식별자 | 필수 · 유니크 · 불변 | **내부 숙소 ID 정본 키** — SavedStay·WishlistItem·StaySearchCache가 참조 [D17] |
| nameKo | 문자열 | 필수 | 숙소명(한국어 정본) |
| coord | 구조체 {lat, lng} | 선택 | 대표 좌표(매칭 기준). NULL=핀 미확정(등록 시 SavedStay.coordConfirmed로 별도 관리) |
| matchConfidence | 수치 [0,1] | 필수 | 좌표+이름 유사도 매칭 신뢰도 |
| operatorOverride | 불리언 | 필수 · 기본 false | 운영자 보정 여부(자동 매칭 오류 수동 교정 여지) [D17] |

**StayIdentityMapping 속성 (StayIdentity 서브엔티티)**

| 속성 | 타입 | 제약 | 의미·근거 |
|---|---|---|---|
| canonicalStayId | 식별자(StayIdentity FK) | 필수 | 귀속 내부 ID |
| source | 열거 {KAKAO, NAVER, TOURAPI, OTA:{partnerCode}} | 필수 | 외부 소스 |
| externalId | 문자열 | 필수 | 소스별 외부 숙소 ID(place_id 등) |

**불변식**
- **INV-SID1 (N:1 라운드트립 — 하드 제약)**: `(source, externalId)`는 정확히 1개 canonicalStayId로 해석된다 — 외부 ID → 내부 ID는 전역·결정적(N:1). 라운드트립(외부 ID 해석 후 재해석 동일) 보장 [D17, U3 DoD, PBT U3-P2].
- **INV-SID2 (복합 유니크)**: `(source, externalId)` 조합은 전역 유일 — 하나의 외부 ID가 두 내부 ID에 매핑될 수 없다.
- **INV-SID3 (매칭 결정성·대칭성)**: 좌표(50m 근접)+명칭 유사도 매칭은 동일 입력에 동일 결과(결정적), 대칭(A가 B에 매칭 ⇔ B가 A에 매칭) — 운영자 보정은 자동 결과를 덮어쓰되 보정 자체도 결정적 [D17, PBT U3-P4].

### 2.6 SavedStay — 등록 숙소 (M4, 계정 레벨 풀)

사용자가 외부에서 예약했거나 직접 찾은 숙소를 '등록'한 **계정 레벨 풀** 항목. 여행 없이 등록 가능하며(D15), 여행 연결(BaseAssignment §2.7) 시 AI 일정 생성의 거점·기간 기준점이 된다(ADR-0002). 예약 상태는 추적하지 않고 사용자 입력 기록만 보관한다.

| 속성 | 타입 | 제약 | 의미·근거 |
|---|---|---|---|
| savedStayId | 식별자 | 필수 · 유니크 | — |
| accountId | 식별자(Account FK) | 필수 | **계정 레벨 풀 귀속** — 여행 없이 등록·유지 [D15] |
| stayIdentityRef | 식별자(StayIdentity 참조) | 필수 | 내부 숙소 ID [D17] |

| checkOut | 날짜 | 필수 | 체크아웃 날짜 — `checkOut > checkIn`(INV-STAY1) [PRD 04-6 예외] |

#### 2.6.1 등록 3경로별 좌표 확보 (registrationPath)

| 경로 | 좌표 확보 | coordConfirmed 초기값 | 폴백 |
|---|---|---|---|
| MAP_SEARCH | 지도/장소 검색 선택 → 자동 확보(M7) | true(단일 확정 선택 시) | 동명 다건 → 후보 선택 강제 [PRD 04-8 예외] |
| LINK_PARSE | URL 패턴 파싱(화이트리스트, fetch 없음) → 부분 프리필 | 대개 false(위치 특정 불가 시) | 추출 실패 → PIN_DROP 안내 [G31] |
| PIN_DROP | 지도 핀 직접 지정 | true(핀 확정 시) | — |
| OTA_RETURN | 복귀 핸드오프 카드 → 빠른 등록(숙소명·위치 자동 채움) | 매칭 성공 시 true | 미확정 시 지도 확인 |

#### 2.6.2 불변식

- **INV-STAY1 (날짜 순서 — 하드 제약)**: `checkOut > checkIn` ∧ 두 날짜 모두 존재. 위반 시 저장 차단·인라인 표시 — 등록 숙소 필수 필드·날짜 순서 검증은 D37 하드 제약, 머지 차단 테스트 [PRD 04-6 예외, U3 DoD, PBT U3-P6].
- **INV-STAY2 (좌표 확정 게이트)**: `coordConfirmed=false`인 SavedStay는 거점 자격이 없다 — 여행 연결(BaseAssignment) 및 일정 생성 기준점 투입은 "지도에서 위치 확인"으로 coordConfirmed=true가 된 후에만 [PRD 04-6·8 예외, CP1 시나리오 2].
- **INV-STAY3 (계정 풀 독립)**: SavedStay는 여행 연결 없이 존재 가능(계정 레벨 풀) — 여행 삭제가 SavedStay를 파기하지 않는다(BaseAssignment만 해제) [D15].
- **INV-STAY4 (예약 상태 부재)**: SavedStay는 예약 진행/확정 상태 필드를 갖지 않는다 — 사용자 입력 기록만(예약·결제는 외부 OTA) [PRD 04-9].
- **INV-STAY5 (선택 항목 명시값)**: otaName·reservationNo·amount가 NULL이면 목록 표시는 "미입력" 명시값(빈칸·입력 강요 금지) [PRD 04-9 예외].

### 2.7 BaseAssignment — 여행 거점 연결 (M4, CP1 공급 · 검증은 U4)

SavedStay(계정 풀)를 특정 여행의 거점으로 연결하는 조인. **데이터·API는 U3(M4) 소유**이나, 여행 컨텍스트가 필요한 **여행 내 거점 날짜 비중첩 검증·거점 UI는 U4(M6)가 완성**한다(명시적 제외 — unit-of-work U3 §제외). U3은 조인 스키마와 비중첩 판정 함수(순수 함수)를 공급한다.

| 속성 | 타입 | 제약 | 의미·근거 |
|---|---|---|---|
| assignmentId | 식별자 | 필수 · 유니크 | — |
| savedStayId | 식별자(SavedStay FK) | 필수 | 연결 대상 숙소(coordConfirmed=true 전제 INV-STAY2) |
| tripId | 식별자(Trip 참조) | 필수 | 연결 여행(엔티티 정본은 U4 M6) |
| dateRange | 구조체 {start, end} | 필수 | 거점 담당 날짜 구간(다박 연속 표현) |
| isSmartDefault | 불리언 | 필수 · 기본 false | 스마트 기본 거점(공백일=직전 숙소, 첫날 공백=여행지 중심 좌표 G41) — 비차단 안내 [PRD 05-6, G41] |
| active | 불리언 | 필수 · 기본 true | 거점 활성 여부(해제 시 false — 기존 일정 유지 G97) |

**불변식**
- **INV-BASE1 (여행 내 비중첩 — U4 소비)**: 같은 tripId에 연결된 active BaseAssignment들의 dateRange는 서로 겹치지 않는다. **검증 실행은 U4**(계정 풀의 미연결 숙소는 검증 대상 아님) — U3은 비중첩 판정 순수 함수와 `ConflictDetected(overlapDates)` 계약을 공급 [D15, PBT U3-P5, CP1].
- **INV-BASE2 (좌표 전제)**: BaseAssignment는 coordConfirmed=true인 SavedStay만 참조(INV-STAY2) — 미확정 좌표 거점 금지.
- **INV-BASE3 (해제 비파괴)**: unlink(active=false)는 기존 일정을 파기하지 않는다 — 리마인드 중단·재생성 유도 배지 이벤트만 발행(U8 소비) [G97].

- **INV-TREND2 (표시 게이트)**: `region`은 홈 대시보드 슬롯(U2)에 공급 — U3은 데이터만 생산, 노출은 U2 소비.

### 2.11 소스별 캐싱 허용/금지 매트릭스 (D13 정본)

> 지도 API 영구 캐싱 금지 약관(ADR-0017)과 후보 풀 적재 요구의 양립을 필드×소스로 명문화한다. **정본 판정**: 셀 값 = 해당 소스 유래 필드의 저장 정책. 법무 확인(P2·P4) 결과로 셀만 교체하면 되도록 데이터 모델에 컬럼 수준 반영. 판정 함수 `cachingPolicy(source, field) → {PERMANENT, TTL_CACHE, LIVE_ONLY, NOT_HELD}`는 총함수(전 셀 정의) — 서버 M7·M3가 공유.

| 필드 | TourAPI(공공) | 카카오/네이버(지도 API) | OTA(딥링크) | 사용자 확정 스냅샷 |
|---|---|---|---|---|

**매트릭스 불변식**
- **INV-MTX1 (지도 API 영구 금지)**: `cachingPolicy(KAKAO|NAVER, {명칭·좌표·영업시간·카테고리})` ≠ PERMANENT — 지도 API 유래 비식별자 필드의 영구 저장 0(TTL 캐시만) [ADR-0017, G193, PBT U3-P3].
- **INV-MTX2 (정확 가격 비보관)**: `cachingPolicy(*, 정확 1박 가격)` = LIVE_ONLY ∨ 미보유 — 어떤 소스의 정확 가격도 저장되지 않는다 [D09, PBT U3-P3].
- **INV-MTX3 (사용자 확정 영구화)**: 사용자 확정 스냅샷 열은 좌표·영업시간을 영구 저장(D13) — 지도 API 실시간 캐싱 금지의 정책적 예외 축(P2 법무 확인 전제) [G124, G70].
- **INV-MTX4 (리뷰 미보유)**: 리뷰·평점은 어떤 소스에서도 앱 내부에 보관·표시하지 않는다(원문·점수 모두 외부 OTA 딥링크 위임) [PRD 04-3].

### 3.1 A. 숙소 탐색

**BR-U3-01 여행지 단일 입력 탐색 — 날짜·인원 미수집**
- **조건**: 숙소 탐색 요청(`searchStays`) 시.
- **동작**: 여행지(또는 '내 주변')만 입력받아 탐색한다 — **체크인/체크아웃 날짜·인원은 탐색 단계에서 받지 않는다**. 여행지가 비어 있으면 탐색 버튼은 비활성 유지. 일정 생성에 필요한 체크인/아웃 날짜는 등록(BR-U3-10)에서 수집한다.
- **위반 시 처리**: 여행지 없는 탐색 요청은 클라이언트 게이트+서버 `ValidationFailed`로 거부(빈 여행지 조회 금지).
- **근거**: US-E3-01, D09, ADR-0002.

**BR-U3-02 탐색 데이터 소스 — TourAPI+지도, OTA는 딥링크만**
- **조건**: 탐색 결과 구성 시.
- **동작**: 결과는 TourAPI+지도 장소 검색(M7 경유)으로 구성한다. **국내 OTA 데이터 크롤링 금지** — 외부 OTA는 숙소명 기반 검색 딥링크로만 연결(부정경쟁 민사 리스크 회피). 소스별 카테고리 코드는 M7이 단일 표준 스키마로 정규화(ADR-0009).
- **위반 시 처리**: OTA 페이지·데이터 크롤링 경로는 구조적으로 존재하지 않음(딥링크 조립만) — 코드 리뷰 차단 항목.
- **근거**: US-E3-01, D09, PRD 04 데이터 모델 註.

**BR-U3-03 대표 가격대 정적 표시 — 정확 가격 일괄 조회·캐싱 금지**
- **조건**: 탐색 결과 카드·목록 표시 시.
- **동작**: 카드는 숙소명·위치·**대표 가격대(정적 구간·등급)**를 기본 표시한다(콘텐츠 메타 — 캐싱 가능, INV-CACHE2 예외). **결과 목록 전체에 대한 정확 1박 가격 일괄 라이브 호출을 하지 않으며, 정확 가격 값은 캐싱하지 않는다**(비용·지연·OTA 약관).
- **위반 시 처리**: 목록 렌더 시 정확 가격을 조회·저장하는 것은 결함(약관 위반급) — INV-CACHE2 방어.
- **근거**: US-E3-01, D09, PRD 04 註 ②, G196.

**BR-U3-04 라이브 가격 조회 — '가격 보기' 명시 액션 한정**
- **조건**: 사용자가 결과 카드·상세의 '가격 보기'를 탭할 때(`getLivePrice`).
- **동작**: 날짜·인원 바텀시트 입력을 받은 뒤 정확 가격을 **라이브 조회**한다. 입력값은 세션 내 재사용. 조회 결과는 표시만 하고 저장하지 않는다(캐싱 0).
- **위반 시 처리**: 라이브 조회 실패 → `UpstreamUnavailable` → "외부 OTA에서 확인" 딥링크 대체 제시(침묵 실패 금지).
- **근거**: US-E3-01, G33, D09, G196.

**BR-U3-05 '내 주변' 위치 권한 폴백**
- **조건**: '내 주변' 탐색 시도 시(위치 just-in-time 발화 1번째 지점 — U1 위치 동의 3층 소비 G182).
- **동작**: 위치 권한 허용이면 단말 좌표로 조회. **권한 거부면 등록 숙소 또는 여행지 중심 좌표로 대체 조회**한다(중단 없음). U1 `effectiveCapabilities`로 서버 위치 서비스 가부를 판정(L1∧L2 필요).
- **위반 시 처리**: 권한 거부만으로 탐색 흐름을 중단하는 것은 결함 — 폴백 좌표로 계속.
- **근거**: US-E3-01 예외, US-E1-04, G182(U1 INV-L1).

### 3.2 B. 필터·정렬

**BR-U3-06 숙소 유형 필터 전면 제공**
- **조건**: 필터 적용 시.
- **동작**: 숙소 유형(호텔·게스트하우스·펜션·모텔·호스텔·아파트·B&B·한옥 등) 필터를 전면 제공한다 — 유형은 가격과 분리된 정적 콘텐츠. 한국형 유형 분류는 **TourAPI cat3를 정본**으로 표준화한다.
- **위반 시 처리**: 소스별 유형 코드 불일치는 M7 표준화로 흡수 — 표준 밖 유형 노출은 결함.
- **근거**: US-E3-02, D09, ADR-0009.

**BR-U3-07 편의시설 필터 — 채움률 검증 항목만 노출**
- **조건**: 편의시설(amenities) 필터 구성 시.
- **동작**: TourAPI 필드 **채움률이 검증된 항목만** 노출한다(핵심 3종 주차·조식·와이파이부터 확대). 편의시설 표시는 **가격 비교 화면이 아닌 곳**에 배치(Booking 약관). 미확보 시 핵심 3종 폴백.
- **위반 시 처리**: 채움률 미검증 항목을 필터로 노출(대량 0건 유발)하는 것은 결함.
- **근거**: US-E3-02, PRD 04-2.

**BR-U3-08 가격순 정렬 미제공 — '대표 가격대순' 대체**
- **조건**: 정렬 옵션 구성 시.
- **동작**: 정확 가격에 의존하는 **가격순 정렬은 제공하지 않는다**(라이브 전수 호출 전제이므로). 대신 **'대표 가격대순'**(정적 구간 기준)으로 대체한다. 가격대 필터도 대표 가격대(구간) 기준으로 적용.
- **위반 시 처리**: 가격순 정렬 노출은 목록 라이브 전수 호출을 유발하는 결함(BR-U3-03 위반).
- **근거**: US-E3-02, D09, PRD 04-2.

**BR-U3-09 거리 필터·정렬 기준점 — 검색 여행지 중심 좌표 직선거리**
- **조건**: 거리 필터·정렬 적용 시.
- **동작**: **검색 여행지 중심 좌표를 기준점**으로 한 **직선거리**로 적용한다. **차량/도보 예상 소요시간은 어떤 화면에도 표시하지 않는다**(거리만) [D25].
- **위반 시 처리**: 소요시간 표기는 D25 위반 결함. 필터 적용 결과 0건이면 원인 필터를 표기하고 완화 제안(BR-U3-10 아님 — 0건 진단).
- **근거**: US-E3-02, G34, D25/Δ1.

### 3.3 C. 숙소 등록

**BR-U3-10 등록 필수 필드·날짜 순서 검증 (하드 제약)**
- **조건**: 숙소 등록(`registerStay`)·수정(`updateStay`) 저장 시.
- **동작**: 숙소(장소)·체크인/체크아웃 날짜·인원을 **필수**로 수집한다(예약번호·금액은 선택). `checkOut > checkIn` ∧ 두 날짜 모두 존재를 검증한다(INV-STAY1).
- **위반 시 처리**: `ValidationFailed` — 누락·역전 항목을 인라인으로 표시하고 저장 차단. **등록 숙소 필수 필드·날짜 순서(체크아웃>체크인) 검증은 D37 하드 제약, 머지 차단 테스트** [U3 DoD].
- **근거**: US-E3-06, D15, PRD 04-6 예외, PBT U3-P6.

**BR-U3-11 좌표 미확정 시 "지도에서 위치 확인" 강제**
- **조건**: 등록 시 좌표 확정 여부(coordConfirmed) 판정 시.
- **동작**: 좌표가 확정되지 않은 등록은 `CoordinateUnresolved`를 반환해 **"지도에서 위치 확인" 단계를 강제**한다. coordConfirmed=true가 되기 전에는 거점 자격·일정 생성 기준점 투입을 보류한다(INV-STAY2). 국내 좌표 범위 검증(G120).
- **위반 시 처리**: 좌표 미확정 숙소를 거점으로 연결·일정 생성에 투입하는 것은 결함 — CP1 소비(U4)에서도 동일 차단 [CP1 시나리오 2].
- **근거**: US-E3-06·08 예외, D15, G120.

**BR-U3-12 계정 레벨 풀 — 여행 없이 등록 가능**
- **조건**: 등록 저장·귀속 시.
- **동작**: 등록 숙소는 **계정 레벨 풀**에 저장한다 — 여행 없이 등록·유지 가능(D15). 여행 연결(BaseAssignment)은 별개 액션이며, 여행 삭제가 SavedStay를 파기하지 않는다(INV-STAY3).
- **위반 시 처리**: 등록을 특정 여행에 강제 귀속시키는 것은 결함(계정 풀 원칙 위반). 여행 밖 등록 시 연결은 U4 화면에서 명시 선택(G150 대안 C).
- **근거**: US-E3-06·09, D15, G128/G150.

**BR-U3-13 등록 3경로 단일 폼 — 자동 채움**
- **조건**: 직접 등록 진입 시(registrationPath).
- **동작**: 세 경로를 단일 등록 폼으로 수렴한다 — (a) 지도/장소 검색 자동 채움(M7), (b) OTA URL 붙여넣기 파싱 프리필(BR-U3-25), (c) 지도 핀 직접 지정. 검색·복귀 핸드오프 시작 시 숙소명·위치·가격대가 자동 채워지고 날짜·인원만 입력하면 완료.
- **위반 시 처리**: 경로별 상이한 폼·중복 정의는 결함(등록 폼은 US-E3-08 정본 — 여행 화면은 진입점만).
- **근거**: US-E3-06·08, G31.

**BR-U3-14 등록 후 [AI 일정 생성하기] 온램프·수정·취소**
- **조건**: 등록 완료 직후 및 사후 관리 시.
- **동작**: 등록 완료 시 해당 숙소를 거점으로 한 **[AI 일정 생성하기] 진입점을 즉시 노출**한다. 등록한 숙소의 날짜·인원·예약번호·금액은 나중에 수정 가능, 등록 취소 가능. 여행 연결 숙소의 삭제·기간 변경은 영향 블록 나열 후 **차단형 확인**(G39), 날짜 변경 시 거점 비중첩 재검증(D15).
- **위반 시 처리**: 연결된 숙소를 확인 없이 삭제·변경하는 것은 결함(파괴적 변경 방어 G39).
- **근거**: US-E3-06, US-E3-09, G39, D15.

### 3.4 D. 거점 연결 준비 (CP1 공급 — 여행 내 비중첩은 U4 소비)

**BR-U3-15 여행 거점 연결 — 일자별 다중 거점**
- **조건**: 숙소를 여행 거점으로 연결(`linkToTrip`) 시.
- **동작**: coordConfirmed=true인 SavedStay를 특정 여행의 dateRange 거점으로 연결한다(BaseAssignment). 하나의 여행에 일자별 서로 다른 거점 등록 지원(다도시). 체크인/아웃이 여행 기간을 이탈하면 `PeriodMismatch(proposal)`로 기간 확장을 경고 반환(자동 변경 없음).
- **위반 시 처리**: 좌표 미확정 숙소 연결은 차단(INV-BASE2). 기간 자동 변경은 금지 — 사용자 확인.
- **근거**: US-E3-07, D15, G41.

**BR-U3-16 여행 내 거점 날짜 비중첩 — 판정 공급(검증 실행은 U4)**
- **조건**: 같은 여행에 연결된 거점 구간의 겹침 판정 시.
- **동작**: 같은 tripId에 연결된 active 거점끼리 **날짜 비중첩을 판정**한다(INV-BASE1). **계정 풀의 미연결 숙소는 검증 대상 아님**. 겹치면 `ConflictDetected(overlapDates)`로 구간 조정을 요구. **U3은 비중첩 판정 순수 함수와 계약을 공급하고, 여행 컨텍스트 검증·UI 완성은 U4가 소비**(unit-of-work U3 §제외).
- **위반 시 처리**: 겹침을 허용해 거점이 둘인 상태로 일정 생성 진입하는 것은 결함(U4 소비 측에서도 방어) [CP1].
- **근거**: US-E3-07, D15, PBT U3-P5, CP1.

**BR-U3-17 스마트 기본 거점 — 비차단 안내**
- **조건**: 거점 타임라인(`getBaseTimeline`) 산출 시.
- **동작**: 다중 숙소는 체크인 순 전환, **공백일은 직전 숙소, 첫날 공백은 여행지 중심 좌표**를 기본 거점으로 채운다(isSmartDefault=true). 스마트 기본 거점은 **비차단 안내**(사후 수정 가능 배지).
- **위반 시 처리**: 스마트 기본 거점으로 사용자 흐름을 차단하는 것은 결함(비차단 원칙 PRD 05-6).
- **근거**: US-E3-07, G41, G50, D15.

### 3.5 E. 숙소 ID 통합

**BR-U3-18 내부 숙소 ID 정본 + 소스별 외부 ID N:1 매핑 (하드 제약)**
- **조건**: 숙소 식별 해석(`resolveStayIdentity`)·등록 시.
- **동작**: **내부 canonical 숙소 ID를 정본**으로 하고 소스별 외부 ID(OTA·지도)를 N:1로 매핑한다(INV-SID1·2). `(source, externalId)`는 정확히 1개 내부 ID로 결정적 해석(라운드트립 보장).
- **위반 시 처리**: 하나의 외부 ID가 두 내부 ID에 매핑되는 것은 하드 결함 — INV-SID2 저장 계층 거부, PBT U3-P2.
- **근거**: US-E3-05, D17, PBT U3-P2, U3 DoD.

**BR-U3-19 좌표+이름 유사도 자동 매칭 — 결정성·대칭성·보수 임계**
- **조건**: 신규 소스 ID의 canonical 귀속 판정 시.
- **동작**: 좌표(50m 근접)+명칭 유사도로 자동 매칭한다 — **결정적**(동일 입력 동일 결과)·**대칭적**(A↔B). 오매칭(동명·근접 병합 오류) 위험에 대비해 **매칭 임계값은 보수 설정**하고 운영자 보정 여지를 둔다(operatorOverride).
- **위반 시 처리**: 매칭이 비결정적이거나 비대칭이면 결함(PBT U3-P4). 임계 미달은 신규 canonical 생성(과병합보다 과분리를 택함 — 보수적).
- **근거**: US-E3-05, D17, G133/G148, PBT U3-P4.

**BR-U3-20 동일 숙소 복수 OTA 묶기**
- **조건**: `getOtaLinkOptions` — OTA 선택 목록 구성 시.
- **동작**: **내부 숙소 ID 기준으로** 동일 숙소에 연결된 OTA를 묶어 OTA명 목록을 반환한다 — 사용자가 어느 링크로 나갈지 선택.
- **위반 시 처리**: 외부 ID 기준 묶기(동일 숙소 분산 표시)는 결함(내부 ID 정본 원칙).
- **근거**: US-E3-05, D17.

### 3.6 F. OTA 딥링크·수수료 고지

**BR-U3-21 숙소명 검색 딥링크만 — 자체 예약·결제 미제공**
- **조건**: OTA 예약 이동(`buildDeeplink`) 시.
- **동작**: **숙소명 기반 OTA 검색 딥링크**만 생성한다(파트너별 URL 템플릿 조립, 실호출 없음). 앱은 자체 예약·객실 선택·예약 확정·인앱 결제를 **제공하지 않으며** 모든 결제 흐름을 외부 OTA로 위임한다.
- **위반 시 처리**: 앱 내 예약·결제 경로 도입은 범위 위반. 딥링크 조립 외 OTA API 실호출은 결함.
- **근거**: US-E3-05, D09, ADR-0003·0012.

**BR-U3-22 제휴 수수료 고지 — 외부 이동 직전 표시**
- **조건**: 외부 OTA 이동 직전.
- **동작**: **"예약·결제는 외부 OTA에서 진행되며 TripPilot은 제휴(어필리에이트) 수수료를 받을 수 있다"**는 제휴 링크 안내·수수료 고지를 표시한다(추가 비용 없음 문안 포함). '다시 보지 않기'(setDisclosureHidden) 설정 연동 — 숨김 설정 시 showDisclosure=false.
- **위반 시 처리**: 고지 없는 외부 이동은 광고·제휴 표시 규제 위반(컴플라이언스 결함).
- **근거**: US-E3-05, PRD 04-5, PRD 12-12.

**BR-U3-23 리뷰·평점 미표시 — 외부 OTA 위임**
- **조건**: 숙소 상세(`getStayDetail`) 구성 시.
- **동작**: 상세는 위치(핀)·사진·대표 가격대·편의시설·체크인/아웃 **시각** 등 정적 콘텐츠만 구성한다. **리뷰·평점은 앱 내부에 표시하지 않고**(원문·점수 모두) "리뷰·평점 보기 → 외부 OTA" 딥링크로 위임한다. 데이터 누락 필드는 빈칸이 아닌 **"미확인"** 명시값.
- **위반 시 처리**: 리뷰·평점 필드를 앱 내부 화면에 표시하는 것은 결함(INV-MTX4). 빈칸 방치는 결함(미확인 표기 강제).
- **근거**: US-E3-03, PRD 04-3, G199, INV-MTX4.

**BR-U3-24 딥링크 실패 우회 경로**
- **조건**: OTA 링크가 깨졌거나(404·만료) 응답하지 않을 때.
- **동작**: "지금 이 OTA로 연결할 수 없어요"를 표시하고, 가능하면 **같은 숙소의 다른 OTA 링크** 또는 **숙소 장소 검색**으로 우회 경로를 제시한다.
- **위반 시 처리**: 링크 실패 시 침묵·막다른 흐름은 결함(ADR-0011 침묵 실패 금지).
- **근거**: US-E3-05 예외, D09, ADR-0011.

### 3.7 G. URL 파싱 (화이트리스트)

**BR-U3-25 OTA 링크 붙여넣기 — 화이트리스트 패턴 파싱만 (fetch 없음)**
- **조건**: OTA 예약 링크 붙여넣기(`parseOtaLink`) 시.
- **동작**: **지원 도메인 화이트리스트 내 URL 패턴 파싱만으로**(페이지 fetch 없음) 숙소명·위치를 추출해 등록 폼 프리필을 만든다. 화이트리스트 밖 URL·패턴 불일치는 전부 거부한다.
- **위반 시 처리**: 화이트리스트 밖 URL 파싱 시도·페이지 fetch는 결함(SSRF·약관 리스크 방어) — 전부 `LinkParseFailed`. **화이트리스트 밖 URL 전부 거부·파싱 실패 시 폴백 경로는 PBT U3-P7 대상** [U3 DoD].
- **근거**: US-E3-08, G31, SECURITY-05, PBT U3-P7.

**BR-U3-26 파싱 실패 시 수동 핀 지정 폴백**
- **조건**: URL 추출 실패·장소 미검색 시.
- **동작**: "위치를 찾지 못했어요. 지도에서 직접 지정해 주세요"로 안내하고 **수동 핀 지정 흐름(PIN_DROP)**으로 넘긴다. 지도/장소 검색 API 무응답도 동일 폴백(침묵 실패 금지).
- **위반 시 처리**: 파싱 실패로 등록 흐름이 막히는 것은 결함(항상 핀 지정 폴백 존재).
- **근거**: US-E3-08 예외, G31, ADR-0011.

### 3.8 H. 복귀 핸드오프 카드

**BR-U3-27 복귀 핸드오프 카드 — 24h·1회 노출**
- **조건**: OTA 딥링크 이탈 후 앱 포그라운드 복귀 시(`getReturnHandoffCard`).
- **동작**: 딥링크 이탈 후 **24시간 내 첫 복귀에 최근 이탈 숙소 1건에 대해 1회** "방금 본 [숙소명], 예약하셨나요? → 거점으로 등록하기" 카드를 노출한다(handoffShownAt 기록). 카드는 빠른 등록(OTA_RETURN 경로, BR-U3-13)으로 연결.
- **위반 시 처리**: 24h 경과분·복수 숙소 동시 노출은 결함. 노출은 정확히 1회(INV-CLICK2, PBT U3-P8).
- **근거**: US-E3-05, G32.

**BR-U3-28 카드 무시 시 재노출 억제 — 비강제**
- **조건**: 사용자가 핸드오프 카드를 무시(디스미스)할 때(`dismissHandoffCard`).
- **동작**: handoffDismissedAt를 기록해 **재노출을 억제**한다. 사용자는 카드를 무시하고 계속 탐색할 수 있다(강제 아님) — 이후 동일 이탈 건에 `null` 반환.
- **위반 시 처리**: 무시 후 재노출은 결함. 등록 강제는 결함(비강제 원칙).
- **근거**: US-E3-05, G32.

### 3.9 I. 위시리스트

**BR-U3-29 위시 저장 — 계정 귀속·기기 전환 유지**
- **조건**: 숙소 저장(`addToWishlist`) 시.
- **동작**: 저장은 **로그인 사용자 계정에 귀속**되어 기기를 바꿔도 유지된다(로그인 필수 전제 D22). `(accountId, stayRef)` 유니크(재저장 멱등). 메모는 SECURITY-05 검증.
- **위반 시 처리**: 비로그인 저장·기기 로컬 저장은 결함(계정 귀속 원칙). 소유권 검증(SECURITY-08).
- **근거**: US-E3-04, D22.

**BR-U3-30 위시 카드 표시·가격 변동 고지**
- **조건**: 위시 목록(`listWishlist`) 조회 시.
- **동작**: 각 항목을 숙소명·대표 사진·가격대·위치 카드로 표시하고, **"외부 OTA 기준, 변동 가능"**을 병기한다(INV-WISH2). 원본 소실 항목은 저장 당시 캐시로 표시+"최신 정보 확인 불가" 플래그(staleFlag).
- **위반 시 처리**: 저장 시점 가격을 확정 가격으로 표시하는 것은 결함. 빈 카드 표시는 결함.
- **근거**: US-E3-04 예외, G103.

**BR-U3-31 위시·저장 POI·등록 숙소 별도 목록**
- **조건**: 목록 관리 전반.
- **동작**: 숙소 위시리스트와 저장 POI(SavedPlace)는 **별도 목록 2개**로 관리한다(G129 원칙). 위시·등록 통합 목록(`listMyStays`)은 출처 라벨('외부 OTA 예약 등록'/'앱 내 저장')로 구분하고, 저장 항목엔 '등록하기' 액션을 둔다.
- **위반 시 처리**: 위시와 저장 POI의 목록 혼합·상호 오염은 결함.
- **근거**: US-E3-04, US-E3-09, G103, G129.

### 3.10 J. POI 정본·캐싱 정책

**BR-U3-32 canonical POI ID 무결성 (하드 제약)**
- **조건**: POI 수집·정규화·식별 해석(`resolvePoiIdentity`) 시.
- **동작**: 내부 canonical POI ID를 정본으로 하고 소스별 place_id를 N:1 매핑한다. **동일 실세계 장소(좌표 50m 근접+명칭 유사도)는 정확히 1개 canonical**(INV-POI1). 한국어명 정본+영문 alias(G148), 한국어 필드 우선(G123).
- **위반 시 처리**: 동일 장소가 복수 canonical로 분열되면 U5 POI 그라운딩 하드 제약의 전제가 무너짐 — **D37 하드 제약, 머지 차단 테스트** [U3 DoD, PBT U3-P1].
- **근거**: US-E3-01·03, D13, G133/G148, PBT U3-P1.

**BR-U3-33 소스별 캐싱 정책 게이트 (D13)**
- **조건**: POI·숙소 필드의 저장·캐싱 판정 시.
- **동작**: §2.11 소스별 캐싱 매트릭스를 정본으로 판정한다(`cachingPolicy(source, field)` 총함수) — **지도 API(카카오·네이버) 유래 비식별자 필드(명칭·좌표·영업시간·카테고리)는 영구 저장 0**(TTL 캐시만, 만료 후 재조회 강제). TourAPI(공공)·사용자 확정 스냅샷만 영구 저장 허용.
- **위반 시 처리**: 지도 API 유래 필드의 영구 저장은 약관 위반급 결함 — **금지 소스 영구저장 0 불변식은 PBT U3-P3** [ADR-0017, G193].
- **근거**: US-E3-01·02, D13, G70/G105/G124/G169/G193, ADR-0017, PBT U3-P3.

**BR-U3-34 사용자 확정 데이터 영구 스냅샷 (D13)**
- **조건**: 사용자 확정 데이터(등록 숙소·저장 POI·방문 체크·필수 방문지) 확정 시(`snapshotUserConfirmed`).
- **동작**: 확정 시점 사본(이름·좌표·카테고리·영업시간·출처)을 **'사용자 입력 데이터'로 영구 저장**한다(PoiSnapshot, INV-SNAP1·3). 지도 API 실시간 캐싱 금지의 정책적 예외 축(P2 법무 확인 전제). 원본 갱신은 스냅샷에 역류하지 않는다(시점 고정).
- **위반 시 처리**: 스냅샷 사후 변경·원본 역류는 결함(대조 무결성 훼손). 탐색·추천 풀을 스냅샷화하는 것은 결함(INV-SNAP3).
- **근거**: US-E3-06(등록 스냅샷), D13, G70, G124, P2.

**BR-U3-35 TTL 캐시 만료 강제**
- **조건**: StaySearchCache·POI 탐색 풀 조회 시.
- **동작**: 모든 캐시 행은 ttlExpiresAt를 가지며, 만료분은 무효로 취급해 **재조회를 강제**한다(INV-CACHE1). 재조회 실패 시 stale 데이터에 '갱신 실패' 표기(허위 최신성 금지). 정적 콘텐츠 캐시 일 1회 갱신(G196).
- **위반 시 처리**: 만료 캐시를 최신으로 표시하는 것은 결함 — **만료 후 재조회 강제는 PBT U3-P3** [U3 DoD].
- **근거**: US-E3-01·03·11, D13, G196, ADR-0017, PBT U3-P3.

**BR-U3-36 탐색 부분 실패·전체 실패 처리**
- **조건**: 탐색 소스(TourAPI·지도) 응답 시.
- **동작**: **일부 소스 실패**면 가용 결과로 목록을 구성하고 "일부 정보가 빠졌을 수 있어요"(degraded)를 표기한다(침묵 실패 금지). **전체 실패**면 "지금 숙소 정보를 불러올 수 없어요"+재시도, 원하면 숙소 직접 등록(US-E3-08)으로 우회. **필터 0건**은 원인 필터 표기+완화 제안, **데이터 부족 지역**은 "직접 등록하면 일정을 만들어 드려요" 안내+등록 경로 연결.
- **위반 시 처리**: 부분 실패 시 전체 실패로 처리하거나 침묵하는 것은 결함. 1차 소스(카카오) 실패 → 2차(네이버) 폴백(D08).
- **근거**: US-E3-10, US-E3-11, D09, D08, ADR-0011.

### 3.11 K. 소실 POI 처리

**BR-U3-37 소실 POI '확인 불가' 배지·시드 투입 제외**
- **조건**: 정본 동기화 배치에서 POI가 소스에서 확인 불가일 때(dataStatus=`LOST`).
- **동작**: '확인 불가' 배지를 유지하고, **필수 방문지 시드 투입에서 제외 안내**한다(G8). 이미 저장된 PoiSnapshot(사용자 확정 사본)은 마지막 스냅샷으로 잔존 표시(INV-SNAP2 — 허위 최신성 금지·배지 병기).
- **위반 시 처리**: 소실 POI를 후보 풀·필수 방문지 시드에 투입하는 것은 결함(U5 그라운딩 오염). 스냅샷 파기는 결함(사용자 데이터 보존).
- **근거**: US-E3(횡단), G8, PBT U3-P9, CP1 시나리오 3.

**BR-U3-38 소실 처리 멱등성**
- **조건**: 정본 동기화 배치 재실행 시.
- **동작**: 소실 판정·시드 제외 신호는 **멱등**하다 — 동일 소실 상태에 배치를 재실행해도 상태·스냅샷·제외 신호가 중복 변경되지 않는다.
- **위반 시 처리**: 배치 재실행이 스냅샷을 중복 생성·상태를 진동시키는 것은 결함 — PBT U3-P9(멱등).
- **근거**: G8, PBT U3-P9.

### 3.12 L. 포스트백 자동등록 이연

**BR-U3-39 포스트백 1탭 자동 등록 후속 이연 — 수동 경로만**
- **조건**: 예약 확인→자동 등록 경로 설계 시.
- **동작**: **포스트백 기반 1탭 자동 등록은 1차 제외**(후속 이연) — 1차는 **복귀 핸드오프 카드(BR-U3-27)+수동 빠른 등록만** 제공한다. PostbackRecord는 **집계용 스키마 여지만** 확보하고 수신 로직은 구현하지 않는다(G29/G108).
- **위반 시 처리**: 1차에서 포스트백 자동 등록을 활성화하는 것은 범위 위반. 수신 엔드포인트 실동작은 결함(스키마 여지만).
- **근거**: US-E3-06, D09, G29/G108.

## 4. 비즈니스 로직 · 처리 흐름 · 테스트 속성(PBT)

핵심 프로세스 플로우 5종과 Testable Properties(PBT-01) 식별표를 정본으로 정의한다. 플로우 ID `FLOW-x`, 단계 `Sx`, 분기 `Bx`, 예외 `Ex`. 메서드 계약은 component-methods M3·M4·M5·M7, 엔티티·불변식은 §2, 규칙은 §3(BR-U3-xx)을 참조한다. 기술 중립 — 전송·저장·캐시 구현체는 §6·§7이 소유한다.

### 4.1 FLOW-1 숙소 탐색 → 필터 → 상세 → 위시

**진입**: 탐색 랜딩 '숙소' 진입 → 여행지(또는 '내 주변') 입력 → `M3.searchStays(query, page)`
**관련**: US-E3-01~04, US-E3-10, US-E3-11 · BR-U3-01~09, 23, 29~31 · D09, D13, G33, G34, D25

```text
S1 여행지 입력 검증 [BR-U3-01]
   ├─ E1 여행지 공백 → 탐색 버튼 비활성 유지(서버 호출 없음)
   └─ B1 '내 주변' 선택 → 위치 권한 판정(U1 effectiveCapabilities) [BR-U3-05]
        └─ 권한 거부 → 등록 숙소/여행지 중심 좌표로 대체 조회(중단 없음)
S2 탐색 실행 — TourAPI+지도(M7 경유) 조회, "숙소를 모으는 중" 로딩 [BR-U3-02]
   ├─ E2 일부 소스 실패 → 가용 결과+degraded "일부 정보가 빠졌을 수 있어요" [BR-U3-36]
   ├─ E3 1차(카카오) 실패 → 2차(네이버) 폴백 [D08]
   └─ E4 전체 실패 → "지금 숙소 정보를 불러올 수 없어요"+재시도 / 직접 등록(FLOW-2) 우회
S3 결과 카드 구성 — 숙소명·위치·대표 가격대(정적) [BR-U3-03]
   (정확 가격 일괄 조회·캐싱 없음 — INV-CACHE2)
S4 필터·정렬 적용 [BR-U3-06~09]
   ├─ 유형 필터(TourAPI cat3 정본, 전면) · 편의시설(채움률 검증 항목만)
   ├─ 대표 가격대순(가격순 정렬 없음) · 거리(여행지 중심 직선거리 — 소요시간 미표시 D25)
   ├─ E5 필터 0건 → 원인 필터 표기+완화 제안(탭 시 조정 재검색) [BR-U3-36]
   └─ E6 결과 0건/데이터 부족 지역 → "직접 등록하면 일정을 만들어 드려요"+등록 경로 [US-E3-10]
S5 페이지네이션/무한스크롤 — N건 이상 추가 노출
S6 상세 진입 — M3.getStayDetail(정적 콘텐츠: 위치·사진·가격대·편의시설·체크인/아웃 시각) [BR-U3-23]
   ├─ 리뷰·평점 미표시 → "외부 OTA에서 확인" 딥링크 [INV-MTX4]
   └─ 누락 필드 "미확인" 명시값(빈칸 금지)
S7 '가격 보기' 탭(선택) → 날짜·인원 바텀시트 → M3.getLivePrice(라이브·비캐싱) [BR-U3-04]
   └─ E7 UpstreamUnavailable → "외부 OTA에서 확인" 딥링크 대체
S8 위시 저장 → M3.addToWishlist(계정 귀속·기기 전환 유지) [BR-U3-29]
   └─ 위시 카드 "외부 OTA 기준, 변동 가능" 병기, 메모 추가/삭제 [BR-U3-30]
```

**사후 조건**: 목록·상세 어디에도 정확 1박 가격이 저장(캐싱)되지 않는다(INV-CACHE2). 소요시간 표기 0(D25). 위시는 계정 귀속으로 기기 전환에 무손실.

### 4.2 FLOW-2 숙소 등록 3경로 (지도 검색 · 링크 파싱 · 핀 지정)

**진입**: 탐색 결과·저장 목록 '[예약했어요/등록]' 또는 직접 등록 진입 → `M4.registerStay(input)`
**관련**: US-E3-06, US-E3-08 · BR-U3-10~14, 25~26 · D09, D13, D15, G31, G120

```text
S1 등록 경로 선택(단일 폼 수렴) [BR-U3-13]
   ├─ 경로 a: 지도/장소 검색(M7) → 숙소 선택
   │   └─ E1 동명·유사 다건 → 후보+핀 목록 제시, 사용자 정확 1곳 선택(좌표 확정)
   ├─ 경로 b: OTA URL 붙여넣기 → M4.parseOtaLink [BR-U3-25]
   │   ├─ 화이트리스트 내 패턴 파싱만(fetch 없음) → 숙소명·위치 프리필
   │   └─ E2 화이트리스트 밖/추출 실패 → LinkParseFailed → 경로 c 폴백 [BR-U3-26]
   └─ 경로 c: 지도 핀 직접 지정 → 좌표 확정(coordConfirmed=true)
S2 폼 자동 채움 — 숙소명·위치·가격대(경로 a/b 성공 시), 사용자는 날짜·인원 입력
S3 저장 검증 [BR-U3-10]
   ├─ 필수: 숙소·체크인/아웃·인원 / 선택: 예약번호·금액
   ├─ E3 checkOut ≤ checkIn 또는 날짜 누락 → ValidationFailed 인라인(저장 차단) [INV-STAY1 하드 제약]
   ├─ E4 coordConfirmed=false → CoordinateUnresolved → "지도에서 위치 확인" 강제 [BR-U3-11, INV-STAY2]
   └─ E5 국내 좌표 범위 이탈 → 차단+사유 [G120]
S4 canonical 숙소 ID 해석 — M4.resolveStayIdentity (FLOW-5) [BR-U3-18]
S5 스냅샷 영구 저장 — M7.snapshotUserConfirmed(REGISTERED_STAY) [BR-U3-34, INV-SNAP1]
S6 SavedStay 생성(계정 레벨 풀 — 여행 없이 등록 가능) [BR-U3-12, INV-STAY3]
S7 등록 완료 → [AI 일정 생성하기] 온램프 즉시 노출 [BR-U3-14]
S8 (사후) 수정·취소 — 연결 숙소는 영향 블록 차단형 확인(G39), 날짜 변경 시 비중첩 재검증(FLOW-3)
```

**가드**: coordConfirmed=false인 SavedStay는 거점 자격·일정 생성 기준점 투입 불가(INV-STAY2) — CP1 소비(U4)에서도 동일 차단. 지도 검색 API 무응답 시에도 항상 핀 지정 폴백 존재(침묵 실패 금지).

### 4.3 FLOW-3 OTA 딥링크 이동 → 복귀 핸드오프

**진입**: 상세·저장 화면 [외부 OTA에서 예약하기] → `M5.getOtaLinkOptions` → `M5.buildDeeplink`
**관련**: US-E3-05 · BR-U3-20~24, 27~28 · D09, D17, G32

```text
S1 OTA 선택 목록 — 내부 숙소 ID 기준 동일 숙소 OTA 묶기 [BR-U3-20]
S2 이동 직전 제휴 수수료 고지 표시 [BR-U3-22]
   └─ '다시 보지 않기' 설정 시 showDisclosure=false (setDisclosureHidden)
S3 딥링크 생성·이동 — 숙소명 검색 딥링크(외부 브라우저/인앱 웹뷰) [BR-U3-21]
   └─ E1 링크 404·만료·무응답 → LinkUnavailable → 다른 OTA 링크/장소 검색 우회 [BR-U3-24]
S4 아웃바운드 클릭 기록 — M5.recordOutboundClick(내부 집계·사용자/LLM 비노출) [INV-CLICK1]
   └─ 기록 실패는 흐름 비차단(best-effort, INV-CLICK3)
S5 【앱 포그라운드 복귀】 M5.getReturnHandoffCard [BR-U3-27]
   ├─ B1 이탈 후 24h 내 첫 복귀 → 최근 이탈 숙소 1건 카드 1회 노출(handoffShownAt 기록)
   │     └─ 카드 탭 → 빠른 등록(OTA_RETURN 경로 — FLOW-2 S2로 자동 채움 진입)
   ├─ B2 이미 노출/무시 이력(handoffDismissedAt) → null(재노출 억제) [BR-U3-28]
   └─ B3 24h 경과 → null
S6 카드 무시 → M5.dismissHandoffCard(재노출 억제) — 계속 탐색(비강제) [BR-U3-28]
```

**사후 조건**: 핸드오프 카드는 이탈 건당 정확히 1회 노출(INV-CLICK2). 아웃바운드 클릭은 사용자·LLM 컨텍스트에 구조적 비노출(SECURITY-11). 앱은 예약 확정 상태를 추적하지 않는다.

### 4.4 FLOW-4 POI 정본 파이프라인 (수집 → 정규화 → canonical ID → 캐싱)

**진입**: (배치) POI 정본 동기화 일 1회 / (온디맨드) 탐색·검색·후보 풀 요청 시 실시간 조회
**관련**: US-E3-01~03(간접), 횡단 · BR-U3-02, 32~38 · D08, D13, G8, G115, G123, G133/G148, G192, ADR-0009·0017

```text
S1 수집 — 카카오 장소 검색·지오코딩(1차)+TourAPI, 실패 시 네이버(2차) [D08]
S2 정규화 — 소스별 카테고리 코드 → 단일 표준 스키마, 한국어명 정본+영문 alias [ADR-0009, G123/G148]
S3 canonical POI ID 매핑 — M7.resolvePoiIdentity [BR-U3-32]
   ├─ 좌표 50m 근접 + 명칭 유사도 → 기존 canonical 귀속(N:1 place_id) [INV-POI1]

   ├─ TourAPI(공공)·식별자 → 영구/정본 저장
   ├─ 지도 API 비식별자(명칭·좌표·영업시간·카테고리) → TTL 캐시만(영구 금지) [INV-MTX1]

   ├─ 좌표·영업시간 확보 → ACTIVE / 미확인 → UNVERIFIED(사용자 확인 후보 분리) [INV-POI3]
   └─ 동기화 확인 불가 → LOST → '확인 불가' 배지, 후보 풀·시드 투입 제외 [BR-U3-37, G8]

```

**소실 처리 멱등(S5·S8)**: 정본 동기화 재실행이 LOST 상태·스냅샷·시드 제외 신호를 중복 변경하지 않는다(BR-U3-38, PBT U3-P9).

- 리뷰·평점 UI는 존재하지 않는다 — 외부 OTA 딥링크로만 위임(INV-MTX4).

### 5.3 탐색 화면군

#### 5.3.1 StaySearchScreen
- **props**: 없음(라우트 루트) · **state**: 여행지 입력값, 검색 중 플래그, 결과 페이지, degraded 소스 목록, 필터 상태.

### 5.7 shared/map — 카카오 지도 SDK 브리지 (U3 신규)

- **책임**: 카카오 지도 SDK 네이티브 모듈의 **재사용 브리지** — 지도 표시·핀·현위치·핀 드래그 좌표 확정. **구현체(Expo config plugin·네이티브 모듈)는 NFR/Infrastructure Design 소유**, 본 절은 브리지 인터페이스와 화면 계약만 규정한다.
- **props**: `KakaoMapBridge{ center: Coord, pins: List<Pin>, onPinDrop?, showMyLocation?: Boolean }` · **state**: 지도 준비 상태, 현위치 가용성(U1 `effectiveCapabilities` 질의 — 서버 위치 서비스=L1∧L2).
- **상호작용·폴백**:
  - `MapView`: 위치 핀 표시(StayDetailScreen), 현위치는 권한 허용 시에만(거부 시 여행지 중심 좌표 폴백).
  - `PinDropController`: 등록 핀 지정(coordConfirmed 확정) — MapSearchPicker 동명 다건 후보의 핀 선택 공용.
  - **SDK 네이티브 모듈 불안정 폴백**: 지도 렌더 실패 시 좌표·주소 텍스트 폴백(핀 지정은 좌표 입력 폴백) — 침묵 실패 금지(ADR-0011, U3 리스크 완화 "지도 표시 범위 축소 폴백").
- **data-testid**: `placestay-map-view`, `placestay-map-pindrop`, `placestay-map-mylocation`.
- **사용 서버 능력**: `M7.searchPoi`, `M7.getPoi`(지오코딩·POI 좌표), U1 `getLocationConsentState`(권한 상태).

## 6. 비기능 요구(NFR) · 설계 패턴 · 기술 스택

시스템 전역 NFR 정본은 requirements.md §6이며, 본 절은 그 정본을 **U3 관점으로 전개**한다. U3는 **외부 의존 최다 유닛(외부 연동 4 + 지도 SDK)**이므로 인프라·플랫폼 결정은 전역 정본을 상속(재정의 금지)하고, 유닛 증분(외부 어댑터 복원력·POI 하이브리드 캐싱·커버리지 게이트)만 확정한다. 크로스커팅 기준은 [NFR 기준](../nfr.md)을 참조한다.

### 6.0 U3 NFR 프로필 — "데이터 척추 · 외부 의존 최다"

U3 NFR 본질은 **제어할 수 없는 외부 데이터 소스 위에서 사용자 경험 품질과 적법성을 지키는 것**이다:

1. **복원력(핵심)** — 외부 API 4종(카카오·네이버·TourAPI·OTA 아웃바운드)의 가변 가용성이 탐색·등록을 마비시키지 않게 한다(RESILIENCY-10, M3·M5 = Medium §6.3 — 폴백 필수).
2. **캐싱·적법성** — D13 하이브리드 저장(TourAPI 정본 영구 + 지도 API TTL·영구 캐싱 금지)을 NFR·인프라로 강제하되, P2 지도 약관·P4 TourAPI 캐싱 조건이 확정될 때까지 영구 저장 표면을 최소화한다.
3. **데이터 품질 게이트** — TourAPI 좌표 확보율·영업시간 채움률이 탐색·일정 품질을 좌우하므로 출시 게이트(G192)를 측정 파이프라인으로 구현한다.
4. **입력 방어** — 사용자 붙여넣기 URL(G31)·외부 응답 스키마는 신뢰할 수 없는 입력이다(SECURITY-05, SSRF·오매칭 방어).

보안·무상태·관측성·배포의 서버 규약은 U1 전역 정본을 그대로 상속하고, U3는 외부 어댑터 계층의 증분만 소유한다.

**워크로드 중요도 (requirements.md §6.3)**

| 컴포넌트 | 중요도 | 불가용 시 영향·U3 대응 |
|---|---|---|
| 숙소 탐색(M3·M5) | **Medium** | 계획 단계 기능 — **수동 등록 우회 존재**(US-E3-08). 외부 소스 전체 실패 시에도 직접 등록으로 여정 지속 |
| POI 정본(M7) 공급 | 기반(Medium) | U4~U8이 이 데이터 위에서 동작 — 캐시 정본/스냅샷으로 외부 순단 흡수 |
| 등록 숙소(M4)·위시리스트 | 조회/저장은 Critical 경로(DB·Auth) 상속 | 자체 데이터 — 외부 의존 없음. 소유권 검증(SECURITY-08)만 증분 |

**핵심 원칙**: 외부 의존은 전부 **Medium 이하 경로**이며, 어떤 외부 소스도 U3의 가용성 바닥이 되어서는 안 된다 — 마지막 폴백은 항상 **수동 등록·직접 입력**(사용자 자체 데이터).

### 6.1 성능 (D38, §6.1)

§6.1이 U3에 직접 부과하는 목표는 **일반 화면 전환 300ms**뿐이며(AI 생성은 U5·U6), U3 성능 본질은 **외부 API 지연이 탐색 체감을 지배하지 않게 타임아웃·부분 응답·캐시로 분리하는 것**이다.

**U3 API 응답 목표 (서버 처리 시간 기준, 네트워크 RTT 별도)**

| API (M3/M7 인터페이스 그룹) | p50 | p95 | 상한(타임아웃) | 비고 · 추적 |
|---|---|---|---|---|
| 숙소·POI 탐색(캐시 히트 — 정본/TTL 캐시 서빙) | 100ms | 300ms | 3초 | 인메모리/정본 조회 — 외부 미호출(§6.4 캐싱) |
| 숙소·POI 탐색(캐시 미스 — 외부 병렬 호출) | 800ms | 3,000ms | **어댑터별 3초 + 부분 응답** | 카카오·TourAPI **병렬** 호출(직렬 금지) — 어느 하나 지연·실패해도 가용 소스로 부분 응답(NFR-U3-PERF-03·US-E3-11) |
| POI 후보 풀 조회(closed-set — M8 그라운딩 공급) | 100ms | 300ms | 3초 | 정본/캐시 조회 — 외부 미호출(G115) |
| 라이브 가격 조회('가격 보기' — 온디맨드 단건) | — | — | **5초(단건)** | **목록 일괄 조회 금지·캐싱 금지**(D09·G33) — 탭 시 날짜·인원 단건만, 세션 내 입력 재사용 |
| 지오코딩·역지오코딩(등록 좌표 확정) | 200ms | 800ms | 3초 | 카카오 지오코딩 → 네이버 폴백(D08). 등록 크리티컬 패스 아님(비동기 확정 가능) |
| 위시리스트·등록 숙소 CRUD(자체 데이터) | 50ms | 200ms | 3초 | 외부 의존 없음 — U1 CRUD 성능 규약 상속 |

- **NFR-U3-PERF-01 (탐색 캐시 우선)**: 탐색은 캐시(정본 영구 + 지도 TTL) 우선 서빙하고, 미스일 때만 외부 병렬 호출한다. 캐시 히트 경로가 지배적이도록 설계해 외부 지연이 상시 노출되지 않게 한다.
- **NFR-U3-PERF-02 (외부 호출 병렬·타임박스)**: 다중 소스(카카오·TourAPI) 탐색은 **병렬 팬아웃 + 어댑터별 3초 타임아웃**. 총 대기는 최장 어댑터 타임아웃(3초)을 넘지 않는다 — 순차 합산 금지.
- **NFR-U3-PERF-03 (부분 응답 허용 — 침묵 실패 금지)**: 일부 소스 미응답 시 나머지로 목록을 구성하고 "일부 정보가 빠졌을 수 있어요"를 표기한다(US-E3-11). 빈칸이 아니라 부분 응답 또는 오류·재시도 안내(ADR-0011).
- **NFR-U3-PERF-04 (라이브 가격 온디맨드)**: '가격 보기'는 탭 시점 단건 조회만 — 목록 전체 일괄 조회·정확 가격 캐싱을 하지 않는다(D09). 입력(날짜·인원)은 세션 내 재사용(G33)하되 결과는 저장하지 않는다.
- **NFR-U3-PERF-05 (지도 렌더 성능·클라이언트)**: 카카오 지도 SDK(config plugin) 로딩·핀 렌더가 상세·핀 지정 화면의 300ms 전환을 차단하지 않게 — 지도는 지연 로드, 로드 실패 시 목록/좌표 텍스트 뷰 폴백.
- **클라이언트 성능**: 탐색 결과는 페이지네이션/무한스크롤(US-E3-01)로 초기 페이로드 제한 — 지역당 POI 5천(G142)을 한 번에 로드하지 않는다. 필터·정렬(직선거리 G34)은 **클라이언트 로컬 계산 우선**(Haversine — 외부 미호출)으로 전환 지연을 네트워크와 분리한다.

### 6.2 복원력 (핵심 · RESILIENCY-10, ADR-0011)

U3 복원력 정본은 **component-dependency §6 외부 어댑터 격리 표**와 **U1 PAT-RES-01 4요소 계약(타임아웃+재시도+서킷+폴백)**이다.

**외부 어댑터 4요소 계약 (RESILIENCY-10 — 계약 누락은 설계 리뷰 차단)**

| 의존 (Port · 소유 M7/M5) | 타임아웃 | 재시도 | 서킷 브레이커 | 폴백 체인 (RESILIENCY-10) |
|---|---|---|---|---|
| **카카오 로컬 장소 검색** (`PlaceSearchPort`) | 3초 | 없음(멱등 GET이나 쿼터 보호 우선 — 사용자 재시도 버튼) | 벤더별 독립 서킷(실패율 임계 초과 시 open) | **① 네이버 2차 폴백(D08) → ② TTL 캐시 잔존분 → ③ '확인 불가' 안내(G8)** |
| **카카오 지오코딩** (`GeocodingPort`) | 3초 | 없음 | 독립 서킷 | ① 네이버 2차(D08) → ② **수동 핀 지정 흐름**(US-E3-08 — 좌표 미확정 등록은 "지도에서 위치 확인" 강제) |
| **TourAPI 콘텐츠** (`TourContentPort`, M3 계약 공유) | 3초 | 없음 | 독립 서킷 | ① 카카오 검색 대체 → ② **정본 테이블 잔존분**(허용 소스 영구) → ③ '미확인' 배지(US-E3-03) |
| **네이버 장소**(2차 폴백 전용) | 3초 | 없음 | 독립 서킷 | 폴백의 폴백 — 실패 시 TTL 캐시 잔존분 → '확인 불가' |
| **OTA 딥링크** (`DeeplinkBuilder`, M5 — **외부 호출 없음**) | N/A(URL 조립만) | N/A | N/A | 링크 생성 불가 시 **타 OTA 링크 또는 숙소명 복사 안내**(US-E3-05·D17 다중 OTA) |
| **클라이언트 카카오 지도 SDK**(`shared/map` 브리지) | 지도 로드 타임아웃(클라) | — | — | 지도 로드 실패 시 **목록 뷰/좌표 텍스트 폴백**(U3 리스크 완화) |

- **NFR-U3-RES-01 (전 어댑터 타임아웃·서킷 의무)**: 카카오·네이버·TourAPI 각 어댑터는 명시적 타임아웃 + 독립 서킷 브레이커를 가진다(벤더 1종 장애가 타 벤더로 전파 금지 — 벌크헤드). 서킷 open/half-open 전이는 계측 이벤트(SI §8.2 A12).
- **NFR-U3-RES-02 (벤더 폴백 체인 — D08)**: 지도 계열은 **카카오 → 네이버 2차 폴백**(D08 정본)을 어댑터 내부에서 처리 — M3/M7 도메인 로직은 벤더 무지(격리 원칙). 폴백 순서는 remote config로 조정 가능(NFR-U1-MT-04 상속).
- **NFR-U3-RES-03 (최종 폴백 = 수동 등록 우회)**: **외부 소스 전체 실패 시** 탐색은 "지금 숙소 정보를 불러올 수 없어요" + 재시도, 그리고 **직접 등록(US-E3-08) 우회 경로**를 제공한다(US-E3-11). 데이터 부족 지역도 동일. **어떤 외부 장애도 사용자 여정을 완전 차단하지 않는다.**
- **NFR-U3-RES-04 (거리 = 직선거리 로컬 계산 · 카카오모빌리티 경계)**: U3 거리 필터·정렬은 **검색 여행지 중심 좌표 기준 직선거리(Haversine — 로컬 계산)**만 사용하며 소요시간을 표시하지 않는다(G34·D25). **따라서 U3는 카카오모빌리티(도로거리)을 호출하지 않는다.** 카카오모빌리티 `RoadDistancePort`·라우팅 실패 시 직선거리×우회계수(G106) 폴백은 **C2 소유로 U5 범위**다. U3의 직선거리는 폴백이 아니라 **1차 정본**이므로 외부 라우팅 의존 자체가 없다(가용성 우위).
- **NFR-U3-RES-05 (부분 실패·오매칭 격리)**: 소스 부분 실패는 부분 응답(NFR-U3-PERF-03), POI 오매칭(동명·근접 병합 오류)은 **매칭 임계 보수 설정 + 운영자 보정 여지**(D17)로 격리 — 오매칭이 일정 생성 그라운딩을 오염시키지 않게.
- **NFR-U3-RES-06 (캐시는 가용성 요소 아님 — NFR-U1-AV-03 상속)**: 탐색 TTL 캐시·인메모리 캐시는 지연 최적화일 뿐 가용성 요소가 아니다 — 캐시 소실은 외부 재호출로 충족한다. 단, TourAPI **정본 테이블**은 캐시가 아닌 정본이며(허용 소스 영구), 외부 순단 시 폴백 데이터 소스로 기능한다.

### 6.3 규모 (§6.8 G142 — DAU 1천 · 지역당 POI 5천)

- **NFR-U3-SC-01 (POI 정본 볼륨)**: 지역당 POI 후보 **5천**(G142). 국내 주요 지역 수십 개 기준 정본 테이블 **수십만 행** 수준 — 단일 PostgreSQL로 충분(SI §3, 샤딩 불요). U3 신규 테이블 ~8~9개(§7.2).
- **NFR-U3-SC-02 (탐색 트래픽)**: 탐색은 계획 단계 저빈도(Medium §6.3) — DAU 1천 기준 정상 피크 수 RPS 미만. **캐시 히트가 지배적**이면 외부 호출은 캐시 미스·라이브 가격에 한정. SI §1.2 태스크 2개로 흡수(추가 컴퓨트·오토스케일 산정 불요).
- **NFR-U3-SC-03 (라이브 가격 호출 상한)**: 라이브 가격 조회는 **파트너 약관 최소치 준수**(G196) — 온디맨드 단건·목록 일괄 금지(D09)로 호출량을 구조적으로 억제. 사용자·세션 단위 rate-limit(U1 PAT-SEC-03 레이트리미터 재사용, LC-3 전역 컴포넌트).
- **NFR-U3-SC-04 (정적 콘텐츠 갱신 주기)**: OTA·TourAPI 정적 콘텐츠는 **일 1회 배치 갱신**(G196) — 실시간 반복 호출 금지. 배치는 기존 ShedLock 스케줄러 프레임(SI §1.2) 재사용.
- **NFR-U3-SC-05 (외부 쿼터 80% 알람)**: 카카오·TourAPI·네이버 각 어댑터 호출량 커스텀 메트릭 대비 문서화된 쿼터의 **80% 알람**(SI §8.2 A11, NFR-U1-SC-04 상속) — U3가 A11·A12의 첫 소비자다. 쿼터 한도는 P4(TourAPI)·벤더 콘솔 확인 후 문서화.

### 6.4 캐싱·데이터 (D13 소스별 정책의 NFR화)

D13 하이브리드 저장을 NFR·인프라 강제 규칙으로 승격한다. **적법성(P2·P4)이 이 절을 지배**한다.

**소스별 캐싱 정책 (D13 · component-dependency §3.1)**

| 소스 | 저장 정책 | NFR 강제 규칙 | 근거·선결 |
|---|---|---|---|
| **TourAPI 등 캐싱 허용 공공 소스** | **정본 테이블 영구 저장**(D13 (a)) | 캐싱 **허용분만** 정본 영구 — 허용 범위는 보존 정책 컬럼으로 소급 축소 가능 | **P4 캐싱 조건 확인 선결** |
| **지도 API(카카오·네이버)** | **실시간 호출 + 약관 허용 TTL 캐시**(D13 (b)) | **영구 캐싱 금지** — TTL 상한 준수, 만료 후 재조회 강제. 영구 저장 표면 최소화(인메모리 권고) | **P2 지도 약관 선결** |
| **사용자 확정 스냅샷**(등록 숙소·필수 방문지·방문 체크) | 확정 시점 '사용자 입력 데이터'로 영구 스냅샷(D13) | **스냅샷 필드 최소화** + 소스별 보존 정책 컬럼 | **P2 스냅샷 적법성 법무 확인 선결** |
| **정확 가격(라이브)** | **저장 안 함** | 캐싱 금지(D09) — 조회 시점 소비 후 폐기 | US-E3-01 |

- **NFR-U3-DATA-01 (영구 캐싱 금지 강제)**: 지도 API 소스 데이터는 **TTL 없는 영구 저장을 금지**한다 — TTL·만료 정리·출처 표기를 데이터 모델·인프라 차원에서 강제. TTL 만료 속성은 PBT 대상(만료 후 재조회 강제 — U3 DoD).
- **NFR-U3-DATA-02 (canonical POI ID 무결성 — 하드 제약 D37)**: 동일 실세계 장소는 **정확히 1개 canonical POI ID**(좌표 근접 50m + 명칭 유사도, G133/G148). 소스별 place_id N:1 매핑(D17 패턴). 매칭 멱등성·대칭성은 PBT·골든 데이터셋 회귀로 상시 검증. 오매칭은 그라운딩 오염이므로 하드 제약.
- **NFR-U3-DATA-03 (스냅샷 필드 최소화 — P2 대응)**: 사용자 확정 스냅샷은 일정·거점 기능에 필요한 **최소 필드만**(명칭·좌표·카테고리·체류 기본값 등) 저장하고, 소스·보존 정책 컬럼(source·retention_policy·cached_at)을 둔다 — 약관 변경 시 정책 컬럼 기준 소급 대응. **P2 확정 전 지도 API 소스 스냅샷은 보류**(TourAPI 허용분만 영구).
- **NFR-U3-DATA-04 (출처 표기)**: 지도 API·TourAPI 데이터는 화면·응답에 **출처 표기**(D13·지도 약관 요건).
- **NFR-U3-DATA-05 (캐시 2계층)**: 캐시는 2계층 — (1) TourAPI **정본 테이블**(RDS 영구, 허용 소스), (2) 지도 API 소스·핫 탐색 결과의 **단기 TTL 캐시**. 단기 캐시는 **인메모리(Caffeine) 권고**(정본 아님·미스=재호출·영구 저장 표면 0 → P2 리스크 격리·신규 인프라 0). **ElastiCache는 미도입** — 규모·정본 아님 대비 이득 0(SI §4 트리거와 통합: 실측 50 RPS 초과 또는 U11 팬아웃 시 재평가).

### 6.5 보안 (Security Baseline 상속 — §6.4)

U3는 SECURITY 규칙의 신규 1차 구현 지점이 아니다 — 전역 보안 설정(deny-by-default·헤더·에러 핸들러·구조화 로깅·시크릿 관리)은 U1 스캐폴드가 소유하고 U3는 상속한다. U3 증분은 **외부 입력·외부 키·소유권** 세 지점이다.

- **NFR-U3-SEC-01 (외부 API 키 = 시크릿)**: 카카오 REST 키·TourAPI 서비스 키·네이버 클라이언트 시크릿(U5용 카카오모빌리티 키는 U5)은 **소스·이미지·설정 파일에 미포함** — Secrets Manager 주입(NFR-U1-SEC-11 상속, PAT-SEC-05). 어댑터별 키 분리, 시크릿 스캔 CI 게이트(SECURITY-10).
- **NFR-U3-SEC-02 (URL 파싱 SSRF 방어 — G31)**: OTA 링크 붙여넣기는 **URL 패턴 파싱만 · 페이지 fetch 없음**(G31) — 사용자 제공 URL로의 아웃바운드 요청이 **구조적으로 발생하지 않으므로 SSRF가 원천 차단**된다. 추가 가드: (a) 지원 도메인 화이트리스트만 파싱, (b) URL 형식·길이·스킴(http/https) 검증(SECURITY-05), (c) 파싱 추출값(숙소명·좌표)도 신뢰하지 않고 지오코딩 재검증, (d) 화이트리스트 밖·파싱 실패는 **수동 핀 지정 흐름 폴백**(US-E3-08). 화이트리스트 밖 URL 전부 거부·폴백 경로는 PBT 대상(U3 DoD).
- **NFR-U3-SEC-03 (외부 응답 스키마 검증)**: 카카오·TourAPI·네이버 응답은 **외부 입력**이다 — 응답 스키마 검증(SECURITY-05), 비정상·부분 응답은 구조화 처리(누락 필드는 '미확인'으로 저장, 오류 fail-safe). 외부 문자열(숙소명)은 저장 전 새니타이즈·저장 후 출력 이스케이프(XSS 방어, NFR-U1-SEC-14 상속).
- **NFR-U3-SEC-04 (크롤링 금지 준수)**: OTA는 **데이터 크롤링 없이 숙소명 검색 딥링크만**(D09) — `DeeplinkBuilder`는 외부 호출 없는 URL 조립만. 리뷰·평점 미표시(OTA 위임), 제휴 수수료 고지(US-E3-05).
- **NFR-U3-SEC-05 (소유권 — SECURITY-08)**: 위시리스트·등록 숙소·저장 POI 접근은 **토큰 주체(accountId) 기준으로만 조회**(IDOR 원천 차단, NFR-U1-SEC-18 상속). 저장은 로그인 계정 귀속(D22 — 기기 변경 유지, US-E3-04). 등록 숙소는 계정 레벨 풀(D15).
- **NFR-U3-SEC-06 (국내 좌표 범위 검증)**: 등록·핀 지정 좌표는 국내 좌표 범위(bounding box/역지오코딩) 검증(G120 — 관련 강제는 U4 완성이나 좌표 확정은 U3) — 범위 밖 차단·사유 안내.

### 6.6 커버리지 게이트 — 데이터 품질 출시 판정 (G192)

TourAPI 데이터 품질(좌표 확보율·영업시간 채움률)이 탐색·일정 품질을 좌우하므로, **출시 게이트를 측정 파이프라인으로 구현**한다.

- **NFR-U3-GATE-01 (출시 게이트 지표)**: **좌표 확보율 ≥ 95% · 영업시간 채움률 ≥ 70%**(G192)를 지역별로 측정한다. 게이트 미달 지역은 출시 판정에서 **차단 또는 제한 노출**(운영 판단) — 게이트는 데이터 품질의 정량 출시 기준이다.
- **NFR-U3-GATE-02 (측정 잡)**: 지역별 POI 정본의 좌표·영업시간 채움률을 **주기 측정하는 배치 잡**(기존 ShedLock 스케줄러 재사용, §7.4)이 커스텀 메트릭으로 발행 — 대시보드·게이트 판정 입력.
- **NFR-U3-GATE-03 ('미확인' 표기)**: 미달·누락 필드는 **빈칸이 아니라 '미확인'으로 표기**(US-E3-03·G199 통일) — 침묵 실패 금지(ADR-0011). 좌표 미확정 등록은 "지도에서 위치 확인" 단계 강제(US-E3-06·08).
- **NFR-U3-GATE-04 (영업시간 변경 감지 경계)**: 정기 영업시간 변경만 자동 감지(TourAPI·지도 재조회, 일 1회), 당일 임시휴무는 수동 트리거(G192) — 자동 감지는 M9 휴무 폴링(U6)으로 승격되나, U3는 **정본의 영업시간 필드 갱신·채움률 측정**까지 소유.

**PAT-U3-01 외부 어댑터 서킷브레이커·폴백 체인 — 벤더 폴백 D08 `[U3]`**
- **문제**: 외부 의존(카카오·TourAPI·네이버)은 제어할 수 없는 가용성을 가진다. 무한 대기는 탐색 스레드 고갈로 증폭되고, 벤더 1종 장애가 타 벤더 호출로 전파되면 격리가 무너지며, 침묵 실패는 사용자를 빈 화면에 방치한다(ADR-0011).
- **적용**: 전 외부 호출은 포트 뒤에서 U1 PAT-RES-01 4요소 계약을 의무화하고(계약 누락은 설계 리뷰 차단), Resilience4j(U3-S1)로 구현한다. 벤더별 **독립 서킷**(벌크헤드) + 폴백 체인(D08 순서 — §6.2 표와 동일).

```text
[탐색 요청]
  → 캐시 조회(정본 + TTL) ──히트──> 응답(외부 미호출)
  → 미스: 카카오·TourAPI 병렬 팬아웃 (어댑터별 3초·독립 서킷)
       ├ 카카오 실패/open ──> 네이버 2차(D08) ──> TTL 캐시 잔존 ──> '확인 불가'(G8)
       └ TourAPI 실패/open ──> 정본 테이블 잔존 ──> '미확인' 배지
  → 일부 성공: 부분 응답 + "일부 정보가 빠졌을 수 있어요"(US-E3-11, 침묵 실패 금지)
  → 전체 실패: "지금 불러올 수 없어요" + 재시도 + [직접 등록] 우회(US-E3-08·10·11)
```

- **설계 원칙**: (a) 재시도는 멱등 GET이라도 쿼터 보호 우선으로 미적용(사용자 버튼이 재시도), (b) 서킷 open/half-open은 계측(A12), (c) 폴백은 반드시 사용자 고지·계측과 짝(침묵 실패 금지), (d) 폴백 순서·타임아웃은 remote config(OPS-01), (e) 최종 폴백은 항상 수동 등록 우회(NFR-U3-RES-03). 카카오모빌리티 경계: U3 거리는 직선거리 로컬 계산(G34)이라 카카오모빌리티 미호출 — 라우팅 폴백(G106)은 C2/U5.
- **U3 적용 지점**: M7 어댑터(`PlaceSearchPort`·`GeocodingPort`·`TourContentPort`)·네이버 폴백 어댑터, M3 탐색 오케스트레이션(병렬 팬아웃·부분 응답), M5 `DeeplinkBuilder`. 구현은 Resilience4j(U3-S1), fake 실패 주입 테스트(D37).
- **검증 기준**: RESILIENCY-10 — 전 어댑터 명시 타임아웃·독립 서킷·폴백 체인(코드 리뷰+계약 테스트), fake로 타임아웃·5xx·서킷 open 주입 시 폴백 실동작(카카오 다운→네이버, 전체 다운→수동 등록 우회 노출) PR CI 통과. RESILIENCY-09 — 서킷/쿼터 계측이 A11·A12에 존재.

**PAT-U3-02 POI 하이브리드 캐싱 — 정본 영구 vs 실시간 TTL (소스별 D13) `[U3]`**
- **문제**: 지도 API 약관은 영구 캐싱을 금지(D13)하고, 그 적법성(스냅샷 영구 저장 포함)은 P2 법무 확인 전까지 미확정이다. 동시에 매 탐색마다 외부를 재호출하면 지연·쿼터(G196)를 소모한다. 소스별 저장 정책이 갈리는데 이를 코드가 혼동하면 약관 위반(법적 리스크)이 된다.
- **적용**: 저장 정책을 **소스 태그로 코드·스키마 레벨에서 분기**(혼동 원천 차단) — TourAPI 허용 공공 소스=정본 테이블 영구(일 1회 배치 갱신), 지도 API=인메모리 TTL만(영구 저장 표면 0), 사용자 확정 스냅샷=최소 필드 영구+보존 정책 컬럼(source·retention_policy·cached_at), 라이브 정확 가격=저장 안 함.
- **설계 원칙**: (a) 영구 캐싱 금지 강제 — 지도 소스는 인메모리 TTL 외 영구 경로가 코드에 존재하지 않게(아키텍처 테스트로 "지도 소스 → 정본 테이블 쓰기" 검출), (b) canonical POI ID 무결성(좌표 50m + 유사도, G133/G148)은 하드 제약(NFR-U3-DATA-02) — 매칭 멱등·대칭 PBT, (c) 캐시는 정본이 아님(미스=재호출) — 단 TourAPI 정본 테이블은 외부 순단 시 폴백 소스(PAT-U3-01), (d) 출처 표기(D13). P2·P4 대응: 정책 컬럼으로 약관 확정 후 보존 범위 소급 조정.
- **U3 적용 지점**: M7 정본 저장소(정본 테이블·소스 매핑 D17)·인메모리 TTL 캐시·스냅샷 발급, canonical ID 발급기(유사도 매칭 U3-S5).
- **검증 기준**: D13 — 소스별 저장 경로 분기 테스트(지도 소스가 정본 테이블에 영구 저장되지 않음 아키텍처 테스트), TTL 만료 후 재조회 강제 PBT. NFR-U3-DATA-02 — canonical 매칭 멱등·대칭 PBT + 골든 데이터셋 회귀.

**PAT-U3-03 라이브 가격 온디맨드 호출 — 목록 일괄 금지 `[U3]`**
- **문제**: 정확 가격·재고는 외부 OTA가 확정하며 변동한다(US-E3-01). 목록 전체를 일괄 조회·캐싱하면 파트너 약관·쿼터(G196)를 위반하고, 낡은 가격을 저장해 사용자를 오도한다(D09).
- **적용**: 온디맨드 단건('가격 보기' 탭 시점에만 날짜·인원 단건 조회 G33, 목록 렌더 시 일괄 조회 금지, 입력값은 세션 내 재사용하되 결과는 저장 안 함, 타임아웃 5초). 호출 억제(사용자·세션 단위 rate-limit — U1 PAT-SEC-03 재사용, LC-3 전역). **MVP 실현 경계**: 1차는 OTA API 계약(P5) 전이므로 '가격 보기'가 날짜·인원 파라미터를 담은 OTA 검색 딥링크 핸드오프로 실현될 수 있다 — 파트너 가격 API 확보 시 동일 온디맨드 계약으로 교체(어댑터 뒤 격리).
- **U3 적용 지점**: M3 '가격 보기' 유스케이스(바텀시트·세션 입력), M5 딥링크(날짜·인원 파라미터), 사용자별 rate-limit 필터.
- **검증 기준**: D09 — 목록 렌더 시 일괄 가격 조회 미발생·정확 가격 미저장 테스트. G196 — rate-limit 발동 테스트. G33 — 세션 내 입력 재사용·결과 비저장 확인.

**PAT-U3-04 URL 파싱 SSRF 가드 — 화이트리스트·fetch 없음 `[U3]`**
- **문제**: 사용자가 붙여넣는 OTA 링크(US-E3-08)는 신뢰할 수 없는 입력이다. 서버가 이 URL을 fetch하면 내부망·메타데이터 엔드포인트 접근(SSRF)의 문이 열리고, 파싱 결과를 무검증 신뢰하면 오염된 좌표·명칭이 등록된다.
- **적용**: fetch 없는 패턴 파싱 + 다층 가드(G31) — SSRF 원천 차단(URL 패턴 파싱만·페이지 fetch 없음, 사용자 제공 URL로의 아웃바운드 요청이 코드 경로에 부재, 아키텍처 테스트로 "붙여넣은 URL을 인자로 하는 HTTP 호출" 부재 검출), 입력 검증(SECURITY-05 — 스킴 http/https만·화이트리스트 remote config 내 URL만·길이 상한·형식 검증·화이트리스트 밖은 수동 핀 지정 폴백), 파싱 결과 불신(추출 숙소명·좌표도 지오코딩 재검증 + 국내 좌표 범위 검증 G120 후 확정).
- **U3 적용 지점**: M4 링크 파싱 유스케이스(US-E3-08 경로 2), 화이트리스트 remote config, 폴백 라우팅(수동 핀).
- **검증 기준**: G31/SECURITY-05 — 화이트리스트 밖 URL 전부 거부·파싱 실패 시 폴백 경로 PBT, 붙여넣은 URL 대상 아웃바운드 호출 부재 아키텍처 테스트(SSRF 구조적 차단), 파싱 좌표의 지오코딩 재검증·좌표 범위 차단 테스트.

**PAT-U3-05 복귀 핸드오프 상태 머신 `[U3]`**
- **문제**: OTA 딥링크 이탈 후 예약했는지 앱은 알 수 없다(포스트백 자동 등록은 1차 제외, D09/G29). 그렇다고 방치하면 등록 온램프가 끊기고, 반복 노출하면 사용자를 괴롭힌다(G32 — 1회만).
- **적용**: 최근 이탈 숙소 1건 기준 상태 머신(G32).

```text
[딥링크 이탈] → 클릭 로그 기록(숙소 ID·이탈 시각) → 상태: PENDING_RETURN(최근 1건만)
   │  (24시간 창)
[24h 내 첫 복귀] → 핸드오프 카드 1회 노출("방금 본 [숙소명], 예약하셨나요? → 거점 등록")
   ├ 사용자 [등록] → 등록 폼 자동 채움(US-E3-06) → 상태: CONSUMED
   └ 사용자 무시/닫기 → 상태: DISMISSED (재노출 없음)
[24h 경과] → 상태: EXPIRED (노출 없음)
```

- **설계 원칙**: (a) 최근 이탈 숙소 1건만 추적(다중 누적 금지 G32), (b) 24h 창·첫 복귀·1회 노출을 상태 전이로 명문화(무시 후 재노출 없음), (c) 클릭 로그(아웃바운드 클릭)는 소유권 스코핑(accountId) + 제휴 고지 계측, (d) 결정성은 clock 주입 순수 함수로 테스트(창 경계·복귀 판정).
- **U3 적용 지점**: M5 클릭 로그·복귀 판정, 핸드오프 카드(클라이언트), 등록 폼 자동 채움 연결(M4).
- **검증 기준**: G32 — 24h 내 첫 복귀 1회 노출·무시 후 재노출 없음·경계(24h 직전/직후) 상태 전이 테스트(clock 주입 결정성), 최근 1건만 추적 확인, 클릭 로그 소유권 스코핑 테스트(SECURITY-08 상속).

**PAT-U3-06 커버리지 게이트 데이터 품질 검증 `[U3]`**
- **문제**: TourAPI 데이터 품질(좌표 확보율·영업시간 채움률)이 낮으면 탐색·일정 생성 그라운딩이 오염되고, 이를 측정하지 않으면 "출시해도 되는가"를 답할 수 없다. 빈 필드를 방치하면 사용자에게 고장 인상을 준다.
- **적용**: 정량 출시 게이트(G192 — 지역별 좌표 확보율 ≥ 95% · 영업시간 채움률 ≥ 70% 측정 배치 잡(ShedLock 재사용, §7.4)이 커스텀 메트릭 발행 → 대시보드·게이트 판정, 미달 지역 출시 제한), '미확인' 표기(ADR-0011 — 누락·미달 필드는 빈칸 아닌 '미확인' US-E3-03·G199, 좌표 미확정 등록은 "지도에서 위치 확인" 강제), 영업시간 갱신 경계(정기 변경만 자동 감지 일 1회, Plan-B 트리거 승격은 M9/U6).
- **U3 적용 지점**: M7 커버리지 측정 잡, 정본 저장의 채움률 계측, '미확인' 표기 규약(클라이언트·M7 DTO).
- **검증 기준**: G192 — 지역별 좌표·영업시간 채움률 산출 정확성 테스트, 게이트 임계(95%/70%) 판정 로직 테스트, 미달·누락 필드의 '미확인' 표기 테스트(빈칸 아님). RESILIENCY-05 — 커버리지 메트릭이 대시보드에 존재(SI §8 편입).

### 6.8 기술 스택 결정 (U3-C·U3-S)

스택 대분류(RN Expo + Spring Boot Kotlin + PostgreSQL, D02)와 U1 라이브러리 정본(S-1~S-13·C-1~C-8)은 기확정이다. U3는 그 위에 **지도 SDK·서버측 외부 API 클라이언트·POI 캐시·유사도 매칭** 4개 지점만 구체화하며, 나머지는 U1 결정을 상속한다. 지도·장소 벤더는 **D08 기확정**(카카오+카카오모빌리티+네이버 — 재확정 없음).

- **U3-C1·C2 클라이언트 지도 SDK**: D02가 "Expo development build + prebuild 운용(국내 지도 SDK config plugin 대비)"을 U1 스캐폴드에서 이미 확정했다. **권고 A(카카오 네이티브 SDK + Expo config plugin)**, 폴백 = B(WebView JS SDK) 또는 목록/좌표 텍스트 뷰. 근거: D08(카카오)+D02(config plugin 대비)의 정합, 핀 지정·바운즈 등 US-E3-08 상호작용의 네이티브 품질. 리스크 완화: config plugin 검증을 U3 초기 스파이크로 선행, 실패 시 지도 표시 범위 축소 폴백 상비. **C(react-native-maps) 비채택**: 구글 지도 기반으로 D08(카카오)과 벤더 불일치 — 국내 POI·지오코딩 정합 저하.
- **U3-S1 HTTP 클라이언트 — Spring RestClient**: 어댑터별 베이스 URL·타임아웃·직렬화 구성. 저빈도·순차 소량 호출(계획 단계 Medium §6.3)이라 동기 블로킹으로 충분. `WebClient`(리액티브)는 대량 병렬 팬아웃 강점이나 U3 규모(수 RPS·소스 2종 병렬)에서 과설계 — 탐색 2소스 병렬은 RestClient + 가상 스레드(JDK 21)/`CompletableFuture`로 충분. `RestTemplate`은 유지보수 모드라 부적절.
- **U3-S1 복원력 라이브러리 — Resilience4j**(`resilience4j-spring-boot3`): 어댑터 포트별 타임아웃 + 서킷 브레이커 + (제한적) 재시도 + 벌크헤드를 선언적 데코레이터로 적용(NFR-U3-RES-01 4요소 계약 구현체). 카카오·TourAPI·네이버 벤더별 독립 서킷 인스턴스(벌크헤드), 폴백 함수로 벤더 폴백 체인(카카오→네이버 D08→캐시)을 코드로 구성(PAT-U3-01), 설정 외부화(remote config). Spring Retry는 서킷·벌크헤드 부재, Spring Cloud Circuit Breaker는 벤더별 세밀 튜닝에 우회 계층, 수동 구현은 서킷 상태 머신 재구현 위험. 메트릭(Micrometer)이 SI §8.2 A12와 직결.
- **U3-S3·S4 POI 캐시 저장**: 정본(영구)=PostgreSQL 정본 테이블(TourAPI 등 캐싱 허용 소스, D13 (a) — 신규 인프라 0, 스키마 확장만) / 단기 TTL 캐시=인메모리 Caffeine(지도 API 소스·핫 탐색, D13 (b)). **캐시 계층 결정(인메모리 vs ElastiCache vs DB TTL)**:

| 기준 | **인메모리 Caffeine (권고)** | ElastiCache Redis | DB TTL 테이블 |
|---|---|---|---|
| 정본 적합성 | 캐시=최적화(미스=외부 재호출, NFR-U1-AV-03 상속) — 태스크 간 불일치 무해 | 충족(공유) | 충족 |
| 지도 약관(P2) | **영구 저장 표면 0** — 프로세스 메모리·TTL만(리스크 최소) | 외부 저장소 상주(TTL이나 표면↑) | DB 영구 스토리지 상주(약관 경계 근접) |
| 규모 정합(G142) | **적합** — 수 RPS·POI 5천/지역, 태스크당 힙 여유(SI §1.2 2GB) | 월 ~$25 + 운영 표면 — 이득 0 | RDS I/O·정리 잡 추가 |
| 신규 인프라 | **0** | +1 서비스·SG·서브넷 | 0(RDS 내) |
| 쿼터 보호(G196) | 태스크당 캐시 → 최대 태스크 수배 호출(0.5 RPS에서 무의미) | 공유로 호출 최소화 | 공유 |

  **확정 권고: 인메모리 Caffeine(단기 TTL).** 지도 API 소스는 영구 저장 금지(D13·P2)라 인메모리 TTL이 약관 리스크를 가장 줄이고, 캐시는 정본이 아니므로 태스크 간 불일치는 재호출로 무해, 신규 인프라 0. **ElastiCache 비채택**: 값·규모 대비 월 비용·운영 표면 증가가 이득 상회 — SI §4와 동일 트리거(실측 50 RPS 초과 또는 U11 팬아웃)로 통합 재평가. **DB TTL 비채택**: 지도 API 데이터를 DB 영구 스토리지에 두는 것이 P2 약관 경계에 근접, TourAPI 허용분은 이미 정본 테이블 존재.
- **U3-S5 canonical POI 유사도 매칭 — 좌표 블로킹 + 문자열 유사도**: 2단 매칭 — (1) 좌표 블로킹(후보를 좌표 근접 50m(Haversine) 이내로 1차 축소 — 전수 비교 회피, POI 5천/지역 성능), (2) 명칭 정규화(공백·특수문자·법인격 접미어 제거, 한국어명 정본+영문 alias) 후 **Jaro-Winkler(접두 가중) 또는 정규화 Levenshtein**(apache commons-text `similarity`)으로 유사도 산출 → 보수적 임계값 이상만 병합. 코사인 유사도는 긴 문장에 강하나 짧은 상호명(2~5음절)에는 Jaro-Winkler가 적합, 완전 일치만 사용하면 표기 변형(호텔/HOTEL·띄어쓰기) 놓쳐 중복 canonical 발생, ML 임베딩은 규모·운영 대비 과설계. 보수적 임계+운영자 보정 여지로 오매칭 격리. 라이브러리는 U1 Gradle version catalog에 추가·잠금 파일 커밋(SECURITY-10 상속).
- **U3-S6 거리 계산 — Haversine 직선거리(로컬, 카카오모빌리티 미사용)**: 기준점은 검색 여행지 중심 좌표(G34), 소요시간 미표시(D25). 카카오모빌리티(도로 거리) `RoadDistancePort`는 C2 소유로 U5 범위 — U3는 카카오모빌리티 미호출·직선거리가 1차 정본(NFR-U3-RES-04), 라우팅 실패 폴백(G106)도 C2/U5. 외부 라우팅 의존 없음(가용성 우위).

**상속·비결정 재확인**: 서버 프레임워크·인증·DB 접근·마이그레이션(U1 S-2·S-3·S-8·S-9), Expo SDK·TS strict·prebuild·상태 관리·HTTP·PBT(U1 C-1·C-4·C-6·C-7), 클라우드·CI/CD·배포·시크릿·로그(GD-1~7·SI·U1 PD-1~9), 지도·장소 벤더(D08 기확정), POI 저장 정책(D13 기확정)은 전부 상속·재결정 금지. **U3 신규 벤더·클라우드 결정 0** — 기존 스택 내 라이브러리 구체화(지도 SDK 연동·Resilience4j·Caffeine·유사도 알고리즘)만 유닛 증분.

### 6.9 미결·선결 항목

**선결 과제 — 설계 차단 요인 (P2·P4)**: U3는 비개발 선결 2건이 Functional Design(스냅샷·정본 테이블 확정 스키마) 착수를 차단한다. 본 NFR 산출물은 벤더·정책 중립 요건까지 확정하며 차단되지 않는다.

| 과제 | 차단 대상 | 중립 대응 |
|---|---|---|
| **P2 지도 API 약관 검토·계약**(D13 스냅샷 적법성 법무 확인) | 지도 API 소스 캐싱 TTL 상한·사용자 확정 스냅샷 영구 저장 가부 | 지도 소스 영구 저장 표면 0(인메모리 TTL), 스냅샷 필드 최소화·정책 컬럼(NFR-U3-DATA-03), P2 확정 전 지도 소스 스냅샷 보류 |
| **P4 TourAPI 활용 신청·캐싱 조건 확인** | 정본 영구 저장 필드 범위(D13 (a)) | `TourContentPort` 격리, 허용분만 정본 영구·보존 정책 컬럼으로 소급 축소(NFR-U3-DATA-01) |
| P5 OTA 제휴 계약(개발 무차단) | 파트너 딥링크 정책·수수료 문안 | 딥링크 템플릿·화이트리스트 remote config화(NFR-U3-SEC-02·04) |

**벤더 종속 미결 — 없음**: U3는 신규 인프라·벤더 종속 미결 항목이 없다. U1이 PD-1~9를 전부 종결했고, U3는 그 종결 결과와 GD-1~7·SI를 상속한다. U3 유닛 로컬 결정(지도 SDK·서킷브레이커 라이브러리·POI 캐시 계층·유사도 알고리즘)은 기존 스택 내 선택이며 신규 벤더가 아니다.

## 7. 인프라

공유 인프라 정본은 [shared-infrastructure.md](../infrastructure.md)(이하 SI)이며, **U3는 SI를 참조만 하고 재정의하지 않는다.** 본 절은 U3가 SI에 추가하는 **인프라 증분**(외부 아웃바운드·POI 정본 저장·단기 캐시·배치 잡)만 기술한다. 배포 파이프라인 정본은 U1 deployment-architecture.md다.

### 7.0 요약 — U3 인프라 증분: 신규 AWS 리소스 사실상 0

U3는 외부 의존 최다 유닛이나 **신규 AWS 리소스는 사실상 0**이다 — SI가 이미 NAT 아웃바운드(§2.1)·시크릿 프레임(§6)·외부 쿼터/실패율 알람 프레임(§8.2 A11·A12)·ShedLock 스케줄러 프레임(§1.2)을 U3(TourAPI·카카오·네이버)을 명시적 소비자로 예약해 두었다. U3 인프라 결정은 넷 — **① 외부 아웃바운드·시크릿 매핑, ② POI 정본 저장(RDS 스키마 확장), ③ 단기 캐시 계층(인메모리 vs ElastiCache), ④ 배치 잡(기존 스케줄러 재사용)** 이며 전부 기존 리소스 내 확장/설정이다.

**선결 경계**: POI **정본 테이블·스냅샷의 확정 스키마**는 **P2(지도 약관)·P4(TourAPI 캐싱 조건)** 선결 후 Functional Design 소관이다. 본 절은 저장 위치·격리·정책 컬럼 구조까지 확정한다.

### 7.1 외부 API 아웃바운드 · 시크릿 보관 (SI §2·§6 상속)

- **아웃바운드 경로**: ECS(프라이빗 앱 서브넷) → **NAT Gateway**(SI §2.1, 기존 1개) → 외부 엔드포인트. `sg-app` 아웃바운드 443은 SI §2.2가 이미 허용("목적지 IP 가변 SaaS 예외 문서화") — **SG 변경 0**. U3 아웃바운드 목적지 문서화(egress 허용 목록):

| 목적지 | 어댑터(포트) | 용도 |
|---|---|---|
| 카카오 로컬 API | `PlaceSearchPort`·`GeocodingPort` | 장소 검색·지오코딩(D08) |
| TourAPI(공공데이터포털) | `TourContentPort` | POI·숙소 정적 콘텐츠(P4 선결) |
| 네이버 장소 API | 네이버 폴백 어댑터 | 2차 폴백(D08) |
| OTA 딥링크 도메인 | `DeeplinkBuilder`(외부 호출 없음) | URL 조립만 — 아웃바운드 미발생(크롤링 금지) |

- **시크릿(SI §6 프레임)**: 외부 API 키를 **Secrets Manager**에 신규 시크릿 항목으로 등록(신규 서비스 아님 — 프레임 내 항목 추가): `external/kakao-rest`(카카오 REST 키), `external/tourapi`(서비스 키), `external/naver`(클라이언트 ID/시크릿). 카카오모빌리티 키(`external/kakao-mobility`)는 **U5에서 등록**(도로거리 C2 소유). ECS 태스크 정의 `secrets` ARN 참조 주입 — 코드·이미지·Terraform state 평문 0(NFR-U3-SEC-01·NFR-U1-SEC-11). 클라이언트 카카오 지도 SDK 앱 키는 앱 config plugin에 주입(네이티브 키 — 서버 시크릿과 별개, EAS 시크릿/환경).
- **어댑터 타임아웃·서킷**(NFR-U3-RES-01)은 앱 소관(Resilience4j) — 인프라는 SI §8.2 **A12(어댑터 실패율·서킷 open)·A11(쿼터 80%)** 알람을 제공(U3가 첫 소비자).

### 7.2 POI 정본 저장 — RDS 스키마 확장 (Flyway)

- **저장소**: SI §3의 **기존 RDS PostgreSQL**(단일 `trippilot` DB·`public` 스키마) 재사용 — **신규 DB 인스턴스·클러스터 없음**. U1 Flyway 파이프라인(배포 전 일회성 `app_migrate` 태스크)으로 U3 마이그레이션 적용.
- **U3 신규 테이블(~8~9개)**: POI 정본, POI 소스 매핑(place_id N:1 D17), POI 스냅샷(사용자 확정), 탐색 캐시(허용 소스 파생 — §7.3 참조), 위시리스트, 등록 숙소, 숙소 ID 소스 매핑, 클릭 로그, 인기 장소 집계.
- **소스별 저장 정책(D13 — §6.4)을 스키마로 강제**:
  - **정본 테이블**: TourAPI 등 **허용 소스만** 영구 저장(D13 (a)) + **보존 정책 컬럼**(`source`·`retention_policy`·`cached_at`) — P4 캐싱 조건 확정 시 소급 축소.
  - **지도 API 소스**: 정본 테이블에 **영구 저장하지 않음**(D13 (b)·P2) — 인메모리 TTL(§7.3). 아키텍처 테스트로 "지도 소스 → 정본 테이블 쓰기" 부재 검증.
  - **스냅샷**: 최소 필드 + 정책 컬럼(NFR-U3-DATA-03) — P2 확정 전 지도 소스 스냅샷 보류.
- **볼륨**: 지역당 POI 5천 × 국내 지역 수십 = 수십만 행(NFR-U3-SC-01) — SI §3 gp3 20GB(오토스케일 100GB 상한) 내 충분. 정본 테이블은 **지역·카테고리 인덱스**(탐색·closed-set 후보 조회 G115), 파티셔닝 여지 확보(NFR-U1-SC-03 상속).
- **소유권·권한**: U3 테이블은 `app_user` 표준 DML(SI §3 롤 모델) — 법정 로그·동의 증적 같은 append-only REVOKE 대상 아님(U3는 일반 CRUD). 위시리스트·등록 숙소 소유권은 앱 레벨 accountId 스코핑(NFR-U3-SEC-05).

### 7.3 단기 캐시 계층 결정 — 인메모리 vs ElastiCache (규모 대비 권고)

지도 API 소스·핫 탐색 결과의 단기 TTL 캐시(D13 (b)·§6.4)의 저장 위치를 비교한다.

| 기준 | **A. 인메모리 Caffeine (권고)** | B. ElastiCache Redis | C. DB TTL 테이블 |
|---|---|---|---|
| 정본 적합성 | 캐시=최적화(미스=외부 재호출, NFR-U1-AV-03 상속) — 태스크 간 불일치 무해 | 충족(공유) | 충족 |
| 지도 약관(P2) | **영구 저장 표면 0** — 프로세스 메모리·TTL만(리스크 최소) | 외부 저장소 상주(TTL이나 표면↑) | DB 영구 스토리지 상주(약관 경계 근접) |
| 신규 AWS 리소스 | **0** | **+1 서비스·SG·서브넷·월 ~$25** | 0(RDS 내, 정리 잡 추가) |
| 규모 정합(G142) | **적합** — 수 RPS·POI 5천/지역, 태스크 2GB 힙 여유(SI §1.2) | 이득 0(운영 표면만) | RDS I/O·정리 부하 추가 |
| 쿼터 보호(G196) | 태스크당 캐시 → 최대 태스크 수배 호출(0.5 RPS에서 무의미) | 공유로 최소화 | 공유 |

- **확정 권고: A(인메모리 Caffeine · 단기 TTL) · ElastiCache 미도입.** 근거: (1) 지도 API 소스는 영구 저장 금지(D13·P2)라 인메모리 TTL이 약관 리스크를 최소화, (2) 캐시는 정본이 아니므로 태스크 간 불일치는 재호출로 무해(NFR-U1-AV-03), (3) 신규 AWS 리소스 0(G142). TourAPI 허용분은 §7.2 정본 테이블이 폴백 소스를 겸한다(별도 캐시 불요).
- **B(ElastiCache) 비채택**: 값·규모 대비 월 비용·운영 표면 증가가 이득 상회 — SI §4와 동일 트리거(인증 계열 포함 실측 50 RPS 초과 또는 U11 WebSocket 팬아웃)로 통합 재평가. **C(DB TTL) 비채택**: 지도 API 데이터를 DB 영구 스토리지에 두는 것이 P2 약관 경계에 근접.

### 7.4 배치 잡 — 기존 ShedLock 스케줄러 재사용 (신규 리소스 0)

U3 스케줄 잡은 SI §1.2 확정 모델(앱 프로세스 내 Spring `@Scheduled` + **PostgreSQL ShedLock 분산 락**)을 재사용한다 — **별도 워커 서비스·EventBridge 없음**(전부 일 1회 저부하).

| 잡 | 주기 | 내용 | 근거 |
|---|---|---|---|
| POI 정본 동기화 | **일 1회** | TourAPI 정적 콘텐츠 수집·정규화·canonical 발급(G133) | G196 |
| 인기 장소 집계 | **일 1회** | 최근 7일 저장+방문 가중합(홈 '인기 장소' 카드 공급 — U2 슬롯) | G2 |
| OTA 정적 콘텐츠 갱신 | **일 1회** | 정적 콘텐츠 갱신(라이브 가격 제외 — 온디맨드 PAT-U3-03) | G196 |
| 커버리지 게이트 측정 | 주기(일 1회) | 지역별 좌표 확보율·영업시간 채움률 산출 → 커스텀 메트릭(§7.5) | G192 |

- ShedLock 락으로 태스크 2개 중복 실행 배제(SI §1.2). 배치 커넥션 풀은 탐색 처리 풀과 분리(U1 PAT-PERF-02 벌크헤드 상속 — 배치가 탐색 커넥션 고갈 방지). **전환 트리거**: 배치가 탐색 API p95에 간섭 시 EventBridge Scheduler + ECS RunTask 분리(SI §1.2 트리거 상속).

### 7.5 커버리지 게이트 측정·외부 알람 (SI §8.2 구체화)

- **커버리지 메트릭(NFR-U3-GATE-02)**: 측정 잡이 지역별 좌표 확보율·영업시간 채움률을 CloudWatch **커스텀 메트릭**(`cloudwatch:PutMetricData` — SI §7.1 태스크 역할 허용)으로 발행 → 대시보드(SI §8.2 `trippilot-{env}-ops` 위젯 추가) + 출시 게이트 판정. 임계 미달(좌표 <95%·영업시간 <70%, G192)은 P3 주의 알람.
- **외부 API 알람(SI §8.2 — U3가 첫 소비자)**:

| 알람(SI 프레임) | U3 소스 | 임계 | 단계 |
|---|---|---|---|
| A11 외부 쿼터 80% | 카카오·TourAPI·네이버 호출량 vs 문서화 쿼터 | 80% | P2 |
| A12 어댑터 실패율·서킷 open | 카카오·TourAPI·네이버 어댑터 | 실패율 >20%(5분) 또는 서킷 open | P2 |
| U3 커버리지 게이트 미달 | 지역별 채움률 | 좌표 <95% / 영업시간 <70% | P3 |

- 알람 라우팅은 SNS `trippilot-{env}-alerts` → OPS-02 프로세스(SI §8.2). 쿼터 한도는 P4(TourAPI)·벤더 콘솔 확인 후 문서화(NFR-U3-SC-05).

### 7.7 배포 — shared 파이프라인 재사용

U3는 **별도 배포 아키텍처 문서를 만들지 않는다** — 전역 배포 파이프라인 정본(U1 스캐폴드 생성)을 재사용한다. 단, **클라이언트 지도 SDK는 네이티브 모듈이므로 EAS 빌드 델타가 있다**(U2와 달리 순수 JS가 아님).

- **서버(M3·M4·M5·M7 모듈 + POI 마이그레이션)**: `server/**` 경로이므로 `server-ci.yml`·`server-deploy.yml` **수정 없이** 커버(U1이 "유닛별 모듈 추가는 워크플로 수정 불요"로 완성). POI 정본·매핑·캐시 테이블 추가는 Flyway forward-only(신규 테이블 추가 = N-1 호환 규칙 "자유", OPS-01). 롤백=버전 고정 재배포(GD-4)·롤링(GD-5) 상속. 외부 시크릿(§7.1)은 배포 전 Secrets Manager 등록.
- **모바일(U3 화면 + `shared/map` 카카오 지도 SDK)**: `apps/mobile/**` 경로 — `mobile-ci.yml`(tsc·ESLint·Jest+fast-check) 커버. **단, 카카오 지도 네이티브 SDK config plugin은 네이티브 모듈 추가라 development/preview EAS Build 재빌드가 필요**하고 **expo-updates OTA로는 반영 불가**(네이티브 변경). 리스크 완화: config plugin 검증을 U3 초기 **스파이크로 선행**, 실패 시 목록/WebView 지도 폴백 빌드. 지도 앱 키는 EAS 시크릿 주입(§7.1).
- **환경**: dev(자동)·prod(승인 게이트) 승격 상속. 외부 API 키·딥링크 화이트리스트·폴백 순서 remote config는 환경별 독립(SI §9).

**U3 배포 델타**: 서버는 기존 워크플로 흡수(0 델타). 모바일은 **카카오 지도 SDK config plugin으로 인한 네이티브 재빌드**(EAS)가 유일한 델타 — OTA 아닌 스토어/개발 빌드 경로.
